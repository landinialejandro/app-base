Objetivo del documento

Este documento fija reglas de trabajo para mantener consistencia técnica y funcional en el proyecto app-base.

Su finalidad es:

unificar criterios de desarrollo

evitar duplicaciones conceptuales

mantener coherencia entre módulos

facilitar escalabilidad

reducir deuda técnica innecesaria

Estas convenciones aplican a todos los módulos nuevos salvo decisión explícita en contrario.

1. Principio general del proyecto

app-base es una plataforma SaaS multi-tenant orientada a gestión empresarial.

La prioridad actual del proyecto es:

construir una base funcional, consistente y extensible

La regla general de esta etapa es:

primero resolver lo operativo de forma clara y usable

luego profundizar estructura, normalización y sofisticación

No se debe complejizar un módulo antes de que exista una necesidad real de negocio o técnica.

2. Convención de arquitectura multi-tenant

La arquitectura multi-tenant definida para el proyecto no debe modificarse salvo decisión explícita.

Reglas obligatorias

cada entidad de negocio debe pertenecer a un tenant

el aislamiento debe resolverse automáticamente por tenant_id

las consultas deben quedar protegidas por scope global

el route model binding debe respetar el tenant activo

las entidades empresariales deben implementar soft delete

Patrón obligatorio para entidades del negocio

Toda entidad del negocio que pertenezca a un tenant debe seguir este patrón:

use TenantScoped

use ResolvesTenantRouteBinding

use SoftDeletes

y su tabla debe incluir:

tenant_id

timestamps

deleted_at

3. Convención sobre tipos de entidades

El sistema distingue cuatro grandes grupos:

Entidades del sistema

Son técnicas o estructurales del SaaS.

Ejemplos:

users

tenants

memberships

roles

permissions

Entidades del negocio

Representan actores o elementos centrales del dominio empresarial.

Ejemplos:

parties

products

orders

documents

payments

Entidades operativas

Representan trabajo, ejecución o seguimiento.

Ejemplos:

projects

tasks

work_orders

Entidades de soporte

Complementan a otras entidades.

Ejemplos:

attachments

notes

addresses

sequences

Regla general

Los modelos deben nombrarse:

en inglés

en singular

en PascalCase

Ejemplos válidos

Project

InventoryMovement

6. Convención de nombres visibles al usuario

La interfaz no está obligada a reflejar literalmente el nombre técnico del modelo o tabla.

Se permite y se recomienda utilizar nombres funcionales más naturales para el usuario.

Ejemplo principal del proyecto

Nombre técnico:

Party

Tabla:

parties

Nombre visible en interfaz:

Contacto / Contactos

Regla

el nombre técnico se conserva en código

el nombre funcional puede adaptarse para mejorar usabilidad

7. Convención sobre Party
Regla central

Party es la entidad base para representar personas u organizaciones vinculadas al negocio.

Antes de crear nuevas tablas como clients, suppliers o employees se debe evaluar si el caso corresponde a una variante de Party.

Party representa conceptualmente

cliente

proveedor

empleado

contratista

empresa

persona

contacto comercial

Regla de interfaz

En pantallas y textos visibles al usuario se utilizará:

Contacto

Contactos

No debe usarse “Party” como término visible en UI salvo motivo técnico excepcional.

8. Convención sobre especialización prematura

No crear tablas separadas cuando el caso todavía pueda resolverse con una entidad base simple.

Ejemplos

No crear todavía:

clients

suppliers

employees

contacts

company_contacts

addresses

salvo que exista una necesidad concreta e irreversible de separación.

Regla

primero resolver con Party

si más adelante se requiere especialización, se agrega por capas

9. Convención de CRUD

Todos los módulos CRUD deben seguir una estructura simple, consistente y predecible.

Archivos mínimos esperados

modelo

migración

controller

requests si corresponde

rutas

vistas

Vistas mínimas esperadas

index

create

edit

show

_form

Rutas

Se prioriza resource controller cuando el módulo realmente sea CRUD estándar.

10. Convención de controllers
Regla general

Los controllers de módulos simples deben seguir el mismo patrón que ProjectController.

Criterios

métodos claros y directos

sin lógica excesiva en controller

validación simple en request o validate()

uso de redirect con mensajes simples

Patrón visual esperado

index()

create()

store()

show()

edit()

update()

destroy()

Regla adicional

Si el módulo pertenece al tenant, el controller debe trabajar dentro del contexto:

auth + tenant

y pasar el tenant a las vistas cuando corresponda.

11. Convención de vistas
Regla actual del proyecto

En esta etapa se utilizarán vistas simples, sin sobrearquitectura visual.

Criterios

HTML Blade simple

sin layouts complejos si aún no existen

sin componentes prematuros

máxima legibilidad

Convención de listados

Los listados deben seguir el patrón de projects:

sin columna de acciones

el nombre enlaza al detalle

editar y eliminar viven en la vista show

Convención de detalle

La vista show debe incluir:

link de edición

botón de eliminación

tenant visible cuando corresponda

link para volver al listado

detalle simple de campos

12. Convención de rutas
Regla general

Las rutas deben nombrarse con el nombre técnico del módulo.

Ejemplos:

projects.index

projects.show

parties.index

parties.show

Regla multi-tenant

Las rutas de entidades del negocio deben quedar dentro del grupo middleware correcto.

Ejemplo:

auth

tenant

Nunca debe exponerse un módulo multi-tenant fuera del contexto tenant.

13. Convención de validación
Regla general

Si el módulo es muy simple, puede usarse validate() directo en controller.

Si el módulo tiene más campos o crecerá, conviene usar FormRequest.

Regla actual

No agregar validaciones complejas si todavía no hay necesidad concreta.

Primero resolver:

required

string

max

nullable

boolean

email

Luego evolucionar a reglas más profundas si el negocio lo exige.

14. Convención sobre tenant_id
Regla obligatoria

Si una entidad depende de tenant, su columna tenant_id debe ser compatible con el tipo real de tenants.id.

En este proyecto:

tenants.id es UUID

Por lo tanto:

cualquier tenant_id vinculado a tenants debe ser UUID compatible

No debe usarse foreignId() para tenant_id mientras tenants.id sea UUID.

15. Convención de soft deletes

Toda entidad empresarial debe incorporar SoftDeletes salvo razón muy justificada.

Motivos

evitar pérdida de información

facilitar recuperación

permitir trazabilidad mínima

Regla

si una entidad es del negocio y opera dentro del tenant, por defecto lleva deleted_at

16. Convención de fields “kind”, “type” o similares

Cuando una entidad base agrupe distintos usos, se puede utilizar un campo simple de clasificación en la primera etapa.

Ejemplo:

Party.kind

Regla

usar clasificación simple primero

evitar submodelado prematuro

Recomendación

preferir kind antes que type cuando ayude a evitar ambigüedad técnica o conceptual

17. Convención de interfaz y lenguaje funcional

La interfaz debe utilizar lenguaje claro, cercano y orientado al usuario final.

Regla

preferir términos funcionales en lugar de nombres técnicos internos

Ejemplos

Party → Contacto

Create Party → Nuevo contacto

Show Party → Detalle del contacto

Regla adicional

El lenguaje visible debe ser consistente entre módulos.

18. Convención para módulos futuros

Antes de crear un módulo nuevo se debe verificar:

si realmente es una entidad nueva

si puede resolverse como especialización de Party

si puede resolverse como una variante documental

si conviene implementarlo ahora o dejarlo en TODO

Regla de decisión

si no es imprescindible ahora, dejarlo registrado y seguir con lo operativo

21. Regla sobre decisiones postergadas

Toda decisión no implementada pero detectada como importante debe registrarse en TODO-DEL-PROYECTO.

Regla

no frenar el avance actual por resolver todo hoy

registrar

continuar

retomar en el momento adecuado

22. Regla final del proyecto

Toda nueva implementación debe priorizar:

coherencia con la arquitectura actual

mínima duplicación

máxima claridad

crecimiento ordenado

consistencia con módulos previos

Si existe duda entre dos caminos, se prioriza el que:

sea más simple

sea más consistente con lo ya implementado

genere menos deuda técnica inmediata

no rompa la arquitectura multi-tenant

onvenciones de base visual

Las vistas del proyecto deben construirse sobre una base visual común.

Toda vista de módulo debe:

extender el layout principal del sistema

respetar la estructura común de páginas

utilizar la hoja de estilos global del proyecto

mantener consistencia visual con el resto de los módulos

La base visual mínima del proyecto se definirá en el documento:

BASE-VISUAL-DEL-PROYECTO – app-base

No deben crearse vistas aisladas con estructuras completamente distintas salvo necesidad justificada.
