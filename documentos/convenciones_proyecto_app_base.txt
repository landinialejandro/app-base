OBJETIVO DEL DOCUMENTO

Este documento define las convenciones estructurales y de desarrollo del proyecto app-base.

Su objetivo es mantener consistencia técnica, previsibilidad arquitectónica y coherencia de código a medida que el sistema crece.

Las convenciones aquí definidas deben respetarse al crear:

nuevos módulos
nuevas entidades
nuevas vistas
nuevos controladores
nuevos flujos operativos
<<END SECTION>>

CONVENCIONES GENERALES

El proyecto sigue los principios de:

simplicidad
claridad estructural
baja duplicación
coherencia entre módulos
previsibilidad del comportamiento del sistema

Cuando existan varias alternativas técnicas, se debe priorizar siempre la opción más simple y consistente con el resto del proyecto.

Convención documental consolidada:

la base documental técnica del proyecto debe tender a una estructura homogénea y navegable

los documentos técnicos activos deben utilizar:

bloque estructural de metadatos
TÍTULO como nombre humano del documento
DOC_SLUG como identificador técnico canónico
DOC_VERSION como versión general del documento

las referencias internas entre documentos deben tender a resolverse mediante:

<DOC_REF slug="...">Título visible</DOC_REF>

no deben utilizarse nombres de archivo físicos como referencia narrativa dentro del cuerpo del documento

las carpetas baks y auditoria no forman parte de la base documental activa navegable

Convención adicional consolidada para contenido orientado a usuario:

los textos dirigidos a usuario final deben utilizar lenguaje no técnico
deben priorizar claridad, accesibilidad y comprensión inmediata
deben construirse con frases cortas y cierre conceptual claro
pueden incorporar líneas de refuerzo tipo “👉 …” como patrón de cierre

este criterio aplica a:

documentación de onboarding
ayudas contextuales
contenidos UX embebidos

Convención operativa adicional:

las pruebas manuales con Tinker forman parte del proceso real de validación técnica del proyecto

cuando el caso toque permisos, constraints, multi-tenancy o runtime real, debe utilizarse la guía operativa vigente y no una prueba abstracta improvisada

<DOC_REF slug="guia_tinker_proyecto_app_base">Guía operativa de pruebas con Tinker</DOC_REF>

Convención modular consolidada:

los módulos del sistema deben entenderse como unidades funcionales cerrables equivalentes a mini apps internas del tenant

cada módulo debe tender a describirse mediante:

núcleo propio
contrato de entrada
contrato de salida
contrato de seguridad
oferta de integración

esta convención no obliga a que todos los módulos publiquen desde el inicio todas sus capacidades reutilizables

sí obliga a que, cuando las publiquen o consuman, lo hagan bajo contrato claro y sin reabrir arbitrariamente el núcleo de otros módulos

cuando una capacidad reusable pertenezca contractualmente a un módulo oferente, su implementación no debe quedar en una ubicación global ambigua que borre esa pertenencia modular

si la capacidad además requiere arbitraje operativo complejo, el módulo oferente debe publicar la unidad completa de integración:

resolución backend
representación visual
consumo homogéneo
ubicación física consistente con su identidad contractual

Caso vigente adicional:

cuando una capacidad transversal madura y adquiere identidad propia, debe evolucionar hacia módulo explícito sin perder compatibilidad con consumidores existentes.

Inventory entra en esta lectura desde su transición de capacidad operativa validada hacia módulo autónomo proyectado.

Convención operativa adicional consolidada:

la complejidad contractual, técnica u operativa debe resolverse prioritariamente dentro del sistema y no trasladarse al usuario como carga mental, burocracia o secuencia innecesaria de pasos manuales

la UX debe tender a acciones simples, comprensibles y de bajo roce aun cuando el backend deba ejecutar consecuencias múltiples, validaciones, trazabilidad o documentos derivados

Convención adicional de ownership funcional:

cuando un frente operativo publique una surface oficial reutilizable, los módulos consumidores deben adelgazarse progresivamente y abandonar variantes locales equivalentes

no corresponde sostener formularios, tablas, historiales o arbitrajes paralelos como resolución principal si ya existe una capacidad oficial del módulo oferente

Convención adicional de semántica funcional:

no toda semántica numérica o direccional puede reutilizarse entre dominios distintos por similitud aparente

cuando un flujo combine stock, ejecución operativa, reversión o pendiente, cada lectura debe resolverse con la semántica propia del dominio correspondiente

no debe asumirse equivalencia automática entre signo de stock y avance operativo de línea

Convención adicional de eventos deduplicados:

cuando el sistema reutilice un registro existente para deduplicar alertas, tareas o eventos repetidos, el nuevo hecho no debe quedar invisible

la reutilización de persistencia puede ser válida

la representación visible del evento debe actualizarse para reflejar la ocurrencia más reciente relevante

Convención adicional de configuración visible para owner:

si un módulo depende de habilitación, permisos o configuración operativa por tenant, el sistema debe ofrecer una vía visible, real y consistente para que owner pueda administrarlo

esa vía no exige necesariamente una pantalla propia aislada por módulo

sí exige que la capacidad sea administrable dentro de la experiencia real del owner sin depender de enlaces forzados, supuestos ocultos o intervención técnica externa

Convención adicional de visibilidad modular:

navbar, dashboard y accesos rápidos no deben forzar la aparición de módulos por existencia técnica

la visibilidad correcta debe resultar de:

módulo habilitado
permiso efectivo
estado real de configuración cuando corresponda

dashboard no reemplaza las surfaces oficiales de un módulo

puede actuar como acceso o resumen, pero no como sustituto de la navegación estructural ni de la administración real del frente

Convención adicional de host visual modular:

cuando una pantalla anfitriona consuma capacidades publicadas por otros módulos, debe mantener separadas:

su identidad nuclear propia
las capacidades embebidas invitadas
los contratos de los módulos oferentes consumidos

el host puede componer la experiencia, pero no apropiarse de la lógica operativa ni recomponer localmente una surface ya publicada por otro módulo

Convención adicional de host show estructurado:

las show pages surface-driven deben leerse como hosts con zonas explícitas de montaje y no como contenedores planos definidos solo por type.

La gramática doctrinal vigente distingue al menos:

header_actions
summary_items
detail_items
zona tab coordinada

Cuando el host trabaja con slots, el contrato publicado por la offer debe declararlos de forma explícita.

No debe suponerse inferencia automática del lugar de montaje.

Convención adicional de referencia evolutiva:

si dos hosts equivalentes muestran distinta madurez contractual, no debe degradarse el más avanzado para alinearlo al más viejo.

La convergencia correcta del ecosistema debe tender hacia el patrón más maduro ya validado.

Caso consolidado actual:

appointments.show expresa una estructura host más avanzada que orders.show para show pages surface-driven.

Convención adicional de raíces internas del tenant:

cuando varios frentes necesiten usuarios internos activos de la empresa, no debe abrirse por inercia un módulo users ni una ability inexistente para resolverlo.

La salida correcta puede ser una pieza reusable del plano tenant con responsabilidad explícita de directorio interno.

Convención adicional de migración arquitectónica:

si una familia reusable ya evolucionó hacia una nueva generación aprobada, no debe cerrarse el trabajo con soluciones híbridas mezclando generación nueva y vieja.

La migración válida puede ser gradual por módulo, pero cada módulo debe cerrar coherencia interna completa.

ESTRUCTURA BASE DEL PROYECTO

Este bloque define la ubicación canónica de código, vistas, assets y documentación técnica relevante del proyecto.

No busca inventariar todo el repositorio.

Busca fijar dónde vive cada responsabilidad principal para evitar búsquedas imprecisas, suposiciones técnicas o dispersión de criterios.

APP

app/ concentra la lógica principal del sistema.

Ubicaciones relevantes:

app/Models
modelos del sistema y del negocio

app/Models/Concerns
traits estructurales del dominio, por ejemplo TenantScoped y ResolvesTenantRouteBinding

app/Http/Controllers
controladores del flujo web

app/Http/Middleware
middleware operativo transversal, por ejemplo ResolveTenant

app/Http/Requests
validaciones por formulario o acción

app/Policies
wrappers mínimos de autorización

app/Support/Auth
capa técnica de acceso y arbitraje, incluyendo Security, resolvers y acceso por tenant o módulo

app/Support/Catalogs
catálogos cerrados del sistema

app/Support/Navigation
motor y builders de navegación contextual y estructural

app/Support/Docs
lectura y escritura técnica de la base documental del proyecto

app/Support/Documents
servicios propios del flujo documental

app/Support/Inventory
servicios, cálculo de inventario y evolución modular de stock

app/Support/Orders
servicios, payloads o arbitrajes propios del flujo de orders y sus integraciones reusables

app/Support/Parties
servicios, payloads o arbitrajes propios del flujo de parties y sus integraciones reusables

app/Support/Projects
lecturas agregadas o utilidades propias de projects

app/Support/Tasks
lecturas y visibilidad operativa de tasks

app/Support/Tenants
contexto funcional del tenant y raíces reusables propias de la empresa activa

Caso consolidado actual:

TenantUserDirectory.php como directorio reusable de usuarios internos activos del tenant

app/Services
servicios operativos específicos cuando no correspondan a Support transversal

app/View/Components
componentes Blade con backing class cuando aplique

app/Actions/Fortify
acciones del flujo de autenticación y cuenta global

app/Providers
providers del sistema y de paquetes integrados

Regla adicional de frontera interna:

tenant pertenece al núcleo interno del sistema, multi-tenancy y seguridad.

No debe tratarse como capacidad surface-driven modular publicada hacia el ecosistema funcional.

RESOURCES

resources/ concentra fuentes editables del frontend Laravel.

Ubicaciones relevantes:

resources/views
vistas Blade del sistema

resources/views/components
componentes visuales reutilizables verdaderamente globales

resources/views/layouts
layouts base del sistema

resources/views/<modulo>
vistas por módulo operativo

resources/views/<modulo>/components
capacidades visuales reutilizables publicadas por un módulo oferente cuando su pertenencia contractual sea propia del módulo y no global del sistema

resources/views/tenants
vistas del contexto tenant, memberships, perfil y permisos

resources/views/docs
visor técnico documental

resources/views/landing
flujo público y marketing

resources/views/print
partials de impresión reutilizables

resources/views/svg
fuente Blade de iconografía y marca SVG

Regla adicional vigente:

cuando inventory publique tablas de saldos, historial, formularios de movimientos o fichas propias, esos recursos deben vivir en resources/views/inventory y no dispersos entre products u orders.

Regla adicional de consumidores:

si products, orders u otros módulos solo consumen inventory, deben hacerlo mediante includes o surfaces oficiales del módulo y no mediante copias locales mantenidas en paralelo.

Regla adicional para hosts surface-driven:

cuando un show page funcione como host modular, la estructura anfitriona debe leerse desde la propia vista del módulo anfitrión y no desde parciales ambiguos globales que oculten su gramática real de montaje.

La organización de loops, slots y armado coordinado de tabs pertenece al host y debe conservarse como responsabilidad visible del módulo anfitrión.

ROUTES

routes/ concentra las rutas del sistema.

Ubicaciones relevantes:

routes/web.php
declaración explícita de rutas del core web

routes/console.php
registro de comandos de consola

Regla:

mientras siga vigente la convención actual del proyecto, las rutas CRUD del core deben declararse explícitamente en web.php.

PUBLIC

public/ concentra assets públicos y archivos servidos directamente.

Ubicaciones relevantes:

public/css/app-base.css
base visual global obligatoria

public/css/app-patterns.css
patrones visuales reutilizables y neutrales

public/css/modules/
estilos específicos por módulo cuando todavía no corresponda promoverlos a patrón global

public/css/landing.css
estilos propios del flujo público

public/js/app-base.js
comportamiento global reutilizable del sistema

public/js/landing.js
comportamiento del flujo público

public/js/print.js
comportamiento específico de impresión

public/brand/
assets públicos de marca

Regla:

los assets públicos consolidados del sistema deben tender a vivir aquí y no dispersarse en ubicaciones paralelas ad hoc.

DATABASE

database/ concentra persistencia estructural y datos base del proyecto.

Ubicaciones relevantes:

database/migrations
estructura persistida del sistema

database/seeders
datos base, módulos y escenarios de demo

database/seeders/Modules
seeders organizados por módulo

database/factories
factories para pruebas y generación de datos

Regla:

toda decisión estructural persistida debe validarse contra migrations reales y no por memoria del modelo.

DOCUMENTOS

documentos/ forma parte activa del proyecto y no debe tratarse como material externo.

Ubicaciones relevantes:

documentos/*.txt
base documental activa del proyecto

documentos/varios
material auxiliar no doctrinal principal

la base documental activa del proyecto está compuesta por los documentos canónicos identificados por DOC_SLUG.

Las carpetas auxiliares no deben confundirse con la base doctrinal principal ni con el visor documental canónico.

REGLA FINAL

CONVENCIONES DE ACCESO POR MÓDULO Y ROLES

La autorización operativa del sistema debe implementarse respetando la arquitectura central definida para acceso y permisos.

Reglas de implementación:

no hardcodear permisos en controllers, policies o blades si existe resolución central

las policies deben consumir la capa central de autorización y no replicar lógica

los controllers deben delegar en authorize(...) o Security y no implementar lógica paralela

las vistas no deben inferir permisos

las vistas deben respetar permisos reales mediante @can solo como verificación final cuando la capacidad no dependa de contexto

los alcances visibles en UI deben resolverse mediante catálogos o resolvers y no reconstruirse manualmente

la habilitación de módulos depende de la capa central de acceso por tenant y no debe asumirse por defecto

la UI de permisos del tenant debe operar únicamente sobre módulos habilitados previamente

Regla crítica validada:

no se debe usar autorización abstracta cuando la capacidad depende de contexto

no utilizar authorize('create', Model::class) en estos casos

debe utilizarse la capa central con parámetros explícitos

La existencia de subject class-string no debe asumirse como condición doctrinal para resolver create contextual.

Si el contrato requiere contexto funcional obligatorio, la autorización válida se resuelve por Security con ese contexto.

Caso consolidado adicional:

cuando el contrato persistido incluya constraints funcionales reales, esas constraints deben arbitrar create, view, update y delete de forma consistente y no limitarse a una sola acción

En orders, constraints.allowed_kinds constituye segmentación efectiva del contrato y no metadata decorativa.

Regla adicional consolidada:

cuando una regla de estado o trazabilidad define si una acción principal existe realmente, esa regla debe reflejarse también en la autorización efectiva central.

No debe quedar únicamente en controller, Blade o Policy si Security es el kernel central del sistema.

Caso vigente:

orders.update y orders.delete deben considerar estado operativo y trazabilidad real además de permisos nominales, scope y constraints.

Regla de dataset:

cuando la visibilidad depende de permisos, el controller debe usar Security::scope(...)

no deben utilizarse queries paralelas ni filtros manuales

Regla de consistencia obligatoria:

view_any debe ser igual o más amplio que view

no debe existir un contrato donde view_any quede más restringido que view para el mismo módulo

Regla modular adicional:

todo módulo debe poseer su propio contrato de seguridad

ese contrato se consume siempre bajo arbitraje de la capa central

la publicación de capacidades reutilizables por parte de un módulo no crea canales paralelos de autorización

las integraciones entre módulos deben respetar el contrato del módulo oferente y no reinterpretarlo desde el módulo consumidor

CONVENCIONES DE CONTRATO MODULAR

Todo módulo operativo del sistema debe poder describirse mediante un contrato modular explícito.

Ese contrato no reemplaza la implementación ni la documentación específica del módulo.

Capas mínimas esperadas del contrato modular:

núcleo
contrato de entrada
contrato de salida
contrato de seguridad
oferta de integración

NÚCLEO

El núcleo del módulo describe aquello que el módulo resuelve por sí mismo.

Incluye, según corresponda:

entidad o agregado principal
flujo base
CRUD o acciones principales
reglas operativas propias
navegación estructural base
recursos hijos propios
comportamientos internos del módulo

Regla:

el núcleo no debe reabrirse por cada integración nueva solicitada desde otros módulos salvo que cambie realmente el contrato interno del propio módulo.

CONTRATO DE ENTRADA

El contrato de entrada define qué puede recibir el módulo desde el sistema o desde otros módulos.

Puede incluir, según el caso:

contexto funcional
ids relacionales válidos
trail compatible
datos mínimos de precompletado
origen de creación contextual
disparadores internos del sistema

Reglas:

no asumir entradas implícitas no declaradas
no abrir canales paralelos de entrada por conveniencia de implementación
no resolver create contextuales con patrones distintos si el mismo contrato ya puede absorberlos

si un create requiere contexto funcional obligatorio, no debe documentarse ni implementarse como create abstracto genérico

CONTRATO DE SALIDA

El contrato de salida define qué devuelve o expone el módulo hacia otros módulos o hacia capas consumidoras del sistema.

Puede incluir, según el caso:

referencia resumida del registro
summary reusable
payload mínimo de lookup
relación enlazable
dataset embebible
acción derivada
compatibilidad con trail contextual
payload arbitrado para integraciones complejas

Regla:

el contrato de salida debe tender a estabilizarse para que otras integraciones consuman capacidades del módulo sin reabrir su núcleo.

OFERTA DE INTEGRACIÓN

La oferta de integración expresa qué capacidades reutilizables publica el módulo para consumo de otros módulos.

Puede incluir, según el caso:

selector reusable
tabla embebida reusable
summary enlazable
acciones contextuales
create contextual compatible
bloques de relación derivados
catálogos o adaptadores de integración
formularios operativos reutilizables cuando pertenezcan al núcleo del módulo oferente
acciones por fila cuando el módulo deba intervenir recursos mostrados por un host externo

Reglas:

un módulo puede publicar oferta de integración aunque todavía no exista un consumidor actual
la ausencia temporal de consumidor no invalida la oferta
la publicación de la oferta pertenece al módulo oferente y no debe imponerse desde afuera como excepción ad hoc

si una capacidad reusable ya fue publicada por el módulo oferente, el consumidor debe priorizar su consumo antes de crear una variante local

si la capacidad reusable requiere arbitraje operativo complejo, su publicación válida no se agota en la vista Blade

debe incluir también una capa de resolución o payload backend publicada por el módulo oferente

si la capacidad pertenece contractualmente al módulo oferente, su ubicación física debe mantener esa pertenencia modular y no diluirse en una carpeta global ambigua

Regla adicional consolidada de identidad modular:

cuando un frente operativo evoluciona desde capacidad dispersa hacia módulo con identidad propia, sus surfaces oficiales deben migrar al propio módulo oferente

las vistas, partials, formularios y tablas que expresen ese frente ya no deben quedar repartidos como solución oficial en módulos consumidores

Regla adicional de abstracción sana:

la homogeneización estructural es deseable, pero no debe sobreabstraer semántica real.

Se admite centralizar infraestructura repetida mediante piezas comunes livianas.

Caso vigente validado:

BuildsSurfaceOffers como convención común para *SurfaceService.

Puede concentrar:

linkedOffer(...)
embeddedOffer(...)
unpackHostPack(...)

No debe absorber:

reglas del dominio
payloads funcionales
resolvers específicos
decisiones UX del módulo
semántica propia de cada contrato publicado

Un SurfaceService puede ser extenso si esa extensión expresa semántica real y no simple boilerplate evitable.

Regla de cierre adicional:

una capacidad reusable no debe considerarse realmente cerrada mientras existan superficies secundarias relevantes que todavía reconstruyan localmente su arbitraje o sus estados operativos

el cierre válido exige revisar también tablas, partials, formularios embebidos y demás superficies derivadas que consuman esa integración

Casos doctrinales actuales:

linked-order-action como superficie reusable de orders
linked-party-action como superficie reusable de parties
linked-asset-action como superficie reusable de assets

Regla adicional de experiencia operativa:

las acciones de alto nivel deben tender a resolver la intención del usuario en un solo paso comprensible

el usuario expresa intención y el sistema ejecuta internamente las consecuencias compatibles con el contrato del módulo

no corresponde trasladar al frontend la lógica contractual crítica, ni exigir al usuario que reconstruya manualmente secuencias internas que el sistema puede arbitrar de forma segura

Caso operativo adicional consolidado:

cuando un consumidor reciba desde un módulo oferente una lectura operativa reusable por fila, no debe reinterpretar localmente qué estado corresponde, qué acción rápida exponer ni cuándo una línea queda readonly

debe consumir la unidad oficial publicada por el módulo oferente

caso vigente relevante:

orders consumiendo contexto operativo por línea publicado por inventory

Regla adicional sobre acciones por fila:

si una tabla nativa del host necesita renderizar acciones pertenecientes a otro módulo, la salida correcta es que el módulo oferente publique también esa capacidad como row action oficial dentro de su oferta contractual

el host presta superficie visual de montaje

el módulo oferente conserva arbitraje operativo, validaciones y semántica de la acción

Aclaración modular adicional:

la lectura contractual del ecosistema debe distinguir entre:

núcleo interno del módulo
puerta de enlace modular oficial
consumo externo por otros módulos

cuando un módulo actúe como superficie anfitriona o hub visual de una capacidad ajena, eso no lo convierte en dueño de esa lógica ni habilita naming, estructuras o responsabilidades centradas en el consumidor para piezas que pertenecen al oferente

Regla adicional de publicación reusable mínima:

cuando una surface reusable embebida se publique para consumo transversal, debe apoyarse en un contrato mínimo estable que permita arbitraje homogéneo del ecosistema.

Como mínimo, esa publicación debe poder describirse mediante:

type
slot
key
label
targets
priority
view
needs
resolver

El campo slot forma parte del contrato explícito de la offer.

No debe tratarse como dato opcional implícito cuando el host consumidor organiza sus zonas de montaje mediante slots.

Regla adicional para resolver reusable:

cuando exista resolver como parte del contrato publicado, debe devolver al menos:

count
data

ese resolver pertenece al módulo oferente y no debe quedar recompuesto por el host consumidor

Regla adicional para hostPack:

cuando una surface embebida reusable madure como patrón transversal, debe priorizarse un hostPack genérico basado en contexto estable del host y no claves ad hoc centradas en un consumidor puntual

caso vigente:

consumo embebido apoyado en record y trailQuery como base genérica del host

Regla adicional de semántica cruzada:

cuando una integración exponga métricas o cantidades pertenecientes a planos funcionales distintos, cada valor debe provenir de la semántica oficial de su dominio

no corresponde reutilizar cálculos de stock como sustituto automático de ejecución operativa, ni viceversa

Regla adicional de compatibilidad host:

si el host consumidor todavía resuelve primero por type y luego por slot, la offer publicada debe respetar esa gramática real.

No alcanza con target correcto ni con view válida si la combinación type + slot no coincide con la estructura efectiva del host.

Caso consolidado actual:

las relaciones compactas hacia appointments.show deben publicarse como linked + summary_items para entrar en la zona correcta del host vigente.

Regla adicional de raíces internas del tenant:

cuando una necesidad repetida no constituye módulo funcional sino dataset interno reusable de la empresa activa, la salida correcta puede ser una pieza del plano tenant.

No debe forzarse un módulo artificial solo por reutilización de consultas internas.

Caso consolidado:

directorio de usuarios internos activos del tenant.

Regla adicional de migración coherente:

si una capacidad reusable ya migró a una nueva generación aprobada, no debe mezclarse soporte nuevo con componente viejo, payload nuevo con vista vieja ni fallback legacy con cierre declarado como definitivo.

La migración válida exige coherencia contractual completa del módulo afectado.

Regla adicional de puerta contractual blanda:

cuando un módulo necesite contacto real con lógicas externas pero aún no corresponda abrir una abstracción mayor, puede consolidar una estación contractual propia y explícita como único punto autorizado de acople extra-módulo.

Esa puerta no reemplaza el flujo principal del módulo ni desplaza su núcleo.

Su función es concentrar delegaciones externas necesarias sin contaminar controllers, blades ni servicios internos nucleares.

Caso vigente:

orders mediante OrdersHooks.

Regla adicional para módulos con estados operativos:

si el estado de un registro define su condición real de operación, ese estado forma parte del contrato efectivo del módulo.

No debe tratarse como una validación lateral.

Caso vigente:

orders.closed bloquea edición y eliminación directa, pero admite reapertura explícita autorizada a approved.

orders.cancelled bloquea edición y eliminación directa y permanece como estado terminal no reabrible dentro de la regla validada.

Regla adicional para agrupadores operativos internos:

cuando un módulo incorpore una entidad persistente agrupadora de eventos o acciones, esa entidad no se convierte automáticamente en surface pública, módulo autónomo ni dependencia directa de consumidores externos.

Debe definirse si pertenece al núcleo interno, a la puerta de enlace modular o a una oferta explícita de integración.

Caso vigente:

inventory_operations pertenece al núcleo interno de inventory como agrupación operativa contextual.

No reemplaza inventory_movements.

No desplaza la trazabilidad unitaria de stock.

Su exposición hacia otros módulos solo corresponde si inventory publica una capacidad oficial para ello.

Regla terminológica vigente para inventory:

operation refiere a agrupación operativa contextual.

movement refiere a impacto unitario de stock.

Una operation puede reunir uno o múltiples movements.

No deben tratarse como equivalentes ni sustituirse conceptualmente.

CONVENCIONES DE CLASIFICACIÓN MODULAR

Los módulos del sistema pueden cumplir distintas posiciones funcionales dentro del ecosistema general.

Esta clasificación no reemplaza el modelado del dominio ni el contrato de seguridad.

Ayuda a evitar decisiones de integración inconsistentes.

Tipos conceptuales principales:

módulo base del dominio
módulo transversal
módulo autónomo
módulo agrupador
módulo derivado

MÓDULO BASE DEL DOMINIO

Es una pieza conceptual fundamental del negocio y no debe tratarse como módulo opcional ordinario si el proyecto ya lo reconoce como base intrínseca del sistema.

Caso actual consolidado:

parties

MÓDULO TRANSVERSAL

Es un módulo cuya función principal es integrarse en otros módulos o recursos y no necesariamente vivir como navegación principal independiente.

Caso actual consolidado:

attachments

MÓDULO AUTÓNOMO

Es un módulo con identidad operativa propia que puede vivir por sí mismo aunque también participe en relaciones con otros módulos.

Regla:

su agrupación o vinculación con otros módulos no debe degradarlo a simple subrecurso si conserva función propia.

Casos doctrinales relevantes:

tasks
appointments
products
assets
orders

MÓDULO AGRUPADOR

Es un módulo cuya función principal es ordenar, contextualizar o dar lectura agregada sobre otros módulos sin agotar completamente la identidad de esos módulos relacionados.

Caso doctrinal relevante:

projects como agrupador de tasks

MÓDULO DERIVADO

Es un módulo que nace o se entiende principalmente como formalización, consecuencia o prolongación operativa de otro módulo más nuclear.

Caso doctrinal relevante:

documents respecto de orders

Reglas generales:

la clasificación funcional del módulo debe ayudar a definir su contrato y su superficie de integración
la clasificación no autoriza excepciones de seguridad
la clasificación no debe forzar jerarquías falsas de dependencia existencial cuando el módulo conserva identidad propia

Aclaración adicional de host visual:

actuar como host visual dentro de una pantalla no constituye una clasificación modular nueva ni altera por sí mismo la identidad del módulo

es una posición de integración dentro del ecosistema

no convierte al host en dueño del contrato del módulo invitado

Aclaración adicional de plano tenant:

no toda pieza reusable del sistema debe clasificarse como módulo funcional.

Cuando la responsabilidad real sea contexto interno de empresa, dataset reusable o directorio operativo del tenant, la clasificación correcta puede pertenecer al plano tenant/transversal y no al catálogo modular del negocio.

Aclaración adicional sobre recursos hijos relevantes:

que un recurso delegado gane peso transversal o trazabilidad fuerte no modifica automáticamente la clasificación modular vigente.

Caso relevante:

order_items continúa siendo entidad hija de orders aunque participe como anclaje operativo frente a inventory.

<<END SECTION>>

<<SECTION: CONVENCIONES DE INTEGRACIÓN ENTRE MÓDULOS>>

CONVENCIONES DE INTEGRACIÓN ENTRE MÓDULOS

La integración entre módulos debe resolverse como consumo de contratos y no como invasión del núcleo del módulo destino.

Reglas:

si un módulo consumidor necesita relacionarse con otro módulo, debe priorizar el consumo de la superficie contractual ya expuesta por el módulo oferente

no debe reabrirse el núcleo del módulo destino por cada integración nueva salvo que cambie su propio contrato interno

no deben proliferar variantes paralelas de:

selectores
tablas embebidas
bloques summary
acciones contextuales
create contextuales
payloads mínimos de referencia

si el módulo oferente ya dispone de una capacidad reusable válida, debe reutilizarse antes de crear una variante específica en el consumidor

si la capacidad reusable todavía no existe y realmente pertenece al módulo oferente, su incorporación debe tratarse como evolución del propio módulo oferente

no debe tratarse como excepción permanente alojada en el módulo consumidor

Regla de consumo seguro:

toda integración entre módulos sigue sujeta al arbitraje central de seguridad

Regla de crecimiento:

un módulo puede dejar publicadas capacidades de integración como oferta futura aun cuando todavía no existan consumidores concretos

esto favorece estabilidad contractual y reduce refactors futuros dispersos

Regla adicional consolidada:

cuando una integración transversal compleja combine:

estado del módulo destino
permiso efectivo
existencia o no de registro relacionado
posible create contextual
estados UX mutuamente excluyentes
navegación contextual requerida

debe priorizarse una capacidad compartida publicada por el módulo oferente antes que duplicar lógica en múltiples consumidores

esa capacidad debe revisarse como unidad completa de integración:

arbitraje backend
payload resuelto
representación visual
ubicación doctrinal correcta
consumo homogéneo

no alcanza con reutilizar solo una vista Blade si la decisión de estados sigue dispersa entre consumidores

cuando la capacidad reusable pertenezca contractualmente a un módulo oferente, su implementación no debe quedar en una ubicación global ambigua que borre su identidad modular

Regla adicional consolidada sobre forma de publicación:

si la relación reusable es fuerte, múltiple o de lectura pesada, debe priorizarse el patrón de embebido modular completo publicado por el módulo oferente

ese patrón debe tender a expresarse mediante wrapper modular tipo partials.embedded-tabs y una pieza interna reusable de listado o tabla

si la relación reusable es compacta, puntual o 1 a 1, debe priorizarse una acción contextual compacta publicada por el propio módulo oferente

no debe forzarse tab pesada cuando la relación no lo necesita

Casos consolidados actuales:

linked-order-action como superficie reusable publicada por orders
linked-party-action como superficie reusable publicada por parties
linked-asset-action como superficie reusable publicada por assets

Caso adicional consolidado:

appointments puede publicar embebido reusable oficial para relaciones fuertes y acción compacta cuando el vínculo consumidor sea puntual

Caso adicional vigente:

inventory debe publicar sus propias superficies de stock, movimientos e historial desde su módulo y los consumidores deben integrarlas sin duplicar tablas locales de stock o variantes paralelas de historial.

documents puede publicar surfaces embebidas oficiales de lectura documental desde su propio módulo para ser consumidas por hosts como orders.show sin volver a tabs relacionales manuales del consumidor.

Regla operativa adicional de cierre:

si una superficie consumidora todavía resuelve localmente acceso, apertura o visualización de una relación ya cerrada por el módulo oferente, el frente de integración no debe considerarse homogéneo ni cerrado

Regla adicional de simplicidad:

la integración correcta no debe exponer al usuario pasos internos, decisiones técnicas o combinaciones contractuales que el sistema pueda resolver por sí mismo

la UI debe presentar acciones simples y comprensibles

la lógica crítica de arbitraje, estado y consecuencias debe permanecer centralizada en backend o en la capacidad oficial publicada por el módulo oferente

Regla adicional para frentes operativos por línea:

cuando una integración reutilizable se consuma por fila dentro de una tabla operativa, el consumidor no debe rearmar arbitraje local de estados, readonly o acciones rápidas

debe recibir y expresar el contexto oficial publicado por el módulo oferente

esto aplica especialmente cuando la línea sigue siendo recurso delegado del padre pero ya posee trazabilidad operativa iniciada

Regla adicional de naming modular:

no nombrar nuevas piezas desde el módulo consumidor cuando la capacidad pertenece al módulo oferente

el naming debe expresar el dominio dueño de la capacidad y no el punto accidental desde el que se la consume

por lo tanto, deben evitarse superficies, controllers, partials o servicios nombrados desde orders, documents u otros consumidores cuando la capacidad pertenece doctrinalmente a inventory u otro módulo oferente

Regla adicional de organización de responsabilidades:

controller expresa la surface HTTP
support o service expresa lógica reusable y arbitraje propio del módulo
consumidor invoca capacidad externa sin absorber su implementación interna

Regla adicional de semántica de datos:

si una integración muestra simultáneamente stock, ejecución, pendiente o reversión, cada indicador debe obtenerse desde la regla oficial de su plano funcional

no deben reciclarse helpers pensados para stock como fuente universal de estados operativos

Regla adicional para host visual modular:

cuando una pantalla anfitriona consuma varias capacidades embebidas desde distintos módulos, debe mantener visible la separación entre su núcleo propio y las capacidades invitadas

el host no debe convertirse en superficie paralela dueña de esas integraciones

debe actuar como compositor visual del ecosistema y no como reemplazo contractual del oferente

Regla adicional de compatibilidad real:

si el host consumidor implementa montaje por zonas, la integración no debe declararse solo en términos abstractos de relación reusable.

Debe verificar compatibilidad real entre:

type
slot
estructura del host
zona efectiva de montaje

Una integración puede estar correctamente orientada al target y seguir siendo invisible si slot no está definido o si la combinación type + slot no coincide con la gramática real del host.

Caso consolidado actual:

las relaciones compactas publicadas hacia appointments.show deben respetar la lectura linked + summary_items del host actual.

Regla adicional para datasets internos del tenant:

si un módulo necesita colaboradores internos activos, validación de pertenencia a la empresa o dataset interno reusable, debe consumir primero la pieza tenant correspondiente antes que reconstruir queries sobre memberships o depender de un módulo users no formalizado.

Regla adicional para hooks blandos:

si un módulo necesita intervención externa sobre un recurso propio, la integración no debe desparramarse entre controllers, blades y servicios nucleares.

Debe concentrarse en la estación contractual propia del módulo y desde allí delegar a piezas externas especializadas del frente correspondiente.

Caso vigente:

orders puede delegar hacia InventoryOrderItemHooks a través de OrdersHooks.

Regla adicional para tablas anfitrionas:

si una tabla del consumidor muestra filas propias pero necesita montar acciones de otro módulo, debe priorizarse el consumo de row actions oficiales publicadas por el oferente y no la reconstrucción manual de botones, permisos o estados dentro del host.

El host monta.

El oferente decide.

Regla adicional para estado superior del host:

cuando una integración actúa sobre recursos hijos dentro de un host operativo, no puede habilitar acciones que contradigan el estado superior del recurso padre.

Caso vigente:

inventory puede publicar operación por línea para order_items, pero esa operación queda subordinada al estado de la order.

Si la order está closed o cancelled, la línea no recupera operabilidad por sí misma.

Regla adicional para agrupaciones internas del módulo oferente:

si una integración dispara una acción que internamente genera o reutiliza una entidad agrupadora propia del módulo oferente, el consumidor no debe reconstruir ni condicionar esa agrupación.

Debe consumir la surface, servicio, hook o contrato publicado por el oferente.

Caso vigente:

orders puede actuar como host visual de acciones de inventory por línea, pero inventory conserva la resolución interna de inventory_operations e inventory_movements.

CONVENCIONES DE FLAGS DE SOPORTE MODULAR

Cuando una pantalla, formulario, partial o componente dependa de la existencia de otro módulo habilitado para el tenant, debe utilizar el flag o señal canónica de soporte modular vigente.

Reglas:

la validación no debe quedarse en la vista principal

debe propagarse hasta la última partial, include, tabla embebida o bloque visual que pueda exponer:

campos relacionales
tabs
enlaces
acciones
botones de alta
referencias visibles

no debe asumirse que una partial profunda conoce el estado modular por contexto implícito

el dato debe llegar explícitamente o resolverse por la vía canónica del sistema

si un módulo no está habilitado:

no deben renderizarse superficies residuales hacia ese módulo

cuando una capacidad reusable publicada por un módulo dependa además de arbitraje backend del oferente, el consumidor no debe recomponer localmente estados alternativos a partir del flag modular

el flag solo define soporte del módulo

la decisión operativa específica debe consumirse desde la capacidad publicada por el módulo oferente

Ejemplos típicos:

supportsOrdersModule
supportsAssetsModule
supportsPartiesModule

Regla adicional consolidada:

la correcta propagación del flag modular no reemplaza la obligación de consumir la capacidad reusable oficial en todas las superficies relevantes

si una partial secundaria conserva una implementación local residual, el cierre del frente modular debe considerarse incompleto aunque la vista principal ya esté alineada

Regla final:

la propagación completa de flags modulares forma parte de la calidad de cierre de un módulo
<<END SECTION>>

<<SECTION: ROLECATALOG Y MODULECATALOG>>
ROLECATALOG Y MODULECATALOG

RoleCatalog es el catálogo base mínimo y reservado del sistema para roles estructurales.

ModuleCatalog es la fuente de verdad para slugs de módulos cerrados del sistema.

Reglas:

no hardcodear slugs funcionales del sistema en controllers, policies, helpers o blades si ya existe catálogo técnico

los roles base del sistema deben resolverse desde RoleCatalog

los módulos base del sistema deben resolverse desde ModuleCatalog

los roles base sembrados al crear el tenant no deben modificarse ni eliminarse

los roles base cerrados del tenant no deben crecer por necesidades funcionales puntuales

las variantes funcionales futuras deben resolverse mediante perfiles funcionales y overrides, no creando nuevos roles base

a futuro el tenant podrá ajustar capacidades dentro de la arquitectura vigente, pero sin alterar la base reservada del sistema
<<END SECTION>>

<<SECTION: EVOLUCIÓN PREVISTA DE PERMISOS>>
EVOLUCIÓN PREVISTA DE PERMISOS

La implementación de permisos del sistema debe apoyarse en una capa centralizada compuesta por resolvers y catálogos del dominio.

Reglas:

toda nueva funcionalidad de autorización debe integrarse en la capa central existente

no deben crearse matrices paralelas ni lógica aislada por módulo

las policies deben mantenerse como wrappers mínimos de consumo de esa capa

los resolvers no definen el contrato de permisos

los resolvers deben resolver y consumir el contrato persistido, incluyendo scopes, constraints y visibilidad de dataset según lo definido en base de datos

la evolución del sistema de permisos debe respetar la arquitectura vigente sin introducir variantes incompatibles

<<SECTION: USO DE TENANT_ID EN FILLABLE>>
USO DE TENANT_ID EN FILLABLE

Como regla general, tenant_id no debe exponerse en formularios ni venir desde input del usuario.

El tenant siempre debe resolverse internamente mediante:

app('tenant')

y asignarse automáticamente por el sistema.

Algunos modelos padre del proyecto exponen tenant_id en fillable por consistencia histórica, pero igualmente no debe venir desde formularios públicos ni desde input del usuario.

En entidades hijas (por ejemplo items o registros dependientes), tenant_id tampoco debe exponerse desde formularios del usuario.

Sin embargo, si el registro hijo se crea internamente desde un flujo controlado del sistema y el tenant se hereda explícitamente desde la entidad padre, puede incluirse tenant_id en fillable para permitir la creación interna consistente mediante create().

Esto permite manejar correctamente casos como:

document_items creados desde documentos
order_items copiados hacia documentos derivados
<<END SECTION>>

<<SECTION: CONVENCIONES DE ENTIDADES HIJAS>>
SECTION_VERSION: 00007

CONVENCIONES DE ENTIDADES HIJAS

Las entidades hijas deben cumplir:

pertenecer a una cabecera
heredar tenant_id de la cabecera
no recibir tenant_id desde formularios del usuario

Ejemplos:

order_items
document_items

Reglas actualizadas:

cuando la entidad hija forme parte natural de la operación de la cabecera, debe tratarse preferentemente como recurso delegado al padre

la autorización de create, update y delete debe resolverse contra la capacidad update del recurso padre correspondiente

no debe modelarse una autorización autónoma innecesaria para la entidad hija si su ciclo de vida depende funcionalmente de la cabecera

si existiera una policy propia, debe limitarse a delegar explícitamente en el padre y no duplicar doctrina paralela

attachments sigue el mismo criterio general de delegación al recurso padre aunque no sea una entidad hija cabecera/detalle clásica

Regla modular adicional:

las entidades hijas no deben confundirse con módulos autónomos ni con módulos agrupados

que una entidad dependa funcionalmente de una cabecera no implica que todo módulo relacionado deba modelarse como entidad hija

no deben agregarse responsabilidades propias de cabecera al hijo por comodidad de implementación técnica

por ejemplo:

auditoría propia
capabilities independientes
ownership artificial
dataset autónomo

Caso consolidado adicional:

order_items debe delegar create, update y delete en orders.update incluyendo controllers, vistas embebidas, navegación de cancelar y verificaciones de pertenencia al padre

cualquier excepción a esta regla debe justificarse por necesidad real del dominio y no por comodidad de implementación

Aclaración adicional consolidada:

order_item puede actuar como anclaje operativo fuerte de un frente externo como inventory sin dejar de ser entidad hija delegada del padre

esa relevancia operativa no lo convierte en módulo autónomo

tampoco transfiere a order_item la semántica material, el cálculo neto ni el arbitraje operativo del módulo invitado

Regla adicional de endurecimiento operativo:

cuando una entidad hija ya posea trazabilidad material iniciada, su edición estructural no debe seguir tratándose como CRUD libre por defecto

las restricciones deben responder al estado real del recurso y al impacto operativo existente

Caso vigente relevante para order_items:

mientras la línea no esté completada ni cancelada puede seguir siendo editable dentro de límites compatibles con lo ya ejecutado

la cantidad no puede reducirse por debajo del ejecutado neto vigente

si existe trazabilidad iniciada no deben mutarse atributos de identidad operativa como product_id, kind o description

Regla adicional de subordinación al padre:

las reglas propias de una entidad hija operan únicamente dentro de la condición operativa permitida por su padre.

Si el padre se encuentra en estado readonly o terminal, la entidad hija no puede recuperar operabilidad por sí misma.

Caso vigente:

order_items queda subordinado al estado operativo de orders.

<<END SECTION>>

<<SECTION: CONVENCIONES DE OWNERSHIP FUNCIONAL Y PERMISOS OPERATIVOS SIMPLES>>
CONVENCIONES DE OWNERSHIP FUNCIONAL Y PERMISOS OPERATIVOS SIMPLES

Cuando un módulo lo permita, puede adoptarse un criterio simple de ownership funcional del registro en lugar de una capa temprana de permisos compleja.

Reglas:

usuario normal gestiona los registros de su responsabilidad operativa

owner y admin pueden gestionar registros ajenos dentro del tenant

los borrados sensibles pueden reservarse a owner y admin aunque la edición o seguimiento sea más amplia

si un módulo tiene asignado directo, assigned_user_id define la responsabilidad operativa actual

si el registro puede reasignarse, el nuevo asignado pasa a ser responsable operativo

si no existe propietario directo, el control sensible queda en owner y admin o en la policy específica del módulo

Este patrón ya se considera válido para tasks.
<<END SECTION>>

<<SECTION: REGLA PARA MÓDULOS OPERATIVOS COMPARTIDOS DEL TENANT>>
REGLA PARA MÓDULOS OPERATIVOS COMPARTIDOS DEL TENANT

Cuando un módulo operativo compartido del tenant no tenga responsable operativo directo:

la policy debe permitir view, create y update a los roles y capacidades habilitados del módulo

delete debe quedar restringido a owner y admin

no debe inventarse ownership artificial mediante assigned_user_id, owner_user_id o restricciones por created_by si el módulo no lo necesita realmente

Esto aplica especialmente a módulos base o compartidos como:

parties
assets
products
orders

Aclaración conceptual consolidada:

orders debe tratarse como módulo operativo interno del tenant

documents no debe absorber órdenes de trabajo operativas ni duplicar la función interna de orders

CONVENCIONES DE SUPPORT Y HELPERS DE ACCESO

La lógica transversal de acceso debe centralizarse en Security.

Reglas:

no implementar lógica de permisos en controllers o blades

no duplicar resolución de acceso

Security es el punto único de consumo de autorización

los resolvers no deben consumirse directamente

los helpers deben facilitar acceso a Security y no redefinir lógica

las classes bajo app/Support/Auth deben actuar como acceso técnico y no como fuente alternativa

Regla operativa adicional:

si una corrección de autorización afecta create contextual, constraints o semántica general de consumo, debe resolverse en la capa central y no mediante helper ad hoc ni parche por módulo.

Regla adicional para ownership funcional:

si una corrección impacta comportamiento operativo propio de un módulo, debe resolverse en service o support del módulo dueño de la capacidad

no corresponde fijar semánticas de inventory, orders u otros dominios desde helpers genéricos del consumidor

Regla adicional de frontera tenant:

las piezas de directorio interno del tenant no reemplazan seguridad, policies ni middleware.

Su función es resolver datasets internos y verificaciones de pertenencia activa a la empresa.

La autorización efectiva continúa perteneciendo a la capa central de acceso.

<<SECTION: NAVEGACIÓN ESTRUCTURAL Y CONTEXTUAL>>
NAVEGACIÓN ESTRUCTURAL Y CONTEXTUAL

La navegación del sistema debe distinguir entre:

navegación estructural
navegación contextual

navegación estructural:
expresa la ubicación estable de la pantalla dentro del sistema o dentro de su módulo

navegación contextual:
expresa el origen funcional desde el cual el usuario llegó a una entidad relacionada, cuando ese origen sigue siendo relevante para la tarea actual

Reglas generales:

toda pantalla debe tener una resolución estructural válida como base, aunque en ciertos flujos pueda mostrarse en modo contextual

la navegación contextual solo debe aplicarse cuando exista un flujo relacionado real y útil para la continuidad de uso

no debe inventarse un origen contextual si la pantalla fue abierta de forma directa, desde index propio, desde búsqueda o desde navegación general del módulo

el breadcrumb del sistema es uno solo, pero debe poder resolverse en modo estructural o en modo contextual según el caso

si ya existe una capa central reutilizable para resolver contexto de navegación, no duplicar esa lógica en múltiples controllers, blades o helpers dispersos

controller resuelve contexto
vista consume contexto

no reconstruir breadcrumb de negocio desde url()->previous() ni desde supuestos del historial del navegador

breadcrumb, retorno al origen y redirects posteriores a acciones deben mantenerse consistentes entre entidades relacionadas

la preservación de contexto UX debe aplicarse con criterio y solo cuando aporte continuidad real de uso

como regla general:

usar navegación estructural por defecto

usar navegación contextual solo como override útil cuando el origen funcional deba preservarse

cuando no exista contexto válido o el origen no agregue valor operativo, debe mostrarse navegación estructural normal
<<END SECTION>>

<<SECTION: RESOLUCIÓN TÉCNICA DEL TRAIL DE NAVEGACIÓN>>
SECTION_VERSION: 00003

RESOLUCIÓN TÉCNICA DEL TRAIL DE NAVEGACIÓN

Reglas:

el mecanismo central vigente del proyecto para breadcrumb contextual es NavigationTrail

los módulos que participen en navegación contextual deben apoyarse en builders o resolutores propios compatibles con NavigationTrail, sin crear variantes paralelas de criterio

controller resuelve trail

vista consume trail

si una pantalla relacionada recibe un trail válido, debe poder heredarlo, extenderlo o colapsarlo sin reconstruir manualmente la navegación desde parámetros sueltos, historial del navegador o deducciones implícitas de relaciones del modelo

los nodos del trail deben almacenar URLs limpias de la entidad actual

no debe serializarse el propio trail dentro del campo url de los nodos

el trail se propaga solo en links, forms y redirects relevantes

cuando el trail vuelva a un nodo ya existente, debe colapsarse de forma limpia y no duplicada

cuando un módulo ya esté migrado a NavigationTrail, no debe seguir usando mecanismos legacy equivalentes o redundantes, incluyendo:

url()->previous()
context_type / context_id
NavigationContext
reconstrucción manual por relaciones del modelo

tasks participa plenamente como origen contextual legítimo hacia orders

appointment y task deben seguir la misma convención técnica para abrir o crear orders relacionadas

en tablas embebidas o listados contextuales, cuando el contexto real de navegación sea la fila y no solo el contenedor, los links profundos a entidades relacionadas deben usar el trail de la fila

en tablas embebidas o relacionales, el link principal de la fila debe construir el trail desde el contexto del contenedor más el nodo de la fila

los links relacionados de esa misma fila deben reutilizar ese trail contextual ya construido y no reiniciar el contexto desde el contenedor ni desde mecanismos manuales

los nodos transitorios de formulario como create, edit, items.create o items.edit no deben persistir al volver a show después de store o update exitoso

las acciones cancelar dentro de formularios contextuales deben respetar el trail completo vigente y no volver por defecto al show estructural del módulo

Regla adicional consolidada para capacidades reusables embebidas:

cuando una tabla reusable o bloque embebido se consuma dentro de otro módulo, no debe reiniciar navegación desde una base local del módulo oferente si el flujo vigente sigue anclado en el contenedor consumidor

debe heredar el trailQuery contextual recibido y propagarlo hacia show, acciones por fila, cancelar y redirects relevantes

esto aplica especialmente a tablas reutilizables consumidas dentro de tabs o cards de relación

las decisiones internas de implementación pueden evolucionar, pero la convención estable es:

motor central reutilizable
builders compatibles por módulo
controller resuelve
vista consume
sin deducción desde historial
sin duplicación por módulo
sin serialización recursiva del trail dentro de nodos

Para describir ese formato dentro de documentos activos debe utilizarse una convención de metarrepresentación y no la sintaxis ejecutable real.

CONVENCIÓN DE METARREPRESENTACIÓN DEL FORMATO

TOKEN_APERTURA

representa el símbolo inicial utilizado por los delimitadores estructurales reales del sistema.

TOKEN_CIERRE

representa el símbolo final utilizado por los delimitadores estructurales reales del sistema.

CABECERA_DE_REEMPLAZO

representa la línea inicial que indica reemplazo de una sección existente en un documento identificado por DOC_SLUG.

CABECERA_DE_NUEVA_SECCION

representa la línea inicial que indica propuesta de alta de una nueva sección en un documento identificado por DOC_SLUG.

SECCION_ANCLA_DE_INSERCION

representa la instrucción obligatoria que indica después de qué sección existente debe insertarse una nueva sección.

Debe identificar la sección ancla por su nombre exacto.

INICIO_DE_SECCION

representa la apertura del bloque estructural de una sección usando el nombre exacto correspondiente y los delimitadores oficiales del sistema.

FIN_DE_SECCION

representa el cierre del bloque estructural de una sección usando los delimitadores oficiales del sistema.

SECTION_VERSION_LINE

representa la línea de versión interna obligatoria de la sección.

REGLAS OPERATIVAS

La salida automatizada real debe materializar estas piezas exactamente con la sintaxis oficial vigente del proyecto.

Dentro de documentos activos no deben escribirse delimitadores reales ni bloques ejecutables de ejemplo si pueden ser interpretados por el parser o por el visor documental como secciones válidas.

Toda respuesta generada en modo automatizado debe seguir el formato oficial real y no esta representación abstracta.

CONTRATO VÁLIDO DE REEMPLAZO

Toda operación de reemplazo debe incluir:

CABECERA_DE_REEMPLAZO
INICIO_DE_SECCION
SECTION_VERSION_LINE
contenido completo de la sección
FIN_DE_SECCION

No se permiten:

reemplazos parciales
fragmentos internos
cambios sin bloque completo
cambios sin SECTION_VERSION

CONTRATO VÁLIDO DE NUEVA SECCIÓN

Toda operación de nueva sección debe incluir obligatoriamente:

CABECERA_DE_NUEVA_SECCION
SECCION_ANCLA_DE_INSERCION
INICIO_DE_SECCION
SECTION_VERSION_LINE
contenido completo de la nueva sección
FIN_DE_SECCION

No se permiten:

altas sin sección ancla
altas con ubicación implícita
altas con posición inferida por contexto
altas sin SECTION_VERSION
altas con nombre de ancla aproximado

REGLAS OBLIGATORIAS DE SALIDA AUTOMATIZADA

devolver siempre bloques completos

usar nombre exacto de sección cuando ya exista

usar nombre exacto de sección ancla cuando se trate de una inserción

usar DOC_SLUG exacto

respetar delimitadores oficiales vigentes

incluir SECTION_VERSION con formato numérico homogéneo del proyecto

no mezclar explicaciones dentro del bloque automatizado

si el modo solicitado es automatizado, priorizar salida limpia y procesable

Cualquier desviación de formato debe considerarse salida no compatible con automatización.

CRITERIOS PARA NUEVAS SECCIONES

Una nueva sección solo debe crearse cuando aporte orden real al documento.

Criterios válidos:

el tema no encaja limpiamente en secciones existentes

el contenido tendrá continuidad futura

se evita mezclar responsabilidades distintas

el nuevo bloque mejora lectura, mantenimiento o auditoría

el tema posee identidad conceptual propia dentro del documento

No crear nuevas secciones cuando:

solo cambia una línea menor

el contenido puede integrarse limpiamente en una sección vigente

la separación agrega ruido estructural

se intenta reflejar una necesidad transitoria de una sola sesión

Reglas adicionales:

cada sección debe tender a una responsabilidad principal

el nombre debe ser claro, estable y descriptivo

evitar renombres cosméticos innecesarios

antes de crear una sección nueva, revisar si el contenido pertenece en realidad a otro documento del sistema

si el documento debe explicar el formato técnico de salida automatizada, debe hacerlo de manera descriptiva y no reproduciendo dentro de su propio cuerpo los delimitadores reales que usa el parser

si una nueva sección va a proponerse para inserción automatizada, debe definirse además una sección ancla exacta ya existente en el documento destino

la inserción automatizada de nuevas secciones no debe depender de ubicación implícita, proximidad temática ni interpretación manual posterior

si no existe una sección ancla clara y estable, primero debe revisarse si corresponde realmente crear la nueva sección o si conviene reorganizar el documento antes de automatizar la inserción
