<<SECTION: RELACIONES CLAVE DEL ECOSISTEMA>>
SECTION_VERSION: 00008

RELACIONES CLAVE DEL ECOSISTEMA

Relaciones estructurales principales del sistema:

users → memberships → tenants

Relaciones operativas del negocio:

orders → order_items
documents → document_items
inventory_operations → inventory_movements
parties → party_roles

Relaciones transversales relevantes:

orders → inventory
documents → inventory
projects → tasks

Relaciones de contexto interno del tenant:

memberships → users
tenants → memberships

Relación conceptual adicional consolidada:

membership → party

Esta relación permite extender la pertenencia operativa con una ficha ampliada del negocio.

Reglas:

no es obligatoria
no altera relaciones estructurales principales
no convierte a party en entidad del sistema
no convierte a party en entidad de acceso

Lectura correcta:

el plano de identidad y acceso permanece en users y memberships

el plano del negocio permanece en parties y demás entidades

Aclaración vigente:

si la membership vinculada está activa, la party funciona como ficha ampliada tenant-scoped del colaborador interno.

Esa lectura no altera la relación principal:

users → memberships → tenants

Cuando una membership activa está vinculada a party, la ficha ampliada queda condicionada por esa relación activa.

Lectura validada:

party.kind = person
party_roles contiene employee
party.email se mantiene coherente con el usuario vinculado cuando corresponda

Aclaración operativa de parties:

parties → party_roles no es solo una relación descriptiva.

party_roles expresa la relación del contacto con la empresa y puede ser utilizada por el contrato de seguridad especializado de parties como dimensión efectiva de segmentación.

Regla:

party_roles no se convierte por eso en módulo autónomo ni en navegación propia.

Sigue siendo entidad hija o complementaria de parties.

Regla:

las relaciones del ecosistema deben leerse respetando los planos conceptuales:

identidad global
pertenencia operativa
operación del negocio

No deben mezclarse por conveniencia de implementación.

<<SECTION: FUNCIÓN DE CADA UNA>>
SECTION_VERSION: 00001

FUNCIÓN DE CADA UNA

users

Representa la identidad global del usuario en todo el sistema.

Un user puede pertenecer a múltiples tenants.

No pertenece a un tenant específico.

No debe leerse como módulo funcional autónomo del ecosistema operativo del tenant.

tenants

Representa cada empresa o espacio de trabajo dentro del SaaS.

Cada tenant contiene sus propios datos operativos.

Además de datos generales, puede contener configuración funcional en:

tenants.settings

Esto permite persistir contexto operativo del tenant cuando todavía no justifica entidad propia.

memberships

Representa la pertenencia de un user a un tenant.

Define acceso real por empresa.

Es la entidad correcta para resolver:

a qué empresa pertenece un usuario
qué estado tiene dentro de esa empresa
qué roles tiene dentro de esa empresa
qué perfil funcional persistido tiene dentro de esa empresa

También constituye la base real para resolver colaboradores internos activos del tenant.

Cuando el sistema necesite reutilizar ese dataset en múltiples frentes operativos, la salida correcta es una raíz reusable del plano tenant y no un módulo users del ecosistema funcional.

signup_requests

Representa solicitudes iniciales de alta o interés desde la landing pública.

No pertenece al dominio operativo interno del tenant.

invitations

Representa invitaciones dentro del flujo de acceso y onboarding del sistema.

Puede operar en dos modos:

owner_signup
member_invite

Cuando type = owner_signup puede existir sin tenant asociado.

Cuando type = member_invite sí se vincula operativamente a un tenant existente.

Por esta razón, no debe tratarse como una entidad tenant-scoped pura del negocio.

<<END SECTION>>

ENTIDADES DEL NEGOCIO

Son las entidades operativas propias de cada tenant.

Estas entidades deben estar aisladas por:

Aclaración vigente:

party_roles es entidad hija o complementaria de parties.

Permite expresar la relación de un contacto con la empresa sin sobrecargar parties.kind.

Aclaración adicional:

party_roles.role tiene incidencia operativa dentro del contrato de parties.

Puede utilizarse para segmentar visibilidad y acciones de parties mediante allowed_party_roles.

Esto no cambia su naturaleza de entidad hija o complementaria de parties.

<<SECTION: ENTIDAD BASE DEL NEGOCIO>>
SECTION_VERSION: 00003

ENTIDAD BASE DEL NEGOCIO

parties

Es la entidad base para personas u organizaciones dentro del sistema.

Debe absorber conceptualmente actores como:

clientes
proveedores
contactos
contratistas
terceros
colaboradores internos cuando corresponda

Regla actual:

no crear entidades nuevas si el caso puede resolverse extendiendo parties mediante su contrato vigente o entidades hijas correspondientes

parties no debe tratarse como un módulo opcional común, sino como entidad base transversal del dominio y módulo base intrínseco del sistema

Nombre técnico:

Party

Nombre visible sugerido en la interfaz:

Contacto

Regla adicional:

las relaciones hacia parties desde otros módulos deben respetar el estado activo del módulo destino y no deben exponerse en UI si el módulo parties no estuviera disponible

Aclaración estructural vigente:

parties.kind representa únicamente el tipo de contacto:

person
company

La relación del contacto con la empresa no debe comprimirse en parties.kind.

Debe expresarse mediante party_roles.

Aclaración operativa vigente:

party_roles.role expresa la relación con la empresa.

Valores vigentes:

customer
supplier
employee
other

Regla:

la segmentación funcional de seguridad de parties debe apoyarse en party_roles.role mediante allowed_party_roles y no en parties.kind.

<<SECTION: MODELO DE ACCESO Y PERTENENCIA>>
SECTION_VERSION: 00007

MODELO DE ACCESO Y PERTENENCIA

La pertenencia y acceso dentro del sistema se estructuran mediante:

users
memberships
tenants

Relación base:

users → memberships → tenants

Lectura:

users define identidad global
memberships define pertenencia operativa
tenants define contexto de empresa

Aclaración consolidada:

membership puede vincular opcionalmente una ficha ampliada en parties:

membership → party

Esta relación:

no altera el modelo de acceso
no convierte a party en entidad de acceso
no reemplaza la identidad del usuario

Reglas:

la operación válida requiere:

usuario autenticado
membership activo
tenant activo

la vinculación a party no interviene en autorización

la ficha ampliada pertenece al plano del negocio

Regla adicional:

la creación o vinculación de party desde membership debe ser explícita

no debe generarse automáticamente por acceso o navegación

Cuando exista membership activa vinculada:

la party debe leerse como ficha ampliada tenant-scoped del colaborador interno

esa ficha puede quedar parcialmente protegida para mantener coherencia con la identidad global y la pertenencia activa

si la membership deja de estar activa o se desvincula, la party deja de estar bloqueada por esa relación activa y vuelve a leerse como contacto común según su contrato vigente

Condiciones estructurales validadas para party vinculada a membership activa:

party.kind = person
party_roles contiene employee
party.email queda protegido respecto de edición libre desde parties cuando corresponde al usuario vinculado

Regla:

el modelo de acceso y pertenencia no debe mezclarse con el modelo de entidades del negocio

cada plano conserva su responsabilidad

<<END SECTION>>

<<SECTION: ROLES Y AUTORIZACIÓN>>
SECTION_VERSION: 00003

ROLES Y AUTORIZACIÓN

El modelo de autorización se encuentra consolidado en base a permisos V2 persistidos.

Componentes activos:

role_permission
membership_permission_overrides

Reglas:

los permisos efectivos se resuelven mediante resolvers y Security

las memberships no owner deben poseer al menos un rol base

el owner puede no tener roles explícitos en la relación many-to-many

el rol owner se resuelve implícitamente mediante lógica del resolver

los scopes y constraints forman parte activa del contrato persistido

Aclaración vigente para parties:

allowed_party_roles es la constraint funcional vigente para segmentar permisos sobre parties.

Se evalúa contra party_roles.role.

Regla:

allowed_kinds no aplica a parties.

<<END SECTION>>

<<SECTION: ACLARACIÓN SOBRE TENANT-SCOPED>>
SECTION_VERSION: 00001

ACLARACIÓN SOBRE TENANT-SCOPED

El trait TenantScoped debe usarse en entidades cuyo ciclo de vida pertenece completamente a un tenant.

Ejemplos de uso correcto:

Product
Order
Document
Branch
Party
Project
Task
Appointment
Asset
Attachment

Aclaración sobre Invitation

Invitation no debe tomarse como ejemplo simple de entidad tenant-scoped pura, porque su comportamiento depende del tipo de invitación.

Cuando type = owner_signup puede operar sin tenant asociado.

Cuando type = member_invite sí se vincula operativamente a un tenant existente.

No debe usarse TenantScoped en entidades del sistema como:

User
Tenant
Membership
SignupRequest

Aclaración adicional consolidada:

que un dato reusable sirva para múltiples módulos del tenant no lo convierte automáticamente en entidad del negocio ni en módulo funcional.

Este criterio aplica especialmente a la lectura de usuarios internos activos del tenant, que debe seguir apoyándose en memberships y en raíces del plano tenant.

<<END SECTION>>

<<SECTION: PARTIES>>
SECTION_VERSION: 00008

PARTIES

Party representa una entidad del negocio dentro del tenant.

Su función principal es actuar como ficha ampliada de contacto o actor relevante dentro del ecosistema operativo.

Puede representar:

clientes
proveedores
colaboradores internos cuando corresponda
otros actores del negocio

Reglas estructurales:

party es tenant-scoped
no pertenece al plano global del sistema
no reemplaza la identidad global de users
no reemplaza la pertenencia operativa de memberships

Clasificación vigente:

parties.kind representa únicamente el tipo de contacto.

Valores vigentes:

person
company

Regla:

parties.kind no debe representar customer, supplier ni employee.

Relación hija consolidada:

party → party_roles

party_roles expresa la relación del contacto con la empresa.

Valores vigentes:

customer
supplier
employee
other

Regla:

employee no es tipo de contacto ni naturaleza de party.

Employee es una relación operativa dentro de party_roles.

Aclaración operativa:

party_roles.role también es la dimensión efectiva utilizada por el contrato de seguridad de parties para segmentar visibilidad y acciones mediante allowed_party_roles.

Regla:

parties.kind no debe utilizarse como dimensión funcional de permisos de parties.

allowed_kinds no aplica a parties.

Relación consolidada:

membership → party

Esta relación es opcional y permite vincular una pertenencia operativa del tenant con una ficha ampliada de datos.

Reglas:

no toda membership debe tener party
no todo usuario interno del tenant debe convertirse en party
la relación no modifica la estructura base:

users → memberships → tenants

Lectura correcta:

users conserva identidad global
membership representa pertenencia operativa
party representa ficha ampliada del negocio

Cuando exista vínculo con membership activa:

party debe representar al colaborador interno
party.kind debe ser person
party_roles debe contener employee
email queda protegido respecto de edición libre desde parties
la ficha queda parcialmente bloqueada por la pertenencia activa

Regla operativa consolidada:

la ficha ampliada no se crea automáticamente

debe crearse o vincularse de forma explícita

la existencia del vínculo no habilita edición libre de atributos críticos sin validación backend

Regla adicional:

party.email no debe tratarse como campo libre cuando represente una ficha vinculada a un usuario interno con membership activa

no debe asumirse sincronización automática sin criterio definido

Distinción vigente:

employee con membership activa

employee manual sin user del sistema

El primer caso representa una ficha ampliada vinculada a pertenencia activa.

El segundo caso representa un contacto común con relación employee, cargable solo por usuarios autorizados.

Regla:

party mantiene su rol como entidad del negocio

no absorbe conceptualmente a users ni a memberships

La complejidad futura de parties debe crecer hacia entidades hijas o complementarias antes que sobrecargar el núcleo.

Líneas conceptuales futuras posibles:

party_relationships
party_emails
party_phones
party_addresses

Estas líneas no deben documentarse como implementación vigente mientras no existan en código real.

<<SECTION: TASKS>>
SECTION_VERSION: 00006

TASKS

Tipo conceptual:

módulo autónomo

Representa unidad operativa interna del sistema.

Funciones relevantes:

asignación
seguimiento
coordinación interna
derivación operativa
alertas o tareas automáticas futuras

Relaciones consumidas:

tenant
users internos resueltos por memberships activas del tenant
project opcional
party opcional
asset opcional

Relaciones ofrecidas:

seguimiento operativo
origen posible de orders
bloques de trabajo relacionados
métricas para projects
canal de alertas automáticas del sistema cuando la operación lo requiera

Superficie contractual conceptual posible:

selector de tarea
tabla embebida
summary operativo
acción derivar a order
lookup mínimo

Reglas:

puede vivir sin projects
su agrupación por projects no lo convierte en subrecurso dependiente

assigned_user_id debe leerse como colaborador interno del tenant y no como relación hacia un módulo users autónomo del ecosistema funcional

party_id sigue siendo relación opcional con actor externo del negocio y no debe confundirse con asignación interna

Aclaración adicional consolidada:

cuando una necesidad operativa automática del sistema requiera alertar o escalar trabajo interno, tasks puede actuar como destino natural de esas alertas sin perder su identidad modular

caso actual reconocido en el ecosistema:

alertas automáticas por stock negativo registradas a nivel metadata del sistema cuando la lógica operativa lo dispare

Aclaración adicional sobre deduplicación operativa:

si el sistema reutiliza una misma task para deduplicar alertas repetidas, esa reutilización no debe volver invisible el evento nuevo

la task persistida puede mantenerse

su contenido visible y legible debe actualizarse para representar el evento más reciente relevante del sistema

Aclaración adicional operativa:

si el flujo no recibe colaborador explícito, puede resolverse asignación al usuario actual como continuidad operativa válida, sin volver inviable el formulario por depender de una fuente externa no formalizada.

<<END SECTION>>

<<SECTION: APPOINTMENTS>>
SECTION_VERSION: 00006

APPOINTMENTS

Tipo conceptual:

módulo autónomo

Representa agenda operativa del tenant.

Relaciones consumidas:

tenant
party opcional
asset opcional
users internos resueltos por memberships activas del tenant cuando corresponda
order opcional mediante appointments.order_id

Relaciones ofrecidas:

agenda
contexto temporal
origen posible de orders
planificación operativa
embebidos relacionales reutilizables
host de relaciones compactas consumidas desde otros módulos

Superficie contractual conceptual posible:

selector de turno
agenda embebible
summary de turno
acción crear order contextual
embebido reusable de turnos relacionados
host show con zonas de montaje explícitas para integraciones compactas o pesadas según el caso

Reglas:

su identidad funcional no depende de orders aunque pueda derivar operaciones hacia ese módulo

cuando exista vínculo con order, ese vínculo real se persiste desde appointments.order_id y no desde orders.appointment_id

su superficie principal debe leerse desde calendar como entrada principal del módulo

appointments.index puede existir como superficie secundaria, pero no debe leerse como base principal promovida

assigned_user_id o lecturas equivalentes de colaborador interno deben apoyarse en pertenencia activa al tenant y no en un módulo users del ecosistema funcional

Aclaración contractual consolidada:

appointments publica oferta reusable oficial para el ecosistema

esa oferta puede consumirse como embebido fuerte desde assets y parties

y como integración compacta desde orders cuando la relación sea puntual

Aclaración estructural consolidada:

appointments.show debe leerse además como host real de relaciones compactas publicadas por otros módulos.

En particular, assets, orders y parties pueden exponer allí relaciones linked de resumen.

Esas relaciones no deben degradarse a pestañas pesadas por defecto.

Deben leerse como surfaces compactas consumidas por el host appointment dentro de su summary.

Aclaración modular consolidada:

products como maestro compartido del artículo

inventory como frente de operación de stock, saldos e historial

Esa evolución no reemplaza a products.

Lo ubica como base maestra estable dentro de una arquitectura más limpia entre catálogo y operación.

<<SECTION: INVENTORY>>
SECTION_VERSION: 00009

INVENTORY

Inventory expresa el frente funcional de stock, movimientos, ejecución material y trazabilidad operativa.

Piezas persistentes relevantes:

inventory_movements
inventory_operations

Lectura interna:

movement = impacto unitario de stock
operation = agrupación contextual de uno o múltiples movimientos relacionados

Inventory puede operar sobre líneas pertenecientes a otros módulos mediante anclajes válidos como:

order_items
document_items

Ese consumo no traslada semántica material al módulo host.

Inventory conserva sus reglas propias de ejecución, reversión, límites netos y trazabilidad.

<<END SECTION>>

<<SECTION: INVENTORY_OPERATIONS>>
SECTION_VERSION: 00001

INVENTORY_OPERATIONS

Tipo conceptual:

entidad interna del frente operativo inventory

Representa una agrupación operativa contextual resuelta automáticamente por Inventory.

No reemplaza a inventory_movements.

Su función es agrupar uno o múltiples movimientos unitarios/atómicos de stock dentro de una misma acción operativa temporal.

Relaciones consumidas:

tenant
origen funcional o documental cuando corresponda
línea origen cuando corresponda
usuario creador

Relaciones ofrecidas:

inventory_movements relacionados
contexto operativo agrupado para auditoría
lectura contextual desde detalle de movimiento

Campos conceptuales vigentes:

tenant
tipo de operación
origen
línea origen
notas
creador
timestamps

Reglas:

pertenece al tenant

participa del aislamiento operativo del sistema

no constituye por sí misma una nueva navegación principal obligatoria

no reemplaza la trazabilidad atómica de inventory_movements

puede reunir uno o múltiples inventory_movements dentro de una misma acción operativa temporal

se crea o resuelve internamente desde Inventory como consecuencia de una acción operativa válida

Aclaración:

inventory_operations expresa contexto operativo agrupado.

inventory_movements expresa impacto unitario/atómico de stock.

<<END SECTION>>

<<SECTION: ORDERS>>
SECTION_VERSION: 00010

ORDERS

orders representa la cabecera operativa principal del flujo comercial u operativo.

Concentra contexto, relación con terceros, lifecycle y agregación de líneas hijas.

Entidad hija natural:

order_items

Estados base persistidos vigentes:

draft
pending_approval
approved
closed
cancelled

Lectura vigente:

draft = cabecera en preparación
pending_approval = cabecera con contenido lista para revisión
approved = habilita operación real
closed = congela operación
cancelled = terminal

Orders mantiene centralidad doctrinal como patrón más maduro del ecosistema para cabeceras equivalentes.

La entidad conserva espacio evolutivo para separar nivel macro y subtipo específico mediante una estructura futura group + kind compatible con contratos existentes.

<<END SECTION>>

<<SECTION: ORDER_ITEMS>>
SECTION_VERSION: 00001

ORDER_ITEMS

Tipo conceptual:

entidad hija delegada de orders con relevancia operativa ampliada

Representa el detalle ejecutable o transaccional de una orden.

Relaciones consumidas:

order padre
product opcional
inventory_movements como trazabilidad operativa iniciada

Relaciones ofrecidas:

anclaje de ejecución por línea
lectura de pendiente
lectura de ejecutado neto
contexto para surtido, recepción y devolución

Reglas:

mantiene dependencia estructural respecto de orders

puede poseer estado propio de línea sin reemplazar el arbitraje superior de la order

sus reglas específicas solo operan si el estado de la orden padre lo permite

si la order está closed o cancelled, la línea no recupera operabilidad por sí misma

al reabrirse la order a approved, vuelven a regir las reglas propias de la línea según su estado y trazabilidad vigente

<<END SECTION>>

<<SECTION: DOCUMENTS>>

DOCUMENTS

documents representa la formalización documental derivada o complementaria de otros flujos operativos del sistema.

Puede vincularse con parties, assets, orders u otros contextos válidos según el caso.

Entidad hija natural:

document_items

Estados base persistidos vigentes:

draft
pending_approval
approved
closed
cancelled

Lectura vigente:

draft = cabecera en preparación
pending_approval = cabecera con contenido lista para revisión
approved = habilita operación documental real según contrato vigente
closed = congela operación
cancelled = terminal

Documents converge doctrinalmente hacia la gramática madura de Orders cuando la semántica coincide, sin perder identidad documental propia.

Las líneas documentales pueden actuar como anclaje de trazabilidad material a través de inventory_movements sin transferir ownership funcional a inventory.

<<END SECTION>>

<<SECTION: DOCUMENT_ITEMS>>
SECTION_VERSION: 00008

DOCUMENT_ITEMS

document_items es la entidad hija perteneciente a documents.

Representa líneas documentales con cantidades, importes y estado operativo de línea.

Estados vigentes de línea:

pending
partial
completed
cancelled

Lectura operativa:

una línea puede registrar ejecuciones parciales y reversiones parciales siempre que respete el ejecutado neto vigente.

El estado visible surge de la situación real de la línea y no solo de la existencia histórica de movimientos.

document_items conserva naturaleza delegada respecto de documents aunque posea relevancia operativa dentro de inventory.

Reglas de consistencia:

tasks y appointments constituyen orígenes funcionales válidos hacia orders dentro del ecosistema actual

la transición tasks → orders y appointments → orders expresa derivación operativa hacia un módulo autónomo y no subordinación conceptual del destino

la relación estructural persistida con tasks se resuelve desde orders.task_id

la relación operativa real con appointments se resuelve desde appointments.order_id

order_items forma parte del flujo operativo base como detalle dependiente de orders

document_items forma parte del flujo documental como detalle dependiente de documents

la transición appointments → orders debe respetar el estado activo del módulo orders

si orders no está habilitado para el tenant, el flujo no debe exponerse en UI

products → inventory_movements expresa la base operativa actual de inventory V1

ese flujo ya se encuentra validado en runtime real

la lectura conceptual vigente empieza además a separar inventory como frente operativo propio de stock sin degradar el rol maestro de products

orders y documents participan como contextos posibles de trazabilidad de movimientos y no como dueños del saldo maestro

order_items y document_items pueden participar como líneas origen de ejecución y trazabilidad, sin convertirse en módulos autónomos ni absorber la semántica de inventory

Aclaración transversal consolidada:

parties no solo inicia el flujo operativo base hacia orders

también actúa como nodo transversal desde el cual pueden leerse otras relaciones operativas relevantes del ecosistema, por ejemplo assets, appointments y documents, según módulos activos y contrato vigente

Aclaración adicional para assets:

assets no solo aporta contexto de datos a otros flujos

también puede actuar como punto de navegación reutilizable mediante capacidades oficiales publicadas por el propio módulo

Aclaración adicional para appointments:

appointments no solo participa en el flujo appointments → orders

también publica una capacidad reusable que permite ser consumido embebidamente desde assets y parties sin perder identidad modular propia

<<SECTION: MEMBERSHIPS, OWNER Y ROLES>>
MEMBERSHIPS, OWNER Y ROLES

La pertenencia operativa de un usuario a un tenant se resuelve por memberships.

Reglas actuales:

memberships.is_owner expresa la condición estructural de owner

owner no debe modelarse como un rol asignable ordinario dentro del flujo normal de edición

owner es único por tenant

la pivote membership_role se utiliza para roles delegables del tenant

admin es el rol delegable de gestión amplia

administrator es el rol delegable administrativo u oficina

operator es el rol operativo base por defecto

sales permanece como rol operativo específico

una membership activa no-owner no debe quedar sin rol

puede existir una membership owner sin registros en membership_role y eso no constituye un error del modelo

como evolución activa de la autorización, membership podrá incorporar además un profile_slug para refinar capacidades funcionales sin multiplicar roles base

La jerarquía efectiva debe entenderse así:

owner como condición estructural
roles delegables como capa operativa base
perfil funcional como especialización de trabajo
permisos concretos como capa de autorización

La complejidad funcional futura no debe resolverse creando nuevos roles base fuera de este conjunto cerrado.
<<END SECTION>>

REGLAS DE LECTURA

Las relaciones aquí expresadas son conceptuales y funcionales.

No implican automáticamente:

foreign keys obligatorias
pantallas existentes
botones visibles
navegación activa
permisos concedidos

Regla adicional consolidada:

cuando una relación se describa aquí como origen funcional, vínculo contextual o flujo operativo real, eso no obliga a una foreign key simétrica en ambos sentidos

puede existir relación estructural persistida desde un solo lado del vínculo y aun así existir integración válida y navegación real entre módulos

Ejemplo consolidado:

orders.task_id expresa vínculo estructural con tasks

appointments.order_id expresa vínculo operativo real con orders sin requerir orders.appointment_id

CONTRATO MODULAR CONCEPTUAL

Cada módulo principal del sistema debe poder describirse conceptualmente mediante:

núcleo
relaciones que consume
relaciones que ofrece
superficie contractual visible

NÚCLEO

Lo que el módulo resuelve por sí mismo.

RELACIONES QUE CONSUME

Lo que toma del ecosistema para ampliar su operación.

RELACIONES QUE OFRECE

Lo que deja disponible a otros módulos.

SUPERFICIE CONTRACTUAL VISIBLE

Piezas conceptuales reutilizables que otros módulos podrían consumir.

Ejemplos:

selector
tabla embebida
summary reusable
acción contextual
lookup mínimo

esta sección no define implementación técnica obligatoria

define cómo pensar el crecimiento del ecosistema modular
