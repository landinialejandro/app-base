<<SECTION: TIPO DE SISTEMA>>
SECTION_VERSION: 00002

TIPO DE SISTEMA

app-base es una plataforma SaaS multi-tenant desarrollada sobre Laravel.

Permite que múltiples empresas utilicen una misma aplicación manteniendo aislamiento lógico de datos y contexto operativo independiente para cada tenant.

El proyecto busca constituir una base ERP modular reutilizable, capaz de crecer por módulos, capacidades transversales y capas productivas satélite sin perder coherencia estructural.

El sistema no debe entenderse como una suma de CRUDs aislados.

Debe entenderse como una plataforma con gobierno global, contexto por tenant, base de dominio, capa transversal, módulos funcionales integrados y capacidades productivas especializadas que pueden orquestar módulos existentes sin duplicar su ownership.

Regla estructural:

app-base no busca copiar un ERP enterprise pesado ni competir por sobredimensionamiento funcional.

Debe adoptar principios robustos de ERP, CMMS y gestión operativa cuando correspondan, pero con una experiencia liviana, automatizable y accesible.

<<END SECTION>>

PROPÓSITO GENERAL DEL PROYECTO

app-base busca proveer una base estable para operar empresas dentro de un entorno SaaS multi-tenant.

No se trata solo de persistir entidades.

El sistema debe sostener una arquitectura común que permita:

ordenar operación
mantener aislamiento por tenant
centralizar gobierno transversal
integrar módulos sin pérdida de identidad
crecer sin rehacer el sistema
habilitar capacidades transversales cuando aporten valor real
habilitar capas productivas satélite cuando correspondan
preparar asistencia IA futura sobre reglas claras y datos estructurados

Principios asociados:

la complejidad operativa debe ser absorbida por el sistema

la experiencia debe tender a claridad

la automatización debe reducir fricción cuando corresponda

seguridad y trazabilidad son parte estructural

la operación real del negocio tiene prioridad sobre formalismos innecesarios

la complejidad debe activarse por necesidad operativa y no imponerse desde el inicio

Criterio general de evolución:

app-base debe servir tanto a operaciones simples con baja tolerancia a complejidad como a organizaciones que necesiten formalidad progresiva, trazabilidad, documentación, costos, proveedores, cierres y automatización.

Regla:

app-base no debe entenderse como una suma de CRUDs aislados ni como copia de un ERP enterprise pesado.

Debe adoptar principios robustos de gestión operativa, ERP y CMMS cuando correspondan, manteniendo una experiencia liviana, modular, automatizable y accesible.

SECTION_VERSION: 00001

ARQUITECTURA SAAS MULTI-TENANT

El sistema utiliza una arquitectura SaaS multi-tenant.

Cada tenant representa una empresa o espacio de trabajo operativo independiente dentro de una misma aplicación.

El aislamiento entre empresas no se resuelve mediante aplicaciones separadas ni bases separadas por defecto, sino mediante una arquitectura común con separación lógica de datos y contexto operativo.

Principios estructurales:

cada tenant constituye una unidad operativa propia
el sistema mantiene una única plataforma común
la operación válida siempre ocurre dentro de un contexto tenant determinado
la pertenencia del usuario y el tenant activo forman parte de la lectura correcta del sistema

<<END SECTION>>

<<SECTION: MODELO DE AISLAMIENTO OPERATIVO>>
SECTION_VERSION: 00004

MODELO DE AISLAMIENTO OPERATIVO

Las entidades operativas del negocio pertenecen a un tenant y deben respetar aislamiento lógico entre empresas.

Ese aislamiento no es solo un detalle de persistencia.

Forma parte de la arquitectura conceptual del sistema.

Principios estables:

los datos de un tenant no deben mezclarse con los de otro
la operación visible y válida del usuario depende del tenant activo
las entidades globales del sistema no deben leerse como entidades del negocio
el contexto tenant no es opcional ni decorativo dentro de la operación interna

El aislamiento alcanza también a capas internas de soporte operativo que sostienen trazabilidad, agrupación contextual o ejecución de acciones dentro de módulos funcionales.

Este criterio incluye movimientos, operaciones internas y líneas operativas cuando participan como anclajes de ejecución o trazabilidad dentro del tenant.

Este documento reconoce el aislamiento como principio estructural general.

Su desarrollo técnico y sus reglas operativas concretas se resuelven en documentos especializados del proyecto.

<<END SECTION>>

<<SECTION: IDENTIDAD GLOBAL Y PERTENENCIA POR EMPRESA>>
SECTION_VERSION: 00007

IDENTIDAD GLOBAL Y PERTENENCIA POR EMPRESA

La relación principal del sistema es:

users
↓
memberships
↓
tenants

Lectura correcta:

users representa identidad global

memberships representa pertenencia operativa por empresa

tenants representa el contexto operativo de cada empresa

Un usuario puede pertenecer a múltiples tenants.

La pertenencia a una empresa no se resuelve desde user como atributo directo único, sino mediante memberships.

Esto forma parte estable del modelo del sistema.

Aclaración consolidada:

membership puede vincular opcionalmente una ficha ampliada en parties.

Esa ficha ampliada pertenece al plano del negocio.

No reemplaza la identidad global.

No reemplaza la pertenencia operativa.

No convierte a party en entidad de acceso.

Cuando exista membership activa vinculada a party, esa party debe leerse como representación operativa ampliada del colaborador dentro del tenant.

Separación vigente:

User = identidad global

Membership = pertenencia operativa

Party = ficha ampliada tenant-scoped cuando existe vínculo explícito

OperationalActivity = trazabilidad transversal tenant-scoped

Aclaración estructural sobre owner/admin:

owner es una condición estructural de gobierno del tenant resuelta desde membership.

admin es un rol delegable de gestión operativa fuerte dentro del tenant.

admin no equivale a owner.

Esta distinción refuerza la separación entre:

condición estructural de gobierno

rol delegable operativo

pertenencia activa al tenant

ficha ampliada opcional del negocio

El detalle operativo, visual y de seguridad de esta distinción se define en documentos especializados.

TENANT ACTIVO COMO CONTEXTO OPERATIVO

El uso operativo del sistema ocurre dentro de un tenant activo.

Ese tenant activo constituye el contexto concreto desde el cual deben leerse las entidades operativas, las relaciones del negocio y las capacidades disponibles para la empresa actual.

El tenant activo define, a nivel estructural:

contexto de empresa vigente
marco operativo de los módulos del negocio
lectura correcta de las relaciones del ecosistema
ámbito de operación del usuario dentro del sistema
marco de gobierno interno del tenant

El tenant activo no debe interpretarse como un dato accesorio de sesión sin relevancia arquitectónica.

Es una pieza estructural del funcionamiento del sistema.

El frente tenant/profile pertenece a este plano de contexto y gobierno interno.

No debe leerse como módulo funcional ordinario del negocio.

Regla estructural:

el tenant activo organiza tanto operación como gobierno interno de la empresa activa, pero el detalle de autorización, permisos y restricciones owner/admin pertenece al documento especializado de seguridad.

CAPAS FUNCIONALES DEL SISTEMA

La arquitectura debe leerse por capas funcionales.

CAPA GLOBAL

Gobierno general de la plataforma.

CAPA TENANT

Contexto operativo de la empresa activa.

Puede incluir capacidades propias del plano empresa que no constituyen módulos funcionales ordinarios.

CAPA DE GOBIERNO INTERNO DEL TENANT

Organiza pertenencia, actores internos y autoridad estructural dentro de la empresa activa.

Lectura vigente:

users mantiene identidad global

memberships resuelve pertenencia operativa

owner es condición estructural de gobierno

admin es rol delegable de gestión operativa fuerte

tenant/profile pertenece a esta capa.

No es módulo funcional ordinario del negocio.

CAPA TRANSVERSAL

Responsabilidades compartidas que sostienen funcionamiento común entre módulos.

Caso vigente:

OperationalActivity

Lectura:

capacidad transversal tenant-scoped de trazabilidad operativa

no es CRUD ordinario

no es módulo funcional ordinario

no es navegación principal por mera existencia técnica

no absorbe ownership funcional de los módulos que producen actividad

CAPA IA OPERATIVA

Capa transversal futura o progresiva de asistencia, reducción de fricción, análisis contextual, preparación operativa y soporte a la automatización.

Lectura:

la IA no es módulo funcional ordinario

no reemplaza seguridad

no reemplaza backend

no reemplaza validaciones

no reemplaza control humano en acciones críticas

debe consumir contratos claros, datos autorizados y reglas explícitas

BASE DEL DOMINIO

Anclajes conceptuales amplios del negocio.

Caso principal vigente:

parties

MÓDULOS FUNCIONALES

Capacidades operativas visibles del tenant.

CAPAS PRODUCTIVAS SATÉLITE

Capacidades especializadas que orquestan módulos existentes para resolver un frente operativo amplio sin convertirse automáticamente en módulos CRUD ordinarios.

Caso vigente:

Servicios y Mantenimiento puede leerse como capa satélite productiva ERP/CMMS liviana.

Regla:

una capa satélite puede orquestar módulos existentes, pero no duplicar ni absorber ownership funcional.

SUBCAPAS CONTEXTUALES INTERNAS

Capacidades alojadas dentro de un módulo funcional existente que amplían su lectura contextual sin constituir módulo nuevo, navegación propia ni entrada independiente del ecosistema.

Caso vigente:

ProductOperationalSummary bajo Products.

Lectura:

subcapa contextual interna

no es módulo nuevo

no tiene CRUD

no tiene navegación propia

no tiene ModuleCatalog propio

no entra al dashboard general

AUTONOMÍA MODULAR

Un módulo funcional puede vincularse con otros módulos sin perder autonomía ni absorber ownership ajeno.

Regla:

los vínculos con otros módulos pueden representar contexto, enriquecimiento, origen, planificación, seguimiento, formalización o evidencia, según corresponda.

No deben convertirse automáticamente en dependencias obligatorias de existencia.

JERARQUÍA GENERAL DEL SISTEMA

La arquitectura conceptual del sistema debe leerse en el siguiente orden:

app-base
↓
capa global
↓
tenant
↓
gobierno interno del tenant
↓
capa transversal
↓

↓

Esta jerarquía expresa orden de gobierno, contexto y operación.

No implica que todos los elementos sean equivalentes como tabla, clase o recurso visible.

el tenant define el contexto operativo de empresa
el gobierno interno del tenant organiza la pertenencia y el uso
la capa transversal arbitra o sostiene funcionamiento compartido
la base del dominio aporta anclajes relacionales amplios
los módulos funcionales ejecutan la operación del negocio

Regla:

no interpretar el sistema como una primera línea plana de módulos al mismo nivel conceptual.

ENTIDADES DEL NEGOCIO

Las entidades del negocio representan la operación propia de cada tenant.

Casos actuales del proyecto:

parties
party_roles
projects
tasks
appointments
products
assets
orders
order_items
documents
document_items
document_sequences
inventory_movements
inventory_operations
operational_activities
attachments

Estas entidades pertenecen al ecosistema operativo de empresa y no a la infraestructura global del sistema.

No todas cumplen el mismo rol conceptual.

Pueden actuar como:

base del dominio
módulos funcionales
entidades hijas
capacidades transversales
soporte operativo interno
entidades complementarias

Reglas de lectura:

users no pasa a ser entidad del negocio por participar en la operación interna de una empresa

users mantiene identidad global del sistema

la pertenencia operativa real se resuelve mediante memberships

si el ecosistema requiere un dataset reusable de colaboradores internos activos, esa necesidad pertenece al plano tenant

inventory_movements expresa impacto unitario real sobre stock

inventory_operations agrupa contexto operativo cuando una misma acción requiere uno o múltiples movimientos

operational_activities registra actividad operativa real dentro del tenant activo como capacidad transversal de trazabilidad

operational_activities no representa una entidad hija de módulos funcionales particulares

operational_activities no reemplaza la lógica interna de los módulos funcionales

order_items y document_items conservan naturaleza de entidades hijas de sus módulos padre

orders y documents comparten una doctrina base de lifecycle para cabeceras operativas

Aclaración estructural para parties:

parties representa contactos o actores del negocio dentro del tenant.

parties.kind representa únicamente tipo de contacto.

Valores vigentes:

person
company

party_roles expresa la relación del contacto con la empresa.

Relaciones vigentes:

customer
supplier
employee
other

Regla:

employee no es tipo de contacto ni naturaleza de party.

Employee es una relación operativa dentro de party_roles.

Aclaración sobre membership → party:

membership puede vincular opcionalmente una ficha ampliada en parties.

Esta relación:

es opcional

es tenant-scoped

no modifica la estructura base del sistema

no reemplaza la identidad global del usuario

no convierte a party en entidad de acceso

Aclaración estructural sobre Orders y Documents:

Orders y Documents pueden conservar vínculos opcionales con Parties sin depender rígidamente de ese vínculo para existir cuando su contrato funcional permita dato propio suficiente de contraparte.

Regla:

la entidad dueña debe conservar la lectura propia de su operación.

Los detalles de campos, helpers, salidas visuales y relaciones concretas corresponden a los documentos especializados de entidades, navegación, visual y convenciones.

Este documento reconoce la posición estructural de estas entidades.

El detalle operativo, visual, de navegación o seguridad debe resolverse en los documentos especializados correspondientes.

DISTINCIÓN ENTRE SISTEMA Y NEGOCIO

La distinción entre entidades del sistema y entidades del negocio es estructural.

Las entidades del sistema sostienen identidad, acceso, onboarding y gobierno general.

Las entidades del negocio sostienen la operación propia de cada tenant.

Regla conceptual:

no debe resolverse una necesidad del negocio trasladándola arbitrariamente a entidades del sistema si esa decisión rompe la separación entre plataforma y operación.

Regla complementaria:

tampoco debe sobredimensionarse el dominio del negocio hasta absorber piezas cuya función real es global o transversal a la plataforma.

<<SECTION: ARQUITECTURA MODULAR DEL ECOSISTEMA>>
SECTION_VERSION: 00033

ARQUITECTURA MODULAR DEL ECOSISTEMA

La arquitectura modular del proyecto debe leerse como un ecosistema reusable y desacoplado.

Los módulos no deben relacionarse como cajas cerradas ni como núcleos invadibles.

La lectura correcta distingue tres planos:

núcleo interno del módulo

puerta de enlace modular oficial

consumo externo

NÚCLEO INTERNO

Corresponde a la lógica y operación propia del módulo.

No debe reabrirse desde consumidores.

PUERTA DE ENLACE MODULAR OFICIAL

Corresponde a la forma publicada para integrarse con el ecosistema.

Debe representar la capacidad del módulo oferente sin trasladar su lógica interna al consumidor.

Puede expresarse mediante contratos explícitos, surfaces, hooks, selectors, services, helpers, catálogos, eventos operativos livianos u otras fronteras formales según corresponda.

CONSUMO EXTERNO

El consumo sigue la secuencia:

consumer solicita

módulo oferente resuelve

componente representa

host monta

El consumidor no debe absorber decisiones del módulo oferente.

ADUANA FUNCIONAL ENTRE MÓDULOS

La integración entre módulos debe funcionar como una frontera explícita.

Un módulo consumidor puede solicitar una capacidad, dato, bloqueo, link, validación o surface.

El módulo oferente debe entregar el resultado mediante contrato oficial.

Reglas:

el consumidor no reconstruye lógica interna del oferente

el oferente no interpreta modelos internos del consumidor

el controller del consumidor no debe acumular conocimiento de dominios externos si existe frontera apropiada

las dependencias activas deben canalizarse mediante contratos explícitos

Caso vigente:

Orders no debe interpretar internamente Task, Appointment, Party, Asset o Document más allá de referencias pasivas, relaciones necesarias o contratos explícitos.

HUB VISUAL

Un módulo puede alojar interacción o lectura de otro sin absorber su lógica.

Regla:

el host conserva superficie visual

el módulo invitado conserva lógica operativa

TRAZABILIDAD TRANSVERSAL

Cuando un módulo funcional completa una acción propia autorizada, puede emitir un evento operativo liviano para trazabilidad.

Lectura arquitectónica:

módulo dueño autoriza

módulo dueño valida

módulo dueño ejecuta

módulo emite evento operativo liviano

capacidad transversal registra si corresponde

OperationalActivity no reemplaza el núcleo del módulo dueño ni se convierte en intérprete principal de su dominio.

SURFACES TENANT O TRANSVERSALES NO MODULARES

Una capacidad tenant-scoped o transversal no modular puede publicar surfaces controladas hacia hosts funcionales sin incorporarse como módulo funcional ordinario.

Esta vía corresponde cuando la capacidad:

no es módulo funcional ordinario

no posee CRUD ordinario

no requiere navegación principal

no se habilita o deshabilita como módulo del negocio

debe ofrecer lectura o información controlada a hosts autorizados

SUBCAPAS CONTEXTUALES INTERNAS

Un módulo puede alojar una subcapa contextual interna cuando la capacidad amplía la lectura del host sin constituir módulo propio.

Caso vigente:

ProductOperationalSummary amplía la lectura operativa de Products.

Regla:

una subcapa contextual interna debe mantenerse bajo ownership del módulo host y consumir datos relacionados sin absorber el ownership de los módulos que los aportan.

AISLAMIENTO OPERATIVO RAZONABLE

El objetivo no es construir aislamiento absoluto artificial entre módulos.

El objetivo es evitar dependencias activas indebidas.

Reglas:

las relaciones pasivas pueden existir cuando representan columnas propias, vínculos estructurales o patrones transversales

las dependencias activas deben pasar por contratos explícitos

una relación de modelo no autoriza al controller a gobernar el módulo relacionado

un módulo puede conservar referencias pasivas sin perder saneamiento, siempre que no interprete ni administre lógica externa desde su núcleo

Aclaración consolidada:

los módulos pueden publicar metadata estructural propia.

El flujo correcto es:

módulo define

catálogo o contrato provee metadata

registry o surface normaliza

host consume sin reinterpretar

Regla final:

la arquitectura modular debe permitir integración sin pérdida de ownership funcional.

BASE DEL DOMINIO

La base del dominio agrupa piezas conceptuales que sostienen múltiples relaciones relevantes dentro del ecosistema operativo.

Caso actual principal:

parties

Su función no se limita a registrar un dato aislado.

Debe leerse como anclaje relacional estable para múltiples frentes del negocio.

parties también puede publicar capacidades oficiales para que otros módulos consuman relaciones compactas o embebidas sin reconstruir localmente su lógica.

Regla estructural:

que una pieza pertenezca a la base del dominio no la convierte en infraestructura global del sistema.

Sigue perteneciendo al ecosistema operativo del tenant, pero con un rol conceptual más amplio que el de un módulo operativo puntual.

Regla vigente para parties:

parties debe mantenerse como núcleo mínimo del actor o contacto del negocio.

La expansión de su complejidad debe tender a entidades hijas o complementarias y no a sobrecargar el núcleo.

Separación consolidada:

parties.kind = tipo de contacto
party_roles.role = relación con la empresa

Valores vigentes de parties.kind:

person
company

Relaciones vigentes de party_roles.role:

customer
supplier
employee
other

Regla:

employee no pertenece a parties.kind.

Employee expresa relación operativa con la empresa dentro de party_roles.

Aclaración sobre colaboradores internos:

los usuarios internos activos del tenant pertenecen al plano tenant y a memberships como pertenencia operativa real.

membership puede vincularse opcionalmente con party como ficha ampliada tenant-scoped, pero esa relación no convierte a party en identidad global ni en entidad de acceso.

Por lo tanto, esa necesidad no debe reinterpretarse como expansión de users ni como apertura de un módulo users dentro de la base del dominio.

<<SECTION: LECTURA CONCEPTUAL DE LOS MÓDULOS ACTUALES>>

LECTURA CONCEPTUAL DE LOS MÓDULOS ACTUALES

Lectura síntesis del ecosistema vigente:

orders es referencia madura para cabeceras operativas equivalentes

documents formaliza flujos documentales y converge hacia patrones sólidos cuando corresponde

inventory gobierna stock, ejecución material y trazabilidad sin absorber ownership ajeno

projects y tasks organizan planificación y seguimiento operativo

parties concentra contactos y actores del negocio, internos o externos según su relación con la empresa

products y assets representan recursos operativos del tenant

appointments coordina agenda y vínculos contextuales con otros módulos

attachments aporta capacidad transversal de evidencias y archivos

OperationalActivity sostiene trazabilidad operativa transversal sin reemplazar la operación original

Regla:

ningún módulo debe expandirse invadiendo el núcleo funcional de otro.

La integración correcta ocurre por contratos publicados y consumo explícito.

Lectura adicional para Orders:

Orders funciona como núcleo operativo autónomo.

Puede recibir contexto desde tasks o appointments, vincularse opcionalmente con parties o assets, derivar documents, consumir capacidades de inventory y publicar surfaces cuando corresponda.

Esa integración no convierte a Orders en dueño de Tasks, Appointments, Parties, Assets, Documents ni Inventory.

Reglas:

Orders no debe almacenar como núcleo propio relaciones persistentes que expresan origen, planificación o seguimiento externo

Tasks y Appointments pueden vincularse a Orders desde su propio ownership

Orders conserva su lifecycle base y su identidad operativa

Orders debe interactuar con otros módulos mediante contratos explícitos o relaciones pasivas justificadas

Orders no debe interpretar internamente modelos consumidores

Lectura adicional para Documents:

Documents formaliza, comunica o registra documentalmente operaciones estructuradas.

Puede derivarse desde Orders.

Puede conservar dato propio de contraparte aunque el vínculo gestionado con Parties sea opcional.

Documents no reemplaza Orders ni absorbe la operación técnica o material.

Lectura adicional para Products:

Products funciona como catálogo maestro del producto.

Puede alojar lecturas contextuales propias cuando no impliquen absorción de ownership de módulos relacionados.

Puede ofrecer capacidades de selección o validación de productos para líneas cuando otros módulos necesiten operar con productos sin consultar directamente su modelo interno.

Caso vigente:

ProductOperationalSummary bajo Products.

Lectura adicional para Inventory:

Inventory conserva ownership de stock, ejecución material y reglas físicas asociadas a líneas cuando corresponda.

Puede intervenir en flujos de Orders o Documents mediante fronteras explícitas sin que esos módulos reconstruyan internamente su lógica.

Lectura adicional para capas satélite:

un frente especializado como Servicios y Mantenimiento puede orquestar módulos actuales, pero no debe modificar la identidad conceptual de cada uno.

Orders no debe absorber Assets.

Orders no debe absorber Inventory.

Orders no debe absorber Documents.

Orders no debe absorber Tasks ni Appointments.

Attachments no debe absorber lógica técnica de mantenimiento.

OperationalActivity no debe convertirse en timeline específico por sí mismo.

La IA no debe convertirse en canal paralelo de operación fuera de contratos oficiales.

Regla final:

los módulos actuales deben conservar ownership propio aun cuando participen en frentes productivos satélite, subcapas contextuales internas, capacidades transversales o automatizaciones futuras.

MODELO GENERAL DE CRECIMIENTO ARQUITECTÓNICO

El sistema debe crecer mediante módulos con identidad clara, contratos explícitos y responsabilidades separadas.

Toda evolución debe:

respetar multi-tenancy
preservar distinción sistema/negocio
mantener identidad conceptual
evitar duplicaciones
consumir capacidades existentes antes de crear variantes
distinguir módulos ordinarios de capacidades transversales
distinguir módulos ordinarios de capas productivas satélite
distinguir módulos ordinarios de subcapas contextuales internas
distinguir automatización asistida de ejecución crítica
distinguir dato propio necesario para operar de vínculo opcional de enriquecimiento
distinguir relación de origen, planificación o seguimiento de dependencia estructural del módulo receptor
distinguir relación pasiva permitida de dependencia activa indebida

Reglas:

no reabrir el núcleo de otro módulo arbitrariamente

las necesidades de colaboradores internos deben resolverse en la capa tenant

las capacidades maduras deben converger hacia contratos oficiales

los acoples externos deben canalizarse mediante puertas contractuales

si una entidad hija crece, debe fortalecerse antes de promoverse a módulo

los frentes transaccionales pueden incorporar agrupación contextual cuando mejore trazabilidad

las capas satélite deben auditar primero qué pueden sostener los módulos actuales antes de proponer piezas nuevas

las piezas satélite no deben crearse por reflejo automático

las subcapas contextuales internas deben alojarse dentro del módulo host cuando no correspondan a módulo, ruta ni CRUD propio

las subcapas contextuales internas deben consumir ownership de módulos relacionados sin absorberlo

la IA futura debe apoyarse en datos estructurados, reglas explícitas y documentación consistente

los vínculos opcionales entre módulos no deben transformarse automáticamente en dependencias obligatorias

los campos o relaciones `*_id` deben interpretarse según contrato funcional real y no por reflejo técnico

cuando un módulo necesite conservar dato propio aunque exista vínculo opcional con otro módulo, debe sostener una lectura autónoma del registro

cuando un módulo publique lecturas oficiales, los módulos externos no deben reconstruir esa lectura desde relaciones internas opcionales

cuando una mecánica sea transversal, el significado de negocio debe permanecer en el catálogo o contrato del módulo correspondiente

cuando un módulo necesite validar, listar o seleccionar datos de otro dominio, debe consumir una capacidad oficial si existe

Casos vigentes:

ProductOperationalSummary se implementa como subcapa contextual de Products.

Orders conserva autonomía operativa y puede relacionarse opcionalmente con otros módulos sin absorber su ownership.

Tasks y Appointments pueden vincularse a Orders desde su propio contrato.

Documents puede derivarse desde Orders y conservar su propia identidad documental.

Orders y Documents pueden conservar dato propio de contraparte aunque el vínculo gestionado con Parties sea opcional.

Products puede ofrecer selección y validación de productos para líneas mediante una capacidad propia consumida por Orders y Documents.

La numeración puede resolverse por servicio transversal, pero el significado de prefijos pertenece a los catálogos de cada módulo.

Regla:

crecer no implica mezclar responsabilidades.

La dirección sigue siendo modular, con fronteras explícitas, operación simple y formalidad progresiva cuando corresponda.

<<SECTION: CONTRATO MODULAR COMO DIRECCIÓN DE CRECIMIENTO>>

CONTRATO MODULAR COMO DIRECCIÓN DE CRECIMIENTO

El crecimiento correcto del sistema debe priorizar evolución contractual y claridad de fronteras antes que expansión desordenada de código.

Cuando una necesidad nueva aparezca en un módulo existente, la primera lectura correcta no es invadir consumidores ni dispersar lógica transversal.

La salida preferente es fortalecer el contrato propio del módulo y ordenar internamente su dominio.

Reglas:

una mejora interna no debe romper contratos externos estables

la complejidad nueva debe resolverse primero dentro del módulo dueño del problema

los consumidores deben depender de capacidades publicadas y no de detalles internos variables

un contrato modular válido debe permitir integrar sin borrar ownership funcional

las capas satélite deben consumir contratos oficiales antes de pedir modificaciones internas

las subcapas contextuales internas deben integrarse al módulo host sin convertirse automáticamente en módulos nuevos

las automatizaciones futuras deben operar sobre contratos explícitos y no sobre scraping visual, inferencias frágiles o labels dispersos

la IA futura debe respetar los contratos modulares del sistema y no crear canales paralelos de operación

los helpers de integración deben ser genéricos desde el punto de vista del módulo oferente y no específicos por consumidor

las surfaces deben expresar capacidades reales y no duplicación de hosts

los hooks deben funcionar como puntos de intervención explícita, no como conocimiento interno disperso en controllers

los selectors o providers deben concentrar capacidades de selección o validación cuando varios módulos necesitan consumir un mismo dominio

los servicios transversales deben ejecutar mecánicas comunes sin decidir significado de negocio propio de cada módulo

Reglas de frontera:

el contrato modular define cómo integrarse

no debe absorber detalle visual

no debe duplicar seguridad

no debe convertirse en inventario de implementación

no debe registrar cada archivo, helper, validación o salida puntual

Casos vigentes:

Servicios y Mantenimiento debe avanzar como capa satélite productiva y no como sobrecarga directa de Orders.

ProductOperationalSummary debe permanecer como subcapa contextual interna de Products mientras no exista justificación real para promoverla a módulo, ruta o frente independiente.

Orders puede exponer relaciones, links y bloqueos mediante contratos oficiales sin interpretar internamente modelos consumidores.

Products puede ofrecer selección y validación de productos para líneas mediante capacidad propia consumida por otros módulos.

La numeración puede centralizarse como mecánica transversal siempre que cada módulo conserve la definición de su significado.

Regla síntesis:

el módulo define el significado.

El servicio transversal ejecuta la mecánica.

Regla final:

la dirección de crecimiento aprobada del proyecto continúa siendo modular, reusable, contractualmente explícita y preparada para asistencia operativa progresiva.
