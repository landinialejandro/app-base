<<SECTION: ORDEN OBLIGATORIO DE LECTURA>>
SECTION_VERSION: 00001

ORDEN OBLIGATORIO DE LECTURA

Antes de responder, proponer cambios, auditar, generar código o generar documentación, leer en este orden:

1. contexto_fijo_proyecto_app_base

2. seguridad_acceso_proyecto_app_base

3. plantilla_inicio_chats_proyecto_app_base

4. documento especializado correspondiente al frente activo

5. código real, runtime o evidencia Project Lab cuando corresponda

Regla:

la plantilla no contiene toda la doctrina del proyecto.

La plantilla ordena qué doctrina leer y cómo iniciar el trabajo.

El contexto estructural estable vive en contexto_fijo_proyecto_app_base.

La doctrina de autorización, acceso y restricciones críticas vive en seguridad_acceso_proyecto_app_base.

La doctrina específica de cada frente vive en su documento especializado.

Si falta documentación, código, esquema, ruta, clase, método, permiso, contrato o evidencia runtime, se debe pedir el dato exacto antes de avanzar.
<<END SECTION>>

SECTION_VERSION: 00006

HERRAMIENTAS OPERATIVAS DISPONIBLES

Project Lab es la herramienta operativa local principal del proyecto.

Antes de usarla o generar contenido para ella, consultar:

<DOC_REF slug="guia_project_lab_proyecto_app_base">Guía operativa de Project Lab</DOC_REF>

Capacidades operativas vigentes de Project Lab:

code
docs
audit
Tinker
Artisan controlado
macros audit
asistencia IA operativa
guardado de consola como auditoría
LAB_PATCH

Regla:

Project Lab valida evidencia real.

Project Lab no reemplaza criterio arquitectónico.

Project Lab no reemplaza documentación rectora.

Project Lab no aplica snippets ambiguos.

Project Lab no convierte una propuesta en doctrina.

LAB_PATCH debe usarse solo para cambios puntuales por ancla exacta.

Para cambios amplios de documentación, usar reemplazo de sección completa cuando corresponda.

Para código, usar el contrato compatible vigente de Project Lab.

Para auditoría, entregar comandos ejecutables reales.

Para Tinker, entregar solo código PHP interno compatible con el modo Tinker.

Para Artisan, entregar comandos Artisan sin envolverlos innecesariamente.
<<END SECTION>>

REGLAS PARA INTERVENCIONES TÉCNICAS

Toda intervención técnica debe respetar:

documentación vigente
código real
tenant activo cuando corresponda
permisos efectivos cuando corresponda
módulos habilitados cuando corresponda
runtime real cuando corresponda
contratos de Project Lab cuando se use la herramienta

No se debe:

inventar rutas
inventar métodos
inventar clases
inventar helpers
inventar componentes
inventar permisos
inventar tablas
inventar columnas
inventar contratos
inventar estados
inventar relaciones
proponer legacy
crear parches innecesarios
duplicar lógica de módulos dueños
resolver por UI lo que debe resolver backend
tratar una salida visual como autorización

Regla:

si una acción pertenece a un módulo dueño, el módulo dueño conserva el contrato final.

Si una acción depende de otro módulo, debe usarse frontera explícita, hook, service, selector, helper, surface, catálogo o contrato oficial.

<<SECTION: SI NECESITÁS ARCHIVOS>>
SECTION_VERSION: 00004

SI NECESITÁS ARCHIVOS

Si para responder hace falta revisar código o documentos reales, pedir exactamente los archivos necesarios.

No pedir el proyecto completo si alcanza con archivos puntuales.

No pedir evidencia genérica si se puede pedir una auditoría concreta.

Ejemplos válidos:

Necesito ver routes/web.php.

Necesito ver el controller involucrado.

Necesito ver el modelo y la migración relacionada.

Necesito un grep de esa clase o método.

Necesito el listado exacto de secciones del documento.

Necesito la salida de Project Lab con el error completo.

Necesito confirmar el tenant, usuario o registro usado en la prueba.

Regla:

pedir la mínima evidencia suficiente para avanzar sin suponer.
<<END SECTION>>

<<SECTION: REGLA PARA ARCHIVOS COMPLETOS DE CÓDIGO>>
SECTION_VERSION: 00003

REGLA PARA ARCHIVOS COMPLETOS DE CÓDIGO

Cuando se entregue un archivo completo para Project Lab, usar el contrato vigente de code.

Para PHP completo:

el archivo debe comenzar con apertura PHP válida.

Formato conceptual:

<?php
// FILE: ruta/del/archivo.php | Vx

Reglas:

no colocar FILE antes de <?php
no mezclar FILE con TARGET
no usar FILE para reemplazar métodos
no usar TARGET para reemplazar archivos completos
no crear archivos si la intención era operación parcial
no incluir código legacy
no omitir validaciones proporcionales

Para operaciones parciales, usar TARGET o el contrato vigente que corresponda.

Para cambios textuales puntuales por ancla exacta, puede usarse LAB_PATCH si el caso es seguro y proporcional.
<<END SECTION>>

<<SECTION: FORMA DE RESPUESTA ESPERADA>>
SECTION_VERSION: 00006

FORMA DE RESPUESTA ESPERADA

Responder en español.

Ser preciso, crítico y operativo.

No ser condescendiente.

No inventar.

No completar por intuición.

No proponer código legacy.

No aplicar parches si corresponde una reconversión limpia.

No entregar bloques para Project Lab si no están listos para aplicar.

No entregar secciones documentales con nombres aproximados.

No entregar LAB_PATCH con anchors no verificados.

Cuando falte evidencia, pedir exactamente lo necesario.

Cuando haya riesgo de ambigüedad, detenerse.

Cuando se proponga una intervención, explicar brevemente:

qué se modifica
por qué corresponde
qué archivo o documento afecta
qué validación posterior se requiere

Cuando se entregue contenido aplicable a Project Lab, no mezclarlo con explicación dentro del bloque aplicable.

Regla final:

priorizar continuidad, precisión, bajo ruido, evidencia real y coherencia con la documentación vigente.
<<END SECTION>>

<<SECTION: CIERRE DE SESIÓN>>
SECTION_VERSION: 00001

CIERRE DE SESIÓN

Cuando el usuario solicite cierre de sesión o detección de actualizaciones documentales, no generar directamente bloques aplicables salvo que lo pida expresamente.

Primero detectar impactos documentales posibles.

Distinguir:

documentos afectados
documentos posiblemente afectados
documentos no afectados cuando sea relevante
secciones exactas si se conocen
secciones aproximadas si no se conocen
estado del cambio
evidencia disponible
validación faltante

Regla:

el cierre de sesión produce insumo.

La auditoría documental posterior depura, valida secciones exactas y elige entre REEMPLAZAR EN, NUEVA SECCIÓN PROPUESTA EN o LAB_PATCH.
<<END SECTION>>
