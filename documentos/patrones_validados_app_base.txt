RECONVERSIÓN LIMPIA DE CONSTRAINTS

Cuando cambia la dimensión real de autorización, la constraint funcional debe migrar a esa dimensión y eliminar la convivencia legacy.

Caso validado:

parties:
antes → allowed_kinds sobre parties.kind
ahora → allowed_party_roles sobre party_roles.role

Regla:

la constraint anterior no debe quedar como fallback operativo si ya no representa la semántica vigente del módulo.

AUTORIZACIÓN SENSIBLE CENTRALIZADA

Cuando una regla sensible aparece duplicada en controller y requests, debe extraerse a una pieza reusable del dominio correspondiente.

Caso validado:

PartyEmployeeContactAuthorization

Ubicación vigente:

app/Support/Parties/PartyEmployeeContactAuthorization.php

Regla:

la UI puede expresar disponibilidad, pero la autorización sensible debe resolverse y bloquearse en backend.

VALIDACIÓN RUNTIME DE SEGURIDAD

Una validación sensible debe combinar:

UI correcta
Security runtime
request validation
intento manipulado
verificación final de persistencia

Caso validado:

protección de party vinculada a membership activa.

La validación no se considera cerrada solo por ocultar campos en UI.

Debe probarse que el backend bloquea intentos manipulados y que el registro persiste intacto.

EVENTOS OPERATIVOS TRANSVERSALES

Patrón validado:

módulo dueño
↓
autoriza, valida y muta
↓
emite evento operativo liviano
↓
listener queued con ShouldQueue y afterCommit
↓
prestador transversal aplica sanidad
↓
persistencia append-only

Reglas:

el evento operativo no autoriza
el evento operativo no valida la operación original
el evento operativo no reemplaza Security ni policies
el módulo dueño conserva ownership funcional
la capacidad transversal registra solo si corresponde
el listener queued no debe bloquear el flujo principal

Caso vigente:

OperationalActivity

Módulos con integración validada a OperationalActivity:

appointments
tasks
orders
documents
projects
assets
products
attachments
parties
inventory

CAMPOS OBSERVABLES POR CATÁLOGO DEL MÓDULO DUEÑO

Cuando una capacidad transversal registra actividad operativa, los campos observables deben pertenecer al catálogo del módulo dueño.

Casos vigentes:

AppointmentCatalog::activityTrackedFields()
TaskCatalog::activityTrackedFields()
OrderCatalog::activityTrackedFields()
DocumentCatalog::activityTrackedFields()

Regla:

el prestador transversal no debe conocer en exceso la semántica interna de cada módulo.

Cuando se integre un nuevo módulo, el catálogo dueño debe declarar explícitamente los campos observables antes de considerar cerrada la trazabilidad operativa.

SANIDAD DE CAMBIOS OPERATIVOS

La trazabilidad operativa debe aplicar sanidad antes de persistir.

Reglas:

no registrar cambios vacíos
ignorar campos técnicos
no registrar por registrar
registrar solo cambios operativamente significativos
clasificar actividad desde el prestador transversal cuando corresponda

Clasificaciones vigentes posibles:

assigned
reassigned
unassigned
status_changed
updated

Regla:

la clasificación no debe dispersarse en controllers.

METADATA OPERATIVA OPCIONAL

Los eventos operativos livianos pueden admitir metadata opcional cuando un cambio operativo relevante no vive directamente como atributo simple del modelo principal.

Reglas:

la metadata debe complementar el cambio operativo
no debe trasladar autorización al evento
no debe convertir entidades hijas en módulos propios
no debe acoplar en exceso OperationalActivity a la semántica interna del módulo dueño

Caso validado:

parties registra cambios de party_roles como metadata operativa de Party.

RELACIONES HIJAS CON SIGNIFICANCIA OPERATIVA

Cuando una relación hija expresa un cambio operativo relevante, puede representarse como metadata operativa del record principal si esa lectura respeta ownership y evita crear un módulo artificial.

Caso validado:

party_roles expresa relación del contacto con la empresa.

Sus cambios pueden registrarse dentro de la actividad operativa de Party.

Regla:

party_roles no se convierte en módulo propio ni en navegación propia.

ENTIDADES AGRUPADORAS Y MOVIMIENTOS ATÓMICOS

Cuando un módulo posee una entidad agrupadora de acción y entidades atómicas de trazabilidad fina, la actividad transversal debe anclarse en la pieza que represente mejor la acción operativa del usuario.

Caso validado:

inventory registra InventoryOperation.

InventoryMovement conserva trazabilidad fina interna de stock.

Regla:

no duplicar en OperationalActivity la trazabilidad atómica que ya pertenece al módulo dueño cuando una operación agrupada expresa mejor el evento operativo.

LECTURA GENERAL DE ACTIVIDAD OPERATIVA

Patrón validado:

capacidad tenant no modular
↓
Security::allows('operational_activity.viewAny')
↓
lectura tenant-scoped desde tenant/profile
↓
tab visual Actividad

Reglas:

la lectura general del log puede existir sin convertir OperationalActivity en módulo funcional ordinario

OperationalActivity no se agrega a ModuleCatalog por este patrón

la vista general inicial no crea navegación principal propia

la tab Actividad en tenant/profile es primera lectura visual concreta, no surface reutilizable todavía

Patrón de resolución de lectura:

operational_activities guarda trazabilidad estable
OperationalActivityLinkResolver resuelve presentación y navegación
Blade consume filas ya resueltas

Campos persistidos usados como base:

module
record_type
record_id

Datos derivados en lectura:

record_label
record_url
actor_label
subject_label
trail

Regla:

el link visual al registro asociado no debe persistirse como URL final.

Debe resolverse dinámicamente desde una pieza de lectura.

Si no existe ruta show canónica segura, se muestra texto plano.

Trail validado desde actividad operativa:

Inicio
Perfil de empresa · Actividad
Registro destino

VALIDACIÓN QUEUED DE ACTIVIDAD OPERATIVA

Secuencia validada para probar eventos queued y OperationalActivity:

disparar evento
verificar jobs > 0
ejecutar queue:work --stop-when-empty --tries=1
verificar operational_activities
verificar jobs = 0
verificar failed_jobs = 0

Regla:

una trazabilidad queued no se considera cerrada solo por disparar el evento.

Debe validarse procesamiento real de cola y persistencia final.

Si la cola no se procesa, la tab puede no mostrar actividades aunque las acciones ya hayan emitido eventos.

Este comportamiento debe leerse como esperado dentro del modelo queued.

VALIDACIÓN RUNTIME DE OPERATIONALACTIVITY

Para cerrar una integración con OperationalActivity debe validarse:

usuario real
membership activa
tenant activo bindeado
evento operativo emitido
job generado cuando corresponda
queue procesada
failed_jobs sin errores
operational_activities persistidas
módulo correcto
tipo correcto
metadata esperada cuando corresponda

EVENT DISCOVERY SIN DUPLICACIÓN MANUAL

Cuando Laravel event discovery ya registra un listener, no debe duplicarse el registro manual mediante Event::listen().

Validación requerida:

event:list debe mostrar la asociación esperada.

Regla:

si event discovery ya resuelve la asociación, duplicar el listener genera jobs y registros repetidos.
