OBJETIVO

Este documento define el uso operativo correcto de Project Lab dentro del proyecto app-base.

Project Lab es la herramienta local unificada para ejecutar tareas técnicas controladas sobre el proyecto.

Su función es:

ejecutar pruebas Tinker sobre runtime real
ejecutar comandos Artisan
ejecutar auditorías simples
aplicar actualizaciones de código
aplicar actualizaciones documentales
usar snippets rápidos
centralizar salidas operativas en consola
reducir errores por formatos ambiguos
evitar cambios manuales dispersos
mantener trazabilidad local de las intervenciones
validar eventos, colas y persistencia cuando el frente técnico lo requiera

Project Lab no reemplaza:

criterio arquitectónico
documentación doctrinal
validación de seguridad
revisión de código real
pruebas funcionales en navegador
auditorías específicas cuando correspondan

Project Lab ejecuta tareas.

La decisión de qué ejecutar, cuándo ejecutar y cómo interpretar el resultado sigue perteneciendo al proceso técnico del proyecto.

Regla:

una validación no queda cerrada solo porque Project Lab ejecute sin error inicial.

Debe interpretarse la salida completa y confirmar las evidencias necesarias para el frente intervenido.

Caso vigente:

OperationalActivity requiere validar runtime real, tenant activo, evento, cola, jobs, failed_jobs y persistencia final cuando corresponda.

Para validar OperationalActivity no alcanza con que el patch tenga sintaxis correcta ni que el evento genere un job.

El cierre requiere confirmar:

evento emitido
job creado cuando corresponda
listener queued procesado
jobs = 0
failed_jobs = 0
actividad persistida
módulo correcto
tipo correcto
metadata esperada cuando corresponda

<<SECTION: REGLA ANTIAMBIGÜEDAD EN HEADERS>>
SECTION_VERSION: 00001

REGLA ANTIAMBIGÜEDAD EN HEADERS

Un encabezado FILE identifica archivo completo.

Un encabezado TARGET identifica operación parcial.

Regla crítica:

FILE no debe contener operadores de método.

No son válidos como archivo completo:

// FILE: app/Providers/AppServiceProvider.php :: boot | V1
// FILE: app/Services/Example.php ++ resolve | V1

Estos casos deben resolverse de una sola manera:

si se acepta como operación parcial, debe procesarse como método
si no se acepta como operación parcial, debe devolver ERROR

Nunca debe crearse un archivo cuya ruta incluya:

::
++

Regla obligatoria:

si un header FILE contiene :: o ++, Project Lab no puede tratarlo como archivo completo.

Debe devolver ERROR explícito o normalizarlo hacia TARGET solo si el parser lo reconoce de forma segura como operación parcial válida.

Regla recomendada:

para operaciones parciales usar siempre TARGET.

Ejemplo correcto:

// TARGET: app/Providers/AppServiceProvider.php :: boot

public function boot(): void
{
    //
}

<<END SECTION>>

<<SECTION: ERRORES OBLIGATORIOS>>
SECTION_VERSION: 00003

ERRORES OBLIGATORIOS

Project Lab debe devolver ERROR y no escribir cambios cuando detecte fallas validables antes de la escritura.

Casos obligatorios:

contenido vacío
herramienta Lab no reconocida
header no compatible
ruta destino vacía
ruta con operadores inválidos para archivo completo
archivo inexistente en operación parcial
método inexistente en reemplazo parcial
método ya existente en alta de método
función JS inexistente en reemplazo parcial
sección documental inexistente
doc_slug inexistente
sección CSS o JS inexistente
intento de usar formato de archivo completo para operación parcial ambigua
bloque documental sin encabezado REEMPLAZAR EN cuando la herramienta vigente lo requiere
bloque documental con DOC_SLUG como encabezado operativo si Project Lab Docs espera REEMPLAZAR EN
header FILE que combine operadores de método como :: o ++
snippet que mezcle FILE y TARGET de forma ambigua

Regla:

un error debe ser explícito, legible y accionable.

Debe indicar:

qué falló
por qué falló
qué formato correcto corresponde usar cuando aplique

Regla adicional:

un ERROR no debe dejar cambios parciales aplicados cuando la operación pueda validarse antes de escribir.

Regla sobre sintaxis PHP:

si la validación de sintaxis PHP posterior a un cambio falla, la intervención no debe considerarse cerrada.

Debe corregirse o revertirse según corresponda.

La salida OK inicial de aplicación no alcanza para cierre si la validación posterior informa error.

Regla adicional sobre errores posteriores a la aplicación:

si la herramienta aplicó un cambio pero una validación posterior falla, el resultado no debe tratarse como cierre exitoso.

Debe corregirse o revertirse antes de avanzar.

SALIDA OPERATIVA ESPERADA

Toda ejecución de Project Lab debe producir salida clara.

Estados esperados:

OK
WARN
ERROR
INFO

OK

Indica operación aplicada correctamente.

Ejemplos:

[OK] Modo: php_method_patch
[OK] Archivo: app/Providers/AppServiceProvider.php
[OK] Método reemplazado: boot
[OK] Sintaxis PHP válida: app/Providers/AppServiceProvider.php

WARN

Indica que la operación se aplicó pero existe una advertencia relevante.

Ejemplo:

[WARN] Archivo existente reemplazado completamente.

ERROR

Indica que la operación no debe considerarse aplicada correctamente.

Ejemplo:

[ERROR] Header FILE inválido para archivo PHP completo.
[ERROR] Se detectó operador de método en el encabezado.
[INFO] Para reemplazar un método use:
[INFO] // TARGET: ruta/del/archivo.php :: nombreMetodo

INFO

Indica orientación complementaria.

Regla:

no debe considerarse cierre técnico una salida sin OK verificable.

No debe considerarse seguro un cambio aplicado con errores no resueltos.

Regla adicional:

una salida OK de aplicación no reemplaza validaciones posteriores necesarias para el frente trabajado.

Ejemplos de validaciones posteriores según caso:

php -l
route:list
optimize:clear
event:list
jobs
failed_jobs
operational_activities
navegador informado por usuario

Caso vigente:

para OperationalActivity, la salida relevante debe permitir confirmar que el evento no solo fue disparado, sino que la cola procesó el listener y persistió el registro esperado sin failed_jobs.

Validación esperada para cierre de OperationalActivity:

evento emitido
jobs generados cuando corresponda
queue:work --stop-when-empty --tries=1 ejecutado
jobs = 0
failed_jobs = 0
operational_activities persistidas
record_type correcto
record_id correcto
módulo correcto
tipo de actividad correcto
metadata esperada cuando corresponda

Regla:

si hay jobs procesados y failed_jobs = 0 pero no hay operational_activities persistidas, no debe cerrarse como éxito.

Debe revisarse primero el mapeo modelo → módulo y módulo → catálogo antes de modificar controllers, eventos o listener.

Aclaración vigente sobre sintaxis PHP:

Project Lab valida sintaxis PHP automáticamente en operaciones compatibles como reemplazo de método PHP, alta de método PHP y archivo PHP completo.

No hace falta pedir php -l redundante si la salida ya informa sintaxis PHP válida, salvo duda puntual o auditoría específica.

USO DE TINKER DENTRO DE PROJECT LAB

Tinker es una capacidad interna de Project Lab para reproducir comportamiento real del sistema.

Debe usarse especialmente para:

No debe pegarse el comando completo:

php artisan tinker --execute

Project Lab ya ejecuta internamente el proceso correspondiente contra el proyecto actual.

Regla adicional:

Ejemplo conceptual:

return User::all();

Regla adicional para bloques con namespaces:

si un bloque ejecutado mediante wrapper falla por el uso de use ..., no debe asumirse automáticamente que falló la lógica funcional.

Alternativas válidas:

usar clases con namespace completo
ejecutar el bloque dentro de Project Lab según corresponda
ajustar la forma de ejecución antes de modificar código funcional

Regla adicional para eventos queued:

si la prueba dispara un evento que debe procesarse en cola, Tinker solo valida el disparo inicial.

El cierre requiere validar también:

jobs generados
queue procesada
operational_activities persistidas cuando corresponda
jobs vaciados
failed_jobs sin errores

Regla adicional para OperationalActivity:

cuando se valide trazabilidad operativa transversal, Tinker debe reproducir runtime real con tenant activo.

La validación debe cubrir:

registro creado o actualizado por el módulo dueño
evento operativo emitido después de autorización y validación
job generado cuando corresponda
procesamiento con queue:work --stop-when-empty --tries=1
persistencia final en operational_activities
metadata esperada cuando corresponda

Casos validados:

parties puede emitir actividad de Party con cambios de party_roles como metadata operativa.

inventory debe registrar actividad sobre InventoryOperation cuando represente la acción operativa agrupada.

Regla:

no debe inferirse cierre por simple ejecución del evento.

El cierre requiere comprobar persistencia final y ausencia de failed_jobs.

<<SECTION: REGLA DE TENANT ACTIVO EN TINKER>>
SECTION_VERSION: 00002

REGLA DE TENANT ACTIVO EN TINKER

En app-base, muchas resoluciones dependen de:

app('tenant')

Por lo tanto, una prueba Tinker no reproduce correctamente el sistema si no existe tenant activo bindeado en el contenedor.

Regla operativa:

antes de probar permisos, visibilidad, queries tenant-aware o lógica que consume Security, debe resolverse y bindearse un tenant activo real.

Forma correcta:

1. obtener un usuario real
2. obtener una membership activa real de ese usuario
3. tomar el tenant de esa membership
4. bindearlo con:

app()->instance('tenant', $tenant)

Sin ese paso, la prueba puede:

fallar artificialmente
devolver falsos negativos
no representar el comportamiento real del sistema

Regla adicional:

no fijar tenant_id como número ni como dato estable copiado desde otra sesión.

Debe obtenerse desde runtime real.

Caso vigente:

si la prueba involucra OperationalActivity, tenant_id debe persistirse como UUID string compatible con tenants.id.

Regla:

tenant activo bindeado no es solo comodidad de prueba.

Es condición necesaria para reproducir el comportamiento real de capacidades tenant-scoped.

<<END SECTION>>

REGLA DE AUTORIZACIÓN EN TINKER

La autorización debe probarse desde la capa central.

inspect(...)

No probar permisos desde UI como fuente final.

No inferir permisos desde campos visibles.

Regla adicional:

Ejemplo conceptual:

si la capacidad no requiere contexto, no inventarlo

<<SECTION: BOOTSTRAP BASE RECOMENDADO PARA TINKER>>

BOOTSTRAP BASE RECOMENDADO PARA TINKER

Bloque base recomendado para comenzar pruebas reales:

use App\Models\User;
use App\Support\Auth\Security;

$user = User::where('email', 'juan@tech.local')->firstOrFail();

$tenant = $user->memberships()
    ->where('status', 'active')
    ->with('tenant')
    ->firstOrFail()
    ->tenant;

app()->instance('tenant', $tenant);

$security = app(Security::class);

Este bloque deja resuelto:

usuario real
membership activa real
tenant real
tenant activo en contenedor
acceso a la capa central de autorización

A partir de ahí pueden ejecutarse pruebas confiables.

Regla:

si el usuario, tenant o dataset cambia, debe validarse nuevamente.

No extender datos de prueba por suposición.

usar TARGET ++ para agregar método PHP nuevo
usar secciones para CSS o JS cuando corresponda
no mezclar header de archivo completo con operación parcial
no usar formatos ambiguos
no aplicar snippets generados sin revisar el encabezado

Regla crítica:

si el objetivo es modificar un método, el snippet debe usar TARGET.

Ejemplo correcto:

// TARGET: app/Providers/AppServiceProvider.php :: boot

public function boot(): void
{
    //
}

Ejemplo incorrecto:

<?php

// FILE: app/Providers/AppServiceProvider.php :: boot | V1

public function boot(): void
{
    //
}

Ese ejemplo incorrecto debe devolver ERROR o ser normalizado de forma segura hacia operación parcial.

Nunca debe crear archivo nuevo.

<<END SECTION>>

FALLAS OPERATIVAS QUE PROJECT LAB DEBE PREVENIR

Project Lab debe prevenir errores derivados de ambigüedad, formato incorrecto o interpretación excesivamente flexible.

Casos que deben considerarse fallas:

crear un archivo cuando la intención era reemplazar un método
interpretar como ruta completa un encabezado que contiene operadores de método
aceptar un bloque documental sin formato real válido
crear secciones documentales inexistentes sin autorización explícita
aplicar snippets incompletos
mezclar contratos de archivo completo y operación parcial
procesar silenciosamente formatos dudosos
devolver OK ante una operación que no cumplió el contrato esperado

Regla:

ante ambigüedad, Project Lab debe fallar.

Regla complementaria:

ante ambigüedad, el chat también debe detenerse y pedir o auditar la evidencia mínima necesaria.

No se debe resolver por intuición.

Caso incorporado:

un encabezado como:

// FILE: app/Providers/AppServiceProvider.php :: boot | V1

no puede ser tratado como archivo completo.

Debe rechazarse o transformarse de forma explícita y segura en operación parcial.

Nunca debe crear un archivo con :: boot como parte de la ruta.

Caso incorporado adicional:

un bloque documental no debe entregarse para Project Lab si no respeta el formato real:

REEMPLAZAR EN: [doc_slug]

<<SECTION: DOCTRINA OPERATIVA DE PROJECT LAB>>
SECTION_VERSION: 00001

DOCTRINA OPERATIVA DE PROJECT LAB

Project Lab no es una herramienta accesoria ni un simple panel de utilidades.

Dentro del proceso técnico de app-base, Project Lab funciona como estación operativa local para ejecutar intervenciones controladas sobre el proyecto.

Su posición doctrinal es la siguiente:

Project Lab no pertenece al producto SaaS final.
Project Lab no forma parte de la experiencia del usuario final.
Project Lab no define reglas de negocio.
Project Lab no reemplaza documentación rectora.
Project Lab no reemplaza criterio arquitectónico.

Pero Project Lab sí forma parte del método interno de trabajo del proyecto.

Project Lab gobierna operativamente:

cómo se aplican snippets de código
cómo se aplican actualizaciones documentales
cómo se ejecutan auditorías simples
cómo se reproducen pruebas Tinker
cómo se ejecutan comandos Artisan controlados
cómo se comparte evidencia técnica en el chat
cómo se valida si una intervención puede considerarse aplicada

Regla central:

toda salida destinada a Project Lab debe respetar el contrato real vigente de Project Lab.

Si el contenido entregado no respeta el formato aceptado por la herramienta, ese contenido es inválido aunque la intención conceptual sea correcta.

Regla adicional:

no se deben entregar bloques para automatizadores si no se conoce o no se verificó el formato real aceptado.

Cuando exista duda sobre el contrato de entrada de Project Lab, debe auditarse primero la herramienta o solicitarse la evidencia mínima necesaria.

No se debe improvisar formato.

No se debe asumir que una variante de formato será aceptada.

No se debe transformar una convención deseada en contrato real sin validación previa.

Project Lab debe ser estricto.

El chat también debe ser estricto cuando genera contenido para Project Lab.

<<END SECTION>>

<<SECTION: ERRORES Y LECCIONES INCORPORADAS>>
SECTION_VERSION: 00002

ERRORES Y LECCIONES INCORPORADAS

Lección incorporada:

un snippet ambiguo puede generar daño si el parser interpreta como ruta lo que en realidad era una operación parcial.

Caso detectado:

un encabezado de este tipo:

// FILE: app/Providers/AppServiceProvider.php :: boot | V1

no debe ser tratado como archivo completo.

Riesgo:

crear archivo indebido con :: boot como parte del nombre
aplicar modo incorrecto
ocultar un error de contrato
dejar residuos en el proyecto

Regla consolidada:

Project Lab debe validar antes de escribir.

FILE corresponde a archivo completo.

TARGET corresponde a operación parcial.

Si aparece operador de método dentro de FILE, debe fallar o normalizarse explícitamente hacia TARGET de forma segura.

Nunca debe crear un archivo con :: o ++ en la ruta.

Lección incorporada adicional:

una ejecución desde clipboard no debe ser invisible para el usuario.

Caso detectado:

las herramientas Lab podían ejecutar contenido leído desde clipboard sin reponer ese contenido en el textarea visible.

Riesgo:

el usuario no podía verificar qué contenido se ejecutó
la consola mostraba salida pero no dejaba visible el input aplicado
la trazabilidad operativa quedaba incompleta
se debilitaba la revisión manual previa o posterior del snippet ejecutado

Regla consolidada:

cuando Project Lab ejecute desde clipboard, debe devolver y mostrar el contenido efectivamente leído.

Esto aplica especialmente a:

code
docs
audit
Tinker desde clipboard

Regla operativa:

el textarea correspondiente debe reflejar el contenido ejecutado desde clipboard.

La consola muestra la salida.

El textarea muestra la entrada.

Ninguna ejecución desde clipboard debe quedar oculta para el usuario.
