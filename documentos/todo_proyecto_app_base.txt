SECTION_VERSION: 00008

CIERRES OPERATIVOS RECIENTES

Se cerró el frente principal de autorización operativa para orders respecto de update y delete.

Validaciones confirmadas:

orders.update bloquea órdenes closed y cancelled

orders.delete bloquea órdenes closed y cancelled

orders.delete bloquea órdenes con inventory_movements asociados

orders.delete permanece disponible solo para órdenes no terminales y sin trazabilidad bloqueante, sujeto al permiso efectivo vigente

closed → approved quedó validado como reapertura explícita operativa

Security y Gate/Policy deben permanecer alineados para estas acciones

Queda registrado como cierre doctrinal y funcional del frente auditado.

Avance operativo adicional consolidado:

inventory_operations fue incorporado como capa persistente interna de agrupación operativa contextual dentro de inventory.

Esta capa permite reunir uno o múltiples inventory_movements dentro de una misma acción operativa temporal, sin reemplazar la trazabilidad unitaria de cada movimiento de stock.

La incorporación no crea por sí misma nuevas pantallas principales, abilities visibles independientes ni ownership funcional para módulos consumidores.

Avance operativo adicional vigente:

documents quedó integrado con inventory mediante surfaces y hooks siguiendo la misma frontera modular aplicada en orders.

documents conserva ownership documental sobre cabecera y líneas.

inventory conserva ownership operativo sobre stock, ejecución, reversión y trazabilidad material.

La ejecución parcial por document_items quedó validada con cálculo derivado de ejecutado y pendiente desde movimientos reales.

Avance operativo adicional consolidado:

Orders y Documents convergieron hacia un lifecycle base común para cabeceras operativas.

Estados persistidos comunes:

draft
pending_approval
approved
closed
cancelled

La transición automática draft → pending_approval por primera línea quedó implementada y validada.

Avance reusable adicional consolidado:

Orders y Documents avanzaron en convergencia de line items mediante componentes visuales compartidos, tabs embebidas, modales comunes y matemática comercial reutilizable.

La convergencia no transfiere ownership entre módulos ni mezcla semántica de inventory con matemática comercial.

Avance UI adicional consolidado:

se unificó la gramática de tabs principales de show pages mediante x-host-tabs y helper HostTabs.

La persistencia del panel activo mediante return_tab quedó alineada para hosts principales validados.

Se completó la alineación de Orders, Documents, Parties, Tasks, Assets, Appointments, Products, Inventory y Projects al patrón común de tabs host.

Avance de limpieza técnica consolidado:

inventory conserva un partial unificado vigente para reversión por línea.

Se eliminaron partials legacy duplicados vinculados a reversión por línea en orders y documents.

La base vigente queda representada por documentación matriz y código actual, no por notas históricas internas.

Cierre operativo adicional de seguridad/auth:

se eliminó la ruta debug /whoami.

LoginResponse quedó alineado con memberships activas para resolver redirección post-login.

User::is_superadmin dejó de ser asignable mediante fillable.

2FA permanece cableado para futura habilitación, pero inactivo en el estado actual del desarrollo.

LogoutResponse limpia tenant_id e invitation_accept_url al cerrar sesión.

El flujo de invitaciones internas quedó validado end-to-end para member_invite.

Las invitaciones pendientes no crean usuarios ni memberships anticipadas.

La membership se materializa recién al aceptar la invitación.

La redirección post-aceptación quedó alineada con memberships activas.

Cierre visual adicional:

x-modal quedó redefinido como shell de overlay, backdrop, centrado, tamaño y scroll.

x-card conserva la responsabilidad de contenedor visual interno del modal.

x-line-operation-modal quedó como único consumidor directo actual de x-modal.

El modal responsive quedó ajustado para mobile con ancho completo, scroll controlado y mejor uso táctil.

Footer global quedó auditado como parte del shell interno mediante x-layout.footer.

El footer no se renderiza en páginas públicas ni en superadmin según condición vigente del layout.

El layout con contenido extenso y tablas largas quedó auditado con base visual general de scroll vertical, scroll horizontal, min-width controlado y horizontal-scroll reusable.

Validaciones recientes informadas:

syntax OK
routes OK
optimize:clear OK
browser OK
route:list sin /whoami
route:list sin two-factor
