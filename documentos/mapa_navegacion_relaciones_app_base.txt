<<SECTION: OBJETIVO>>
SECTION_VERSION: 00002

OBJETIVO

Este documento define el mapa vigente de navegación, recorridos funcionales y relaciones visibles entre módulos de app-base.

Su función es:

describir recorridos reales del sistema
ordenar navegación principal y contextual
documentar accesos entre módulos
evitar flujos inconsistentes
guiar futuras ampliaciones de navegación

No define:

seguridad doctrinal
modelo conceptual completo de entidades
contratos modulares internos
detalle visual de componentes

Esos aspectos se resuelven en otros documentos.

Regla consolidada:

este documento describe navegación real o navegación objetivo ya validada

no debe cargarse con teoría arquitectónica general que corresponda a otros documentos
<<END SECTION>>

<<SECTION: REGLA GENERAL>>
SECTION_VERSION: 00005

REGLA GENERAL

Toda navegación del sistema debe responder a una necesidad funcional real.

No debe crearse navegación por simetría visual, comodidad técnica o existencia de tablas auxiliares.

Reglas obligatorias:

navegación principal expresa estructura estable
navegación contextual expresa relaciones reales de trabajo
dataset interno no equivale a módulo navegable
host visual no equivale a ownership funcional
el trail preserva contexto entre pantallas relacionadas
la UI no debe prometer accesos no operables

Regla adicional consolidada:

si una necesidad puede resolverse como pieza reusable del plano tenant o como capacidad publicada por un módulo existente, esa salida debe evaluarse antes de abrir una nueva ruta mental en el sistema mediante navegación adicional.

<<END SECTION>>

<<SECTION: REGLAS DE LECTURA>>
SECTION_VERSION: 00002

REGLAS DE LECTURA

Una relación en este documento significa acceso, recorrido o vínculo visible para el usuario dentro del sistema.

No implica automáticamente:

foreign key obligatoria
permiso concedido
implementación ya terminada
visibilidad universal
módulo siempre habilitado

Toda navegación real sigue condicionada por:

módulo habilitado para el tenant
permiso efectivo
estado funcional del recurso
flujo operativo aplicable

Tipos de navegación relevantes:

estructural
contextual
embebida
derivada

NAVEGACIÓN ESTRUCTURAL

Recorrido estable propio del módulo.

Ejemplo:

menú principal → index → show → edit

NAVEGACIÓN CONTEXTUAL

Recorrido originado desde otro módulo conservando contexto.

Ejemplo:

appointment.show → crear order contextual

NAVEGACIÓN EMBEBIDA

Consumo de tablas, bloques o relaciones dentro de otra pantalla.

Ejemplo:

orders embebidas dentro de asset.show

NAVEGACIÓN DERIVADA

Flujo donde una acción en un módulo genera o abre otro módulo como consecuencia operativa.

Ejemplo:

orders.show → crear document

Regla:

cuando existan varias rutas posibles, debe priorizarse la más consistente con NavigationTrail y con el flujo real del usuario
<<END SECTION>>

<<SECTION: NAVEGACIÓN PRINCIPAL TENANT>>
SECTION_VERSION: 00003

NAVEGACIÓN PRINCIPAL TENANT

La navegación principal tenant expresa los módulos funcionales visibles de la empresa activa.

Debe responder al contrato real del sistema y no a mera existencia técnica de código.

Casos funcionales actuales:

dashboard
projects
tasks
appointments
parties
products
assets
orders
documents
inventory

La aparición real de cada entrada depende de:

módulo habilitado
permiso efectivo
estado operativo cuando corresponda

Regla adicional consolidada:

users no debe aparecer por esta necesidad como módulo navegable autónomo dentro del tenant.

La gestión interna de usuarios, pertenencias, roles y permisos continúa anclada al frente tenant/profile/accesses y no a una navegación principal users separada.

Regla:

la navegación principal no reemplaza flujos contextuales ni relaciones entre módulos.

Su función es dar acceso estructural al ecosistema operativo vigente.

<<END SECTION>>

<<SECTION: NAVEGACIÓN CONTEXTUAL ENTRE MÓDULOS>>
SECTION_VERSION: 00009

NAVEGACIÓN CONTEXTUAL ENTRE MÓDULOS

La navegación entre módulos debe preservar intención, contexto y continuidad de trabajo.

Reglas:

no saltar de contexto sin necesidad
no perder breadcrumb útil
no enviar al usuario a pantallas genéricas cuando existe destino contextual mejor
no duplicar rutas equivalentes con comportamientos distintos
las acciones embebidas deben volver al punto correcto del host cuando corresponda

Cuando una relación nace dentro de otro módulo, debe priorizarse navegación contextual compatible con trail vigente.

La navegación contextual puede resolverse mediante:

trailQuery
return_url cuando aplique
return_tab cuando exista contexto interno del host
acciones compactas enlazadas
surfaces embebidas con continuidad local

Caso consolidado:

una acción ejecutada dentro de orders.show debe tender a volver a la misma order y, si aplica, a la tab técnica desde la que nació la intención.

Regla adicional para estados operativos:

la navegación disponible también depende del estado real del recurso.

No debe ofrecerse como acción navegable una operación inexistente por estado, trazabilidad o autorización efectiva.

Caso vigente:

en orders.show una order closed no debe ofrecer edición ni eliminación directa, pero puede ofrecer reapertura explícita autorizada a approved.

<<END SECTION>>

<<SECTION: RELACIONES EMBEBIDAS EN SHOW PAGES>>
SECTION_VERSION: 00008

RELACIONES EMBEBIDAS EN SHOW PAGES

Las relaciones embebidas dentro de show pages deben mejorar continuidad operativa sin confundir ownership funcional.

Reglas:

el host conserva identidad principal
el módulo invitado conserva su lógica y semántica
las acciones del invitado no se reinterpretan en el host
la navegación derivada debe mantener contexto cuando corresponda
las relaciones compactas y las relaciones fuertes no deben mezclarse sin criterio

Criterios de uso:

summary_items para relaciones puntuales o de baja densidad
detail_items para bloques breves informativos
tabs o paneles amplios para relaciones fuertes
row actions oficiales para operaciones dentro de tablas anfitrionas

Caso vigente en orders.show:

inventory puede operar por línea dentro del host sin perder identidad modular.

documents y attachments pueden consumirse como capacidades invitadas.

Regla adicional para estados del host:

si el host entra en estado readonly o terminal, las relaciones embebidas no deben reabrir operabilidad incompatible con ese estado.

Las capacidades de lectura pueden permanecer.

Las capacidades de mutación deben respetar la condición superior del host.

<<END SECTION>>

FLUJOS DERIVADOS RELEVANTES

order

→ document_items por copia funcional cuando corresponda

signup_request aprobado
→ invitation owner_signup

→ owner onboarding

Reglas:

un flujo derivado expresa continuidad operativa entre módulos con identidad propia

no implica subordinación conceptual absoluta del módulo destino respecto del origen

task → order y appointment → order representan derivación hacia un módulo autónomo ya consolidado

si el flujo derivado requiere contexto, debe preservar seguridad y trail correspondiente

si existe una order ya vinculada al origen funcional, debe priorizarse apertura consistente de esa order antes de ofrecer alta duplicada

Aclaración relacional consolidada:

en task → order el vínculo estructural persistido se resuelve desde orders.task_id

en appointment → order el vínculo operativo real se resuelve desde appointments.order_id y no requiere orders.appointment_id

<<SECTION: REGLAS DEL TRAIL>>
SECTION_VERSION: 00007

REGLAS DEL TRAIL

El trail representa contexto externo entre recursos y pantallas.

No reemplaza estados internos del host.

Reglas:

cada salto relevante debe poder reconstruir origen útil
no usar trail como contenedor ambiguo de todo estado UI
preservar breadcrumb comprensible
evitar cadenas innecesarias o redundantes
el destino final debe coincidir con la intención original del usuario

Cuando además exista contexto interno del host, debe complementarse con return_tab u otro mecanismo explícito equivalente.

Distinción doctrinal:

trail = contexto externo entre recursos
return_tab = contexto interno de una pantalla anfitriona

Caso vigente:

acción disparada desde una tab de orders.show puede requerir ambos:
trail para volver a la order
return_tab para restaurar el panel correcto

Regla adicional:

las transiciones especiales de estado no deben confundirse con edición ordinaria.

Si una acción cambia estado terminal o readonly hacia uno operable, debe tratarse como acción específica del flujo y no como simple navegación a edit.

Caso vigente:

reapertura closed → approved en orders.

<<END SECTION>>

<<SECTION: MAPA DE NAVEGACIÓN PRINCIPAL>>
SECTION_VERSION: 00005

MAPA DE NAVEGACIÓN PRINCIPAL

La navegación principal del sistema organiza acceso a frentes funcionales y no solo a tablas CRUD.

Entradas vigentes del ecosistema:

dashboard
projects
tasks
appointments
parties
products
assets
orders
documents
inventory

Reglas:

cada entrada principal debe corresponder a un frente reconocible del negocio
dashboard resume o deriva, no reemplaza módulos
los módulos con identidad propia deben tender a tener acceso claro
la navegación principal no obliga a que toda relación secundaria tenga menú propio

Aclaración adicional:

un módulo puede tener navegación principal y además publicar capacidades reutilizables consumidas dentro de otros hosts.

Aclaración sobre hosts avanzados:

la centralidad de un módulo en el menú no lo convierte automáticamente en patrón rector de estructura show.

La madurez contractual de cada host se evalúa por su propia gramática de montaje.

Caso vigente:

appointments.show expresa una estructura host más avanzada que otros shows aún en transición.

<<END SECTION>>

<<SECTION: INVENTORY>>
SECTION_VERSION: 00006

INVENTORY

La navegación principal del frente inventory debe apoyarse en sus surfaces oficiales y no en accesos residuales dispersos.

Superficies vigentes:

inventory.index
inventory.show

Lectura esperada:

inventory.index como tablero operativo de saldos y acceso a productos

inventory.show como historial y detalle contextual de movimientos para un producto

Reglas:

los consumidores externos deben enlazar a inventory mediante estas surfaces cuando requieran lectura completa

si el caso requiere solo contexto breve, debe priorizarse embebido oficial o acción compacta

las vistas filtradas secundarias deben tender a reutilizar surfaces canónicas y no abrir pantallas paralelas innecesarias

Cuando inventory opere dentro de orders.show:

la navegación no debe romper continuidad del host salvo intención explícita del usuario

las acciones por línea deben permanecer en contexto local cuando sea posible

la apertura de historial filtrado puede usar surface oficial de inventory conservando trail de retorno

Regla adicional por estado superior:

si la order anfitriona está closed o cancelled, inventory no debe exponer desde ese host acciones operativas incompatibles con dicho estado.

<<END SECTION>>

<<SECTION: REGLA PARA LINKS RELACIONALES DESDE TABLAS CONTEXTUALES>>
SECTION_VERSION: 00005

REGLA PARA LINKS RELACIONALES DESDE TABLAS CONTEXTUALES

Cuando una tabla enlaza desde una entidad fila hacia una entidad relacionada, debe evaluarse cuál es el contexto real que el breadcrumb necesita preservar.

Regla práctica:

si el trabajo sigue anclado en la fila desde la que se abrió el vínculo, el breadcrumb esperado puede ser:

módulo origen
↓
entidad fila
↓
entidad relacionada

Esto aplica especialmente en:

tablas embebidas dentro de show

listados contextuales dentro de tabs

acciones por fila hacia entidades relacionadas

sublistados donde el contenedor no alcanza por sí solo para conservar el contexto de origen

Regla adicional consolidada:

en una tabla contextual, el link principal de la fila debe consolidar primero el trail completo de esa fila

los links relacionales secundarios de esa misma fila deben reutilizar ese trail ya construido y no reiniciar contexto desde el contenedor

Regla adicional para integraciones reusables:

si la fila expone una capacidad compleja ya publicada por otro módulo, no debe reconstruirse localmente por conveniencia dentro de la tabla

debe consumirse la capacidad oficial del módulo oferente también en esa superficie contextual

Casos vigentes relevantes:

linked-order-action
linked-party-action
linked-asset

Regla adicional para tablas anfitrionas con row actions:

si la tabla pertenece al host pero expone micro-acciones de otro módulo, los links y acciones secundarios deben respetar al mismo tiempo:

contexto de fila
trail del host
retorno interno del panel cuando aplique

Esto evita que la navegación termine desviando al usuario hacia el recurso delegado como destino final de trabajo.

<<END SECTION>>

DETALLE DE FLUJOS RELEVANTES

TASKS → ORDERS

La tarea puede derivar una orden dentro del flujo operativo.

El usuario conserva contexto mediante trail.

APPOINTMENTS → ORDERS

El turno puede originar una orden contextual asociada a la atención planificada.

ASSETS → APPOINTMENTS

Desde un activo puede consultarse o gestionar agenda relacionada.

ASSETS → ORDERS

Desde un activo puede abrirse historial operativo o crear nueva orden contextual.

ASSETS → DOCUMENTS

Desde un activo puede abrirse documentación relacionada cuando corresponda.

PARTIES → APPOINTMENTS

Desde un contacto puede consultarse agenda asociada.

PARTIES → ORDERS

Desde un contacto puede abrirse o crear orden contextual.

PROJECTS → TASKS

El proyecto actúa como agrupador y punto de acceso contextual a tareas relacionadas.

ORDERS → INVENTORY POR LÍNEA

La orden puede consumir inventory como frente operativo embebido sobre sus líneas.

Flujos vigentes consolidados:

orders.show
→ tab operativa de inventory
→ acción por fila
→ redirect al mismo host con misma tab

orders.show
→ embebido de inventory por línea
→ ver movimientos filtrados
→ inventory.show con filtro `order_item_id`

orders.show
→ modal Surtir línea
→ confirmar o cancelar
→ retorno local al mismo host

orders.show
→ acción Devolver por línea
→ registro de contramovimiento
→ retorno al mismo host con recálculo operativo visible

Regla:

cuando el flujo siga anclado en la orden anfitriona, la navegación no debe cerrar accidentalmente en el recurso delegado de línea como si fuera pantalla final de trabajo.

<<SECTION: CRITERIOS DE AUDITORÍA FUTURA>>
CRITERIOS DE AUDITORÍA FUTURA

Al auditar breadcrumb, trail o navegación entre módulos, verificar:

si la pantalla debía abrirse en modo estructural o contextual

si el trail se propagó correctamente

si hubo colapso de nodos repetidos

si volver y cancelar respetan el nodo previo real

si redirects post acción mantienen el contexto útil

si el breadcrumb visible sigue siendo compacto

si navbar y sidebar marcan el área funcional esperada y no solo la hoja final abierta

si en tablas contextuales el trail correcto era el del contenedor o el de la fila

si hay pantallas donde el contexto se pierde por filtros, tabs, paginación o acciones secundarias

si los enlaces, acciones y campos relacionales hacia módulos relacionados verifican primero que el módulo destino esté habilitado para el tenant

si la deshabilitación de módulos destino elimina completamente:

accesos
bloques visuales
referencias indirectas

especialmente en flujos desde appointments hacia orders y assets
<<END SECTION>>

<<SECTION: CRITERIO FINAL>>
SECTION_VERSION: 00002

CRITERIO FINAL

La navegación correcta del sistema no es la que muestra más enlaces.

Es la que permite al usuario llegar de forma clara, segura y coherente a la acción que necesita.

Una relación visible entre módulos se justifica cuando:

mejora continuidad operativa
reduce fricción real
respeta permisos
respeta módulos habilitados
mantiene contexto útil
no duplica caminos innecesarios

Si una relación existe técnicamente pero no cumple estos criterios, no debe priorizarse como navegación visible.
