CONVENCIONES-DEL-PROYECTO – app-base


OBJETIVO DEL DOCUMENTO

Este documento define las convenciones estructurales y de desarrollo del proyecto app-base.

Su objetivo es mantener consistencia técnica, previsibilidad arquitectónica y coherencia de código a medida que el sistema crece.

Las convenciones aquí definidas deben respetarse al crear:

nuevos módulos
nuevas entidades
nuevas vistas
nuevos controladores
nuevos flujos operativos


CONVENCIONES GENERALES

El proyecto sigue los principios de:

simplicidad
claridad estructural
baja duplicación
coherencia entre módulos
previsibilidad del comportamiento del sistema

Cuando existan varias alternativas técnicas, se debe priorizar siempre la opción más simple y consistente con el resto del proyecto.


CONVENCIONES DE NOMBRES

MODELOS

Los modelos deben estar en singular.

Ejemplos:

User
Tenant
Membership
Party
Project
Order
Document
Asset

TABLAS

Las tablas deben utilizar plural en snake_case.

Ejemplos:

users
tenants
memberships
parties
projects
orders
documents
assets

CONTROLADORES

Los controladores utilizan el sufijo:

Controller

Ejemplos:

UserController
OrderController
DocumentController
AssetController

RUTAS

Las rutas utilizan plural.

Ejemplos:

/users
/orders
/documents
/assets


CONVENCIÓN ACTUAL DE DECLARACIÓN DE RUTAS CRUD

Como convención actual del proyecto, las rutas CRUD del core funcional deben declararse explícitamente en routes/web.php.

No se utilizará Route::resource(...) mientras el proyecto mantenga compatibilidad objetivo con hosting compartido básico sin CLI, salvo revisión técnica posterior en un entorno más robusto.

Esta decisión busca priorizar previsibilidad operativa y reducir riesgo en despliegues cerrados preparados en local.


CONVENCIONES DE MULTI-TENANCY

Las entidades del negocio deben incluir siempre:

tenant_id

El tenant determina el aislamiento lógico de los datos dentro del sistema.

Las entidades del sistema no deben incluir tenant_id.

Ejemplos de entidades del sistema:

User
Tenant
Membership


ACCESO OPERATIVO POR TENANT

La restricción operativa de acceso a empresa debe validarse por memberships.status = active.

No debe utilizarse desactivación global de users para resolver bajas, bloqueos o pérdida de acceso específicos de un tenant.


CONVENCIONES DE ACCESO POR MÓDULO Y ROLES

La autorización operativa del sistema debe resolverse diferenciando:

módulo habilitado para el tenant
módulo permitido para el rol del usuario
visibilidad del registro
acción permitida

Reglas actuales consolidadas:

owner y admin equivalen funcionalmente a nivel de control operativo del tenant

roles operativos base esperados:

sales
operator

como regla mínima por módulo:

owner y admin tienen control total operativo

los roles operativos pueden ver el módulo habilitado y crear o editar registros propios o de su responsabilidad operativa, según defina la policy del módulo

la eliminación sensible debe reservarse a owner y admin, salvo necesidad funcional justificada

la visibilidad del dashboard y de métricas agregadas debe respetar el rol y no asumir siempre conteos globales del tenant


ROLECATALOG Y MODULECATALOG

RoleCatalog es el catálogo base mínimo y reservado del sistema para roles estructurales.

ModuleCatalog es la fuente de verdad para slugs de módulos cerrados del sistema.

Reglas:

no hardcodear slugs funcionales del sistema en controllers, policies, helpers o blades si ya existe catálogo técnico

los roles base del sistema deben resolverse desde RoleCatalog

los módulos base del sistema deben resolverse desde ModuleCatalog

los roles base sembrados al crear el tenant no deben modificarse ni eliminarse

a futuro el tenant podrá agregar roles propios en base de datos, pero sin alterar la base reservada del sistema


EVOLUCIÓN PREVISTA DE PERMISOS

La arquitectura de permisos debe prepararse para incorporar una capa centralizada compuesta por:

RolePermissionMatrix
RolePermissionResolver
TenantModuleAccess

Reglas de diseño:

las policies no deben hardcodear permisos dispersos cuando la capa central ya exista

la policy debe consumir la capa central de permisos en lugar de multiplicar lógica por módulo

la matriz actual puede ser hardcodeada, pero debe diseñarse para migrar más adelante a persistencia en base de datos

cuando exista resolución central disponible, no hardcodear permisos por rol directamente en controllers, blades o componentes

toda autorización funcional nueva debe apoyarse, según corresponda, en:

RolePermissionMatrix
RolePermissionResolver
TenantModuleAccess
Policy

el acceso a módulo no reemplaza la policy por registro

el acceso a módulo, la visibilidad de registros y la capacidad de acción deben tratarse como capas separadas

cuando un módulo tenga reglas específicas por ownership, asignación o participación operativa, la policy debe resolver esa diferencia reutilizando la capa central y el criterio de visibilidad del módulo


TRAIT TENANTSCOPED

Las entidades del negocio deben utilizar el trait:

TenantScoped

Este trait se encarga de:

filtrar automáticamente por tenant_id
asignar tenant_id al crear registros

Esto garantiza que los datos de diferentes empresas no se mezclen.


USO DE TENANT_ID EN FILLABLE

Como regla general, tenant_id no debe exponerse en formularios ni venir desde input del usuario.

El tenant siempre debe resolverse internamente mediante:

app('tenant')

y asignarse automáticamente por el sistema.

Algunos modelos padre del proyecto exponen tenant_id en fillable por consistencia histórica, pero igualmente no debe venir desde formularios públicos ni desde input del usuario.

En entidades hijas (por ejemplo items o registros dependientes), tenant_id tampoco debe exponerse desde formularios del usuario.

Sin embargo, si el registro hijo se crea internamente desde un flujo controlado del sistema y el tenant se hereda explícitamente desde la entidad padre, puede incluirse tenant_id en fillable para permitir la creación interna consistente mediante create().

Esto permite manejar correctamente casos como:

document_items creados desde documentos
order_items copiados hacia documentos derivados


CONVENCIONES DE ENTIDADES HIJAS

Las entidades hijas deben cumplir:

pertenecer a una cabecera
heredar tenant_id de la cabecera
no recibir tenant_id desde formularios del usuario

Ejemplos:

order_items
document_items

Cuando la autorización de la entidad hija dependa naturalmente de la cabecera:

debe heredar autorización desde la entidad padre

debe evitarse una policy independiente compleja si no aporta valor real

Esto ya se considera válido para entidades como:

order_items respecto de orders
document_items respecto de documents


CONVENCIONES DE OWNERSHIP FUNCIONAL Y PERMISOS OPERATIVOS SIMPLES

Cuando un módulo lo permita, puede adoptarse un criterio simple de ownership funcional del registro en lugar de una capa temprana de permisos compleja.

Reglas:

usuario normal gestiona los registros de su responsabilidad operativa

owner y admin pueden gestionar registros ajenos dentro del tenant

los borrados sensibles pueden reservarse a owner y admin aunque la edición o seguimiento sea más amplia

si un módulo tiene asignado directo, assigned_user_id define la responsabilidad operativa actual

si el registro puede reasignarse, el nuevo asignado pasa a ser responsable operativo

si no existe propietario directo, el control sensible queda en owner y admin o en la policy específica del módulo

Este patrón ya se considera válido para tasks.


REGLA PARA MÓDULOS OPERATIVOS COMPARTIDOS DEL TENANT

Cuando un módulo operativo compartido del tenant no tenga responsable operativo directo:

la policy debe permitir view, create y update a los roles habilitados del módulo

delete debe quedar restringido a owner y admin

no debe inventarse ownership artificial mediante assigned_user_id, owner_user_id o restricciones por created_by si el módulo no lo necesita realmente

Esto aplica especialmente a módulos base o compartidos como:

parties
assets
products
orders


CONVENCIONES DE SUPPORT Y HELPERS DE ACCESO

La lógica transversal de acceso tenant/roles no debe dispersarse en controllers o blades.

Debe centralizarse preferentemente en clases de soporte ubicadas bajo:

app/Support/Auth

Clases conceptuales ya alineadas con esta convención:

TenantAccess
TenantModuleAccess
RoleModuleAccess

Cuando la visibilidad de registros dependa del módulo, debe resolverse preferentemente en clases específicas de soporte, por ejemplo:

Support/<Modulo>/<Entidad>Visibility

Esto ya se considera válido para la visibilidad operativa de projects.


CONVENCIONES DE POLICIES

La autorización por acción debe resolverse preferentemente mediante Policies.

Acciones típicas:

viewAny
view
create
update
delete

Reglas:

evitar lógica de permisos duplicada en controllers

usar controllers delgados y delegar la autorización a policies o helpers de acceso

si una policy necesita conceptos cerrados del sistema, reutilizar ModuleCatalog, RoleCatalog y support de acceso en lugar de strings hardcodeados

cuando exista una policy, no alcanza con crearla:
el controller debe invocar authorize(...) o authorizeResource(...)
la vista debe respetar permisos reales con @can al mostrar acciones

una policy que no se consume desde controller y vista no protege efectivamente el sistema


REGLA DE NAVEGACIÓN Y VISIBILIDAD DE UI

La interfaz no debe mostrar accesos que el usuario no puede usar realmente.

Reglas:

navbar, dashboard y accesos rápidos deben respetar acceso real del usuario al módulo

las acciones visibles en listados, show y sublistados deben respetar permisos reales mediante @can

la navegación no debe asumir acceso total al tenant si el módulo o la acción están restringidos por rol

la navegación principal del tenant debe construirse desde ModuleCatalog cuando exista una capa de navegación central disponible

no se deben hardcodear módulos navegables en componentes de layout cuando el sistema ya pueda resolverlos desde catálogo y acceso efectivo


CONVENCIONES DE BLADE Y FRONTEND LIVIANO

Las vistas Blade deben mantenerse simples, explícitas y previsibles.

Reglas:

evitar lógica de negocio en Blade

permitir condicionales simples de interfaz

extraer bloques repetidos a componentes o partials cuando el patrón ya sea claro

no introducir complejidad de frontend innecesaria si la interacción puede resolverse con HTML, Blade, CSS y JavaScript mínimo

cuando una interacción visual se repita en varias pantallas, priorizar componente o patrón reusable antes que copiar markup

para comportamiento JavaScript simple, usar data-* claros y estables en lugar de acoplar JS a clases puramente visuales cuando eso ayude a separar estilo y comportamiento

Separación recomendada:

clases CSS = estilo
data-* = comportamiento

Preferir soluciones simples y declarativas antes que lógica distribuida en múltiples blades.


CONVENCIONES DE PRESERVACIÓN DE CONTEXTO UX

En vistas con tabs, filtros o acciones internas, debe preservarse el contexto funcional al refrescar o redirigir cuando eso mejore la UX.

Ejemplos de contexto a preservar:

tab activa
filtros
paginación cuando aplique

Las tabs mencionadas en esta convención refieren a tabs internas de contenido dentro de una pantalla.

No debe asumirse todavía como convención general del sistema una navegación entre rutas dentro de tabs de app o tipo workspace.


NAVEGACIÓN ESTRUCTURAL Y CONTEXTUAL

La navegación del sistema debe distinguir entre:

navegación estructural
navegación contextual

navegación estructural:
expresa la ubicación estable de la pantalla dentro del sistema o dentro de su módulo

navegación contextual:
expresa el origen funcional desde el cual el usuario llegó a una entidad relacionada, cuando ese origen sigue siendo relevante para la tarea actual

Reglas generales:

toda pantalla debe tener una resolución estructural válida como base, aunque en ciertos flujos pueda mostrarse en modo contextual

la navegación contextual solo debe aplicarse cuando exista un flujo relacionado real y útil para la continuidad de uso

no debe inventarse un origen contextual si la pantalla fue abierta de forma directa, desde index propio, desde búsqueda o desde navegación general del módulo

el breadcrumb del sistema es uno solo, pero debe poder resolverse en modo estructural o en modo contextual según el caso

si ya existe una capa central reutilizable para resolver contexto de navegación, no duplicar esa lógica en múltiples controllers, blades o helpers dispersos

controller resuelve contexto
vista consume contexto

no reconstruir breadcrumb de negocio desde url()->previous() ni desde supuestos del historial del navegador

breadcrumb, retorno al origen y redirects posteriores a acciones deben mantenerse consistentes entre entidades relacionadas

la preservación de contexto UX debe aplicarse con criterio y solo cuando aporte continuidad real de uso

como regla general:

usar navegación estructural por defecto

usar navegación contextual solo como override útil cuando el origen funcional deba preservarse

cuando no exista contexto válido o el origen no agregue valor operativo, debe mostrarse navegación estructural normal


RESOLUCIÓN TÉCNICA DEL TRAIL DE NAVEGACIÓN

Cuando la navegación contextual ya exista como patrón real del sistema, debe resolverse mediante un mecanismo central reutilizable y estable.

Reglas:

preferir un motor genérico de trail reutilizable antes que variantes dispersas por módulo

cuando un módulo necesite lógica propia de navegación contextual, debe apoyarse en resolutores, builders o adaptadores compatibles con el mecanismo central vigente

si una pantalla relacionada recibe un trail válido, debe poder heredarlo, extenderlo o colapsarlo sin reconstruir manualmente la navegación desde parámetros sueltos o historial del navegador

cuando el trail vuelva a un nodo ya existente, debe priorizarse una resolución limpia y no duplicada

no sostener en paralelo mecanismos redundantes de breadcrumb contextual si ya existe una forma central adoptada por el proyecto

los nodos del trail deben almacenar URLs limpias de la entidad actual

no debe serializarse el propio trail dentro del campo url de los nodos

el trail se propaga solo en links, forms y redirects

si un módulo recibe trail válido, debe heredarlo y colapsarlo; no debe reinyectarlo dentro de la definición interna del nodo

cuando un módulo ya esté migrado al trail central, deben eliminarse mecanismos legacy de contexto equivalentes o redundantes

las decisiones técnicas concretas de clases o nombres de implementación pueden evolucionar, pero la convención estable es:

motor central reutilizable
controller resuelve
vista consume
sin deducción desde historial
sin duplicación por módulo
sin serialización recursiva del trail dentro de nodos


CONVENCIONES DE VOCABULARIO UI

Nombre genérico visible de la entidad parties en interfaz:

Contacto

Cliente debe utilizarse solo como categoría, clase o tipo específico dentro del catálogo correspondiente, no como nombre genérico de la entidad.

Para asignación interna de tareas, el término visible recomendado es:

Colaborador

El verbo destructivo estándar en botones, confirmaciones y mensajes del sistema es:

Eliminar


CONVENCIONES DE ACCIONES VISUALES

Fuera de tablas, los botones principales pueden usar icono + texto cuando eso refuerce la asociación visual de la acción.

Dentro de tablas o celdas compact-actions-cell, las acciones deben mostrarse solo con icono.

Todo botón de solo icono debe incluir:

title
aria-label


CONVENCIONES DE SEGURIDAD

El sistema debe evitar que un usuario pueda acceder a registros de otro tenant.

Esto se logra mediante:

TenantScoped
ResolvesTenantRouteBinding

Nunca se deben realizar consultas que ignoren el tenant actual.


CONVENCIONES DE FLUJO DE INVITACIONES

Las invitaciones forman parte del flujo de onboarding y acceso al sistema.

Cuando una invitación corresponde a un usuario que ya posee cuenta en el sistema, el flujo debe:

redirigir al login si el usuario no está autenticado

una vez autenticado, reanudar el flujo original de la invitación

En estos casos no debe aplicarse la redirección estándar posterior al login.

El sistema debe retornar al flujo de aceptación de la invitación.

Esto garantiza continuidad del proceso de incorporación al tenant correspondiente.


CONVENCIÓN DE SIMPLICIDAD

Antes de introducir una nueva abstracción, evaluar si realmente es necesaria.

El sistema prioriza:

claridad
consistencia
bajo nivel de complejidad accidental

Las soluciones simples y previsibles deben preferirse sobre arquitecturas excesivamente abstractas.


CONVENCIONES DE CSS Y PATRONES VISUALES

La organización del CSS visual del proyecto debe seguir esta base:

public/css/app-base.css
public/css/app-patterns.css
public/css/modules/<modulo>.css

Reglas:

app-base.css concentra la base visual global obligatoria

app-patterns.css concentra patrones visuales reutilizables y neutrales compartidos entre módulos

modules/<modulo>.css debe reservarse para estilos específicos de módulos complejos

no duplicar variantes visuales equivalentes con prefijos de módulo si ya existe un patrón reusable neutral

cuando un patrón visual deje de ser exclusivo de un módulo, debe migrarse a nomenclatura neutral reusable

preferir nombres neutrales cuando el patrón ya se utilice en más de un módulo


CONVENCIONES DE NAVEGACIÓN CONTEXTUAL

Cuando exista navegación cruzada entre entidades relacionadas, no debe resetearse el contexto de origen sin necesidad funcional real.

Reglas:

si ya existe una capa central reutilizable para resolver contexto de navegación, no duplicar esa lógica en múltiples controllers

no hardcodear variantes divergentes por módulo si la regla de navegación ya existe a nivel reutilizable

la preservación de contexto UX debe aplicarse con criterio y solo cuando aporte continuidad real de uso

breadcrumb, retorno al origen y preservación de contexto deben resolverse de forma consistente entre módulos relacionados
