<<SECTION: OBJETIVO DEL DOCUMENTO>>

OBJETIVO DEL DOCUMENTO

Este documento define la base visual común del sistema.

Su objetivo es asegurar una interfaz coherente, reusable, limpia y predecible entre módulos.

No documenta decisiones de negocio ni reglas funcionales profundas.

Documenta:

estructura visual base
componentes reutilizables
criterios de composición de vistas
patrones visuales compartidos
jerarquía visual entre módulo principal, relaciones embebidas y capacidades auxiliares reutilizables

Aclaración consolidada:

la base visual debe acompañar la arquitectura modular vigente del proyecto

esto implica expresar visualmente:

qué pertenece al módulo principal de la pantalla
qué corresponde a relaciones operativas relevantes
qué corresponde a capacidades transversales o de integración reutilizable

No corresponde definir aquí contratos doctrinales ni reglas de autorización.

Esos aspectos se resuelven en otros documentos del proyecto.
<<END SECTION>>

<<SECTION: ESTRUCTURA GENERAL DE PÁGINA>>
ESTRUCTURA GENERAL DE PÁGINA

Toda pantalla principal del sistema debe construirse sobre una estructura consistente.

Base esperada:

x-page
x-breadcrumb
x-page-header
contenido principal dentro de cards o bloques equivalentes
footer reusable del shell general de la app cuando corresponda al layout interno del sistema

La estructura debe mantener:

jerarquía visual clara
encabezado consistente
acciones principales visibles
lectura simple del contenido

Reglas:

el footer general de la app forma parte del shell reusable de pantallas internas no públicas

no debe resolverse como bloque inline aislado por vista o por módulo

su presencia debe mantenerse consistente dentro del layout general y fuera del flujo público o de layouts especiales que no compartan ese shell

el footer debe considerarse parte estructural del layout base y no un componente opcional por pantalla

el header/navbar forma parte del shell global y debe mantener una identidad visual consistente basada en iconografía de marca
<<END SECTION>>

COMPONENTES DISPONIBLES

El sistema utiliza una base visual reusable compuesta por layouts, componentes Blade y patrones UI consistentes.

Componentes estructurales vigentes:

x-page
x-page-header
x-breadcrumb
x-card
x-modal
x-empty-state

Componentes de lectura show vigentes:

x-show-summary
x-show-summary-item
x-show-summary-item-detail-block
x-host-tabs

Componentes de acción frecuentes:

botones primarios
botones secundarios
botones icon-only
acciones inline
toolbars contextuales

Componentes de iconografía reutilizable:

resources/views/components/icons

Los íconos SVG reutilizables deben vivir como componentes Blade dentro de esta familia cuando su uso sea transversal.

Los nombres de íconos deben ser genéricos por figura o representación visual, no por módulo, para permitir reutilización entre pantallas.

Ejemplos de nombres válidos:

calendar
user-group
screen
list-check
folder
archive-box
file-text
paperclip
grid

Componentes modulares reusables vigentes:

linked-order-action
linked-party-action
linked-asset-action

Nueva línea evolutiva consolidada:

linked-order
linked-party
linked-asset
linked-project

Cuando un módulo migra a la nueva generación, el componente visual debe limitarse a representar el payload oficial resuelto por el módulo oferente.

No debe reinterpretar permisos, rutas ni estados internos.

Componentes reutilizables vigentes para line items:

x-line-item-form
x-line-item-table
x-line-operation-modal
x-tabs-embedded
x-dev-component-version

x-dev-component-version puede utilizarse como ayuda visual de versionado en Blades intervenidos.

Props vigentes:

name
version
align

No debe usarse la prop file.

Debe ubicarse al final visible del componente o card cuando corresponda, sin insertarlo dentro de tablas ni formularios.

Estos componentes pueden ser consumidos por orders y documents cuando la gramática visual sea equivalente.

Su existencia no transfiere ownership funcional entre módulos.

Componentes de formularios frecuentes:

inputs de texto
textareas
selects
date pickers
checkboxes
radios
mensajes de error
help text contextual

Componentes de interacción contextual vigentes:

stepper numérico
acciones por Enter
micro-superficies operativas en modal
acciones por fila publicadas por módulos oferentes

Comportamientos frontend reutilizables vigentes:

formularios progresivos activados mediante data-action="app-progressive-form"

Este comportamiento no reemplaza componentes Blade ni valida reglas de negocio.

Solo organiza visualmente la aparición progresiva de secciones de un formulario cuando la carga guiada reduce error operativo.

Caso vigente:

parties._form utiliza formulario progresivo para guiar tipo de contacto, relación con la empresa y datos del contacto.

Lectura vigente en parties._form:

Tipo de contacto expresa parties.kind.

Relación con la empresa expresa party_roles.role.

Regla:

el formulario puede mostrar relaciones disponibles según allowedPartyRoles, pero no decide autorización.

La UI expresa permisos ya resueltos.

El backend autoriza y bloquea.

Regla adicional consolidada:

cuando un selector dependa de dataset interno reusable del tenant, su fuente lógica puede compartirse sin convertir automáticamente esa necesidad en componente global nuevo ni en módulo visual independiente.

Aclaración adicional consolidada:

x-modal ya no se limita a diálogos simples.

Forma parte de la base visual para micro-superficies operativas contextualizadas dentro de show pages y tabs embebidas.

Caso vigente:

modal de Surtir línea dentro de orders.show.

Caso vigente adicional:

modal de operación por línea documental dentro de documents.show, consumido como micro-superficie operativa de inventory y no como componente exclusivo de documents.

Aclaración adicional para métricas embebidas:

x-tabs-embedded puede mostrar valor principal y valor secundario contextual mediante subvalue cuando el bloque requiera lectura económica o métrica dual.

Caso vigente:

monto estructural principal y monto ejecutado o entregado como dato secundario de menor jerarquía visual.

Aclaración adicional sobre tabs host:

x-host-tabs es el componente visual estándar para tabs principales de páginas show.

Su función es centralizar estructura visual, toolbar, scroll horizontal, paneles y estado activo de tabs del host.

Puede representar icon cuando venga explícito en la tab.

Cuando la tab provenga de una surface modular enriquecida, puede utilizar module_icon como fallback.

No reemplaza x-tabs-embedded.

x-tabs-embedded continúa correspondiendo a segmentaciones internas de bloques embebidos o capacidades invitadas.

PATRÓN DE INTERACCIÓN MODAL

Los modales deben utilizarse para ejecutar acciones puntuales con foco operativo y mínima fricción.

Reglas generales:

resolver tareas breves sin abandonar contexto
mostrar solo datos necesarios
mantener confirmaciones claras
evitar sobrecarga visual
cerrar con retorno natural al flujo origen

Regla adicional de evolución interna:

cuando una mejora funcional pueda resolverse en dominio o servicios sin exigir nueva interacción del usuario, debe priorizarse esa vía antes de agregar nuevas pantallas o pasos visuales.

Caso vigente:

la agrupación interna de inventory mediante operaciones contextuales se resuelve backend-first y no exige nuevos modales obligatorios para el usuario.

Caso vigente adicional:

las operaciones por línea compartidas entre orders y documents deben sostener una experiencia visual homogénea cuando representan la misma intención operativa.

El modal debe actuar como micro-superficie reusable de operación por línea, con payload contextual según host, sin duplicar componentes visuales por módulo cuando la gramática de uso sea equivalente.

Elementos esperados para operaciones por línea:

producto o referencia operativa
cantidad ejecutada
cantidad pendiente
stock actual cuando corresponda
selector o stepper numérico
ayuda contextual inline
confirmación clara desde el header o acción principal del modal

Regla:

la homogeneidad visual no transfiere ownership funcional.

El modal puede ser compartido o reusable, pero la semántica operativa permanece en el módulo oferente correspondiente.

<<SECTION: JERARQUÍA VISUAL MODULAR>>
SECTION_VERSION: 00004

JERARQUÍA VISUAL MODULAR

La UI debe expresar distintos niveles funcionales dentro de una pantalla.

NIVEL 1 — MÓDULO PRINCIPAL

Corresponde al recurso central de la pantalla actual.

Debe tener mayor jerarquía visual mediante:

título claro
summary principal
acciones principales
detalle dominante

NIVEL 2 — RELACIONES OPERATIVAS RELEVANTES

Corresponde a módulos relacionados que aportan continuidad de trabajo.

Debe resolverse mediante:

tabs
cards secundarias
tablas embebidas
bloques relacionados

NIVEL 3 — CAPACIDADES AUXILIARES O TRANSVERSALES

Corresponde a funciones reutilizables o de soporte.

Ejemplos:

attachments
bloques de ayuda
lookups
acciones contextuales
capacidades embebidas reutilizables
lecturas de actividad operativa cuando correspondan

Reglas:

no todos los bloques deben competir en jerarquía visual
la pantalla debe dejar claro qué pertenece al módulo actual y qué corresponde a una capacidad consumida
si todo se presenta con igual peso visual, la pantalla pierde claridad operativa

Aclaración consolidada para hosts visuales:

cuando una show page hospede capacidades embebidas de otros módulos, la UI debe distinguir con claridad entre:

núcleo del host
capa estructural propia del host
capacidad operativa o documental invitada del módulo oferente

el usuario debe poder entender qué pertenece al recurso actual y qué está siendo consumido desde otro frente del ecosistema

Aclaración adicional para summary del host:

las cards o bloques summary propios del módulo anfitrión no deben desaparecer por incorporar surfaces externas.

El summary del host conserva su núcleo visual y las integraciones ocupan slots adicionales compatibles.

Aclaración adicional para actividad operativa:

una lectura de actividad no debe competir visualmente con el núcleo del recurso actual.

Debe presentarse como capacidad transversal de lectura, no como segundo módulo principal dentro de la pantalla.

Si se incorpora en una show page, debe mantener jerarquía secundaria y lectura escaneable.

<<END SECTION>>

SELECTORES REUTILIZABLES

Los selectores reutilizables deben resolver necesidades repetidas sin duplicar consultas, copy ni comportamiento visual.

Reglas:

usar selector reusable cuando la necesidad se repite en múltiples módulos
mantener naming claro y orientado al dominio
no convertir datasets internos en módulo visual por inercia
la fuente de datos reusable no obliga a una única representación visual

Casos consolidados:

selector de parties
selector de assets
selector de products

Caso adicional consolidado:

selectores de colaboradores internos del tenant pueden consumir una misma fuente lógica reusable basada en pertenencia activa de la empresa.

Eso no exige todavía un componente visual global único.

Cada formulario puede adaptar copy o UX mientras respete la fuente común y la validación central.

Caso UX validado:

en tasks el selector puede ofrecer una opción funcional equivalente a “Asignarme a mí” alineada con fallback backend al usuario actual.

OFERTAS DE INTEGRACIÓN VISUAL

Las ofertas de integración visual permiten que un módulo publique capacidades consumibles por otras pantallas del ecosistema.

Deben preservar:

ownership funcional del módulo oferente
baja fricción visual
coherencia con el host

la información complementaria debe sumar contexto y no competir con el dato principal de la pantalla anfitriona

no debe requerirse una nueva pantalla si la capacidad puede resolverse en una surface existente

IDENTIDAD VISUAL Y BRANDING DEL HEADER

El sistema utiliza una identidad visual basada en iconografía SVG en el header/navbar.

no deben agregarse efectos visuales adicionales sobre el contenedor del brand

favicon y logo cumplen roles distintos y no deben confundirse

SECTION_VERSION: 00006

PATRÓN GENERAL PARA VISTAS INDEX

Toda vista index debe priorizar lectura rápida, foco operativo y consistencia con el sistema.

Estructura esperada:

header claro
acciones principales visibles
filtros consistentes
tabla o listado principal dominante
paginación consistente
estados vacíos claros

Reglas:

la tabla principal debe ser el centro visual de la pantalla

los bloques secundarios no deben competir con el listado principal

los indicadores resumidos solo deben existir si agregan valor operativo real

no convertir index operativos en dashboards pesados por inercia

mantener jerarquía visual simple y escaneable

Aclaración consolidada para inventory.index:

si incorpora métricas o indicadores rápidos, estos deben tener bajo protagonismo visual

deben acompañar la tabla principal de saldos y no desplazarla como foco de trabajo

tipografía sobria
bajo peso visual
lectura rápida

Aclaración vigente para parties.index:

el filtro visual de parties debe respetar la separación conceptual vigente.

Tipo de contacto corresponde a parties.kind.

Relación con la empresa corresponde a party_roles.role.

Cuando la visibilidad dependa de allowed_party_roles, la UI debe expresar la relación con la empresa permitida para el usuario sin reconstruir seguridad localmente.

Regla:

parties.index no debe decidir creación u operación por allowed_kinds.

Debe expresar la disponibilidad ya resuelta por el backend y por el contrato vigente de allowed_party_roles.

<<SECTION: SUMMARY Y BLOQUES VISUALES REUTILIZABLES>>
SUMMARY Y BLOQUES VISUALES REUTILIZABLES

Los bloques de summary, métricas o visualización rápida deben poder resolverse mediante wrappers o partials reusables cuando el patrón ya sea claro.

No deben desarrollarse variantes aisladas por módulo si la misma estructura visual puede reutilizarse.

Cuando un patrón visual deje de ser exclusivo de un módulo, debe migrarse a una forma neutral reusable.

Regla específica:

los bloques analíticos no deben resolverse utilizando x-show-summary ni sus derivados cuando la información a representar sea de tipo métrico, agregado o comparativo.

Para esos casos debe utilizarse un patrón visual específico de analytics separado del summary del registro.
<<END SECTION>>

SISTEMA VISUAL DE ANALYTICS

Los indicadores visuales del sistema deben ayudar a decidir, no decorar.

<<SECTION: PATRÓN VISUAL PARA VISTAS SHOW>>
SECTION_VERSION: 00016

PATRÓN VISUAL PARA VISTAS SHOW

Las vistas show deben responder a una gramática común y reconocible.

Estructura anfitriona vigente:

x-page
x-breadcrumb
x-page-header
x-show-summary
zonas adicionales según módulo
zona tab coordinada cuando corresponda

Zonas mínimas consolidadas de montaje:

header_actions
summary_items
detail_items

Cuando exista navegación tabular, no debe resolverse como renders aislados sin estructura.

Debe construirse como una zona tab coordinada del host con:

navegación
toolbar contextual
acciones auxiliares
paneles de contenido

Cuando se trate de tabs principales del host, debe priorizarse x-host-tabs como componente estándar.

El host debe entregar:

colección de tabs
key activa validada
label contextual del toolbar
contenido de cada panel

El componente visual no decide semántica de dominio ni ownership funcional.

Reglas:

el host conserva identidad visual propia
las capacidades invitadas mantienen identidad modular
las relaciones compactas se muestran en zonas breves
las relaciones fuertes pueden vivir en tabs o bloques amplios
el patrón más maduro validado del ecosistema debe servir de referencia evolutiva

Caso vigente:

appointments.show expresa una referencia positiva de madurez host para otras show pages surface-driven.

Aclaración adicional consolidada:

cuando una show page consuma una surface operativa madura de otro módulo, debe invitarla con limpieza visual y sin reconstrucciones redundantes del consumidor.

Caso vigente:

orders.show consumiendo inventory como frente operativo por línea.

Caso vigente adicional:

documents.show consumiendo inventory como frente operativo por línea documental.

En este caso, documents conserva su jerarquía visual como cabecera y líneas documentales, mientras inventory aporta acciones y lectura operativa sin convertirse en núcleo visual del documento.

Aclaración vigente para parties.show:

parties.show debe mantener la lectura separada entre:

tipo de contacto
relación con la empresa

La decisión visual sobre relaciones embebidas o compactas debe revisarse según party_roles.role y no según parties.kind.

Regla:

no agregar embebidos manuales en parties.show.

Si falta una relación visible, debe detectarse si existe surface oficial del módulo oferente o si corresponde evolucionar la oferta del módulo dueño.

Aclaración adicional para actividad en parties.show:

la futura tab Actividad no debe construirse como embebido manual ni como inferencia visual desde campos actuales del contacto.

Debe consumir actividad operativa persistida cuando exista surface o lectura autorizada definida.

La lectura mínima esperada para filas de actividad debe poder representar:

fecha
módulo
registro enlazado
tipo de actividad

La columna Registro debe contener el enlace al registro relacionado.

No debe agregarse una columna adicional solo para el link.

Aclaración adicional para detalle de movimientos:

cuando inventory.movements.show exponga una operación asociada mediante inventory_operations, esa información debe leerse como contexto adicional del movimiento y no como nueva pantalla principal obligatoria.

La vista puede incorporar datos de operación asociada, como identificador, tipo, notas o auditoría técnica, sin desplazar el foco del show principal del movimiento.

Aclaración adicional para line items:

orders y documents pueden compartir componentes visuales de líneas cuando la estructura de lectura sea equivalente.

La tabla, el formulario, el modal operativo y las tabs embebidas pueden reutilizar una gramática común sin borrar la identidad visual del host ni la semántica propia de cada módulo.

Aclaración adicional para tabs persistentes:

las tabs principales del host deben poder conservar el panel activo mediante return_tab cuando el host lo soporte.

La key recibida debe validarse contra las tabs disponibles del recurso actual.

No debe usarse la posición del loop como criterio de tab activa cuando existe estado explícito calculado por el host.

Aclaración adicional sobre iconografía en tabs host:

las tabs principales pueden incorporar iconografía para mejorar lectura visual sin modificar la lógica de activación ni la persistencia por return_tab.

La metadata visual de una tab debe llegar resuelta al componente mediante el objeto de tab correspondiente.

El host no debe reconstruir localmente qué ícono corresponde a una surface modular cuando esa metadata ya fue enriquecida por el flujo modular.

<<END SECTION>>

<<SECTION: REGLAS PARA X-SHOW-SUMMARY Y X-SHOW-SUMMARY-ITEM>>
REGLAS PARA X-SHOW-SUMMARY Y X-SHOW-SUMMARY-ITEM

x-show-summary es el contenedor visual base para el resumen principal de una vista show.

x-show-summary-item representa cada bloque interno de resumen.

Reglas:

x-show-summary-item debe seguir siendo la unidad visual reusable del summary principal

cuando el contenido del bloque requiera más ancho para análisis, métricas, visualización o composición interna más compleja, x-show-summary-item puede admitir span variable

ese span puede utilizarse para ocupar ancho doble o ancho completo dentro del summary, según la grilla disponible

no crear variantes visuales aisladas por módulo cuando el mismo resultado pueda resolverse extendiendo x-show-summary-item con span

el uso de span debe reservarse a bloques que realmente necesiten mayor ancho visual y no como excepción arbitraria de maquetado

si un bloque analítico requiere más espacio horizontal, debe priorizarse esta capacidad reusable antes que crear wrappers específicos del módulo

la capacidad de span en x-show-summary-item forma parte de la base visual reusable del sistema y debe validarse en uso real antes de extenderla a casos más complejos

Regla operativa:

si el bloque representa una relación hacia otro módulo:

no debe renderizarse si el módulo destino está deshabilitado

la unidad visual completa debe desaparecer y no solo el contenido interno
<<END SECTION>>

<<SECTION: ORGANIZACIÓN DE RELACIONES EN VISTAS SHOW>>
SECTION_VERSION: 00013

ORGANIZACIÓN DE RELACIONES EN VISTAS SHOW

Las relaciones dentro de una show page deben ordenarse por relevancia operativa y no por acumulación indiscriminada.

Criterios:

primero relaciones críticas para la tarea actual
luego relaciones frecuentes
luego bloques secundarios o informativos

Reglas:

usar tabs, cards o bloques según densidad real del contenido

evitar repetir la misma relación en múltiples lugares de la pantalla

si existe componente reusable oficial, priorizarlo

una relación contextual no debe convertirse visualmente en segundo módulo principal dentro de la misma pantalla

Aclaración consolidada para orders.show:

la capa estructural de ítems debe mantenerse separada de la operación material publicada por inventory

documents y attachments deben presentarse como capacidades embebidas de módulos oferentes y no como prolongaciones manuales del host

Aclaración consolidada para products.show:

los bloques de inventory deben actuar como acceso o resumen contextual y no como surface maestra paralela

Aclaración adicional para documents.show:

la capa estructural de líneas documentales debe mantenerse separada de la operación material publicada por inventory.

Las acciones por línea pueden nacer visualmente en la fila documental, pero no deben presentarse como lógica propia de documents si pertenecen a inventory.

Aclaración adicional para parties.show:

las relaciones embebidas o compactas de parties.show deben responder a relaciones reales publicadas por surfaces oficiales.

La lectura semántica debe considerar party_roles.role:

customer
supplier
employee
other

Regla:

parties.kind no debe utilizarse para decidir embebidos.

parties.kind solo expresa tipo de contacto:

person
company

Aclaración adicional para actividad operativa en parties.show:

si se incorpora una lectura de actividad para contactos con relación employee y membership activa vinculada, esa lectura debe consumir OperationalActivity persistido y autorizado.

No debe inferirse desde party_id.

No debe inferirse desde created_by, updated_by ni assigned_user_id actuales.

No debe agregarse como relación manual reconstruida en el controller del host.

La fila de actividad debe permitir lectura compacta de:

fecha
módulo
registro
tipo de actividad

El enlace al recurso debe estar integrado en Registro.

Aclaración adicional para relaciones compactas:

las relaciones puntuales, 1 a 1 o de baja densidad deben priorizar summary_items, details o acciones compactas antes que tabs pesadas.

Caso vigente:

assets, parties y orders hacia appointments.show deben visualizarse como relaciones compactas de summary y no como pestañas por defecto.

Aclaración adicional para embebidos operativos:

si una capacidad invitada ya incluye tabla, tabs internas o toolbar propia, el host no debe volver a encapsularla como si fuera una relación secundaria genérica.

Debe reconocerse su jerarquía como frente invitado del módulo oferente y no como card auxiliar duplicada.

Aclaración adicional para capas internas de inventory:

la agrupación inventory_operations puede aportar lectura contextual o auditoría técnica, pero no exige crear jerarquía visual nueva equivalente a módulo principal, tab pesada o pantalla autónoma si el flujo real del usuario continúa centrado en movimientos y acciones operativas.

Aclaración adicional sobre acciones por fila:

cuando una tabla anfitriona combine acciones propias y acciones invitadas, la lectura visual debe priorizar primero las acciones nativas del host y luego las acciones externas integradas.

Esto permite sostener una jerarquía clara entre núcleo del recurso actual y capacidades consumidas desde módulos oferentes.

SUPERFICIES CONDICIONALES POR MÓDULO

Toda superficie visual que dependa de otro módulo habilitado debe comportarse de forma consistente en todo el sistema.

Se considera superficie visual:

index
show
create
edit
partials
tabs
calendarios
formularios embebidos
tablas embebidas
componentes reutilizables
modales si existieran

Reglas:

no alcanza con ocultar la relación en la pantalla principal

la condición debe mantenerse en toda superficie derivada

si el módulo relacionado no está habilitado:

no deben verse accesos residuales
no deben quedar tabs vacías
no deben mostrarse filtros inútiles
no deben aparecer campos no operables
no deben persistir enlaces muertos

la UI debe recomponerse de forma limpia con lo disponible

Regla adicional consolidada:

si además existe una capacidad reusable oficial publicada por el módulo relacionado, la superficie derivada no debe sustituirla por una resolución local improvisada

<<SECTION: TABS INTERNAS DEL EMBEBIDO>>
SECTION_VERSION: 00005

TABS INTERNAS DEL EMBEBIDO

Las tabs internas de un bloque embebido deben usarse solo cuando exista una dimensión visible, estable y útil para segmentar el contenido.

Reglas:

no introducir tabs internas en embebidos por anticipación o por estética

si la segmentación visible responde a una dimensión clara del dominio o de la operación, las tabs internas son válidas

si no existe una dimensión clara, debe priorizarse listado único con filtros o lectura directa

las tabs internas del embebido no deben duplicar sin necesidad una segmentación ya resuelta por tabs del padre

cuando un embebido requiera tabs internas y además una acción principal contextual, debe priorizarse el patrón tab-toolbar reusable

el embebido sigue siendo un bloque subordinado al show principal, aunque internamente utilice tabs

Aclaración adicional para host principal:

las tabs internas de un embebido no deben confundirse con la zona tab estructural del host show.

La tab del host organiza relaciones de primer nivel de la pantalla.

Las tabs internas segmentan solo el contenido del bloque invitado.

Solo las tabs principales del host deben modificar return_tab.

Las tabs internas de un embebido no deben alterar el contexto host ni apropiarse de la persistencia principal del recurso mostrado.

Aclaración adicional de retorno visual:

si una acción se dispara desde una tab interna del embebido y luego redirige, la UX correcta debe restaurar la tab técnica correspondiente del host cuando ese panel siga siendo el punto real de continuidad del trabajo.

Aclaración adicional sobre x-tabs-embedded:

cuando un bloque embebido comparte gramática entre módulos, puede utilizar x-tabs-embedded como componente transversal.

Este componente puede mostrar métricas principales y métricas secundarias mediante subvalue, siempre que la segunda lectura sea contextual y de menor jerarquía visual.

<<END SECTION>>

<<SECTION: ACCIONES VISUALES EN INDEX Y SHOW>>

ACCIONES VISUALES EN INDEX Y SHOW

Las acciones visibles deben expresar intención clara y estado operable real.

Reglas:

no mostrar acciones imposibles
no prometer permisos inexistentes
no duplicar CTAs equivalentes
priorizar acciones principales sobre ruido secundario
preservar contexto de navegación cuando corresponda

Acciones frecuentes:

crear
editar
eliminar
volver
abrir relación vinculada
crear relacionado contextual
acciones rápidas por fila
micro-operaciones locales en modal

Regla adicional UX:

si el sistema puede resolver una intención mediante fallback sano, la UI no debe bloquear innecesariamente al usuario.

Caso consolidado:

en tasks, no seleccionar colaborador explícito puede resolverse asignando al usuario actual.

La interfaz debe acompañar ese comportamiento con copy claro y consistente.

Regla adicional para componentes linked:

la decisión de readonly, apertura o visibilidad debe venir resuelta desde el módulo oferente.

El componente visual solo representa el estado contractual recibido.

Regla adicional para show pages:

las acciones contextuales dentro de una show page deben aparecer donde nace la intención del usuario.

Regla adicional para ficha ampliada:

la acción Más datos debe leerse como acción contextual de apertura de información ampliada.

Si existe ficha vinculada, debe abrir lectura mediante parties.show.

No debe llevar directamente a edición.

Si no existe ficha vinculada, no debe crearla silenciosamente.

Debe existir confirmación explícita antes de crear o vincular la ficha ampliada.

Regla adicional para parties:

la acción visual o el campo que permite marcar relación Colaborador debe mostrarse solo cuando corresponda al estado autorizado o bloqueado del recurso.

Si la party está bloqueada por membership activa, la opción puede mostrarse como lectura protegida con ayuda contextual.

Si el usuario no está autorizado para asignar employee, la opción no debe presentarse como acción disponible.

Regla:

la ocultación o bloqueo visual no constituye autorización.

El backend debe seguir validando la operación.

<<END SECTION>>

<<SECTION: FRAGMENTACIÓN DE BLADES>>
SECTION_VERSION: 00002

FRAGMENTACIÓN DE BLADES

Las vistas grandes deben dividirse en partials cuando superen complejidad razonable.

Especialmente cuando mezclen:

filtros
tablas
resúmenes
formularios secundarios
bloques visuales repetidos

La fragmentación debe buscar:

legibilidad
mantenibilidad
reutilización

No debe fragmentarse artificialmente si el archivo sigue siendo claro y simple.

Criterio consolidado para formularios operativos:

las vistas create y edit deben actuar como contenedor completo del formulario dentro de la card o bloque principal de la pantalla

los parciales _form deben funcionar como bloque interno reusable centrado en los campos

cuando un _form agrupe múltiples campos mediante un wrapper raíz, ese wrapper puede y debe usar la clase .form para mantener el espaciado vertical uniforme del sistema, aunque no coincida con la etiqueta <form> principal

no debe fragmentarse un formulario de modo que se disperse entre partials la responsabilidad del <form>, @csrf, @method y form-actions

Regla validada en comportamiento real:

cuando un campo está bloqueado por reglas de negocio, no debe depender de old()

el valor visible debe reflejar el estado actual del modelo

no debe priorizarse el valor previo del request si ese campo no es editable

esto evita inconsistencias visuales en formularios como órdenes numeradas

Criterio adicional para formularios progresivos:

un _form puede organizar campos en secciones progresivas cuando eso mejore la carga guiada sin convertir el formulario en un wizard externo ni dispersar la responsabilidad del formulario principal.

El formulario progresivo debe:

mantener create y edit como contenedores del formulario completo
mantener _form como bloque interno de campos
mostrar las secciones necesarias según valores existentes en edición
no trasladar reglas de negocio ni autorización al frontend
acompañar la carga con copy claro y no técnico

Caso vigente:

parties._form guía la carga en tres momentos visuales:

¿Qué tipo de contacto vas a agregar?
¿Qué relación tiene con tu empresa?
Datos del contacto

Regla:

la progresión visual ayuda a reducir error de carga, pero no reemplaza validaciones backend ni autorización efectiva.

ORGANIZACIÓN DE CSS VISUAL

La organización del CSS visual del proyecto debe seguir esta estructura:

public/css/app-base.css
public/css/app-patterns.css

Criterios:

<<SECTION: BREADCRUMB ESTRUCTURAL Y CONTEXTUAL>>
BREADCRUMB ESTRUCTURAL Y CONTEXTUAL

x-breadcrumb representa la ruta visible de ubicación actual de la pantalla.

Debe poder resolverse en dos modos:

breadcrumb estructural
breadcrumb contextual

breadcrumb estructural:
expresa la ubicación normal y estable de la pantalla dentro del módulo o del sistema

breadcrumb contextual:
expresa el retorno al contexto de origen cuando el usuario llegó desde una entidad relacionada y ese origen sigue siendo relevante para la tarea actual

Reglas:

x-breadcrumb es un único componente visual del sistema

no deben coexistir dos breadcrumb distintos en una misma pantalla

el breadcrumb estructural debe actuar como base por defecto

el breadcrumb contextual solo debe utilizarse cuando exista un flujo relacionado real, válido y útil para la continuidad de uso

no usar breadcrumb contextual si no existe origen funcional claro

no reconstruir jerarquías artificiales ni cadenas excesivas solo porque exista una relación entre entidades

el breadcrumb contextual debe mejorar orientación y retorno, no agregar ruido visual

cuando el usuario acceda a una pantalla desde navegación propia del módulo, búsqueda, acceso directo o index del mismo módulo, debe priorizarse breadcrumb estructural

cuando exista navegación cruzada entre entidades relacionadas, puede priorizarse una salida visual de retorno al origen siempre que esa salida mejore realmente la UX

breadcrumb no debe depender del historial del navegador para representar relaciones de negocio

cuando exista navegación cruzada entre pantallas relacionadas, el breadcrumb puede resolverse mediante un trail heredable previamente validado por controller

cuando una pantalla use breadcrumb contextual mediante trail heredable, el retorno visible no debe duplicar nodos ya recorridos

la salida visual debe mantenerse compacta y limpia aunque el usuario navegue entre entidades relacionadas en ambos sentidos
<<END SECTION>>

PANTALLAS TIPO CALENDARIO

Cuando un módulo necesite lectura temporal, agenda visual o navegación por fechas, se admite una pantalla tipo calendario dentro de la misma base visual del sistema.

Estructura recomendada:

x-page

x-page-header

Estas pantallas pueden fragmentarse en partials específicas del módulo, por ejemplo:

partials/calendar-toolbar.blade.php
partials/calendar-grid.blade.php
partials/calendar-day-cell.blade.php

Cuando la UX lo requiera, el calendario puede utilizar:

cabecera sticky interna
scroll interno del bloque
columna auxiliar de contexto temporal, como número de semana

No corresponde documentar aquí detalles finos de CSS o ajuste visual puntual de implementación.

TABS INTERNAS EN VISTAS SHOW

Las tabs internas ordenan relaciones y capacidades secundarias dentro de una show page sin desplazar la identidad principal del recurso.

Reglas vigentes:

la entidad principal de la pantalla conserva prioridad visual

las tabs deben agrupar información relacionada y no fragmentar innecesariamente la experiencia

una capacidad nueva no obliga automáticamente a crear una nueva pantalla si puede resolverse dentro de una tab o detalle existente

los hosts visuales pueden consumir capacidades de otros módulos sin absorber su ownership funcional

Caso vigente:

inventory incorpora una capa interna de operaciones contextuales sin requerir una nueva navegación principal obligatoria.

El detalle de movement continúa como lectura primaria, enriquecido con contexto adicional cuando exista.

<<SECTION: TABS REUTILIZABLES Y TOOLBAR DE TABS>>
SECTION_VERSION: 00003

TABS REUTILIZABLES Y TOOLBAR DE TABS

El proyecto puede reutilizar la misma base visual de tabs en distintos niveles de la interfaz.

Reglas:

no crear variantes nuevas de tabs si el patrón base existente ya resuelve correctamente navegación y estado activo

cuando un bloque de tabs necesite compartir fila con acciones contextuales, debe resolverse mediante un wrapper reusable de toolbar de tabs

ese toolbar debe poder usarse:

con acciones a la derecha
sin acciones, cuando solo se necesite mantener el mismo contenedor visual

el toolbar de tabs debe vivir dentro de un card o contenedor visual compartido para evitar que las acciones queden flotando respecto del grupo de tabs

si el card del toolbar requiere una densidad visual más compacta, el ajuste debe resolverse mediante una clase específica del wrapper y no modificando el card global del sistema

las acciones contextuales deben percibirse como propias del panel activo y no como CTA principal de toda la pantalla

Regla modular adicional:

cuando un módulo publique una capacidad reusable de integración que se consuma mediante tabs, toolbar o bloque embebido, esa capacidad debe reutilizar la base visual existente y no crear una familia nueva de tabs o toolbar específica del consumidor
<<END SECTION>>

<<SECTION: TABS REUTILIZABLES, TOOLBAR Y OVERFLOW HORIZONTAL>>
TABS REUTILIZABLES, TOOLBAR Y OVERFLOW HORIZONTAL

Los tabs reutilizables del proyecto deben sostener una misma base visual y de comportamiento, tanto en tabs de primer nivel como en tabs internos o embebidos.

Criterios consolidados:

el mismo wrapper visual de tab-toolbar puede utilizarse:
con acciones contextuales a la derecha
sin acciones, cuando solo se necesite el grupo de tabs dentro del mismo contenedor visual

no conviene crear familias distintas de tabs superiores e inferiores si la misma base ya resuelve correctamente el patrón visual y el comportamiento

cuando el conjunto de tabs pueda desbordar horizontalmente, el overflow debe resolverse mediante un wrapper reusable de scroll horizontal y no con soluciones ad hoc por módulo

el wrapper reusable de scroll horizontal debe:
ocultar el scroll nativo visible
permitir desplazamiento horizontal del contenido
mostrar affordance clara mediante chevrons laterales cuando exista overflow real
recalcular visibilidad de chevrons según ancho real disponible del viewport y del track

las acciones contextuales deben quedar fuera del área desplazable de tabs y mantener ancho propio, mientras el bloque de tabs ocupa solo el espacio restante del contenedor

el efecto visual del viewport horizontal puede reforzarse con una sombra interna muy sutil, sin alterar la legibilidad ni competir con el tab activo

no deben mezclarse dentro del área scrollable:
botones de acción
controles contextuales del panel
navegación horizontal de tabs

las acciones contextuales de tabs deben permanecer fijas a la derecha y los tabs desplazarse únicamente dentro de su viewport
<<END SECTION>>

EMBEDDED TABS EN SHOW RELACIONADOS

Los bloques embebidos dentro de show relacionados deben tender a una misma estructura:

tab-toolbar reutilizable
tabs internos con posibilidad de overflow horizontal reusable
acción contextual fija a la derecha
tabla o listado debajo en card independiente

Cuando un embedded-tabs tenga riesgo razonable de desborde en pantallas chicas o por crecimiento futuro de etiquetas, conviene aplicar desde el inicio el wrapper reusable de scroll horizontal.

No esperar a que el layout rompa para recién introducir un patrón especial si ya se sabe que el bloque puede crecer.

Attachments se considera el caso estándar actual de embebido tabulado por kind dentro de una vista show relacionada.

Regla adicional:

si otros módulos publican tablas embebidas, bloques summary o acciones contextuales reutilizables, deben priorizar esta misma familia visual antes de introducir layouts paralelos incompatibles
