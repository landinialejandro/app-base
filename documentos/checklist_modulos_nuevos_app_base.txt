<<SECTION: OBJETIVO>>
SECTION_VERSION: 00004

OBJETIVO

Este documento define el checklist obligatorio para crear, incorporar o reconvertir módulos dentro de app-base.

Su función es:

reducir omisiones
forzar consistencia arquitectónica
alinear seguridad, dominio y UI
evitar módulos incompletos
mantener coherencia entre implementaciones nuevas y existentes

No es una guía opcional.

Es un control mínimo obligatorio antes de considerar un módulo, reconversión profunda, capacidad transversal o capa satélite como integrada al sistema.

Aclaración doctrinal consolidada:

este checklist aplica tanto a módulos nuevos como a reconversiones profundas de módulos existentes cuando cambie su contrato funcional, contractual, de modelado o de seguridad.

También aplica, con la adaptación correspondiente, a:

capacidades transversales
capacidades tenant no modulares
surfaces tenant/transversales no modulares
piezas satélite productivas
frentes operativos especializados
automatizaciones relevantes
integraciones futuras con IA

Regla:

si una capacidad no es módulo ordinario, el checklist no debe forzarla a convertirse en módulo.

Debe ayudar a clasificarla correctamente antes de implementar.

<<SECTION: REGLA GENERAL>>
SECTION_VERSION: 00011

REGLA GENERAL

Ningún módulo debe considerarse terminado por tener tablas, rutas o CRUD funcionando.

Un módulo solo puede considerarse integrado cuando supera este checklist en forma coherente.

Orden correcto:

definir contrato conceptual
definir contrato de seguridad
definir relaciones reales necesarias
definir superficie de integración
definir si participa en trazabilidad operativa transversal
definir si consume lectura contextual de actividad operativa, directa, agregada o contextual especial cuando corresponda
definir si la necesidad corresponde a módulo funcional ordinario, capacidad transversal, pieza satélite, entidad hija, subcapa interna de módulo existente, surface tenant/transversal o frente operativo especializado
implementar
auditar
ajustar documentación afectada

Nunca al revés.

Regla adicional consolidada:

si una integración nueva requiere reabrir arbitrariamente el núcleo de otro módulo, primero debe revisarse si la necesidad puede resolverse consumiendo una capacidad ya publicada por ese módulo o evolucionando su superficie contractual propia.

Regla adicional para reconversiones profundas:

este checklist también aplica cuando no se crea un módulo nuevo, pero se reconvierte un frente funcional, visual, contractual o de seguridad ya existente.

Caso vigente:

tenant/profile/accesses fue reconvertido en relación con owner/admin y edición de funciones de memberships.

Regla:

una reconversión de acceso o gobierno interno debe validarse con el mismo rigor que un módulo sensible.

Debe incluir UI correcta, backend centralizado, requests manipulados, persistencia final y rollback o limpieza de pruebas.

Regla adicional para subcapas internas de módulos existentes:

si una capacidad contextual pertenece a un módulo host, no posee CRUD, no tiene navegación propia, no requiere ModuleCatalog propio, no entra al dashboard general y no posee rutas propias, debe evaluarse como subcapa interna del módulo antes de tratarla como módulo nuevo.

Caso vigente:

ProductOperationalSummary se define como subcapa contextual de Products.

No debe tratarse como módulo nuevo.

No posee CRUD.

No posee navegación propia.

No posee ModuleCatalog propio.

No entra al dashboard general.

No posee rutas propias.

Se publica mediante ProductSurfaceService como summary_items dentro de products.show.

Regla:

una subcapa interna puede tener contrato propio de acceso y visibilidad sin convertirse en módulo funcional ordinario.

Debe quedar claro:

qué módulo host la aloja
qué módulos aporta datos
qué módulo conserva ownership de cada dato
qué condiciones de visibilidad aplican
qué surfaces la publican
qué vistas representan el payload
qué queda fuera de su alcance inicial

Regla adicional para capas satélite productivas:

si una necesidad organiza un frente operativo amplio sobre módulos existentes, no debe asumirse que corresponde crear un módulo CRUD ordinario.

Debe evaluarse si corresponde:

módulo funcional
pieza satélite
capacidad transversal
surface tenant/transversal
dashboard operativo especializado
experiencia enfocada
lectura consolidada
contrato nuevo del módulo dueño
subcapa contextual interna de un módulo existente

Caso vigente:

Servicios y Mantenimiento debe leerse como capa satélite productiva ERP/CMMS liviana y no como CRUD aislado.

Reglas:

no duplicar ownership de módulos existentes

no absorber Orders, Assets, Inventory, Products, Appointments, Tasks, Documents, Attachments, Parties ni OperationalActivity

si una necesidad es propia de mantenimiento pero no corresponde al núcleo de un módulo existente, debe evaluarse como pieza satélite antes de reabrir módulos cerrados

si una necesidad puede resolverse mediante lectura consolidada, surfaces, contratos existentes o capacidades oficiales, no debe abrirse una pieza nueva por reflejo automático

Regla adicional para clasificaciones:

si una clasificación mezcla dimensiones conceptuales distintas, debe frenarse la implementación hasta revisar el contrato funcional.

No deben mezclarse en un mismo campo:

propiedad o relación del objeto
ubicación de ejecución
estado operativo
lifecycle
custodia material
consumo material
prioridad
tipo técnico
origen de necesidad
cierre técnico
cierre administrativo
cierre final

Regla adicional para actividad operativa:

no todo módulo debe registrar actividad por defecto.

La participación en OperationalActivity debe justificarse por valor operativo real.

No debe registrarse actividad solo por existir CRUD, eventos técnicos o cambios persistidos sin significancia operativa.

La integración con OperationalActivity debe definirse como parte del contrato del módulo dueño.

Debe quedar claro:

qué operaciones emiten eventos
qué campos son observables
qué campos quedan excluidos por ruido técnico
qué cambios tienen valor operativo real
si existen relaciones hijas que deban expresarse como metadata operativa
si existe entidad agrupadora que deba actuar como ancla operativa
si el módulo consume actividad operativa contextual
si la lectura contextual será directa, agregada o contextual especial

Casos validados:

parties puede registrar cambios de party_roles como metadata operativa de Party.

inventory debe registrar InventoryOperation como ancla operativa agrupada, no InventoryMovement como actividad principal.

projects puede declarar un activity_record_set para lectura agregada de actividad contextual de Project + Tasks.

parties puede declarar un activity_context para lectura contextual especial de Party employee con membership activa vinculada.

Regla adicional para lectura contextual de actividad operativa:

la lectura directa no requiere provider adicional.

Se limita al record host autorizado.

La lectura agregada requiere que el módulo dueño declare explícitamente su frontera mediante un provider compatible con ActivityRecordSetProvider.

La lectura contextual especial requiere que el módulo dueño declare explícitamente su criterio mediante un provider compatible con ActivityContextProvider.

Antes de declarar un activity_record_set debe quedar claro:

qué records forman parte de la actividad contextual del host
por qué esos records pertenecen funcionalmente al host
si esa agregación altera o no ownership
si el host ya autorizó el record principal
si la lectura agregada no equivale a lectura general del tenant
si no se está reconstruyendo historia desde campos actuales en lugar de leer OperationalActivity persistido

Antes de declarar un activity_context debe quedar claro:

qué criterio amplía el dataset
qué módulo dueño define la frontera
qué condición habilita esa lectura especial
si respeta tenant activo
si no habilita lectura general del tenant
si no reemplaza permisos del host
si no contamina OperationalActivity con lógica del módulo

Regla:

OperationalActivity no debe modificarse con condiciones específicas por módulo.

No corresponde agregar lógica como:

if Project
if Order
if Document
if Party employee

La frontera debe vivir en el módulo dueño o en su contrato publicado.

Caso validado de activity_record_set:

projects.show
ProjectActivityRecordSet
Project + Tasks

Caso validado de activity_context:

parties.show
PartyEmployeeActivityContext
Party employee con membership activa vinculada
actividad directa de Party
actividad donde el user vinculado sea actor o sujeto

Regla adicional para capacidades tenant no modulares:

no toda capacidad tenant-scoped debe incorporarse a ModuleCatalog.

Si una capacidad:

no es módulo funcional
no tiene navegación propia
no tiene CRUD completo
no se habilita ni deshabilita como módulo ordinario

debe evaluarse como capacidad tenant no modular.

Caso validado:

operational_activity.view_any

Regla:

las capacidades tenant no modulares pueden ser evaluadas por Security sin convertirse en módulos funcionales ordinarios.

Regla adicional para surfaces tenant/transversales no modulares:

una capacidad tenant-scoped o transversal puede publicar una surface hacia hosts funcionales sin convertirse en módulo funcional.

Esta integración no representa un módulo nuevo si la capacidad:

no posee navegación principal
no posee CRUD ordinario
no se incorpora a ModuleCatalog
no pasa por TenantModuleAccess
no se habilita ni deshabilita como módulo ordinario

Caso validado:

OperationalActivitySurfaceService publica una tab contextual Actividad para hosts autorizados mediante TenantSurfaceRegistry.

Antes de crear una surface para una capacidad tenant/transversal se debe verificar:

si la capacidad es módulo funcional o no
si corresponde ModuleCatalog o TenantSurfaceRegistry
si necesita navegación propia o solo consumo embebido
si la autorización proviene de permiso propio o del host
si la surface representa lectura general o contextual
si el host ya autorizó el registro antes de consumir la surface

Regla:

no debe quedar como cierre estable un push manual de tabs en vistas host si la capacidad ya puede publicarse mediante registry reusable.

Regla adicional para surfaces modulares:

una capacidad interna de un módulo host puede publicarse mediante el surface service del módulo cuando deba integrarse de forma natural al show del host.

Caso vigente:

ProductOperationalSummary se publica mediante ProductSurfaceService.

Publica tres summary_items:

Compras
Ventas
Entradas / salidas

Regla:

si la infraestructura surface-driven vigente puede publicar la capacidad, no debe dejarse como cierre estable un include directo en el Blade del host.

Regla adicional para IA:

si una capacidad produce información relevante para IA futura, debe exponerla mediante contratos claros, datos estructurados, catálogos, surfaces o lecturas autorizadas.

No debe depender de scraping visual, labels dispersos, texto libre inevitable ni inferencia frágil.

La IA no debe ejecutar acciones críticas sin autorización, confirmación humana o regla explícita previamente aprobada.

Regla adicional para estado cerrado:

una orden cerrada no debe quedar operable por interfaces satélite, dashboards, automatizaciones ni IA.

Toda reapertura debe respetar autorización vigente.

<<END SECTION>>

Validar:

nombre del módulo o capacidad
problema real que resuelve
naturaleza principal
posición conceptual en el ecosistema

si es módulo funcional, entidad hija, capacidad transversal de soporte, capacidad tenant no modular, pieza interna del tenant, pieza satélite productiva, subcapa interna de módulo existente, frente operativo especializado o surface tenant/transversal no modular

Clasificación conceptual posible:

módulo base del dominio
módulo transversal
módulo autónomo
módulo agrupador
módulo derivado
capacidad transversal de soporte
capacidad tenant no modular
capacidad tenant/transversal no modular con surface publicada
pieza interna del tenant
pieza satélite productiva
subcapa interna de módulo existente
frente operativo especializado
experiencia enfocada de ejecución
lectura consolidada

Preguntas obligatorias:

este módulo tiene identidad propia o es solo una entidad hija
es realmente un módulo o una evolución de otro ya existente
debe vivir por sí mismo o principalmente embebido
qué rol ocupa dentro del ecosistema general
registra, acompaña o sostiene actividad de otros módulos sin absorber su ownership
debe tener navegación principal o solo consumo interno/transversal
la necesidad requiere ModuleCatalog o puede resolverse como capacidad tenant no modular
si necesita publicar UI reusable, corresponde ModuleCatalog, TenantSurfaceRegistry o surface service del módulo host
la surface representa un módulo funcional o una capacidad tenant/transversal no modular
la necesidad corresponde a una capa satélite productiva o a un módulo CRUD ordinario
la necesidad corresponde a una subcapa contextual interna de un módulo existente
si pertenece a una subcapa interna, qué módulo host la aloja
si pertenece a una capa satélite, qué módulo conserva ownership funcional de cada parte
si la necesidad puede resolverse con contrato actual, lectura consolidada o surface antes de crear tabla, campo o módulo nuevo

Si estas respuestas no están claras, no debe avanzarse a implementación.

Caso vigente:

OperationalActivity se lee como capacidad transversal de soporte tenant-scoped para trazabilidad operativa.

No se lee como CRUD ordinario ni como módulo navegable principal.

Caso vigente adicional:

operational_activity.view_any se lee como capacidad tenant no modular para lectura general tenant-scoped del log operativo.

No se incorpora a ModuleCatalog.

Caso vigente adicional:

OperationalActivitySurfaceService se lee como surface tenant/transversal no modular publicada mediante TenantSurfaceRegistry.

No convierte OperationalActivity en módulo funcional ordinario.

Caso doctrinal vigente:

Servicios y Mantenimiento se lee como capa satélite productiva ERP/CMMS liviana sobre módulos existentes.

No se lee como módulo CRUD ordinario por defecto.

Puede tener dashboard operativo, orden electrónica liviana, paquete de trabajo o piezas satélite específicas sin duplicar Orders ni absorber ownership de módulos existentes.

Caso vigente de subcapa interna:

ProductOperationalSummary se lee como subcapa contextual de Products.

No es módulo nuevo.

No tiene CRUD.

No tiene navegación propia.

No tiene ModuleCatalog propio.

No entra al dashboard general.

No tiene rutas propias.

Se monta mediante ProductSurfaceService como summary_items dentro de products.show.

<<SECTION: BLOQUE 1 — CONTRATO MODULAR>>

BLOQUE 1 — CONTRATO MODULAR

Antes de aprobar un módulo nuevo, reconversión profunda, capacidad transversal, pieza satélite o subcapa interna debe poder explicarse con claridad:

núcleo propio
contrato de entrada
contrato de salida
contrato de seguridad
oferta de integración
participación o no en trazabilidad operativa transversal
participación o no en lectura contextual de actividad operativa
relación con módulos dueños existentes
límite de ownership funcional
si requiere datos estructurados para reportes, trazabilidad, automatización o IA futura

Validar además:

si realmente es módulo y no dataset reusable del tenant
si resuelve un problema propio del dominio
si evita invadir núcleos existentes
si sus integraciones futuras pueden publicarse por contrato
si el ownership funcional queda claro
si debe emitir eventos operativos livianos después de acciones propias autorizadas
si debe ser consumido por una capacidad transversal sin trasladarle lógica del dominio
si necesita publicar una frontera explícita para lectura contextual agregada de actividad
si necesita publicar un criterio contextual especial para lectura contextual de actividad
si corresponde una pieza satélite en lugar de modificar el núcleo de un módulo existente
si corresponde una subcapa interna en lugar de crear un módulo nuevo
si corresponde una lectura consolidada antes que una tabla nueva
si corresponde un dashboard o experiencia especializada sin CRUD ordinario

Criterio adicional validado:

si una necesidad es solo consulta interna de colaboradores, directorio activo o pertenencia operativa, evaluar primero una raíz reusable del plano tenant antes de abrir módulo nuevo.

Criterio adicional validado:

si la necesidad aparece dentro de tablas anfitrionas de otros módulos, evaluar si corresponde publicar row actions oficiales desde el módulo oferente antes de crear lógica local en el consumidor.

Criterio adicional para subcapas internas:

si la capacidad contextual pertenece al módulo host y no posee identidad suficiente para módulo nuevo, puede organizarse como subcapa interna con carpeta propia y servicios propios.

Debe definirse:

módulo host
servicio de acceso
servicio lector
surface de publicación
vistas internas
módulos que aportan datos
condiciones de visibilidad
exclusiones de alcance
ausencia de rutas propias
ausencia de navegación propia
ausencia de CRUD

Caso vigente:

ProductOperationalSummary

Módulo host:

Products

Ubicación vigente:

app/Support/Products/OperationalSummary/
resources/views/products/operational-summary/

Surface de publicación:

ProductSurfaceService

Salida visual:

summary_items en products.show

Ítems vigentes:

Compras
Ventas
Entradas / salidas

Módulos que aportan datos:

Orders para compras y ventas reales cuando corresponde

Inventory para entradas, salidas y movimientos

Parties para proveedor o cliente cuando existe relación real

OperationalActivity no se usa como fuente principal de métricas

Regla:

una subcapa interna no debe calcular datos en Blade.

El Blade debe representar payload ya preparado por servicios.

Regla adicional:

si la subcapa puede publicarse mediante surface del módulo host, no debe quedar como include directo en el show como cierre estable.

Criterio adicional para reconversiones:

si una clasificación mezcla dimensiones conceptuales distintas, debe revisarse el contrato funcional del módulo antes de seguir agregando excepciones.

La salida correcta puede requerir separar:

núcleo de la entidad
relaciones operativas
entidades hijas o complementarias
dimensiones satélite
catálogos mínimos
labels visibles
ubicaciones configurables

Caso validado:

parties.kind no debe mezclar tipo de contacto con relación comercial o vínculo interno.

Criterio adicional para constraints funcionales:

si una reconversión cambia la dimensión real de autorización, la constraint funcional debe migrar hacia esa dimensión.

No debe conservarse la constraint anterior como compatibilidad operativa si ya no representa el dominio vigente.

Caso validado:

parties dejó de usar allowed_kinds como constraint funcional.

parties usa allowed_party_roles sobre party_roles.role.

Criterio adicional para capas satélite productivas:

si una necesidad pertenece a una capa satélite, debe quedar definido:

qué módulo conserva ownership funcional
qué módulo actúa como host visual si corresponde
qué datos se leen desde relaciones existentes
qué pieza satélite se justifica realmente
qué no debe duplicarse
qué no debe resolverse con nota libre si luego se espera automatización
qué parte corresponde a dashboard, experiencia enfocada o lectura consolidada
qué parte requiere seguridad específica
qué parte requiere trazabilidad operativa
qué parte requiere supervisión humana

Caso doctrinal vigente:

Servicios y Mantenimiento no debe resolverse solo modificando Orders.

Debe auditarse qué puede sostenerse con Orders, Assets, Inventory, Products, Appointments, Tasks, Documents, Attachments, Parties y OperationalActivity antes de crear piezas nuevas.

Criterio adicional para Orders dentro de mantenimiento:

Orders conserva lifecycle base:

draft
pending_approval
approved
closed
cancelled

No debe usarse Orders.status para estados técnicos de ejecución.

El estado operativo de ejecución debe vivir en dimensión satélite.

Una orden closed no debe quedar operable por interfaces satélite, automatizaciones ni IA sin reapertura autorizada.

Criterio adicional para costos operativos:

fletes, mano de obra, kilómetros, viajes, prorrateos, costos accesorios, costos indirectos y rentabilidad contable completa no deben incorporarse automáticamente dentro de una lectura contextual inicial de producto.

Pueden requerir:

proveedor distinto
orden distinta
criterio de imputación distinto
prorrateo
tratamiento como servicio
tratamiento como costo directo operativo
formalización documental específica

Regla:

si estos conceptos se vuelven necesarios, deben auditarse como línea futura separada y no mezclarse por reflejo con ProductOperationalSummary.

Criterio adicional para OperationalActivity:

si un módulo participa en actividad operativa transversal, debe quedar definido:

qué eventos operativos livianos emite
en qué momento los emite
qué campos son observables
qué campos deben excluirse por ruido técnico
qué cambios tienen significancia operativa real
qué catálogo del módulo dueño declara esos campos
qué datos mínimos necesita el registro transversal
qué no debe delegarse al prestador transversal
si la operación debe registrar el modelo principal o una entidad agrupadora
si existen relaciones hijas relevantes que deban representarse como metadata operativa

Reglas:

el prestador transversal no debe conocer en exceso la semántica interna del módulo dueño

el módulo dueño no debe llamar directamente al logger desde controllers o services

el módulo dueño debe autorizar, validar y mutar antes de emitir el evento operativo

no se debe registrar actividad técnica sin valor operativo

no se debe duplicar trazabilidad propia de otro módulo

Casos validados:

parties registra cambios de party_roles como metadata operativa de Party.

inventory registra InventoryOperation como acción operativa agrupada.

InventoryMovement conserva trazabilidad fina interna de stock.

Criterio adicional para lectura contextual agregada de OperationalActivity:

si un módulo necesita que la tab Actividad muestre más que el record host directo, debe publicar un activity_record_set propio.

Ese activity_record_set debe ser definido por el módulo dueño.

No debe ser definido dentro de OperationalActivity.

Validar:

si la lectura será directa o agregada
qué records integran el conjunto
si esos records pertenecen funcionalmente al host
si el conjunto respeta tenant activo
si el conjunto no amplía acceso general al log
si el conjunto no altera ownership de los records incluidos
si el host principal ya fue autorizado antes de leer actividad
si corresponde declarar provider compatible con ActivityRecordSetProvider
si corresponde registrar el provider en ModuleCatalog

Caso validado:

Projects declara ProjectActivityRecordSet.

ProjectActivityRecordSet define Project + Tasks como dataset de actividad visible del proyecto.

Regla:

si no existe activity_record_set, la lectura contextual debe permanecer directa sobre el record host.

Criterio adicional para lectura contextual especial de OperationalActivity:

si un módulo necesita que la tab Actividad muestre actividad por un criterio especial que no se expresa como simple conjunto de records, debe publicar un activity_context propio.

Ese activity_context debe ser definido por el módulo dueño.

No debe ser definido dentro de OperationalActivity.

Validar:

qué criterio amplía el dataset
qué condición habilita la lectura especial
si la condición pertenece realmente al módulo dueño
si el criterio respeta tenant activo
si el criterio no amplía acceso general al log
si el criterio no reemplaza permisos del host
si el host principal ya fue autorizado antes de leer actividad
si corresponde declarar provider compatible con ActivityContextProvider
si corresponde registrar el provider en ModuleCatalog
si la lectura no contamina OperationalActivity con lógica interna del módulo

Caso validado:

Parties declara PartyEmployeeActivityContext.

PartyEmployeeActivityContext define la lectura de actividad contextual para Party employee con membership activa vinculada.

Incluye actividad directa de Party y actividad donde el user vinculado sea actor o sujeto.

Regla:

activity_context no reemplaza activity_record_set.

activity_record_set corresponde a conjuntos explícitos de records.

activity_context corresponde a criterios contextuales especiales definidos por el módulo dueño.

Criterio adicional para capacidades tenant no modulares:

si la capacidad pertenece al plano tenant o transversal, pero no constituye módulo funcional ordinario, debe definirse fuera de ModuleCatalog.

Validar:

si no tiene navegación propia
si no tiene CRUD completo
si no se habilita o deshabilita como módulo
si no debe generar create, update ni delete
si puede ser evaluada por Security como capability específica

Caso validado:

operational_activity.view_any

Criterio adicional para surfaces tenant/transversales no modulares:

si una capacidad tenant o transversal no modular necesita publicar UI reusable hacia hosts funcionales, debe definirse una vía de publicación distinta de ModuleCatalog.

Validar:

si la capacidad conserva naturaleza no modular
si la surface representa lectura general o lectura contextual
si el host consumidor ya autorizó el registro cuando la lectura sea contextual
si corresponde TenantSurfaceRegistry
si ModuleSurfaceRegistry puede fusionar esa surface con las modulares sin alterar ownership
si la key de la tab es compatible con return_tab
si se evita duplicación por pushes manuales en cada host

Caso validado:

OperationalActivitySurfaceService publica tab Actividad mediante TenantSurfaceRegistry.

Regla:

la surface tenant/transversal no convierte la capacidad en módulo funcional ordinario.

Criterio adicional para IA:

si una capacidad será asistida por IA o producirá información para IA futura, debe quedar definido:

si el módulo publica una relación navegable reusable, no alcanza con implementarla en su show principal

también deben contemplarse tablas, sublistados y superficies secundarias consumidoras relevantes

Validación adicional de versionado visual Blade:

todo Blade nuevo o modificado debe revisar si corresponde incorporar o actualizar:

x-dev-component-version

Aplicar cuando:

la vista tenga entidad visual propia

el partial represente una card, bloque, tab, tabla, summary, toolbar o componente visible estable

la modificación altere estructura visual, composición, jerarquía, integración de componentes o comportamiento visible

el Blade ya tenga x-dev-component-version y se realice una modificación relevante

No aplicar cuando:

el fragmento sea puramente interno y no tenga entidad visual propia

la ubicación natural caiga dentro de tablas, tbody, formularios, loops o estructuras repetitivas donde pueda romper HTML o generar ruido visual

el cambio sea mínimo y sin impacto real en la vista

Reglas:

si se incorpora x-dev-component-version, usar props vigentes:

name
version
align

no usar prop file

si ya existe, revisar e incrementar version cuando corresponda

si no se incorpora, debe ser porque no corresponde por estructura, ubicación o entidad visual

<<SECTION: CHECKLIST DE INTEGRACIÓN>>

CHECKLIST DE INTEGRACIÓN

Cuando un módulo se relacione con otros, verificar:

consumo de contratos existentes antes de crear variantes nuevas

uso de capacidades reusables oficiales del módulo oferente

trail y retorno consistentes

flags modulares correctamente propagados

homogeneidad entre show pages, tablas y partials secundarias

diferenciación explícita entre integración pesada embebida e integración compacta

alineación con el patrón visual ya consolidado en el módulo consumidor cuando exista

si el módulo reemplaza surfaces históricas dispersas, plan de migración claro hacia la nueva surface oficial

si la relación reusable se apoya en registry o mecanismo equivalente, el host consumidor no recompone localmente payload ni naming ad hoc

compatibilidad real entre target, type, slot y gramática efectiva del host consumidor

Regla de cierre:

no considerar terminada una integración por haber corregido solo la pantalla principal del consumidor.

Debe revisarse también el ecosistema visible que consume esa relación.

Regla adicional para show pages surface-driven:

si el host monta surfaces por zonas explícitas, el checklist debe verificar que cada offer publique slot correcto y que el host tenga montaje compatible con esa combinación.

Una offer sin slot o con slot incoherente puede quedar invisible aunque el resto del contrato parezca correcto.

Regla adicional para usuarios internos del tenant:

si una integración necesita usuarios internos activos, validar explícitamente que el dataset provenga del plano tenant reusable correspondiente y no de un módulo inexistente ni de una query duplicada reconstruida por cada consumidor.

VALIDACIONES BACKEND VS UI

<<SECTION: ENTIDADES HIJAS Y SUBRECURSOS>>

ENTIDADES HIJAS Y SUBRECURSOS

Verificar:

si el recurso hijo tiene autonomía real o depende funcionalmente del padre

si corresponde delegar autorización en el recurso padre

si evita crear un módulo artificial donde alcanza con una entidad hija bien modelada

si la UI del hijo no compite con la jerarquía del padre sin necesidad

si las restricciones reales del hijo provienen del padre o de un frente operativo relacionado ya existente

Preguntas obligatorias:

este recurso necesita identidad modular propia o sigue siendo detalle del padre
su seguridad puede delegarse completamente
su navegación debe ser estructural o contextual
su edición libre sigue siendo válida cuando ya existe trazabilidad operativa iniciada
si gana relevancia operativa, esa relevancia exige módulo nuevo o solo endurecer su contrato como hijo delegado

Regla adicional consolidada:

un subrecurso delegado puede ganar relevancia operativa transversal sin dejar de ser hijo

eso ocurre cuando pasa a actuar como ancla explícita de ejecución, trazabilidad o restricciones dentro de otro frente operativo

caso vigente relevante:

order_items respecto de inventory

En esos casos, validar además:

que no se le inventen permisos autónomos innecesarios
que la trazabilidad no rompa la delegación doctrinal al padre
que la UI exprese su nuevo peso operativo sin promoverlo falsamente a módulo
que edición y borrado dejen de ser libres cuando ya existan movimientos ejecutados o equivalentes

<<SECTION: NAVEGACIÓN Y BREADCRUMB>>
SECTION_VERSION: 00003

NAVEGACIÓN Y BREADCRUMB

Definir desde el inicio si el módulo necesita solo navegación estructural o también navegación contextual.

Preguntas de control:

la entidad puede abrirse desde otra entidad relacionada como continuación natural de trabajo

el usuario necesita volver al origen sin perder el hilo operativo

ese origen es semántico y estable, no solo la página anterior

Si alguna de estas respuestas es sí, el módulo debe evaluar e implementar navegación contextual explícita.

Si corresponde, el módulo debe:

tener builder o resolutor de trail propio compatible con el mecanismo central vigente del proyecto

aceptar contexto de navegación de forma explícita

resolver ese contexto desde la capa reutilizable central vigente

propagar ese contexto en tablas, tabs, acciones por fila, links relacionales, paginación, filtros, botones volver, botones cancelar y redirects relevantes

resolver create -> store -> show sin persistencia de nodos transitorios de “Nuevo …”

Validar además:

cuál es la superficie principal real del módulo

si existe una superficie secundaria que permanece accesible pero no debe actuar como base promovida del trail

si calendarios, tabs o embebidos contextuales necesitan actuar como entrada principal efectiva del módulo

Regla adicional consolidada:

cuando una capacidad reusable oficial del módulo oferente se consuma dentro de otro módulo, también debe propagar el trail válido del contenedor y no reiniciar navegación desde una lectura estructural local del oferente

<<END SECTION>>

<<SECTION: CIERRE>>
SECTION_VERSION: 00003

CIERRE

Un módulo no debe considerarse terminado solo porque funciona.

Debe quedar:

consistente con el sistema
alineado con permisos reales
alineado con multi-tenancy
alineado con la base visual
alineado con la documentación vigente

Además debe quedar explícitamente resuelto:

si el dataset se gobierna desde la capa central de autorización

si el módulo usa solo scopes o también constraints funcionales

si las entidades hijas o embebidas se autorizan por delegación al padre

si un create contextual requiere contexto obligatorio y cuál es su canal válido de autorización

si la validación manual del módulo puede reproducirse de forma canónica con Project Lab sin ambigüedad operativa
