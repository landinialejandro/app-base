Objetivo

Este documento define el mapa general de entidades del proyecto app-base en su estado actual y su dirección de crecimiento.

Su finalidad es:

visualizar la estructura general del sistema
distinguir entidades del sistema y entidades del negocio
evitar duplicación de modelos
orientar la creación de nuevos módulos
mantener una visión coherente del crecimiento del ERP SaaS

Este documento no redefine la arquitectura multi-tenant, sino que organiza conceptualmente el dominio.

1. Criterio general de modelado

El sistema distingue entre:

entidades del sistema
entidades del negocio
entidades operativas
entidades documentales
entidades futuras de soporte

La regla general es:

users pertenece al sistema
parties es la entidad base del negocio para personas y organizaciones
en la interfaz y en el lenguaje funcional del sistema, parties se presentará como contactos
los módulos operativos deben relacionarse preferentemente con parties cuando intervengan actores externos o internos del negocio
antes de crear nuevas tablas base debe evaluarse si el caso corresponde a una variante de una entidad ya existente

2. Entidades del sistema

Estas entidades forman el núcleo técnico y de acceso del SaaS.

users

Representa usuarios con acceso al sistema.

Responsabilidades:

autenticación
acceso
permisos
sesiones
operación interna del sistema

tenants

Representa empresas o cuentas independientes dentro del SaaS.

Responsabilidades:

aislamiento lógico de datos
contexto de operación
pertenencia empresarial

memberships

Vincula usuarios con tenants.

Responsabilidades:

pertenencia de un usuario a una o varias empresas
eventual relación con roles/permisos dentro del tenant

3. Entidad base del negocio

parties

Entidad general para personas u organizaciones vinculadas a la operación de un tenant.

Nombre técnico interno:

parties

Nombre funcional e interfaz:

contactos

Ejemplos de uso:

cliente
proveedor
empleado
contacto
contratista
tercero

Criterio actual:

implementación simple y funcional
diferenciación inicial mediante campo de tipo o clase
evolución futura posible hacia perfiles especializados o múltiples roles

Campos conceptuales esperables:

tenant_id
kind
name
display_name opcional
document_type nullable
document_number nullable
tax_id nullable
email nullable
phone nullable
address nullable
notes nullable
is_active
timestamps
soft deletes

Criterio de interfaz:

en pantallas, menús y textos visibles al usuario se utilizará contacto o contactos
el término party queda reservado al modelo técnico, tabla, rutas y convenciones internas

4. Entidades operativas principales

Estas entidades representan la operación diaria del negocio.

projects

Agrupa trabajos, servicios, iniciativas o procesos internos/comerciales.

Relaciones principales:

pertenece a tenant
puede vincularse a contacts
puede contener múltiples tasks
puede vincular documentos en el futuro

tasks

Unidad operativa de trabajo.

Representa actividades concretas que deben realizarse dentro del sistema.

Las tareas pueden existir:

dentro de un proyecto
asociadas a un contacto
asignadas a un usuario
o de manera independiente

Campos relevantes:

tenant_id
project_id nullable
party_id nullable
assigned_user_id nullable
name
description
status
due_date
timestamps
soft deletes

Relaciones principales:

tenant → tasks
project → tasks
party → tasks
user → tasks

tasks constituye el núcleo operativo del sistema para la gestión del trabajo diario.

products

Representa productos, artículos, repuestos, insumos o ítems comercializables o utilizables.

Relaciones probables:

pertenece a tenant
puede participar en orders
puede participar en documents
puede vincularse a inventory

orders

Representa pedidos u órdenes comerciales u operativas.

Relaciones probables:

pertenece a tenant
se vincula con un contacto
puede tener múltiples productos
puede derivar en documents
puede relacionarse con payments

5. Entidades documentales

El sistema tenderá a una familia documental unificada.

documents

Entidad genérica para documentos comerciales u operativos.

Ejemplos de tipos:

factura
remito
proforma
orden de trabajo
presupuesto
nota de crédito
recibo
comprobante interno

Criterio actual:

no profundizar todavía en exceso
pensar la familia documental como una misma línea conceptual
diferenciar inicialmente por document_type
dejar para una segunda etapa la profundización de numeración, series, estados complejos y relaciones avanzadas

Relaciones probables:

pertenece a tenant
puede vincularse a contacts
puede vincularse a orders
puede vincularse a projects
puede vincularse a products mediante detalle
puede vincularse a payments

6. Entidades financieras futuras

payments

Representa cobros, pagos o movimientos relacionados con documentos u órdenes.

Relaciones probables:

pertenece a tenant
puede vincularse a documents
puede vincularse a contacts
puede vincularse a orders

7. Entidades de inventario futuras

inventory

Representa stock o existencia de productos.

Relaciones probables:

pertenece a tenant
se vincula con products
puede vincular movimientos futuros

inventory_movements

Entidad futura para entradas, salidas, ajustes o transferencias.

Relaciones probables:

pertenece a tenant
se vincula con inventory o products
puede relacionarse con documents o orders

8. Entidades futuras de soporte

Estas no son prioridad en la etapa actual, pero forman parte del crecimiento natural del sistema.

addresses

Para múltiples direcciones por entidad.

contacts

Reservado conceptualmente para múltiples medios de contacto o contactos secundarios.

Nota importante:

aunque la interfaz del sistema utilizará la palabra contactos para referirse a parties,
no se creará una tabla contacts en esta etapa.

party_profiles

Para especializaciones futuras por tipo de party.

document_sequences

Para numeración por tipo documental, sucursal o tenant.

branches

Sucursales o sedes del tenant.

roles

Roles internos de acceso o de negocio.

permissions

Permisos del sistema.

attachments

Archivos adjuntos relacionados con contacts, projects, tasks o documents.

notes

Notas internas vinculables a distintas entidades.

Orden sugerido de crecimiento:

orders

inventory

11. Regla de racionalización

Antes de crear una nueva entidad base debe evaluarse:

si pertenece al sistema o al negocio
si puede resolverse dentro de parties
si puede resolverse dentro de documents
si es una verdadera entidad nueva o una especialización futura
si conviene implementarla ahora o dejarla como TODO

13. Regla final

El crecimiento del sistema debe seguir una lógica de:

núcleo estable
entidades reutilizables
especialización progresiva
mínima duplicación
evolución por capas

La prioridad actual es construir una base operativa y adaptable, evitando profundizaciones innecesarias en esta etapa.
