OBJETIVO DEL DOCUMENTO

Este documento define la lectura funcional y doctrinal de Shopping Autoservicio dentro de app-base.

Shopping Autoservicio se identifica técnicamente como:

self_service_sales

Su función es ordenar una capacidad comercial autogestiva por tenant, orientada a permitir que clientes externos puedan registrarse, acceder a una tienda pública controlada, seleccionar una tienda vinculada, comprar o cargar saldo, consumir fichas mediante QR, consultar movimientos y operar dentro del alcance comercial habilitado por cada tenant.

Este documento busca:

definir la naturaleza de self_service_sales
separar clientes externos de memberships internas
definir la relación con Parties
ordenar la tienda pública del tenant
distinguir /shop/{tenant:slug} como tienda pública tenant-scoped
distinguir /tienda como entrada pública general externa
distinguir /tienda/seleccionar como selector externo temporal de tiendas
distinguir /tienda/salir como salida externa del plano customer
definir la identidad externa customer
definir la relación customer-tienda
definir el token temporal de selección de tienda
definir la acción mínima de selección externa de tienda
definir la sesión externa mínima namespaced
definir el middleware externo pasivo de resolución contextual
definir la administración interna inicial de tienda
distinguir configuración interna de tienda y consumo público externo
definir la tienda interna como SelfServiceShop
definir los artículos publicados como SelfServiceShopItem
definir publicación inicial de productos activos
definir saldo real, fichas, fichas promocionales y consumo futuros
establecer límites frente a Products, Inventory y Documents
preparar integración futura con Mercado Pago
preparar integración futura con controladores externos
definir alcance inicial de implementación
registrar los cortes operativos validados
evitar que la capacidad se resuelva como parches dispersos en módulos existentes
evitar que la confirmación de email se confunda con habilitación operativa completa
evitar que la selección de tienda se confunda con sesión interna
evitar que la sesión externa mínima se confunda con guard externo definitivo
evitar que el middleware externo pasivo se confunda con autorización obligatoria
evitar que el selector externo prometa operación comercial real
evitar que la administración interna de tienda se confunda con operación externa customer

No define:

implementación técnica completa
migraciones finales de todas las fases futuras
controllers definitivos de saldo, QR, pagos o minimercado
campos finales de saldo, ledger, fichas o consumos
integración real con Mercado Pago
integración real con hardware
diseño visual final de tienda completa
contratos fiscales definitivos
promociones finales
minimercado completo
guard externo definitivo
middleware externo definitivo de autorización obligatoria
remember token externo
validación final de identidad operativa
operación comercial real

Estado operativo vigente:

self_service_sales cuenta con una primera base implementada y validada como registro externo pendiente, confirmación por token y creación de Party customer person sin User, sin Membership y sin acceso interno al tenant.

Corte operativo vigente del selector externo:

Corte operativo vigente del middleware externo pasivo:

self_service_sales incorpora middleware externo pasivo de resolución contextual para /shop/{tenant:slug}.

Middleware vigente:

ResolveSelfServiceExternalCustomer

Lectura:

el middleware resuelve contexto externo solo cuando existe sesión externa mínima válida y corresponde al tenant público de la ruta.

El middleware comparte payload externo en el request para que la tienda pública pueda representar customer externo activo.

El middleware no protege /shop/{tenant:slug} como ruta privada.

El middleware no redirige visitantes públicos.

El middleware no exige login externo.

El middleware no habilita operación comercial.

El middleware no usa Auth.

El middleware no escribe session('tenant_id').

El middleware no crea User.

El middleware no crea Membership.

El middleware no reemplaza guard externo definitivo.

El middleware no reemplaza middleware externo definitivo de autorización obligatoria.

Si la sesión externa corresponde a otro tenant, /shop/{tenant:slug} se muestra como tienda pública visitante.

Si no existe sesión externa, /shop/{tenant:slug} se muestra como tienda pública visitante.

self_service_sales incorpora una primera administración interna UI de Tienda.

Lectura:

SelfServiceShopItem representa productos publicados en la tienda.

ProductLineItemSelector funciona como base para seleccionar productos activos publicables.

La administración interna de tienda no habilita compra, saldo, fichas, QR, pagos, carrito ni operación comercial real.

La selección válida no crea:

Regla:

Toda evolución futura debe validarse contra documentación vigente, código real, seguridad, tenant activo, permisos, esquema real, runtime y evidencia técnica.

NATURALEZA DE SHOPPING AUTOSERVICIO

Shopping Autoservicio es una capa satélite comercial autogestiva por tenant.

No debe leerse como:

módulo CRUD ordinario
reemplazo de Products
reemplazo de Inventory
reemplazo de Parties
reemplazo de Documents
reemplazo de Security
portal interno del tenant
membership del cliente externo
pasarela de pago propia
controlador de hardware
sistema fiscal completo por sí mismo

Debe leerse como:

capacidad comercial autogestiva
tienda pública controlada por tenant
frente externo para clientes
capa de registro externo
capa de identidad externa customer
capa de selección externa de tienda
capa de sesión externa mínima
capa de publicación comercial
capa de administración interna de tienda
capa de saldo prepago futura
capa de consumo por ficha futura
capa de movimientos comerciales futura
capa preparada para pagos externos
capa preparada para consumo por QR
capa preparada para integración con controlador externo

Nombre doctrinal:

Shopping Autoservicio

Key técnica:

self_service_sales

Ruta pública tenant-scoped vigente:

/shop/{tenant:slug}

Entrada pública general vigente:

/tienda

Rutas internas vigentes de administración:

/self-service/shops...

Nombres internos de ruta vigentes:

self_service_sales.shops.*

Label visible:

Tienda

Regla:

Shopping Autoservicio orquesta módulos y capacidades existentes sin absorber su ownership funcional.

Products conserva catálogo.

Inventory conserva stock físico y movimientos materiales.

Parties conserva clientes, contactos y sujetos fiscales.

Documents conserva formalización documental y comprobantes cuando correspondan.

Security conserva autorización.

self_service_sales conserva la experiencia comercial autogestiva, la tienda, el registro externo, la identidad externa, la relación customer-tienda, la publicación comercial, el saldo futuro, las fichas futuras, los movimientos comerciales futuros y el consumo por QR futuro.

Primer corte operativo validado:

self_service_sales ya cuenta con una primera implementación centrada en:

registro externo pendiente
confirmación por token
creación de Party kind = person
creación de PartyRole role = customer
ausencia de User interno
ausencia de Membership
ausencia de acceso al panel interno

Corte operativo vigente de administración interna:

self_service_sales incorpora administración interna inicial para tiendas mediante SelfServiceShop y SelfServiceShopItem.

Lectura:

SelfServiceShop configura la tienda del tenant desde el plano interno.

SelfServiceShopItem representa un producto publicado en esa tienda.

SelfServiceShopItem no reemplaza Product.

SelfServiceShop no reemplaza Tenant.

La tienda pública muestra únicamente lo que el contrato público permita exponer.

Regla:

la existencia de self_service_sales no modifica la doctrina vigente de Parties.

El cliente externo confirmado se representa como Party person con relación customer.

La condición autogestiva pertenece a self_service_sales, no a parties.kind.

Regla adicional:

la entrada pública general /tienda no es login interno.

No habilita panel interno.

No habilita navegación tenant interna.

Su función es separar el acceso externo de customers del acceso interno de usuarios del sistema.

Regla adicional de administración interna:

la administración interna de tienda pertenece al plano interno autenticado.

No puede ser operada por customer externo.

No puede ser autorizada por sesión externa self_service_sales.*.

No puede confundirse con la tienda pública /shop/{tenant:slug}.

Regla central:

el cliente externo no es membership del tenant.

El cliente externo no usa el login interno de app-base.

El cliente externo no crea User interno.

El cliente externo no accede al panel interno.

El cliente externo no posee roles internos del tenant.

El cliente externo no participa de owner, admin, operator, sales ni administrator.

El cliente externo no administra SelfServiceShop ni SelfServiceShopItem.

El cliente externo solo opera dentro de Shopping Autoservicio y únicamente sobre sus propios datos comerciales autorizados.

Regla adicional:

las rutas públicas de Shopping Autoservicio deben permanecer separadas del grupo interno auth + tenant.

Rutas públicas externas vigentes:

/shop/{tenant:slug}

/tienda

/tienda/seleccionar

/tienda/salir

/shop/{tenant:slug} resuelve el tenant por slug público y no habilita operación interna del tenant.

/tienda funciona como entrada externa general y no como login interno.

/tienda/seleccionar funciona como selector externo temporal de tiendas y no como selector interno de tenants.

/tienda/salir funciona como salida externa del acceso customer y no como logout interno del usuario app-base.

POST /tienda/seleccionar ejecuta selección mínima externa y redirige hacia la tienda pública tenant-scoped cuando el token y la relación customer-tienda son válidos.

La selección válida puede iniciar sesión externa mínima namespaced.

ResolveSelfServiceExternalCustomer puede resolver contexto externo pasivo para /shop/{tenant:slug} cuando existe sesión externa mínima válida para el mismo tenant público.

Regla:

operar en la tienda no equivale a pertenecer al tenant.

La autenticación externa no debe confundirse con autorización interna.

La sesión externa de self_service_sales no debe confundirse con sesión interna app-base.

La selección externa de tienda no debe confundirse con tenants/select.

El token temporal de selección no equivale a sesión externa persistente por sí mismo.

Marcar used_at no equivale a iniciar sesión interna.

Iniciar sesión externa mínima namespaced no equivale a crear guard externo definitivo.

Resolver contexto externo pasivo no equivale a proteger la tienda como ruta privada.

Redirigir a /shop/{tenant:slug} no equivale a habilitar operación comercial.

Ver catálogo publicado no equivale a habilitar operación comercial.

POST /tienda/salir limpia solo la sesión externa de self_service_sales.

POST /tienda/salir no debe invalidar la sesión Laravel completa.

POST /tienda/salir no debe cerrar sesión interna si existiera.

La entrada pública externa no debe mostrar ni prometer acceso a dashboard, tenant/profile, parties.show ni rutas internas.

Regla adicional para administración interna:

las rutas self_service_sales.shops.* pertenecen al plano interno.

No son rutas públicas.

No son rutas customer.

No pertenecen a sesión externa mínima.

No deben consumir autorización externa.

Deben requerir usuario interno autenticado, tenant activo y autorización backend efectiva.

<<SECTION: CLIENTE EXTERNO Y PARTIES>>

CLIENTE EXTERNO Y PARTIES

El cliente externo de Shopping Autoservicio debe representarse dentro del tenant como Party.

Lectura vigente para el caso base:

party.kind = person

party_roles.role = customer

No debe crearse:

party.kind = auto-customer

party.kind = self-service

membership para el cliente externo

rol interno para el cliente externo

usuario interno del tenant para el cliente externo

Regla:

la condición autogestiva pertenece a self_service_sales, no a parties.kind.

Party representa el cliente comercial dentro del tenant.

La cuenta externa/autogestiva representa el acceso del cliente al Shopping Autoservicio.

Membership queda reservado para usuarios internos del tenant.

Regla adicional:

el cliente externo se mantiene como persona.

Si necesita comprobante fiscal a nombre de una empresa, se debe resolver mediante datos fiscales o Party company vinculada para facturación, sin reemplazar la identidad principal del cliente person.

Primer corte validado:

la confirmación del registro externo crea una Party person dentro del tenant y asigna PartyRole customer.

El flujo validado no crea User.

El flujo validado no crea Membership.

El flujo validado no vincula la Party customer a una membership.

Corte vigente adicional:

la confirmación externa puede crear o vincular una cuenta externa customer y una relación customer-tienda según contrato vigente.

Relaciones vigentes:

SelfServiceCustomerAccount = cuenta externa customer

SelfServiceStoreCustomer = relación customer-tienda

Party = representación comercial tenant-scoped

Lectura:

SelfServiceCustomerAccount no reemplaza User.

SelfServiceStoreCustomer no reemplaza Membership.

Party no reemplaza cuenta externa.

Party no reemplaza relación customer-tienda.

Regla de duplicados:

el alta autogestiva debe bloquearse si ya existe cualquier Party dentro del mismo tenant con el mismo email o teléfono.

Si el flujo incluye DNI/document_number, también debe bloquearse ante duplicado por ese dato.

Esta regla no se limita a Parties con rol customer.

Regla:

self_service_sales puede crear una Party customer person como resultado de confirmación externa validada, pero no debe reutilizar PartyController como contrato de alta externa.

PartyController pertenece al panel interno y a permisos internos de Parties.

Lectura vigente del registro liviano:

el registro inicial externo puede crear Party con datos mínimos confirmados.

Datos mínimos actuales para Party:

kind = person

name = display_name

display_name = display_name

email

phone

document_type = null

document_number = null

Regla:

que Party permita datos mínimos no implica habilitación operativa comercial completa.

La habilitación operativa del customer externo debe resolverse por contrato propio de self_service_sales.

Identidad operativa vigente:

identity_stage = operational_identity_completed

Lectura:

el customer completó la identidad requerida para este corte.

No significa operación comercial habilitada.

Operación vigente:

operation_enabled = false

significa:

customer completo, pero todavía no aprobado para operar.

operation_enabled = true

junto con canOperate() = true

significa:

customer aprobado por owner/admin para operar en tienda según contrato vigente.

Reglas:

confirmar email no habilita operación comercial completa

completar identidad operativa no habilita operación comercial completa

crear contraseña externa no habilita operación comercial completa

validar credencial externa no habilita operación comercial completa

seleccionar tienda no habilita operación comercial completa

iniciar sesión externa mínima no habilita operación comercial completa

resolver contexto externo pasivo no habilita operación comercial completa

ver catálogo publicado no habilita operación comercial completa

Regla final:

el cliente externo existe comercialmente como Party dentro del tenant, pero su acceso y operación pertenecen a self_service_sales.

<<SECTION: REGISTRO DEL CUSTOMER>>
SECTION_VERSION: 00004

REGISTRO DEL CUSTOMER

El registro del customer externo debe ser autogestivo, validado y separado del registro interno de usuarios del tenant.

La Party customer se genera al confirmar el email o completar la validación principal definida para el registro externo.

Registro inicial liviano vigente:

nombre para mostrar
email
teléfono

Lectura:

email es el vínculo fuerte inicial del customer externo.

teléfono es dato obligatorio de contacto.

teléfono no reemplaza al email como vínculo principal.

DNI, nombre real completo, datos fiscales y contraseña externa completa no forman parte obligatoria del registro inicial liviano vigente.

Pueden incorporarse luego en una validación posterior de identidad operativa o en fases posteriores.

Datos opcionales o posteriores:

DNI
nombre real completo
domicilio
datos fiscales
observaciones
contraseña externa completa
otros datos comerciales futuros

Entidad operativa vigente para el registro pendiente:

self_service_customer_registrations

Función:

guardar registros externos pendientes de customer antes de crear la Party definitiva.

Estados conceptuales vigentes del registro externo:

pending
confirmed
expired
cancelled

Modelo vigente:

SelfServiceCustomerRegistration

Request vigente:

StoreSelfServiceCustomerRegistrationRequest

Reglas:

el backend debe validar lo máximo posible para evitar errores de carga

el email debe tener formato válido

el teléfono debe tener formato estricto

el registro debe evitar duplicados dentro del tenant según el contrato vigente

el cliente no debe quedar habilitado plenamente sin validación de contacto

la creación de Party debe respetar tenant resuelto desde la tienda pública

no debe crearse membership

no debe crearse User

no debe habilitarse acceso al panel interno

Regla de duplicados:

la verificación de duplicados debe ejecutarse al recibir los datos iniciales del registro externo.

También debe repetirse defensivamente al confirmar el token.

Debe bloquearse el alta si ya existe cualquier Party del tenant con el mismo email o teléfono.

Si el flujo incluye DNI/document_number, también debe bloquearse ante duplicado por ese dato.

Regla sobre invitations:

el registro externo del customer no debe usar invitations.

invitations pertenece semánticamente al plano interno de invitaciones de usuarios, owner signup y memberships.

Regla sobre controllers internos:

el registro externo del customer no debe usar InvitationAcceptanceController.

Ese flujo pertenece al plano interno y puede crear o vincular User, Membership y login interno.

Regla sobre requests:

StorePartyRequest puede servir como referencia de campos.

No debe usarse como contrato del registro externo de self_service_sales.

El registro externo requiere reglas propias, mayor separación de seguridad y validaciones específicas del plano público.

Flujo vigente de registro pendiente:

/shop/{tenant:slug}/register

Este flujo crea un registro pending en self_service_customer_registrations.

No crea todavía Party.

Flujo vigente de confirmación:

/shop/{tenant:slug}/register/confirm/{token}

Este flujo valida token, estado, vencimiento y duplicados.

Si la validación es correcta:

crea Party kind = person
crea PartyRole role = customer
marca el registro como confirmed
vincula party_id
crea o vincula SelfServiceCustomerAccount según contrato vigente
crea SelfServiceStoreCustomer según contrato vigente
mantiene operation_enabled = false

Validación runtime vigente:

registration_status = confirmed
party_kind = person
party_roles = ["customer"]

Regla de finalización de registro:

luego de confirmar email, el customer debe completar datos críticos y crear contraseña externa dentro de un trayecto único, sin caminos ambiguos.

Datos críticos de finalización:

nombre completo
tipo de documento
número de documento
teléfono
contraseña
confirmación de contraseña
aceptación de términos

Reglas:

la finalización no debe crear User

la finalización no debe crear Membership

la finalización no debe usar Fortify interno

la finalización no debe habilitar panel interno

la finalización no debe habilitar operación comercial completa

El token de confirmación puede retomar onboarding incompleto cuando corresponda al contrato vigente.

Retomar onboarding incompleto no debe crear duplicados.

Lectura vigente de identidad operativa:

identity_stage = operational_identity_completed

significa:

customer con identidad operativa completa para el corte vigente.

No significa:

operación comercial habilitada.

Lectura vigente de operación:

operation_enabled = false

significa:

customer completo, pero todavía no aprobado para operar.

operation_enabled = true

junto con canOperate() = true

significa:

customer aprobado por owner/admin para operar en tienda según contrato vigente.

Regla de experiencia:

la validación debe reducir fricción, pero no sacrificar consistencia de datos.

La prioridad es crear un customer correcto, trazable y vinculado al tenant correspondiente, sin mezclarlo con usuarios internos ni memberships.

Regla adicional:

la confirmación de email no equivale a habilitación operativa completa.

La finalización de identidad operativa no equivale a habilitación comercial completa.

El flujo debe distinguir:

registro inicial liviano
confirmación de email
finalización de identidad operativa
creación de contraseña externa
habilitación interna para operar
operación comercial real futura
