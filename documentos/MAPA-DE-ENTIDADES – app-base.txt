MAPA-DE-ENTIDADES – app-base

Objetivo

Este documento define el mapa conceptual de entidades del proyecto app-base.

Su finalidad es:

visualizar la estructura general del sistema
evitar duplicación de modelos
orientar la creación de nuevos módulos
mantener coherencia en el crecimiento del SaaS

Criterio general de modelado

El sistema distingue entre:

entidades del sistema
entidades del negocio
entidades operativas
entidades documentales
entidades futuras de soporte

Reglas principales:

users es entidad del sistema
parties es la entidad base del negocio
antes de crear una entidad nueva evaluar si puede resolverse con una entidad existente

Entidades del sistema

Estas entidades forman el núcleo técnico del SaaS.

SIGNUP_REQUESTS

Entidad que representa la solicitud pública inicial para crear una nueva empresa en el sistema.

Se utiliza como etapa previa de validación manual o comercial antes de crear un tenant.

Campos principales:

id
requested_name
requested_email
company_name
phone_whatsapp
status
review_notes
approved_at
rejected_at
approved_by
created_at
updated_at

Notas:

No crea por sí sola ni un user ni un tenant.

Representa un proceso previo de validación administrativa o comercial.

Puede dar origen a una o más invitations.

Relación de flujo:

signup_requests aprobada
↓
genera invitation owner_signup
↓
owner acepta invitación
↓
se crea tenant y membership owner

INVITATIONS

Entidad que representa una invitación formal de ingreso o vinculación al sistema.

Es el mecanismo central de onboarding y alta de usuarios relevantes dentro del sistema.

Campos principales:

id
tenant_id (nullable)
type
status
email
token
signup_request_id (nullable)
invited_by_user_id (nullable)
sent_at
expires_at
accepted_at
accepted_ip
user_agent
meta
created_at
updated_at

Relaciones:

belongsTo tenant (nullable)
belongsTo signupRequest (nullable)
belongsTo invitedBy user (nullable)

Dominios cerrados:

type

owner_signup
member_invite

status

pending
accepted
expired
cancelled

Interpretación de campos del flujo:

sent_at
momento en que el superadmin marca la invitación como enviada al owner

accepted_at
momento en que el owner acepta la invitación

Notas importantes:

tenant_id puede ser NULL cuando type = owner_signup.

owner_signup se utiliza para completar el alta de una nueva empresa aprobada desde signup_requests.

member_invite se utiliza para sumar un usuario a un tenant existente.

No existe registro público libre en el sistema; el alta de usuarios relevantes parte desde invitations.

TENANTS

Entidad que representa una organización dentro del sistema.

Define el aislamiento principal del modelo multi-tenant.

Campos principales:

id
name
slug
settings
created_at
updated_at

Notas importantes:

id utiliza UUID (char(36)) como primary key.

slug es único.

Todas las relaciones hacia tenants deben utilizar CHAR(36).

Esto evita incompatibilidades con migraciones que utilicen bigint por defecto.

USERS

Entidad que representa la identidad global del sistema.

Responsabilidades:

autenticación
identidad única del usuario
acceso al sistema

Campo global relevante:

is_superadmin

Define acceso administrativo global al sistema.
No depende de memberships ni de pertenencia a tenants.

Notas:

email es único global.

Un mismo user puede pertenecer a muchos tenants.

Un mismo user puede ser owner de muchos tenants.

La pertenencia real no se resuelve desde users, sino desde memberships.

MEMBERSHIPS

Entidad que representa la pertenencia real entre un user y un tenant.

Es la tabla puente central del modelo multi-tenant.

Campos principales:

id
tenant_id
user_id
status
is_owner
joined_at
blocked_at
blocked_reason
created_at
updated_at

Notas:

Define si un usuario pertenece o no a una empresa.

Permite múltiples memberships para un mismo user en distintos tenants.

Es la fuente válida de pertenencia user-tenant.

Regla sobre TenantScoped

El trait TenantScoped se utiliza únicamente en entidades cuyo ciclo de vida pertenece completamente a un tenant.

Ejemplos de uso correcto:

Product
Order
Document
Branch
Role
Invitation
Party
Project
Task

No debe utilizarse en entidades del sistema como:

User
Tenant
Membership
SignupRequest

Entidad base del negocio

PARTIES

Entidad general para personas u organizaciones relacionadas con el negocio.

Nombre técnico:

parties

Nombre visible en interfaz:

contactos

Ejemplos de uso:

cliente
proveedor
empleado
contratista
tercero

La diferenciación inicial se realiza mediante un campo de tipo.

Dominio funcional:

kind

kind es un dominio cerrado que define el tipo de actor representado por la entidad.

Campos conceptuales esperados:

tenant_id
kind
name
display_name opcional
document_type nullable
document_number nullable
tax_id nullable
email nullable
phone nullable
address nullable
notes nullable
is_active
timestamps
softDeletes

Entidades operativas

PROJECTS

Agrupa trabajos o iniciativas.

Relaciones principales:

tenant → projects
party → projects
projects → tasks

Nota:

projects no posee catálogo funcional propio por el momento.

TASKS

Unidad operativa de trabajo.

Representa actividades concretas dentro del sistema.

Relaciones principales:

tenant → tasks
project → tasks
party → tasks
user → tasks

Dominio funcional:

status

status es un dominio cerrado.

PRODUCTS

Representa productos, artículos o insumos.

Relaciones posibles:

tenant → products
products → orders
products → inventory
products → documents

Dominio funcional:

kind

ORDERS

Representa pedidos u órdenes comerciales.

Características:

representan intención comercial
los totales se calculan dinámicamente a partir de los ítems

Relaciones principales:

tenant → orders
orders → parties
orders → products
orders → documents

Dominios funcionales:

kind
status

ORDER_ITEMS

Detalle de productos dentro de una orden.

Relaciones principales:

order → order_items
product → order_items

Características:

heredan tenant desde la cabecera
no exponen tenant_id en fillable

Dominio funcional:

kind

Regla importante:

kind reutiliza el mismo dominio definido en products.

Entidades documentales

DOCUMENTS

Entidad genérica para documentos comerciales u operativos.

Ejemplos:

factura
remito
presupuesto
orden de trabajo
nota de crédito
recibo

Características:

representan documentos formales
los totales se persisten en la cabecera

Relaciones principales:

tenant → documents
documents → parties
documents → products
documents → projects
documents → orders

Dominios funcionales:

kind
status

DOCUMENT_ITEMS

Detalle de líneas dentro de un documento.

Relaciones principales:

document → document_items
product → document_items

Características:

heredan tenant desde cabecera
eliminación de cabecera elimina ítems

Dominio funcional:

kind

Regla:

kind reutiliza el mismo dominio de products.

Entidades financieras futuras

PAYMENTS

Representa pagos o cobros asociados a documentos.

Relaciones posibles:

tenant → payments
payments → documents
payments → parties

Entidades de inventario futuras

INVENTORY

Representa stock de productos.

INVENTORY_MOVEMENTS

Registra movimientos de stock.

Entidades futuras de soporte

addresses
attachments
notes
roles
permissions
branches
document_sequences
party_profiles

Nota importante

Aunque la interfaz utiliza el término contactos, no se creará una tabla contacts.

La entidad real continúa siendo:

parties

Regla final

El crecimiento del sistema debe seguir estos principios:

núcleo estable
entidades reutilizables
especialización progresiva
mínima duplicación
evolución gradual del dominio.
