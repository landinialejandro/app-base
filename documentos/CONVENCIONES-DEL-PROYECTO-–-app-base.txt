CONVENCIONES DEL PROYECTO – app-base
------------------------------------

Este documento define las convenciones de desarrollo que deben seguirse
en el proyecto app-base para mantener consistencia, claridad y facilidad
de mantenimiento del sistema.


OBJETIVO

Establecer reglas claras sobre:

- organización del código
- nomenclatura
- estructura de módulos
- diseño de vistas
- convenciones de interfaz

Estas convenciones deben aplicarse a todos los módulos nuevos del sistema.


ARQUITECTURA GENERAL

El sistema está construido con Laravel y utiliza una arquitectura basada en:

- controladores
- modelos Eloquent
- vistas Blade
- middleware
- migraciones

Cada módulo debe seguir la estructura estándar de Laravel.


MULTI-TENANT

El sistema utiliza un modelo multi-tenant basado en empresa.

Entidad central:

Tenant

Cada registro de datos debe pertenecer a un tenant.


RESOLUCIÓN DE TENANT

El tenant actual se resuelve mediante:

Middleware:
ResolveTenant

El tenant puede obtenerse desde:

1. sesión del usuario
2. header X-Tenant (modo API/debug)


ACCESO AL TENANT ACTUAL

Dentro de la aplicación el tenant actual puede obtenerse mediante:

app('tenant')

Los controladores deben utilizar siempre este tenant para filtrar datos.


CONVENCIONES DE BASE DE DATOS

Cada tabla relacionada a datos de negocio debe incluir:

tenant_id

Ejemplo:

projects
parties
tasks
documents


CONVENCIONES DE CONTROLADORES

Los controladores deben seguir el patrón REST estándar de Laravel.

Ejemplo:

index
create
store
show
edit
update
destroy

Los controladores deben ser responsables de:

- obtener datos
- aplicar filtros por tenant
- enviar datos a las vistas


CONVENCIONES DE MODELOS

Los modelos deben utilizar:

- Eloquent
- relaciones explícitas
- fillable o guarded correctamente definidos


CONVENCIONES DE RUTAS

Las rutas deben definirse en:

routes/web.php

Utilizar resource routes siempre que sea posible.

Ejemplo:

Route::resource('projects', ProjectController::class);


CONVENCIONES DE VISTAS

Las vistas deben utilizar Blade.

Cada módulo debe tener su carpeta:

resources/views/projects
resources/views/parties
etc.


BASE VISUAL

El sistema utiliza un layout principal:

layouts/app.blade.php

Las páginas internas deben seguir la siguiente estructura:

<x-page>
    <x-breadcrumb />
    <x-page-header />
    <x-card />
</x-page>


COMPONENTES BLADE

Componentes visuales disponibles:

x-page
Contenedor base de página.

x-breadcrumb
Ruta de navegación.

x-page-header
Encabezado de página con título y acciones.

x-card
Contenedor visual de contenido.


LISTADOS

Los listados deben utilizar:

- x-card
- table-wrap
- table

Las tablas deben incluir encabezados y filas de datos.

Los estados vacíos deben mostrarse dentro de la card.


FORMULARIOS

Los formularios deben seguir estas reglas:

- reutilizar parciales (_form)
- botones dentro de form-actions
- validaciones mostradas cerca del campo

Ejemplo:

projects/_form
parties/_form


TERMINOLOGÍA DE INTERFAZ

La interfaz del sistema debe utilizar términos orientados al usuario.

Ejemplos:

Tenant → Empresa
Party → Contacto

En el código se puede mantener la nomenclatura técnica,
pero en la interfaz se deben usar los términos amigables.


EMPRESA ACTUAL

La empresa actual se muestra en el header global de la aplicación.

Por lo tanto no es necesario repetir la empresa dentro de cada vista.


USUARIOS Y EMPRESAS

Reglas del sistema:

- un usuario debe pertenecer al menos a una empresa
- un usuario puede pertenecer a múltiples empresas
- si un usuario tiene una sola empresa, no debe pasar por el selector
- si tiene varias, puede cambiar entre ellas desde la interfaz


CAMBIO DE EMPRESA

El cambio de empresa se realiza mediante:

selector en el header de la aplicación.

Esto evita obligar al usuario a cerrar sesión.


PÁGINAS PÚBLICAS

Algunas páginas son públicas y no utilizan el layout completo.

Ejemplos:

welcome
login
register
selección de empresa

Estas páginas utilizan:

$publicPage = true


CREACIÓN DE NUEVOS MÓDULOS

Al crear un módulo nuevo se debe:

1. crear modelo
2. crear migración
3. crear controlador resource
4. registrar rutas
5. crear vistas siguiendo la base visual
6. filtrar datos por tenant


PRINCIPIOS GENERALES

1. Simplicidad
Evitar complejidad innecesaria.

2. Consistencia
Seguir siempre las convenciones del sistema.

3. Escalabilidad
Las decisiones deben permitir crecimiento futuro.

4. Claridad
El código debe ser fácil de entender.


CONCLUSIÓN

Estas convenciones permiten mantener el sistema ordenado,
predecible y fácil de mantener.

Todos los módulos nuevos deben seguir estas reglas para
garantizar coherencia en todo el proyecto.
