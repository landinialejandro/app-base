Proyecto: app-base (Laravel multi-tenant)

leer contexto vigente
trabajar sobre código real
pedir evidencia mínima necesaria
no inventar estructura
no inferir APIs
no suponer tablas, rutas o contratos

Regla adicional consolidada:

ante integraciones entre users, memberships y parties, separar siempre:

draft → pending_approval

return_tab se usa como contexto interno del host para preservar el panel activo cuando corresponde.

trail continúa representando contexto externo entre recursos.

HERRAMIENTAS OPERATIVAS DISPONIBLES

El proyecto dispone de automatizadores internos para acelerar tareas repetitivas, mantener formato canónico y reducir errores manuales.

Caso vigente:

normalizador y actualizador de código por STDIN.

Capacidades actuales:

* creación o reemplazo de archivos completos PHP
* creación o reemplazo de archivos completos Blade
* normalización automática de encabezado FILE y versión
* actualización parcial de métodos PHP existentes
* alta de métodos nuevos dentro de clases PHP
* mantenimiento de log operativo de cambios
* salida de consola con estado y trazabilidad
* validaciones básicas de ruta para evitar escrituras accidentales

Criterio de uso:

usar actualización parcial cuando el cambio sea puntual y localizado dentro de un método PHP existente

usar archivo completo cuando existan imports, métodos nuevos, cambios estructurales de clase, Blade completo o riesgo de que el parser parcial no cubra correctamente el caso

si una tarea consiste principalmente en aplicar código ya definido sobre archivos existentes, priorizar formato compatible con automatizador antes que edición manual fragmentada

Regla:

la existencia de una herramienta no reemplaza auditoría técnica ni criterio arquitectónico.

La herramienta ejecuta cambios.
La decisión de usarlos sigue siendo del proceso técnico.

SI NECESITÁS ARCHIVOS

Pedir solo archivos mínimos necesarios.

Usar formato claro y copiable.

Cuando se requiera auditoría masiva, solicitar salida ordenada hacia carpeta auditoria.

Ejemplo:

(
find app/Models -type f -name "*.php";
find app/Http/Controllers -type f -name "*.php";
find app/Http/Middleware -type f -name "*.php";
find database/migrations -type f -name "*.php";
find resources/views/orders -type f -name "*.blade.php";
find resources/views/documents -type f -name "*.blade.php";
) | sort | while read file; do
echo "===== $file ====="
cat "$file"
echo
echo
done > documentos/auditoria/auditoria-modulo.txt

<<SECTION: REGLA PARA ARCHIVOS COMPLETOS DE CÓDIGO>>
SECTION_VERSION: 00003

REGLA PARA ARCHIVOS COMPLETOS DE CÓDIGO

Cuando devuelvas un archivo completo, la primera referencia del archivo debe usar siempre este formato:

FILE: ruta/del/archivo.ext | VX

Reglas:

si el archivo es nuevo, usar V1
si el archivo ya existe y no tiene comentario de versión, al devolverlo completo agregarlo como V2
si el archivo ya tiene comentario de versión, incrementar en 1 la última versión existente
esta regla aplica solo cuando se devuelve el archivo completo
si se devuelve solo un fragmento o bloque parcial, no agregar ni modificar versión
el comentario debe escribirse con la sintaxis correcta del archivo correspondiente

Formato canónico adicional para PHP completo:

la salida debe respetar esta estructura exacta:

<?php

// FILE: ruta/del/archivo.php | VX

...resto del código...

Es decir:

- apertura PHP en primera línea
- una línea en blanco
- comentario FILE con versión
- una línea en blanco
- resto del archivo
