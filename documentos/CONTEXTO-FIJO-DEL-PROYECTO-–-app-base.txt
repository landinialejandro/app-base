<<SECTION: Tipo de sistema>>
Tipo de sistema

app-base es una plataforma SaaS multi-tenant desarrollada en Laravel.

Permite que múltiples empresas utilicen la misma aplicación manteniendo aislamiento lógico de datos dentro de una única base de datos.

Cada empresa dentro del sistema se denomina tenant.

El objetivo del proyecto es construir una base ERP SaaS modular que permita gestionar operaciones empresariales reutilizando una arquitectura común y estable.
<<END SECTION>>

<<SECTION: Arquitectura SaaS>>
Arquitectura SaaS

El sistema utiliza una arquitectura multi-tenant basada en una columna:

tenant_id

Todas las entidades del negocio incluyen:

tenant_id

Esto permite aislar los datos de cada empresa dentro de la misma base de datos.
<<END SECTION>>

<<SECTION: Convención real de identificador de tenant>>
Convención real de identificador de tenant

tenants.id utiliza char(36)

Las foreign keys hacia tenant deben respetar ese tipo real validado del proyecto.

No debe asumirse tenant_id como foreignId() por convención de Laravel sin verificar primero el patrón real existente.
<<END SECTION>>

<<SECTION: Relación principal de identidad>>
Relación principal de identidad

La relación de pertenencia del sistema es:

users
↓
memberships
↓
tenants

Características:

Un usuario puede pertenecer a múltiples empresas.
Cada empresa puede tener múltiples usuarios.
Un mismo user puede pertenecer a tenants no relacionados entre sí.
users representa identidad global.
memberships representa acceso por empresa.
<<END SECTION>>

<<SECTION: Acceso efectivo por empresa>>
Acceso efectivo por empresa

El estado de acceso efectivo a un tenant se resuelve por memberships.status y no por users.

El bloqueo o inhabilitación de acceso debe resolverse por membership y no por user.

Esto permite que un usuario continúe activo en un tenant y al mismo tiempo se encuentre bloqueado o inactivo en otro.
<<END SECTION>>

<<SECTION: RESOLUCIÓN DEL TENANT ACTIVO>>
RESOLUCIÓN DEL TENANT ACTIVO

En el flujo tenant, el tenant activo se resuelve por middleware.

Luego de esa resolución, el tenant vigente del request debe consumirse desde app('tenant').

No deben mantenerse helpers, soportes legacy o mecanismos paralelos redundantes para resolver el tenant activo si duplican o contradicen esa fuente vigente.
<<END SECTION>>

<<SECTION: Reglas operativas del tenant activo>>
Reglas operativas del tenant activo

El selector de empresa debe mostrar únicamente memberships activas del usuario autenticado.

ResolveTenant debe validar pertenencia activa al tenant antes de permitir acceso operativo.
<<END SECTION>>

<<SECTION: Aislamiento automático de datos>>
Aislamiento automático de datos

Las entidades del negocio utilizan el trait:

TenantScoped

Este trait aplica automáticamente:

WHERE tenant_id = tenant_actual

Además asigna automáticamente el tenant_id al crear nuevos registros.
<<END SECTION>>

<<SECTION: Regla de uso del trait>>
Regla de uso del trait

TenantScoped solo debe aplicarse a modelos cuyo ciclo de vida pertenece completamente a un tenant.

No debe utilizarse en entidades del sistema como:

User
Tenant
Membership
<<END SECTION>>

<<SECTION: Protección de acceso por URL>>
Protección de acceso por URL

Las entidades empresariales utilizan el trait:

ResolvesTenantRouteBinding

Esto protege el route model binding para evitar acceso a registros de otros tenants.

Consulta efectiva:

id = ?
AND tenant_id = tenant_actual
AND deleted_at IS NULL
<<END SECTION>>

<<SECTION: Soft delete obligatorio>>
Soft delete obligatorio

Las entidades del negocio utilizan:

SoftDeletes

Reglas:

Los registros eliminados no aparecen en listados.
No son accesibles mediante URL.
Siguen existiendo en la base de datos.
<<END SECTION>>

<<SECTION: Aclaración sobre invitations>>
Aclaración sobre invitations

invitations forma parte del flujo de acceso y onboarding del sistema.

Es una entidad híbrida:

puede existir sin tenant asociado cuando type = owner_signup

<<SECTION: Entidad base del negocio>>
Entidad base del negocio

La entidad base para personas u organizaciones es:

parties

Representa cualquier actor relacionado con la operación del tenant.

Ejemplos:

cliente
proveedor
contacto
contratista
tercero

En la interfaz del sistema se utiliza el término:

contactos

El término party queda reservado al modelo técnico.

La asignación interna de responsabilidad operativa no se modela desde Party, sino desde User + Membership.

Para la asignación interna de tareas, el término visible recomendado es:

colaborador

Projects como capa de seguimiento agregado

<<SECTION: Tasks como unidad operativa interna>>
Tasks como unidad operativa interna

<<SECTION: Appointments como agenda operativa reutilizable>>
Appointments como agenda operativa reutilizable

appointments es un módulo core de agenda o reserva operativa reutilizable por tenant.

Reglas actuales consolidadas:

pertenece a un tenant

requiere assigned_user_id como responsable operativo

puede vincularse opcionalmente con parties

puede vincularse opcionalmente con assets

puede vincularse opcionalmente con orders

su capa visible puede adaptar vocabulario funcional según el rubro configurado del tenant

el módulo ya dispone de una vista calendario mensual operativa

la vista calendario reutiliza el mismo contexto funcional del tenant que usa el resto del módulo

desde calendario no se permite crear turnos en fechas anteriores a hoy

los turnos en fechas pasadas siguen siendo visibles y consultables

un turno pasado puede reprogramarse a hoy o futuro si no está completado

un turno en estado completado no debe poder moverse de fecha ni modificar su horario
<<END SECTION>>

<<SECTION: Perfil funcional del tenant>>
Perfil funcional del tenant

El tenant ya incorpora un perfil funcional de rubro persistido en:

Este rubro define etiquetas funcionales visibles en módulos adaptables, comenzando por appointments.

La personalización por tenant no se resuelve mediante etiquetas libres.

Debe apoyarse en:

un catálogo cerrado de rubros
el contexto del tenant
catálogos funcionales del módulo correspondiente
<<END SECTION>>

<<SECTION: OWNER Y ADMIN>>
OWNER Y ADMIN

Conforman el nivel privilegiado actual dentro del tenant.

Reglas vigentes:

owner es único por tenant

owner no es un rol delegable y se resuelve estructuralmente desde memberships.is_owner

admin es el rol delegable de gestión amplia dentro del tenant

admin no equivale a owner como condición estructural, aunque puede compartir capacidades elevadas según la configuración de permisos del tenant
<<END SECTION>>

<<SECTION: ROLES OPERATIVOS BASE>>
ROLES OPERATIVOS BASE

Roles base actuales del tenant:

sales
operator
administrator

Reglas conceptuales:

sales representa el perfil comercial operativo

operator representa el perfil operativo de ejecución o campo

administrator representa el perfil operativo administrativo u oficina

estos roles no deben interpretarse como capas globales del sistema, sino como roles base propios del tenant

las diferencias funcionales finas futuras no deben resolverse creando nuevos roles base, sino mediante perfiles funcionales y overrides cuando corresponda
<<END SECTION>>

<<SECTION: Reglas generales de autorización ya consolidadas>>
Reglas generales de autorización ya consolidadas

La autorización del sistema debe entenderse mediante una separación obligatoria entre:

dataset visible
registro individual
acción permitida
render visible en UI

Reglas base:

cuando exista una policy, no alcanza con crearla:
el controller debe invocar authorize(...) o authorizeResource(...)
la vista debe respetar permisos reales con @can al mostrar acciones

la tríada obligatoria del sistema es:

policy
authorize en controller
@can en vistas

la resolución del dataset no debe vivir repartida entre controllers o blades

debe apoyarse en la capa central de autorización y en sus resolvers efectivos

las policies actúan como wrapper mínimo de consumo de esa capa central

delete de registros principales compartidos debe reservarse a owner y admin, salvo excepción funcional justificada

los recursos hijos o embebidos cuya vida operativa dependa de una cabecera no deben tratarse como autorización autónoma

esto aplica explícitamente a:

order_items respecto de orders
document_items respecto de documents
attachments respecto de su attachable

create, update y delete de order_items, document_items y attachments deben depender de la capacidad update sobre el recurso padre correspondiente

La doctrina detallada de implementación segura, Fortify, onboarding, hardening y controles de acceso debe concentrarse en:

SEGURIDAD-Y-ACCESO-DEL-PROYECTO – app-base
<<END SECTION>>

<<SECTION: Patrón de modelado cabecera / detalle>>
Patrón de modelado cabecera / detalle

Cuando un módulo represente una operación compuesta por múltiples renglones, debe modelarse preferentemente utilizando el patrón cabecera/detalle.

Reglas del patrón:

la cabecera representa el acto principal
los ítems representan el detalle operativo
los ítems heredan el tenant de la cabecera

los ítems, por regla general, no exponen tenant_id en fillable desde formularios públicos.

Si el alta del ítem se realiza internamente desde un flujo controlado del sistema y requiere asignación explícita del tenant heredado de la cabecera, puede incluirse tenant_id en fillable para permitir la creación consistente del registro.

la eliminación de la cabecera elimina sus ítems
la posición del ítem es responsabilidad del módulo
los importes pueden calcularse dinámicamente o persistirse según la naturaleza funcional del documento
<<END SECTION>>

<<SECTION: Numeración operativa>>
Numeración operativa

El sistema implementa numeración automática para órdenes y documentos.

Reglas actuales:

la numeración se asigna al guardar la entidad

no se reserva numeración antes del guardado

la numeración se controla mediante document_sequences

la secuencia se define por:

tenant
tipo documental
punto de venta

orders y documents no permiten edición manual del número

una vez asignado el número, el tipo de entidad no debe modificarse
<<END SECTION>>

<<SECTION: Flujo tasks → orders>>
Flujo tasks → orders

El sistema contempla una derivación operativa opcional desde tasks hacia orders.

Reglas actuales:

una tarea puede originar una única orden

la relación operativa 1 a 1 se expresa desde orders.task_id

la orden derivada suele corresponder a una operación de servicio u otra acción formalizada a partir de la tarea
<<END SECTION>>

<<SECTION: Flujo assets → orders → documents>>
Flujo assets → orders → documents

El sistema ya contempla una relación operativa opcional entre activos, órdenes y documentos.

Reglas actuales:

orders puede vincularse opcionalmente a assets

documents puede vincularse opcionalmente a assets

cuando un document se genera desde un order, hereda automáticamente:

tenant_id
party_id
order_id
asset_id si la orden lo posee
<<END SECTION>>

<<SECTION: Flujo orders → documents>>
Flujo orders → documents

Desde orders.show pueden generarse documentos derivados de una orden.

La relación entre ambos módulos debe entenderse así:

orders representa la operación interna del tenant

documents representa la formalización documental, comercial o fiscal derivada de esa operación

El flujo actual contempla como tipos documentales activos al menos:

quote
delivery_note
invoice

work_order permanece solo como kind legacy inhabilitado y no debe formar parte del flujo operativo normal ni de la UI corriente.

Cuando un documento se genera desde una orden:

documents hereda automáticamente:

tenant_id
party_id
order_id
asset_id si la orden lo posee

Los order_items pueden copiarse automáticamente a document_items.

Una vez copiados los ítems, los totales del documento deben recalcularse según la lógica del módulo documental.
<<END SECTION>>

<<SECTION: Reglas de fechas del flujo operativo>>
Reglas de fechas del flujo operativo

Para mantener coherencia en las operaciones comerciales:

orders siempre deben tener:

party_id
ordered_at

documents siempre deben tener:

party_id
issued_at

En orders la fecha es editable con restricciones razonables hacia el futuro.

En documents la fecha se controla según el tipo documental y su coherencia con la orden asociada.

Por ejemplo:

un documento asociado a una orden no debe tener una fecha anterior a la orden.
<<END SECTION>>

<<SECTION: Capa inicial de impresión y PDF>>
Capa inicial de impresión y PDF

El sistema ya incorpora una capa inicial de impresión y exportación PDF aplicada sobre módulos operativos existentes.

Alcance actual validado:

Reglas y decisiones actuales:

la capa de impresión no se modela como módulo de negocio independiente

las vistas Blade imprimibles se reutilizan como base común para salida HTML de impresión y para generación de PDF

<<SECTION: Factories y seeders>>
Factories y seeders

En factories utilizadas para testing es válido crear tenants automáticamente, por ejemplo:

tenant_id => Tenant::factory()

En seeders reales del sistema el tenant normalmente se asigna explícitamente desde el seeder.

No es necesario modificar este comportamiento.
<<END SECTION>>

<<SECTION: Criterio actual de deploy y declaración de rutas>>
Criterio actual de deploy y declaración de rutas

En despliegues sobre hosting compartido sin CLI, Composer ni Artisan en servidor, el proyecto prioriza estrategia de deploy por versión cerrada preparada en local.

En ese contexto, las rutas del core operativo se mantienen declaradas explícitamente en routes/web.php en lugar de depender de Route::resource(...), debido a un incidente real de registro inconsistente de rutas durante la primera prueba operativa.
<<END SECTION>>

Además del flujo multi-tenant principal, el sistema incorpora una capa global de administración denominada superadmin.

<<SECTION: Configuración básica de empresa>>
Configuración básica de empresa

tenants.settings se utiliza para persistir configuración básica editable de empresa cuando esa configuración no justifica entidad propia.

El perfil de empresa ya existe como pantalla operativa para owner.

Además de datos generales, settings puede contener configuración funcional del tenant cuando esta no justifique todavía una entidad específica.

Esto incluye, entre otros casos, el perfil de rubro en:

settings.business_profile.type

La pantalla tenant.profile incorpora además una primera UI real de gestión de permisos V2 por tenant mediante un tab interno permissions.

Reglas actuales de esa pantalla:

solo owner puede operar esta primera gestión

el owner no habilita módulos desde tenant.profile, solo ajusta permisos sobre módulos ya habilitados previamente para el tenant

la edición actual se limita a roles base delegables:

admin
sales
operator
administrator

la persistencia de esta fase se realiza sobre role_permission

membership_permission_overrides queda fuera de alcance en esta primera UI

la edición efectiva de permisos por rol depende de que los roles base existan materialmente en cada tenant y no solo en catálogo o código

si un tenant quedó desincronizado respecto de los roles base persistidos, la UI puede requerir sincronización previa de datos para operar correctamente
<<END SECTION>>

<<SECTION: Decisión actual de frontend base>>
Decisión actual de frontend base

El frontend base del proyecto ya migró a un patrón declarativo con data-action="app-..." y public/js/app-base.js como capa mínima reutilizable de comportamiento.

La capa de impresión puede incorporar además recursos propios acotados a esa responsabilidad, por ejemplo CSS y JavaScript específicos del módulo print, sin romper la base declarativa general del proyecto.

El shell general de la app ya incorpora un footer reusable en pantallas internas no públicas.

Ese footer puede exponer identificación visible de versión o estado del sistema.

En entorno local, esa identificación se resuelve automáticamente desde .git como identificador de desarrollo.

Este comportamiento forma parte efectiva del shell del sistema y no debe tratarse como funcionalidad futura.
