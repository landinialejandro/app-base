OBJETIVO DEL DOCUMENTO

Este documento define las convenciones estructurales y de desarrollo del proyecto app-base.

Su objetivo es mantener consistencia técnica, previsibilidad arquitectónica y coherencia de código a medida que el sistema crece.

Las convenciones aquí definidas deben respetarse al crear:

nuevos módulos
nuevas entidades
nuevas vistas
nuevos controladores
nuevos flujos operativos
<<END SECTION>>

CONVENCIONES GENERALES

El proyecto sigue los principios de:

simplicidad
claridad estructural
baja duplicación
coherencia entre módulos
previsibilidad del comportamiento del sistema

Cuando existan varias alternativas técnicas, se debe priorizar siempre la opción más simple y consistente con el resto del proyecto.

Reglas generales:

no asumir estructura, rutas, APIs, tablas ni contratos sin evidencia real

trabajar sobre código vigente y documentación activa

pedir solo la evidencia mínima necesaria antes de intervenir

evitar soluciones híbridas que mezclen patrones nuevos con patrones legacy

no cerrar una migración, refactor o reconversión con convivencia indefinida entre generaciones

mantener ownership funcional claro entre módulos

consumir capacidades oficiales del módulo oferente antes de crear variantes locales

no trasladar complejidad contractual, técnica u operativa al usuario cuando el sistema pueda resolverla internamente

la UI debe tender a acciones simples, comprensibles y de bajo roce

las validaciones críticas deben sostenerse en backend, servicios, policies, Security o contratos oficiales, no en Blade ni frontend

las reglas owner/admin del frente tenant/profile deben centralizarse en una pieza reusable del plano tenant y no repetirse en controllers, policies ni Blades

Pieza vigente:

app/Support/Tenants/TenantProfileAccess.php

Regla:

las vistas pueden expresar disponibilidad, readonly, botones visibles o estados bloqueados, pero no decidir autorización final

Convención documental consolidada:

la base documental técnica del proyecto debe tender a una estructura homogénea y navegable

los documentos técnicos activos deben utilizar:

bloque estructural de metadatos
TÍTULO como nombre humano del documento
DOC_SLUG como identificador técnico canónico
DOC_VERSION como versión general del documento

las referencias internas entre documentos deben tender a resolverse mediante:

<DOC_REF slug="...">Título visible</DOC_REF>

no deben utilizarse nombres de archivo físicos como referencia narrativa dentro del cuerpo del documento

las carpetas baks y auditoria no forman parte de la base documental activa navegable

Convención adicional consolidada para contenido orientado a usuario:

los textos dirigidos a usuario final deben utilizar lenguaje no técnico
deben priorizar claridad, accesibilidad y comprensión inmediata
deben construirse con frases cortas y cierre conceptual claro
pueden incorporar líneas de refuerzo tipo “👉 …” como patrón de cierre

este criterio aplica a:

documentación de onboarding
ayudas contextuales
contenidos UX embebidos

Convención operativa adicional:

Project Lab forma parte del proceso real de validación técnica, auditoría, actualización de código y actualización documental del proyecto.

Cuando el caso toque permisos, constraints, multi-tenancy, runtime real, snippets, auditorías, actualización de código o actualización documental, debe utilizarse la guía operativa vigente y no una prueba abstracta improvisada.

<DOC_REF slug="guia_project_lab_proyecto_app_base">Guía operativa de Project Lab</DOC_REF>

Convención modular consolidada:

los módulos del sistema deben entenderse como unidades funcionales cerrables equivalentes a mini apps internas del tenant

cada módulo debe tender a describirse mediante:

núcleo propio
contrato de entrada
contrato de salida
contrato de seguridad
oferta de integración

esta convención no obliga a que todos los módulos publiquen desde el inicio todas sus capacidades reutilizables

sí obliga a que, cuando las publiquen o consuman, lo hagan bajo contrato claro y sin reabrir arbitrariamente el núcleo de otros módulos

Regla de convergencia:

si dos módulos o hosts equivalentes muestran distinta madurez contractual, no debe degradarse el más avanzado para alinearlo al más viejo.

La convergencia correcta del ecosistema debe tender hacia el patrón más maduro ya validado.

Regla de autonomía modular:

un módulo debe poder distinguir entre:

dato propio necesario para existir
vínculo gestionado opcional
contexto transitorio de navegación o request
relación de origen, planificación o seguimiento
formalización derivada
enriquecimiento futuro

Regla:

un campo `*_id` no debe interpretarse automáticamente como dependencia obligatoria.

Su sentido debe definirse por contrato funcional real, esquema vigente, validación backend y doctrina del módulo.

Caso vigente:

party_id en Orders y Documents es vínculo gestionado opcional.

No es condición obligatoria de existencia funcional cuando el registro conserva dato propio suficiente de contraparte.

Convención de snapshots propios:

cuando un módulo puede vincularse opcionalmente a Parties pero necesita conservar dato propio de contraparte, debe utilizar un snapshot textual propio.

Patrón vigente:

party_id = vínculo gestionado opcional

counterparty_name = snapshot textual propio del registro

Casos vigentes:

orders.counterparty_name
documents.counterparty_name

Reglas:

el snapshot no reemplaza autorización

el snapshot no reemplaza Party

el snapshot no convierte el vínculo gestionado en obligatorio

el snapshot permite que el registro conserve lectura autónoma aunque no exista Party vinculada

los listados, show summaries, prints, surfaces y salidas visuales deben priorizar el método de presentación propio cuando exista

Caso vigente:

displayCounterpartyName()

Regla:

no debe mostrarse únicamente party->name si el módulo posee snapshot propio de contraparte y fallback controlado.

Convención de relaciones externas hacia Orders:

Orders debe conservar autonomía funcional.

Las relaciones de seguimiento, planificación u origen pueden vivir en el módulo que las origina o acompaña.

Casos vigentes:

tasks.order_id

appointments.order_id

Regla:

Orders no debe conservar task_id ni appointment_id como parte de su núcleo persistente.

task_id o appointment_id pueden viajar como contexto transitorio de request o navegación cuando la creación se origine desde Task o Appointment.

Ese contexto no debe convertirse automáticamente en campo persistente del módulo receptor.

Convención de derivación entre módulos:

un módulo puede derivar otro registro sin transferir ownership ni autorización.

Caso vigente:

Order → Document

Reglas:

Orders aporta contexto operativo

Documents conserva ownership documental

la creación derivada debe pasar por autorización efectiva del módulo receptor

el documento derivado debe conservar sus propios datos, líneas, totales, lifecycle y snapshot cuando corresponda

Convención de migraciones:

las migraciones deben respetar el orden real de creación de tablas.

No debe agregarse una foreign key hacia una tabla que todavía no existe en la secuencia real de migraciones.

Caso vigente:

asset_id no debe quedar directamente dentro de create_orders_table si assets se crea después.

Regla:

si el vínculo es válido pero la tabla destino aparece después, debe resolverse en una migración posterior compatible con el orden real.

Convención de seeders:

los seeders que crean registros directamente deben respetar el contrato vigente del módulo.

No deben depender de controllers para completar datos obligatorios o snapshots.

Caso vigente:

si Orders o Documents usan counterparty_name como snapshot propio, los seeders correspondientes deben poblar ese dato cuando generen registros que lo requieran.

Regla de subcapas contextuales internas:

cuando una capacidad contextual pertenece a un módulo host pero no constituye CRUD, navegación propia ni módulo independiente, puede organizarse como subcapa interna del módulo.

Caso vigente:

ProductOperationalSummary

Lectura:

subcapa contextual de Products
sin CRUD
sin navegación propia
sin ModuleCatalog propio
sin dashboard general
sin rutas propias

Ubicación vigente:

app/Support/Products/OperationalSummary/
resources/views/products/operational-summary/

Regla:

una subcapa interna debe mantenerse bajo el módulo host, pero separada de partials comunes si posee identidad funcional propia.

No debe degradarse a include local disperso si puede publicarse mediante la infraestructura surface-driven vigente.

Caso corregido:

ProductOperationalSummary dejó de resolverse como include directo o partial común dentro de products/partials.

La salida estable se resolvió mediante ProductSurfaceService.

Patrón aplicado:

consumer solicita
módulo oferente resuelve
surface publica
host monta

Caso vigente:

ProductOperationalSummary publica summary_items en products.show mediante ProductSurfaceService.

Summary items vigentes:

Compras
Ventas
Entradas / salidas

Regla:

los hosts deben consumir información mediante surfaces oficiales cuando el sistema ya posee infraestructura surface-driven.

No debe usarse include directo en el host como cierre estable si existe una surface adecuada del módulo oferente.

Regla de modelado:

una entidad no debe comprimir en un único campo conceptos que pertenecen a dimensiones distintas.

Cuando una clasificación mezcle naturaleza, tipo, relación operativa o vínculo comercial, debe separarse el núcleo de la entidad de sus relaciones o tablas hijas.

Caso vigente:

parties.kind representa únicamente tipo de contacto.

party_roles.role representa relación del contacto con la empresa.

No deben volver a usarse customer, supplier ni employee como valores de parties.kind.

Regla adicional de clasificación:

las taxonomías operativas deben ser mínimas, objetivas, no ambiguas y útiles para filtros, reportes, trazabilidad, historial técnico e interpretación futura por IA.

La cantidad de valores no debe ser el diferencial.

El diferencial debe ser claridad, explotabilidad estadística, utilidad operativa y ausencia de ambigüedad.

Regla:

no deben mezclarse dimensiones conceptuales distintas en un mismo campo o taxonomía.

Deben mantenerse separadas, cuando correspondan:

propiedad o relación del objeto
ubicación de ejecución
lifecycle administrativo/base
estado operativo de ejecución
custodia material
consumo material
prioridad
tipo técnico
origen de necesidad
cierre técnico
cierre administrativo
cierre final

Caso doctrinal vigente para mantenimiento:

interno
externo
remoto
indefinido

puede funcionar como clasificación estructural mínima de ubicación de ejecución.

Valores como taller, box, base, planta, obra, cliente, predio o locación deben tratarse primero como ubicaciones específicas, configuración o labels visibles, no necesariamente como tags estructurales universales.

Convención adicional de lenguaje de UI para parties:

en la interfaz de parties debe evitarse la palabra rol para no confundir relación del contacto con roles de permisos o memberships.

La lectura visible correcta debe priorizar:

Tipo de contacto
Relación con la empresa

Convención adicional de lenguaje de UI para contraparte:

cuando un módulo distinga entre vínculo gestionado y dato textual propio, la interfaz debe reflejar esa diferencia.

Lectura vigente:

Contacto gestionado = party_id opcional

Contraparte = snapshot textual propio del registro

Casos vigentes:

Orders
Documents

Regla:

la UI no debe presentar party_id como si fuera obligatorio cuando el contrato del módulo permite operar con contraparte textual propia.

Regla de constraints funcionales:

las constraints funcionales deben operar sobre la dimensión real del dominio del módulo.

No deben sostenerse por compatibilidad si la dimensión conceptual fue reconvertida.

Caso vigente:

orders conserva allowed_kinds porque kind representa una segmentación operativa válida del módulo.

parties no debe usar allowed_kinds porque parties.kind representa únicamente tipo de contacto.

parties debe usar allowed_party_roles porque la relación con la empresa vive en party_roles.role.

Convención de actividad operativa transversal:

la integración con OperationalActivity debe realizarse mediante eventos operativos livianos emitidos por el módulo dueño después de autorizar, validar y ejecutar su operación.

No corresponde llamar directamente al logger desde controllers o services del módulo dueño.

No corresponde registrar actividad solo por existir CRUD.

No corresponde registrar cambios técnicos sin valor operativo.

No corresponde registrar deletes, vistas, descargas, previews, prints o PDFs salvo decisión explícita posterior.

Los campos observables deben declararse en el catálogo del módulo dueño mediante activityTrackedFields() o contrato equivalente vigente.

El listener de actividad operativa debe mantenerse queued cuando corresponda al patrón vigente.

OperationalActivity puede recibir metadata operativa adicional cuando un cambio compuesto no viva directamente como atributo simple del modelo principal.

Caso validado:

parties puede registrar cambios de party_roles como metadata operativa de Party.

Regla:

la metadata operativa no convierte una entidad hija en módulo propio.

Convención de métricas contextuales:

una subcapa contextual puede leer datos operativos de otros módulos cuando esa lectura respete ownership, permisos, módulos habilitados y contratos de los módulos dueños.

Caso vigente:

ProductOperationalSummary lee compras y ventas desde Orders cuando corresponde, y entradas/salidas desde Inventory.

Reglas:

Products presenta el resumen contextual

Orders aporta transacciones reales cuando está habilitado y autorizado

Inventory aporta movimientos y flujo material cuando está habilitado

Parties aporta proveedor o cliente cuando existe relación real

OperationalActivity no debe usarse como fuente principal de métricas

los cálculos no deben hacerse en Blade

las vistas deben representar payload ya resuelto

Convención de exclusión de costos accesorios:

fletes, mano de obra, kilómetros, viajes, prorrateos, costos accesorios, costos indirectos y rentabilidad contable completa no deben incorporarse dentro de una primera lectura contextual de producto si requieren criterios de imputación, proveedor u orden distinta.

Deben tratarse como posible línea futura de costos directos operativos, no como parte automática del resumen operativo inicial del producto.

Convención de automatización:

toda automatización relevante debe respetar tenant activo, módulos habilitados, permisos efectivos, lifecycle, ownership funcional, validaciones backend y trazabilidad.

La automatización no debe crear datos imposibles para el contrato vigente.

La automatización no debe ocultar decisiones críticas.

La automatización debe reducir carga manual, no aumentar confusión.

Cuando una automatización impacte operación real, debe dejar rastro suficiente para auditoría, revisión humana, línea de tiempo, análisis posterior e interpretación por IA cuando corresponda.

Convención de IA operativa:

la IA debe leerse como capa transversal de asistencia y reducción de fricción, no como módulo funcional ordinario ni como canal paralelo de operación.

No debe implementarse IA sin:

caso de uso claro
datos autorizados
separación entre sugerencia y ejecución
fallback
validación backend posterior
respeto por Security
respeto por tenant activo
respeto por lifecycle
respeto por ownership modular

La IA no debe operar por scraping visual, inferir permisos, inventar datos faltantes ni cerrar procesos críticos sin confirmación humana o regla explícita previamente aprobada.

La arquitectura debe separar intención IA de proveedor IA.

No debe asumirse API paga, proveedor único ni dependencia temprana con OpenAI u otro servicio externo.

Regla:

las nuevas capacidades deben evitar reglas implícitas, labels dispersos, textos especializados quemados en Blades y decisiones no documentadas cuando esas reglas deban ser interpretables por una futura capa IA.

Convención de capas satélite productivas:

una capacidad especializada puede organizar un frente operativo amplio sobre módulos existentes sin convertirse automáticamente en módulo CRUD ordinario.

Caso vigente:

Servicios y Mantenimiento debe leerse como capa satélite productiva ERP/CMMS liviana.

Reglas:

no duplicar ownership de módulos existentes

no absorber Orders, Assets, Inventory, Products, Appointments, Tasks, Documents, Attachments, Parties ni OperationalActivity

auditar primero si la necesidad puede resolverse mediante lectura consolidada, surfaces, contratos existentes o pieza satélite

no abrir módulos o campos por reflejo automático

no reabrir arbitrariamente módulos ya cerrados para introducir complejidad propia de una capa satélite

cuando una necesidad pertenezca claramente a un módulo existente, debe consumirse la capacidad oficial del módulo oferente

cuando una necesidad pertenezca específicamente al universo de la capa satélite, debe evaluarse como pieza satélite antes de modificar el núcleo de módulos cerrados

Convención de seeders demo operativos:

los seeders demo que representan carga operativa real no deben funcionar como simples pobladores silenciosos de tablas si el sistema ya posee eventos operativos y trazabilidad transversal.

Cuando un seeder demo represente una operación real del sistema, debe tender a utilizar código interno vigente, modelos, servicios y eventos reales.

No debe depender de controllers, requests web, navegador ni sesión HTTP.

Reglas:

no sostener soluciones híbridas con compatibilidad legacy indefinida

no conservar adaptadores silenciosos para datos antiguos

no reinterpretar internamente payloads legacy hacia contratos nuevos

no crear datos que el sistema vigente ya no reconoce

no corregir silenciosamente datos mal formados

fallar fuerte si el payload no cumple contrato vigente

validar estados contra catálogos vigentes

emitir eventos operativos cuando la carga demo represente actividad operativa real

procesar cola y validar persistencia si participa OperationalActivity

poblar snapshots propios cuando el módulo los requiera y el seeder no pase por controller

Caso vigente:

si el contrato de Orders exige group y kind, el seeder debe cargar group y kind explícitamente.

No debe aceptar entradas antiguas para reinterpretarlas internamente.

Contrato vigente para orders demo:

group = sale|purchase|service
kind = standard

Regla:

la carga demo debe hablar el contrato vigente del módulo.

No debe simular una compatibilidad que el sistema ya no debe sostener.

Convención de lectura contextual agregada de actividad:

un módulo puede declarar opcionalmente un activity_record_set para indicar qué records deben formar parte de una lectura contextual agregada de actividad.

Esta declaración pertenece al módulo dueño.

No pertenece a OperationalActivity.

El provider debe implementar el contrato común:

ActivityRecordSetProvider

La convención vigente es:

si el módulo declara activity_record_set
OperationalActivity usa ese conjunto de records

si el módulo no declara activity_record_set
OperationalActivity usa lectura directa del record host autorizado

Caso validado:

projects declara ProjectActivityRecordSet.

ProjectActivityRecordSet define Project + Tasks como dataset de actividad visible del proyecto.

Regla:

OperationalActivity no debe conocer internamente relaciones propias de módulos como Project → Tasks, Order → Items o Document → Items.

Cada módulo debe publicar esa frontera si corresponde.

No deben agregarse condiciones específicas dentro de OperationalActivity como if Project, if Order o if Document para decidir lectura agregada.

Convención de lectura contextual especial de actividad:

un módulo puede declarar opcionalmente un activity_context para indicar un criterio contextual especial de lectura de actividad cuando la frontera no pueda representarse correctamente como simple conjunto de records.

Esta declaración pertenece al módulo dueño.

No pertenece a OperationalActivity.

El provider debe implementar el contrato común vigente:

ActivityContextProvider

La convención vigente es:

si el módulo declara activity_context
OperationalActivity usa el criterio contextual publicado por el módulo dueño

si el módulo no declara activity_context ni activity_record_set
OperationalActivity usa lectura directa del record host autorizado

Caso validado:

parties declara PartyEmployeeActivityContext para Party con relación employee y membership activa vinculada.

PartyEmployeeActivityContext permite incluir:

actividad directa de Party
actividad donde el user vinculado sea actor_user_id
actividad donde el user vinculado sea subject_user_id

Reglas:

activity_context no reemplaza activity_record_set

activity_context no autoriza records por sí mismo

activity_context no habilita lectura general del tenant

activity_context no debe definirse dentro de OperationalActivity

activity_context no debe hardcodearse como condición interna de OperationalActivity

activity_context debe respetar tenant activo y host autorizado

Convención de presentación de cambios operativos:

la vista no debe interpretar directamente metadata.changes.

La transformación de cambios persistidos a lectura visual debe resolverse mediante una pieza de presentación.

Caso validado:

OperationalActivityChangePresenter

Responsabilidad:

transformar metadata.changes en change_summary y change_details.

Regla:

la tabla de actividad puede mostrar una síntesis compacta de cambios y ofrecer detalle bajo demanda, pero no debe incorporar reglas de campos ni lógica de interpretación del JSON.

Convención para capacidades tenant no modulares:

no toda capacidad tenant-scoped debe incorporarse a ModuleCatalog.

Si una capacidad no es módulo funcional, no posee navegación propia, no tiene CRUD completo y no se habilita o deshabilita como módulo ordinario, debe evaluarse como capacidad tenant no modular.

Caso validado:

operational_activity.view_any

Esta capability autoriza lectura general tenant-scoped de OperationalActivity desde Security sin convertir OperationalActivity en módulo funcional ordinario.

Convención para surfaces tenant/transversales no modulares:

una capacidad tenant-scoped o transversal no modular puede publicar surfaces reutilizables mediante una vía específica distinta de ModuleCatalog.

Caso validado:

TenantSurfaceRegistry

Este canal permite que una capacidad no modular publique información controlada hacia hosts funcionales sin convertirse en módulo ordinario.

Caso vigente:

OperationalActivity
TenantSurfaceRegistry
OperationalActivitySurfaceService
ModuleSurfaceRegistry
hosts show

Reglas:

no incorporar una capacidad transversal a ModuleCatalog solo para publicar UI

no insertar tabs manualmente en cada host consumidor si la capacidad puede publicarse por registry reusable

no pasar una capacidad tenant/transversal no modular por TenantModuleAccess como si fuera módulo funcional ordinario

no crear navegación principal ni CRUD por el solo hecho de publicar una surface

ModuleSurfaceRegistry puede fusionar surfaces modulares desde ModuleCatalog y surfaces tenant/transversales desde TenantSurfaceRegistry.

La surface tenant/transversal debe respetar la misma gramática esperada por los hosts.

Si la surface se monta como tab principal del host, su key debe ser compatible con return_tab.

Caso validado:

key = activity

No debe usarse un identificador técnico largo como key visible cuando genere navegación poco legible o inadecuada.

Caso corregido:

operational_activity.contextual

Regla:

el identificador técnico conceptual de una capacidad transversal puede estar dado por la clase oferente, el registry tenant y el contexto.

No necesariamente debe trasladarse como key visible de tab.

Convención de surfaces modulares:

cuando una capacidad contextual pertenece al mismo módulo host y puede integrarse mediante la infraestructura surface-driven, debe publicarse desde el surface service del módulo correspondiente.

Caso vigente:

ProductOperationalSummary se publica mediante ProductSurfaceService.

La salida entra a products.show como summary_items y no mediante include directo.

Regla:

una surface modular puede publicar summary_items, tabs u otros bloques compatibles con la gramática del host cuando el módulo oferente conserva ownership de la lectura.

El host monta payload ya resuelto.

El Blade no calcula métricas ni reconstruye contratos.

Convención de navegación para actividad operativa:

no persistir URLs finales en registros de actividad operativa.

La navegación hacia registros asociados debe resolverse desde una pieza de lectura.

Caso vigente:

OperationalActivityLinkResolver resuelve label, URL canónica cuando existe show seguro, fallback textual y trail contextual.

Regla:

si no existe ruta canónica segura, se muestra texto plano y no se inventa link.

Convención de componentes Blade:

no usar componentes Blade no existentes.

Si el componente no está en resources/views/components ni tiene clase correspondiente, no debe usarse.

Caso detectado:

x-empty-state no existe actualmente como componente runtime disponible.

Regla adicional:

no asumir utilities CSS no validadas como parte de la base visual real.

Si se necesita un comportamiento estructural seguro, como salto de línea, debe preferirse markup HTML simple antes que depender de una utility no confirmada.

Caso detectado:

d-block no produjo el salto de línea esperado en Blades compactos de ProductOperationalSummary.

Corrección aplicada:

usar div para garantizar estructura visual en líneas separadas.

Convención de Project Lab y colas:

Project Lab puede tener acciones operativas locales fuera del producto, como procesar cola pendiente en entorno local.

Esto no implica funcionalidad SaaS productiva.

Regla:

no ejecutar queue:work desde UI del producto.

En producción debe definirse worker supervisado por infraestructura, no botón operativo dentro del SaaS.

ESTRUCTURA BASE DEL PROYECTO

Este bloque define la ubicación canónica de código, vistas, assets y documentación técnica relevante del proyecto.

No busca inventariar todo el repositorio.

Busca fijar dónde vive cada responsabilidad principal para evitar búsquedas imprecisas, suposiciones técnicas o dispersión de criterios.

APP

app/ concentra la lógica principal del sistema.

Ubicaciones relevantes:

app/Models
modelos del sistema y del negocio

app/Models/Concerns
traits estructurales del dominio

app/Events
eventos del sistema y eventos operativos livianos cuando correspondan

app/Listeners
listeners del sistema, incluyendo listeners queued cuando correspondan

app/Http/Controllers
controladores del flujo web

app/Http/Middleware
middleware operativo transversal

app/Http/Requests
validaciones por formulario o acción

app/Policies
wrappers mínimos de autorización

app/Support/Auth
capa técnica de acceso y arbitraje

app/Support/Catalogs

CONVENCIONES DE ACCESO POR MÓDULO Y ROLES

La autorización operativa del sistema debe implementarse respetando la arquitectura central definida para acceso y permisos.

Reglas de implementación:

no hardcodear permisos en controllers, policies o blades si existe resolución central

las policies deben consumir la capa central de autorización y no replicar lógica

los controllers deben delegar en authorize(...) o Security y no implementar lógica paralela

las vistas no deben inferir permisos

las vistas deben respetar permisos reales mediante @can solo como verificación final cuando la capacidad no dependa de contexto

los alcances visibles en UI deben resolverse mediante catálogos o resolvers y no reconstruirse manualmente

la habilitación de módulos depende de la capa central de acceso por tenant y no debe asumirse por defecto

la UI de permisos del tenant debe operar únicamente sobre módulos habilitados previamente

Regla crítica validada:

no se debe usar autorización abstracta cuando la capacidad depende de contexto

no utilizar authorize('create', Model::class) en estos casos

debe utilizarse la capa central con parámetros explícitos

La existencia de subject class-string no debe asumirse como condición doctrinal para resolver create contextual.

Si el contrato requiere contexto funcional obligatorio, la autorización válida se resuelve por Security con ese contexto.

Caso consolidado adicional:

cuando el contrato persistido incluya constraints funcionales reales, esas constraints deben arbitrar create, view, update y delete de forma consistente y no limitarse a una sola acción

En orders, constraints.allowed_kinds constituye segmentación efectiva del contrato y no metadata decorativa.

En parties, constraints.allowed_party_roles constituye segmentación efectiva del contrato y no metadata decorativa.

Regla:

allowed_kinds no debe utilizarse para parties.

allowed_party_roles debe utilizarse para parties y evaluarse contra party_roles.role.

Regla adicional para capacidades contextuales internas:

ver el módulo host no implica automáticamente ver todas sus subcapas contextuales.

Caso vigente:

ver Products no implica ver ProductOperationalSummary.

ProductOperationalSummary posee condiciones propias de acceso y visibilidad sin convertirse por eso en módulo funcional ordinario ni requerir permisos persistidos nuevos en su primera versión.

Condiciones base vigentes:

tenant activo
membership activa
Products habilitado
Inventory habilitado
rol compatible

Condición adicional:

Orders habilitado permite métricas transaccionales de compras y ventas cuando el usuario posee dataset visible según Security.

Si Orders no está habilitado, no deben mostrarse métricas transaccionales de compra/venta.

Regla:

una subcapa contextual interna puede tener una compuerta de acceso propia, siempre que no reemplace Security ni duplique permisos persistidos innecesarios.

Caso vigente:

ProductOperationalSummaryAccess actúa como compuerta contextual interna de la subcapa.

No reemplaza Security.

No convierte ProductOperationalSummary en módulo.

No introduce capabilities persistidas nuevas en esta primera versión.

Regla adicional:

las métricas de compras y ventas deben usar Security::scope() sobre Orders para no leer órdenes fuera del dataset visible del usuario.

Las vistas compactas solo representan datos ya preparados.

La UI no decide autorización.

Matriz inicial vigente de acceso para ProductOperationalSummary:

owner: compras, ventas, inventory
admin: compras, ventas, inventory
administrator: compras e inventory
sales: ventas
operator: sin acceso

Regla final:

una capacidad contextual interna puede tener visibilidad más restrictiva que el módulo host cuando sus datos dependen de otros módulos, datasets o roles.

CONVENCIONES DE CONTRATO MODULAR

Todo módulo operativo del sistema debe poder describirse mediante un contrato modular explícito.

Ese contrato no reemplaza la implementación ni la documentación específica del módulo.

Capas mínimas esperadas del contrato modular:

núcleo
contrato de entrada
contrato de salida
contrato de seguridad
oferta de integración

NÚCLEO

El núcleo del módulo describe aquello que el módulo resuelve por sí mismo.

Incluye, según corresponda:

entidad o agregado principal
flujo base
CRUD o acciones principales
reglas operativas propias
navegación estructural base
recursos hijos propios
comportamientos internos del módulo

Regla:

el núcleo no debe reabrirse por cada integración nueva solicitada desde otros módulos salvo que cambie realmente el contrato interno del propio módulo.

CONTRATO DE ENTRADA

El contrato de entrada define qué puede recibir el módulo desde el sistema o desde otros módulos.

Puede incluir, según el caso:

contexto funcional
ids relacionales válidos
trail compatible
return_tab cuando el host lo entregue como contexto interno de navegación
datos mínimos de precompletado
origen de creación contextual
disparadores internos del sistema
eventos operativos livianos emitidos después de acciones propias autorizadas
payloads demo cuando el seeder represente carga operativa real

Reglas:

no asumir entradas implícitas no declaradas
no abrir canales paralelos de entrada por conveniencia de implementación
no resolver create contextuales con patrones distintos si el mismo contrato ya puede absorberlos
no aceptar payloads legacy si el contrato vigente ya exige otra estructura
no reinterpretar internamente entradas antiguas hacia el contrato nuevo

si un create requiere contexto funcional obligatorio, no debe documentarse ni implementarse como create abstracto genérico

Caso vigente:

crear una Party employee desde un flujo de membership requiere contexto funcional employee porque la autorización de parties se orienta por party_roles.role mediante allowed_party_roles.

Caso vigente adicional:

OrderModuleSeeder debe declarar directamente group y kind según contrato vigente.

No debe aceptar kind = sale/purchase/service como entrada conceptual legacy para convertirlo internamente.

CONTRATO DE SALIDA

El contrato de salida define qué devuelve o expone el módulo hacia otros módulos o hacia capas consumidoras del sistema.

Puede incluir, según el caso:

referencia resumida del registro
summary reusable
payload mínimo de lookup
relación enlazable
dataset embebible
acción derivada
compatibilidad con trail contextual
compatibilidad con return_tab cuando la acción deba preservar el panel host desde el cual nació
payload arbitrado para integraciones complejas
evento operativo liviano para trazabilidad transversal
activity_record_set para lectura contextual agregada de actividad
activity_context para lectura contextual especial de actividad

Regla:

el contrato de salida debe tender a estabilizarse para que otras integraciones consuman capacidades del módulo sin reabrir su núcleo.

OFERTA DE INTEGRACIÓN

La oferta de integración expresa qué capacidades reutilizables publica el módulo para consumo de otros módulos.

Puede incluir, según el caso:

surfaces
row actions
componentes embebibles
lookup providers
summary providers
servicios operativos
events
activity_record_set
activity_context

Reglas:

la oferta de integración pertenece al módulo oferente

el consumidor no debe duplicarla localmente

si la integración exige lógica interna del módulo oferente, debe evolucionarse la oferta del oferente y no crear bypass en el consumidor

CONTRATO DE SEGURIDAD

El contrato de seguridad describe cómo se autoriza el módulo y sus capacidades.

Debe contemplar:

permisos
scopes
constraints
estado operativo
relaciones sensibles
tenancy
lectura directa
lectura contextual
lectura agregada
lectura contextual especial

Reglas:

el contrato de seguridad no debe resolverse desde UI

las constraints deben operar sobre la dimensión real del dominio

si una regla sensible aparece en múltiples capas, debe centralizarse

la lectura contextual de OperationalActivity no reemplaza permisos del módulo host

la lectura agregada debe ser declarada por el módulo dueño mediante activity_record_set

la lectura contextual especial debe ser declarada por el módulo dueño mediante activity_context

CONTRATO DE SEEDERS DEMO OPERATIVOS

Cuando un módulo tenga seeders demo que representen operaciones reales del sistema, esos seeders deben respetar el contrato modular vigente.

Reglas:

el seeder no debe ser una vía paralela para cargar estados inválidos

el seeder no debe usar payloads legacy por comodidad

el seeder no debe sostener adaptadores silenciosos hacia versiones anteriores del contrato

el seeder debe validar contra catálogos vigentes cuando corresponda

el seeder debe respetar lifecycle real del módulo

el seeder debe utilizar modelos, servicios y eventos internos cuando represente actividad operativa real

el seeder puede preparar contexto tenant explícito en CLI cuando un servicio operativo dependa de app('tenant')

el seeder debe restaurar o limpiar bindings temporales cuando corresponda

el seeder debe fallar fuerte ante datos incompletos, inválidos o fuera de catálogo

el seeder debe emitir eventos operativos cuando la carga demo represente actividad operativa real y el módulo participe en OperationalActivity

el cierre de un seeder demo operativo debe validar cola y persistencia cuando emita eventos queued

Caso validado:

PartyModuleSeeder emite actividad para parties demo nuevas y registra metadata operativa de party_roles.

Caso validado:

OrderModuleSeeder crea órdenes demo con contrato vigente group/kind, incluye una orden testigo de ciclo completo y utiliza OrderInventoryOperationService para ejecutar inventario.

Caso validado:

DocumentModuleSeeder utiliza modelos Document y DocumentItem, estados vigentes de DocumentCatalog e ítems documentales en estado pending.

Regla:

un seeder demo operativo no reemplaza UI ni requests web.

Tampoco debe degradarse a carga dura silenciosa si representa una operación real que el sistema ya puede trazar.

parties

Es un módulo cuya función principal es integrarse en otros módulos o recursos y no necesariamente vivir como navegación principal independiente.

Caso actual consolidado:

attachments

Casos doctrinales relevantes:

tasks
appointments
products

orders

MÓDULO AGRUPADOR

Es un módulo cuya función principal es ordenar, contextualizar o dar lectura agregada sobre otros módulos sin agotar completamente la identidad de esos módulos relacionados.

Caso doctrinal relevante:

projects como agrupador de tasks

MÓDULO DERIVADO

Es un módulo que nace o se entiende principalmente como formalización, consecuencia o prolongación operativa de otro módulo más nuclear.

Caso doctrinal relevante:

documents respecto de orders

CAPACIDAD TRANSVERSAL DE SOPORTE

Es una pieza que registra, acompaña o sostiene funcionamiento común del ecosistema sin constituir necesariamente módulo navegable principal ni CRUD ordinario.

Caso vigente:

OperationalActivity

puede ser indispensable para trazabilidad o lectura operativa

no absorbe ownership funcional de los módulos que producen la actividad

no autoriza ni valida operaciones originales

no debe abrir navegación principal por mera existencia técnica

no debe registrarse actividad por default solo por existir CRUD

la participación de un módulo debe justificarse por valor operativo real

Módulos con integración validada a OperationalActivity:

appointments
tasks
orders
documents
projects
assets
products
attachments
parties
inventory

Aclaración vigente para inventory:

inventory mantiene identidad de módulo nuclear funcional.

Su integración con OperationalActivity se ancla en InventoryOperation como agrupación operativa contextual.

InventoryMovement conserva trazabilidad fina y atómica de stock.

CAPACIDAD TENANT NO MODULAR

Es una capacidad propia del plano tenant o transversal que puede ser evaluable por seguridad sin convertirse en módulo funcional ordinario.

Caso vigente:

operational_activity.view_any

Reglas:

no posee navegación principal propia

no implica CRUD ordinario

no se incorpora automáticamente a ModuleCatalog

no pasa por TenantModuleAccess como módulo funcional

puede ser evaluada desde Security si existe contrato explícito

puede autorizar una lectura general tenant-scoped sin convertir la capacidad en módulo navegable

CAPACIDAD TENANT/TRANSVERSAL NO MODULAR CON SURFACE PUBLICADA

Es una capacidad tenant-scoped o transversal que, sin ser módulo funcional ordinario, publica una surface controlada hacia hosts funcionales.

Caso vigente:

OperationalActivity mediante TenantSurfaceRegistry y OperationalActivitySurfaceService.

Reglas:

no se incorpora a ModuleCatalog por publicar una surface

no se habilita ni deshabilita como módulo ordinario

no crea navegación principal

no crea CRUD

no pasa por TenantModuleAccess

puede ser consumida por hosts funcionales autorizados

debe mantener ownership en el plano tenant/transversal

debe publicar la surface mediante canal específico y no mediante inserciones manuales permanentes en vistas host

CONVENCIONES DE INTEGRACIÓN ENTRE MÓDULOS

La integración entre módulos debe resolverse como consumo de contratos y no como invasión del núcleo del módulo destino.

Reglas:

si un módulo consumidor necesita relacionarse con otro módulo, debe priorizar el consumo de la superficie contractual ya expuesta por el módulo oferente

no debe reabrirse el núcleo del módulo destino por cada integración nueva salvo que cambie su propio contrato interno

no deben proliferar variantes paralelas de:

selectores
tablas embebidas
bloques summary
acciones contextuales
create contextuales
payloads mínimos de referencia

si el módulo oferente ya dispone de una capacidad reusable válida, debe reutilizarse antes de crear una variante específica en el consumidor

si la capacidad reusable todavía no existe y realmente pertenece al módulo oferente, su incorporación debe tratarse como evolución del propio módulo oferente

no debe tratarse como excepción permanente alojada en el módulo consumidor

Regla de consumo seguro:

toda integración entre módulos sigue sujeta al arbitraje central de seguridad

Regla de crecimiento:

un módulo puede dejar publicadas capacidades de integración como oferta futura aun cuando todavía no existan consumidores concretos

esto favorece estabilidad contractual y reduce refactors futuros dispersos

Regla adicional consolidada:

cuando una integración transversal compleja combine:

estado del módulo destino
permiso efectivo
existencia o no de registro relacionado
posible create contextual
estados UX mutuamente excluyentes
navegación contextual requerida

debe priorizarse una capacidad compartida publicada por el módulo oferente antes que duplicar lógica en múltiples consumidores

esa capacidad debe revisarse como unidad completa de integración:

arbitraje backend
payload resuelto
representación visual
ubicación doctrinal correcta
consumo homogéneo

no alcanza con reutilizar solo una vista Blade si la decisión de estados sigue dispersa entre consumidores

cuando la capacidad reusable pertenezca contractualmente a un módulo oferente, su implementación no debe quedar en una ubicación global ambigua que borre su identidad modular

Regla adicional consolidada sobre forma de publicación:

si la relación reusable es fuerte, múltiple o de lectura pesada, debe priorizarse el patrón de embebido modular completo publicado por el módulo oferente

ese patrón debe tender a expresarse mediante wrapper modular tipo partials.embedded-tabs y una pieza interna reusable de listado o tabla

si la relación reusable es compacta, puntual o 1 a 1, debe priorizarse una acción contextual compacta publicada por el propio módulo oferente

no debe forzarse tab pesada cuando la relación no lo necesita

Casos consolidados actuales:

linked-order-action como superficie reusable publicada por orders
linked-party-action como superficie reusable publicada por parties
linked-asset-action como superficie reusable publicada por assets

Caso adicional consolidado:

appointments puede publicar embebido reusable oficial para relaciones fuertes y acción compacta cuando el vínculo consumidor sea puntual

Caso adicional vigente:

inventory debe publicar sus propias superficies de stock, movimientos e historial desde su módulo y los consumidores deben integrarlas sin duplicar tablas locales de stock o variantes paralelas de historial.

documents puede publicar surfaces embebidas oficiales de lectura documental desde su propio módulo para ser consumidas por hosts como orders.show sin volver a tabs relacionales manuales del consumidor.

Regla operativa adicional de cierre:

si una superficie consumidora todavía resuelve localmente acceso, apertura o visualización de una relación ya cerrada por el módulo oferente, el frente de integración no debe considerarse homogéneo ni cerrado

Regla adicional de simplicidad:

la integración correcta no debe exponer al usuario pasos internos, decisiones técnicas o combinaciones contractuales que el sistema pueda resolver por sí mismo

la UI debe presentar acciones simples y comprensibles

la lógica crítica de arbitraje, estado y consecuencias debe permanecer centralizada en backend o en la capacidad oficial publicada por el módulo oferente

Regla adicional para frentes operativos por línea:

cuando una integración reutilizable se consuma por fila dentro de una tabla operativa, el consumidor no debe rearmar arbitraje local de estados, readonly o acciones rápidas

debe recibir y expresar el contexto oficial publicado por el módulo oferente

esto aplica especialmente cuando la línea sigue siendo recurso delegado del padre pero ya posee trazabilidad operativa iniciada

Regla adicional de naming modular:

no nombrar nuevas piezas desde el módulo consumidor cuando la capacidad pertenece al módulo oferente

el naming debe expresar ownership funcional real

Regla adicional para actividad operativa:

OperationalActivity puede ser consumido visualmente solo como lectura persistida y autorizada.

No debe reconstruirse historia operativa desde campos actuales de otros modelos.

No debe inferirse actividad desde:

created_by
updated_by
assigned_user_id
party_id

si existe actividad persistida específica.

La lectura general vigente se resuelve desde tenant/profile mediante tab Actividad.

Esa tab no convierte OperationalActivity en surface reusable cerrada.

OperationalActivityLinkResolver queda como pieza reusable de lectura.

Regla:

no persistir URLs finales dentro de operational_activities.

La navegación hacia el registro asociado debe resolverse dinámicamente y solo cuando exista ruta show canónica segura.

Casos actuales sin link seguro:

InventoryOperation
Attachment

Regla:

la falta de link seguro no debe resolverse inventando rutas ni columnas auxiliares.

Debe mostrarse fallback textual.

<<SECTION: CONVENCIONES DE FLAGS DE SOPORTE MODULAR>>

CONVENCIONES DE FLAGS DE SOPORTE MODULAR

Cuando una pantalla, formulario, partial o componente dependa de la existencia de otro módulo habilitado para el tenant, debe utilizar el flag o señal canónica de soporte modular vigente.

Reglas:

la validación no debe quedarse en la vista principal

debe propagarse hasta la última partial, include, tabla embebida o bloque visual que pueda exponer:

campos relacionales
tabs
enlaces
acciones
botones de alta
referencias visibles

no debe asumirse que una partial profunda conoce el estado modular por contexto implícito

el dato debe llegar explícitamente o resolverse por la vía canónica del sistema

si un módulo no está habilitado:

no deben renderizarse superficies residuales hacia ese módulo

cuando una capacidad reusable publicada por un módulo dependa además de arbitraje backend del oferente, el consumidor no debe recomponer localmente estados alternativos a partir del flag modular

el flag solo define soporte del módulo

la decisión operativa específica debe consumirse desde la capacidad publicada por el módulo oferente

Ejemplos típicos:

supportsOrdersModule
supportsAssetsModule
supportsPartiesModule

Regla adicional consolidada:

la correcta propagación del flag modular no reemplaza la obligación de consumir la capacidad reusable oficial en todas las superficies relevantes

si una partial secundaria conserva una implementación local residual, el cierre del frente modular debe considerarse incompleto aunque la vista principal ya esté alineada

Regla final:

la propagación completa de flags modulares forma parte de la calidad de cierre de un módulo

<<SECTION: ROLECATALOG Y MODULECATALOG>>
ROLECATALOG Y MODULECATALOG

RoleCatalog es el catálogo base mínimo y reservado del sistema para roles estructurales.

ModuleCatalog es la fuente de verdad para slugs de módulos cerrados del sistema.

Reglas:

no hardcodear slugs funcionales del sistema en controllers, policies, helpers o blades si ya existe catálogo técnico

los roles base del sistema deben resolverse desde RoleCatalog

los módulos base del sistema deben resolverse desde ModuleCatalog

los roles base sembrados al crear el tenant no deben modificarse ni eliminarse

los roles base cerrados del tenant no deben crecer por necesidades funcionales puntuales

las variantes funcionales futuras deben resolverse mediante perfiles funcionales y overrides, no creando nuevos roles base

a futuro el tenant podrá ajustar capacidades dentro de la arquitectura vigente, pero sin alterar la base reservada del sistema

<<SECTION: USO DE TENANT_ID EN FILLABLE>>
USO DE TENANT_ID EN FILLABLE

Como regla general, tenant_id no debe exponerse en formularios ni venir desde input del usuario.

El tenant siempre debe resolverse internamente mediante:

app('tenant')

y asignarse automáticamente por el sistema.

Algunos modelos padre del proyecto exponen tenant_id en fillable por consistencia histórica, pero igualmente no debe venir desde formularios públicos ni desde input del usuario.

En entidades hijas (por ejemplo items o registros dependientes), tenant_id tampoco debe exponerse desde formularios del usuario.

Sin embargo, si el registro hijo se crea internamente desde un flujo controlado del sistema y el tenant se hereda explícitamente desde la entidad padre, puede incluirse tenant_id en fillable para permitir la creación interna consistente mediante create().

Esto permite manejar correctamente casos como:

document_items creados desde documentos
order_items copiados hacia documentos derivados

<<SECTION: CONVENCIONES DE ENTIDADES HIJAS>>
SECTION_VERSION: 00008

CONVENCIONES DE ENTIDADES HIJAS

Las entidades hijas deben cumplir:

pertenecer a una cabecera
heredar tenant_id de la cabecera
no recibir tenant_id desde formularios del usuario

Ejemplos:

order_items
document_items

Reglas actualizadas:

cuando la entidad hija forme parte natural de la operación de la cabecera, debe tratarse preferentemente como recurso delegado al padre

la autorización de create, update y delete debe resolverse contra la capacidad update del recurso padre correspondiente

no debe modelarse una autorización autónoma innecesaria para la entidad hija si su ciclo de vida depende funcionalmente de la cabecera

si existiera una policy propia, debe limitarse a delegar explícitamente en el padre y no duplicar doctrina paralela

attachments sigue el mismo criterio general de delegación al recurso padre aunque no sea una entidad hija cabecera/detalle clásica

Regla modular adicional:

las entidades hijas no deben confundirse con módulos autónomos ni con módulos agrupados

que una entidad dependa funcionalmente de una cabecera no implica que todo módulo relacionado deba modelarse como entidad hija

no deben agregarse responsabilidades propias de cabecera al hijo por comodidad de implementación técnica

por ejemplo:

auditoría propia
capabilities independientes
ownership artificial
dataset autónomo

Caso consolidado adicional:

order_items debe delegar create, update y delete en orders.update incluyendo controllers, vistas embebidas, navegación de cancelar y verificaciones de pertenencia al padre

cualquier excepción a esta regla debe justificarse por necesidad real del dominio y no por comodidad de implementación

Aclaración adicional consolidada:

order_item puede actuar como anclaje operativo fuerte de un frente externo como inventory sin dejar de ser entidad hija delegada del padre

esa relevancia operativa no lo convierte en módulo autónomo

tampoco transfiere a order_item la semántica material, el cálculo neto ni el arbitraje operativo del módulo invitado

Aclaración adicional para document_items:

document_item puede actuar como anclaje operativo fuerte de inventory cuando una línea documental genere movimientos trazables.

Esa participación no lo convierte en módulo autónomo ni transfiere a document_items o documents la semántica material del frente inventory.

El recurso sigue delegado a documents, pero puede requerir restricciones operativas adicionales cuando existan movimientos asociados.

Regla adicional de endurecimiento operativo:

cuando una entidad hija ya posea trazabilidad material iniciada, su edición estructural no debe seguir tratándose como CRUD libre por defecto

las restricciones deben responder al estado real del recurso y al impacto operativo existente

Caso vigente relevante para order_items:

mientras la línea no esté completada ni cancelada puede seguir siendo editable dentro de límites compatibles con lo ya ejecutado

la cantidad no puede reducirse por debajo del ejecutado neto vigente

si existe trazabilidad iniciada no deben mutarse atributos identitarios que rompan lectura histórica de los movimientos

Caso vigente relevante para document_items:

cuando una línea documental ya posee movimientos de inventory asociados, su edición o eliminación debe respetar la trazabilidad existente.

La lectura operativa de ejecutado, pendiente y estado de línea debe derivarse de movimientos reales y no de columnas duplicadas en la tabla base.

Regla:

las restricciones adicionales de una entidad hija no alteran su delegación principal al recurso padre.

Solo endurecen el ciclo de vida cuando la operación real ya generó efectos trazables.

<<END SECTION>>

<<SECTION: CONVENCIONES DE OWNERSHIP FUNCIONAL Y PERMISOS OPERATIVOS SIMPLES>>
CONVENCIONES DE OWNERSHIP FUNCIONAL Y PERMISOS OPERATIVOS SIMPLES

Cuando un módulo lo permita, puede adoptarse un criterio simple de ownership funcional del registro en lugar de una capa temprana de permisos compleja.

Reglas:

usuario normal gestiona los registros de su responsabilidad operativa

owner y admin pueden gestionar registros ajenos dentro del tenant

los borrados sensibles pueden reservarse a owner y admin aunque la edición o seguimiento sea más amplia

si un módulo tiene asignado directo, assigned_user_id define la responsabilidad operativa actual

si el registro puede reasignarse, el nuevo asignado pasa a ser responsable operativo

si no existe propietario directo, el control sensible queda en owner y admin o en la policy específica del módulo

Este patrón ya se considera válido para tasks.

<<SECTION: REGLA PARA MÓDULOS OPERATIVOS COMPARTIDOS DEL TENANT>>
REGLA PARA MÓDULOS OPERATIVOS COMPARTIDOS DEL TENANT

Cuando un módulo operativo compartido del tenant no tenga responsable operativo directo:

la policy debe permitir view, create y update a los roles y capacidades habilitados del módulo

delete debe quedar restringido a owner y admin

no debe inventarse ownership artificial mediante assigned_user_id, owner_user_id o restricciones por created_by si el módulo no lo necesita realmente

Esto aplica especialmente a módulos base o compartidos como:

parties
assets
products
orders

orders debe tratarse como módulo operativo interno del tenant

documents no debe absorber órdenes de trabajo operativas ni duplicar la función interna de orders

CONVENCIONES DE SUPPORT Y HELPERS DE ACCESO

La lógica transversal de acceso debe centralizarse en Security.

Reglas:

no implementar lógica de permisos en controllers o blades

no duplicar resolución de acceso

Security es el punto único de consumo de autorización

los resolvers no deben consumirse directamente

las piezas de directorio interno del tenant no reemplazan seguridad, policies ni middleware.

La autorización efectiva continúa perteneciendo a la capa central de acceso.

CONVENCIONES DE POLICIES

La autorización por acción debe resolverse preferentemente mediante Policies.

Reglas:

las policies deben actuar como wrappers mínimos sobre Security

no deben contener lógica de negocio ni reconstruir permisos

<<SECTION: REGLA DE NAVEGACIÓN Y VISIBILIDAD DE UI>>
REGLA DE NAVEGACIÓN Y VISIBILIDAD DE UI

La interfaz no debe mostrar accesos que el usuario no puede usar realmente.

Reglas:

navbar, dashboard y accesos rápidos deben respetar acceso real del usuario al módulo

las acciones visibles en listados, show y sublistados deben respetar permisos reales mediante @can

la navegación principal del tenant debe construirse desde ModuleCatalog cuando exista una capa de navegación central disponible

no se deben hardcodear módulos navegables en componentes de layout cuando el sistema ya pueda resolverlos desde catálogo y acceso efectivo

distinción obligatoria en UI:

la visibilidad de listados completos debe depender de view_any

la visibilidad de acciones sobre registros individuales debe depender de view, update o delete según corresponda

no debe utilizarse view como criterio para mostrar listados completos

lo que el listado muestra
lo que el usuario puede abrir

no deben renderizarse enlaces
no deben renderizarse campos
no deben renderizarse bloques completos asociados
no deben mantenerse accesos residuales

el breadcrumb representa recorrido contextual o estructural de la pantalla actual

el navbar representa anclaje funcional estable del sistema

el navbar no debe copiar ni duplicar el breadcrumb

cuando exista navegación contextual entre módulos relacionados, el módulo activo del navbar puede resolverse por origen funcional del flujo y no solo por la pantalla hoja actualmente abierta

<<SECTION: CONVENCIONES DE BLADE Y FRONTEND LIVIANO>>

CONVENCIONES DE BLADE Y FRONTEND LIVIANO

Las vistas Blade deben mantenerse simples, explícitas y previsibles.

Reglas:

evitar lógica de negocio en Blade

permitir condicionales simples de interfaz

no reconstruir lógica de permisos

la UI debe consumir resultados del backend

@can puede utilizarse solo cuando la capacidad no dependa de contexto

Regla crítica:

cuando una capacidad depende de constraints

la UI no debe evaluarla directamente

no utilizar @can('create') en módulos con constraints

la UI debe consumir flags resueltos o datasets ya filtrados

los botones de creación deben evaluarse según contexto válido

los datasets visibles deben provenir de Security::scope(...)

Convención visual consolidada para índices:

cuando se utilice x-list-filters-card, la alineación correcta del módulo no implica exponer todos los filtros en el bloque principal.

filtros principales visibles en primer plano
filtros secundarios agrupados en bloque colapsable tipo Más filtros cuando corresponda

si una vista consume una surface oficial publicada por otro módulo, Blade no debe reconstruir localmente formularios, tablas ni decisiones de estado equivalentes

debe limitarse a componer la experiencia visual usando la salida oficial recibida

cuando una show page actúe como host de capacidades embebidas publicadas por otros módulos, Blade debe dejar visible la separación entre:

núcleo propio del recurso anfitrión
relaciones estructurales del host
capacidades operativas invitadas de módulos oferentes

la composición visual puede organizar tabs, cards o bloques, pero no debe mezclar en una misma superficie del host métricas, estados o acciones que pertenecen semánticamente al módulo invitado

Regla adicional para show pages surface-driven:

una show page no debe leerse ni implementarse como punto plano único de inserción.

Debe respetar la estructura anfitriona real y sus zonas explícitas de montaje.

Casos vigentes consolidados:

header_actions
summary_items
detail_items
zona tab coordinada construida desde tabItems

Debe leerse y renderizarse como bloque coordinado del host para navegación, toolbar y paneles.

Cuando Blade consuma surfaces externas en un host con slots, no debe suponer que el target alcanza.

Debe respetar la combinación efectiva de type + slot que el host utiliza para montar la integración.

Regla adicional de preservación del núcleo anfitrión:

la incorporación de surfaces externas no debe hacer desaparecer las cards propias, summaries propios ni bloques estructurales del módulo anfitrión.

El host conserva su núcleo visual.

Las surfaces externas agregan contenido por zonas compatibles; no reemplazan el frente principal del recurso.

CONVENCIONES DE FORMULARIOS OPERATIVOS

los modales no deben utilizarse para operaciones principales de creación o edición

<form> @csrf @method cuando corresponda form-actions

campos
ayudas

bloques internos estrictamente propios de los campos

Regla adicional:

un módulo transversal también puede publicar oferta contractual reusable para otros módulos

esa publicación debe pertenecer al propio módulo transversal y no a implementaciones sueltas de cada consumidor

attachments

<<SECTION: NAVEGACIÓN ESTRUCTURAL Y CONTEXTUAL>>
NAVEGACIÓN ESTRUCTURAL Y CONTEXTUAL

La navegación del sistema debe distinguir entre:

navegación estructural
navegación contextual

navegación estructural:
expresa la ubicación estable de la pantalla dentro del sistema o dentro de su módulo

navegación contextual:
expresa el origen funcional desde el cual el usuario llegó a una entidad relacionada, cuando ese origen sigue siendo relevante para la tarea actual

Reglas generales:

toda pantalla debe tener una resolución estructural válida como base, aunque en ciertos flujos pueda mostrarse en modo contextual

la navegación contextual solo debe aplicarse cuando exista un flujo relacionado real y útil para la continuidad de uso

no debe inventarse un origen contextual si la pantalla fue abierta de forma directa, desde index propio, desde búsqueda o desde navegación general del módulo

el breadcrumb del sistema es uno solo, pero debe poder resolverse en modo estructural o en modo contextual según el caso

si ya existe una capa central reutilizable para resolver contexto de navegación, no duplicar esa lógica en múltiples controllers, blades o helpers dispersos

controller resuelve contexto
vista consume contexto

no reconstruir breadcrumb de negocio desde url()->previous() ni desde supuestos del historial del navegador

breadcrumb, retorno al origen y redirects posteriores a acciones deben mantenerse consistentes entre entidades relacionadas

la preservación de contexto UX debe aplicarse con criterio y solo cuando aporte continuidad real de uso

como regla general:

usar navegación estructural por defecto

usar navegación contextual solo como override útil cuando el origen funcional deba preservarse

cuando no exista contexto válido o el origen no agregue valor operativo, debe mostrarse navegación estructural normal
<<END SECTION>>

<<SECTION: RESOLUCIÓN TÉCNICA DEL TRAIL DE NAVEGACIÓN>>
SECTION_VERSION: 00003

RESOLUCIÓN TÉCNICA DEL TRAIL DE NAVEGACIÓN

Reglas:

el mecanismo central vigente del proyecto para breadcrumb contextual es NavigationTrail

los módulos que participen en navegación contextual deben apoyarse en builders o resolutores propios compatibles con NavigationTrail, sin crear variantes paralelas de criterio

controller resuelve trail

vista consume trail

si una pantalla relacionada recibe un trail válido, debe poder heredarlo, extenderlo o colapsarlo sin reconstruir manualmente la navegación desde parámetros sueltos, historial del navegador o deducciones implícitas de relaciones del modelo

los nodos del trail deben almacenar URLs limpias de la entidad actual

no debe serializarse el propio trail dentro del campo url de los nodos

el trail se propaga solo en links, forms y redirects relevantes

cuando el trail vuelva a un nodo ya existente, debe colapsarse de forma limpia y no duplicada

cuando un módulo ya esté migrado a NavigationTrail, no debe seguir usando mecanismos legacy equivalentes o redundantes, incluyendo:

url()->previous()
context_type / context_id
NavigationContext
reconstrucción manual por relaciones del modelo

tasks participa plenamente como origen contextual legítimo hacia orders

appointment y task deben seguir la misma convención técnica para abrir o crear orders relacionadas

en tablas embebidas o listados contextuales, cuando el contexto real de navegación sea la fila y no solo el contenedor, los links profundos a entidades relacionadas deben usar el trail de la fila

en tablas embebidas o relacionales, el link principal de la fila debe construir el trail desde el contexto del contenedor más el nodo de la fila

los links relacionados de esa misma fila deben reutilizar ese trail contextual ya construido y no reiniciar el contexto desde el contenedor ni desde mecanismos manuales

los nodos transitorios de formulario como create, edit, items.create o items.edit no deben persistir al volver a show después de store o update exitoso

las acciones cancelar dentro de formularios contextuales deben respetar el trail completo vigente y no volver por defecto al show estructural del módulo

Regla adicional consolidada para capacidades reusables embebidas:

cuando una tabla reusable o bloque embebido se consuma dentro de otro módulo, no debe reiniciar navegación desde una base local del módulo oferente si el flujo vigente sigue anclado en el contenedor consumidor

debe heredar el trailQuery contextual recibido y propagarlo hacia show, acciones por fila, cancelar y redirects relevantes

esto aplica especialmente a tablas reutilizables consumidas dentro de tabs o cards de relación

las decisiones internas de implementación pueden evolucionar, pero la convención estable es:

motor central reutilizable
builders compatibles por módulo
controller resuelve
vista consume
sin deducción desde historial
sin duplicación por módulo
sin serialización recursiva del trail dentro de nodos

<<END SECTION>>

CONVENCIONES DE NAVEGACIÓN CONTEXTUAL

Cuando exista navegación cruzada entre entidades relacionadas, no debe resetearse el contexto de origen sin necesidad funcional real.

Reglas:

si ya existe una capa central reutilizable para resolver contexto de navegación, no duplicar esa lógica en múltiples controllers

no hardcodear variantes divergentes por módulo si la regla de navegación ya existe a nivel reutilizable

la preservación de contexto UX debe aplicarse con criterio y solo cuando aporte continuidad real de uso

breadcrumb, retorno al origen y preservación de contexto deben resolverse de forma consistente entre módulos relacionados

cuando una navegación surja desde una tabla contextual, debe verificarse si el contexto real a preservar es:

solo el módulo contenedor

o el módulo contenedor más la entidad fila desde la cual se abre la entidad relacionada

esa decisión no debe dejarse implícita ni variar accidentalmente entre módulos

en tablas embebidas o relacionales, el link principal de la fila debe consolidar primero el contexto completo de esa fila

Regla adicional consolidada:

cuando la interacción no abandona el host pero sí requiere redirect posterior a una acción, debe distinguirse entre:

contexto externo de navegación
contexto interno del host

El contexto externo se preserva con trail.

El contexto interno del host puede preservarse con return_tab cuando exista una tab activa relevante que deba restaurarse luego de la acción.

<<SECTION: VERSIONADO DOCUMENTAL>>

VERSIONADO DOCUMENTAL

El versionado documental distingue nivel documento y nivel sección.

DOC_VERSION

Representa cambios globales del documento.

Debe incrementarse cuando ocurra alguno de estos casos:

reorganización estructural relevante

cambio doctrinal mayor

múltiples secciones impactadas en una misma revisión

cambio de alcance del documento

normalización general del formato

SECTION_VERSION

Representa cambios internos de una sección específica.

Debe incrementarse cuando cambie:

contenido

criterio operativo

redacción sustancial

estructura interna de esa sección

Reglas:

no incrementar versiones por cambios cosméticos mínimos

toda sección modificada debe conservar continuidad de nombre salvo necesidad real

si una sección reemplaza completamente otra con cambio conceptual fuerte, debe evaluarse nueva sección en lugar de renombre artificial

la numeración debe mantener formato homogéneo del proyecto

<<SECTION: FORMATO DE INTERCAMBIO AUTOMATIZADO>>
SECTION_VERSION: 00004

FORMATO DE INTERCAMBIO AUTOMATIZADO

Las actualizaciones documentales automatizadas deben respetar formato estricto compatible con el proceso PHP vigente del proyecto.

La doctrina documental debe reflejar exactamente el contrato real del automatizador.

No deben describirse formatos alternativos que el script no soporte.

Para describir ese formato dentro de documentos activos debe utilizarse una convención de metarrepresentación y no la sintaxis ejecutable real.

CONVENCIÓN DE METARREPRESENTACIÓN DEL FORMATO

TOKEN_APERTURA

representa el símbolo inicial utilizado por los delimitadores estructurales reales del sistema.

TOKEN_CIERRE

representa el símbolo final utilizado por los delimitadores estructurales reales del sistema.

CABECERA_DE_REEMPLAZO

representa la línea inicial que indica reemplazo de una sección existente en un documento identificado por DOC_SLUG.

CABECERA_DE_NUEVA_SECCION

representa la línea inicial que indica propuesta de alta de una nueva sección en un documento identificado por DOC_SLUG.

SECCION_ANCLA_DE_INSERCION

representa la instrucción obligatoria que indica después de qué sección existente debe insertarse una nueva sección.

Debe identificar la sección ancla por su nombre exacto.

INICIO_DE_SECCION

representa la apertura del bloque estructural de una sección usando el nombre exacto correspondiente y los delimitadores oficiales del sistema.

FIN_DE_SECCION

representa el cierre del bloque estructural de una sección usando los delimitadores oficiales del sistema.

SECTION_VERSION_LINE

representa la línea de versión interna obligatoria de la sección.

REGLAS OPERATIVAS

La salida automatizada real debe materializar estas piezas exactamente con la sintaxis oficial vigente del proyecto.

Dentro de documentos activos no deben escribirse delimitadores reales ni bloques ejecutables de ejemplo si pueden ser interpretados por el parser o por el visor documental como secciones válidas.

Toda respuesta generada en modo automatizado debe seguir el formato oficial real y no esta representación abstracta.

CONTRATO VÁLIDO DE REEMPLAZO

Toda operación de reemplazo debe incluir:

CABECERA_DE_REEMPLAZO
INICIO_DE_SECCION
SECTION_VERSION_LINE
contenido completo de la sección
FIN_DE_SECCION

No se permiten:

reemplazos parciales
fragmentos internos
cambios sin bloque completo
cambios sin SECTION_VERSION

CONTRATO VÁLIDO DE NUEVA SECCIÓN

Toda operación de nueva sección debe incluir obligatoriamente:

CABECERA_DE_NUEVA_SECCION
SECCION_ANCLA_DE_INSERCION
INICIO_DE_SECCION
SECTION_VERSION_LINE
contenido completo de la nueva sección
FIN_DE_SECCION

No se permiten:

altas sin sección ancla
altas con ubicación implícita
altas con posición inferida por contexto
altas sin SECTION_VERSION
altas con nombre de ancla aproximado

REGLAS OBLIGATORIAS DE SALIDA AUTOMATIZADA

devolver siempre bloques completos

usar nombre exacto de sección cuando ya exista

usar nombre exacto de sección ancla cuando se trate de una inserción

usar DOC_SLUG exacto

respetar delimitadores oficiales vigentes

incluir SECTION_VERSION con formato numérico homogéneo del proyecto

no mezclar explicaciones dentro del bloque automatizado

si el modo solicitado es automatizado, priorizar salida limpia y procesable

Cualquier desviación de formato debe considerarse salida no compatible con automatización.

<<END SECTION>>

<<SECTION: CRITERIOS PARA NUEVAS SECCIONES>>
SECTION_VERSION: 00003

CRITERIOS PARA NUEVAS SECCIONES

Una nueva sección solo debe crearse cuando aporte orden real al documento.

Criterios válidos:

el tema no encaja limpiamente en secciones existentes

el contenido tendrá continuidad futura

se evita mezclar responsabilidades distintas

el nuevo bloque mejora lectura, mantenimiento o auditoría

el tema posee identidad conceptual propia dentro del documento

No crear nuevas secciones cuando:

solo cambia una línea menor

el contenido puede integrarse limpiamente en una sección vigente

la separación agrega ruido estructural

se intenta reflejar una necesidad transitoria de una sola sesión

Reglas adicionales:

cada sección debe tender a una responsabilidad principal

el nombre debe ser claro, estable y descriptivo

evitar renombres cosméticos innecesarios

antes de crear una sección nueva, revisar si el contenido pertenece en realidad a otro documento del sistema

si el documento debe explicar el formato técnico de salida automatizada, debe hacerlo de manera descriptiva y no reproduciendo dentro de su propio cuerpo los delimitadores reales que usa el parser

si una nueva sección va a proponerse para inserción automatizada, debe definirse además una sección ancla exacta ya existente en el documento destino

la inserción automatizada de nuevas secciones no debe depender de ubicación implícita, proximidad temática ni interpretación manual posterior

si no existe una sección ancla clara y estable, primero debe revisarse si corresponde realmente crear la nueva sección o si conviene reorganizar el documento antes de automatizar la inserción
