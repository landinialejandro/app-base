<<SECTION: TIPO DE SISTEMA>>
SECTION_VERSION: 00001

TIPO DE SISTEMA

app-base es una plataforma SaaS multi-tenant desarrollada sobre Laravel.

Permite que múltiples empresas utilicen una misma aplicación manteniendo aislamiento lógico de datos y contexto operativo independiente para cada tenant.

El proyecto busca constituir una base ERP modular reutilizable, capaz de crecer por módulos sin perder coherencia estructural.

El sistema no debe entenderse como una suma de CRUDs aislados.

Debe entenderse como una plataforma con gobierno global, contexto por tenant, base de dominio, capa transversal y módulos funcionales integrados.

<<END SECTION>>

<<SECTION: PROPÓSITO GENERAL DEL PROYECTO>>
SECTION_VERSION: 00004

PROPÓSITO GENERAL DEL PROYECTO

app-base busca proveer una base estable para operar empresas dentro de un entorno SaaS multi-tenant.

No se trata solo de persistir entidades, sino de sostener una arquitectura común que permita:

ordenar operación
mantener aislamiento por tenant
centralizar gobierno transversal
integrar módulos sin pérdida de identidad
crecer sin rehacer el sistema

Principios asociados:

la complejidad operativa debe ser absorbida por el sistema

la experiencia debe tender a claridad

la automatización debe reducir fricción cuando corresponda

seguridad y trazabilidad son parte estructural

la operación real del negocio tiene prioridad sobre formalismos innecesarios

El sistema debe poder sostener múltiples frentes operativos sin degradarse en soluciones aisladas.

<<SECTION: ARQUITECTURA SAAS MULTI-TENANT>>
SECTION_VERSION: 00001

ARQUITECTURA SAAS MULTI-TENANT

El sistema utiliza una arquitectura SaaS multi-tenant.

Cada tenant representa una empresa o espacio de trabajo operativo independiente dentro de una misma aplicación.

El aislamiento entre empresas no se resuelve mediante aplicaciones separadas ni bases separadas por defecto, sino mediante una arquitectura común con separación lógica de datos y contexto operativo.

Principios estructurales:

cada tenant constituye una unidad operativa propia
el sistema mantiene una única plataforma común
la operación válida siempre ocurre dentro de un contexto tenant determinado
la pertenencia del usuario y el tenant activo forman parte de la lectura correcta del sistema

<<END SECTION>>

<<SECTION: MODELO DE AISLAMIENTO OPERATIVO>>
SECTION_VERSION: 00004

MODELO DE AISLAMIENTO OPERATIVO

Las entidades operativas del negocio pertenecen a un tenant y deben respetar aislamiento lógico entre empresas.

Ese aislamiento no es solo un detalle de persistencia.

Forma parte de la arquitectura conceptual del sistema.

Principios estables:

los datos de un tenant no deben mezclarse con los de otro
la operación visible y válida del usuario depende del tenant activo
las entidades globales del sistema no deben leerse como entidades del negocio
el contexto tenant no es opcional ni decorativo dentro de la operación interna

El aislamiento alcanza también a capas internas de soporte operativo que sostienen trazabilidad, agrupación contextual o ejecución de acciones dentro de módulos funcionales.

Este criterio incluye movimientos, operaciones internas y líneas operativas cuando participan como anclajes de ejecución o trazabilidad dentro del tenant.

Este documento reconoce el aislamiento como principio estructural general.

Su desarrollo técnico y sus reglas operativas concretas se resuelven en documentos especializados del proyecto.

<<SECTION: IDENTIDAD GLOBAL Y PERTENENCIA POR EMPRESA>>
SECTION_VERSION: 00006

IDENTIDAD GLOBAL Y PERTENENCIA POR EMPRESA

La relación principal del sistema es:

users
↓
memberships
↓
tenants

Lectura correcta:

users representa identidad global
memberships representa pertenencia y acceso por empresa
tenants representa el contexto operativo de cada empresa

Un usuario puede pertenecer a múltiples tenants.

La pertenencia a una empresa no se resuelve desde user como atributo directo único, sino mediante la capa intermedia de memberships.

Esto forma parte estable del modelo del sistema.

Aclaración consolidada:

membership puede vincular opcionalmente una ficha ampliada en parties.

Esa ficha ampliada pertenece al plano del negocio y no reemplaza la identidad global ni la pertenencia operativa.

Cuando la membership activa esté vinculada a una party, esa party debe leerse como representación operativa ampliada del colaborador dentro del tenant.

Regla:

la relación membership → party no convierte a party en identidad del sistema ni en entidad de acceso.

Cuando exista membership activa vinculada a party, la ficha ampliada debe mantener coherencia estructural con esa pertenencia activa.

Esta protección no convierte a parties en fuente de identidad ni de acceso.

Refuerza únicamente la separación entre:

identidad global
pertenencia operativa
ficha ampliada tenant-scoped

Aclaración estructural adicional:

cuando una party con relación employee tenga membership activa vinculada, esa ficha ampliada puede funcionar como centro contextual de lectura de actividad del colaborador dentro del tenant.

Esa lectura no altera el modelo de identidad.

No convierte a user en entidad del negocio.

No convierte a membership en módulo funcional.

No convierte a party en entidad de acceso.

No convierte a OperationalActivity en módulo funcional ordinario.

Separación vigente:

User = identidad global
Membership = pertenencia operativa
Party = ficha ampliada tenant-scoped cuando existe vínculo explícito
OperationalActivity = trazabilidad transversal tenant-scoped

Aclaración estructural sobre owner/admin:

owner es una condición estructural de gobierno del tenant resuelta desde membership.

admin es un rol delegable de gestión operativa fuerte dentro del tenant.

admin no equivale a owner.

Esta distinción refuerza la separación entre:

condición estructural de gobierno
rol delegable operativo
pertenencia activa al tenant
ficha ampliada opcional del negocio

El detalle operativo, visual y de seguridad de esa ficha ampliada y del gobierno owner/admin se define en los documentos especializados correspondientes.

<<END SECTION>>

TENANT ACTIVO COMO CONTEXTO OPERATIVO

El uso operativo del sistema ocurre dentro de un tenant activo.

Ese tenant activo constituye el contexto concreto desde el cual deben leerse las entidades operativas, las relaciones del negocio y las capacidades disponibles para la empresa actual.

El tenant activo define, a nivel estructural:

contexto de empresa vigente
marco operativo de los módulos del negocio
lectura correcta de las relaciones del ecosistema
ámbito de operación del usuario dentro del sistema
marco de gobierno interno del tenant

El tenant activo no debe interpretarse como un dato accesorio de sesión sin relevancia arquitectónica.

Es una pieza estructural del funcionamiento del sistema.

El frente tenant/profile pertenece a este plano de contexto y gobierno interno.

No debe leerse como módulo funcional ordinario del negocio.

Regla estructural:

el tenant activo organiza tanto operación como gobierno interno de la empresa activa, pero el detalle de autorización, permisos y restricciones owner/admin pertenece al documento especializado de seguridad.

<<SECTION: CAPAS FUNCIONALES DEL SISTEMA>>
SECTION_VERSION: 00028

CAPAS FUNCIONALES DEL SISTEMA

La arquitectura debe leerse por capas funcionales.

CAPA GLOBAL
Gobierno general de la plataforma.

CAPA TENANT
Contexto operativo de la empresa activa.

La capa tenant puede incluir capacidades propias del plano empresa que no constituyen módulos funcionales ordinarios.

Estas capacidades pueden ser tenant-scoped, evaluables por seguridad y visibles dentro de frentes del tenant sin incorporarse automáticamente al conjunto de módulos funcionales ordinarios.

CAPA DE GOBIERNO INTERNO DEL TENANT
Organiza pertenencia, actores internos y autoridad estructural dentro de la empresa activa.

users mantiene identidad global.

memberships resuelve pertenencia operativa.

owner se resuelve como condición estructural de gobierno mediante memberships.is_owner.

admin se resuelve como rol delegable de gestión operativa fuerte.

Lectura vigente:

owner gobierna estructuralmente el tenant

admin opera con autoridad amplia pero no reemplaza al owner

admin no gobierna la estructura superior del tenant

El frente tenant/profile pertenece a esta capa de gobierno interno.

No es módulo funcional ordinario del negocio.

Una membership puede vincularse opcionalmente con una ficha ampliada tenant-scoped en parties, sin reemplazar identidad ni acceso.

La complejidad del gobierno interno debe ser absorbida por el sistema mediante reglas backend centralizadas, sin trasladar lógica contractual crítica al usuario.

CAPA TRANSVERSAL
Responsabilidades compartidas que sostienen funcionamiento común entre módulos.

OperationalActivity pertenece a esta capa como capacidad transversal tenant-scoped de trazabilidad operativa.

No debe leerse como CRUD ordinario, módulo navegable principal ni responsabilidad propia de los módulos funcionales que producen actividad.

OperationalActivity registra actividad operativa emitida por módulos dueños luego de que estos autorizan, validan y ejecutan su operación propia.

No absorbe ownership funcional ni ejecuta la operación original.

La lectura general de OperationalActivity se resuelve como capacidad tenant no modular.

La lectura contextual de OperationalActivity puede publicarse hacia hosts funcionales autorizados sin convertir OperationalActivity en módulo funcional.

La lectura contextual puede operar en tres modalidades:

lectura directa del record host autorizado

lectura agregada mediante un conjunto explícito de records definido por el módulo dueño

lectura contextual especial mediante un criterio definido por el módulo dueño

Regla estructural:

OperationalActivity lee y presenta actividad persistida, pero no decide la frontera funcional de actividad agregada o contextual especial de cada módulo.

Esa frontera pertenece al módulo dueño.

Caso validado de lectura agregada:

projects define que la actividad contextual de un Project puede incluir el propio Project y sus Tasks.

Esa definición no queda hardcodeada dentro de OperationalActivity.

Caso validado de lectura contextual especial:

parties puede definir una lectura contextual especial para Party con relación employee y membership activa vinculada.

En ese caso, la actividad contextual puede incluir:

actividad directa de Party
actividad donde el user vinculado sea actor
actividad donde el user vinculado sea sujeto

Esta modalidad no convierte users, memberships ni party_roles en módulos funcionales nuevos.

Tampoco traslada ownership funcional hacia OperationalActivity.

BASE DEL DOMINIO
Anclajes conceptuales amplios del negocio.

MÓDULOS FUNCIONALES
Capacidades operativas visibles.

Reglas:

esta lectura debe prevalecer en decisiones arquitectónicas

tenant es contexto, no módulo funcional

tenant/profile es frente de gobierno interno, no módulo funcional ordinario

no toda capacidad tenant-scoped debe incorporarse como módulo funcional ordinario

las capacidades tenant no modulares pueden existir cuando la responsabilidad no sea módulo funcional, no tenga CRUD ordinario, no requiera navegación propia y no deba habilitarse o deshabilitarse como módulo del negocio

las capacidades tenant/transversales no modulares pueden publicar superficies controladas hacia hosts funcionales cuando exista contrato validado para ello

esa publicación no altera su naturaleza arquitectónica ni las convierte en módulos funcionales ordinarios

las necesidades de colaboradores internos deben resolverse primero en la capa tenant antes de crear módulos nuevos

la ficha ampliada en parties no altera el plano de identidad global ni el modelo de pertenencia operativa

las capacidades transversales pueden registrar, acompañar o sostener actividad operativa sin absorber ownership funcional de los módulos que producen esa actividad

La separación interna de parties refuerza esta lectura por capas:

parties.kind pertenece al núcleo estructural del contacto

party_roles.role expresa la relación del contacto con la empresa

Las reglas de autorización derivadas de esa relación pertenecen al documento especializado de seguridad.

JERARQUÍA GENERAL DEL SISTEMA

La arquitectura conceptual del sistema debe leerse en el siguiente orden:

app-base
↓
capa global
↓
tenant
↓
gobierno interno del tenant
↓
capa transversal
↓

la capa transversal arbitra o sostiene funcionamiento compartido
la base del dominio aporta anclajes relacionales amplios
los módulos funcionales ejecutan la operación del negocio

no interpretar el sistema como una primera línea plana de módulos al mismo nivel conceptual.

ENTIDADES DEL NEGOCIO

Las entidades del negocio representan la operación propia de cada tenant.

Casos actuales del proyecto:

parties
party_roles
projects
tasks
appointments
products
assets
orders
order_items
documents
document_items
document_sequences
inventory_movements
inventory_operations
operational_activities
attachments

Estas entidades pertenecen al ecosistema operativo de empresa y no a la infraestructura global del sistema.

No todas cumplen el mismo rol conceptual.

Pueden actuar como:

base del dominio
módulos autónomos
entidades hijas
capacidades transversales
soporte operativo interno
entidades complementarias de una base del dominio

Reglas de lectura relevantes:

users no pasa a ser entidad del negocio por participar en la operación interna de una empresa

users mantiene identidad global del sistema

la pertenencia operativa real se resuelve mediante memberships

si el ecosistema requiere un dataset reusable de colaboradores internos activos, esa necesidad pertenece al plano tenant y no implica crear una nueva entidad del negocio

inventory_movements expresa impacto unitario real sobre stock

inventory_operations agrupa contexto operativo cuando una misma acción requiere uno o múltiples movimientos

operational_activities registra actividad operativa real del sistema dentro del tenant activo como capacidad transversal de trazabilidad

operational_activities no representa una entidad hija de módulos funcionales particulares

operational_activities no reemplaza la lógica interna de los módulos funcionales

order_items y document_items conservan naturaleza de entidades hijas de sus módulos padre aunque puedan actuar como anclaje de trazabilidad o ejecución dentro de otros flujos

orders y documents comparten una doctrina base de lifecycle para cabeceras operativas

Aclaración estructural para OperationalActivity:

OperationalActivity registra actividad operativa sobre registros de módulos funcionales cuando el módulo dueño emite eventos operativos livianos luego de una operación autorizada, validada y ejecutada.

La integración con OperationalActivity no convierte a esos módulos en dependientes del log ni altera su ownership funcional.

OperationalActivity puede exponer lectura contextual directa sobre un record host autorizado, lectura contextual agregada sobre un conjunto explícito de records declarado por el módulo dueño o lectura contextual especial mediante un criterio definido por el módulo dueño.

Regla:

la frontera funcional de una lectura agregada o contextual especial pertenece al módulo dueño.

Caso validado de lectura agregada:

Projects puede declarar Project + Tasks como dataset de actividad contextual agregada del proyecto.

Caso validado de lectura contextual especial:

Parties puede declarar un criterio contextual especial para Party employee con membership activa vinculada.

Ese contexto puede permitir leer actividad directa de Party y actividad asociada al user vinculado como actor o sujeto.

Esto no convierte User en entidad del negocio.

No convierte Membership en módulo funcional.

No convierte party_roles en módulo funcional propio.

No convierte OperationalActivity en dueño de esa relación.

Aclaración estructural para inventory:

inventory_operations representa la acción operativa agrupada frente a OperationalActivity.

inventory_movements conserva la trazabilidad fina, unitaria y atómica del stock.

Regla:

OperationalActivity no debe duplicar como actividad principal cada inventory_movement cuando el evento operativo relevante está representado por InventoryOperation.

Aclaración estructural para parties:

parties representa contactos o actores del negocio dentro del tenant.

parties.kind representa únicamente tipo de contacto.

Valores vigentes:

person
company

party_roles expresa la relación del contacto con la empresa.

Relaciones vigentes:

customer
supplier
employee
other

Regla:

employee no es tipo de contacto ni naturaleza de party.

Employee es una relación operativa dentro de party_roles.

La incidencia de party_roles.role en seguridad se define en el documento especializado de seguridad.

Aclaración sobre membership → party:

membership puede vincular opcionalmente una ficha ampliada en parties.

Esta relación:

es opcional
es tenant-scoped
no modifica la estructura base del sistema
no reemplaza la identidad global del usuario
no convierte a party en entidad de acceso

Cuando exista vínculo con membership activa, party representa al colaborador interno en el plano operativo del negocio.

Cuando esa party tenga relación employee y membership activa vinculada, puede funcionar además como centro contextual de lectura de actividad del colaborador, siempre mediante contrato específico del módulo dueño.

Este documento reconoce la posición estructural de estas entidades.

El detalle operativo, visual, de navegación o seguridad debe resolverse en los documentos especializados correspondientes.

DISTINCIÓN ENTRE SISTEMA Y NEGOCIO

La distinción entre entidades del sistema y entidades del negocio es estructural.

Las entidades del sistema sostienen identidad, acceso, onboarding y gobierno general.

Las entidades del negocio sostienen la operación propia de cada tenant.

Regla conceptual:

no debe resolverse una necesidad del negocio trasladándola arbitrariamente a entidades del sistema si esa decisión rompe la separación entre plataforma y operación.

Regla complementaria:

tampoco debe sobredimensionarse el dominio del negocio hasta absorber piezas cuya función real es global o transversal a la plataforma.

CLASIFICACIÓN CONCEPTUAL DE MÓDULOS

Los módulos y capacidades del ecosistema no cumplen todos el mismo rol.

Clasificación doctrinal vigente:

MÓDULOS NUCLEARES

Representan frentes funcionales principales del tenant.

Ejemplos actuales:

orders
documents
inventory
projects
tasks
parties
products
assets
appointments

MÓDULOS TRANSVERSALES

Aportan capacidades reutilizables consumibles por múltiples frentes sin redefinir el núcleo de cada módulo.

Ejemplo actual:

attachments

CAPACIDADES TRANSVERSALES DE SOPORTE

Registran, acompañan o sostienen funcionamiento común del ecosistema sin constituir necesariamente módulo navegable principal ni CRUD ordinario.

Caso vigente:

OperationalActivity

OperationalActivity se clasifica como capacidad transversal tenant-scoped de trazabilidad operativa.

Reglas:

no abre navegación principal por mera existencia técnica

no es CRUD ordinario

no absorbe ownership funcional de los módulos que emiten eventos

no debe registrar actividad solo por existir CRUD

puede ofrecer lectura contextual directa, agregada o contextual especial cuando exista contrato válido

Regla:

OperationalActivity no define por sí mismo qué records componen la actividad agregada de un módulo ni qué criterio contextual especial corresponde aplicar.

La participación de un módulo en OperationalActivity debe definirse explícitamente y justificarse por valor operativo real.

CAPACIDADES TENANT NO MODULARES

Representan capacidades del plano tenant que pueden ser evaluadas por seguridad sin constituir módulos funcionales ordinarios.

Caso vigente:

lectura general de OperationalActivity mediante capability específica.

Reglas:

no se incorporan automáticamente como módulos funcionales ordinarios

no se leen como módulos navegables

no poseen CRUD completo

no pasan por habilitación modular ordinaria

pueden ser evaluadas por la capa central de seguridad

pueden tener permisos asignables cuando exista contrato explícito

CAPACIDADES TENANT/TRANSVERSALES NO MODULARES CON SURFACE PUBLICADA

Representan capacidades tenant-scoped o transversales que publican lectura o información controlada hacia hosts funcionales sin convertirse en módulos ordinarios.

Caso vigente:

OperationalActivity publica lectura contextual mediante surface tenant/transversal.

Reglas:

no se incorporan como módulos funcionales por publicar una surface

no generan navegación principal

no poseen CRUD ordinario

no pasan por TenantModuleAccess como módulo funcional

conservan ownership en el plano tenant/transversal

Aclaraciones vigentes:

projects puede declarar lectura contextual agregada de actividad sobre Project + Tasks mediante activity_record_set

parties puede declarar lectura contextual especial de actividad para Party employee con membership activa vinculada mediante activity_context

inventory es módulo nuclear funcional y, frente a OperationalActivity, el ancla operativa vigente es InventoryOperation

parties es módulo base del dominio y party_roles no constituye módulo funcional independiente

Regla:

la clasificación funcional de un módulo no autoriza excepciones de seguridad ni navegación automática.

La navegación, visualización y lectura autorizada se resuelven en los documentos especializados correspondientes.

<<SECTION: ARQUITECTURA MODULAR DEL ECOSISTEMA>>
SECTION_VERSION: 00030

ARQUITECTURA MODULAR DEL ECOSISTEMA

La arquitectura modular del proyecto debe leerse como un ecosistema reusable y desacoplado.

Los módulos no deben relacionarse como cajas cerradas ni como núcleos invadibles.

La lectura correcta distingue tres planos:

núcleo interno del módulo
puerta de enlace modular oficial
consumo externo

NÚCLEO INTERNO

Corresponde a la lógica y operación propia del módulo.

No debe reabrirse desde consumidores.

PUERTA DE ENLACE MODULAR OFICIAL

Corresponde a la forma publicada para integrarse con el ecosistema.

Puede expresarse mediante:

acciones
embebidos
payloads
surfaces
row actions
resolvers
eventos operativos livianos cuando corresponda
record sets explícitos cuando corresponda lectura contextual agregada
criterios contextuales especiales cuando corresponda lectura contextual especial

Debe representar la capacidad completa del módulo oferente.

CONSUMO EXTERNO

El consumo sigue la secuencia:

consumer solicita
módulo oferente resuelve
componente representa
host monta

El consumidor no debe absorber decisiones del módulo oferente.

HUB VISUAL

Un módulo puede alojar interacción de otro sin absorber su lógica.

Regla:

el host conserva superficie visual

el módulo invitado conserva lógica operativa

TRAZABILIDAD TRANSVERSAL

Cuando un módulo funcional completa una acción propia autorizada, puede emitir un evento operativo liviano para trazabilidad.

La lectura arquitectónica correcta es:

módulo dueño autoriza
módulo dueño valida
módulo dueño ejecuta
módulo emite evento operativo liviano
OperationalActivity registra si corresponde

OperationalActivity no reemplaza el núcleo del módulo dueño ni se convierte en intérprete principal de su dominio.

La lógica de sanidad, normalización y persistencia del registro operativo pertenece a la capacidad transversal.

El significado operativo primario de la acción sigue perteneciendo al módulo dueño.

LECTURA CONTEXTUAL DE ACTIVIDAD

OperationalActivity puede leer actividad contextual directa, agregada o contextual especial.

Lectura directa:

record host autorizado
↓
lectura de actividad asociada al host

Lectura agregada:

módulo dueño
↓
conjunto explícito de records
↓
lectura contextual agregada

Lectura contextual especial:

módulo dueño
↓
criterio contextual especial
↓
lectura contextual ampliada

Regla:

OperationalActivity no decide la frontera funcional agregada ni contextual especial.

No debe conocer internamente relaciones propias de módulos concretos para ampliar el dataset.

Caso validado de lectura agregada:

Projects declara Project + Tasks como conjunto de actividad contextual visible desde projects.show.

Esta decisión pertenece al módulo Projects y no a OperationalActivity.

Caso validado de lectura contextual especial:

Parties declara el criterio de Party employee con membership activa vinculada.

Esta decisión pertenece al módulo Parties y no a OperationalActivity.

SURFACES TENANT O TRANSVERSALES NO MODULARES

Una capacidad tenant-scoped o transversal no modular puede publicar surfaces controladas hacia hosts funcionales sin incorporarse como módulo funcional ordinario.

Esta vía corresponde cuando la capacidad:

no es módulo funcional ordinario
no posee CRUD ordinario
no requiere navegación principal
no se habilita o deshabilita como módulo del negocio
debe ofrecer lectura o información controlada a hosts autorizados

Caso validado:

OperationalActivity publica lectura contextual mediante una surface tenant/transversal.

Esta publicación no convierte OperationalActivity en módulo funcional, no lo incorpora al conjunto de módulos ordinarios y no crea navegación principal.

Aclaración consolidada:

los módulos pueden publicar metadata estructural propia.

El flujo correcto es:

módulo define
catálogo o contrato provee metadata
registry normaliza
host consume sin reinterpretar

Cuando existan surfaces tenant/transversales no modulares, el sistema puede integrar ese origen sin degradar la frontera entre módulos funcionales ordinarios y capacidades transversales.

Regla:

la arquitectura modular debe permitir integración sin pérdida de ownership funcional.

BASE DEL DOMINIO

La base del dominio agrupa piezas conceptuales que sostienen múltiples relaciones relevantes dentro del ecosistema operativo.

Caso actual principal:

parties

Su función no se limita a registrar un dato aislado.

Debe leerse como anclaje relacional estable para múltiples frentes del negocio.

parties también puede publicar capacidades oficiales para que otros módulos consuman relaciones compactas o embebidas sin reconstruir localmente su lógica.

Regla estructural:

que una pieza pertenezca a la base del dominio no la convierte en infraestructura global del sistema.

Sigue perteneciendo al ecosistema operativo del tenant, pero con un rol conceptual más amplio que el de un módulo operativo puntual.

Regla vigente para parties:

parties debe mantenerse como núcleo mínimo del actor o contacto del negocio.

La expansión de su complejidad debe tender a entidades hijas o complementarias y no a sobrecargar el núcleo.

Separación consolidada:

parties.kind = tipo de contacto
party_roles.role = relación con la empresa

Valores vigentes de parties.kind:

person
company

Relaciones vigentes de party_roles.role:

customer
supplier
employee
other

Regla:

employee no pertenece a parties.kind.

Employee expresa relación operativa con la empresa dentro de party_roles.

Aclaración sobre colaboradores internos:

los usuarios internos activos del tenant pertenecen al plano tenant y a memberships como pertenencia operativa real.

membership puede vincularse opcionalmente con party como ficha ampliada tenant-scoped, pero esa relación no convierte a party en identidad global ni en entidad de acceso.

Por lo tanto, esa necesidad no debe reinterpretarse como expansión de users ni como apertura de un módulo users dentro de la base del dominio.

MODELO GENERAL DE CRECIMIENTO ARQUITECTÓNICO

El sistema debe crecer mediante módulos con identidad clara, contratos explícitos y responsabilidades separadas.

respetar multi-tenancy
preservar distinción sistema/negocio
mantener identidad conceptual
evitar duplicaciones
consumir capacidades existentes antes de crear variantes

no reabrir el núcleo de otro módulo arbitrariamente

las necesidades de colaboradores internos deben resolverse en la capa tenant

las capacidades maduras deben converger hacia contratos oficiales

los acoples externos deben canalizarse mediante puertas contractuales

si una entidad hija crece, debe fortalecerse antes de promoverse a módulo

los frentes transaccionales pueden incorporar agrupación contextual cuando mejore trazabilidad

Regla:

crecer no implica mezclar responsabilidades.

La dirección sigue siendo modular y con fronteras explícitas.

CONTRATO MODULAR COMO DIRECCIÓN DE CRECIMIENTO

El crecimiento correcto del sistema debe priorizar evolución contractual y claridad de fronteras antes que expansión desordenada de código.

Cuando una necesidad nueva aparezca en un módulo existente, la primera lectura correcta no es invadir consumidores ni dispersar lógica transversal.

La salida preferente es fortalecer el contrato propio del módulo y ordenar internamente su dominio.

Reglas:

una mejora interna no debe romper contratos externos estables

la complejidad nueva debe resolverse primero dentro del módulo dueño del problema

los consumidores deben depender de capacidades publicadas y no de detalles internos variables

un contrato modular válido debe permitir integrar sin borrar ownership funcional

Caso vigente consolidado:

orders continúa integrándose con inventory mediante servicios y puertas contractuales explícitas, sin absorber lógica interna del frente stock.

Regla final:

la dirección de crecimiento aprobada del proyecto continúa siendo modular, reusable y contractualmente explícita.
