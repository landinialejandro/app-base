OBJETIVO

Este documento define el mapa vigente de navegación, recorridos funcionales y relaciones visibles entre módulos de app-base.

Su función es:

describir recorridos reales del sistema
ordenar navegación principal y contextual
documentar accesos entre módulos
evitar flujos inconsistentes
guiar futuras ampliaciones de navegación

No define:

seguridad doctrinal
modelo conceptual completo de entidades
contratos modulares internos
detalle visual de componentes

Esos aspectos se resuelven en otros documentos.

Regla consolidada:

este documento describe navegación real o navegación objetivo ya validada

no debe cargarse con teoría arquitectónica general que corresponda a otros documentos

<<SECTION: REGLA GENERAL>>
SECTION_VERSION: 00021

REGLA GENERAL

Toda navegación del sistema debe responder a una necesidad funcional real.

No debe crearse navegación por simetría visual, comodidad técnica o existencia de tablas auxiliares.

Reglas obligatorias:

navegación principal expresa estructura estable

navegación contextual expresa relaciones reales de trabajo

navegación pública externa expresa acceso externo controlado y no panel interno

dataset interno no equivale a módulo navegable

host visual no equivale a ownership funcional

universo semántico no equivale a ownership funcional

entrada pública externa no equivale a navegación interna del tenant

selector externo no equivale a selector interno de tenants

salida externa no equivale a logout interno

token temporal de selección no equivale a sesión interna

sesión externa namespaced no equivale a sesión interna

sesión externa namespaced no equivale a tenant activo interno

middleware externo pasivo no equivale a ruta privada

middleware externo pasivo no equivale a autorización obligatoria

selección externa de tienda no equivale a operación comercial habilitada

redirección a tienda pública no equivale a operación comercial habilitada

catálogo publicado no equivale a operación comercial habilitada

administración interna de tienda no equivale a tienda pública customer

customer externo no equivale a administrador interno de tienda

administración interna de shops pertenece al plano interno

shops no pertenece al plano externo customer

self_service_sales no administra internamente shops

el trail preserva contexto entre pantallas relacionadas

la UI no debe prometer accesos no operables

Regla adicional consolidada:

si una necesidad puede resolverse como pieza reusable del plano tenant, como capacidad publicada por un módulo existente o como universo semántico contextual, esa salida debe evaluarse antes de abrir una nueva ruta mental en el sistema mediante navegación adicional.

Regla adicional para actividad operativa:

la existencia de operational_activities no habilita por sí sola navegación principal nueva.

OperationalActivity debe leerse primero como capacidad transversal de soporte.

Cualquier lectura visual debe responder a una necesidad funcional real, autorización explícita y superficie definida.

La integración backend de nuevos módulos con OperationalActivity no modifica por sí misma la navegación principal, las show pages, las tabs ni los recorridos visibles.

Regla:

no crear tab, vista general, menú ni acceso de actividad sin contrato de lectura autorizado y decisión visual específica.

Regla adicional para lectura contextual:

una tab Actividad dentro de un host autorizado no crea navegación principal de OperationalActivity.

La navegación visible permanece dentro del show del host.

La lectura contextual debe limitarse a la actividad asociada al registro host autorizado, al conjunto explícito de records definido por el módulo dueño cuando exista activity_record_set o al criterio contextual especial definido por el módulo dueño cuando exista activity_context.

Si no existe activity_record_set ni activity_context, la lectura contextual es directa sobre el record host.

Si existe activity_record_set, la lectura contextual es agregada sobre el conjunto definido por el módulo dueño.

Si existe activity_context, la lectura contextual se adapta al criterio especial definido por el módulo dueño.

Regla:

la existencia de una tab Actividad no implica por sí misma lectura agregada ni lectura contextual especial.

La lectura agregada debe estar declarada explícitamente por el módulo dueño mediante activity_record_set.

La lectura contextual especial debe estar declarada explícitamente por el módulo dueño mediante activity_context.

Regla adicional para capas satélite productivas:

una capa satélite productiva puede tener entrada visual propia, dashboard operativo especializado, experiencia enfocada o universo semántico propio sin convertirse automáticamente en navegación CRUD ordinaria.

Caso doctrinal vigente:

Servicios y Mantenimiento puede requerir dashboard operativo, orden electrónica liviana o paquete de trabajo como frentes especializados.

Estos frentes no deben crear navegación por simetría ni duplicar módulos existentes.

Deben orquestar recorridos reales entre Orders, Assets, Inventory, Products, Appointments, Tasks, Documents, Attachments, Parties y OperationalActivity respetando ownership funcional.

Caso operativo vigente:

Servicio y mantenimiento cuenta con un universo semántico propio:

/service

Lectura:

/service representa el frente Servicio y mantenimiento

/service no es CRUD ordinario

/service no es módulo dueño de Orders

/service expresa contexto funcional, navegación especializada y acceso operativo al frente satélite

Orders conserva ownership de las órdenes

Reglas:

un frente especializado no autoriza por sí mismo acciones sobre módulos participantes

la autorización debe seguir perteneciendo al módulo dueño o a la capacidad central correspondiente

la existencia de entrada visual no convierte a la capa satélite en CRUD ordinaria

una entrada de Servicio y mantenimiento no debe duplicar la navegación estructural de Orders si solo está filtrando o contextualizando órdenes de servicio

un universo semántico debe conservar coherencia de trail, breadcrumbs y redirecciones físicas cuando el usuario entra por ese universo

Regla adicional para capas satélite comerciales autogestivas:

una capa satélite comercial autogestiva puede tener entrada pública externa, selector externo, sesión externa mínima namespaced, middleware externo pasivo, tienda tenant-scoped, salida externa y consumo de catálogo publicado sin convertirse automáticamente en navegación interna ordinaria del tenant ni en CRUD externo customer.

Lectura del recorrido:

ResolveSelfServiceExternalCustomer puede compartir payload externo en el request cuando existe sesión externa válida para el tenant público de la ruta.

Si la sesión externa corresponde a otro tenant, /shop/{tenant:slug} se muestra como visitante.

Si no existe sesión externa, /shop/{tenant:slug} se muestra como visitante.

POST /tienda/salir limpia la sesión externa self_service_sales.*.

POST /tienda/salir devuelve al acceso externo sin cerrar sesión interna.

un token temporal de selección no debe leerse como sesión interna

marcar used_at no equivale a crear sesión interna

la sesión externa mínima debe leerse como self_service_sales.*, no como session('tenant_id')

ResolveSelfServiceExternalCustomer no convierte /shop/{tenant:slug} en ruta privada

ResolveSelfServiceExternalCustomer no exige login externo

ResolveSelfServiceExternalCustomer no bloquea visitantes públicos

ResolveSelfServiceExternalCustomer no redirige por ausencia de sesión externa

seleccionar tienda no habilita por sí mismo operación comercial

mostrar customer externo activo no habilita por sí mismo operación comercial

la navegación pública externa se mantiene accesible sin login externo obligatorio

REGLAS DE LECTURA

Una relación en este documento significa acceso, recorrido o vínculo visible para el usuario dentro del sistema.

No implica automáticamente:

foreign key obligatoria
permiso concedido
implementación ya terminada
visibilidad universal
módulo siempre habilitado

Toda navegación real sigue condicionada por:

módulo habilitado para el tenant
permiso efectivo
estado funcional del recurso
flujo operativo aplicable

Tipos de navegación relevantes:

estructural
contextual
embebida
derivada

NAVEGACIÓN ESTRUCTURAL

Recorrido estable propio del módulo.

Ejemplo:

menú principal → index → show → edit

NAVEGACIÓN CONTEXTUAL

Recorrido originado desde otro módulo conservando contexto.

Ejemplo:

appointment.show → crear order contextual

NAVEGACIÓN EMBEBIDA

Consumo de tablas, bloques o relaciones dentro de otra pantalla.

Ejemplo:

orders embebidas dentro de asset.show

NAVEGACIÓN DERIVADA

Flujo donde una acción en un módulo genera o abre otro módulo como consecuencia operativa.

Ejemplo:

orders.show → crear document

Regla:

cuando existan varias rutas posibles, debe priorizarse la más consistente con NavigationTrail y con el flujo real del usuario
<<END SECTION>>

NAVEGACIÓN PRINCIPAL TENANT

La navegación principal tenant expresa los módulos funcionales visibles de la empresa activa y los frentes operativos especializados aprobados.

Debe responder al contrato real del sistema y no a mera existencia técnica de código.

Casos funcionales actuales:

dashboard

projects

tasks

appointments

parties

products

assets

orders

documents

inventory

shops

Frentes operativos especializados vigentes:

service

La aparición real de cada entrada depende de:

módulo o frente habilitado

permiso efectivo

estado operativo cuando corresponda

Regla adicional consolidada:

users no debe aparecer por esta necesidad como módulo navegable autónomo dentro del tenant.

La gestión interna de usuarios, pertenencias, roles y permisos continúa anclada al frente tenant/profile/accesses y no a una navegación principal users separada.

Regla adicional consolidada:

tenant/profile es frente de gobierno interno del tenant.

No es módulo funcional ordinario del negocio.

Su acceso visible puede estar disponible para owner y admin según autorización central.

Lectura vigente:

owner puede acceder a tenant/profile como autoridad estructural de gobierno

admin puede acceder a tenant/profile con restricciones de gobierno

sales/operator no deben ver ni acceder a ese frente si no poseen autorización efectiva

La autorización de acceso debe resolverse por la pieza central correspondiente y no por condición visual local.

Caso vigente:

TenantProfileAccess::canViewProfile

Regla adicional consolidada:

la navegación principal puede incorporar iconografía por módulo o frente para mejorar lectura visual.

Esa iconografía debe provenir de metadata modular o definición del frente y no definirse localmente de forma dispersa en la vista.

Antes de asignar un ícono de módulo debe verificarse que exista el componente correspondiente en:

resources/views/components/icons

Regla:

no usar íconos inexistentes en ModuleCatalog.

Un ícono inexistente puede romper dashboard/navbar cuando se renderiza mediante x-dynamic-component.

Aclaración vigente para parties:

parties mantiene su lugar como módulo visible de contactos.

El alta de contacto puede resolverse mediante experiencia progresiva dentro de la misma pantalla, sin crear una navegación nueva.

NAVEGACIÓN CONTEXTUAL ENTRE MÓDULOS

Los recorridos entre módulos deben respetar identidad propia, contexto funcional y contratos vigentes.

Relaciones vigentes destacadas:

orders consume capacidades oficiales de inventory sin absorber su lógica interna

orders puede actuar como host visual para inventory, documents, attachments, tasks y appointments cuando existan relaciones o surfaces válidas

products consume lectura contextual de inventory mediante saldos e historial

products puede ofrecer selección y validación de productos para líneas operativas y documentales mediante capacidad propia

documents puede contextualizar trazabilidad de inventory cuando exista vínculo documental

documents puede actuar como host visual de acciones oficiales de inventory sobre document_items

tasks puede crear o vincular orders mediante contexto transitorio de navegación y persistencia real en tasks.order_id

appointments puede crear o vincular orders mediante contexto transitorio de navegación y persistencia real en appointments.order_id

orders puede derivar documents mediante documents.order_id

assets y parties mantienen sus relaciones propias dentro del ecosistema general

tenant/profile actúa como frente de gobierno interno del tenant para usuarios, memberships, roles y permisos

service puede actuar como universo semántico de acceso contextual a órdenes de servicio sin absorber ownership de Orders

Regla de frontera:

que un módulo o universo hospede visualmente una capacidad ajena no lo convierte en dueño de su lógica operativa.

El módulo oferente conserva reglas, validaciones y evolución interna.

Aduana funcional:

cuando una navegación, link, surface o acción contextual involucra otro módulo, debe usarse una frontera explícita del módulo oferente.

Reglas:

el host no debe reconstruir lógica interna del oferente

el oferente no debe interpretar modelos internos del host consumidor

el host entrega contexto o recurso compatible

el oferente arma payload, link, acción o surface según su contrato

Aclaración vigente para tenant/profile:

tenant/profile no abre un módulo users.

La gestión de usuarios, memberships, roles y permisos continúa anclada al frente tenant/profile/accesses.

Owner y admin pueden visualizar ese frente cuando la autorización central lo permita.

La diferencia entre ambos no se resuelve creando navegaciones separadas, sino aplicando restricciones dentro del mismo frente.

Lectura vigente:

owner gobierna

admin opera fuerte con restricciones

Aclaración vigente para edición de funciones:

la edición de funciones de una membership puede resolverse como micro-interacción modal dentro de la pantalla de roles/accesos.

Patrón visible:

acción por fila

apertura de modal

edición del conjunto de funciones

cancelar o guardar

recarga natural tras submit cuando corresponda

Esta interacción no constituye navegación nueva ni cambio de recurso.

Aclaración vigente para inventory:

la capa inventory_operations no modifica por sí misma los recorridos principales entre módulos.

Su función es enriquecer el contexto interno y la trazabilidad del frente inventory.

Frente a OperationalActivity, InventoryOperation actúa como ancla operativa agrupada.

InventoryMovement conserva la trazabilidad fina de stock y no genera por sí mismo navegación contextual adicional en el log transversal.

InventoryOperation no tiene todavía show canónico confirmado para navegación desde el log.

Aclaración vigente para documents:

cuando documents hospeda operaciones de inventory por línea documental, la navegación sigue perteneciendo al flujo documental anfitrión salvo que el usuario decida abrir una surface canónica de inventory.

Aclaración vigente para lifecycle de cabeceras:

la creación de una cabecera en orders o documents no implica por sí sola que el recurso esté listo para operar.

La cabecera debe respetar lifecycle, líneas, validaciones y acciones disponibles del módulo dueño.

Aclaración vigente para Orders embebidas:

OrderSurfaceService publica orders.related.embedded como surface embebida única para assets.show y parties.show.

La surface usa configuración contextual para decidir columnas visibles.

Reglas vigentes:

en assets.show, Orders debe mostrar Contraparte y ocultar Activo porque el host ya representa el activo

en parties.show, Orders debe ocultar Contraparte y mostrar Activo porque el host ya representa el contacto

en orders.index, Orders puede mostrar Contraparte y Activo como datos operativos principales de lectura

Contraparte se resuelve desde:

Order::displayCounterpartyName()

Activo se resuelve desde:

Order::displayAssetReference()

La condición manual o gestionada se resuelve desde helpers del modelo dueño y no desde el Blade.

<<SECTION: REGLAS DEL TRAIL>>

REGLAS DEL TRAIL

El trail representa contexto externo entre recursos y pantallas.

No reemplaza estados internos del host.

Reglas:

cada salto relevante debe poder reconstruir origen útil
no usar trail como contenedor ambiguo de todo estado UI
preservar breadcrumb comprensible
evitar cadenas innecesarias o redundantes
el destino final debe coincidir con la intención original del usuario

Cuando además exista contexto interno del host, debe complementarse con return_tab u otro mecanismo explícito equivalente.

Distinción doctrinal:

trail = contexto externo entre recursos
return_tab = contexto interno de una pantalla anfitriona

Caso vigente:

acción disparada desde una tab de orders.show puede requerir ambos:
trail para volver a la order
return_tab para restaurar el panel correcto

Regla adicional:

las transiciones especiales de estado no deben confundirse con edición ordinaria.

Si una acción cambia estado terminal o readonly hacia uno operable, debe tratarse como acción específica del flujo y no como simple navegación a edit.

Caso vigente:

reapertura closed → approved en orders.

Regla adicional sobre tabs host:

return_tab debe usarse para conservar el contexto visual interno de una show page cuando el host posea tabs principales.

El host es responsable de validar que la key recibida exista dentro de sus tabs disponibles.

Si la key no existe, debe resolver un fallback seguro sin romper la navegación.

CRITERIO FINAL

La navegación correcta del sistema no es la que muestra más enlaces.

Es la que permite al usuario llegar de forma clara, segura y coherente a la acción que necesita.

Una relación visible entre módulos se justifica cuando:

mejora continuidad operativa
reduce fricción real
respeta permisos
respeta módulos habilitados
mantiene contexto útil
no duplica caminos innecesarios

Si una relación existe técnicamente pero no cumple estos criterios, no debe priorizarse como navegación visible.
