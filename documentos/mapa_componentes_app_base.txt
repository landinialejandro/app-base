<<SECTION: COMPONENTES BLADE GLOBALES>>

COMPONENTES BLADE GLOBALES

Ubicación:

resources/views/components/

Responsabilidad:

componentes reutilizables en todo el sistema sin pertenencia a un módulo específico.

Ejemplos vigentes:

x-page
x-page-header
x-breadcrumb
x-card
x-modal
x-dev-component-version

Reglas:

deben ser genéricos
no deben contener lógica de negocio
no deben depender de un módulo específico
deben ser reutilizables en cualquier contexto

Regla adicional:

si un componente comienza a depender de un módulo específico, debe migrarse al módulo correspondiente.

Aclaración vigente sobre x-modal:

x-modal es componente global válido para modales operativos e informativos cuando el contenido ya llega preparado por la capa correspondiente.

Casos vigentes:

detalle de cambios en actividad operativa

edición de funciones de una membership desde tenant/profile

Regla:

x-modal no decide autorización, no consulta datos y no interpreta reglas de dominio.

Solo representa una interacción visual reutilizable.

En el caso de edición de funciones tenant, el modal representa la micro-interacción visual con datos y permisos ya preparados por backend.

No decide reglas owner/admin ni resuelve autoridad.

Aclaración vigente sobre x-dev-component-version:

x-dev-component-version es un componente global de ayuda visual y control interno de versiones de Blades.

Responsabilidad:

identificar Blades intervenidos
mostrar versión visual del componente, partial, card o vista
facilitar auditorías posteriores
favorecer continuidad entre cambios visuales sucesivos

Props vigentes:

name
version
align

Reglas:

debe usarse en Blades nuevos con entidad visual propia

debe actualizarse en Blades modificados que ya lo tengan

debe evaluarse su incorporación en Blades existentes intervenidos que todavía no lo tengan

no debe usarse con prop file

no debe ubicarse dentro de tablas, tbody, formularios, loops ni estructuras repetitivas

no reemplaza versionado documental ni control de código

no define reglas de negocio, permisos ni navegación

Caso vigente:

los partials compactos de ProductOperationalSummary no requieren x-dev-component-version porque funcionan como fragmentos internos de summary_items, sin entidad visual independiente mayor.

Regla de validación runtime:

no debe utilizarse un componente Blade si no existe realmente en resources/views/components o como clase correspondiente.

Caso detectado:

x-empty-state no existe actualmente como componente disponible.

Por lo tanto, no debe listarse ni usarse como componente global vigente.

Si una vista requiere estado vacío y no existe componente específico, debe resolverlo con componentes disponibles o markup inline consistente.

Caso vigente:

la tab Actividad de tenant/profile resolvió su estado vacío inline con detail-block.

Regla adicional:

no deben promoverse partials modulares a componentes globales si la necesidad no es transversal.

Caso vigente:

los partials de ProductOperationalSummary permanecen dentro de resources/views/products/operational-summary/ y no se convierten en componentes globales.

Regla adicional para componentes linked:

un componente global o reusable de vínculo no debe reemplazar el dato textual propio de un módulo cuando el vínculo gestionado no existe.

Caso vigente:

Orders y Documents pueden conservar counterparty_name aunque party_id sea null.

Reglas:

si existe Party vinculada, puede representarse el vínculo gestionado cuando corresponda

si no existe Party vinculada pero existe counterparty_name, debe representarse la contraparte textual propia

si no existe ni vínculo ni snapshot, recién corresponde estado vacío o fallback visual

un componente linked no debe inventar enlace hacia Party

un componente linked no debe ocultar el snapshot propio del registro

PartyLinked representa únicamente el vínculo gestionado cuando existe Party vinculada y corresponde mostrar enlace.

No debe usarse como fuente principal del texto visible de contraparte cuando Orders o Documents ofrecen helper oficial de lectura.

Casos vigentes:

Order::displayCounterpartyName()

Document::displayCounterpartyName()

Regla adicional para componentes linked dentro de tablas/listados:

los componentes linked pueden utilizarse dentro de tablas/listados cuando el registro dueño posee un vínculo gestionado opcional hacia otro módulo.

Caso validado:

orders.partials.table consume parties.components.linked-party

orders.partials.table consume assets.components.linked-asset

Reglas:

el componente linked representa el vínculo gestionado cuando existe y corresponde

el texto visible puede venir resuelto por el módulo dueño

el componente linked no reemplaza la referencia textual propia del registro

si no hay vínculo gestionado, debe mostrarse texto plano propio

el componente linked no debe inventar enlaces

el componente linked no debe ocultar snapshots o referencias propias del registro dueño

Aclaración vigente:

en el uso validado dentro de Orders, el prop correcto para linked-asset es linked.

No debe reemplazarse por action en ese contexto.

Regla:

los componentes globales representan payload recibido.

No deben decidir ni inferir relaciones de dominio.

Aclaración vigente sobre vistas de universos funcionales:

una vista raíz de universo funcional no debe promoverse automáticamente a componente global.

Caso vigente:

resources/views/service/dashboard.blade.php

Lectura:

service.dashboard es vista del universo Servicio y mantenimiento.

No es componente Blade global.

No debe ubicarse dentro de resources/views/components.

No debe contener autorización propia.

Debe representar accesos y datos ya preparados por la capa correspondiente.

Regla:

un dashboard de área pertenece al universo o frente que representa, no a la familia de componentes globales, salvo que se extraigan piezas genéricas reutilizables.

Aclaración vigente sobre componentes y vistas de Orders en contexto /service:

orders.index, orders.create, orders._form y orders.partials.table pueden adaptarse al universo /service mediante payload, flags o rutas contextuales.

Esto no los convierte en componentes de service.

Orders conserva ownership visual del recurso Order.

Regla:

si el mismo partial reusable opera en más de un universo, debe recibir configuración explícita y no inferir rutas por su cuenta cuando el contexto sea relevante.

Caso vigente:

orders.partials.table puede recibir showRouteName para apuntar a orders.show o service.orders.show según el universo de navegación.

Aclaración vigente sobre componentes de Assets y Documents en formularios vinculados:

assets.components.order-form-asset puede participar del formulario de Orders como componente oferente de Assets.

documents._form puede consumir el contrato JS genérico de selects vinculados cuando corresponda.

Estas integraciones no trasladan ownership entre módulos.

Aclaración vigente sobre comportamiento JS reusable:

Contrato vigente:

app-linked-select-sync

Lectura:

comportamiento frontend reusable para sincronizar selects vinculados.

Contrato de datos vigente:

data-action="app-linked-select-sync"
data-source-select
data-target-select
data-source-value

Reglas:

no usar nombres específicos como app-party-asset-sync si la mecánica es genérica

el comportamiento JS no decide permisos

el comportamiento JS no reemplaza validación backend

el comportamiento JS no debe ser fuente única de consistencia de datos críticos

VISTAS Y PARTIALS DE FRENTES SATÉLITE Y CONTEXTOS ESPECIALIZADOS

Los frentes satélite y universos semánticos pueden requerir vistas propias o adaptaciones contextuales de vistas existentes.

no debe promoverse automáticamente a componente global

no debe contener autorización propia

si opera sobre un módulo dueño, debe respetar la vista, partial o contrato visual del módulo dueño

si el mismo partial reusable opera en más de un universo, debe recibir configuración explícita

no debe inferir rutas por su cuenta cuando el contexto sea relevante

COMPONENTES DE TABS

El sistema distingue dos tipos de tabs con roles diferentes.

TABS PRINCIPALES DEL HOST

Componente:

x-host-tabs

Responsabilidad:

organizar tabs principales de páginas show.

Incluye:

estructura de navegación
toolbar
scroll horizontal
paneles de contenido
estado activo

Reglas:

representa navegación principal del recurso
puede renderizar icon si viene en metadata
puede usar module_icon como fallback
no define lógica de negocio

Aclaración vigente:

x-host-tabs puede recibir tabs provenientes de surfaces modulares y de surfaces tenant/transversales no modulares, siempre que respeten la gramática esperada.

Gramática esperada:

key
label
view
data
count
priority

Caso validado:

tab Actividad contextual publicada desde OperationalActivity mediante TenantSurfaceRegistry.

Regla:

cuando la tab participe de return_tab, key debe ser estable y compatible con navegación del host.

Caso vigente:

activity

Hosts validados con tab Actividad:

tasks.show
parties.show
projects.show
assets.show
products.show
appointments.show
orders.show
documents.show

Regla:

la presencia de una tab Actividad no define por sí misma si la lectura es directa, agregada o contextual especial.

La modalidad depende del contrato de lectura preparado por la capa correspondiente.

Caso validado de lectura agregada:

projects.show usa ProjectActivityRecordSet para mostrar actividad de Project + Tasks.

Caso validado de lectura contextual especial:

parties.show usa PartyEmployeeActivityContext para mostrar actividad directa de Party employee con membership activa vinculada y actividad del user vinculado como actor o sujeto.

La tab Actividad puede usar:

icon = activity

declarado por la surface correspondiente.

TABS INTERNAS EMBEBIDAS

Componente:

x-tabs-embedded

Responsabilidad:

organizar contenido interno dentro de una capacidad embebida.

Casos:

segmentación por estado
segmentación interna de datasets
lectura agrupada

Reglas:

no participan de navegación principal
no usan return_tab
no representan cambio de recurso

Distinción obligatoria:

x-host-tabs → navegación del recurso
x-tabs-embedded → organización interna de contenido

No deben mezclarse.

COMPONENTES MODULARES

Ubicación general:

resources/views/<modulo>/

Responsabilidad:

componentes, partials o fragmentos visuales propios de un módulo funcional.

Reglas:

pertenecen al módulo que los aloja
no son componentes globales
no deben ubicarse en resources/views/components si no son reutilizables transversalmente
no deben contener lógica de negocio
no deben decidir autorización final
deben representar payload ya preparado por la capa correspondiente
deben respetar ownership funcional del módulo oferente

Caso vigente:

Products puede alojar una subcapa contextual interna llamada ProductOperationalSummary.

Ubicación visual vigente:

resources/views/products/operational-summary/

Partials vigentes:

resources/views/products/operational-summary/purchases-summary.blade.php
resources/views/products/operational-summary/sales-summary.blade.php
resources/views/products/operational-summary/flow-summary.blade.php

Lectura:

estos partials son propios de Products

no son componentes Blade globales

no deben documentarse como resources/views/components

no constituyen componentes reutilizables transversales

no representan un módulo nuevo

no tienen navegación propia

no contienen autorización final

no calculan métricas

representan payload ya resuelto por ProductOperationalSummaryService y publicado por ProductSurfaceService

Regla:

cuando una subcapa interna de un módulo tenga identidad funcional propia, puede alojarse en carpeta específica dentro del módulo para evitar confundirla con partials comunes.

Caso vigente:

products/operational-summary

Regla adicional:

no debe crearse un componente global nuevo si la necesidad visual es propia de un módulo y se resuelve correctamente con partials internos.

Regla adicional para snapshots propios:

cuando un módulo posee un snapshot textual propio y un vínculo gestionado opcional, sus componentes, partials y fragmentos visuales deben representar esa distinción.

Lectura vigente:

party_id = vínculo gestionado opcional con Parties

counterparty_name = dato textual propio del registro

Casos vigentes:

Orders

Documents

Reglas:

los partials propios del módulo no deben depender exclusivamente de party->name si existe método de presentación propio

no deben mostrar vacío o guion cuando exista counterparty_name

no deben usar un componente linked de Party como única representación si el vínculo gestionado no existe

no deben hacer parecer obligatorio un vínculo que el contrato del módulo define como opcional

si existe método de presentación propio, el partial debe consumirlo o recibir su resultado ya preparado por la capa correspondiente

Casos vigentes:

Order::displayCounterpartyName()

Document::displayCounterpartyName()

Regla:

el componente o partial representa el payload correcto.

No reconstruye el contrato de dominio.

Regla adicional para tablas modulares de Orders y Documents:

cuando una tabla de Orders o Documents representa el dato operativo visible del registro, la columna debe llamarse Contraparte.

Cuando la tabla filtra por party_id o representa vínculo gestionado, puede usar Contacto.

Contrato visual vigente:

showCounterparty

Uso:

indica que la tabla o surface debe mostrar la columna Contraparte.

showParty queda solo como fallback transitorio interno en Blades auditados.

Blades alineados:

resources/views/orders/partials/table.blade.php

resources/views/documents/partials/table.blade.php

resources/views/orders/partials/embedded-tabs.blade.php

resources/views/documents/partials/embedded-tabs.blade.php

resources/views/orders/index.blade.php

resources/views/documents/index.blade.php

Reglas:

los partials de Orders y Documents no deben usar showParty como contrato principal si el dato visible esperado es Contraparte

showCounterparty debe ser el contrato publicado por surfaces cuando corresponda mostrar contraparte

el fallback showParty no debe convertirse en nuevo contrato estable

Regla adicional para OrderLinked:

OrderLinked pertenece al módulo Orders y representa acceso contextual hacia Orders.

No debe exigir party_id como condición rígida para permitir creación contextual si Orders puede resolver counterparty_name en el formulario.

Archivos vigentes relacionados:

app/Support/Orders/OrderLinked.php

resources/views/orders/components/linked-order.blade.php

Patrón vigente:

OrderLinked::forOrder(?Order $order, ...)

Reglas:

OrderLinked no reemplaza autorización de Orders

OrderLinked no convierte Task ni Appointment en Order encubierta

OrderLinked no debe bloquear creación contextual por ausencia de party_id

OrderLinked no debe interpretar modelos consumidores

si el origen no tiene party_id, la Order resultante debe completar contraparte según contrato vigente del módulo Orders

Validación vigente:

Task sin party_id y sin order_id puede devolver state = creatable.

Appointment sin party_id y sin order_id puede devolver state = creatable.

Regla adicional para OrderSurfaceService:

OrderSurfaceService pertenece al módulo Orders y debe organizar ofertas por capacidades reales.

Ofertas vigentes:

orders.related.embedded

orders.associated.linked

Lectura:

orders.related.embedded representa órdenes relacionadas embebidas en hosts como assets.show y parties.show.

orders.associated.linked representa link asociado hacia una Order cuando el consumidor entrega Order|null.

Reglas:

no duplicar offers por host si la capacidad es la misma

si solo cambia host, filtro, tabsId, flags visuales o mensaje vacío, usar configuración contextual

el host aloja la surface

Orders prepara el payload o la view de Orders

el host no reconstruye internamente el contenido del oferente

Patrón vigente:

relatedHosts()

Regla adicional para hooks y servicios modulares:

los hooks y servicios de soporte ubicados en un módulo deben funcionar como frontera explícita de su dominio.

Casos vigentes:

OrdersHooks funciona como frontera estratégica de consulta o intervención del flujo de Orders.

InventoryOrderItemHooks conserva la lógica material de bloqueo de cierre por ejecución física de líneas.

ProductLineItemSelector ofrece selección y validación de productos para líneas.

orders.associated.linked

showCounterparty

el host aporta contexto compatible

no debe duplicarse una offer por cada host cuando la capacidad es la misma

Regla final:

los componentes de show pages deben expresar autonomía del host, relaciones invitadas y snapshots propios sin mezclar responsabilidades.

COMPONENTES REUTILIZABLES TRANSVERSALES

Componentes utilizados por múltiples módulos con gramática común.

Ejemplos:

x-line-item-form
x-line-item-table
x-line-operation-modal
x-tabs-embedded

Reglas:

pueden ser compartidos

no deben consultar modelos externos por sí mismos

no deben decidir permisos efectivos

no deben reconstruir reglas de módulos consumidores u oferentes

Aclaración vigente para line items:

los componentes transversales de líneas pueden ser reutilizados por Orders y Documents cuando la gramática visual y operativa sea equivalente.

Eso no transfiere ownership de los ítems.

Lectura vigente:

Orders administra order_items.

Documents administra document_items.

Products ofrece selección y validación de productos mediante ProductLineItemSelector.

Inventory puede ofrecer operaciones materiales sobre líneas mediante capacidades oficiales.

Reglas:

x-line-item-form no debe decidir por sí mismo qué productos son seleccionables

x-line-item-table no debe interpretar ownership de productos, documentos u órdenes

x-line-operation-modal no debe ejecutar operación material sin backend autorizado

la selección y validación de productos debe venir preparada desde la capacidad correspondiente

Caso vigente:

ProductLineItemSelector concentra opciones, reglas y verificación de acceso a Products para líneas.

Aclaración vigente para x-dev-component-version:

x-dev-component-version es un componente transversal de control visual y continuidad técnica.

Responsabilidad:

identificar Blades intervenidos

mostrar versión del componente, partial, card o vista

facilitar auditoría posterior

favorecer continuidad entre cambios visuales sucesivos

Props vigentes:

name

version

align

Reglas:

no usar prop file

no ubicarlo dentro de tablas, tbody, formularios, loops ni estructuras repetitivas

no reemplaza SECTION_VERSION documental

no reemplaza versionado de código

no decide permisos ni lógica de negocio

debe agregarse o actualizarse cuando un Blade modificado tenga entidad visual propia

Regla de frontera:

un componente reusable puede unificar forma visual, pero no unificar dominio.

La lógica debe permanecer en:

módulo dueño

servicio de frontera

selector

hook

surface service

controller autorizado

payload preparado

Regla final:

la reutilización transversal debe reducir duplicación visual sin crear acoplamiento funcional oculto.
