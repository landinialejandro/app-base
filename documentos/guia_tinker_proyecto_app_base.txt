<<SECTION: OBJETIVO>>

OBJETIVO

Este documento define criterios operativos para validar comportamiento real del sistema mediante Tinker.

Su función es:

probar runtime real
verificar permisos efectivos
auditar constraints
inspeccionar relaciones activas
reproducir flujos técnicos controlados
reducir suposiciones durante auditorías

No reemplaza:

tests automatizados
documentación doctrinal
auditorías funcionales completas

Aclaración consolidada:

cuando exista diferencia entre una hipótesis teórica y lo observado en Tinker sobre runtime real, debe investigarse el código vigente y no sostener supuestos no verificados
<<END SECTION>>

<<SECTION: 🆕 PRINCIPIOS DE USO>>
SECTION_VERSION: 00001

PRINCIPIOS DE USO

Regla central:

no asumir
no suponer
verificar

Toda prueba debe intentar reproducir el comportamiento real del sistema con el menor nivel posible de simulación artificial.

Priorizar:

modelos reales
usuarios reales de prueba
tenants reales de prueba
datos persistidos reales
servicios del sistema cuando corresponda

Evitar:

arrays inventados si existe dato real disponible
resolvers aislados cuando debe probarse Security
validaciones parciales si el flujo real exige contexto completo
conclusiones basadas en una sola observación débil
<<END SECTION>>

<<SECTION: REGLA CENTRAL>>
REGLA CENTRAL

Las pruebas en Tinker no deben ejecutarse de forma abstracta ni incompleta.

Toda prueba debe respetar el runtime real del sistema.

En especial:

si el flujo real depende de tenant activo, Tinker debe bindear tenant activo
si el flujo real depende de Security, Tinker debe consumir Security
si el flujo real depende de contexto funcional, Tinker debe pasar ese contexto de forma explícita

NO SE ASUME.
NO SE SUPONE.

Si una prueba requiere un dato no validado:

no inventarlo
no inferirlo
no completarlo por memoria
primero obtenerlo de base o de una validación previa
<<END SECTION>>

<<SECTION: REGLA DE TENANT ACTIVO EN TINKER>>
REGLA DE TENANT ACTIVO EN TINKER

En este proyecto, muchas resoluciones dependen de:

app('tenant')

Por lo tanto, una prueba en Tinker no reproduce correctamente el sistema si no existe tenant activo bindeado en el contenedor.

Regla operativa:

antes de probar permisos, visibilidad, queries tenant-aware o lógica que consume Security, debe resolverse y bindearse un tenant activo real.

Forma correcta:

1. obtener un usuario real
2. obtener una membership activa real de ese usuario
3. tomar el tenant de esa membership
4. bindearlo con:

app()->instance('tenant', $tenant)

Sin ese paso, la prueba puede:

fallar artificialmente
devolver falsos negativos
no representar el comportamiento real del sistema
<<END SECTION>>

<<SECTION: REGLA DE AUTORIZACIÓN EN TINKER>>
REGLA DE AUTORIZACIÓN EN TINKER

La autorización debe probarse desde la capa central.

Regla operativa:

para permisos, visibilidad o acciones permitidas, usar:

app(Security::class)

No probar permisos reconstruyendo lógica manualmente.

Usos válidos:

inspect(...)
allows(...)
authorize(...)
scope(...)

Regla adicional:

si el módulo utiliza constraints, la prueba debe incluir el contexto funcional real requerido.

Ejemplo conceptual:

si la capacidad requiere kind, probar con kind explícito
si la capacidad no requiere contexto, no inventarlo
<<END SECTION>>

<<SECTION: REGLA PARA CREATE CONTEXTUAL>>
REGLA PARA CREATE CONTEXTUAL

Cuando un módulo tenga create contextual, la prueba no debe resolverse mediante evaluación abstracta.

Regla operativa:

no usar pruebas equivalentes a:

authorize('create', Model::class)

cuando el módulo requiere contexto funcional

En esos casos, la prueba correcta debe pasar por Security con contexto explícito.

Ejemplo conceptual:

Security::inspect($user, 'orders.create', null, ['kind' => 'service'])

La prueba debe verificar:

contrato persistido
constraints efectivas
resultado allowed
consistencia entre subject nulo y class-string si ambas formas están soportadas por la capa central
<<END SECTION>>

<<SECTION: CASOS PRIORITARIOS DE USO>>

CASOS PRIORITARIOS DE USO

Usar Tinker especialmente para:

permisos V2 efectivos
constraints y allowed_kinds
módulos habilitados por tenant
dataset real bajo scope
bindings multi-tenant
inspección de relaciones
debug de create contextual
delegación de entidades hijas
flujos entre módulos
auditorías rápidas sobre runtime
validación de agrupadores operativos internos cuando una acción visible pueda generar uno o múltiples registros relacionados
validación backend de campos ocultos o relaciones sensibles que la UI no muestra a todos los usuarios

Cuando se requiera registrar varias salidas, copiar resultados para el chat, ejecutar comandos rápidos o conservar trazabilidad visual de una sesión, debe utilizarse Project Lab local.

Project Lab permite ejecutar Tinker, comandos Artisan, herramientas de actualización de código, actualización documental y auditorías desde una misma interfaz operativa.

Regla adicional consolidada:

cuando una integración entre módulos genere dudas, priorizar prueba directa del contrato consumido y del arbitraje efectivo antes de modificar código

Caso vigente:

para inventory, las pruebas deben poder distinguir entre inventory_operations como agrupación operativa contextual e inventory_movements como trazabilidad unitaria de stock.

Caso vigente adicional:

para parties, las pruebas deben validar que StorePartyRequest y UpdatePartyRequest bloqueen la asignación manipulada de employee cuando el usuario no esté autorizado, aunque la UI oculte la opción.

<<SECTION: 🆕 PATRÓN GENERAL DE PRUEBA>>
SECTION_VERSION: 00003

PATRÓN GENERAL DE PRUEBA

Toda prueba relevante debería intentar responder:

qué quiero validar
con qué usuario
en qué tenant
sobre qué módulo
con qué contexto funcional
cuál sería el resultado esperado
cuál fue el resultado real

Secuencia recomendada:

resolver usuario
resolver tenant
setear contexto si aplica
cargar recurso real
ejecutar consulta real
comparar resultado esperado vs real
anotar hallazgo

Si falta alguno de esos elementos, la conclusión puede quedar incompleta.

Validación adicional para acciones con agrupación interna:

cuando una acción pueda crear o reutilizar una entidad agrupadora y luego generar registros unitarios asociados, la prueba debe verificar ambas capas.

Controlar:

que la agrupación exista cuando corresponda
que los registros unitarios queden asociados si corresponde
que cada registro unitario conserve trazabilidad propia
que el tenant sea consistente en todas las capas
que la acción visible no dependa de datos inventados ni de supuestos sobre cantidad de registros generados

Caso vigente:

inventory_operations e inventory_movements.

Validación adicional para relaciones sensibles ocultas en UI:

cuando una opción no sea visible para ciertos usuarios, no alcanza con validar navegador.

Debe probarse intento manipulado contra la validación backend correspondiente.

Controlar:

usuario real
tenant real
request o servicio real cuando corresponda
resultado esperado permitido o bloqueado
mensaje o error de validación cuando aplique

Caso vigente:

asignación de employee en parties mediante StorePartyRequest y UpdatePartyRequest.

<<END SECTION>>

<<SECTION: BOOTSTRAP BASE RECOMENDADO PARA TINKER>>
BOOTSTRAP BASE RECOMENDADO PARA TINKER

Bloque base recomendado para comenzar pruebas reales:

use App\Models\User;
use App\Support\Auth\Security;

$user = User::where('email', 'juan@tech.local')->firstOrFail();

$tenant = $user->memberships()
    ->where('status', 'active')
    ->with('tenant')
    ->firstOrFail()
    ->tenant;

app()->instance('tenant', $tenant);

$security = app(Security::class);

Este bloque deja resuelto:

usuario real
membership activa real
tenant real
tenant activo en contenedor
acceso a la capa central de autorización

A partir de ahí ya pueden ejecutarse pruebas confiables.
<<END SECTION>>

<<SECTION: COMANDOS BASE RECOMENDADOS>>
COMANDOS BASE RECOMENDADOS

PRUEBA DE CONTEXTO ACTIVO

Verificar primero:

tenant_id
tenant_name

Ejemplo:

echo $tenant->id;
echo $tenant->name;

PRUEBA DE INSPECCIÓN DE PERMISOS

Usar:

$security->inspect($user, 'modulo.capacidad', $subject, $context);

PRUEBA DE PERMISO BOOLEANO

Usar:

$security->allows($user, 'modulo.capacidad', $subject, $context);

PRUEBA DE AUTORIZACIÓN DURA

Usar:

$security->authorize($user, 'modulo.capacidad', $subject, $context);

PRUEBA DE DATASET

Usar:

$security->scope($user, 'modulo.viewAny', Modelo::query())

y luego aplicar count(), get(), paginate(), etc. según el caso.

<<SECTION: TINKER LAB LOCAL>>

TINKER LAB LOCAL

El proyecto dispone actualmente de Project Lab como herramienta local auxiliar para ejecutar pruebas Tinker y tareas técnicas vinculadas.

Ubicación vigente:

tools/project-lab

La herramienta permite:

pegar código PHP interno de Tinker
ejecutar pruebas dentro del proyecto actual
ejecutar Tinker desde textarea
ejecutar Tinker desde clipboard del sistema mediante backend
ver salida unificada en consola
copiar salida para chat
persistir salida visible en localStorage
ejecutar comandos Artisan rápidos
ejecutar herramientas Lab de código, documentación y auditoría
utilizar snippets rápidos para pruebas frecuentes

La herramienta no forma parte del sistema SaaS productivo.

Es una utilidad local de depuración, auditoría y mantenimiento del proyecto.

No debe publicarse ni exponerse en entornos accesibles desde internet.

Regla crítica:

en el editor Tinker de Project Lab debe pegarse solo el código PHP interno.

No debe pegarse el comando completo:

php artisan tinker --execute

La herramienta ya ejecuta internamente el proceso correspondiente contra el proyecto actual.

Regla adicional:

si se quiere visualizar el resultado de una expresión en comandos rápidos o snippets, debe utilizarse return o una salida explícita mediante echo, dump o equivalente.

Ejemplo conceptual:

return User::all();

<<SECTION: USO OPERATIVO DE TINKER LAB>>
SECTION_VERSION: 00002

USO OPERATIVO DE TINKER LAB

Para levantar Project Lab local desde la raíz del proyecto:

php -S 127.0.0.1:8787 -t tools/project-lab

Luego abrir:

http://127.0.0.1:8787

Flujo recomendado para Tinker:

1. abrir la pestaña Editor Tinker
2. pegar código PHP interno o insertar un snippet rápido
3. ejecutar desde textarea o desde clipboard backend
4. revisar la salida en la consola unificada
5. copiar la salida si debe compartirse en el chat
6. conservar o limpiar la consola según necesidad

Flujo recomendado para automatizador:

1. abrir la pestaña Herramientas Lab
2. pegar el bloque a aplicar en textarea o copiarlo al clipboard del sistema
3. elegir la herramienta correspondiente:

code
docs
audit

4. revisar la salida en la consola unificada
5. validar que el resultado informe OK o ERROR de forma explícita

Herramientas disponibles:

code:
aplica actualizaciones de código compatibles con el automatizador vigente

docs:
aplica actualizaciones documentales por bloques completos de sección

audit:
ejecuta auditorías simples o comandos de inspección definidos para el proyecto

Regla para clipboard:

el clipboard operativo de Project Lab se resuelve desde backend mediante herramientas del sistema cuando corresponde.

No depende del permiso de lectura del portapapeles del navegador para ejecutar Tinker desde clipboard ni herramientas Lab desde clipboard.

Regla para snippets rápidos:

si el snippet debe mostrar resultado visible, debe insertar return o producir salida explícita.

Una expresión sin return puede ejecutarse correctamente y aun así no mostrar resultado propio, dejando visible solo la depuración SQL.

Regla:

la consola unificada de Project Lab es la salida operativa principal para compartir resultados, errores y auditorías durante el trabajo técnico.

<<SECTION: CASO BASE YA VALIDADO: ORDERS.CREATE>>
CASO BASE YA VALIDADO: ORDERS.CREATE

Estado actual validado:

orders.create ya no depende indebidamente de subject para funcionar

Comportamiento validado:

con kind válido:
allowed = true

sin kind:
allowed = false

con kind inválido:
allowed = false

Regla operativa derivada:

orders.create debe probarse siempre como create contextual

Ejemplos válidos:

$security->inspect($user, 'orders.create', null, ['kind' => 'service'])

$security->inspect($user, 'orders.create', Order::class, ['kind' => 'service'])

Ejemplos inválidos como prueba doctrinal del contrato:

Gate::allows('create', Order::class)

o cualquier evaluación abstracta equivalente sin contexto kind
<<END SECTION>>

<<SECTION: PATRÓN DE PRUEBA RECOMENDADO PARA MÓDULOS CON CONSTRAINTS>>
PATRÓN DE PRUEBA RECOMENDADO PARA MÓDULOS CON CONSTRAINTS

Cuando un módulo use constraints funcionales, conviene probar siempre la misma matriz mínima:

1. caso válido con contexto completo
2. caso inválido sin contexto
3. caso inválido con valor fuera de constraints
4. caso de control de no regresión sobre view
5. caso de control de no regresión sobre update
6. caso de control de no regresión sobre viewAny o scope

Objetivo:

confirmar que la autorización:

requiere el contexto correcto
respeta constraints
no se rompe en acciones vecinas
no altera el dataset del módulo
<<END SECTION>>

<<SECTION: REGLA PARA SUJETOS EN PRUEBAS>>
REGLA PARA SUJETOS EN PRUEBAS

Distinguir siempre entre:

subject nulo
subject class-string
subject record real

subject nulo:
útil para create contextual o inspecciones donde no existe registro real todavía

subject class-string:
solo usar si el caso realmente lo admite o si se quiere comparar comportamiento

subject record real:
usar para view, update, delete y verificaciones sobre registros existentes

Regla práctica:

no pasar modelos o clases por inercia

el tipo de subject debe responder al tipo real de prueba
<<END SECTION>>

<<SECTION: ERRORES COMUNES A EVITAR>>
ERRORES COMUNES A EVITAR

No ejecutar pruebas de permisos sin tenant activo.

No probar autorización directamente contra policies cuando el caso depende de contexto y el contrato exige Security.

No interpretar un false en Tinker como problema del sistema si antes no se confirmó:

usuario real
membership activa
tenant activo
contexto funcional requerido
constraints efectivas

No usar usuarios o tenant ids recordados por memoria sin revalidarlos.

No mezclar pruebas doctrinales con pruebas de conveniencia.

No concluir que un módulo está roto si la prueba fue abstracta cuando el contrato exigía contexto.
<<END SECTION>>

<<SECTION: CUÁNDO USAR GATE O POLICY EN TINKER>>
CUÁNDO USAR GATE O POLICY EN TINKER

Gate o Policy pueden usarse para:

viewAny
view
update
delete
create simples

No deben usarse como mecanismo principal de prueba para:

create contextuales
casos con constraints obligatorias
casos donde el contrato ya define consumo por Security

Regla práctica:

si el create necesita un dato previo del negocio, la prueba principal debe hacerse con Security.
<<END SECTION>>

<<SECTION: PLANTILLA DE PRUEBA RÁPIDA>>

PLANTILLA DE PRUEBA RÁPIDA

Código base recomendado para ejecutar una prueba simple en Tinker o en Tinker Lab:

use App\Models\User;
use App\Support\Auth\Security;

$user = User::where("email", "juan@tech.local")->firstOrFail();

$tenant = $user->memberships()
    ->where("status", "active")
    ->with("tenant")
    ->firstOrFail()
    ->tenant;

app()->instance("tenant", $tenant);

$security = app(Security::class);

echo "tenant_id: {$tenant->id}\n";
echo "tenant_name: {$tenant->name}\n";

Esta plantilla solo prepara contexto.

A partir de ella deben agregarse las pruebas específicas del caso.

Regla actualizada:

si se utiliza consola manual, envolver este bloque en:

php artisan tinker --execute='...'

si se utiliza Tinker Lab local, pegar únicamente el código interno sin el wrapper del comando.

REGLA FINAL

usuario real
membership activa real
tenant activo real

Sin eso, la prueba puede servir como exploración, pero no como validación del sistema.
