LECCIONES VIGENTES

Ability sobre módulo inexistente puede fallar por inexistencia contractual y no por falta de permiso.

Dataset interno reusable y autorización efectiva son problemas distintos.

Una falla visual reusable puede deberse a hostPack, slot, type, shape o view desalineados y no a permisos.

Sin tenant activo en Tinker pueden aparecer falsos negativos.

No revisar solo controller principal: también tabs, modales, partials y superficies secundarias.

Un host visual no se convierte en dueño funcional de la capacidad invitada.

No toda necesidad repetida exige módulo nuevo.

Cambiar estados operativos requiere alinear Security, Gate, UI y ciclo de vida real.

Si una decisión importante no queda documentada, reaparece como discusión futura.

Si una entidad separa tipo base y relación operativa, la seguridad no debe seguir usando la dimensión anterior por compatibilidad.

Caso vigente:

parties.kind no debe arbitrar permisos.

party_roles.role debe arbitrar permisos mediante allowed_party_roles.

Si una opción sensible no aparece en UI, eso no alcanza como protección.

Caso vigente:

aunque el formulario de una party vinculada a membership activa no exponga de forma simple campos manipulables, debe probarse intento manipulado y bloqueo backend.

Las validaciones sensibles deben verificarse por persistencia final, no solo por mensaje visual.

Caso vigente:

tras intentar manipular kind, email y roles de una party vinculada a membership activa, debe verificarse que el registro siga intacto.

Lectura esperada:

kind: person
roles: employee

Los snippets para Project Lab deben respetar estrictamente el contrato del automatizador.

Error detectado:

usar encabezados no compatibles o mezclar FILE con TARGET puede hacer que el automatizador interprete incorrectamente el bloque como archivo completo.

Regla:

no mezclar formatos.

Usar FILE solo para archivo completo.

Usar TARGET solo para operación parcial.

Formatos correctos:

// FILE: ruta | Vn
// TARGET: ruta :: metodoExistente
// TARGET: ruta ++ metodoNuevo

No usar:

// FILE: ruta :: metodo

<?php // FILE: ruta | V1

Lección:

un formato ambiguo no debe tratarse como éxito operativo.

Debe corregirse el snippet antes de aplicar.

tenant_id debe respetar el tipo real de tenants.id.

Error detectado:

OperationalActivityLogger trató tenant_id como entero, pero tenants.id es UUID char(36).

Corrección:

conservar tenant_id como string.

Lección:

no asumir tipos de claves.

Validar esquema real antes de persistir datos transversales.

Event discovery puede registrar listeners automáticamente.

Error detectado:

RegisterOperationalActivity@handle ya estaba registrado por event discovery y además se registró manualmente en AppServiceProvider.

Resultado:

dos jobs
dos registros idénticos

Corrección:

eliminar el registro manual.

Lección:

si event:list ya muestra la asociación, no duplicar con Event::listen() manual.

php artisan tinker --execute puede fallar con use según el wrapper.

Error detectado:

bloques con use ... pueden parsearse mal dentro del wrapper de ejecución.

Alternativas:

usar clases con namespace completo
ejecutar el bloque en Project Lab según corresponda

Lección:

si el problema es el wrapper y no la lógica probada, ajustar la forma de ejecución antes de cambiar código funcional.

OperationalActivity no debe bloquear la operación principal.

Lección:

la trazabilidad operativa queued debe registrar si corresponde, pero un fallo del log no debe romper el flujo principal ya autorizado y ejecutado por el módulo dueño.

Jobs procesados sin failed_jobs no garantizan persistencia de OperationalActivity.

Error detectado:

los jobs de OperationalActivity se procesaban correctamente, sin failed_jobs, pero no se persistía actividad.

Causa:

OperationalActivityCatalog no reconocía todavía nuevos modelos en el mapeo modelo → módulo y módulo → catálogo.

Lección:

si hay jobs procesados sin errores pero no hay registros en operational_activities, revisar primero:

moduleForRecordClass()
catalogForModule()
catálogo dueño del módulo
activityTrackedFields()

antes de tocar controllers, eventos o listener.

Project Lab audit debe recibir comandos ejecutables, no prompts.

Error detectado:

una auditoría de Project Lab intentó ejecutar texto de prompt como Bash y generó errores de comandos inexistentes.

Lección:

en la herramienta audit debe pegarse solo comando ejecutable o bloque compatible con auditoría, no texto documental ni prompts de chat.

Las relaciones hijas pueden requerir metadata operativa explícita.

Caso detectado:

en parties no alcanza con registrar solo atributos de Party, porque los roles operativos viven en party_roles.

Lección:

cuando una relación hija expresa un cambio operativo relevante, debe representarse explícitamente mediante metadata operativa o patrón equivalente.

Esto no convierte la relación hija en módulo propio.

extra_changes no debería persistirse como duplicación innecesaria.

Caso detectado:

en parties.updated la información de party_roles quedó duplicada dentro de changes y extra_changes.

Lección:

extra_changes puede funcionar como insumo interno para construir changes, pero no necesariamente debe persistirse como metadata final si genera duplicación.

InventoryMovement no debe duplicar la trazabilidad operativa agrupada.

Caso validado:

inventory debe registrar InventoryOperation como actividad transversal cuando representa la acción operativa agrupada.

Lección:

InventoryMovement conserva trazabilidad fina de stock.

No todo movimiento atómico debe duplicarse como OperationalActivity si existe una operación agrupada que representa mejor la acción del usuario.

No usar componentes Blade no validados.

Error detectado:

se usó x-empty-state en una vista, pero el componente no existía.

Resultado:

Unable to locate a class or view for component [empty-state]

Corrección:

reemplazar por estado vacío inline con detail-block.

No crear componente global como parche si no corresponde cerrar esa abstracción.

Lección:

la documentación visual no alcanza para asumir existencia runtime de un componente.

Antes de usar un componente Blade en código real debe validarse su existencia en resources/views/components o clase correspondiente.

OperationalActivity queued no aparece en vistas hasta procesar la cola.

Caso validado:

antes de procesar cola: operational_activities = 0
queue:work ejecutó RegisterOperationalActivity
después: jobs = 0, failed_jobs = 0, operational_activities = 2

Lección:

si la tab de actividad no muestra registros, revisar primero:

jobs
failed_jobs
procesamiento de cola
persistencia final

antes de tocar vista, logger, resolver o Blade.

El primer piloto funcional no siempre es cierre arquitectónico.

Caso detectado:

se agregó tab Actividad mediante $tabItems->push directamente en tasks.show y parties.show.

Aunque funcionó, duplicaba montaje local y evitaba el sistema surface-driven.

Corrección:

crear TenantSurfaceRegistry
crear OperationalActivitySurfaceService
fusionar surfaces tenant/transversales desde ModuleSurfaceRegistry

Lección:

si una capacidad transversal puede publicarse mediante registry reusable, no debe quedar como cierre estable una inserción manual por vista host.

No conviene hardcodear conocimiento interno de un módulo dentro de una capacidad transversal.

Caso detectado:

la lectura agregada de actividad de Project no debía resolverse con una condición interna dentro de OperationalActivity.

Corrección:

Projects declara ProjectActivityRecordSet.

OperationalActivity consume el record set publicado.

Lección:

si la frontera funcional pertenece al módulo dueño, no debe vivir en OperationalActivity.

activity_record_set y activity_context resuelven problemas distintos.

Caso validado:

Project + Tasks se resolvió con activity_record_set porque es un conjunto explícito de records.

Party employee con membership activa vinculada se resolvió con activity_context porque es un criterio contextual especial basado en el user vinculado como actor o sujeto.

Lección:

no forzar activity_record_set cuando la frontera no es un conjunto simple de records.

No hardcodear esa excepción dentro de OperationalActivity.

El módulo dueño debe declarar el criterio mediante provider compatible.

La actividad histórica no aparece retroactivamente si el evento no existía o no se emitió.

Caso detectado:

Party #25 Carlos Demo ya existía antes de emitir actividad operativa para ciertos flujos.

Se emitió evento manual posterior para generar actividad de prueba.

Lección:

ese registro debe leerse como backfill manual de entorno de desarrollo, no como acción real del flujo original.

Si una acción ocurrió antes de que el flujo emitiera evento operativo, la actividad no aparece retroactivamente.

Ver actividad de un user en lectura general no implica que aparezca automáticamente en una Party vinculada.

Caso detectado:

actividad donde Carlos actuaba como user no aparecía en Party #25 mientras parties.show solo tenía lectura directa de Party.

Corrección:

PartyEmployeeActivityContext habilita lectura contextual especial para employee con membership activa vinculada.

Lección:

si la actividad pertenece a records donde el colaborador actúa como user, la ficha Party necesita contrato contextual especial para incluirla.

No alcanza con tener membership vinculada ni actividad persistida.

jobs = 0 y failed_jobs = 0 son condición necesaria pero no suficiente.

Lección:

también hay que validar persistencia y dataset esperado.

Caso vigente:

en activity_context de Party employee debe validarse que aparezcan las filas esperadas para:

actividad directa de Party
actividad donde el user vinculado sea actor
actividad donde el user vinculado sea sujeto

Crear una Party employee desde membership requiere contexto funcional employee.

Caso detectado:

autorizar creación de Party employee desde TenantMembershipPartyController informando solo kind = person no alcanza para el contrato vigente.

Motivo:

en parties, kind expresa tipo de contacto.

La relación funcional sensible vive en party_roles.role.

Lección:

si la autorización depende de allowed_party_roles, el contexto funcional debe informar employee cuando esa es la relación que se intenta crear.

No crear clases visuales nuevas si ya existe variante equivalente.

Caso detectado:

el botón Ver cambios necesitaba reducir impacto visual dentro de la tabla.

Corrección:

usar variante existente:

btn btn-secondary btn-icon

Lección:

las mejoras visuales deben priorizar patrones existentes antes de crear clases nuevas.
