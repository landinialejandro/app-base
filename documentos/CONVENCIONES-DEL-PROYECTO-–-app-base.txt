Objetivo

Este documento define las convenciones de desarrollo del proyecto app-base con el objetivo de:

mantener consistencia en el código
evitar decisiones ad-hoc entre módulos
facilitar el mantenimiento del sistema
permitir que el proyecto escale a múltiples módulos

Estas convenciones no modifican la arquitectura SaaS existente.

Convención de modelado del dominio

El sistema distingue entre:

entidades del sistema
entidades del negocio

users pertenece al sistema.

Las entidades del negocio deben modelarse preferentemente mediante estructuras reutilizables.

La entidad base para personas u organizaciones del negocio es:

parties

Antes de crear nuevas tablas como clients, suppliers o employees se debe evaluar si el caso corresponde a una variante de parties.

En esta etapa del proyecto la diferenciación funcional podrá resolverse mediante un campo simple de tipo o rol.

En el futuro podrán agregarse:

roles múltiples
perfiles especializados
atributos dinámicos

sin modificar la entidad base.

Convenciones de base de datos

Nombres de tablas:

snake_case
plural

Ejemplos:

users
tenants
memberships
projects
parties
tasks
products
orders

Claves primarias:

id
bigint unsigned auto_increment

Claves foráneas:

entidad_id

Ejemplos:

tenant_id
user_id
project_id
party_id

Columna obligatoria de multi-tenancy

Todas las entidades empresariales deben incluir:

tenant_id

Ejemplo:

$table->foreignId('tenant_id')->index();

Timestamps

Todas las tablas empresariales deben incluir:

$table->timestamps();

Soft delete

Todas las entidades empresariales utilizan:

$table->softDeletes();

Convenciones de modelos

PascalCase
singular

Ejemplos:

User
Tenant
Project
Party
Task
Product
Order

Traits obligatorios en entidades empresariales:

use TenantScoped;
use SoftDeletes;
use ResolvesTenantRouteBinding;

Convenciones de rutas

Ejemplo:

URLs:

plural
kebab-case

Ejemplos:

/parties

Convenciones de Requests

Nunca se valida directamente en el controlador.

Se utilizan Form Requests.

Convención:

StoreEntidadRequest
UpdateEntidadRequest

Ejemplo:

StorePartyRequest
UpdatePartyRequest

Ubicación:

app/Http/Requests
