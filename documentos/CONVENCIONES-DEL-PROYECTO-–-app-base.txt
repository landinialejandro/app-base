CONVENCIONES-DEL-PROYECTO – app-base

Objetivo

Definir las convenciones técnicas y de desarrollo que deben seguir todos los módulos del proyecto app-base.

Estas reglas permiten mantener consistencia, previsibilidad y facilidad de mantenimiento del sistema.

Arquitectura base

El sistema utiliza la arquitectura estándar de Laravel basada en:

Modelos Eloquent
Controladores
Vistas Blade
Migraciones
Middleware
Rutas REST

Multi-tenancy

El sistema es SaaS multi-tenant.

El aislamiento de datos se realiza mediante la columna:

tenant_id

Todas las entidades del negocio deben pertenecer a un tenant.

Fuente del tenant operativo

La única fuente válida del tenant operativo es:

app('tenant')

No se debe utilizar directamente:

auth()->user()->tenant_id
request()->tenant_id

Todas las operaciones del sistema deben resolver el tenant actual mediante:

app('tenant')

Acceso al tenant actual

El tenant activo puede obtenerse mediante:

app('tenant')

Los controladores deben utilizar siempre este tenant como contexto de operación.

Trait obligatorio en modelos empresariales

Los modelos del negocio deben utilizar:

TenantScoped
ResolvesTenantRouteBinding
SoftDeletes

Objetivos:

TenantScoped
aplica automáticamente el filtro por tenant.

ResolvesTenantRouteBinding
protege el acceso por URL.

SoftDeletes
permite eliminación lógica.

Uso de ResolvesTenantRouteBinding

Todo modelo accesible mediante route model binding que pertenezca a un tenant debe utilizar el trait:

ResolvesTenantRouteBinding

Esto evita acceso cruzado entre tenants mediante URLs manipuladas.

La resolución efectiva del modelo debe garantizar que el registro pertenece al tenant actual.

Convenciones de base de datos

Todas las tablas del negocio deben incluir:

tenant_id
timestamps
softDeletes

Reglas para tenant_id:

tenant_id no debe crearse con foreignId('tenant_id')

debe definirse manualmente con un tipo compatible con tenants.id

antes de crear una nueva migración se debe copiar el patrón real ya utilizado en módulos existentes del proyecto

Esto garantiza consistencia con la arquitectura multi-tenant real del sistema.

Asignación de tenant_id en entidades

Entidades padre

Cuando el modelo utiliza el trait TenantScoped, tenant_id no debe asignarse manualmente en el controller.

El trait se encarga automáticamente de asignarlo al crear el registro.

Entidades hijas

En entidades hijas dependientes de una cabecera (por ejemplo items de documentos u órdenes), tenant_id debe asignarse automáticamente a partir de la cabecera correspondiente.

Regla para entidades hijas con tenant_id

En entidades hijas (items de documentos, items de órdenes, etc):

tenant_id existe en la tabla para facilitar índices y consultas.

tenant_id NO debe exponerse en $fillable.

tenant_id debe asignarse automáticamente desde la cabecera correspondiente o desde el contexto del tenant.

Convenciones de validación multi-tenant

En validaciones que utilicen exists o validaciones de claves foráneas sensibles, siempre se debe restringir al tenant actual.

La validación debe garantizar que la entidad referenciada pertenece al mismo tenant activo.

Esto evita referencias cruzadas entre tenants.

Protección de tenant_id en modelos

tenant_id no debe quedar expuesto innecesariamente en el atributo fillable de entidades hijas.

La asignación de tenant_id debe realizarse de forma controlada desde el modelo o desde el controller utilizando el contexto del tenant actual.

Convenciones de controladores

Los controladores deben seguir el patrón REST estándar de Laravel.

Métodos esperados:

index
create
store
show
edit
update
destroy

Convenciones de rutas

Las rutas deben definirse en:

routes/web.php

Utilizar resource routes siempre que sea posible.

Ejemplo:

Route::resource('projects', ProjectController::class);

Las rutas deben ejecutarse dentro de los middlewares:

auth
tenant

Convenciones de modelos

Los modelos deben:

usar Eloquent
definir relaciones explícitas
definir fillable o guarded correctamente
utilizar los traits de multi-tenancy

Convenciones de vistas

Cada módulo debe tener su carpeta de vistas:

resources/views/nombre_modulo

Ejemplos:

resources/views/projects
resources/views/parties
resources/views/tasks

Estructura visual de páginas

Todas las páginas internas deben seguir esta estructura:

<x-page> <x-breadcrumb /> <x-page-header /> <x-card /> </x-page>

Componentes Blade disponibles

x-page
Contenedor principal de página.

x-breadcrumb
Ruta de navegación.

x-page-header
Título de página y acciones.

x-card
Contenedor visual de contenido.

La base visual del sistema se define en el documento:

BASE-VISUAL-DEL-PROYECTO – app-base. 

BASE-VISUAL-DEL-PROYECTO – app-…

Convención de listados

Las vistas index deben utilizar el patrón de lista del sistema.

Estructura visual:

page.list-page
card.list-card
table-wrap.list-scroll

Esto garantiza:

scroll interno en tablas
encabezado visible
layout estable

Convención de formularios

Los formularios deben:

utilizar parciales _form
agrupar campos de forma consistente
mostrar errores de validación cerca del campo
ubicar botones dentro de form-actions

Convención de entidades padre/hijo

Cuando una entidad padre tenga registros hijos relevantes:

la eliminación debe advertir explícitamente al usuario que existen registros relacionados.

Si el usuario confirma la acción, la eliminación puede ejecutarse en cascada.

No se debe obligar por defecto al usuario a eliminar manualmente cada registro hijo, salvo que el módulo requiera trazabilidad estricta.

Convención para cálculos de totales

Existen dos estrategias posibles para manejar totales en el sistema:

Estrategia dinámica

Los totales se calculan dinámicamente a partir de los ítems mediante métodos del modelo (por ejemplo calculateTotal()).

Esto se utiliza cuando el registro no cumple función documental.

Estrategia persistida

Los totales se almacenan en la tabla principal del documento.

Ejemplo de campos:

subtotal
tax_total
total

Esto se utiliza cuando el registro cumple función documental o contable.

Convención para catálogos funcionales

Los estados, tipos y variantes funcionales de cada módulo no deben repetirse como strings sueltos en controllers, vistas, seeders o filtros.

Cada módulo que tenga catálogos funcionales (status, kind, type, etc.) debe centralizarlos en una única fuente reutilizable.

Implementación recomendada:

app/Support/Catalogs

Esa fuente debe ser utilizada por:

validaciones
formularios
badges
filtros
seeders

Si un dominio funcional ya existe en otro módulo, debe reutilizarse y no duplicarse.

No deben crearse catálogos nuevos si uno existente cubre el mismo dominio conceptual.

Gestión de módulos reemplazados

Cuando un módulo sea reemplazado por otro más general o por una entidad más adecuada, el módulo anterior debe eliminarse.

No deben mantenerse módulos “obsoletos” dentro del sistema.

Terminología de interfaz

La interfaz debe utilizar términos orientados al usuario.

Equivalencias:

Tenant → Empresa
Party → Contacto

La nomenclatura técnica puede mantenerse en el código.

Selector de empresa

Un usuario puede pertenecer a múltiples empresas.

Reglas:

si tiene una sola empresa se accede directamente
si tiene varias puede cambiar mediante selector en el header

Páginas públicas

Algunas páginas no utilizan el layout completo.

Ejemplos:

welcome
login
register
selección de empresa

Estas páginas utilizan:

$publicPage = true

Principios generales

Simplicidad
Evitar complejidad innecesaria.

Consistencia
Seguir siempre las convenciones definidas.

Escalabilidad
Permitir evolución futura del sistema.

Claridad
El código debe ser fácil de entender y mantener.

Regla de decisiones arquitectónicas

Si durante el desarrollo aparece una decisión que afectará a todo el sistema:

no improvisar.

Registrar la decisión en este documento antes de continuar.
