Objetivo

Este checklist define el procedimiento estándar para crear nuevos módulos dentro del proyecto app-base.

Su finalidad es:

mantener consistencia entre módulos
garantizar compatibilidad multi-tenant
evitar errores estructurales
acelerar el desarrollo
mantener coherencia en todo el sistema

Este checklist aplica a todos los módulos CRUD del sistema.

Evaluación de la entidad (ANTES de crear el módulo)

Antes de crear una nueva entidad verificar:

si el caso puede resolverse con la entidad parties

si corresponde a una extensión de contacts en la interfaz

si realmente es una entidad nueva del sistema

Recordar:

parties es la entidad base del negocio.

Si la entidad no es necesaria en esta etapa:

registrarla en TODO-DEL-PROYECTO.

Crear migración

Crear la migración correspondiente.

Ejemplo:

php artisan make:migration create_products_table

Reglas obligatorias para entidades del negocio:

toda tabla debe incluir:

tenant_id
timestamps
softDeletes

Ejemplo mínimo:

$table->uuid('tenant_id')->index();
$table->string('name');
$table->timestamps();
$table->softDeletes();

Índices recomendados:

siempre indexar tenant_id
indexar campos utilizados en búsquedas

Ejemplo:

$table->index(['tenant_id','name']);

Crear modelo

Crear modelo:

php artisan make:model Product

El modelo debe incluir:

HasFactory
TenantScoped
SoftDeletes
ResolvesTenantRouteBinding

Ejemplo de traits utilizados:

use Illuminate\Database\Eloquent\Factories\HasFactory;
use App\Models\Concerns\TenantScoped;
use App\Models\Concerns\ResolvesTenantRouteBinding;
use Illuminate\Database\Eloquent\SoftDeletes;

Motivación:

HasFactory permite generar datos de prueba
TenantScoped garantiza aislamiento multi-tenant
SoftDeletes permite eliminación lógica
ResolvesTenantRouteBinding protege el acceso por URL

Definir fillable.

Ejemplo:

protected $fillable = [
'tenant_id',
'name',
'description',
];

Crear controller

Crear controller tipo resource.

Ejemplo:

php artisan make:controller ProductController

El controller debe implementar:

index
create
store
show
edit
update
destroy

Regla de tenant:

cuando sea necesario obtener el tenant actual:

$tenant = app('tenant');

Definir rutas

Registrar rutas en:

routes/web.php

Ejemplo:

Route::resource('products', ProductController::class);

Las rutas deben quedar dentro del grupo protegido por:

auth
tenant

Crear vistas

Crear carpeta del módulo:

resources/views/products

Vistas mínimas:

index.blade.php
create.blade.php
edit.blade.php
show.blade.php
_form.blade.php

Convención de listados (index)

Los listados deben seguir el patrón visual del sistema.

Estructura:

page.list-page
card.list-card
table-wrap.list-scroll

Reglas del listado:

el nombre enlaza a la vista show
incluir botón "Nuevo"
mostrar estados vacíos dentro de la card

Convención de show

La vista de detalle debe incluir:

editar
eliminar
volver al listado
detalle de campos

El botón eliminar debe utilizar:

POST + method DELETE

y confirmación de usuario.

Convención de formularios

Los formularios deben reutilizar:

_form.blade.php

Este archivo debe ser utilizado por:

create
edit

Validación

La validación se realiza inicialmente en el controller.

Ejemplo:

$request->validate([
'name' => ['required','string','max:255'],
]);

Si el módulo crece se puede migrar a FormRequest.

Mensajes de éxito

Usar mensajes simples y consistentes.

Ejemplos:

Entidad creada correctamente
Entidad actualizada correctamente
Entidad eliminada correctamente

Prueba funcional mínima

Verificar funcionamiento del módulo:

listado
creación
edición
eliminación

Verificación de aislamiento multi-tenant

Confirmar que:

un tenant no puede ver datos de otro tenant.

Este comportamiento es gestionado por:

TenantScoped

pero debe verificarse manualmente.

Datos de prueba

Si el módulo es relevante para el sistema:

crear factory.

Ejemplo:

php artisan make:factory ProductFactory

Agregar datos al DemoSeeder.

Esto permite reconstruir el entorno con:

php artisan migrate:fresh --seed

Actualizar documentos del proyecto

Si se crea un módulo nuevo actualizar:

MAPA-DE-ENTIDADES
TODO-DEL-PROYECTO

Regla de simplicidad

Antes de agregar:

roles complejos
subtablas
polimorfismos
arquitecturas complejas

preguntarse:

¿esto es necesario ahora?

Si no lo es:

registrarlo en TODO-DEL-PROYECTO.

Regla de coherencia

Cada módulo nuevo debe seguir el mismo patrón que:

projects
parties

Esto garantiza coherencia en:

estructura
rutas
vistas
flujo de trabajo

Flujo resumido para crear módulo

evaluar entidad
crear migración
crear modelo
crear controller
definir rutas
crear vistas
validar funcionamiento
probar aislamiento tenant
actualizar documentos

Regla final

Si una decisión afecta a toda la arquitectura del sistema:

no improvisar.

Registrar la decisión en:

CONVENCIONES-DEL-PROYECTO

y luego continuar el desarrollo.
