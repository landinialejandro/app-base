<<SECTION: FLUJO GLOBAL DE ACCESO>>
FLUJO GLOBAL DE ACCESO

El flujo base del sistema es:

login
↓
selección de empresa (/tenants/select)
↓
session('tenant_id')
↓
ResolveTenant
↓
acceso operativo al sistema

Reglas:

el usuario siempre se autentica a nivel global

el acceso real ocurre recién al seleccionar un tenant válido

no debe existir acceso operativo sin tenant activo

ResolveTenant debe validar membership activa antes de permitir acceso

La descripción estructural base de este flujo debe mantenerse alineada con:

CONTEXTO-FIJO-DEL-PROYECTO – app-base
MAPA-DE-ENTIDADES – app-base
<<END SECTION>>

<<SECTION: MODELO DE IDENTIDAD Y PERTENENCIA>>
MODELO DE IDENTIDAD Y PERTENENCIA

El acceso del sistema se apoya en la relación:

users
↓
memberships
↓
tenants

Reglas de seguridad:

users representa identidad global

memberships representa acceso real por empresa

memberships.status = active define acceso efectivo

memberships.is_owner define condición estructural de owner

owner no es un rol asignable ordinario

owner es único por tenant

La descripción conceptual de estas entidades y sus relaciones debe mantenerse alineada con:

MAPA-DE-ENTIDADES – app-base
CONTEXTO-FIJO-DEL-PROYECTO – app-base
<<END SECTION>>

<<SECTION: AUTENTICACIÓN Y FORTIFY>>
AUTENTICACIÓN Y FORTIFY

El sistema utiliza Laravel Fortify como base de autenticación.

Responsabilidades:

login
logout
recuperación de contraseña
remember session cuando corresponda

Reglas:

el login autentica identidad global (User), no acceso a tenant
la selección de tenant es obligatoria posterior al login
LoginResponse debe redirigir únicamente a tenants con membership activa
no debe existir autoingreso a tenants bloqueados

Logout:

debe eliminar completamente el contexto de tenant
no deben quedar residuos de session('tenant_id')
no debe mantenerse contexto operativo luego del cierre

Two-Factor Authentication:

Actualmente el sistema no incorpora un flujo activo de Two-Factor Authentication.

Fortify debe operar sin 2FA habilitado hasta que exista una implementación completa, consistente y validada en modelo, columnas y flujo.

No debe quedar ningún estado parcialmente activo.
<<END SECTION>>

<<SECTION: MULTI-TENANCY Y AISLAMIENTO>>
MULTI-TENANCY Y AISLAMIENTO

La seguridad multi-tenant se basa en:

tenant_id en entidades del negocio
TenantScoped
ResolvesTenantRouteBinding
resolución de tenant mediante middleware

Reglas:

todas las consultas deben respetar tenant_id

no deben existir queries sin tenant activo

no debe exponerse tenant_id en formularios

tenant se obtiene exclusivamente desde app('tenant')

route model binding debe validar tenant

no se permite acceso a registros de otro tenant por URL

los registros eliminados mediante SoftDeletes no deben ser accesibles

La doctrina estructural de multi-tenancy debe mantenerse alineada con:

CONTEXTO-FIJO-DEL-PROYECTO – app-base
CONVENCIONES-DEL-PROYECTO – app-base
<<END SECTION>>

<<SECTION: ROLES BASE Y MODELO DE AUTORIZACIÓN>>
ROLES BASE Y MODELO DE AUTORIZACIÓN

Los roles base del tenant son:

owner
admin
sales
operator
administrator

Reglas:

owner se resuelve por memberships.is_owner

owner no es delegable

owner es único por tenant

los roles delegables actuales son:

admin
sales
operator
administrator

administrator representa rol administrativo de oficina, no superadmin

la membership puede incluir profile_slug como especialización funcional

la autorización no depende solo de rol:

debe considerar capacidad
scope
constraints

La definición estructural de roles y pertenencia debe mantenerse alineada con:

<<SECTION: CAPA V2 DE PERMISOS>>
CAPA V2 DE PERMISOS

La autorización se basa en una capa central compuesta por:

TenantModuleAccess
RolePermissionResolver
RecordScopeResolver
CapabilityCatalog
ModuleCatalog

Capacidades vigentes:

view_any
view
create
update
delete

Reglas:

create es booleano

view_any, view, update y delete dependen de scope válido

los scopes válidos del estado actual son:

tenant_all
own_assigned
limited

no deben usarse scopes legacy como:

all
mine

view_any no es booleano

los scopes se definen por módulo + capacidad

limited requiere resolver explícito

constraints forman parte obligatoria de la autorización

allowed_kinds es la primera constraint consolidada

orders y documents son módulos compartidos del tenant

no deben resolverse por ownership artificial del registro

su segmentación fina se basa en constraints.allowed_kinds

La doctrina completa de V2 debe mantenerse alineada con:

CONTEXTO-FIJO-DEL-PROYECTO – app-base
CONVENCIONES-DEL-PROYECTO – app-base
MAPA-DE-ENTIDADES – app-base
<<END SECTION>>

<<SECTION: FLUJO DE GESTIÓN Y CONSUMO DE PERMISOS>>
OBJETIVO

Definir el flujo operativo de gestión y consumo de permisos dentro del sistema.

FLUJO GENERAL

superadmin define los módulos habilitados por tenant

owner configura los permisos sobre los módulos habilitados

los permisos se definen por:

rol
módulo
capacidad
scope

el backend resuelve:

acceso al módulo
universo visible de registros
acciones permitidas sobre cada registro

la UI:

no redefine permisos
no altera el universo visible
solo filtra dentro del universo definido por el backend
solo muestra acciones que el usuario puede ejecutar realmente

REGLA DE UNIVERSO Y ACCIONES

el universo visible y la capacidad de operar sobre un registro son conceptos distintos

un usuario puede:

ver un registro sin poder modificarlo

las acciones deben validarse siempre contra la capacidad correspondiente (view, update, delete, create)

REGLA DE SCOPES

los scopes no se aplican de forma uniforme a todas las capacidades

cada módulo define explícitamente qué scopes admite para cada capacidad

un mismo módulo puede tener:

view basado en limited
update basado en own_assigned

los scopes representan subconjuntos de tenant_all

no todos los módulos deben usar limited

si un módulo no utiliza limited, su contrato debe definirse explícitamente con los scopes que sí admite

REGLA DE MÓDULOS ACTIVOS

solo deben considerarse los módulos habilitados por superadmin para el tenant

no deben mostrarse ni evaluarse permisos sobre módulos inactivos

toda relación, acción o campo que dependa de otro módulo debe verificar este estado antes de resolverse

si el módulo destino está deshabilitado:

no debe evaluarse autorización
no debe exponerse en UI
no debe mantenerse acceso residual

REGLA DE CONSISTENCIA

la lógica de permisos debe mantenerse alineada entre:

resolver
policy
controller
Blade

no debe existir lógica paralela o contradictoria entre capas

view_any debe ser igual o más amplio que view
<<END SECTION>>

<<SECTION: REGLAS OPERATIVAS POR MÓDULO>>
REGLAS OPERATIVAS POR MÓDULO

OBJETIVO

Definir de forma explícita el comportamiento funcional y de autorización de cada módulo del sistema.

Cada módulo debe establecer:

su universo visible

cómo interpreta el alcance LIMITED

reglas de creación

reglas de actualización

reglas de eliminación

atributo que define responsabilidad operativa (si aplica)

REGLA GENERAL

Para todos los módulos:

el backend define completamente el universo visible del usuario

la UI solo puede aplicar filtros dentro de ese universo

no debe existir lógica paralela de visibilidad en controllers o Blade

no debe utilizarse ownership artificial si el dominio no lo requiere

los scopes representan subconjuntos de tenant_all

LIMITED no es un patrón genérico reutilizable automáticamente

LIMITED debe definirse explícitamente por módulo

REGLA DE CONSISTENCIA GLOBAL

Cada módulo debe:

definir explícitamente su comportamiento

no depender de interpretaciones implícitas de scope

no delegar lógica de visibilidad a la UI

mantener alineación entre:

resolver
policy
controller
Blade

view_any debe ser igual o más amplio que view

Cualquier nuevo módulo que utilice LIMITED debe documentar su comportamiento en una sección independiente dentro de este documento.
<<END SECTION>>

<<SECTION: MÓDULO: PROJECTS>>
MÓDULO: PROJECTS

UNIVERSO VISIBLE

tenant_all:
todos los proyectos del tenant

limited:
solo proyectos donde el usuario tiene al menos una tarea asignada dentro de ese proyecto

REGLAS

la visibilidad de proyectos depende de la participación operativa en tareas del proyecto

no se utiliza ownership del proyecto

no se utiliza created_by como criterio de acceso

<<END SECTION>>

<<SECTION: MÓDULO: TASKS>>
MÓDULO: TASKS

UNIVERSO VISIBLE

tenant_all:
todas las tareas del tenant

limited:
tareas asignadas al usuario
+
todas las tareas de proyectos donde el usuario participa por tener al menos una tarea asignada

own_assigned:
solo tareas asignadas al usuario

REGLAS DE VISIBILIDAD

la visibilidad no depende del creador

la visibilidad se basa en asignación operativa (assigned_user_id)

las tareas sin proyecto solo son visibles para el usuario asignado

las tareas con proyecto visible se amplían a todo el proyecto

CONFIGURACIÓN DE CAPACIDADES

view_any:
limited

view:
limited

update:
own_assigned

delete:
own_assigned

REGLA DE CREACIÓN

toda tarea debe crearse con assigned_user_id obligatorio

si no se define explícitamente un usuario asignado:

la tarea debe autoasignarse al usuario que la crea

no deben existir tareas sin asignación

REGLA DE ACTUALIZACIÓN

la actualización se limita a tareas asignadas al usuario

no se permite editar tareas visibles por participación en proyecto si no están asignadas al usuario

REGLA DE ELIMINACIÓN

la eliminación se limita a tareas asignadas al usuario

no se permite eliminar tareas visibles por participación en proyecto si no están asignadas al usuario

<<END SECTION>>

<<SECTION: MÓDULO: APPOINTMENTS>>
MÓDULO: APPOINTMENTS

appointments no utiliza semántica limited como contrato del módulo

create:
true

tenant_all

delete:
tenant_all

roles operativos habilitados, por ejemplo operator y sales

view_any:
tenant_all

own_assigned

create:
true

update:
own_assigned

delete:
own_assigned

REGLA FUNCIONAL

los roles operativos pueden ver el listado completo de agenda cuando así se configure

REGLA DE CONSISTENCIA

<<SECTION: REGLAS DE IMPLEMENTACIÓN SEGURA>>
REGLAS DE IMPLEMENTACIÓN SEGURA

La autorización debe implementarse obligatoriamente en cuatro niveles:

resolver central
policy
controller
Blade

Regla obligatoria:

policy
authorize(...) en controller
@can en Blade

Roles de cada capa:

resolver central
resuelve acceso efectivo, scope, constraints y dataset

policy
wrapper mínimo sobre la capa central

controller
ejecuta authorize(...)
no implementa lógica paralela de permisos
consume dataset ya resuelto desde la capa central

Blade
usa @can para visibilidad final
no reconstruye datasets ni permisos por fuera de la capa central

Distinción obligatoria:

view_any controla dataset
view controla acceso a registro individual

No usar view para controlar listados.
<<END SECTION>>

<<SECTION: RECURSOS DELEGADOS AL PADRE>>
RECURSOS DELEGADOS AL PADRE

Los siguientes recursos no deben tratarse como autorización autónoma:

attachments
order_items
document_items

Reglas:

no deben tener lógica de permisos propia cuando su ciclo de vida dependa naturalmente del padre

create, update y delete dependen de update del recurso padre

Ejemplos:

attachment depende de su attachable
order_item depende de order
document_item depende de document

En UI:

no evaluar create Attachment

evaluar update del modelo padre correspondiente

Esta regla debe mantenerse alineada con:

CONTEXTO-FIJO-DEL-PROYECTO – app-base
CONVENCIONES-DEL-PROYECTO – app-base
MAPA-DE-ENTIDADES – app-base
<<END SECTION>>

<<SECTION: FLUJO DE INVITACIONES Y ONBOARDING>>
FLUJO DE INVITACIONES Y ONBOARDING

Tipos de invitación:

owner_signup
member_invite

Escenarios:

usuario nuevo
usuario existente

Reglas:

si el usuario no está autenticado → redirigir a login
luego del login → retomar flujo original

El sistema debe:

crear Membership correspondiente
no dejar usuarios sin acceso a tenant
no romper continuidad del flujo de onboarding

El flujo de invitación es parte crítica del acceso seguro al sistema.
<<END SECTION>>

<<SECTION: PANTALLAS SENSIBLES DE SEGURIDAD>>
PANTALLAS SENSIBLES DE SEGURIDAD

Se consideran pantallas críticas:

login
tenant select
tenant.profile
tab de permisos
flujo de invitaciones
pantallas de superadmin

Reglas:

solo owner puede gestionar permisos del tenant
los módulos visibles dependen de superadmin
no exponer configuraciones críticas a roles no autorizados

Las pantallas sensibles deben:

usar catálogos cerrados
evitar combinaciones libres peligrosas
priorizar claridad sobre flexibilidad
<<END SECTION>>

<<SECTION: HARDENING Y AUDITORÍA>>
HARDENING Y AUDITORÍA

Puntos obligatorios de revisión:

LoginResponse
validar acceso solo a memberships activas

Logout
limpiar contexto tenant completamente

User
proteger atributos sensibles (ej. is_superadmin)

Creación de usuarios
garantizar creación de membership

Fortify
definir estado real de 2FA

Rutas de debug
no exponer en producción

Permisos
no permitir scopes inválidos en base

Dataset
no usar lógica paralela fuera del resolver

Members
no dejar memberships sin rol

Seguridad es responsabilidad transversal del sistema y no de un único módulo.
<<END SECTION>>

<<SECTION: REGLA FINAL>>
REGLA FINAL

El sistema debe garantizar que:

ningún usuario acceda a datos de otro tenant
ninguna acción se ejecute sin autorización real
ninguna UI muestre acciones no permitidas
ninguna capa duplique lógica de permisos

La seguridad no es una capa adicional.

Es una responsabilidad transversal que debe cumplirse en:

modelo
middleware
resolver
policy
controller
Blade
UI

Este documento es la fuente principal del proyecto para acceso y seguridad.

Toda implementación nueva que toque autenticación, tenant, permisos, Fortify, onboarding o autorización debe respetar este documento y mantenerse alineada con los documentos estructurales del proyecto.
<<END SECTION>>
