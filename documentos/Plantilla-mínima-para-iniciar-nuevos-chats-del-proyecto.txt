ARQUITECTURA DEL SISTEMA

Tipo de sistema:

Plataforma SaaS multi-tenant desarrollada en Laravel.

Aislamiento de datos mediante:

tenant_id

Traits obligatorios en entidades del negocio:

TenantScoped
ResolvesTenantRouteBinding
SoftDeletes

Fuente del tenant operativo:

app('tenant')

La arquitectura multi-tenant ya está implementada y no debe modificarse.

Convención de base de datos:

tenants.id es char(36)
tenant_id no debe asumirse como foreignId()
copiar siempre el patrón real validado del proyecto antes de crear nuevas migraciones

MODELO DEL NEGOCIO

users es entidad del sistema.

parties es la entidad base del negocio.

Nombre técnico:

parties

Nombre visible en la interfaz:

contactos

El acceso por empresa se resuelve por memberships.

users representa identidad global.
memberships representa acceso por tenant.

projects ya está validado como módulo con estado mínimo active/closed.

projects.index funciona como listado operativo con métricas por proyecto y filtro por estado.

BASE VISUAL

Las páginas internas utilizan el layout estándar:

<x-page> <x-breadcrumb /> <x-page-header /> <x-card /> </x-page>

Patrón de listados:

page.list-page
card.list-card
table-wrap.list-scroll

Existen patrones recientes validados para vistas show:

summary-inline-grid
detail-grid
detail-grid--3
tabs internas para información secundaria
badges de estado resueltos desde catálogo en el core

Convenciones recientes de frontend:

evitar handlers JS inline en Blade
preferir data-action="app-..." para comportamiento reutilizable
resolver comportamiento reusable en app-base.js
resolver comportamiento puntual de pantalla con @push('scripts')
en vistas con tabs, preservar contexto activo cuando mejore la UX
en tablas con compact-actions, usar solo iconos
fuera de tablas, los botones principales pueden usar icono + texto
usar Contacto como nombre visible genérico de parties
usar Eliminar como verbo destructivo estándar
usar title y aria-label en botones compactos con solo icono

Criterios operativos recientes de UX:

en index, priorizar toolbar compacto con hasta 2 filtros visibles

en index, cuando mejore la lectura, separar filtros y resultados en dos cards

en show, priorizar card principal compacta y colapsar detalle secundario cuando reduzca altura visual

usar paginación propia del proyecto y evitar markup visual externo al sistema

Convenciones reusables recientes por módulo:

usar partials/table.blade.php para listados puros del módulo

usar partials/embedded-tabs.blade.php cuando el mismo listado deba agruparse por tabs dentro de una vista embebida o show

reutilizar la misma tabla base del módulo antes de crear variantes duplicadas en index y show

usar tabs por tipo o por estado solo cuando el módulo realmente lo justifique

REGLA DE DESARROLLO

Los nuevos módulos deben seguir el patrón utilizado en:

models
controllers
routes
vistas
estructura CRUD

Si el módulo tiene cabecera y detalle:

diseñar primero la estructura y relaciones
luego crear migraciones
luego implementar cabecera
luego implementar ítems

CATÁLOGOS FUNCIONALES

Antes de implementar campos con valores cerrados (por ejemplo status, kind o type):

verificar si el sistema ya dispone de un catálogo reutilizable
si corresponde, reutilizar o extender los catálogos existentes en:

app/Support/Catalogs

Evitar definir strings hardcodeados directamente en controladores, vistas o seeders.
