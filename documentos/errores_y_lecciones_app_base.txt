LECCIONES VIGENTES

Las lecciones vigentes del proyecto deben orientar decisiones futuras para evitar repetir errores ya detectados.

NO ASUMIR UTILITIES CSS NO VALIDADAS

Una clase visual tipo utility no debe asumirse disponible si no está validada en la base CSS real.

Caso detectado:

class="d-block"

No produjo el salto de línea esperado en los Blades compactos de ProductOperationalSummary.

Corrección aplicada:

reemplazar span + d-block por div

Lección:

cuando se necesita comportamiento estructural HTML seguro, preferir markup semántico simple antes que depender de una utility CSS no validada.

Regla:

si una clase CSS no está confirmada en la base visual real, no debe usarse como solución estructural obligatoria.

PRIMER PILOTO FUNCIONAL NO SIEMPRE ES CIERRE ARQUITECTÓNICO

Una primera implementación puede funcionar visual o técnicamente y, aun así, no representar el cierre arquitectónico correcto.

Caso detectado:

la primera implementación de ProductOperationalSummary como include directo funcionaba, pero no respetaba plenamente el patrón surface-driven ni la identidad de subcapa contextual.

Corrección aplicada:

mover salida visual a ProductSurfaceService
eliminar include directo
organizar archivos en carpeta propia

Ubicaciones finales:

app/Support/Products/OperationalSummary/
resources/views/products/operational-summary/

Lección:

si existe infraestructura surface-driven para el host, la salida estable debe integrarse mediante surface del módulo oferente y no quedar como include local disperso.

Regla:

funcionar no alcanza.

Debe respetar arquitectura, ownership, superficie de integración, seguridad y documentación vigente.

NO CALCULAR MÉTRICAS EN BLADE

Las vistas deben representar payload ya preparado por servicios o capas de lectura.

Caso vigente:

ProductOperationalSummary publica vistas compactas para summary_items.

Regla:

los Blades compactos no deben calcular compras, ventas, entradas, salidas, proveedor actual, proveedor anterior ni promedios.

Deben recibir datos ya resueltos.

Lección:

cuando una vista necesita métricas operativas, debe existir una capa de lectura previa.

El Blade no debe transformarse en servicio, query builder ni autorizador.

NO CONFUNDIR SUBCAPA CONTEXTUAL CON MÓDULO NUEVO

Una capacidad visual o analítica alojada dentro de un módulo host no debe convertirse automáticamente en módulo nuevo.

Caso vigente:

ProductOperationalSummary

Lectura correcta:

subcapa contextual interna de Products
sin CRUD
sin navegación propia
sin ModuleCatalog propio
sin dashboard general
sin rutas propias

Lección:

antes de abrir módulo, tabla, ruta o dashboard, debe revisarse si la necesidad puede vivir como subcapa interna del módulo host y publicarse por surface.

NO USAR OPERATIONALACTIVITY COMO FUENTE DE MÉTRICAS CUANDO EXISTE MÓDULO DUEÑO

OperationalActivity registra trazabilidad transversal.

No debe usarse como fuente principal de métricas si existen módulos dueños de los datos transaccionales u operativos.

Caso vigente:

ProductOperationalSummary debe usar Orders e Inventory como fuentes principales.

OperationalActivity conserva trazabilidad, pero no aporta la métrica principal.

Lección:

trazabilidad no reemplaza datos operativos del módulo dueño.

MIGRATE FRESH SEED NO CIERRA FLUJOS RUNTIME

Un flujo puede pasar lint, migrate:fresh y seeders, pero fallar en runtime web real.

Caso detectado:

Order → Document falló por dependencia a DocumentTotalsCalculator::apply(), método inexistente para ese flujo.

El error apareció en prueba web real.

Corrección aplicada:

el flujo Order → Document recalcula totales desde document_items luego de copiar ítems.

Lección:

migrate:fresh --seed no alcanza para cerrar flujos derivados con controllers, requests, autorización, copia de líneas, cálculos y redirecciones.

Regla:

todo flujo derivado relevante debe validarse en runtime web real antes de considerarse cerrado.

UN VÍNCULO NULLABLE NO DEBE REINTRODUCIRSE COMO REQUIRED SIN DOCTRINA

Un campo relacional nullable puede ser vínculo opcional y no dependencia obligatoria.

Caso detectado:

Orders necesitaba poder crearse sin party_id usando counterparty_name.

Documents necesitaba poder formalizarse desde Orders autónomas sin depender rígidamente de party_id.

Corrección aplicada:

se separó vínculo gestionado opcional y snapshot textual propio.

Patrón vigente:

party_id = vínculo gestionado opcional

counterparty_name = snapshot textual propio

Lección:

un `*_id` no debe tratarse automáticamente como required por existir en el modelo o por conveniencia visual.

Regla:

antes de exigir un vínculo en controller o request, validar esquema, contrato funcional, doctrina y autonomía del módulo.

NO UBICAR RELACIONES DE ORIGEN O SEGUIMIENTO EN EL MÓDULO RECEPTOR POR COMODIDAD

La relación persistente debe vivir en el módulo que conserva ownership real del concepto.

Caso detectado:

orders.task_id ubicaba en Orders una relación de seguimiento que pertenece funcionalmente a Tasks.

Corrección aplicada:

la relación se movió a tasks.order_id.

Lectura vigente:

Tasks organiza pendientes y seguimiento.

Orders ejecuta intervención operativa.

Una Order puede tener múltiples Tasks asociadas desde la lectura funcional del ecosistema.

Lección:

si una relación expresa origen, planificación o seguimiento, no debe ubicarse automáticamente en el módulo receptor.

Regla:

validar siempre qué módulo conserva ownership conceptual antes de persistir el vínculo.

LOS SNAPSHOTS DEBEN LLEGAR A SEEDERS, PRINTS, LISTADOS Y SURFACES

Agregar un snapshot al modelo no cierra el saneamiento si las demás salidas siguen dependiendo solo del vínculo gestionado.

Caso vigente:

Orders y Documents incorporan counterparty_name.

Riesgo detectado:

listados, prints, show summaries, surfaces o componentes linked pueden seguir mostrando solo party->name y dejar vacío cuando party_id sea null.

Lección:

la autonomía del módulo debe verse en todas sus salidas.

Regla:

cuando exista método de presentación propio, debe usarse en listados, show summaries, prints, surfaces y salidas contextuales.

Casos vigentes:

Order::displayCounterpartyName()

Document::displayCounterpartyName()

PARTYLINKED NO ES FUENTE PRINCIPAL DE CONTRAPARTE

PartyLinked representa vínculo gestionado con Parties.

No debe usarse como fuente principal de texto visible cuando el módulo dueño posee snapshot propio de contraparte.

Caso vigente:

Orders y Documents pueden tener counterparty_name aunque party_id sea null.

Lección:

un componente linked no debe ocultar la autonomía visual del módulo dueño.

Reglas:

PartyLinked se usa cuando existe party_id y corresponde mostrar vínculo gestionado

si no existe party_id pero existe counterparty_name, debe mostrarse texto propio

no inventar link hacia Party

no mostrar vacío si el módulo dueño posee snapshot

ASSETLINKED NO ES FUENTE PRINCIPAL DE ACTIVO SI EXISTE REFERENCIA PROPIA

AssetLinked representa vínculo gestionado con Assets.

No debe usarse como fuente principal de texto visible cuando el módulo dueño posee referencia propia de activo.

Caso vigente:

Orders puede representar Activo mediante asset_id y asset_reference.

Lección:

un componente linked no debe ocultar la autonomía visual del módulo dueño.

Reglas:

AssetLinked se usa cuando existe asset_id y corresponde mostrar vínculo gestionado

si no existe asset_id pero existe asset_reference, debe mostrarse texto propio

no inventar link hacia Asset

no mostrar vacío si el módulo dueño posee referencia propia

SHOWPARTY NO DEBE SEGUIR COMO CONTRATO PRINCIPAL SI EL DATO ES CONTRAPARTE

Un nombre de contrato visual incorrecto genera confusión conceptual y acoplamiento.

Caso detectado:

showParty podía interpretarse como columna de vínculo gestionado a Party, cuando el dato visible real era Contraparte.

Corrección vigente:

showCounterparty es el contrato visual para columnas de contraparte en Orders y Documents.

Lección:

el nombre del contrato visual debe expresar el dato operativo visible, no necesariamente la entidad gestionada relacionada.

Regla:

usar Contraparte cuando se representa dato propio del registro.

Usar Contacto cuando se filtra o selecciona Party gestionada.

NO DEJAR FLAGS VISUALES SIN REPRESENTACIÓN EN TABLAS REUSABLES

Un flag publicado por SurfaceService o payload no puede quedar sin efecto real en el partial consumidor.

Caso detectado:

showCounterparty y showAsset debían tener representación efectiva en orders.partials.table.

Lección:

un flag visual sin representación real es contrato fantasma.

Regla:

si se publica un flag visual, debe implementarse en la vista final o eliminarse del contrato.

PUBLICAR CATÁLOGO NO ES OPERAR COMERCIALMENTE

Una tienda pública puede mostrar catálogo sin habilitar todavía compra, saldo, fichas, QR, pago ni carrito.

Caso vigente:

self_service_sales muestra catálogo publicado inicial mediante SelfServiceShop y SelfServiceShopItem.

Lección:

la publicación visual de productos no debe confundirse con operación comercial real.

Reglas:

publicar producto no equivale a vender

publicar producto no mueve stock

publicar producto no crea documento

publicar producto no habilita carrito

activar tienda no habilita operación comercial completa

catálogo publicado no habilita operación comercial

operation_enabled = false mantiene bloqueada la operación comercial

NO MEZCLAR ADMINISTRACIÓN INTERNA DE TIENDA CON PLANO CUSTOMER

Una capa satélite con entrada pública puede tener administración interna, pero ambos planos deben permanecer estrictamente separados.

Caso vigente:

self_service_sales tiene plano externo customer y administración interna de tienda.

Plano externo:

/tienda
/tienda/seleccionar
/shop/{tenant:slug}
/tienda/salir
SelfServiceExternalSession
ResolveSelfServiceExternalCustomer

Plano interno:

self_service_sales.shops.*
SelfServiceShop
SelfServiceShopItem
TenantProfileAccess::canManageSelfServiceSales()

Lección:

la existencia de sesión externa mínima no autoriza gestión interna.

Reglas:

customer externo no administra tienda

SelfServiceExternalSession no autoriza self_service_sales.shops.*

ResolveSelfServiceExternalCustomer no autoriza self_service_sales.shops.*

/shop/{tenant:slug} no es administración interna

/self-service/shops... no es tienda pública

NO USAR ENTIDADES SATÉLITE COMO REEMPLAZO DEL MÓDULO DUEÑO

Una entidad auxiliar de capa satélite puede representar una publicación, configuración o relación específica, pero no reemplaza al módulo dueño del dato base.

Caso vigente:

SelfServiceShopItem publica productos en tienda.

Product conserva catálogo maestro.

Lección:

la publicación satélite no debe absorber catálogo, stock, documento ni operación.

Reglas:

SelfServiceShop no reemplaza Tenant

SelfServiceShopItem no reemplaza Product

self_service_sales no redefine catálogo maestro

Products conserva ownership de productos

Inventory conserva ownership de stock

Documents conserva ownership documental

NO PROMETER EN UI ACCIONES NO OPERABLES

La UI puede anticipar una capacidad futura solo si deja claro que no está operativa.

Caso vigente:

botón deshabilitado de operación en preparación dentro del catálogo publicado inicial.

Lección:

la claridad visual evita ambigüedad operativa y reduce falsas expectativas.

Reglas:

si compra no existe, no mostrar compra como acción disponible

si pago no existe, no mostrar pago como acción disponible

si saldo no existe, no mostrar saldo como operativo

si QR no existe, no mostrar consumo QR como operativo

si operación está bloqueada, debe expresarse como bloqueada o pendiente
