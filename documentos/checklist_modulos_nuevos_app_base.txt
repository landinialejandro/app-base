<<SECTION: OBJETIVO>>
SECTION_VERSION: 00003

OBJETIVO

Este documento define el checklist obligatorio para crear, incorporar o reconvertir módulos dentro de app-base.

Su función es:

reducir omisiones
forzar consistencia arquitectónica
alinear seguridad, dominio y UI
evitar módulos incompletos
mantener coherencia entre implementaciones nuevas y existentes

No es una guía opcional.

Es un control mínimo obligatorio antes de considerar un módulo como integrado al sistema.

Aclaración doctrinal consolidada:

este checklist aplica tanto a módulos nuevos como a reconversiones profundas de módulos existentes cuando cambie su contrato funcional, contractual, de modelado o de seguridad

<<END SECTION>>

<<SECTION: REGLA GENERAL>>
SECTION_VERSION: 00003

REGLA GENERAL

Ningún módulo debe considerarse terminado por tener tablas, rutas o CRUD funcionando.

Un módulo solo puede considerarse integrado cuando supera este checklist en forma coherente.

Orden correcto:

definir contrato conceptual
definir contrato de seguridad
definir relaciones reales necesarias
definir superficie de integración
definir si participa en trazabilidad operativa transversal
implementar
auditar
ajustar documentación afectada

Nunca al revés.

Regla adicional consolidada:

si una integración nueva requiere reabrir arbitrariamente el núcleo de otro módulo, primero debe revisarse si la necesidad puede resolverse consumiendo una capacidad ya publicada por ese módulo o evolucionando su superficie contractual propia

Regla adicional para actividad operativa:

no todo módulo debe registrar actividad por defecto.

La participación en OperationalActivity debe justificarse por valor operativo real.

No debe registrarse actividad solo por existir CRUD, eventos técnicos o cambios persistidos sin significancia operativa.

<<END SECTION>>

<<SECTION: BLOQUE 0 — IDENTIDAD DEL MÓDULO>>
SECTION_VERSION: 00002

BLOQUE 0 — IDENTIDAD DEL MÓDULO

Antes de cualquier decisión técnica debe quedar definida la identidad del módulo.

Validar:

nombre del módulo
problema real que resuelve
naturaleza principal del módulo
posición conceptual en el ecosistema
si es módulo nuevo o reconversión de uno existente
si es módulo funcional, entidad hija, capacidad transversal de soporte o pieza interna del tenant

Clasificación conceptual posible:

módulo base del dominio
módulo transversal
módulo autónomo
módulo agrupador
módulo derivado
capacidad transversal de soporte

Preguntas obligatorias:

este módulo tiene identidad propia o es solo una entidad hija
es realmente un módulo o una evolución de otro ya existente
debe vivir por sí mismo o principalmente embebido
qué rol ocupa dentro del ecosistema general
registra, acompaña o sostiene actividad de otros módulos sin absorber su ownership
debe tener navegación principal o solo consumo interno/transversal

Si estas respuestas no están claras, no debe avanzarse a implementación.

Caso vigente:

OperationalActivity se lee como capacidad transversal de soporte tenant-scoped para trazabilidad operativa.

No se lee como CRUD ordinario ni como módulo navegable principal.

<<END SECTION>>

<<SECTION: BLOQUE 1 — CONTRATO MODULAR>>
SECTION_VERSION: 00007

BLOQUE 1 — CONTRATO MODULAR

Antes de aprobar un módulo nuevo debe poder explicarse con claridad:

núcleo propio
contrato de entrada
contrato de salida
contrato de seguridad
oferta de integración
participación o no en trazabilidad operativa transversal

Validar además:

si realmente es módulo y no dataset reusable del tenant
si resuelve un problema propio del dominio
si evita invadir núcleos existentes
si sus integraciones futuras pueden publicarse por contrato
si el ownership funcional queda claro
si debe emitir eventos operativos livianos después de acciones propias autorizadas
si debe ser consumido por una capacidad transversal sin trasladarle lógica del dominio

Criterio adicional validado:

si una necesidad es solo consulta interna de colaboradores, directorio activo o pertenencia operativa, evaluar primero una raíz reusable del plano tenant antes de abrir módulo nuevo.

Criterio adicional validado:

si la necesidad aparece dentro de tablas anfitrionas de otros módulos, evaluar si corresponde publicar row actions oficiales desde el módulo oferente antes de crear lógica local en el consumidor.

Criterio adicional para reconversiones:

si una clasificación mezcla dimensiones conceptuales distintas, debe revisarse el contrato funcional del módulo antes de seguir agregando excepciones.

La salida correcta puede requerir separar:

núcleo de la entidad
relaciones operativas
entidades hijas o complementarias

Caso validado:

parties.kind no debe mezclar tipo de contacto con relación comercial o vínculo interno.

Criterio adicional para constraints funcionales:

si una reconversión cambia la dimensión real de autorización, la constraint funcional debe migrar hacia esa dimensión.

No debe conservarse la constraint anterior como compatibilidad operativa si ya no representa el dominio vigente.

Caso validado:

parties dejó de usar allowed_kinds como constraint funcional.

parties usa allowed_party_roles sobre party_roles.role.

Criterio adicional para OperationalActivity:

si un módulo participa en actividad operativa transversal, debe quedar definido:

qué eventos operativos livianos emite
en qué momento los emite
qué campos son observables
qué campos deben excluirse por ruido técnico
qué cambios tienen significancia operativa real
qué catálogo del módulo dueño declara esos campos
qué datos mínimos necesita el registro transversal
qué no debe delegarse al prestador transversal

Regla:

el prestador transversal no debe conocer en exceso la semántica interna del módulo dueño.

<<END SECTION>>

<<SECTION: BLOQUE 2 — SEGURIDAD OBLIGATORIA>>
SECTION_VERSION: 00008

BLOQUE 2 — SEGURIDAD OBLIGATORIA

Ningún módulo operativo puede entrar al sistema sin contrato de seguridad explícito.

Validar:

módulo registrado
capacidades válidas
scopes válidos
constraints necesarias
dataset visible
reglas de create
reglas de update
reglas de delete
si delega autorización al padre
si requiere contexto funcional obligatorio

Validar además:

consumo correcto mediante Security
ausencia de lógica paralela en controller
ausencia de inferencias en Blade
coherencia entre view_any y view

Preguntas obligatorias:

cómo autoriza realmente este módulo
qué rompe si falta contexto
qué acciones dependen de constraints
qué dataset debe devolver Security::scope(...)

Regla adicional consolidada:

si el módulo utiliza constraints funcionales, validar explícitamente que arbitren no solo create sino también view, update y delete cuando corresponda al contrato del módulo

no deben tratarse como metadata decorativa ni como validación parcial

Validación adicional para flujos operativos con estado:

si una operación depende del estado del recurso padre, de una orden, de una línea o de una etapa del flujo, ese estado debe auditarse como parte real de la seguridad efectiva

no debe quedar como simple bloqueo visual opcional

Validación adicional para módulos transaccionales o con estados operativos:

Security, Gate/Policy, controller y UI deben devolver la misma lectura operativa

los estados readonly deben bloquear actualización y eliminación en la autorización central

el borrado debe considerar trazabilidad operativa real y no solo permiso nominal del rol

las acciones sobre recursos hijos deben respetar primero el estado del padre

las validaciones con Project Lab deben incluir matriz por estado, permiso y trazabilidad cuando corresponda

Validación adicional para administración owner:

si el módulo depende de activación, permisos o configuración tenant, debe existir una vía visible y real para que owner administre ese frente

auditar explícitamente:

si existen permisos del módulo en el catálogo efectivo
si esos permisos pueden asignarse realmente
si la habilitación del módulo produce consecuencia visible coherente en UI
si owner dispone de una surface real para entender y administrar ese comportamiento

Validación adicional para campos ocultos o relaciones sensibles:

si una opción se oculta en UI por permisos, estado o relación activa, debe existir validación backend equivalente.

No alcanza con no mostrar el campo.

Debe probarse intento manipulado desde request o runtime real.

Caso validado:

parties bloquea asignación manipulada de relación employee desde StorePartyRequest y UpdatePartyRequest para usuarios no autorizados.

Validación adicional para reconversión de constraints:

cuando un módulo migra una constraint funcional hacia una nueva dimensión, debe validarse el contrato completo y no solo la UI o el formulario.

Controlar:

resolver central
contrato persistido
seeder
UI de permisos
controller
requests
vistas
Security::scope(...)
create
view
update
delete
usuarios reales del seed
grep de residuos legacy
prueba de manipulación cuando exista relación sensible

Caso validado:

parties migró desde allowed_kinds hacia allowed_party_roles.

Regla:

PARTIES + allowed_kinds no debe quedar como combinación activa cuando la doctrina vigente exige allowed_party_roles.

<<END SECTION>>

<<SECTION: BLOQUE 3 — RELACIONES E INTEGRACIONES>>
SECTION_VERSION: 00007

BLOQUE 3 — RELACIONES E INTEGRACIONES

Todo módulo debe definir claramente sus relaciones e integraciones dentro del ecosistema.

Validaciones obligatorias:

no reabrir núcleos de otros módulos
consumir capacidades oficiales antes de crear variantes locales
publicar capacidades reutilizables cuando correspondan
respetar ownership del módulo oferente
definir si emite eventos operativos livianos hacia una capacidad transversal
definir si consume actividad operativa persistida como lectura y bajo qué autorización

Regla adicional consolidada:

si el módulo participa en navegación, dashboard o surfaces visuales, debe declarar metadata visual mínima en ModuleCatalog.

Como mínimo:

icon lógico del módulo

Reglas:

la metadata visual no debe duplicarse en SurfaceService
no debe hardcodearse en vistas consumidoras
debe derivarse desde el catálogo modular

Regla adicional:

las surfaces publicadas no deben duplicar metadata que pueda resolverse desde ModuleCatalog

la metadata debe viajar enriquecida a través de ModuleSurfaceRegistry

Regla adicional:

las integraciones surface-driven deben permitir que el host consuma metadata ya resuelta

incluyendo iconografía cuando corresponda

sin reconstrucción local en el host

Caso consolidado:

tabs principales del host pueden renderizar icon mediante metadata de tab enriquecida desde surfaces

Regla:

toda integración debe poder describirse contractualmente y mantenerse reusable sin acoples innecesarios

Regla adicional para entidades base con relaciones hijas:

cuando una entidad base requiere expresar múltiples relaciones con la empresa, esas relaciones deben documentarse y modelarse como dimensión separada del tipo base.

Caso vigente:

parties → party_roles

parties.kind expresa tipo de contacto.

party_roles.role expresa relación con la empresa.

Regla adicional para trazabilidad operativa:

la integración con OperationalActivity debe realizarse mediante eventos operativos livianos y no mediante acoples directos de controllers consumidores al log.

El módulo dueño emite.

La capacidad transversal registra si corresponde.

El futuro consumidor visual lee actividad persistida bajo autorización definida.

Regla:

no inferir historia operativa desde campos actuales si existe actividad persistida específica.

<<END SECTION>>

<<SECTION: BLOQUE 4 — UI Y EXPERIENCIA OPERATIVA>>
SECTION_VERSION: 00006

BLOQUE 4 — UI Y EXPERIENCIA OPERATIVA

Todo módulo debe cumplir criterios mínimos de experiencia operativa consistente.

Validaciones obligatorias:

usar componentes base del sistema
no duplicar patrones visuales existentes
mantener coherencia con show pages host
no generar UI paralela para capacidades ya resueltas

Regla adicional consolidada:

los estilos específicos de módulo o pantalla deben ubicarse en:

public/css/modules/

no deben reemplazar app-base.css ni app-patterns.css

Regla adicional:

los íconos deben existir como componentes Blade reutilizables

ubicados en:

resources/views/components/icons

Reglas:

los nombres deben ser genéricos
no deben nombrarse por módulo
deben permitir reutilización transversal

Regla adicional:

las vistas deben consumir metadata visual ya resuelta

no deben decidir localmente iconografía ni atributos visuales de módulos

Regla adicional:

las tabs principales deben poder consumir metadata visual enriquecida

incluyendo iconografía desde surfaces o catálogo modular

Regla:

la experiencia operativa debe mantenerse homogénea en todo el sistema

Validación adicional para formularios de módulos base:

si una entidad base concentra decisiones iniciales relevantes, puede usarse un formulario progresivo para guiar la carga sin exponer complejidad técnica.

El formulario progresivo debe:

reducir error de carga
mantener copy comprensible
no reemplazar autorización backend
no trasladar reglas de negocio al frontend

Caso vigente:

parties guía la carga mediante:

tipo de contacto
relación con la empresa
datos del contacto

Regla de lenguaje:

evitar términos que generen confusión con seguridad o permisos.

Caso vigente:

en parties debe preferirse Relación con la empresa antes que rol.

<<END SECTION>>

<<SECTION: BLOQUE 5 — DOCUMENTACIÓN Y CIERRE>>
SECTION_VERSION: 00008

BLOQUE 5 — DOCUMENTACIÓN Y CIERRE

Todo módulo debe cerrar con documentación alineada a la doctrina vigente del sistema.

Validaciones obligatorias:

actualizar documentos doctrinales si corresponde
no dejar decisiones implícitas sin registrar
no duplicar información entre documentos
respetar estructura documental vigente
registrar si participa o no en trazabilidad operativa transversal cuando corresponda

Regla adicional consolidada:

las decisiones sobre metadata modular como iconografía u otros atributos visuales deben documentarse en:

convenciones
contexto arquitectónico
mapas de componentes

no deben quedar únicamente reflejadas en código

Regla adicional:

si una capacidad puede derivarse de un catálogo estructural, no debe documentarse como dato manual duplicado en múltiples capas

Regla:

el cierre de un módulo no es solo código funcionando

implica coherencia entre arquitectura, UI, navegación y documentación

Validación adicional para reconversiones profundas:

cuando se eliminen constantes, valores de catálogo o lecturas anteriores del modelo, debe auditarse que no queden residuos en:

app
resources
database

Caso vigente:

la reconversión de parties elimina la lectura de customer, supplier y employee como valores de parties.kind.

El cierre debe verificar que esos valores no reaparezcan como constantes, migraciones, seeders, requests, controller o vistas bajo la semántica anterior.

Validación adicional para reconversión de constraints:

cuando se reemplaza una constraint funcional anterior por una nueva, el cierre documental debe registrar:

dimensión anterior eliminada
dimensión nueva vigente
módulo afectado
capabilities alcanzadas
capa central que resuelve el contrato
validaciones runtime realizadas
residuos legacy eliminados

Caso validado:

parties dejó de usar allowed_kinds y consolidó allowed_party_roles.

La documentación no debe dejar como línea futura una revisión ya implementada y validada.

Validación adicional para actividad operativa transversal:

cuando un módulo emita eventos hacia OperationalActivity, el cierre debe registrar:

módulo dueño
eventos emitidos
momento de emisión
campos observables definidos por catálogo del módulo
campos excluidos por ruido técnico
validación de tenant activo
validación de queue si corresponde
validación de jobs y failed_jobs si corresponde
validación de operational_activities persistidas
ausencia de duplicación de listeners si event discovery ya resuelve la asociación

Regla:

no cerrar trazabilidad operativa solo por crear tabla o listener.

Debe validarse runtime real.

<<END SECTION>>

CHECKLIST MÍNIMO EXISTENTE

Además de los bloques obligatorios anteriores, verificar como mínimo:

modelo consistente
migraciones correctas
tenant scope correcto
rutas coherentes
controllers limpios
requests correctos
policies alineadas
Security correctamente consumido
vistas consistentes
componentes reusables aprovechados
relaciones necesarias reales
documentación impactada revisada
sin código muerto evidente
sin duplicaciones evitables
sin bypasses de seguridad

Para módulos con estados operativos o trazabilidad, verificar además:

estados terminales o readonly bloquean update/delete en Security y Gate/Policy
delete considera trazabilidad operativa real
controller no sostiene reglas críticas ausentes en Security
UI no muestra acciones que la autorización efectiva no permite
recursos hijos respetan el estado operativo del padre
validaciones con Project Lab incluyen matriz por estado, permiso y trazabilidad cuando corresponda

Para reconversiones de clasificación o catálogo, verificar además:

que los valores anteriores no sigan vivos como constantes
que migraciones y seeders expresen la semántica nueva
que requests validen contra el contrato vigente
que controllers no reconstruyan lógica legacy
que vistas y copy no usen términos anteriores de forma ambigua
que se haya probado intento manipulado cuando exista relación sensible
que se haya ejecutado grep de residuos sobre app, resources y database

Para reconversiones de constraints funcionales, verificar además:

que la constraint anterior no siga operando para el módulo reconvertido
que la nueva constraint exista en el contrato persistido
que el resolver central la consuma
que Security::scope(...) devuelva dataset coherente
que create, view, update y delete respeten la constraint cuando corresponda
que la UI de permisos muestre la dimensión correcta
que los formularios operativos expresen permisos ya resueltos
que no queden helpers legacy activos en controller
que no queden combinaciones residuales entre módulo y constraint anterior

Caso validado:

parties debe usar allowed_party_roles.

parties no debe usar allowed_kinds.

Este bloque resume mínimos técnicos, pero no reemplaza el checklist completo.

<<SECTION: CRITERIO FINAL DE APROBACIÓN>>
SECTION_VERSION: 00007

CRITERIO FINAL DE APROBACIÓN

Una capacidad nueva no se considera madura solo porque renderiza correctamente.

El cierre real exige revisar de forma integral:

contrato modular
seguridad efectiva
multi-tenancy
persistencia
controllers
services
hooks
eventos
listeners
colas cuando correspondan
cálculos
UI real
rutas
runtime
consistencia doctrinal

No alcanza con una vista compartida ni con una refactorización parcial.

Se considera cierre sólido cuando desaparecen duplicaciones relevantes sin perder claridad, ownership funcional, trazabilidad ni seguridad.

Criterio adicional para reconversiones profundas:

una reconversión se considera cerrada solo cuando la semántica anterior deja de operar en todas las capas relevantes.

Validaciones mínimas esperadas:

código aplicado
migraciones o migrate:fresh --seed según corresponda
Tinker backend para restricciones críticas
validación en navegador informada
grep de residuos
copy UX coherente con la nueva lectura

Caso vigente de referencia:

parties.kind debe quedar limitado a tipo de contacto.

party_roles.role debe expresar relación con la empresa.

Criterio adicional para reconversión de constraints funcionales:

una reconversión de permisos no se considera cerrada si la constraint anterior sigue operando como fallback legacy.

Debe validarse que la nueva constraint gobierne el contrato efectivo completo.

Caso validado:

parties consolidó allowed_party_roles como constraint funcional.

allowed_party_roles se evalúa contra party_roles.role.

allowed_kinds no aplica a parties.

El cierre incluyó validación de Security::scope(...), create, view, update, delete, UI de permisos, requests, controller, migrate:fresh --seed, Tinker, navegador y grep de residuos legacy.

Criterio adicional para trazabilidad operativa transversal:

una capacidad como OperationalActivity no se considera cerrada solo por existir tabla, modelo, evento o listener.

Debe validarse:

tenant activo real
usuario real
membership activa real
evento emitido después de operación autorizada
queue ejecutada cuando corresponda
jobs procesados
failed_jobs sin errores
operational_activities persistidas
sin duplicación de listeners
sin escritura manual desde UI
sin bloqueo del flujo principal ante falla de log
sin asumir autorización de lectura futura

Regla:

la trazabilidad transversal debe reforzar lectura operativa sin convertirse en autorización, CRUD ordinario ni módulo navegable principal por defecto.

<<SECTION: RUTAS Y CONTROLADOR>>

RUTAS Y CONTROLADOR

Verificar:

rutas CRUD declaradas explícitamente
no usar Route::resource(...) mientras siga vigente la convención actual del proyecto
nombres de ruta consistentes
controller delgado
validación separada cuando haga falta
authorize(...) o authorizeResource(...) aplicado cuando corresponda
no duplicar permisos en controller si la policy ya resuelve la acción
si el módulo tiene listados o datasets con visibilidad variable, resolverlos desde la capa central de autorización y no con lógica paralela ad hoc
si el módulo requiere constraints funcionales, declararlas y consumirlas explícitamente desde la capa central

Regla adicional para create contextual:

si create depende de contexto funcional obligatorio, debe quedar definido explícitamente en controller cuál es el canal válido de autorización

no debe asumirse que authorize('create', Model::class) resuelve el caso

debe consumirse Security con el contexto real requerido por el contrato

Todo módulo nuevo debe contar, como mínimo, con superficies visuales coherentes con el sistema.

Verificar:

index
show
create

partials reutilizables necesarias
componentes modulares propios cuando publiquen capacidades reusables

si el módulo publica una relación navegable reusable, no alcanza con implementarla en su show principal

Verificar coherencia entre backend y superficie visible.

Controlar:

acciones visibles alineadas con permisos reales

dataset proveniente de la capa central cuando aplique

ausencia de botones inaccesibles

ausencia de links residuales hacia módulos no habilitados

create contextuales resueltos con contexto válido

capacidades reusables sin canal paralelo de autorización

<<SECTION: CHECKLIST DE INTEGRACIÓN>>
SECTION_VERSION: 00008

CHECKLIST DE INTEGRACIÓN

Cuando un módulo se relacione con otros, verificar:

consumo de contratos existentes antes de crear variantes nuevas

uso de capacidades reusables oficiales del módulo oferente

trail y retorno consistentes

flags modulares correctamente propagados

homogeneidad entre show pages, tablas y partials secundarias

ausencia de links locales remanentes hacia relaciones ya cerradas doctrinalmente por otro módulo

diferenciación explícita entre integración pesada embebida e integración compacta

alineación con el patrón visual ya consolidado en el módulo consumidor cuando exista

si el módulo reemplaza surfaces históricas dispersas, plan de migración claro hacia la nueva surface oficial

si la relación reusable se apoya en registry o mecanismo equivalente, el host consumidor no recompone localmente payload ni naming ad hoc

compatibilidad real entre target, type, slot y gramática efectiva del host consumidor

Regla de cierre:

no considerar terminada una integración por haber corregido solo la pantalla principal del consumidor.

Debe revisarse también el ecosistema visible que consume esa relación.

Regla adicional para show pages surface-driven:

si el host monta surfaces por zonas explícitas, el checklist debe verificar que cada offer publique slot correcto y que el host tenga montaje compatible con esa combinación.

Una offer sin slot o con slot incoherente puede quedar invisible aunque el resto del contrato parezca correcto.

Regla adicional para usuarios internos del tenant:

si una integración necesita usuarios internos activos, validar explícitamente que el dataset provenga del plano tenant reusable correspondiente y no de un módulo inexistente ni de una query duplicada reconstruida por cada consumidor.

<<END SECTION>>

<<SECTION: VALIDACIONES BACKEND VS UI>>
SECTION_VERSION: 00003

VALIDACIONES BACKEND VS UI

Verificar:

la UI no sea la única barrera de control

campos ocultos también estén validados en backend

acciones no visibles también estén protegidas en backend

datasets respeten scope real desde backend

links relacionales dependan de permiso y módulo activo

si un módulo relacionado está deshabilitado, el backend rechace referencias hacia ese módulo

tabs, tablas parciales, calendarios y formularios embebidos respeten la misma lógica que la pantalla principal

no existan rutas operables residuales aunque la UI ya no las muestre

Regla adicional:

cuando una opción visual se oculte por rol, permiso, estado operativo o relación activa, debe probarse que el backend bloquee el mismo caso ante manipulación del request.

Caso vigente:

el checkbox o selección visual de Colaborador en parties no es fuente de autorización.

La restricción efectiva para asignar employee debe vivir en StorePartyRequest, UpdatePartyRequest o capa backend equivalente.

<<SECTION: SUPERFICIES SECUNDARIAS Y PARTIALS>>
SECTION_VERSION: 00008

SUPERFICIES SECUNDARIAS Y PARTIALS

Verificar:

partials profundas alineadas con la pantalla principal

calendarios alineados con reglas del módulo

formularios embebidos alineados con permisos reales

componentes reutilizables alineados con contrato del módulo

cancelar y redirects respetan trail vigente

no existen botones, filtros o enlaces residuales en vistas secundarias

el módulo puede considerarse cerrado también fuera de index/controller/show principal

si una capacidad reusable publicada por un módulo ya reemplaza una implementación anterior, no deben quedar superficies secundarias relevantes sosteniendo la variante vieja

si la integración depende de slots o zonas del host, las partials consumidoras no deben perder esa compatibilidad real

Regla adicional:

no considerar cerrada una integración reusable solo porque funciona en la vista principal del host.

Debe revisarse también:

sublistados
cards compactas
partials de summary
details
tabs secundarias
bloques relacionales derivados

Si una relación puede aparecer en más de una zona del host, cada una debe auditarse según la gramática real de montaje y no por similitud visual aparente.

<<SECTION: ENTIDADES HIJAS Y SUBRECURSOS>>
SECTION_VERSION: 00004

ENTIDADES HIJAS Y SUBRECURSOS

Verificar:

si el recurso hijo tiene autonomía real o depende funcionalmente del padre

si corresponde delegar autorización en el recurso padre

si evita crear un módulo artificial donde alcanza con una entidad hija bien modelada

si la UI del hijo no compite con la jerarquía del padre sin necesidad

si las restricciones reales del hijo provienen del padre o de un frente operativo relacionado ya existente

Preguntas obligatorias:

este recurso necesita identidad modular propia o sigue siendo detalle del padre
su seguridad puede delegarse completamente
su navegación debe ser estructural o contextual
su edición libre sigue siendo válida cuando ya existe trazabilidad operativa iniciada
si gana relevancia operativa, esa relevancia exige módulo nuevo o solo endurecer su contrato como hijo delegado

Regla adicional consolidada:

un subrecurso delegado puede ganar relevancia operativa transversal sin dejar de ser hijo

eso ocurre cuando pasa a actuar como ancla explícita de ejecución, trazabilidad o restricciones dentro de otro frente operativo

caso vigente relevante:

order_items respecto de inventory

En esos casos, validar además:

que no se le inventen permisos autónomos innecesarios
que la trazabilidad no rompa la delegación doctrinal al padre
que la UI exprese su nuevo peso operativo sin promoverlo falsamente a módulo
que edición y borrado dejen de ser libres cuando ya existan movimientos ejecutados o equivalentes

<<SECTION: NAVEGACIÓN Y BREADCRUMB>>
SECTION_VERSION: 00003

NAVEGACIÓN Y BREADCRUMB

Definir desde el inicio si el módulo necesita solo navegación estructural o también navegación contextual.

Preguntas de control:

la entidad puede abrirse desde otra entidad relacionada como continuación natural de trabajo

el usuario necesita volver al origen sin perder el hilo operativo

ese origen es semántico y estable, no solo la página anterior

Si alguna de estas respuestas es sí, el módulo debe evaluar e implementar navegación contextual explícita.

Si corresponde, el módulo debe:

tener builder o resolutor de trail propio compatible con el mecanismo central vigente del proyecto

aceptar contexto de navegación de forma explícita

resolver ese contexto desde la capa reutilizable central vigente

propagar ese contexto en tablas, tabs, acciones por fila, links relacionales, paginación, filtros, botones volver, botones cancelar y redirects relevantes

resolver create -> store -> show sin persistencia de nodos transitorios de “Nuevo …”

Validar además:

cuál es la superficie principal real del módulo

si existe una superficie secundaria que permanece accesible pero no debe actuar como base promovida del trail

si calendarios, tabs o embebidos contextuales necesitan actuar como entrada principal efectiva del módulo

Regla adicional consolidada:

cuando una capacidad reusable oficial del módulo oferente se consuma dentro de otro módulo, también debe propagar el trail válido del contenedor y no reiniciar navegación desde una lectura estructural local del oferente

<<END SECTION>>

<<SECTION: CIERRE>>
SECTION_VERSION: 00003

CIERRE

Un módulo no debe considerarse terminado solo porque funciona.

Debe quedar:

consistente con el sistema
alineado con permisos reales
alineado con multi-tenancy
alineado con la base visual
alineado con la documentación vigente

Además debe quedar explícitamente resuelto:

si el dataset se gobierna desde la capa central de autorización

si el módulo usa solo scopes o también constraints funcionales

si las entidades hijas o embebidas se autorizan por delegación al padre

si un create contextual requiere contexto obligatorio y cuál es su canal válido de autorización

si la validación manual del módulo puede reproducirse de forma canónica con Project Lab sin ambigüedad operativa

<<END SECTION>>
