BASE-VISUAL-DEL-PROYECTO – app-base


OBJETIVO DEL DOCUMENTO

Este documento define la base visual común del sistema.

Su objetivo es asegurar una interfaz coherente, reusable, limpia y predecible entre módulos.

No documenta decisiones de negocio ni reglas funcionales profundas.

Documenta:

estructura visual base
componentes reutilizables
criterios de composición de vistas
patrones visuales compartidos


ESTRUCTURA GENERAL DE PÁGINA

Toda pantalla principal del sistema debe construirse sobre una estructura consistente.

Base esperada:

x-page
x-breadcrumb
x-page-header
contenido principal dentro de cards o bloques equivalentes

La estructura debe mantener:

jerarquía visual clara
encabezado consistente
acciones principales visibles
lectura simple del contenido


COMPONENTES DISPONIBLES

Componentes Blade reutilizables utilizados en todas las vistas.

x-page
Contenedor general de la página.

x-breadcrumb
Navegación contextual y estructural de la pantalla.

x-page-header
Título de la pantalla y acciones principales.

x-card
Contenedor visual del contenido principal.


PATRÓN GENERAL PARA VISTAS INDEX

Las vistas index deben usar, como criterio general, una estructura de dos bloques principales:

bloque superior para filtros, búsqueda o acciones secundarias
bloque principal para listado, tabla o grilla

Cuando corresponda, debe priorizarse el uso de partials o componentes reutilizables.


CARD DE FILTROS REUTILIZABLE

Cuando una vista index requiera filtros, debe priorizarse el patrón reusable correspondiente.

Este bloque debe funcionar como card superior de filtros o búsqueda y mantener consistencia entre módulos.

No debe mezclarse con summaries, métricas o bloques de navegación secundaria salvo necesidad justificada.


SUMMARY Y BLOQUES VISUALES REUTILIZABLES

Los bloques de summary, métricas o visualización rápida deben poder resolverse mediante wrappers o partials reusables cuando el patrón ya sea claro.

No deben desarrollarse variantes aisladas por módulo si la misma estructura visual puede reutilizarse.

Cuando un patrón visual deje de ser exclusivo de un módulo, debe migrarse a una forma neutral reusable.


FRAGMENTACIÓN DE BLADES

Las vistas grandes deben dividirse en partials cuando superen complejidad razonable.

Especialmente cuando mezclen:

filtros
tablas
resúmenes
formularios secundarios
bloques visuales repetidos

La fragmentación debe buscar:

legibilidad
mantenibilidad
reutilización

No debe fragmentarse artificialmente si el archivo sigue siendo claro y simple.


ORGANIZACIÓN DE CSS VISUAL

La organización del CSS visual del proyecto debe seguir esta estructura:

public/css/app-base.css
public/css/app-patterns.css
public/css/modules/<modulo>.css

Criterios:

app-base.css contiene la base global obligatoria del sistema

app-patterns.css concentra patrones visuales reutilizables y neutrales compartidos entre módulos

modules/<modulo>.css debe reservarse para estilos específicos de módulos complejos cuando realmente no corresponda promoverlos todavía a patrón global

Cuando un patrón visual deja de ser exclusivo de un módulo, debe migrarse a nomenclatura neutral reusable y salir del archivo específico del módulo.


PATRONES VISUALES NEUTRALES REUTILIZABLES

Cuando un patrón visual se reutilice en más de un módulo, debe adoptar una nomenclatura neutral y reusable.

Familias válidas ya alineadas con este criterio:

visual-grid
visual-card
visual-title
progress
progress-bar
progress-meta
pie-layout
pie-chart
pie-track
pie-segment--*
pie-center
pie-legend
pie-dot--*
visual-kpi
visual-note

Regla:

no mantener prefijos de módulo en clases que ya funcionan como patrón transversal del sistema.


BREADCRUMB ESTRUCTURAL Y CONTEXTUAL

x-breadcrumb representa la ruta visible de ubicación actual de la pantalla.

Debe poder resolverse en dos modos:

breadcrumb estructural
breadcrumb contextual

breadcrumb estructural:
expresa la ubicación normal y estable de la pantalla dentro del módulo o del sistema

breadcrumb contextual:
expresa el retorno al contexto de origen cuando el usuario llegó desde una entidad relacionada y ese origen sigue siendo relevante para la tarea actual

Reglas:

x-breadcrumb es un único componente visual del sistema

no deben coexistir dos breadcrumb distintos en una misma pantalla

el breadcrumb estructural debe actuar como base por defecto

el breadcrumb contextual solo debe utilizarse cuando exista un flujo relacionado real, válido y útil para la continuidad de uso

no usar breadcrumb contextual si no existe origen funcional claro

no reconstruir jerarquías artificiales ni cadenas excesivas solo porque exista una relación entre entidades

el breadcrumb contextual debe mejorar orientación y retorno, no agregar ruido visual

cuando el usuario acceda a una pantalla desde navegación propia del módulo, búsqueda, acceso directo o index del mismo módulo, debe priorizarse breadcrumb estructural

cuando exista navegación cruzada entre entidades relacionadas, puede priorizarse una salida visual de retorno al origen siempre que esa salida mejore realmente la UX

breadcrumb no debe depender del historial del navegador para representar relaciones de negocio


PANTALLAS TIPO CALENDARIO

Cuando un módulo necesite lectura temporal, agenda visual o navegación por fechas, se admite una pantalla tipo calendario dentro de la misma base visual del sistema.

Estructura recomendada:

x-page
x-breadcrumb
x-page-header
card superior de navegación o filtros
card principal con contenedor scrolleable interno

Estas pantallas pueden fragmentarse en partials específicas del módulo, por ejemplo:

partials/calendar-toolbar.blade.php
partials/calendar-grid.blade.php
partials/calendar-day-cell.blade.php

Cuando la UX lo requiera, el calendario puede utilizar:

cabecera sticky interna
scroll interno del bloque
columna auxiliar de contexto temporal, como número de semana

No corresponde documentar aquí detalles finos de CSS o ajuste visual puntual de implementación.


TABS INTERNAS EN VISTAS SHOW

Cuando una vista show concentre demasiada información, se permite organizar el contenido mediante tabs internas dentro del cuerpo principal de la página.

Las tabs no son obligatorias.
Solo deben utilizarse cuando ayudan a reducir altura visual o separar información secundaria real.

Clases utilizadas:

tabs
tabs-nav
tabs-link
tab-panel
tab-panel-stack

Comportamiento:

controlado por JavaScript mínimo global
una solapa activa visible por vez
útil para separar detalle, documentos asociados, historial u otros bloques secundarios

Las tabs internas pueden contener múltiples cards apiladas por sección.

Cuando la UX lo requiera, puede preservarse la tab activa al recargar o luego de acciones POST, PUT o DELETE.

La persistencia de tab activa no debe asumirse como regla global por query string si existen tabs anidadas o múltiples contenedores de tabs en una misma pantalla.

Debe resolverse con criterio por contenedor, o no aplicarse, cuando la preservación complique la UX.

La navegación de tabs debe ser visualmente reconocible como patrón de solapas.

La pestaña activa debe distinguirse con claridad.

El contenedor tabs-nav debe tener jerarquía visual suficiente para que el usuario perciba que puede cambiar de sección.

Las tabs documentadas en esta base visual corresponden a tabs internas de contenido.

No implican todavía un sistema general de navegación entre rutas dentro de tabs de app.

No deben confundirse con navegación cruzada entre entidades relacionadas ni con retorno contextual al origen.


REGLA GENERAL DE SOBRIEDAD VISUAL

La interfaz debe privilegiar claridad y continuidad visual.

Evitar:

sobrecarga de cards sin necesidad
soluciones visuales demasiado particulares por módulo
fragmentación artificial de patrones equivalentes
exceso de variaciones tipográficas o espaciales sin valor funcional

Cuando dos pantallas resuelven el mismo problema visual, deben parecer parte del mismo sistema.
