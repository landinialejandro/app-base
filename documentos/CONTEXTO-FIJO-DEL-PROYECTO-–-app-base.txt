<<SECTION: Tipo de sistema>>
Tipo de sistema

app-base es una plataforma SaaS multi-tenant desarrollada en Laravel.

Permite que múltiples empresas utilicen la misma aplicación manteniendo aislamiento lógico de datos dentro de una única base de datos.

Cada empresa dentro del sistema se denomina tenant.

El objetivo del proyecto es construir una base ERP SaaS modular que permita gestionar operaciones empresariales reutilizando una arquitectura común y estable.
<<END SECTION>>

<<SECTION: Arquitectura SaaS>>
Arquitectura SaaS

El sistema utiliza una arquitectura multi-tenant basada en una columna:

tenant_id

Todas las entidades del negocio incluyen:

tenant_id

Esto permite aislar los datos de cada empresa dentro de la misma base de datos.
<<END SECTION>>

<<SECTION: Convención real de identificador de tenant>>
Convención real de identificador de tenant

tenants.id utiliza char(36)

Las foreign keys hacia tenant deben respetar ese tipo real validado del proyecto.

No debe asumirse tenant_id como foreignId() por convención de Laravel sin verificar primero el patrón real existente.
<<END SECTION>>

<<SECTION: Relación principal de identidad>>
Relación principal de identidad

La relación de pertenencia del sistema es:

users
↓
memberships
↓
tenants

Características:

Un usuario puede pertenecer a múltiples empresas.
Cada empresa puede tener múltiples usuarios.
Un mismo user puede pertenecer a tenants no relacionados entre sí.
users representa identidad global.
memberships representa acceso por empresa.
<<END SECTION>>

<<SECTION: Acceso efectivo por empresa>>
Acceso efectivo por empresa

El estado de acceso efectivo a un tenant se resuelve por memberships.status y no por users.

El bloqueo o inhabilitación de acceso debe resolverse por membership y no por user.

Esto permite que un usuario continúe activo en un tenant y al mismo tiempo se encuentre bloqueado o inactivo en otro.
<<END SECTION>>

<<SECTION: RESOLUCIÓN DEL TENANT ACTIVO>>
RESOLUCIÓN DEL TENANT ACTIVO

En el flujo tenant, el tenant activo se resuelve por middleware.

Luego de esa resolución, el tenant vigente del request debe consumirse desde app('tenant').

No deben mantenerse helpers, soportes legacy o mecanismos paralelos redundantes para resolver el tenant activo si duplican o contradicen esa fuente vigente.
<<END SECTION>>

<<SECTION: Reglas operativas del tenant activo>>
Reglas operativas del tenant activo

El selector de empresa debe mostrar únicamente memberships activas del usuario autenticado.

ResolveTenant debe validar pertenencia activa al tenant antes de permitir acceso operativo.
<<END SECTION>>

<<SECTION: Aislamiento automático de datos>>
Aislamiento automático de datos

Las entidades del negocio utilizan el trait:

TenantScoped

Este trait aplica automáticamente:

WHERE tenant_id = tenant_actual

Además asigna automáticamente el tenant_id al crear nuevos registros.
<<END SECTION>>

<<SECTION: Regla de uso del trait>>
Regla de uso del trait

TenantScoped solo debe aplicarse a modelos cuyo ciclo de vida pertenece completamente a un tenant.

No debe utilizarse en entidades del sistema como:

User
Tenant
Membership
<<END SECTION>>

<<SECTION: Protección de acceso por URL>>
Protección de acceso por URL

Las entidades empresariales utilizan el trait:

ResolvesTenantRouteBinding

Esto protege el route model binding para evitar acceso a registros de otros tenants.

Consulta efectiva:

id = ?
AND tenant_id = tenant_actual
AND deleted_at IS NULL

<<SECTION: MODELO ACTUAL DE ACCESO, VISIBILIDAD Y AUTORIZACIÓN OPERATIVA>>
MODELO ACTUAL DE ACCESO, VISIBILIDAD Y AUTORIZACIÓN OPERATIVA

El sistema ya incorpora una capa base real de acceso y visibilidad por módulo.

La autorización efectiva debe entenderse como una combinación de niveles distintos:

tenant activo válido
módulo habilitado para el tenant
permiso del rol del usuario
visibilidad del registro
acción permitida sobre ese registro
restricciones futuras por tipo o contexto

Estos niveles no deben mezclarse entre sí.

Reglas actuales consolidadas:

la habilitación de módulos del tenant se resuelve mediante una capa central específica de acceso por tenant

el tenant define qué módulos están operativamente disponibles

sobre esos módulos habilitados, el permiso efectivo se resuelve por rol

owner no debe tratarse como un rol delegable ordinario

owner se define estructuralmente por memberships.is_owner = true

admin es el rol administrativo delegable exclusivo

sales y operator son los roles operativos base actuales

una membership activa no-owner no debe quedar sin al menos un rol válido

el rol por defecto para nuevas memberships activas no-owner es operator

la visibilidad no necesariamente coincide con la capacidad de edición

roles base del sistema se definen en RoleCatalog

los roles base reservados no deben modificarse ni eliminarse

a futuro el tenant podrá incorporar más flexibilidad de roles sin romper esta jerarquía base
<<END SECTION>>

<<SECTION: Capa central prevista para acceso y permisos>>
Capa central prevista para acceso y permisos

La arquitectura de acceso y permisos debe prepararse para incorporar una capa centralizada compuesta por:

ModuleCatalog
TenantModuleAccess
RolePermissionMatrix
RolePermissionResolver

Rol conceptual de cada pieza:

ModuleCatalog define la base de módulos cerrados del sistema y actúa como fuente de verdad para slugs funcionales y módulos navegables del core.

TenantModuleAccess debe resolver la habilitación efectiva de módulos por tenant y por usuario, reutilizando una única lógica tanto para acceso operativo como para navegación visible.

RolePermissionMatrix define la matriz base de permisos por rol y módulo.

RolePermissionResolver debe resolver el permiso efectivo por usuario, tenant y módulo, reutilizando la matriz base y la capa de acceso al módulo.

Estado actual de esta capa:

conceptualmente aprobada
pendiente de implementación formal consolidada en todo el core

Dirección de diseño:

la matriz inicial puede ser hardcodeada

la lógica debe diseñarse para migrar luego a base de datos

las policies no deben hardcodear permisos dispersos cuando esta capa exista formalmente

la navegación y accesos visibles deben reutilizar la misma lógica real de acceso

la habilitación efectiva de módulos puede admitir capas de override controladas, siempre subordinadas a la base definida en ModuleCatalog
<<END SECTION>>

<<SECTION: Precedencia conceptual prevista para habilitación de módulos>>
Precedencia conceptual prevista para habilitación de módulos

La habilitación efectiva de módulos puede resolverse, cuando el sistema lo necesite, con una precedencia como la siguiente:

base desde ModuleCatalog

override global externo

override externo por tenant

override persistido en settings del tenant

Esta precedencia debe entenderse como criterio de diseño para crecimiento controlado del sistema, no como obligación de implementación inmediata en todos los entornos.
<<END SECTION>>

<<SECTION: Overrrides externos controlados>>
Overrrides externos controlados

La matriz base de permisos y la habilitación base de módulos pueden admitir overrides externos parciales cuando haga falta realizar ajustes rápidos y controlados sin modificar directamente las clases principales.

Estos overrides deben respetar siempre:

los roles definidos en RoleCatalog
los módulos definidos en ModuleCatalog
la estructura base del sistema

No deben utilizarse para inventar roles o módulos inexistentes.
<<END SECTION>>

<<SECTION: Reglas operativas actuales por módulo>>
Reglas operativas actuales por módulo

orders, parties, assets y products siguen actualmente un patrón compartido del tenant:

owner y admin pueden ver, crear, editar y eliminar

sales y operator pueden ver, crear y editar

la eliminación queda reservada a owner y admin

por ahora no se usa ownership artificial en módulos base o compartidos como parties, assets y products

tasks mantiene ownership operativo real por assigned_user_id

roles operativos pueden ver tareas del tenant según la policy vigente

roles operativos solo editan sus propias tareas, salvo permisos superiores

projects mantiene visibilidad restringida por participación operativa:

owner y admin ven todos los proyectos

roles operativos ven solo proyectos donde tengan al menos una tarea asignada

la visibilidad del dashboard y de métricas agregadas puede variar según rol

el dashboard analítico queda reservado al nivel privilegiado owner y admin

documents queda deliberadamente postergado como módulo de endurecimiento posterior por su mayor sensibilidad de trazabilidad, flujo y acciones visibles en UI
<<END SECTION>>

<<SECTION: Reglas generales de autorización ya consolidadas>>
Reglas generales de autorización ya consolidadas

la autorización por acción debe resolverse preferentemente mediante policies

cuando exista una policy, no alcanza con crearla:
el controller debe invocar authorize(...) o authorizeResource(...)
la vista debe respetar permisos reales con @can al mostrar acciones

la tríada obligatoria del sistema es:

policy
authorize en controller
@can en vistas

delete de registros principales compartidos debe reservarse a owner y admin, salvo excepción funcional justificada

los ítems embebidos pueden heredar autorización desde la cabecera cuando la acción dependa naturalmente de ella

navbar, dashboard y accesos rápidos deben respetar acceso real del usuario al módulo

la navegación no debe asumir acceso total al tenant si el módulo o la acción están restringidos por rol

el acceso al módulo, la visibilidad del dataset y las acciones permitidas deben tratarse como capas distintas

el navbar no debe depender de un hardcode manual de módulos cuando exista una capa central disponible para resolver acceso y navegación
<<END SECTION>>

<<SECTION: Patrón de modelado cabecera / detalle>>
Patrón de modelado cabecera / detalle

Cuando un módulo represente una operación compuesta por múltiples renglones, debe modelarse preferentemente utilizando el patrón cabecera/detalle.

Reglas del patrón:

la cabecera representa el acto principal
los ítems representan el detalle operativo
los ítems heredan el tenant de la cabecera

los ítems, por regla general, no exponen tenant_id en fillable desde formularios públicos.

Si el alta del ítem se realiza internamente desde un flujo controlado del sistema y requiere asignación explícita del tenant heredado de la cabecera, puede incluirse tenant_id en fillable para permitir la creación consistente del registro.

la eliminación de la cabecera elimina sus ítems
la posición del ítem es responsabilidad del módulo
los importes pueden calcularse dinámicamente o persistirse según la naturaleza funcional del documento
<<END SECTION>>

<<SECTION: Numeración operativa>>
Numeración operativa

El sistema implementa numeración automática para órdenes y documentos.

Reglas actuales:

la numeración se asigna al guardar la entidad

no se reserva numeración antes del guardado

la numeración se controla mediante document_sequences

la secuencia se define por:

tenant
tipo documental
punto de venta

orders y documents no permiten edición manual del número

una vez asignado el número, el tipo de entidad no debe modificarse
<<END SECTION>>

<<SECTION: Flujo orders → documents>>

<<SECTION: Reglas de fechas del flujo operativo>>
Reglas de fechas del flujo operativo

Para mantener coherencia en las operaciones comerciales:

orders siempre deben tener:

party_id
ordered_at

documents siempre deben tener:

party_id
issued_at

En orders la fecha es editable con restricciones razonables hacia el futuro.

En documents la fecha se controla según el tipo documental y su coherencia con la orden asociada.

Por ejemplo:

un documento asociado a una orden no debe tener una fecha anterior a la orden.
<<END SECTION>>

<<SECTION: Factories y seeders>>
Factories y seeders

En factories utilizadas para testing es válido crear tenants automáticamente, por ejemplo:

tenant_id => Tenant::factory()

En seeders reales del sistema el tenant normalmente se asigna explícitamente desde el seeder.

No es necesario modificar este comportamiento.
<<END SECTION>>

<<SECTION: Criterio actual de deploy y declaración de rutas>>
Criterio actual de deploy y declaración de rutas

En despliegues sobre hosting compartido sin CLI, Composer ni Artisan en servidor, el proyecto prioriza estrategia de deploy por versión cerrada preparada en local.

En ese contexto, las rutas del core operativo se mantienen declaradas explícitamente en routes/web.php en lugar de depender de Route::resource(...), debido a un incidente real de registro inconsistente de rutas durante la primera prueba operativa.
<<END SECTION>>
