OBJETIVO DEL DOCUMENTO

Este documento describe la organización de los componentes visuales del sistema app-base.

Su función es:

identificar componentes reutilizables  
definir su ubicación canónica  
diferenciar responsabilidades visuales  
evitar duplicación de UI  
servir como referencia para desarrollo y refactor  

NO CORRESPONDE DEFINIR AQUÍ

reglas de negocio  
contratos de seguridad  
arquitectura modular  
navegación funcional  
decisiones de permisos  

Este documento se limita a la capa visual y estructural de componentes.
<<END SECTION>>


<<SECTION: COMPONENTES BLADE GLOBALES>>
SECTION_VERSION: 00002

COMPONENTES BLADE GLOBALES

Ubicación:

resources/views/components/

Responsabilidad:

componentes reutilizables en todo el sistema sin pertenencia a un módulo específico.

Ejemplos vigentes:

x-page
x-page-header
x-breadcrumb
x-card
x-modal

Reglas:

deben ser genéricos
no deben contener lógica de negocio
no deben depender de un módulo específico
deben ser reutilizables en cualquier contexto

Regla adicional:

si un componente comienza a depender de un módulo específico, debe migrarse al módulo correspondiente.

Regla de validación runtime:

no debe utilizarse un componente Blade si no existe realmente en resources/views/components o como clase correspondiente.

Caso detectado:

x-empty-state no existe actualmente como componente disponible.

Por lo tanto, no debe listarse ni usarse como componente global vigente.

Si una vista requiere estado vacío y no existe componente específico, debe resolverlo con componentes disponibles o markup inline consistente.

Caso vigente:

la tab Actividad de tenant/profile resolvió su estado vacío inline con detail-block.

<<END SECTION>>

COMPONENTES DE LAYOUT

resources/views/components/layout/

<<SECTION: COMPONENTES DE ÍCONOS>>
SECTION_VERSION: 00001

COMPONENTES DE ÍCONOS

Ubicación canónica:

resources/views/components/icons/

Responsabilidad:

centralizar iconografía SVG reutilizable del sistema.

Reglas:

los íconos reutilizables no deben permanecer inline en blades complejos  
deben definirse como componentes Blade  
los nombres deben ser genéricos por forma visual  
no deben nombrarse por módulo  

Ejemplos válidos:

calendar  
user-group  
screen  
list-check  
folder  
archive-box  
file-text  
paperclip  
grid  

Regla crítica:

las vistas no deben decidir qué ícono usar  
la iconografía debe venir resuelta desde metadata modular  

Aclaración:

no existe `<x-icons.user />`  
debe utilizarse `<x-icons.user-group />` u otro ícono existente  

Objetivo:

evitar duplicación de SVG  
permitir consistencia visual  
facilitar mantenimiento y evolución visual
<<END SECTION>>

COMPONENTES DE TABS

El sistema distingue dos tipos de tabs con roles diferentes.

TABS PRINCIPALES DEL HOST

Componente:

x-host-tabs

Responsabilidad:

organizar tabs principales de páginas show.

Incluye:

estructura de navegación
toolbar
scroll horizontal
paneles de contenido
estado activo

Reglas:

representa navegación principal del recurso
puede renderizar icon si viene en metadata
puede usar module_icon como fallback
no define lógica de negocio

Aclaración vigente:

x-host-tabs puede recibir tabs provenientes de surfaces modulares y de surfaces tenant/transversales no modulares, siempre que respeten la gramática esperada.

Gramática esperada:

key
label
view
data
count
priority

Caso validado:

tab Actividad contextual publicada desde OperationalActivity mediante TenantSurfaceRegistry.

Regla:

cuando la tab participe de return_tab, key debe ser estable y compatible con navegación del host.

Caso vigente:

activity

TABS INTERNAS EMBEBIDAS

Componente:

x-tabs-embedded

Responsabilidad:

organizar contenido interno dentro de una capacidad embebida.

Casos:

segmentación por estado
segmentación interna de datasets
lectura agrupada

Reglas:

no participan de navegación principal
no usan return_tab
no representan cambio de recurso

Distinción obligatoria:

x-host-tabs → navegación del recurso
x-tabs-embedded → organización interna de contenido

No deben mezclarse.

<<END SECTION>>


<<SECTION: COMPONENTES MODULARES>>
SECTION_VERSION: 00001

COMPONENTES MODULARES

Ubicación:

resources/views/<modulo>/components/

Responsabilidad:

componentes reutilizables propios de un módulo específico.

Ejemplos:

tablas embebidas  
formularios específicos  
partials operativos  
componentes de integración  

Reglas:

pertenecen al módulo oferente  
no deben moverse a global si contienen semántica del módulo  
deben ser consumidos por otros módulos sin duplicación  

Regla adicional:

si una capacidad reusable pertenece a un módulo, su componente debe vivir en ese módulo.

El consumidor no debe copiar ni recrear ese componente.

Caso vigente:

inventory publica componentes propios para operaciones por línea  
orders y documents los consumen sin duplicar lógica
<<END SECTION>>


<<SECTION: COMPONENTES DE SHOW PAGES>>
SECTION_VERSION: 00002

COMPONENTES DE SHOW PAGES

Responsabilidad:

estructurar la lectura visual de una entidad.

Componentes principales:

x-show-summary
x-show-summary-item
x-show-summary-item-detail-block

Integración con tabs:

x-host-tabs

Reglas:

la show page actúa como host visual
los componentes definen zonas de lectura
las capacidades externas se montan como invitadas

Zonas principales:

header_actions
summary_items
detail_items
zona tab coordinada

Regla:

el host organiza

los módulos invitados ejecutan

No debe mezclarse ownership.

Aclaración vigente para tabs contextuales:

una show page puede montar una tab proveniente de una capacidad tenant/transversal no modular si la surface fue publicada por canal reusable y respeta la gramática del host.

Caso validado:

tab Actividad en tasks.show y parties.show.

Reglas:

la tab no convierte OperationalActivity en módulo principal del host

la tab no crea navegación principal propia

la tab no debe insertarse manualmente como cierre estable si puede publicarse mediante registry

la tab debe mantener key compatible con return_tab

Caso vigente:

key = activity

<<END SECTION>>


<<SECTION: COMPONENTES DE ACCIÓN>>
SECTION_VERSION: 00002

COMPONENTES DE ACCIÓN

Responsabilidad:

representar acciones disponibles para el usuario.

Incluye:

botones primarios
botones secundarios
acciones icon-only
acciones por fila
acciones contextuales

Reglas:

las acciones deben reflejar estado real
no deben mostrar acciones inválidas
no deben decidir permisos

Regla crítica:

el estado de la acción debe venir resuelto desde backend o surface

el componente solo representa

No debe haber lógica de autorización en el componente.

Aclaración vigente para parties:

las acciones visibles en parties deben expresar la autorización ya resuelta por el contrato efectivo.

Cuando la creación o gestión dependa de allowed_party_roles, el componente o vista solo debe representar disponibilidad ya calculada.

Regla:

no decidir acciones de parties por allowed_kinds.

No reconstruir permisos desde parties.kind.

<<END SECTION>>


<<SECTION: COMPONENTES REUTILIZABLES TRANSVERSALES>>
SECTION_VERSION: 00005

COMPONENTES REUTILIZABLES TRANSVERSALES

Componentes utilizados por múltiples módulos con gramática común.

Ejemplos:

x-line-item-form
x-line-item-table
x-line-operation-modal
x-tabs-embedded
x-dev-component-version

Reglas:

pueden ser compartidos
no deben absorber lógica de dominio
no deben romper ownership modular

Regla adicional:

si la semántica cambia por módulo, no debe forzarse reutilización artificial

Objetivo:

maximizar reutilización sin perder claridad funcional.

Comportamientos frontend reutilizables asociados a formularios:

data-action="app-progressive-form"

Este comportamiento pertenece a la capa visual/interactiva general.

No es un componente Blade nuevo.

Su función es organizar visualmente formularios progresivos sin reemplazar autorización ni validación backend.

Partial reusable transversal vigente:

resources/views/tenants/partials/operational-activity-table.blade.php

Responsabilidad:

representar tabla de actividad operativa ya resuelta por la capa de lectura correspondiente.

Uso vigente:

lectura general de actividad en tenant/profile
lectura contextual de actividad en hosts autorizados

Columnas:

Fecha
Módulo
Actividad
Registro
Actor
Sujeto

Reglas:

no decide autorización
no consulta permisos
no reconstruye historia operativa
no resuelve ownership funcional
no inventa links
representa filas ya preparadas por OperationalActivitySurfaceService, OperationalActivityContextReader o lectura equivalente vigente

Aunque vive bajo tenants/partials, su uso actual es transversal para representar actividad operativa en distintos contextos autorizados.

<<END SECTION>>


<<SECTION: REGLAS GENERALES DE COMPONENTES>>
SECTION_VERSION: 00003

REGLAS GENERALES DE COMPONENTES

Reglas obligatorias:

no duplicar componentes existentes
no mezclar lógica de negocio en componentes visuales
no decidir permisos en frontend
no reconstruir autorización en componentes
no trasladar ownership funcional al componente
no convertir comportamiento visual en regla de seguridad

Los componentes deben representar información ya resuelta por backend, services, payloads o surfaces oficiales.

Si un componente necesita saber si una acción o relación está disponible, debe recibir esa lectura ya calculada.

Regla adicional:

la ocultación visual no reemplaza autorización backend.

Caso vigente:

parties._form puede ocultar employee cuando el usuario no puede gestionarlo, pero el bloqueo real debe permanecer en StorePartyRequest, UpdatePartyRequest y PartyEmployeeContactAuthorization.

Regla:

los componentes pueden mejorar claridad, progresividad y lectura operativa.

No pueden convertirse en fuente final de autorización, validación sensible o seguridad efectiva.

<<END SECTION>>
