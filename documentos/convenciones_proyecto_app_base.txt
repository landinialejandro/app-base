<<SECTION: OBJETIVO DEL DOCUMENTO>>
OBJETIVO DEL DOCUMENTO

Este documento define las convenciones estructurales y de desarrollo del proyecto app-base.

Su objetivo es mantener consistencia técnica, previsibilidad arquitectónica y coherencia de código a medida que el sistema crece.

Las convenciones aquí definidas deben respetarse al crear:

nuevos módulos
nuevas entidades
nuevas vistas
nuevos controladores
nuevos flujos operativos
<<END SECTION>>

<<SECTION: CONVENCIONES GENERALES>>
SECTION_VERSION: 00003

CONVENCIONES GENERALES

El proyecto sigue los principios de:

simplicidad
claridad estructural
baja duplicación
coherencia entre módulos
previsibilidad del comportamiento del sistema

Cuando existan varias alternativas técnicas, se debe priorizar siempre la opción más simple y consistente con el resto del proyecto.

Convención documental consolidada:

la base documental técnica del proyecto debe tender a una estructura homogénea y navegable

los documentos técnicos activos deben utilizar:

bloque <<SECTION: METADATOS>>
TÍTULO como nombre humano del documento
DOC_SLUG como identificador técnico canónico
DOC_VERSION como versión general del documento

las referencias internas entre documentos deben tender a resolverse mediante:

<DOC_REF slug="...">Título visible</DOC_REF>

no deben utilizarse nombres de archivo físicos como referencia narrativa dentro del cuerpo del documento

las carpetas baks y auditoria no forman parte de la base documental activa navegable

Convención adicional consolidada para contenido orientado a usuario:

los textos dirigidos a usuario final deben utilizar lenguaje no técnico
deben priorizar claridad, accesibilidad y comprensión inmediata
deben construirse con frases cortas y cierre conceptual claro
pueden incorporar líneas de refuerzo tipo “👉 …” como patrón de cierre

este criterio aplica a:

documentación de onboarding
ayudas contextuales
contenidos UX embebidos
<<END SECTION>>

<<SECTION: CONVENCIÓN ACTUAL DE DECLARACIÓN DE RUTAS CRUD>>
CONVENCIÓN ACTUAL DE DECLARACIÓN DE RUTAS CRUD

Como convención actual del proyecto, las rutas CRUD del core funcional deben declararse explícitamente en routes/web.php.

No se utilizará Route::resource(...) mientras el proyecto mantenga compatibilidad objetivo con hosting compartido básico sin CLI, salvo revisión técnica posterior en un entorno más robusto.

Esta decisión busca priorizar previsibilidad operativa y reducir riesgo en despliegues cerrados preparados en local.
<<END SECTION>>

<<SECTION: CONVENCIONES DE MULTI-TENANCY>>
CONVENCIONES DE MULTI-TENANCY

Las entidades del negocio deben incluir siempre:

tenant_id

El tenant determina el aislamiento lógico de los datos dentro del sistema.

Las entidades del sistema no deben incluir tenant_id.

Ejemplos de entidades del sistema:

User
Tenant
Membership
<<END SECTION>>

<<SECTION: ACCESO OPERATIVO POR TENANT>>
ACCESO OPERATIVO POR TENANT

La restricción operativa de acceso a empresa debe validarse por memberships.status = active.

No debe utilizarse desactivación global de users para resolver bajas, bloqueos o pérdida de acceso específicos de un tenant.

<<SECTION: CONVENCIONES DE ACCESO POR MÓDULO Y ROLES>>
SECTION_VERSION: 00003

CONVENCIONES DE ACCESO POR MÓDULO Y ROLES

La autorización operativa del sistema debe implementarse respetando la arquitectura central definida para acceso y permisos.

Reglas de implementación:

no hardcodear permisos en controllers, policies o blades si existe resolución central

las policies deben consumir la capa central de autorización y no replicar lógica

los controllers deben delegar en authorize(...) y no implementar lógica paralela

las vistas no deben inferir permisos

las vistas deben respetar permisos reales mediante @can solo como verificación final cuando la capacidad no dependa de contexto

los alcances visibles en UI deben resolverse mediante catálogos o resolvers y no reconstruirse manualmente

la habilitación de módulos depende de la capa central de acceso por tenant y no debe asumirse por defecto

la UI de permisos del tenant debe operar únicamente sobre módulos habilitados previamente

Regla crítica validada:

no se debe usar autorización abstracta cuando la capacidad depende de contexto

no utilizar authorize('create', Model::class) en estos casos

debe utilizarse la capa central con parámetros explícitos

Regla de consistencia obligatoria:

view_any debe ser igual o más amplio que view

no debe existir un contrato donde view_any quede más restringido que view para el mismo módulo

Reglas adicionales consolidadas:

los permisos pueden diferenciarse por capability dentro de un mismo módulo

los roles son acumulables y el permiso efectivo surge de su combinación

la capa de autorización debe contemplar la fusión de constraints por capability

la UI de configuración de permisos no debe simplificar el modelo a matrices planas de enabled + scope

cuando el dominio del módulo lo requiera, deben utilizarse catálogos dinámicos como fuente única de verdad para valores configurables (por ejemplo OrderCatalog para allowed_kinds)

no deben hardcodearse valores de dominio en frontend

La definición completa de roles, capacidades, scopes y constraints se encuentra en:

<DOC_REF slug="seguridad_acceso_proyecto_app_base">Seguridad y acceso del proyecto</DOC_REF>

<<END SECTION>>

<<SECTION: ROLECATALOG Y MODULECATALOG>>
ROLECATALOG Y MODULECATALOG

RoleCatalog es el catálogo base mínimo y reservado del sistema para roles estructurales.

ModuleCatalog es la fuente de verdad para slugs de módulos cerrados del sistema.

Reglas:

no hardcodear slugs funcionales del sistema en controllers, policies, helpers o blades si ya existe catálogo técnico

los roles base del sistema deben resolverse desde RoleCatalog

los módulos base del sistema deben resolverse desde ModuleCatalog

los roles base sembrados al crear el tenant no deben modificarse ni eliminarse

los roles base cerrados del tenant no deben crecer por necesidades funcionales puntuales

las variantes funcionales futuras deben resolverse mediante perfiles funcionales y overrides, no creando nuevos roles base

a futuro el tenant podrá ajustar capacidades dentro de la arquitectura vigente, pero sin alterar la base reservada del sistema
<<END SECTION>>

<<SECTION: EVOLUCIÓN PREVISTA DE PERMISOS>>
EVOLUCIÓN PREVISTA DE PERMISOS

La implementación de permisos del sistema debe apoyarse en una capa centralizada compuesta por resolvers y catálogos del dominio.

Reglas:

toda nueva funcionalidad de autorización debe integrarse en la capa central existente

no deben crearse matrices paralelas ni lógica aislada por módulo

las policies deben mantenerse como wrappers mínimos de consumo de esa capa

los resolvers no definen el contrato de permisos

los resolvers deben resolver y consumir el contrato persistido, incluyendo scopes, constraints y visibilidad de dataset según lo definido en base de datos

la evolución del sistema de permisos debe respetar la arquitectura vigente sin introducir variantes incompatibles

La doctrina completa de permisos y su modelo operativo se define en:

<<SECTION: TRAIT TENANTSCOPED>>
TRAIT TENANTSCOPED

Las entidades del negocio deben utilizar el trait:

TenantScoped

Este trait se encarga de:

filtrar automáticamente por tenant_id
asignar tenant_id al crear registros

Esto garantiza que los datos de diferentes empresas no se mezclen.
<<END SECTION>>

<<SECTION: USO DE TENANT_ID EN FILLABLE>>
USO DE TENANT_ID EN FILLABLE

Como regla general, tenant_id no debe exponerse en formularios ni venir desde input del usuario.

El tenant siempre debe resolverse internamente mediante:

app('tenant')

y asignarse automáticamente por el sistema.

Algunos modelos padre del proyecto exponen tenant_id en fillable por consistencia histórica, pero igualmente no debe venir desde formularios públicos ni desde input del usuario.

En entidades hijas (por ejemplo items o registros dependientes), tenant_id tampoco debe exponerse desde formularios del usuario.

Sin embargo, si el registro hijo se crea internamente desde un flujo controlado del sistema y el tenant se hereda explícitamente desde la entidad padre, puede incluirse tenant_id en fillable para permitir la creación interna consistente mediante create().

Esto permite manejar correctamente casos como:

document_items creados desde documentos
order_items copiados hacia documentos derivados
<<END SECTION>>

<<SECTION: CONVENCIONES DE ENTIDADES HIJAS>>
CONVENCIONES DE ENTIDADES HIJAS

Las entidades hijas deben cumplir:

pertenecer a una cabecera
heredar tenant_id de la cabecera
no recibir tenant_id desde formularios del usuario

Ejemplos:

order_items
document_items

Reglas actualizadas:

cuando la entidad hija forme parte natural de la operación de la cabecera, debe tratarse preferentemente como recurso delegado al padre

la autorización de create, update y delete debe resolverse contra la capacidad update del recurso padre correspondiente

no debe modelarse una autorización autónoma innecesaria para la entidad hija si su ciclo de vida depende funcionalmente de la cabecera

si existiera una policy propia, debe limitarse a delegar explícitamente en el padre y no duplicar doctrina paralela

attachments sigue el mismo criterio general de delegación al recurso padre aunque no sea una entidad hija cabecera/detalle clásica

cualquier excepción a esta regla debe justificarse por necesidad real del dominio y no por comodidad de implementación
<<END SECTION>>

<<SECTION: CONVENCIONES DE OWNERSHIP FUNCIONAL Y PERMISOS OPERATIVOS SIMPLES>>
CONVENCIONES DE OWNERSHIP FUNCIONAL Y PERMISOS OPERATIVOS SIMPLES

Cuando un módulo lo permita, puede adoptarse un criterio simple de ownership funcional del registro en lugar de una capa temprana de permisos compleja.

Reglas:

usuario normal gestiona los registros de su responsabilidad operativa

owner y admin pueden gestionar registros ajenos dentro del tenant

los borrados sensibles pueden reservarse a owner y admin aunque la edición o seguimiento sea más amplia

si un módulo tiene asignado directo, assigned_user_id define la responsabilidad operativa actual

si el registro puede reasignarse, el nuevo asignado pasa a ser responsable operativo

si no existe propietario directo, el control sensible queda en owner y admin o en la policy específica del módulo

Este patrón ya se considera válido para tasks.
<<END SECTION>>

<<SECTION: REGLA PARA MÓDULOS OPERATIVOS COMPARTIDOS DEL TENANT>>
REGLA PARA MÓDULOS OPERATIVOS COMPARTIDOS DEL TENANT

Cuando un módulo operativo compartido del tenant no tenga responsable operativo directo:

la policy debe permitir view, create y update a los roles y capacidades habilitados del módulo

delete debe quedar restringido a owner y admin

no debe inventarse ownership artificial mediante assigned_user_id, owner_user_id o restricciones por created_by si el módulo no lo necesita realmente

Esto aplica especialmente a módulos base o compartidos como:

parties
assets
products
orders

Aclaración conceptual consolidada:

orders debe tratarse como módulo operativo interno del tenant

documents no debe absorber órdenes de trabajo operativas ni duplicar la función interna de orders
<<END SECTION>>

<<SECTION: CONVENCIONES DE SUPPORT Y HELPERS DE ACCESO>>
CONVENCIONES DE SUPPORT Y HELPERS DE ACCESO

La lógica transversal de acceso tenant/roles debe centralizarse en clases de soporte reutilizables.

Reglas:

no implementar lógica de permisos en controllers o blades

no duplicar resolución de acceso en múltiples capas

utilizar siempre la capa central de autorización para resolver permisos efectivos

los helpers o clases de soporte deben limitarse a facilitar el acceso a esa capa y no a redefinir su comportamiento

las clases bajo app/Support/Auth deben considerarse punto de acceso técnico a la lógica central, no fuente alternativa de definición

La definición completa del modelo de autorización se encuentra en:

<<END SECTION>>

<<SECTION: CONVENCIONES DE POLICIES>>
CONVENCIONES DE POLICIES

La autorización por acción debe resolverse preferentemente mediante Policies.

Acciones típicas:

viewAny
view
create
update
delete

Reglas:

evitar lógica de permisos duplicada en controllers

usar controllers delgados y delegar la autorización a policies o a la capa central

si una policy necesita conceptos cerrados del sistema, reutilizar catálogos y resolvers

no utilizar casts booleanos ambiguos sobre scopes

no usar fallbacks implícitos cuando la capa central ya resuelve la capacidad

cuando exista una policy:

el controller debe invocar authorize(...) o authorizeResource(...)
la vista debe respetar permisos reales con @can

las policies deben trabajar con capacidades concretas, scopes efectivos y constraints aplicables

las policies deben tender a wrappers mínimos sobre el kernel central de autorización

distinción obligatoria:

viewAny controla acceso al dataset (listados)
view controla acceso a registros individuales

no debe utilizarse view para resolver visibilidad de listados

no deben resolverse listados mediante lógica indirecta o reinterpretación de scopes

viewAny debe consumir directamente el scope resuelto por el resolver

viewAny y view no deben resolverse como booleanos genéricos si el resolver ya entrega scope válido

view_any debe ser igual o más amplio que view

no deben resolver lectura únicamente con canUseModule

no deben reinterpretar tenant_all, own_assigned o limited como aliases legacy

las entidades hijas no deben asumir herencia automática de autorización

order_items, document_items y attachments deben resolverse como recursos delegados al padre cuando la acción dependa naturalmente de la cabecera

una policy que no se consume desde controller y vista no protege efectivamente el sistema
<<END SECTION>>

<<SECTION: REGLA DE NAVEGACIÓN Y VISIBILIDAD DE UI>>
REGLA DE NAVEGACIÓN Y VISIBILIDAD DE UI

La interfaz no debe mostrar accesos que el usuario no puede usar realmente.

Reglas:

navbar, dashboard y accesos rápidos deben respetar acceso real del usuario al módulo

las acciones visibles en listados, show y sublistados deben respetar permisos reales mediante @can

la navegación no debe asumir acceso total al tenant si el módulo o la acción están restringidos por rol

la navegación principal del tenant debe construirse desde ModuleCatalog cuando exista una capa de navegación central disponible

no se deben hardcodear módulos navegables en componentes de layout cuando el sistema ya pueda resolverlos desde catálogo y acceso efectivo

distinción obligatoria en UI:

la visibilidad de listados completos debe depender de view_any

la visibilidad de acciones sobre registros individuales debe depender de view, update o delete según corresponda

no debe utilizarse view como criterio para mostrar listados completos

view_any debe ser igual o más amplio que view

no deben existir inconsistencias entre:

lo que el listado muestra
lo que el usuario puede abrir

toda acción derivada, campo relacional o navegación hacia otra entidad modular por *_id debe verificar primero si el módulo relacionado está habilitado para el tenant

si el módulo destino está deshabilitado:

no deben renderizarse enlaces
no deben renderizarse campos
no deben renderizarse bloques completos asociados
no deben mantenerse accesos residuales

el breadcrumb representa recorrido contextual o estructural de la pantalla actual

el navbar representa anclaje funcional estable del sistema

el navbar no debe copiar ni duplicar el breadcrumb

cuando exista navegación contextual entre módulos relacionados, el módulo activo del navbar puede resolverse por origen funcional del flujo y no solo por la pantalla hoja actualmente abierta
<<END SECTION>>

<<SECTION: CONVENCIONES DE BLADE Y FRONTEND LIVIANO>>
CONVENCIONES DE BLADE Y FRONTEND LIVIANO

Las vistas Blade deben mantenerse simples, explícitas y previsibles.

Reglas:

evitar lógica de negocio en Blade

permitir condicionales simples de interfaz

extraer bloques repetidos a componentes o partials cuando el patrón ya sea claro

no introducir complejidad innecesaria de frontend

Separación recomendada:

clases CSS = estilo
data-* = comportamiento

Las acciones visibles deben respetar siempre permisos reales

no deben renderizarse acciones no permitidas aunque el backend las bloquee

Blade puede utilizar @can como verificación final solo cuando la capacidad no dependa de contexto

Regla crítica validada:

cuando una capacidad depende de constraints, la UI no debe evaluarla directamente

no utilizar @can('create') en módulos con constraints

la UI debe consumir flags ya resueltos desde backend

la resolución del dataset no debe reconstruirse en Blade

debe venir ya resuelta desde controllers y capa central de autorización

en vistas embebidas o sublistados:

cuando la autorización dependa del recurso padre, las acciones deben evaluarse contra el modelo padre real

esto aplica explícitamente a:

attachments
order_items
document_items

en attachments:

la UI no debe evaluar create Attachment

debe evaluar update sobre el attachable

los botones de alta, edición y eliminación deben alinearse con esa lógica

las vistas no deben reconstruir lógica de permisos por fuera de la capa central

la UI no debe usar ni reinterpretar scopes legacy como all o mine

si la interfaz necesita mostrar o seleccionar alcances, debe consumir únicamente scopes válidos del catálogo vigente

esto incluye en especial:

tenant_all
own_assigned
limited

toda acción derivada, campo relacional o navegación hacia otra entidad modular por *_id debe verificar primero si el módulo relacionado está habilitado para el tenant

si el módulo relacionado está deshabilitado:

el bloque completo no debe renderizarse
no debe mostrarse versión deshabilitada ni placeholder
no debe mantenerse acceso indirecto

cuando una pieza global del layout exponga información del sistema, debe resolverse desde provider o capa central y no inline

esto incluye:

footer del shell
capas de permisos visibles
pantallas de configuración del owner
<<END SECTION>>

<<SECTION: CONVENCIONES DE FORMULARIOS OPERATIVOS>>
CONVENCIONES DE FORMULARIOS OPERATIVOS

Regla general:

los formularios operativos de create y edit deben resolverse en pantallas dedicadas

los modales no deben utilizarse para operaciones principales de creación o edición

los modales pueden utilizarse únicamente para:

confirmaciones simples
visualización puntual y liviana

Objetivos de esta convención:

reducir complejidad de JavaScript
evitar problemas de estado, foco y validación
mantener consistencia de navegación y breadcrumb

Si una operación requiere validación completa, navegación consistente o interacción de formulario real, debe resolverse como pantalla y no como modal.

Regla de composición:

las vistas create y edit deben contener el formulario completo, incluyendo:

<form> @csrf @method cuando corresponda form-actions

los parciales _form deben contener únicamente:

campos
ayudas
mensajes de error
bloques internos estrictamente propios de los campos

el parcial _form no debe abrir ni cerrar el <form> principal ni contener acciones finales del formulario

cuando un _form utilice un wrapper raíz para agrupar múltiples .form-group, ese wrapper debe usar class="form" para heredar el espaciado vertical base del sistema
<<END SECTION>>

<<SECTION: CONVENCIONES DE MÓDULOS TRANSVERSALES>>
CONVENCIONES DE MÓDULOS TRANSVERSALES

Un módulo transversal es aquel que:

no tiene navegación principal propia
se integra dentro del show de otros módulos
puede tener pantallas propias para acciones como create o edit

Reglas:

debe diseñarse desacoplado de módulos específicos

no debe implementar navegación paralela propia mientras su uso principal siga siendo embebido o contextual

puede tener pantallas dedicadas para acciones operativas si eso mejora consistencia y mantenibilidad

no debe depender de nombres de clase del backend en el frontend

debe poder evolucionar en el futuro hacia una vista independiente sin romper su base conceptual ni su integración actual

cuando un patrón transversal se repita en varios módulos, debe tender a resolverse con partials, componentes o catálogos reusables antes que con variantes aisladas por módulo

Ejemplo actual de este patrón:

attachments
<<END SECTION>>

<<SECTION: CONVENCIONES ESPECÍFICAS PARA ATTACHMENTS>>
CONVENCIONES ESPECÍFICAS PARA ATTACHMENTS

Attachments se considera un módulo transversal embebido alineado al patrón general del proyecto.

Reglas actuales:

attachments debe integrarse principalmente dentro del show de otros módulos y no como navegación principal paralela

create y edit de attachments deben resolverse en pantallas dedicadas y no en modales operativos

cuando el patrón de attachments se repita entre módulos, debe resolverse con partials o componentes reusables

AttachmentCatalog debe actuar como fuente de verdad para kinds, etiquetas y segmentaciones visibles del módulo

en el embebido visible de attachments, kind es la dimensión principal y estable de segmentación

no debe agregarse una segunda dimensión visible equivalente si no aporta valor operativo real

attachments no debe depender en frontend de nombres de clase del backend para resolver integración, comportamiento o segmentación
<<END SECTION>>

<<SECTION: CONVENCIONES DE PRESERVACIÓN DE CONTEXTO UX>>
CONVENCIONES DE PRESERVACIÓN DE CONTEXTO UX

En vistas con tabs, filtros o acciones internas, debe preservarse el contexto funcional al refrescar o redirigir cuando eso mejore la UX.

Ejemplos de contexto a preservar:

tab activa
filtros
paginación cuando aplique

Las tabs mencionadas en esta convención refieren a tabs internas de contenido dentro de una pantalla.

No debe asumirse todavía como convención general del sistema una navegación entre rutas dentro de tabs de app o tipo workspace.
<<END SECTION>>

<<SECTION: NAVEGACIÓN ESTRUCTURAL Y CONTEXTUAL>>
NAVEGACIÓN ESTRUCTURAL Y CONTEXTUAL

La navegación del sistema debe distinguir entre:

navegación estructural
navegación contextual

navegación estructural:
expresa la ubicación estable de la pantalla dentro del sistema o dentro de su módulo

navegación contextual:
expresa el origen funcional desde el cual el usuario llegó a una entidad relacionada, cuando ese origen sigue siendo relevante para la tarea actual

Reglas generales:

toda pantalla debe tener una resolución estructural válida como base, aunque en ciertos flujos pueda mostrarse en modo contextual

la navegación contextual solo debe aplicarse cuando exista un flujo relacionado real y útil para la continuidad de uso

no debe inventarse un origen contextual si la pantalla fue abierta de forma directa, desde index propio, desde búsqueda o desde navegación general del módulo

el breadcrumb del sistema es uno solo, pero debe poder resolverse en modo estructural o en modo contextual según el caso

si ya existe una capa central reutilizable para resolver contexto de navegación, no duplicar esa lógica en múltiples controllers, blades o helpers dispersos

controller resuelve contexto
vista consume contexto

no reconstruir breadcrumb de negocio desde url()->previous() ni desde supuestos del historial del navegador

breadcrumb, retorno al origen y redirects posteriores a acciones deben mantenerse consistentes entre entidades relacionadas

la preservación de contexto UX debe aplicarse con criterio y solo cuando aporte continuidad real de uso

como regla general:

usar navegación estructural por defecto

usar navegación contextual solo como override útil cuando el origen funcional deba preservarse

cuando no exista contexto válido o el origen no agregue valor operativo, debe mostrarse navegación estructural normal
<<END SECTION>>

<<SECTION: RESOLUCIÓN TÉCNICA DEL TRAIL DE NAVEGACIÓN>>
RESOLUCIÓN TÉCNICA DEL TRAIL DE NAVEGACIÓN

Cuando la navegación contextual ya exista como patrón real del sistema, debe resolverse mediante un mecanismo central reutilizable y estable.

Reglas:

el mecanismo central vigente del proyecto para breadcrumb contextual es NavigationTrail

los módulos que participen en navegación contextual deben apoyarse en builders o resolutores propios compatibles con NavigationTrail, sin crear variantes paralelas de criterio

controller resuelve trail

vista consume trail

si una pantalla relacionada recibe un trail válido, debe poder heredarlo, extenderlo o colapsarlo sin reconstruir manualmente la navegación desde parámetros sueltos, historial del navegador o deducciones implícitas de relaciones del modelo

los nodos del trail deben almacenar URLs limpias de la entidad actual

no debe serializarse el propio trail dentro del campo url de los nodos

el trail se propaga solo en links, forms y redirects relevantes

cuando el trail vuelva a un nodo ya existente, debe colapsarse de forma limpia y no duplicada

cuando un módulo ya esté migrado a NavigationTrail, no debe seguir usando mecanismos legacy equivalentes o redundantes, incluyendo:

url()->previous()

context_type / context_id

NavigationContext

reconstrucción manual por relaciones del modelo

en tablas embebidas o listados contextuales, cuando el contexto real de navegación sea la fila y no solo el contenedor, los links profundos a entidades relacionadas deben usar el trail de la fila

en tablas embebidas o relacionales, el link principal de la fila debe construir el trail desde el contexto del contenedor más el nodo de la fila

los links relacionados de esa misma fila deben reutilizar ese trail contextual ya construido y no reiniciar el contexto desde el contenedor ni desde mecanismos manuales

los nodos transitorios de formulario como create, edit, items.create o items.edit no deben persistir al volver a show después de store o update exitoso

las acciones cancelar dentro de formularios contextuales deben respetar el trail completo vigente y no volver por defecto al show estructural del módulo

las decisiones internas de implementación pueden evolucionar, pero la convención estable es:

motor central reutilizable

builders compatibles por módulo

controller resuelve

vista consume

sin deducción desde historial

sin duplicación por módulo

sin serialización recursiva del trail dentro de nodos

<<SECTION: CONVENCIONES DE NAVEGACIÓN CONTEXTUAL>>
CONVENCIONES DE NAVEGACIÓN CONTEXTUAL

Cuando exista navegación cruzada entre entidades relacionadas, no debe resetearse el contexto de origen sin necesidad funcional real.

Reglas:

si ya existe una capa central reutilizable para resolver contexto de navegación, no duplicar esa lógica en múltiples controllers

no hardcodear variantes divergentes por módulo si la regla de navegación ya existe a nivel reutilizable

la preservación de contexto UX debe aplicarse con criterio y solo cuando aporte continuidad real de uso

breadcrumb, retorno al origen y preservación de contexto deben resolverse de forma consistente entre módulos relacionados

cuando una navegación surja desde una tabla contextual, debe verificarse si el contexto real a preservar es:

solo el módulo contenedor

o el módulo contenedor más la entidad fila desde la cual se abre la entidad relacionada

esa decisión no debe dejarse implícita ni variar accidentalmente entre módulos

en tablas embebidas o relacionales, el link principal de la fila debe consolidar primero el contexto completo de esa fila

los links secundarios o relacionados de esa fila deben reutilizar ese mismo contexto y no reconstruirse desde la pantalla contenedora
