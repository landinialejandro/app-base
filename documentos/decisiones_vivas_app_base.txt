DECISIONES VIGENTES

No abrir módulo users por necesidades de colaboradores internos.

La resolución vigente es:

users = identidad global
memberships = pertenencia operativa
raíces tenant reusables para datasets internos

Inventory debe leerse como frente funcional real y no como efecto lateral de products.

Orders conserva centralidad operativa pero puede hostear capacidades de otros módulos sin absorber su lógica.

La evolución reusable correcta prioriza surfaces oficiales del módulo oferente antes que soluciones locales del consumidor.

Security permanece como kernel central de autorización efectiva.

Las show pages deben converger hacia slots explícitos y montaje coordinado.

NavigationTrail resuelve contexto externo.

return_tab complementa contexto interno del host cuando aplica.

No backward patching como cierre estable de migraciones.

Para parties, la constraint funcional de seguridad es allowed_party_roles.

Lectura vigente:

allowed_party_roles se evalúa contra party_roles.role.

allowed_kinds no aplica a parties.

Motivo:

parties.kind representa únicamente tipo de contacto y no relación comercial ni vínculo interno.

La gestión de contactos con relación employee se autoriza mediante PartyEmployeeContactAuthorization.

Regla vigente:

employee es relación sensible.

No se protege solo en UI.

Debe bloquearse en backend.

No se notifica al owner por cada intento individual de manipulación.

Criterio acordado:

primero registrar o auditar eventos sensibles
notificar solo ante patrón, reiteración o severidad

Este criterio no implica implementación vigente de auditoría o notificación.

OperationalActivity queda decidido como capacidad transversal tenant-scoped de trazabilidad operativa.

Lectura vigente:

no es CRUD ordinario
no es navegación principal
no pertenece a parties, tasks, orders, documents, appointments, projects, assets, products, attachments ni inventory
no autoriza operaciones
no reemplaza Security
no reemplaza policies
no valida la operación original
no debe bloquear el flujo principal si falla el registro

Regla vigente:

el módulo dueño autoriza, valida y ejecuta su operación.

Luego emite un evento operativo liviano.

OperationalActivity registra si corresponde.

La escritura de OperationalActivity es system-owned y append-only.

No se implementa create manual desde UI.

No se implementa edit.

No se implementa update.

No se implementa delete.

No se implementa destroy.

No se implementa carga manual por formularios.

Módulos con integración validada a OperationalActivity:

appointments
tasks
orders
documents
projects
assets
products
attachments
parties
inventory

Decisión vigente sobre inventory:

OperationalActivity registra InventoryOperation como ancla operativa agrupada.

InventoryMovement no se registra como actividad principal cuando la operación agrupada representa la acción del usuario.

InventoryMovement conserva la trazabilidad fina y atómica de stock dentro de inventory.

Decisión vigente sobre parties:

OperationalActivity registra actividad sobre Party.

Los cambios de party_roles pueden registrarse como metadata operativa dentro de la actividad de Party.

party_roles no se convierte en módulo propio.

party_roles no abre navegación propia.

Decisión vigente sobre attachments:

attachments puede registrar creación y actualización de metadata operativa.

No se registra como actividad operativa vigente:

download
preview
vista
print
PDF
destroy

salvo decisión explícita posterior.

Decisión vigente sobre lectura general de OperationalActivity:

la lectura general tenant-scoped del log operativo se autoriza mediante:

operational_activity.view_any

Esta capability se resuelve desde Security sin incorporar OperationalActivity a ModuleCatalog.

Motivo:

ModuleCatalog gobierna módulos habilitados, navegación, seeders, surfaces y permisos funcionales.

Agregar OperationalActivity allí lo convertiría en módulo funcional ordinario.

Por eso, OperationalActivity se mantiene como capacidad transversal tenant-scoped y su lectura general se resuelve mediante capacidad tenant no modular.

Estado de asignación inicial:

owner
admin

Decisión vigente sobre ubicación visual inicial:

la primera lectura visual general de OperationalActivity se ubica en tenant/profile mediante tab Actividad.

No se agrega navegación principal para OperationalActivity.

No se implementa vista general independiente del log.

No se considera todavía surface reutilizable.

Decisión vigente sobre acceso a tenant/profile:

tenant/profile sigue restringido a owner en el estado actual.

admin tiene permiso sembrado para operational_activity.view_any, pero no accede todavía a tenant/profile por la restricción vigente del controlador.

El acceso futuro de admin a tenant/profile queda pendiente explícito y debe distinguir visualización permitida de edición restringida.

Decisión vigente sobre navegación desde actividad:

no se persisten URLs finales en operational_activities.

La navegación al registro asociado se resuelve en capa de lectura mediante OperationalActivityLinkResolver.

OperationalActivityLinkResolver puede resolver links solo cuando exista ruta show canónica segura.

Casos sin link por ahora:

InventoryOperation
Attachment

Decisión vigente sobre Project Lab y colas:

Project Lab puede tener acción externa al producto para procesar cola localmente.

Esto no debe trasladarse al SaaS productivo.

No debe implementarse botón de queue dentro del producto SaaS.

Project Lab es la herramienta operativa local vigente para aplicar, auditar y validar intervenciones técnicas.

Regla vigente:

todo snippet destinado a Project Lab debe respetar el contrato real de la herramienta.

FILE corresponde a archivo completo.

TARGET corresponde a operación parcial.

No se deben mezclar FILE y TARGET.

No se deben entregar encabezados ambiguos que obliguen al usuario a corregir el formato.

Ante falta de datos reales:

NO SE ASUME
NO SE SUPONE
