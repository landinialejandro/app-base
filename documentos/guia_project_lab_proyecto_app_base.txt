NATURALEZA DE PROJECT LAB

Project Lab es la estación operativa local de desarrollo, auditoría, validación y mantenimiento técnico del proyecto.

Ubicación vigente:

tools/project-lab

Project Lab no forma parte del sistema SaaS productivo.

No debe publicarse ni exponerse en entornos accesibles desde internet.

Su uso corresponde únicamente en entorno local controlado.

Funciones principales:

ejecución de Tinker
ejecución de Tinker desde clipboard
ejecución de comandos Artisan permitidos
herramienta Lab de código
herramienta Lab documental
herramienta Lab de auditoría
snippets rápidos
búsqueda operativa de métodos o fragmentos
consola unificada
guardado de consola como auditoría
copiado de salida para compartir evidencia
persistencia auxiliar local cuando corresponda
procesamiento local de cola pendiente cuando se requiera validar eventos queued

Project Lab debe leerse como herramienta técnica local del proyecto.

No debe confundirse con una funcionalidad del producto final.

Reglas:

Project Lab no define reglas de negocio.
Project Lab no define autorización del producto.
Project Lab no define navegación del usuario final.
Project Lab no reemplaza documentación rectora.
Project Lab no reemplaza criterio arquitectónico.
Project Lab no reemplaza validación runtime.
Project Lab no reemplaza pruebas en navegador cuando correspondan.

Project Lab define únicamente el contrato operativo local para aplicar, auditar, ejecutar y validar tareas técnicas sobre el proyecto.

Regla específica para colas:

Project Lab puede facilitar el procesamiento manual de colas en local cuando sea necesario validar eventos queued, jobs y persistencia final.

Esta capacidad no debe trasladarse como botón, acción o pantalla dentro del producto SaaS.

HERRAMIENTAS DISPONIBLES

Project Lab dispone de tres herramientas Lab principales:

code

CODE

Se utiliza para aplicar actualizaciones de código compatibles con los contratos vigentes de Project Lab.

Puede operar sobre:

archivos PHP completos
archivos Blade completos
archivos CSS completos
archivos JS completos
archivos TPL completos
métodos PHP existentes
métodos PHP nuevos dentro de clases existentes
funciones JS existentes
secciones CSS o JS delimitadas

Reglas:

debe trabajar siempre dentro del projectRoot actual
debe resolver rutas mediante la aduana central de rutas vigente
debe escribir archivos mediante la aduana central de escritura vigente
debe rechazar rutas fuera del proyecto
debe validar antes de escribir cuando el error sea detectable
debe validar sintaxis PHP cuando corresponda
debe restaurar o eliminar cambios PHP si la validación de sintaxis posterior falla
no debe aplicar parches ambiguos
no debe mezclar contratos de archivo completo con contratos parciales

Regla específica para TPL:

los archivos TPL completos deben usarse para plantillas operativas controladas por Project Lab.

Caso vigente:

tools/project-lab/macros/audit/*.sh.tpl

Estos archivos deben llevar encabezado FILE compatible y versionado.

DOCS

Se utiliza para aplicar actualizaciones documentales mediante bloques completos de sección.

Debe operar con:

encabezado REEMPLAZAR EN: [doc_slug]
doc_slug existente
nombre exacto de sección
bloque completo delimitado
SECTION_VERSION
contenido compatible con el documento destino

Reglas:

no debe inventar secciones
no debe reemplazar fragmentos internos
no debe aplicar cambios sin SECTION_VERSION
no debe aceptar nombres aproximados
no debe mezclar explicación narrativa con bloque automatizado
debe generar backup documental antes de reemplazar
debe usar backup con timestamp para evitar pisar respaldos anteriores
debe devolver OK o ERROR explícito

AUDIT

Se utiliza para ejecutar auditorías locales, inspecciones, comandos de apoyo y macros audit sobre el proyecto real.

Lectura vigente:

audit ejecuta bash libre local desde projectRoot.

No es sandbox del sistema operativo.

Su uso queda a riesgo del usuario técnico local.

Reglas:

debe ejecutarse desde el projectRoot actual
debe utilizarse para obtener evidencia real
puede ejecutar auditorías fuertes cuando el usuario técnico lo solicite
no debe endurecerse con bloqueos heurísticos que rompan auditorías necesarias
no debe tratarse como protección de sistema operativo
no debe ejecutar prompts de chat como si fueran comandos
debe normalizar CRLF antes de ejecutar scripts cuando corresponda

Uso esperado:

inspeccionar archivos reales
buscar métodos o residuos
generar auditorías en documentos/auditoria
validar rutas
validar contratos de código
obtener evidencia antes de intervenir
guardar salidas de consola como auditoría
ejecutar macros audit internas mediante prefijo controlado

Macros audit:

Project Lab puede interpretar comandos rápidos iniciados con el prefijo:

Estos comandos no reemplazan audit.

identifica el nombre de macro

REGLA ESTRICTA DE FORMATOS

Project Lab debe rechazar formatos ambiguos.

Un snippet solo puede aplicarse si coincide claramente con un contrato válido.

Contratos válidos principales:

archivo PHP completo
archivo Blade completo
archivo CSS completo
archivo JS completo
reemplazo de método PHP
alta de método PHP
reemplazo de función JS
reemplazo de sección CSS o JS
reemplazo de sección documental

Regla crítica:

Project Lab debe fallar antes de escribir cuando el error sea detectable.

Si un bloque no cumple exactamente un contrato válido:

no debe aplicar cambios
no debe crear archivos
no debe crear directorios
no debe escribir fuera del projectRoot
no debe normalizar silenciosamente casos peligrosos
debe devolver ERROR explícito

No se aceptan cambios ambiguos como éxito parcial.

No se aceptan escrituras preventivas para luego validar.

Orden obligatorio:

validar contrato
resolver ruta segura dentro del projectRoot
validar existencia cuando corresponda
aplicar cambio
validar resultado cuando corresponda
mostrar salida operativa

Regla:

un formato pensado para archivo completo no debe usarse para reemplazar métodos.

Un formato pensado para operación parcial no debe usarse para crear archivos completos.

Un bloque documental no debe usarse para modificar código.

Un bloque de código no debe usarse para modificar documentación.

<<SECTION: REGLA ANTIAMBIGÜEDAD EN HEADERS>>
SECTION_VERSION: 00002

REGLA ANTIAMBIGÜEDAD EN HEADERS

Un encabezado FILE identifica archivo completo.

Un encabezado TARGET identifica operación parcial.

Regla crítica:

FILE no debe contener operadores de método.

No son válidos como archivo completo los encabezados que mezclen ruta de archivo con operadores de método.

Estos casos deben resolverse de una sola manera:

si corresponde operación parcial, debe usarse TARGET
si corresponde archivo completo, debe usarse FILE sin operadores parciales
si el parser no puede clasificarlo con seguridad, debe devolver ERROR

Nunca debe crearse un archivo cuya ruta incluya operadores de método.

Regla obligatoria:

si un header FILE contiene operadores parciales, Project Lab no puede tratarlo como archivo completo.

Debe devolver ERROR explícito o normalizarlo hacia TARGET solo si el parser lo reconoce de forma segura como operación parcial válida.

Regla recomendada:

para operaciones parciales usar siempre TARGET.

Para archivos completos usar siempre FILE.

No mezclar ambos contratos en un mismo snippet.

<<END SECTION>>

<<SECTION: FORMATO DE ARCHIVO PHP COMPLETO>>
SECTION_VERSION: 00001

FORMATO DE ARCHIVO PHP COMPLETO

Cuando Project Lab recibe un snippet para crear o reemplazar un archivo PHP completo, el contenido debe comenzar como archivo PHP válido.

Formato correcto:

<?php
// FILE: app/Support/Ejemplo.php | V1

Reglas:

la apertura PHP debe estar al inicio del archivo

el encabezado FILE debe ubicarse luego de <?php

el encabezado FILE identifica archivo completo

la ruta debe ser una ruta de archivo PHP

la ruta debe resolverse dentro del projectRoot

el snippet no debe mezclar FILE con TARGET

el snippet no debe contener operadores de método si se declara como archivo completo

el archivo final debe validar sintaxis PHP cuando corresponda

Formato incorrecto para archivo PHP completo:

// FILE: app/Support/Ejemplo.php | V1
<?php

Regla:

un snippet PHP completo no debe comenzar con comentario FILE antes de la apertura PHP si la herramienta espera archivo PHP válido desde el inicio.

Si el parser no puede clasificar el bloque con seguridad, debe devolver ERROR antes de escribir.

Caso detectado:

un modelo PHP no se creó correctamente por usar formato inicial incorrecto del snippet.

Corrección vigente:

para archivo PHP completo, comenzar siempre con <?php y luego declarar el marcador FILE.

Regla final:

Project Lab debe validar el contrato antes de escribir.

No debe normalizar silenciosamente un formato ambiguo si puede provocar creación incorrecta, ruta inválida, archivo inválido o interpretación errónea del snippet.

<<END SECTION>>

<<SECTION: ERRORES OBLIGATORIOS>>

ERRORES OBLIGATORIOS

Project Lab debe devolver ERROR y no escribir cambios cuando detecte fallas validables antes de la escritura.

Casos obligatorios:

contenido vacío
herramienta Lab no reconocida
header no compatible
ruta destino vacía
ruta fuera del projectRoot
ruta con operadores inválidos para archivo completo
archivo inexistente en operación parcial
método inexistente en reemplazo parcial
método ya existente en alta de método
función JS inexistente en reemplazo parcial
sección documental inexistente
doc_slug inexistente
sección CSS o JS inexistente
intento de usar formato de archivo completo para operación parcial ambigua
bloque documental sin encabezado REEMPLAZAR EN cuando la herramienta vigente lo requiere
bloque documental sin SECTION_VERSION válido
bloque documental con DOC_SLUG como encabezado operativo si Project Lab Docs espera REEMPLAZAR EN
header FILE que combine operadores de método
snippet que mezcle FILE y TARGET de forma ambigua
archivo PHP completo que no comience como PHP válido
archivo PHP completo con apertura PHP ubicada después del encabezado FILE cuando el contrato espera PHP válido desde el inicio
intento de escritura fuera del projectRoot
falla al crear backup documental antes de reemplazar
falla de validación de sintaxis PHP posterior a la escritura
falla de restauración o limpieza luego de lint PHP inválido
macro audit inexistente
macro audit con argumentos inválidos
comando rápido con prefijo >> no reconocido
intento de ejecutar prompt narrativo como si fuera comando shell o audit válido

Reglas:

si el error es detectable antes de escribir, Project Lab debe fallar antes de modificar archivos

si el error aparece después de una escritura PHP y el lint falla, debe restaurar el archivo anterior o eliminar el archivo nuevo inválido

si el contrato es ambiguo, no debe aceptarse como éxito parcial

si el formato no corresponde a ninguno de los contratos vigentes, no debe aplicarse

si el bloque corresponde a documentación, no debe procesarse como código

si el bloque corresponde a código, no debe procesarse como documentación

Regla final:

un ERROR explícito es preferible a una escritura ambigua.

<<SECTION: SALIDA OPERATIVA ESPERADA>>
SECTION_VERSION: 00011

SALIDA OPERATIVA ESPERADA

Toda ejecución de Project Lab debe producir salida clara.

Estados esperados:

OK
WARN
ERROR
INFO

OK

Indica operación aplicada correctamente.

Ejemplos:

archivo actualizado
método reemplazado
documento actualizado
auditoría generada
sintaxis PHP válida
backup generado
archivo inválido eliminado luego de rollback validado
archivo anterior restaurado luego de lint fallido

WARN

Indica que la operación se aplicó o se procesó, pero existe una advertencia relevante.

ERROR

Indica que la operación no debe considerarse aplicada correctamente.

INFO

Indica orientación complementaria o detalle técnico útil para diagnóstico.

Reglas:

no debe considerarse cierre técnico una salida sin OK verificable

no debe considerarse seguro un cambio aplicado con errores no resueltos

una salida OK de aplicación no reemplaza validaciones posteriores necesarias para el frente trabajado

si Project Lab informa OK de sintaxis PHP válida dentro de un patch PHP, no hace falta repetir php -l de forma redundante salvo auditoría externa o duda puntual

si Project Lab informa ERROR de sintaxis PHP y restauración o eliminación del archivo inválido, la operación falló pero el proyecto debe quedar protegido frente a ese cambio inválido

La consola debe permitir reconstruir:

qué se ejecutó
qué entrada se usó
qué salida produjo
qué archivo se modificó cuando corresponda
qué validación se realizó
qué error quedó pendiente si corresponde

Regla de claridad:

cuando una operación falle por lint PHP, la salida debe separar:

resultado operativo principal
detalle técnico del lint

Lectura esperada:

[ERROR] Sintaxis PHP inválida. Se restauró el archivo anterior.

o

[ERROR] Sintaxis PHP inválida. Se eliminó el archivo nuevo generado.

seguido de:

[INFO] Detalle lint:

Regla:

la salida de Project Lab es evidencia operativa de herramienta.

La interpretación funcional o arquitectónica de esa evidencia corresponde al proceso técnico del proyecto y a la documentación especializada.

USO DE TINKER DENTRO DE PROJECT LAB

Tinker es una capacidad interna de Project Lab para ejecutar código PHP sobre el proyecto actual y observar comportamiento runtime.

Debe usarse para:

reproducir lógica real del proyecto
inspeccionar modelos, servicios o helpers
validar queries
verificar persistencia
probar contratos backend
diagnosticar errores runtime
ejecutar pruebas controladas con salida explícita

Regla crítica:

en el editor Tinker de Project Lab debe pegarse solo código PHP interno.

No debe pegarse el comando completo:

php artisan tinker --execute

Project Lab ya ejecuta internamente el proceso correspondiente contra el proyecto actual.

Reglas:

si se quiere visualizar el resultado de una expresión, debe utilizarse return, echo, dump o salida explícita

si una salida no muestra el resultado esperado, primero debe revisarse si el bloque genera salida visible

si un bloque falla por imports, namespaces o wrapper, debe corregirse el formato del snippet antes de concluir que falló la lógica probada

si la prueba requiere contexto runtime, ese contexto debe inicializarse dentro del propio snippet

si la prueba modifica datos reales, debe planificarse rollback, limpieza o uso de datos descartables cuando corresponda

Tinker ayuda a reproducir runtime.

No reemplaza validación en navegador cuando el cierre dependa de interacción visual.

No reemplaza tests automatizados cuando el frente los requiera.

No reemplaza documentación especializada sobre seguridad, permisos, módulos o datos del sistema.

<<END SECTION>>

<<SECTION: REGLA DE TENANT ACTIVO EN TINKER>>
SECTION_VERSION: 00004

REGLA DE TENANT ACTIVO EN TINKER

Algunos proyectos requieren contexto runtime activo para que una prueba Tinker represente el comportamiento real del sistema.

Ese contexto puede incluir, según el proyecto:

usuario autenticado
tenant activo
organización activa
módulo habilitado
rol o membresía vigente
dataset real
configuración runtime
servicio central registrado en el contenedor

Regla operativa:

si la lógica que se desea probar depende de contexto runtime, ese contexto debe resolverse explícitamente dentro del snippet Tinker.

Macros audit:

Project Lab puede interpretar comandos rápidos internos cuando el textarea comienza con el prefijo:

>>

Lectura:

estos comandos no se ejecutan como texto literal.

Project Lab debe:

identificar el nombre de macro
validar el nombre de macro
buscar la plantilla correspondiente
validar argumentos
renderizar el script audit final
ejecutar el script generado mediante audit
devolver salida explícita

tools/project-lab/macros/audit

Formato vigente de plantilla:

nombre.sh.tpl

Ejemplo vigente:

find.sh.tpl

Sintaxis vigente de uso:

>> find archivo termino +lineas

Ejemplos:

>> find web.php service +30

>> find projectlab.php executeTinker +50

Reglas:

si la macro no existe, Project Lab debe devolver ERROR y listar macros disponibles
si los argumentos no cumplen la regla de la macro, Project Lab debe devolver ERROR y no ejecutar audit
si el comando es válido, Project Lab puede reemplazar el textarea con el script generado luego de recibir la respuesta
la salida debe dejar claro cuál fue la entrada original
la salida debe indicar la macro ejecutada
la salida debe ser interpretable como evidencia operativa

Regla específica para find:

find busca de forma case-insensitive.

El primer parámetro indica archivo, patrón o asterisco.

El segundo parámetro indica término de búsqueda.

El tercer parámetro opcional indica líneas posteriores mediante +N.

El término de búsqueda no puede estar vacío ni ser asterisco.

La cantidad de líneas debe permanecer dentro del límite aceptado por Project Lab.

Regla para cierres de fase:

La lista concreta depende del caso.

Project Lab ayuda a obtener evidencia.

No reemplaza el análisis técnico de esa evidencia.

USO DE CODE

La herramienta code debe usarse para aplicar cambios de código cuando el snippet cumpla un contrato válido.

Reglas:

usar TARGET con operador de reemplazo para reemplazar método PHP existente o función JS existente

no usar formatos ambiguos

no aplicar snippets generados sin revisar el encabezado

Regla crítica:

si el objetivo es modificar un método, el snippet debe usar TARGET.

Si el objetivo es reemplazar un archivo completo, el snippet debe usar FILE.

Si el encabezado mezcla ambas intenciones, Project Lab debe devolver ERROR o normalizar de forma explícita y segura cuando el contrato lo permita.

Nunca debe crear un archivo nuevo por interpretar mal una operación parcial.

Regla vigente para PHP completo:

el snippet debe comenzar como archivo PHP válido.

Formato correcto:

<?php
// FILE: app/Support/Ejemplo.php | V1

Reglas:

la apertura PHP debe aparecer antes del marcador FILE

el marcador FILE debe ubicarse después de <?php

el archivo debe validar como PHP cuando corresponda

Project Lab no debe aceptar como archivo PHP completo un bloque que comience con FILE y luego abra PHP si el contrato vigente espera PHP válido desde el inicio

Regla vigente para PHP:

cuando Project Lab aplica una actualización PHP compatible, debe ejecutar validación de sintaxis PHP.

Si la validación falla:

si el archivo existía, debe restaurar el contenido anterior

si el archivo era nuevo, debe eliminar el archivo inválido generado

La salida debe informar el rollback o la eliminación.

Regla vigente para operaciones parciales:

cuando se reemplaza o agrega un método PHP, debe usarse TARGET.

Ejemplos:

// TARGET: app/Support/Ejemplo.php :: metodoExistente

// TARGET: app/Support/Ejemplo.php ++ metodoNuevo

Regla:

no usar FILE para operaciones parciales.

no usar TARGET para archivo completo.

no mezclar ambos contratos en el mismo snippet.

Esta sección conserva únicamente lecciones operativas directamente vinculadas al uso de Project Lab.

Lección incorporada:

un snippet ambiguo puede generar daño si el parser interpreta como ruta lo que en realidad era una operación parcial.

Regla consolidada:

Project Lab debe validar antes de escribir.

Nunca debe crear un archivo con operadores parciales en la ruta.

Lección incorporada:

una ejecución desde clipboard no debe ser invisible para el usuario.

Regla consolidada:

cuando Project Lab ejecute desde clipboard, debe devolver y mostrar el contenido efectivamente leído.

Esto aplica especialmente a:

code
docs
audit
Tinker desde clipboard

La consola muestra la salida.

El textarea muestra la entrada.

Ninguna ejecución desde clipboard debe quedar oculta para el usuario.

Lección incorporada:

Project Lab no debe recibir texto narrativo como si fuera script ejecutable.

Regla:

las respuestas narrativas no son comandos.

comandos shell válidos
patches completos compatibles
formatos aceptados por la herramienta
bloques documentales válidos
snippets de auditoría realmente ejecutables
macros audit válidas

un cambio PHP aplicado por Project Lab no debe dejar persistido código inválido cuando la herramienta puede detectarlo.

Regla consolidada:

las vías legacy internas de Project Lab no deben conservarse si ya no pertenecen al contrato operativo vigente.

ESTADO VIGENTE

Project Lab es la herramienta operativa local principal para tareas técnicas controladas dentro de app-base.

Tinker queda incorporado como una capacidad interna de Project Lab.

La guía operativa principal ya no debe organizarse alrededor de Tinker como herramienta aislada.

Lectura vigente:

Project Lab = estación operativa local
Tinker = capacidad runtime dentro de Project Lab
code = herramienta de aplicación de código
docs = herramienta de actualización documental
audit = herramienta de auditoría simple
Artisan = ejecución controlada de comandos permitidos
consola = evidencia operativa guardable como auditoría

Regla:

las referencias futuras deben priorizar esta guía como documento operativo principal de herramienta local.

Regla de mantenimiento documental:

esta guía debe permanecer enfocada en el uso de Project Lab.

Cuando acumule doctrina general del proyecto, debe sanearse y remitir al documento especializado correspondiente.

Estado técnico validado reciente:

las acciones legacy fueron desconectadas

los métodos legacy fueron eliminados físicamente

las propiedades internas de estado fueron declaradas explícitamente

la creación de directorios internos fue centralizada

docs exige SECTION_VERSION

los backups documentales usan timestamp

las operaciones PHP gestionadas por code ejecutan lint

si el lint PHP falla, Project Lab restaura el archivo anterior o elimina el archivo nuevo inválido

la salida de rollback conserva detalle técnico bajo [INFO] Detalle lint

audit fue recuperado y validado luego de incorporar normalización de script

las lecturas críticas con projectRoot directo fueron saneadas o justificadas bajo resolución segura

no se detectaron métodos privados huérfanos luego del saneamiento

no se detectaron acciones legacy residuales en PHP/UI

Regla:

este estado no convierte a Project Lab en sandbox.

Project Lab sigue siendo herramienta local técnica bajo uso responsable del usuario técnico.
