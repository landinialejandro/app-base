<<SECTION: OBJETIVO>>

OBJETIVO

Este documento define el checklist obligatorio para crear, incorporar o reconvertir módulos dentro de app-base.

Su función es:

reducir omisiones
forzar consistencia arquitectónica
alinear seguridad, dominio y UI
evitar módulos incompletos
mantener coherencia entre implementaciones nuevas y existentes

No es una guía opcional.

Es un control mínimo obligatorio antes de considerar un módulo como integrado al sistema.

Aclaración doctrinal consolidada:

este checklist aplica tanto a módulos nuevos como a reconversiones profundas de módulos existentes cuando cambie su contrato funcional, contractual, de modelado o de seguridad

<<SECTION: REGLA GENERAL>>
SECTION_VERSION: 00007

REGLA GENERAL

Ningún módulo debe considerarse terminado por tener tablas, rutas o CRUD funcionando.

Un módulo solo puede considerarse integrado cuando supera este checklist en forma coherente.

Orden correcto:

definir contrato conceptual
definir contrato de seguridad
definir relaciones reales necesarias
definir superficie de integración
definir si participa en trazabilidad operativa transversal
definir si consume lectura contextual de actividad operativa, directa o agregada, cuando corresponda
implementar
auditar
ajustar documentación afectada

Nunca al revés.

Regla adicional consolidada:

si una integración nueva requiere reabrir arbitrariamente el núcleo de otro módulo, primero debe revisarse si la necesidad puede resolverse consumiendo una capacidad ya publicada por ese módulo o evolucionando su superficie contractual propia

Regla adicional para actividad operativa:

no todo módulo debe registrar actividad por defecto.

La participación en OperationalActivity debe justificarse por valor operativo real.

No debe registrarse actividad solo por existir CRUD, eventos técnicos o cambios persistidos sin significancia operativa.

La integración con OperationalActivity debe definirse como parte del contrato del módulo dueño.

Debe quedar claro:

qué operaciones emiten eventos
qué campos son observables
qué campos quedan excluidos por ruido técnico
qué cambios tienen valor operativo real
si existen relaciones hijas que deban expresarse como metadata operativa
si existe entidad agrupadora que deba actuar como ancla operativa
si el módulo consume actividad operativa contextual
si la lectura contextual será directa o agregada

Casos validados:

parties puede registrar cambios de party_roles como metadata operativa de Party.

inventory debe registrar InventoryOperation como ancla operativa agrupada, no InventoryMovement como actividad principal.

projects puede declarar un activity_record_set para lectura agregada de actividad contextual de Project + Tasks.

Regla adicional para lectura contextual de actividad operativa:

la lectura directa no requiere provider adicional.

Se limita al record host autorizado.

La lectura agregada requiere que el módulo dueño declare explícitamente su frontera mediante un provider compatible con ActivityRecordSetProvider.

Antes de declarar un activity_record_set debe quedar claro:

qué records forman parte de la actividad contextual del host
por qué esos records pertenecen funcionalmente al host
si esa agregación altera o no ownership
si el host ya autorizó el record principal
si la lectura agregada no equivale a lectura general del tenant
si no se está reconstruyendo historia desde campos actuales en lugar de leer OperationalActivity persistido

Regla:

OperationalActivity no debe modificarse con condiciones específicas por módulo.

No corresponde agregar lógica como:

if Project
if Order
if Document

La frontera debe vivir en el módulo dueño o en su contrato publicado.

Caso validado:

projects.show
ProjectActivityRecordSet
Project + Tasks

Regla adicional para capacidades tenant no modulares:

no toda capacidad tenant-scoped debe incorporarse a ModuleCatalog.

Si una capacidad:

no es módulo funcional
no tiene navegación propia
no tiene CRUD completo
no se habilita ni deshabilita como módulo ordinario

debe evaluarse como capacidad tenant no modular.

Caso validado:

operational_activity.view_any

Regla:

las capacidades tenant no modulares pueden ser evaluadas por Security sin convertirse en módulos funcionales ordinarios.

Regla adicional para surfaces tenant/transversales no modulares:

una capacidad tenant-scoped o transversal puede publicar una surface hacia hosts funcionales sin convertirse en módulo funcional.

Esta integración no representa un módulo nuevo si la capacidad:

no posee navegación principal
no posee CRUD ordinario
no se incorpora a ModuleCatalog
no pasa por TenantModuleAccess
no se habilita ni deshabilita como módulo ordinario

Caso validado:

OperationalActivitySurfaceService publica una tab contextual Actividad para hosts autorizados mediante TenantSurfaceRegistry.

Antes de crear una surface para una capacidad tenant/transversal se debe verificar:

si la capacidad es módulo funcional o no
si corresponde ModuleCatalog o TenantSurfaceRegistry
si necesita navegación propia o solo consumo embebido
si la autorización proviene de permiso propio o del host
si la surface representa lectura general o contextual
si el host ya autorizó el registro antes de consumir la surface

Regla:

no debe quedar como cierre estable un push manual de tabs en vistas host si la capacidad ya puede publicarse mediante registry reusable.

<<SECTION: BLOQUE 0 — IDENTIDAD DEL MÓDULO>>
SECTION_VERSION: 00004

BLOQUE 0 — IDENTIDAD DEL MÓDULO

Antes de cualquier decisión técnica debe quedar definida la identidad del módulo.

Validar:

nombre del módulo
problema real que resuelve
naturaleza principal del módulo
posición conceptual en el ecosistema
si es módulo nuevo o reconversión de uno existente
si es módulo funcional, entidad hija, capacidad transversal de soporte, capacidad tenant no modular o pieza interna del tenant

Clasificación conceptual posible:

módulo base del dominio
módulo transversal
módulo autónomo
módulo agrupador
módulo derivado
capacidad transversal de soporte
capacidad tenant no modular
capacidad tenant/transversal no modular con surface publicada
pieza interna del tenant

Preguntas obligatorias:

este módulo tiene identidad propia o es solo una entidad hija
es realmente un módulo o una evolución de otro ya existente
debe vivir por sí mismo o principalmente embebido
qué rol ocupa dentro del ecosistema general
registra, acompaña o sostiene actividad de otros módulos sin absorber su ownership
debe tener navegación principal o solo consumo interno/transversal
la necesidad requiere ModuleCatalog o puede resolverse como capacidad tenant no modular
si necesita publicar UI reusable, corresponde ModuleCatalog o TenantSurfaceRegistry
la surface representa un módulo funcional o una capacidad tenant/transversal no modular

Si estas respuestas no están claras, no debe avanzarse a implementación.

Caso vigente:

OperationalActivity se lee como capacidad transversal de soporte tenant-scoped para trazabilidad operativa.

No se lee como CRUD ordinario ni como módulo navegable principal.

Caso vigente adicional:

operational_activity.view_any se lee como capacidad tenant no modular para lectura general tenant-scoped del log operativo.

No se incorpora a ModuleCatalog.

Caso vigente adicional:

OperationalActivitySurfaceService se lee como surface tenant/transversal no modular publicada mediante TenantSurfaceRegistry.

No convierte OperationalActivity en módulo funcional ordinario.

<<END SECTION>>

<<SECTION: BLOQUE 1 — CONTRATO MODULAR>>
SECTION_VERSION: 00011

BLOQUE 1 — CONTRATO MODULAR

Antes de aprobar un módulo nuevo debe poder explicarse con claridad:

núcleo propio
contrato de entrada
contrato de salida
contrato de seguridad
oferta de integración
participación o no en trazabilidad operativa transversal
participación o no en lectura contextual de actividad operativa

Validar además:

si realmente es módulo y no dataset reusable del tenant
si resuelve un problema propio del dominio
si evita invadir núcleos existentes
si sus integraciones futuras pueden publicarse por contrato
si el ownership funcional queda claro
si debe emitir eventos operativos livianos después de acciones propias autorizadas
si debe ser consumido por una capacidad transversal sin trasladarle lógica del dominio
si necesita publicar una frontera explícita para lectura contextual agregada de actividad

Criterio adicional validado:

si una necesidad es solo consulta interna de colaboradores, directorio activo o pertenencia operativa, evaluar primero una raíz reusable del plano tenant antes de abrir módulo nuevo.

Criterio adicional validado:

si la necesidad aparece dentro de tablas anfitrionas de otros módulos, evaluar si corresponde publicar row actions oficiales desde el módulo oferente antes de crear lógica local en el consumidor.

Criterio adicional para reconversiones:

si una clasificación mezcla dimensiones conceptuales distintas, debe revisarse el contrato funcional del módulo antes de seguir agregando excepciones.

La salida correcta puede requerir separar:

núcleo de la entidad
relaciones operativas
entidades hijas o complementarias

Caso validado:

parties.kind no debe mezclar tipo de contacto con relación comercial o vínculo interno.

Criterio adicional para constraints funcionales:

si una reconversión cambia la dimensión real de autorización, la constraint funcional debe migrar hacia esa dimensión.

No debe conservarse la constraint anterior como compatibilidad operativa si ya no representa el dominio vigente.

Caso validado:

parties dejó de usar allowed_kinds como constraint funcional.

parties usa allowed_party_roles sobre party_roles.role.

Criterio adicional para OperationalActivity:

si un módulo participa en actividad operativa transversal, debe quedar definido:

qué eventos operativos livianos emite
en qué momento los emite
qué campos son observables
qué campos deben excluirse por ruido técnico
qué cambios tienen significancia operativa real
qué catálogo del módulo dueño declara esos campos
qué datos mínimos necesita el registro transversal
qué no debe delegarse al prestador transversal
si la operación debe registrar el modelo principal o una entidad agrupadora
si existen relaciones hijas relevantes que deban representarse como metadata operativa

Reglas:

el prestador transversal no debe conocer en exceso la semántica interna del módulo dueño

el módulo dueño no debe llamar directamente al logger desde controllers o services

el módulo dueño debe autorizar, validar y mutar antes de emitir el evento operativo

no se debe registrar actividad técnica sin valor operativo

no se debe duplicar trazabilidad propia de otro módulo

Casos validados:

parties registra cambios de party_roles como metadata operativa de Party.

inventory registra InventoryOperation como acción operativa agrupada.

InventoryMovement conserva trazabilidad fina interna de stock.

Criterio adicional para lectura contextual agregada de OperationalActivity:

si un módulo necesita que la tab Actividad muestre más que el record host directo, debe publicar un activity_record_set propio.

Ese activity_record_set debe ser definido por el módulo dueño.

No debe ser definido dentro de OperationalActivity.

Validar:

si la lectura será directa o agregada
qué records integran el conjunto
si esos records pertenecen funcionalmente al host
si el conjunto respeta tenant activo
si el conjunto no amplía acceso general al log
si el conjunto no altera ownership de los records incluidos
si el host principal ya fue autorizado antes de leer actividad
si corresponde declarar provider compatible con ActivityRecordSetProvider
si corresponde registrar el provider en ModuleCatalog

Caso validado:

Projects declara ProjectActivityRecordSet.

ProjectActivityRecordSet define Project + Tasks como dataset de actividad visible del proyecto.

Regla:

si no existe activity_record_set, la lectura contextual debe permanecer directa sobre el record host.

Criterio adicional para capacidades tenant no modulares:

si la capacidad pertenece al plano tenant o transversal, pero no constituye módulo funcional ordinario, debe definirse fuera de ModuleCatalog.

Validar:

si no tiene navegación propia
si no tiene CRUD completo
si no se habilita o deshabilita como módulo
si no debe generar create, update ni delete
si puede ser evaluada por Security como capability específica

Caso validado:

operational_activity.view_any

Criterio adicional para surfaces tenant/transversales no modulares:

si una capacidad tenant o transversal no modular necesita publicar UI reusable hacia hosts funcionales, debe definirse una vía de publicación distinta de ModuleCatalog.

Validar:

si la capacidad conserva naturaleza no modular
si la surface representa lectura general o lectura contextual
si el host consumidor ya autorizó el registro cuando la lectura sea contextual
si corresponde TenantSurfaceRegistry
si ModuleSurfaceRegistry puede fusionar esa surface con las modulares sin alterar ownership
si la key de la tab es compatible con return_tab
si se evita duplicación por pushes manuales en cada host

Caso validado:

OperationalActivitySurfaceService publica tab Actividad mediante TenantSurfaceRegistry.

Regla:

la surface tenant/transversal no convierte la capacidad en módulo funcional ordinario.

<<END SECTION>>

BLOQUE 2 — SEGURIDAD OBLIGATORIA

Ningún módulo operativo puede entrar al sistema sin contrato de seguridad explícito.

Validar:

módulo registrado
capacidades válidas
scopes válidos
constraints necesarias
dataset visible
reglas de create
reglas de update
reglas de delete
si delega autorización al padre
si requiere contexto funcional obligatorio

Validar además:

consumo correcto mediante Security
ausencia de lógica paralela en controller
ausencia de inferencias en Blade
coherencia entre view_any y view

Preguntas obligatorias:

cómo autoriza realmente este módulo
qué rompe si falta contexto
qué acciones dependen de constraints
qué dataset debe devolver Security::scope(...)

Regla adicional consolidada:

si el módulo utiliza constraints funcionales, validar explícitamente que arbitren no solo create sino también view, update y delete cuando corresponda al contrato del módulo

no deben tratarse como metadata decorativa ni como validación parcial

Validación adicional para flujos operativos con estado:

si una operación depende del estado del recurso padre, de una orden, de una línea o de una etapa del flujo, ese estado debe auditarse como parte real de la seguridad efectiva

no debe quedar como simple bloqueo visual opcional

Validación adicional para módulos transaccionales o con estados operativos:

Security, Gate/Policy, controller y UI deben devolver la misma lectura operativa

los estados readonly deben bloquear actualización y eliminación en la autorización central

el borrado debe considerar trazabilidad operativa real y no solo permiso nominal del rol

las acciones sobre recursos hijos deben respetar primero el estado del padre

las validaciones con Project Lab deben incluir matriz por estado, permiso y trazabilidad cuando corresponda

Validación adicional para administración owner:

si el módulo depende de activación, permisos o configuración tenant, debe existir una vía visible y real para que owner administre ese frente

auditar explícitamente:

si existen permisos del módulo en el catálogo efectivo
si esos permisos pueden asignarse realmente
si la habilitación del módulo produce consecuencia visible coherente en UI
si owner dispone de una surface real para entender y administrar ese comportamiento

Validación adicional para campos ocultos o relaciones sensibles:

si una opción se oculta en UI por permisos, estado o relación activa, debe existir validación backend equivalente.

No alcanza con no mostrar el campo.

Debe probarse intento manipulado desde request o runtime real.

Caso validado:

parties bloquea asignación manipulada de relación employee desde StorePartyRequest y UpdatePartyRequest para usuarios no autorizados.

Validación adicional para reconversión de constraints:

cuando un módulo migra una constraint funcional hacia una nueva dimensión, debe validarse el contrato completo y no solo la UI o el formulario.

Controlar:

resolver central
contrato persistido
seeder
UI de permisos
controller
requests
vistas
Security::scope(...)
create
view
update
delete
usuarios reales del seed
grep de residuos legacy
prueba de manipulación cuando exista relación sensible

Caso validado:

parties migró desde allowed_kinds hacia allowed_party_roles.

Regla:

PARTIES + allowed_kinds no debe quedar como combinación activa cuando la doctrina vigente exige allowed_party_roles.

Validación adicional para actividad operativa transversal:

si el módulo emite eventos hacia OperationalActivity, verificar que la autorización ocurra antes de la mutación y antes de emitir el evento.

OperationalActivity no debe ser usado para decidir permisos.

OperationalActivity no debe reemplazar Security, Gate, Policy, requests ni validaciones del módulo dueño.

La lectura de operational_activities requiere autorización explícita.

No debe asumirse acceso de lectura solo porque exista escritura append-only.

Caso vigente:

la lectura general tenant-scoped se autoriza mediante capability específica:

operational_activity.view_any

Regla:

esta capability no convierte OperationalActivity en módulo funcional ordinario ni habilita operaciones originales.

Validación adicional para lectura de logs:

definir autorización explícita antes de crear vista, tab, menú o surface.

Si se crea una lectura general, debe diferenciarse de lectura contextual por registro.

La lectura general del log del tenant no debe confundirse con la lectura de actividad asociada a un registro que el usuario ya puede ver.

Validación adicional para lectura contextual heredada:

si una surface muestra actividad operativa asociada a un registro host, debe validarse que el host ya autorizó view sobre ese registro.

La lectura contextual no debe usar operational_activity.view_any.

operational_activity.view_any queda reservado para lectura general tenant-scoped.

Reglas:

la surface contextual no abre acceso general al log del tenant
la surface contextual no autoriza el registro host
la surface contextual no reconstruye permisos del host
la surface contextual no debe mostrar actividad de otros registros
el reader contextual debe recibir o trabajar sobre el record ya autorizado

Caso validado:

OperationalActivityContextReader lee actividad asociada al record host autorizado.

BLOQUE 3 — RELACIONES E INTEGRACIONES

Todo módulo debe definir claramente sus relaciones e integraciones dentro del ecosistema.

Validaciones obligatorias:

no reabrir núcleos de otros módulos
consumir capacidades oficiales antes de crear variantes locales
publicar capacidades reutilizables cuando correspondan
respetar ownership del módulo oferente
definir si emite eventos operativos livianos hacia una capacidad transversal
definir si consume actividad operativa persistida como lectura y bajo qué autorización
definir si su lectura contextual de actividad es directa o agregada

Regla adicional consolidada:

si el módulo participa en navegación, dashboard o surfaces visuales, debe declarar metadata visual mínima en ModuleCatalog.

Como mínimo:

icon lógico del módulo

Reglas:

la metadata visual no debe duplicarse en SurfaceService
no debe hardcodearse en vistas consumidoras
debe derivarse desde el catálogo modular

Regla adicional:

las surfaces publicadas no deben duplicar metadata que pueda resolverse desde ModuleCatalog

la metadata debe viajar enriquecida a través de ModuleSurfaceRegistry

Regla adicional:

las integraciones surface-driven deben permitir que el host consuma metadata ya resuelta

incluyendo iconografía cuando corresponda

sin reconstrucción local en el host

Caso consolidado:

tabs principales del host pueden renderizar icon mediante metadata de tab enriquecida desde surfaces

Regla:

toda integración debe poder describirse contractualmente y mantenerse reusable sin acoples innecesarios

Regla adicional para entidades base con relaciones hijas:

cuando una entidad base requiere expresar múltiples relaciones con la empresa, esas relaciones deben documentarse y modelarse como dimensión separada del tipo base.

Caso vigente:

parties → party_roles

parties.kind expresa tipo de contacto.

party_roles.role expresa relación con la empresa.

Regla adicional para trazabilidad operativa:

la integración con OperationalActivity debe realizarse mediante eventos operativos livianos y no mediante acoples directos de controllers consumidores al log.

El módulo dueño emite.

La capacidad transversal registra si corresponde.

El futuro consumidor visual lee actividad persistida bajo autorización definida.

Regla:

no inferir historia operativa desde campos actuales si existe actividad persistida específica.

Validación adicional para relaciones hijas:

si una relación hija expresa un cambio operativo relevante, debe definirse si ese cambio se registra como metadata operativa del registro principal.

Caso validado:

party_roles puede registrarse como metadata operativa de Party.

Regla:

la relación hija no se convierte por eso en módulo propio ni en navegación autónoma.

Validación adicional para entidades agrupadoras:

si el módulo tiene una capa agrupadora y movimientos atómicos, debe elegirse el ancla correcta para OperationalActivity.

Caso validado:

inventory registra InventoryOperation.

InventoryMovement conserva trazabilidad atómica interna de stock.

Regla:

no duplicar en OperationalActivity la trazabilidad fina que ya pertenece al módulo dueño si existe una operación agrupada que representa mejor la acción del usuario.

Validación adicional para lectura contextual agregada de actividad operativa:

si un host necesita mostrar actividad de un conjunto de records y no solo del record host autorizado, el módulo dueño debe publicar explícitamente ese conjunto.

La publicación debe realizarse mediante activity_record_set y provider compatible con ActivityRecordSetProvider.

Validar:

que el conjunto sea finito y explícito
que la frontera esté definida por el módulo dueño
que la frontera no sea inferida por OperationalActivity
que no exista recursividad automática
que no se amplíe acceso general al tenant
que no se altere ownership funcional de los records incluidos
que el host principal haya sido autorizado antes de consumir la lectura
que el reader consuma records ya definidos y no reconstruya relaciones funcionales por su cuenta

Caso validado:

projects.show
ProjectActivityRecordSet
Project + Tasks

Regla:

OperationalActivity no debe contener condiciones específicas por módulo para decidir activity_record_set.

No corresponde incorporar lógica como:

if Project
if Order
if Document

Si un módulo requiere lectura agregada, debe declararla desde su propio contrato publicado.

Validación adicional para lectura visual de actividad operativa:

cuando se cree una lectura visual de OperationalActivity:

no se deben persistir URLs finales
los links deben resolverse desde capa de lectura
si no hay ruta canónica segura, mostrar texto plano
el enlace al registro debe integrarse en la columna Registro
no debe agregarse una columna adicional solo para link
debe preservarse trail contextual cuando exista navegación

Caso vigente:

OperationalActivityLinkResolver resuelve label, URL, fallback y trail.

Casos actuales sin link seguro:

InventoryOperation
Attachment

Validación adicional para surfaces tenant/transversales no modulares:

cuando una capacidad tenant/transversal no modular publique una surface hacia hosts funcionales:

debe hacerlo mediante registry específico del plano tenant/transversal
no debe incorporarse a ModuleCatalog si no es módulo funcional ordinario
no debe resolverse con pushes manuales permanentes en cada host
no debe crear navegación principal por el solo hecho de publicar UI
no debe pasar por TenantModuleAccess
debe respetar la gramática esperada por ModuleSurfaceRegistry y x-host-tabs cuando corresponda

Caso validado:

OperationalActivitySurfaceService se publica mediante TenantSurfaceRegistry.

Regla:

ModuleSurfaceRegistry puede fusionar surfaces modulares y tenant/transversales, pero cada origen debe conservar su ownership y frontera conceptual.

BLOQUE 4 — UI Y EXPERIENCIA OPERATIVA

Todo módulo debe cumplir criterios mínimos de experiencia operativa consistente.

Validaciones obligatorias:

usar componentes base del sistema
no duplicar patrones visuales existentes
mantener coherencia con show pages host
no generar UI paralela para capacidades ya resueltas

Regla adicional consolidada:

los estilos específicos de módulo o pantalla deben ubicarse en:

public/css/modules/

no deben reemplazar app-base.css ni app-patterns.css

Regla adicional:

los íconos deben existir como componentes Blade reutilizables

ubicados en:

resources/views/components/icons

Reglas:

los nombres deben ser genéricos
no deben nombrarse por módulo
deben permitir reutilización transversal

Regla adicional:

las vistas deben consumir metadata visual ya resuelta

no deben decidir localmente iconografía ni atributos visuales de módulos

Regla adicional:

las tabs principales deben poder consumir metadata visual enriquecida

incluyendo iconografía desde surfaces o catálogo modular

Regla:

la experiencia operativa debe mantenerse homogénea en todo el sistema

Validación adicional para formularios de módulos base:

si una entidad base concentra decisiones iniciales relevantes, puede usarse un formulario progresivo para guiar la carga sin exponer complejidad técnica.

El formulario progresivo debe:

reducir error de carga
mantener copy comprensible
no reemplazar autorización backend
no trasladar reglas de negocio al frontend

Caso vigente:

parties guía la carga mediante:

tipo de contacto
relación con la empresa
datos del contacto

Regla de lenguaje:

evitar términos que generen confusión con seguridad o permisos.

Caso vigente:

en parties debe preferirse Relación con la empresa antes que rol.

no usar componentes no existentes aunque parezcan patrón documentado.
