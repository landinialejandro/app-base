BASE VISUAL DEL PROYECTO – app-base
-----------------------------------

OBJETIVO

Definir una base visual simple, consistente y escalable para la aplicación SaaS app-base.
El objetivo es contar con una interfaz limpia, profesional y mantenible, sin introducir
complejidad innecesaria en las primeras etapas del desarrollo.

La base visual debe permitir:
- consistencia entre módulos
- facilidad para agregar nuevas pantallas
- separación clara entre lógica y presentación
- evolución futura sin reescrituras grandes.


PRINCIPIOS DE DISEÑO

1. Simplicidad primero
La interfaz debe ser simple y funcional. Se prioriza claridad sobre estética avanzada.

2. Consistencia
Todos los módulos deben usar la misma estructura visual.

3. Componentización mínima
Se utilizan componentes Blade reutilizables cuando aportan claridad o evitan repetición.

4. Evolución gradual
La base visual debe poder evolucionar hacia un diseño más completo sin romper lo existente.


ARQUITECTURA VISUAL ACTUAL

La aplicación utiliza un layout principal basado en Blade.

layouts/app.blade.php

Este layout define:

- barra superior de la aplicación
- contenedor principal
- área de contenido
- carga del CSS global
- navegación básica del sistema


ESTRUCTURA GENERAL DE PÁGINA

Las páginas internas siguen esta estructura:

<x-page>
    <x-breadcrumb />
    <x-page-header />
    <x-card />
</x-page>

Componentes utilizados:

x-page
Contenedor principal de una pantalla.

x-breadcrumb
Navegación jerárquica dentro del sistema.

x-page-header
Encabezado de la página con título y acciones.

x-card
Contenedor visual para contenido o secciones.


COMPONENTES BLADE DISPONIBLES

Actualmente existen los siguientes componentes visuales reutilizables:

x-page
Define el layout base de una pantalla.

x-page-header
Contiene el título de la pantalla y acciones (botones).

x-card
Caja visual que contiene contenido.

x-breadcrumb
Ruta de navegación de la página.


BARRA SUPERIOR DE LA APLICACIÓN

El layout principal incluye una barra superior con:

- marca de la aplicación (app-base)
- empresa actual
- selector de empresa (cuando el usuario tiene más de una)
- usuario autenticado
- acceso para cerrar sesión

La empresa actual se muestra en el header global, evitando repetirla en cada vista.


PÁGINAS PÚBLICAS

Algunas páginas son consideradas públicas y no muestran el layout completo.

Ejemplos:

- welcome
- login
- register
- selección de empresa

Estas páginas utilizan la variable:

$publicPage = true

Esto permite ocultar elementos de navegación interna.


DASHBOARD

Se implementó un dashboard inicial con:

- empresa actual
- cantidad de proyectos
- accesos rápidos a módulos principales

El dashboard actúa como punto de entrada al sistema.


LISTADOS

Los listados utilizan el siguiente patrón visual:

- x-card
- table-wrap
- table

Las tablas deben incluir encabezado y filas de datos.

Los estados vacíos deben mostrarse dentro de la card.


FORMULARIOS

Los formularios siguen estas reglas:

- campos agrupados con clases consistentes
- botones en un bloque form-actions
- formularios reutilizan archivos parciales (_form)

Ejemplo:

projects/_form
parties/_form


MÓDULOS ADAPTADOS A LA BASE VISUAL

Actualmente los siguientes módulos ya utilizan la base visual:

- projects
- parties (contactos)

Esto define el patrón que deben seguir los nuevos módulos.


NOMENCLATURA DE INTERFAZ

En la interfaz se utilizan los siguientes términos:

Tenant → Empresa
Party → Contacto

La terminología técnica puede mantenerse en el código, pero la interfaz debe usar
los términos orientados al usuario.


EVOLUCIÓN FUTURA

La base visual permite incorporar en el futuro:

- sidebar de navegación
- mejoras en el dashboard
- sistema de iconos
- temas visuales
- dark mode
- mejoras de UX


CONCLUSIÓN

La base visual actual define una estructura clara y consistente para el sistema.

Los nuevos módulos deben seguir el mismo patrón implementado en projects y parties
para mantener coherencia visual y facilitar el mantenimiento del sistema.
