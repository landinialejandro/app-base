RECONVERSIÓN LIMPIA DE CONSTRAINTS

Cuando cambia la dimensión real de autorización, la constraint funcional debe migrar a esa dimensión y eliminar la convivencia legacy.

Caso validado:

parties:
antes → allowed_kinds sobre parties.kind
ahora → allowed_party_roles sobre party_roles.role

Regla:

la constraint anterior no debe quedar como fallback operativo si ya no representa la semántica vigente del módulo.

AUTONOMÍA MODULAR CON VÍNCULOS OPCIONALES

Patrón validado:

un módulo puede conservar autonomía funcional aunque posea vínculos opcionales hacia otros módulos.

Lectura:

el vínculo opcional enriquece, contextualiza o formaliza

no define por sí solo existencia funcional

no concede autorización

no debe convertirse automáticamente en required

no debe forzar dependencia visual

Casos vigentes:

Orders puede existir sin party_id

Documents puede existir o derivarse sin party_id cuando conserva contraparte textual propia

Regla:

un campo `*_id` debe interpretarse según contrato funcional real, no por reflejo técnico.

SNAPSHOT TEXTUAL PROPIO + VÍNCULO GESTIONADO OPCIONAL

Patrón validado:

cuando un módulo puede vincularse opcionalmente con Parties pero necesita conservar dato propio de contraparte, debe separar:

party_id = vínculo gestionado opcional

counterparty_name = snapshot textual propio del registro

Casos vigentes:

Orders:

party_id
counterparty_name

Documents:

party_id
counterparty_name

Reglas:

el snapshot no reemplaza Party

el snapshot no reemplaza Security

el snapshot no crea permiso

el snapshot permite lectura autónoma del registro

la salida visual debe usar método de presentación propio cuando exista

Caso vigente:

displayCounterpartyName()

Regla visual:

no mostrar solo party->name si el módulo posee snapshot propio y fallback controlado.

RELACIÓN EXTERNA HACIA ORDERS

Patrón validado:

cuando otro módulo origina, planifica o acompaña una Order, la relación persistente debe vivir en el módulo que conserva ownership de ese origen, planificación o seguimiento.

Casos vigentes:

Tasks usa tasks.order_id

Appointments usa appointments.order_id

Orders no conserva task_id ni appointment_id como parte de su núcleo persistente.

Reglas:

task_id puede viajar como contexto transitorio de request o navegación

appointment_id puede viajar como contexto transitorio de request o navegación

el contexto transitorio no equivale a permiso concedido

el contexto transitorio no debe convertirse automáticamente en campo persistente del módulo receptor

Tasks conserva ownership de pendientes y seguimiento

Appointments conserva ownership de agenda y planificación temporal

Orders conserva ownership de la intervención operativa

Una Order puede tener múltiples Tasks asociadas.

DERIVACIÓN ENTRE MÓDULOS SIN TRANSFERENCIA DE OWNERSHIP

Patrón validado:

un módulo puede derivar otro registro aportando contexto sin transferir ownership.

Caso vigente:

Order → Document

Lectura:

Orders aporta contexto operativo

Documents conserva ownership documental

documents.order_id conserva vínculo de derivación cuando corresponde

Reglas:

la derivación exige autorización del módulo receptor

Documents conserva sus propios ítems

Documents conserva sus propios totales

Documents conserva su propio lifecycle

Documents conserva su propia contraparte textual cuando corresponda

La creación derivada debe validar runtime web real, no solo seeders o lint.

Caso detectado:

una dependencia a método inexistente puede aparecer en runtime real aunque migrate:fresh --seed sea exitoso.

Corrección vigente:

Order → Document recalcula totales desde document_items sin depender de DocumentTotalsCalculator::apply().

AUTORIZACIÓN SENSIBLE CENTRALIZADA

Cuando una regla sensible aparece duplicada en controller y requests, debe extraerse a una pieza reusable del dominio correspondiente.

Caso validado:

PartyEmployeeContactAuthorization

Ubicación vigente:

app/Support/Parties/PartyEmployeeContactAuthorization.php

Caso validado adicional:

TenantProfileAccess

Ubicación vigente:

app/Support/Tenants/TenantProfileAccess.php

Regla:

la UI puede expresar disponibilidad, pero la autorización sensible debe resolverse y bloquearse en backend.

Regla adicional:

cuando un frente tenant concentra reglas owner/admin, esa autoridad debe centralizarse en una pieza reusable del plano tenant y no repetirse en policies, controllers o Blades.

VALIDACIÓN RUNTIME DE SEGURIDAD

Una validación sensible debe combinar:

UI correcta
Security runtime
request validation
intento manipulado
verificación final de persistencia

Caso validado:

protección de party vinculada a membership activa.

Caso validado adicional:

contrato owner/admin en tenant/profile/accesses.

Validaciones realizadas:

admin intentando editar General bloqueado

admin intentando modificar permisos del rol admin bloqueado

admin modificando permisos de rol operativo permitido

admin intentando asignar admin bloqueado

admin asignando rol operativo permitido

admin intentando autogestionarse o vaciar sus roles bloqueado

admin intentando combinar admin con rol operativo vía sync bloqueado

membership activa sin roles con error funcional controlado

rollback confirmado luego de pruebas positivas

La validación no se considera cerrada solo por ocultar campos en UI.

Debe probarse que el backend bloquea intentos manipulados y que el registro persiste intacto.

VALIDACIÓN RUNTIME DE FLUJOS DERIVADOS

Un flujo derivado no se considera cerrado solo porque cree el registro destino.

Debe validarse:

autorización del módulo receptor

datos propios del registro destino

líneas copiadas cuando corresponda

cálculos o totales reales

lifecycle inicial correcto

salida visual posterior

ausencia de dependencia a métodos inexistentes

prueba runtime web real

Caso validado:

Order → Document funciona desde runtime web real después de permitir Documents con counterparty_name, copiar ítems y recalcular totales desde document_items.

EVENTOS OPERATIVOS TRANSVERSALES

Patrón validado:

↓

↓

↓

↓

↓

Reglas:

el evento operativo no autoriza
el evento operativo no valida la operación original
el evento operativo no reemplaza Security ni policies
el módulo dueño conserva ownership funcional
la capacidad transversal registra solo si corresponde
el listener queued no debe bloquear el flujo principal

CAMPOS OBSERVABLES POR CATÁLOGO DEL MÓDULO DUEÑO

Cuando una capacidad transversal registra actividad operativa, los campos observables deben pertenecer al catálogo del módulo dueño.

Casos vigentes:

AppointmentCatalog::activityTrackedFields()
TaskCatalog::activityTrackedFields()
OrderCatalog::activityTrackedFields()
DocumentCatalog::activityTrackedFields()

Regla:

el prestador transversal no debe conocer en exceso la semántica interna de cada módulo.

Cuando se integre un nuevo módulo, el catálogo dueño debe declarar explícitamente los campos observables antes de considerar cerrada la trazabilidad operativa.

SANIDAD DE CAMBIOS OPERATIVOS

La trazabilidad operativa debe aplicar sanidad antes de persistir.

Reglas:

no registrar cambios vacíos
ignorar campos técnicos
no registrar por registrar
registrar solo cambios operativamente significativos
clasificar actividad desde el prestador transversal cuando corresponda

Clasificaciones vigentes posibles:

assigned
reassigned
unassigned
status_changed
updated

Regla:

la clasificación no debe dispersarse en controllers.

METADATA OPERATIVA OPCIONAL

Los eventos operativos livianos pueden admitir metadata opcional cuando un cambio operativo relevante no vive directamente como atributo simple del modelo principal.

Reglas:

la metadata debe complementar el cambio operativo
no debe trasladar autorización al evento
no debe convertir entidades hijas en módulos propios
no debe acoplar en exceso OperationalActivity a la semántica interna del módulo dueño

Caso validado:

parties registra cambios de party_roles como metadata operativa de Party.

RELACIONES HIJAS CON SIGNIFICANCIA OPERATIVA

Cuando una relación hija expresa un cambio operativo relevante, puede representarse como metadata operativa del record principal si esa lectura respeta ownership y evita crear un módulo artificial.

Caso validado:

party_roles expresa relación del contacto con la empresa.

Sus cambios pueden registrarse dentro de la actividad operativa de Party.

Regla:

party_roles no se convierte en módulo propio ni en navegación propia.

ENTIDADES AGRUPADORAS Y MOVIMIENTOS ATÓMICOS

Cuando un módulo posee una entidad agrupadora de acción y entidades atómicas de trazabilidad fina, la actividad transversal debe anclarse en la pieza que represente mejor la acción operativa del usuario.

Caso validado:

inventory registra InventoryOperation.

InventoryMovement conserva trazabilidad fina interna de stock.

Regla:

no duplicar en OperationalActivity la trazabilidad atómica que ya pertenece al módulo dueño cuando una operación agrupada expresa mejor el evento operativo.
