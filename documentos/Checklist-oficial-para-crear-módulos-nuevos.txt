Objetivo

Este checklist define el procedimiento estándar para crear un nuevo módulo dentro del proyecto app-base.

Su finalidad es:

mantener consistencia entre módulos
evitar errores estructurales
garantizar compatibilidad multi-tenant
acelerar desarrollo

Este checklist aplica a todos los módulos CRUD del sistema.

1. Evaluación inicial (ANTES de crear el módulo)

Antes de crear una entidad nueva verificar:

si puede resolverse con Party (Contactos)
si puede resolverse con Documents
si es una verdadera entidad nueva

Si no es imprescindible ahora:

registrarlo en TODO-DEL-PROYECTO.

2. Crear migración

Crear la migración.

Ejemplo:

php artisan make:migration create_products_table

Reglas de migración

Toda entidad del negocio debe tener:

tenant_id
timestamps
softDeletes

Ejemplo mínimo:

$table->uuid('tenant_id')->index();
$table->string('name');
$table->timestamps();
$table->softDeletes();

Índices recomendados

Siempre indexar:

tenant_id

y campos de búsqueda relevantes.

Ejemplo:

$table->index(['tenant_id','name']);

3. Crear modelo

Crear modelo:

php artisan make:model Product

El modelo debe incluir:

use HasFactory;
use TenantScoped;
use SoftDeletes;
use ResolvesTenantRouteBinding;

Ejemplo:

use Illuminate\Database\Eloquent\Factories\HasFactory;
use App\Models\Concerns\TenantScoped;
use App\Models\Concerns\ResolvesTenantRouteBinding;
use Illuminate\Database\Eloquent\SoftDeletes;

Motivación:

HasFactory permite generar datos de prueba mediante factories
TenantScoped garantiza aislamiento automático multi-tenant
SoftDeletes permite eliminación lógica
ResolvesTenantRouteBinding asegura resolución segura de rutas

fillable

Definir los campos editables.

Ejemplo:

protected $fillable = [
'tenant_id',
'name',
'description',
];

4. Crear controller

Crear controller tipo resource:

php artisan make:controller ProductController

Métodos mínimos

El controller debe incluir:

index
create
store
show
edit
update
destroy

Regla de tenant

Siempre obtener el tenant:

$tenant = app('tenant');

y pasarlo a la vista cuando sea necesario.

5. Definir rutas

Agregar en:

routes/web.php

Ejemplo:

Route::resource('products', ProductController::class);

Las rutas deben quedar dentro del grupo:

auth
tenant

6. Crear vistas

Crear carpeta:

resources/views/products

Vistas mínimas

Crear:

index.blade.php
create.blade.php
edit.blade.php
show.blade.php
_form.blade.php

7. Convención de listados (index)

Los listados deben seguir el patrón usado en projects.

Reglas:

no usar columna de acciones
el nombre enlaza al detalle
crear botón "Nuevo"

Ejemplo:

Nombre → link a show

8. Convención de show

La vista de detalle debe incluir:

Editar
Eliminar
Volver al listado
Detalle de campos

El botón eliminar debe usar:

POST + method DELETE

con confirmación.

9. Convención de formularios

Usar archivo compartido:

_form.blade.php

utilizado por:

create
edit

10. Validación

Validar en el controller inicialmente.

Ejemplo:

$request->validate([
'name' => ['required','string','max:255'],
]);

Si el módulo crece, migrar a FormRequest.

11. Mensajes de éxito

Usar mensajes simples.

Ejemplo:

Entidad creada correctamente
Entidad actualizada correctamente
Entidad eliminada correctamente

Ejemplo:

->with('success', 'Producto creado correctamente');

12. Prueba mínima del módulo

Verificar:

✔ listado
✔ creación
✔ edición
✔ eliminación
✔ aislamiento por tenant

13. Verificar aislamiento multi-tenant

Confirmar que:

un tenant no puede ver datos de otro tenant.

Esto debe resolverse automáticamente por:

TenantScoped

pero siempre debe verificarse.

14. Datos de prueba

Cuando un módulo nuevo es relevante para el funcionamiento del sistema:

crear factory correspondiente

php artisan make:factory ProductFactory

y agregar datos al DemoSeeder para facilitar pruebas.

Esto permite reconstruir el entorno de desarrollo mediante:

php artisan migrate:fresh --seed

15. Actualizar documentos del proyecto

Si el módulo es nuevo, actualizar:

MAPA-DE-ENTIDADES
TODO-DEL-PROYECTO

16. Regla de simplicidad

Antes de agregar:

roles
estados complejos
subtablas
polimorfismos
arquitecturas complejas

preguntarse:

¿esto es necesario ahora?

Si no lo es:

registrarlo en TODO y seguir con el sistema funcional.

17. Regla de coherencia

Cada módulo nuevo debe parecer parte del mismo sistema.

Debe respetar:

estructura
rutas
vistas
convenciones
flujo de trabajo

18. Flujo resumido para crear módulo

Flujo recomendado:

evaluar entidad
crear migración
crear modelo
crear controller
crear rutas
crear vistas
validar funcionamiento
actualizar documentos

Regla final

Si al implementar un módulo aparece una decisión que afectará a todo el sistema:

no improvisar.

Registrar la decisión en:

CONVENCIONES-DEL-PROYECTO

y luego continuar el desarrollo.
