<<SECTION: ENTIDADES DEL SISTEMA>>
ENTIDADES DEL SISTEMA

Son las entidades que permiten que la plataforma funcione como SaaS multi-tenant.

Incluyen:

users
tenants
memberships
signup_requests
invitations
<<END SECTION>>

<<SECTION: FUNCIÓN DE CADA UNA>>
FUNCIÓN DE CADA UNA

users

Representa la identidad global del usuario en todo el sistema.

Un user puede pertenecer a múltiples tenants.

No pertenece a un tenant específico.

tenants

Representa cada empresa o espacio de trabajo dentro del SaaS.

Cada tenant contiene sus propios datos operativos.

Además de datos generales, puede contener configuración funcional en:

tenants.settings

Esto permite persistir contexto operativo del tenant cuando todavía no justifica entidad propia.

memberships

Representa la pertenencia de un user a un tenant.

Define acceso real por empresa.

Es la entidad correcta para resolver:

a qué empresa pertenece un usuario
qué estado tiene dentro de esa empresa
qué roles tiene dentro de esa empresa
qué perfil funcional persistido tiene dentro de esa empresa

signup_requests

Representa solicitudes iniciales de alta o interés desde la landing pública.

No pertenece al dominio operativo interno del tenant.

invitations

Representa invitaciones dentro del flujo de acceso y onboarding del sistema.

Puede operar en dos modos:

owner_signup
member_invite

Cuando type = owner_signup puede existir sin tenant asociado.

Cuando type = member_invite sí se vincula operativamente a un tenant existente.

Por esta razón, no debe tratarse como una entidad tenant-scoped pura del negocio.
<<END SECTION>>

<<SECTION: ENTIDADES DEL NEGOCIO>>
ENTIDADES DEL NEGOCIO

Son las entidades operativas propias de cada tenant.

Estas entidades deben estar aisladas por:

tenant_id

Actualmente incluyen:

parties
projects
tasks
appointments
products
assets
orders
order_items
documents
document_items
document_sequences
attachments
<<END SECTION>>

<<SECTION: ENTIDAD BASE DEL NEGOCIO>>
ENTIDAD BASE DEL NEGOCIO

parties

Es la entidad base para personas u organizaciones dentro del sistema.

Debe absorber conceptualmente actores como:

clientes
proveedores
contactos
contratistas
terceros

Regla actual:

no crear entidades nuevas si el caso puede resolverse extendiendo parties

parties no debe tratarse como un módulo opcional común, sino como entidad base transversal del dominio y módulo base intrínseco del sistema

Nombre técnico:

Party

Nombre visible sugerido en la interfaz:

Contacto

Regla adicional:

las relaciones hacia parties desde otros módulos deben respetar el estado activo del módulo destino y no deben exponerse en UI si el módulo parties no estuviera disponible
<<END SECTION>>

<<SECTION: MODELO DE ACCESO Y PERTENENCIA>>
MODELO DE ACCESO Y PERTENENCIA

Relación principal:

users
↓
memberships
↓
tenants

Regla:

users resuelve identidad global
memberships resuelve acceso real por tenant

No usar user.tenant_id porque rompería el modelo multi-tenant actual.
<<END SECTION>>

<<SECTION: ROLES Y AUTORIZACIÓN>>
SECTION_VERSION: 00002

ROLES Y AUTORIZACIÓN

El modelo de autorización se encuentra consolidado en base a permisos V2 persistidos.

Componentes activos:

role_permission
membership_permission_overrides

Reglas:

los permisos efectivos se resuelven mediante resolvers y Security

las memberships no owner deben poseer al menos un rol base

el owner puede no tener roles explícitos en la relación many-to-many

el rol owner se resuelve implícitamente mediante lógica del resolver

los scopes y constraints forman parte activa del contrato persistido

<<END SECTION>>

<<SECTION: ACLARACIÓN SOBRE TENANT-SCOPED>>
ACLARACIÓN SOBRE TENANT-SCOPED

El trait TenantScoped debe usarse en entidades cuyo ciclo de vida pertenece completamente a un tenant.

Ejemplos de uso correcto:

Product
Order
Document
Branch
Party
Project
Task
Appointment
Asset
Attachment

Aclaración sobre Invitation

Invitation no debe tomarse como ejemplo simple de entidad tenant-scoped pura, porque su comportamiento depende del tipo de invitación.

Cuando type = owner_signup puede operar sin tenant asociado.

Cuando type = member_invite sí se vincula operativamente a un tenant existente.

No debe usarse TenantScoped en entidades del sistema como:

User
Tenant
Membership
SignupRequest
<<END SECTION>>

MÓDULOS ACTUALES DEL NEGOCIO

Los módulos actuales del sistema corresponden directamente a las entidades principales del dominio.

Cada uno de estos módulos posee:

representación operativa en UI
correspondencia directa con entidad del negocio
presencia en documentación funcional de usuario

Listado actual:

parties
projects
tasks
appointments
products
assets
orders
documents
attachments

Relación consolidada:

entidad del negocio
↓
módulo del sistema
↓
sección de documentación de usuario

Esta correspondencia 1 a 1 forma parte del modelo mental estable del sistema y debe mantenerse en futuras extensiones.

<<SECTION: PARTIES>>
PARTIES

Entidad base transversal del dominio y módulo base del sistema.

Relaciones actuales y futuras:

un Party puede vincularse a muchos Orders
un Party puede vincularse a muchos Documents
un Party puede vincularse a muchos Assets
un Party puede representar cliente, proveedor, contacto u otro actor

Reglas:

no duplicar esta función en entidades paralelas como Client o Supplier

no debe tratarse como módulo opcional apagable por tenant en el mismo sentido que otros módulos operativos
<<END SECTION>>

<<SECTION: PROJECTS>>
PROJECTS

Agrupador operativo de trabajo.

Relaciones:

un Project tiene muchas Tasks

Estado actual:

usa estado mínimo active/closed

Función actual:

seguimiento agregado del trabajo

Regla operativa actual:

owner y admin ven todos los proyectos

roles operativos ven solo proyectos donde tengan al menos una tarea asignada
<<END SECTION>>

<<SECTION: TASKS>>
TASKS

Unidad operativa interna fina.

Relaciones:

una Task pertenece a un Project
una Task puede asignarse a un User
una Task puede derivar opcionalmente en un Order

Reglas actuales:

assigned_user_id define responsabilidad operativa actual

toda tarea debe crearse asignada

owner y admin pueden gestionar tareas ajenas

los usuarios operativos:

pueden ver tareas asignadas a ellos

pueden ver tareas de proyectos donde participan

solo editan tareas asignadas a ellos

solo eliminan tareas asignadas a ellos

<<END SECTION>>

<<SECTION: APPOINTMENTS>>
APPOINTMENTS

Agenda operativa reutilizable por tenant.

Relaciones actuales:

pertenece a un tenant
requiere assigned_user_id
puede vincularse opcionalmente a un Party
puede vincularse opcionalmente a un Asset
puede vincularse opcionalmente a un Order

Regla funcional actual:

su capa visible puede adaptar vocabulario o etiquetas según el rubro configurado del tenant, sin permitir etiquetas libres por empresa

Regla operativa actual:

appointments utiliza assigned_user_id como referencia de responsabilidad operativa

puede resolver lectura con tenant_all o own_assigned según la capacidad configurada

no utiliza semántica limited como contrato del módulo

Regla de relaciones:

los vínculos hacia parties, assets y orders deben depender del estado activo del módulo destino

si el módulo destino está deshabilitado para el tenant, la relación no debe exponerse ni operarse en UI
<<END SECTION>>

<<SECTION: PRODUCTS>>
PRODUCTS

Catálogo operativo de productos o ítems comercializables.

Relaciones:

puede aparecer en OrderItems
puede aparecer en DocumentItems
<<END SECTION>>

<<SECTION: ASSETS>>
ASSETS

Representa activos o equipos asociados a un tenant.

Relaciones actuales:

un Asset puede vincularse opcionalmente a un Party
un Asset puede vincularse opcionalmente a muchos Orders
un Asset puede vincularse opcionalmente a muchos Documents

Relación futura posible:

historial de asociación Asset ↔ Party
<<END SECTION>>

<<SECTION: ORDERS>>
SECTION_VERSION: 00002

ORDERS

Cabecera operativa de órdenes.

Relaciones:

una Order pertenece a un Party
una Order puede vincularse opcionalmente a un Asset
una Order tiene muchos OrderItems
una Order puede derivar de una Task
una Order puede originar muchos Documents
una Order puede vincularse opcionalmente desde un Appointment

Relación Task → Order actual:

una Task puede derivar en una única Order

el vínculo operativo se expresa desde orders.task_id

Función conceptual consolidada:

orders representa la operación interna del tenant

debe absorber tanto la dimensión comercial operativa como la dimensión de servicio u operación técnica

la orden de servicio real del sistema pertenece a orders y no a documents

Regla de autorización consolidada:

orders debe tratarse como módulo compartido del tenant

no depende de assigned_user_id

no utiliza ownership como base de permisos

created_by es un dato estructural sin impacto en autorización

no utiliza own_assigned como contrato del módulo

su segmentación funcional se resuelve exclusivamente mediante constraints.allowed_kinds

cualquier segmentación futura por usuario deberá definirse explícitamente y no inferirse del modelo actual

Regla de relaciones:

las relaciones hacia otros módulos deben considerarse opcionales a nivel UI según estado activo del módulo destino

si un módulo relacionado está deshabilitado, la relación no debe exponerse ni operar en UI

<<END SECTION>>

<<SECTION: ORDER_ITEMS>>
ORDER_ITEMS

Detalle de una Order.

Reglas:

pertenece a una Order
hereda tenant_id desde la cabecera
no debe tratarse como recurso autónomo de autorización
su create, update y delete deben delegarse a la capacidad update de Order
<<END SECTION>>

<<SECTION: DOCUMENTS>>
DOCUMENTS

Cabecera documental.

Relaciones:

un Document pertenece a un Party
un Document puede vincularse opcionalmente a un Asset
un Document puede vincularse opcionalmente a un Order
un Document tiene muchos DocumentItems

Tipos documentales activos previstos al menos:

quote
delivery_note
invoice

work_order permanece como kind legacy inhabilitado.

Regla conceptual consolidada:

documents no debe representar la operación interna diaria del tenant

documents debe concentrar la dimensión documental, formal, comercial o fiscal derivada de la operación

documents debe tratarse como módulo compartido del tenant

su segmentación funcional no debe resolverse por ownership artificial del registro

cuando se requiera segmentación por tipo, debe apoyarse en constraints.allowed_kinds

Regla actual importante:

si un Document nace desde una Order, hereda automáticamente:

tenant_id
party_id
order_id
asset_id si corresponde
<<END SECTION>>

<<SECTION: DOCUMENT_ITEMS>>
DOCUMENT_ITEMS

Detalle de un Document.

Reglas:

pertenece a un Document
hereda tenant_id desde la cabecera
puede originarse manualmente o por copia desde OrderItems
no debe tratarse como recurso autónomo de autorización
su create, update y delete deben delegarse a la capacidad update de Document
<<END SECTION>>

<<SECTION: DOCUMENT_SEQUENCES>>
DOCUMENT_SEQUENCES

Entidad de soporte para numeración operativa.

Función:

gestionar numeración secuencial de Orders y Documents

Dimensiones actuales de secuencia:

tenant
tipo documental
punto de venta

Regla:

no modelar numeración manual o unique global en cabeceras operativas cuando ya existe este sistema.
<<END SECTION>>

<<SECTION: ATTACHMENTS>>
SECTION_VERSION: 00002

ATTACHMENTS

Tipo:

módulo transversal

Estado:

implementado y consolidado

Reglas:

no posee autonomía de autorización

su acceso depende del recurso padre

la autorización se delega en policies y controllers del padre

este patrón se encuentra aplicado en implementación real

<<END SECTION>>

<<SECTION: CONTEXTO FUNCIONAL DEL TENANT>>
CONTEXTO FUNCIONAL DEL TENANT

El tenant puede incorporar configuración funcional en:

settings.business_profile.type

Ese valor no representa texto libre de interfaz.

Debe resolverse a partir de un catálogo cerrado de rubros y servir como contexto para módulos adaptables, por ejemplo appointments.
<<END SECTION>>

<<SECTION: MEMBERSHIPS, OWNER Y ROLES>>
MEMBERSHIPS, OWNER Y ROLES

La pertenencia operativa de un usuario a un tenant se resuelve por memberships.

Reglas actuales:

memberships.is_owner expresa la condición estructural de owner

owner no debe modelarse como un rol asignable ordinario dentro del flujo normal de edición

owner es único por tenant

la pivote membership_role se utiliza para roles delegables del tenant

admin es el rol delegable de gestión amplia

administrator es el rol delegable administrativo u oficina

operator es el rol operativo base por defecto

sales permanece como rol operativo específico

una membership activa no-owner no debe quedar sin rol

puede existir una membership owner sin registros en membership_role y eso no constituye un error del modelo

como evolución activa de la autorización, membership podrá incorporar además un profile_slug para refinar capacidades funcionales sin multiplicar roles base

La jerarquía efectiva debe entenderse así:

owner como condición estructural
roles delegables como capa operativa base
perfil funcional como especialización de trabajo
permisos concretos como capa de autorización

La complejidad funcional futura no debe resolverse creando nuevos roles base fuera de este conjunto cerrado.
<<END SECTION>>

<<SECTION: REGLA GENERAL DE CRECIMIENTO>>
REGLA GENERAL DE CRECIMIENTO

Antes de crear una nueva entidad, verificar:

si el caso puede resolverse con una entidad ya existente
si parties puede absorber el nuevo caso
si la relación puede modelarse como extensión y no como nueva tabla
si la necesidad corresponde realmente al dominio o solo a una vista específica
