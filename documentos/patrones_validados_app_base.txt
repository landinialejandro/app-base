ADUANA FUNCIONAL ENTRE MÓDULOS

Patrón validado:

un módulo consumidor solicita una capacidad
↓
una frontera explícita procesa la necesidad
↓
el módulo oferente conserva ownership de su lógica
↓
el consumidor recibe payload, decisión, bloqueo, link, selector o surface

Formas válidas de frontera:

hooks
selectors
services
linked helpers
surface services
catálogos
registries
contratos explícitos

Reglas:

el consumidor no reconstruye lógica interna del oferente

el oferente no interpreta modelos internos del consumidor

el controller consumidor no debe acumular conocimiento de dominios externos si existe frontera apropiada

una relación de modelo no autoriza por sí misma a gobernar un módulo externo

Caso vigente:

Orders no debe interpretar Task, Appointment, Party, Asset o Document más allá de referencias pasivas, relaciones necesarias o contratos explícitos.

AISLAMIENTO OPERATIVO RAZONABLE

Patrón validado:

un módulo saneado no requiere aislamiento absoluto artificial.

Requiere:

núcleo propio claro
relaciones pasivas justificadas
dependencias activas saneadas
contratos explícitos para integración
controllers sin lógica interna de dominios externos
helpers genéricos del módulo oferente
hooks o services para bloqueos externos
selectors para selección o validación de dominios externos
surface services por capacidad real

Caso vigente:

Orders queda como isla funcional saneada apta para iniciar Fase 1 de mantenimiento.

Reglas:

las relaciones pasivas pueden conservarse si corresponden a columnas propias, derivaciones aceptadas o patrones transversales

las dependencias activas deben pasar por frontera explícita

no debe confundirse relación pasiva con ownership funcional

no debe confundirse aislamiento razonable con desconexión artificial del ecosistema

HOOKS COMO FRONTERA FUNCIONAL

Patrón validado:

el módulo dueño del flujo consulta un hook en puntos estratégicos.

El módulo externo puede:

bloquear
ajustar
dejar pasar
informar condición externa
aportar validación contextual

Reglas:

el módulo dueño del flujo no conoce la lógica interna del módulo externo

el módulo externo conserva ownership de su regla

si una verificación de bloqueo crítico falla, debe asumirse el escenario seguro

el hook no reemplaza autorización principal del módulo dueño

Caso vigente:

OrdersHooks funciona como frontera estratégica del flujo de Orders.

InventoryOrderItemHooks conserva la lógica material de bloqueo de cierre por ejecución física de líneas.

LINKED HELPER GENÉRICO

Patrón validado:

el módulo oferente arma links para su propio modelo o null.

No interpreta modelos consumidores.

Caso vigente:

OrderLinked::forOrder(?Order $order, ...)

Reglas:

si Tasks necesita mostrar una Order, Tasks entrega Order|null

si Appointments necesita mostrar una Order, Appointments entrega Order|null

Orders arma el link de su propio recurso

no deben existir métodos linked específicos por consumidor

No deben reaparecer:

OrderLinked::forTask()
OrderLinked::forAppointment()
OrderLinked::forDocument()
OrderLinked::forAsset()
OrderLinked::forParty()

SURFACE AGRUPADA POR CAPACIDAD REAL

Patrón validado:

una misma capacidad con igual formato y distinto host, filtro o configuración contextual debe representarse como una sola offer con múltiples targets.

Caso vigente:

OrderSurfaceService ofrece orders.related.embedded como surface embebida única para assets.show y parties.show.

Configuración contextual:

relatedHosts()

Puede concentrar:

clase esperada
columna de filtro
tabsId
flags visuales
mensaje vacío
parámetros de montaje

Reglas:

no duplicar offers por host si la capacidad es la misma

la surface debe expresar la capacidad real

el host aloja

el oferente prepara payload o vista

SURFACE SERVICE DECLARATIVO CON LÓGICA TRANSVERSAL EXTRAÍDA

Patrón validado:

un SurfaceService debe declarar ofertas, hosts y configuración propia.

La lógica transversal reusable no debe quedar incrustada dentro de un único SurfaceService si puede servir como mecánica común para otros módulos.

Piezas vigentes:

app/Support/Modules/Concerns/BuildsHostPacks.php
app/Support/Modules/Concerns/ResolvesRelatedEmbeddedSurfaces.php

Lectura:

BuildsHostPacks concentra la construcción genérica de hostPack.

ResolvesRelatedEmbeddedSurfaces concentra la resolución reusable de relacionados embebidos.

El SurfaceService conserva:

ofertas propias
hosts soportados
configuración contextual
relaciones propias
flags visuales propios
mensajes propios

Caso validado:

OrderSurfaceService conserva su configuración propia de hosts y relaciones, pero ya no contiene toda la mecánica genérica de validación, consulta y armado de payload.

Hosts validados:

orders.show
assets.show
parties.show

Surface validada:

orders.related.embedded

Payload validado:

orders
tabsId
trailQuery
emptyMessage
showCounterparty
showAsset

Reglas:

no vaciar un SurfaceService por reflejo

no mover lógica propia del módulo a un concern transversal

extraer solo mecánica reusable

aplicar el patrón progresivamente y con validación runtime

no migrar services de forma masiva sin evidencia

SELECTOR O PROVEEDOR DE DOMINIO

Patrón validado:

cuando varios módulos necesitan seleccionar o validar entidades de un mismo dominio, debe existir una capacidad central del módulo dueño.

Caso vigente:

ProductLineItemSelector

Responsabilidad:

opciones de productos
reglas de validación de productos
verificación de acceso a productos para líneas o publicaciones cuando corresponda

Consumidores vigentes:

OrderItemController

DocumentItemController

self_service_sales como publicación de tienda cuando corresponda al contrato vigente

Reglas:

Products ofrece selección y validación de productos

Orders administra sus propios order_items

Documents administra sus propios document_items

self_service_sales administra sus propias publicaciones de tienda

los controllers consumidores no deben reconstruir Product::query(), products.viewAny ni Rule::exists('products') cuando existe selector oficial

PUBLICACIÓN INTERNA DE TIENDA SIN OPERACIÓN COMERCIAL

Patrón validado:

una capa satélite comercial puede publicar un catálogo visible sin habilitar todavía operación transaccional.

Caso vigente:

self_service_sales publica catálogo inicial mediante:

SelfServiceShop
SelfServiceShopItem
SelfServiceShopCatalogReader
SelfServiceShopPublisher

Lectura:

SelfServiceShop configura la tienda.

SelfServiceShopItem publica productos.

Product conserva catálogo maestro.

Reglas:

publicar producto no equivale a vender

publicar producto no mueve stock

publicar producto no crea documento

publicar producto no habilita carrito

activar tienda no habilita operación comercial completa

catálogo visible no equivale a operación comercial habilitada

SEPARACIÓN DE ADMINISTRACIÓN INTERNA Y PLANO CUSTOMER

Patrón validado:

una capacidad satélite con entrada pública puede tener administración interna, pero ambos planos deben permanecer separados.

Caso vigente:

self_service_sales

Plano interno:

self_service_sales.shops.*

SelfServiceShop

SelfServiceShopItem

TenantProfileAccess::canManageSelfServiceSales()

Plano externo:

/tienda

/tienda/seleccionar

/shop/{tenant:slug}

/tienda/salir

SelfServiceExternalSession

ResolveSelfServiceExternalCustomer

Reglas:

customer externo no administra tienda

sesión externa mínima no autoriza administración interna

middleware externo pasivo no autoriza administración interna

vista pública no debe exponer rutas internas

vista interna no debe depender de sesión externa

NUMERACIÓN TRANSVERSAL CON SIGNIFICADO EN CATÁLOGOS

Patrón validado:

un servicio transversal puede ejecutar mecánica común de numeración, pero no decidir significado de negocio propio de cada módulo.

Regla síntesis:

el módulo define el significado.

el servicio transversal ejecuta la mecánica.

Caso vigente:

RecordNumberGenerator ejecuta numeración.

OrderCatalog define prefijos Orders.

DocumentCatalog define prefijos Documents.
