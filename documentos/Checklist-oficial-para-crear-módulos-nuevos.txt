CHECKLIST OFICIAL PARA CREAR MÓDULOS NUEVOS


OBJETIVO

Este checklist define los mínimos obligatorios al crear un módulo nuevo en el proyecto.

Debe utilizarse como control de consistencia antes de considerar el módulo integrado al core.

No reemplaza la revisión técnica.
Funciona como piso común obligatorio.


ESTRUCTURA GENERAL

Verificar:

modelo creado
migración creada
controller creado
rutas declaradas explícitamente
vistas base creadas
enlace en navegación si corresponde
validaciones mínimas implementadas
tenant resuelto correctamente
autorización revisada
documentación actualizada


MODELO Y MIGRACIÓN

Verificar:

la tabla usa naming consistente
la entidad incluye tenant_id si corresponde al negocio
la entidad no incluye tenant_id si es del sistema
soft delete aplicado cuando corresponda
fillable revisado
casts revisados
relaciones definidas
trait TenantScoped aplicado si corresponde


RUTAS Y CONTROLADOR

Verificar:

rutas CRUD declaradas explícitamente
no usar Route::resource(...) mientras siga vigente la convención actual del proyecto
nombres de ruta consistentes
controller delgado
validación separada cuando haga falta
authorize(...) o authorizeResource(...) aplicado cuando corresponda
no duplicar permisos en controller si la policy ya resuelve la acción


VISTAS

Verificar:

index consistente con la base visual
show consistente con la base visual
create y edit consistentes entre sí
uso de x-page
uso de x-breadcrumb
uso de x-page-header
uso de cards o bloques visuales consistentes
acciones visibles alineadas con permisos reales
partials creados si la vista crece demasiado


AUTORIZACIÓN Y UI

Verificar:

policy creada si el módulo lo necesita
policy realmente aplicada desde controller
@can aplicado en botones, acciones y accesos sensibles
la UI no muestra acciones no permitidas
navbar o dashboard no exponen accesos irreales
permisos alineados con rol, módulo y tipo de acción


DOCUMENTACIÓN OBLIGATORIA

Verificar si corresponde actualizar:

CONTEXTO-FIJO-DEL-PROYECTO
MAPA-DE-ENTIDADES
MAPA-DE-NAVEGACION-Y-RELACIONES
CONVENCIONES-DEL-PROYECTO
BASE-VISUAL-DEL-PROYECTO
TODO-DEL-PROYECTO
DEPLOY-Y-COMPATIBILIDAD-DEL-PROYECTO

No actualizar documentos por inercia.

Solo tocar los realmente afectados.


NAVEGACIÓN Y BREADCRUMB

Definir desde el inicio si el módulo necesita solo navegación estructural o también navegación contextual.

Preguntas de control:

la entidad puede abrirse desde otra entidad relacionada como continuación natural de trabajo

el usuario necesita volver al origen sin perder el hilo operativo

ese origen es semántico y estable, no solo la página anterior

Si alguna de estas respuestas es sí, el módulo debe evaluar e implementar navegación contextual explícita.

Si corresponde, el módulo debe:

tener builder o resolutor de trail propio compatible con el mecanismo central vigente del proyecto

aceptar contexto de navegación de forma explícita

resolver ese contexto desde la capa reutilizable central vigente

propagar ese contexto en tablas, tabs, acciones por fila, links relacionales, paginación, filtros, botones volver, botones cancelar y redirects relevantes

resolver create -> store -> show sin persistencia de nodos transitorios de “Nuevo …”

resolver show -> edit -> update -> show sin persistencia de nodos transitorios de “Editar …”

resolver correctamente submódulos tipo ítems, de modo que item create o item edit retornen al show del padre sin contaminar el trail final

si participa en navegación cruzada real, aceptar y propagar el trail o mecanismo contextual vigente del proyecto

si ya existe un resolutor, builder o adaptador central del módulo, reutilizarlo en lugar de crear una variante paralela

verificar que el builder o resolutor del módulo guarde URLs limpias en los nodos

verificar que el trail no quede serializado dentro del propio campo url de los nodos

verificar que, al volver a una entidad ya presente, el trail colapse sin duplicarse

verificar que el módulo no conserve mecanismos anteriores de contexto redundantes una vez migrado al esquema central vigente

verificar links relacionales en tablas y sublistados:
si el contexto real es la fila, usar el trail de la fila y no solo el del contenedor

verificar index contextual:
si desde un index se navega a una entidad relacionada, decidir explícitamente si el breadcrumb debe conservar solo módulo origen o módulo origen más entidad fila

verificar navbar:
no debe saltar automáticamente solo por el módulo profundo actual si el flujo funcional sigue anclado en otro módulo de origen

Reglas:

usar breadcrumb estructural por defecto

usar breadcrumb contextual solo cuando el origen siga siendo útil

no reconstruir breadcrumb funcional desde url()->previous()

no deducir breadcrumb funcional desde relaciones implícitas del modelo ni desde historial del navegador

no introducir mecanismos paralelos de contexto si ya existe trail central vigente

no duplicar en blades lógica que ya fue resuelta por controller o capa central de navegación

CIERRE

Un módulo no debe considerarse terminado solo porque funciona.

Debe quedar:

consistente con el sistema
alineado con permisos reales
alineado con multi-tenancy
alineado con la base visual
alineado con la documentación vigente
