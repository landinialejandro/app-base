OBJETIVO DEL DOCUMENTO

Este documento define las convenciones estructurales y de desarrollo del proyecto app-base.

Su objetivo es mantener consistencia técnica, previsibilidad arquitectónica y coherencia de código a medida que el sistema crece.

Las convenciones aquí definidas deben respetarse al crear:

nuevos módulos
nuevas entidades
nuevas vistas
nuevos controladores
nuevos flujos operativos

<<END SECTION>>

[USO]

No es rector arquitectónico general.
No es documento de seguridad.
No es mapa visual.
No es navegación.
No es memoria histórica.
No es backlog.

[REGLA 1]
Si una regla no orienta implementación futura, sale.

[REGLA 2]
Si una regla pertenece a arquitectura general, va a contexto_fijo.

[REGLA 3]

[REGLA 4]
Si una regla pertenece a navegación, UI, entidades, componentes, Project Lab, mantenimiento, shopping, IA, errores o decisiones vivas, va al documento especializado.

[REGLA 5]
Convención válida = criterio repetible de implementación.

[REGLA 6]
Caso vigente solo queda si evita una deriva técnica concreta.

[REGLA 7]
NO SE ASUME.
NO SE SUPONE.
Si falta estructura, ruta, clase, tabla, método, contrato, capability, scope, schema o runtime: pedir evidencia exacta.

[REGLA 8]
Project Lab valida evidencia real.
No reemplaza criterio arquitectónico.
No reemplaza documentación rectora.
No aplica snippets ambiguos.

[REGLA 9]

implementación consistente
bajo acoplamiento

continuidad IA
prevención de legacy
contratos explícitos

[PRINCIPIOS]
Simplicidad.
Claridad estructural.
Baja duplicación.
Coherencia entre módulos.
Previsibilidad del comportamiento.

[REGLA BASE]
Ante varias alternativas, elegir la más simple compatible con arquitectura vigente.

[NO ASUMIR]
No asumir:
estructura
rutas
APIs
tablas
contratos
helpers
componentes
permisos
capabilities
scopes
constraints
métodos
runtime

[ANTES DE INTERVENIR]
Leer documentación vigente.
Trabajar sobre código real.
Pedir evidencia mínima necesaria.
Validar contra estructura real.
No completar por memoria.
No completar por conveniencia.

[NO LEGACY]
No cerrar migraciones, refactors o reconversiones con convivencia indefinida entre patrón nuevo y patrón viejo.

[OWNERSHIP]
Cada módulo conserva núcleo propio.
Consumer no invade oferente.
Host visual no absorbe ownership.
Universo semántico no absorbe ownership.
Capa satélite no absorbe ownership.
Entrada pública no absorbe plano interno.

[INTEGRACIÓN]
Consumidor solicita.
Frontera explícita procesa.
Módulo oferente resuelve.
Host monta.
UI representa.

[Fronteras válidas]
hooks
selectors
providers
services
linked helpers
surface services
catálogos
registries
contratos explícitos

[CONTROLLERS]
Controller = flujo propio del módulo.
No intérprete de dominios externos.
No reconstruye permisos.
No reconstruye bloqueos externos.
No decide significado de catálogos ajenos.
No concentra cierre/cancelación/derivación si existe hook o service de frontera.

[SEGURIDAD]
Validación crítica vive en backend, Security, Policy, service o contrato oficial.
Blade no autoriza.
Frontend no autoriza.
JS no autoriza.
UI solo expresa resultado autorizado.

[OWNER / ADMIN]
Reglas tenant/profile centralizadas en:
app/Support/Tenants/TenantProfileAccess.php

Vistas pueden expresar:
disponibilidad
readonly
botones visibles
estados bloqueados

Backend decide autorización final.

[DOCUMENTACIÓN]
Documentos activos deben usar:
METADATOS
TÍTULO
DOC_SLUG
DOC_VERSION
SECTION_VERSION cuando corresponda

Referencias internas:
<DOC_REF slug="...">Título visible</DOC_REF>

No usar nombres físicos de archivo como referencia narrativa si existe DOC_SLUG.

baks y auditoria no son base documental activa navegable.

[CONTENIDO USUARIO FINAL]
Lenguaje no técnico.
Frases claras.
Cierre conceptual.
Puede usar refuerzo tipo “👉 …”.
Aplica a onboarding, ayudas contextuales y contenidos UX.

[PROJECT LAB]
Project Lab = herramienta local de evidencia, auditoría, código y documentación.
Usar cuando el caso toque:
permisos
constraints
multi-tenancy
runtime real
snippets
auditorías
actualización de código
actualización documental

Referencia:
<DOC_REF slug="guia_project_lab_proyecto_app_base">Guía operativa de Project Lab</DOC_REF>

[CONTRATO MODULAR]
Módulo = mini app interna cerrable del tenant.

Debe poder describirse por:
núcleo propio
contrato de entrada
contrato de salida
contrato de seguridad
oferta de integración

[CONVERGENCIA]
Si dos módulos/hosts tienen distinta madurez contractual, no degradar el más avanzado.
La convergencia debe tender al patrón maduro validado.

[HELPERS LINKED]
Helper linked pertenece al módulo oferente.
Recibe modelo propio o null.
No métodos específicos por consumidor.

Patrón vigente:
OrderLinked::forOrder(?Order $order, ...)

No reintroducir:
OrderLinked::forTask()
OrderLinked::forAppointment()
OrderLinked::forDocument()
OrderLinked::forAsset()
OrderLinked::forParty()

[SURFACES]
SurfaceService declara:
ofertas
hosts
configuración propia
payload preparado
vistas consumidas

Capacidad real > duplicación por host.

Si solo cambia host/filtro/flags/mensaje/configuración:
una offer con múltiples targets.

Mecánica transversal reusable vive en Support/Modules.
Lógica propia queda en SurfaceService.

Piezas vigentes:
app/Support/Modules/Concerns/BuildsHostPacks.php
app/Support/Modules/Concerns/ResolvesRelatedEmbeddedSurfaces.php

No vaciar SurfaceService por reflejo.

[HOOKS]
Hook = frontera funcional.
Puede bloquear, ajustar, dejar pasar o informar condición externa.
Módulo dueño no conoce lógica interna externa.
Módulo externo conserva ownership de su regla.
Falla en bloqueo crítico = escenario seguro.

Casos:
OrdersHooks
InventoryOrderItemHooks

[SELECTORS / PROVIDERS]
Si varios módulos seleccionan o validan entidades de un dominio, crear capacidad central del módulo dueño.

Caso:
ProductLineItemSelector

Regla:
Products ofrece selección/validación.
Orders/Documents administran sus ítems.
Consumidores no reconstruyen Product::query(), products.viewAny ni Rule::exists('products') si existe selector oficial.

[SERVICIOS TRANSVERSALES]
Servicio transversal ejecuta mecánica común.
Módulo/catálogo define significado.

Caso:
RecordNumberGenerator ejecuta numeración.
OrderCatalog define prefijos Orders.
DocumentCatalog define prefijos Documents.

No usar service nombrado por módulo si la mecánica ya es transversal.

[UNIVERSOS SEMÁNTICOS]
Universo semántico = frente funcional de uso, navegación y contexto.
No módulo dueño.
No CRUD por sí mismo.
No autorización.
No ownership.

Caso:
/service = Servicio y mantenimiento.
Orders conserva ownership.
Rutas pueden expresar universo y recurso operado.

[MODELOS]
Relación pasiva ≠ ownership.
Relación de consulta ≠ autorización.
Relación inversa innecesaria debe eliminarse si induce ownership incorrecto.

Caso:
Order no expone tasks().
Dirección vigente: Task -> Order mediante tasks.order_id.

[AUTONOMÍA MODULAR]
Distinguir:
dato propio necesario
vínculo gestionado opcional
contexto transitorio
origen
planificación
seguimiento
formalización derivada
enriquecimiento
universo de entrada

*_id no implica dependencia obligatoria.
Universo semántico no implica ownership.

[SNAPSHOTS]
party_id = vínculo gestionado opcional.
counterparty_name = snapshot textual propio.

Casos:
orders.counterparty_name
documents.counterparty_name

Snapshot:
no reemplaza autorización
no reemplaza Party
no vuelve obligatorio el vínculo
permite lectura autónoma

Usar helpers:
Order::displayCounterpartyName()
Document::displayCounterpartyName()

No mostrar solo party->name si existe snapshot propio.

[REFERENCIAS PROPIAS]
*_id = vínculo gestionado opcional.
*_reference / snapshot = referencia propia.
record_metadata = contexto, excepción o manualidad.

Blade representa.
Blade no interpreta JSON.
Salida visual usa helper, presenter o payload preparado.

Caso Orders:
party_id
asset_id
asset_reference
record_metadata

Nombre vigente para contraparte:
counterparty_name

counterparty_reference no es contrato vigente sin auditoría contra código real.

[CONTRAPARTE]
Contacto gestionado = party_id.
Contraparte = dato visible propio del registro.

Casos:
Orders
Documents

UI no debe presentar party_id como obligatorio si el contrato permite counterparty_name.

[FLAGS VISUALES]
showCounterparty = mostrar Contraparte.
showAsset = mostrar Activo.
showParty no es contrato visual vigente de Orders.

Flags publicados deben tener representación real en partial final.
No flags fantasma.

[FORMULARIOS CON FRONTERAS]
Formulario host no renderiza foreach abierto de cualquier oferta externa.

Patrón:
proveedor externo válido → componente proveedor
sin proveedor externo → campo manual propio

Caso:
orders.form

Ofertas:
party.order.form-context
asset.order.form-context

Carga manual no se transforma automáticamente en gestionada.

[RECORD_METADATA]
Metadata contextual o excepcional.
No dato central.
No dato operativo principal.
No vínculo gestionado.
No fuente principal visual.

Si flujo gestionado normal:
puede ser null.

Si frontera cae a carga manual:
puede registrar excepción/manualidad.

Blade no interpreta record_metadata.

[RELACIONES EXTERNAS HACIA ORDERS]
Orders conserva autonomía.
Seguimiento/planificación/origen viven en módulo que los origina.

Casos:
tasks.order_id
appointments.order_id

Orders no conserva:
task_id
appointment_id

task_id / appointment_id pueden viajar como contexto transitorio.
No se convierten automáticamente en persistencia del receptor.

[ORDERLINKED]
OrderLinked no bloquea creación contextual por ausencia de party_id.
Task/Appointment sin party_id pueden originar Order contextual si Orders permite counterparty_name.

OrderLinked:
expresa intención contextual
no reemplaza autorización
no exige Party si Orders permite contraparte textual

[DERIVACIÓN ENTRE MÓDULOS]
Derivar no transfiere ownership.

Caso:
Order → Document

Orders aporta contexto.
Documents conserva ownership.
Crear derivado requiere autorización del módulo receptor.
Documento conserva datos, líneas, totales, lifecycle y snapshot propio.

[MIGRACIONES]
Respetar orden real de creación de tablas.
No foreign key hacia tabla futura en migración previa.

Caso:
asset_id no debe ir en create_orders_table si assets se crea después.
Resolver en migración posterior.

[SEEDERS]
Seeder que crea registros debe respetar contrato vigente del módulo.
No depender de controller para completar obligatorios.
No aceptar payload legacy silencioso.
No crear datos inválidos.
Poblar snapshots si no pasa por controller.

[SUBCAPAS INTERNAS]
Capacidad contextual dentro de módulo host.
Sin CRUD.
Sin navegación.
Sin ModuleCatalog.
Sin dashboard general.
Sin rutas propias.

Caso:
ProductOperationalSummary bajo Products.

Debe publicarse por surface si existe infraestructura surface-driven.
No include directo como cierre estable.

[MODELADO]
No comprimir dimensiones distintas en un único campo.

Separar:
naturaleza
tipo
relación operativa
vínculo comercial
lifecycle
estado operativo
ubicación
custodia
prioridad
origen
cierre

Caso:
parties.kind = person/company.
party_roles.role = customer/supplier/employee/other.

[CONSTRAINTS]
Constraint opera sobre dimensión real del dominio.
No sostener compatibilidad legacy si la dimensión cambió.

Casos:
orders → allowed_kinds.
parties → allowed_party_roles.
allowed_kinds no aplica a parties.

[OPERATIONAL ACTIVITY]
Emitir eventos operativos livianos después de autorizar, validar y ejecutar.
No llamar logger directo desde controllers/services.
No registrar por tener CRUD.
No registrar ruido técnico.
No registrar deletes/vistas/downloads/previews/prints/PDF salvo decisión explícita.

Campos observables en catálogo del módulo dueño.
Metadata compuesta permitida si agrega valor operativo.

[METRÍCAS CONTEXTUALES]
Subcapa puede leer otros módulos si respeta ownership, permisos, módulos habilitados y contratos.

Caso:
ProductOperationalSummary:
Products presenta.
Orders aporta compras/ventas.
Inventory aporta movimientos.
Parties aporta proveedor/cliente si existe.
OperationalActivity no es fuente principal.

No cálculos en Blade.

[AUTOMATIZACIÓN]
Debe respetar:
tenant activo
módulos habilitados
permisos efectivos
lifecycle
ownership
validaciones backend
trazabilidad

No oculta decisiones críticas.
Reduce carga manual.
Deja rastro auditable cuando impacta operación real.

[IA]
IA = capa transversal asistiva.
No módulo.
No canal paralelo.
No scraping visual.
No permisos inferidos.
No datos inventados.
No cierre crítico sin confirmación o regla explícita.

Separar intención IA de proveedor IA.
No asumir API paga ni proveedor único.

[CAPAS SATÉLITE]
Capa satélite orquesta módulos.
No CRUD ordinario por reflejo.
No absorbe ownership.
No reabre módulos cerrados por reflejo.

Casos:
Servicios y Mantenimiento = /service sobre Orders group=service.
Shopping Autoservicio = /shop/{tenant:slug} plano externo self_service_sales.

Detalle funcional de capas satélite pertenece a documentos especializados.

[CSS / COMPONENTES / JS]
No usar componentes Blade inexistentes.
No asumir utilities CSS no validadas.
Preferir HTML simple para estructura segura.
JS reusable se nombra por mecánica, no por caso puntual.

Caso:
app-linked-select-sync

JS no decide permisos ni consistencia crítica.

[REGLA FINAL]
Convención válida = regla repetible de implementación que evita acoplamiento, legacy, ambigüedad o duplicación.

[APLICA A]

[SERVICIO Y MANTENIMIENTO]
Tipo = capa satélite productiva ERP/CMMS liviana.
Universo = /service.
Visible = Servicio y mantenimiento.
Owner real órdenes = Orders.
Base operativa = Orders.group = service.

Rutas vigentes:

Reglas:

No crear ownership mantenimiento sobre Orders.
No reabrir Orders por reflejo si la necesidad pertenece a Assets, Inventory, Products, Appointments, Tasks, Documents, Attachments, Parties, OperationalActivity o pieza satélite propia.

[CONTEXTO FÍSICO /SERVICE]
Ruta física contextual puede diferir del host funcional.

Caso:
orders.show = host funcional.
service.orders.show = ruta física contextual.

Reglas:
no crear service.orders.items.* si operación real sigue en Orders.
no duplicar controllers por contexto si el ownership real no cambia.
no depender de query params manipulables.
links, redirects y trails pueden conservar /service.
retorno contextual debe resolverse por trail, helper o contrato explícito.
las rutas internas de operación pueden seguir perteneciendo al módulo dueño cuando el ownership real no cambia.

Pieza vigente:
OrderNavigationTrail::showRouteName()

Lectura:
OrderNavigationTrail::showRouteName() resuelve dinámicamente si el retorno físico corresponde a orders.show o service.orders.show según ruta actual o trail recibido.

Regla:
el host surface interno puede seguir usando orders.show como identificador funcional aunque la ruta física contextual sea service.orders.show.
