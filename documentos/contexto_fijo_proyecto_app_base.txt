<<SECTION: TIPO DE SISTEMA>>
SECTION_VERSION: 00001

TIPO DE SISTEMA

app-base es una plataforma SaaS multi-tenant desarrollada sobre Laravel.

Permite que múltiples empresas utilicen una misma aplicación manteniendo aislamiento lógico de datos y contexto operativo independiente para cada tenant.

El proyecto busca constituir una base ERP modular reutilizable, capaz de crecer por módulos sin perder coherencia estructural.

El sistema no debe entenderse como una suma de CRUDs aislados.

Debe entenderse como una plataforma con gobierno global, contexto por tenant, base de dominio, capa transversal y módulos funcionales integrados.

<<END SECTION>>

<<SECTION: PROPÓSITO GENERAL DEL PROYECTO>>
SECTION_VERSION: 00002

PROPÓSITO GENERAL DEL PROYECTO

El propósito general de app-base es ofrecer una base estable para construir operaciones empresariales dentro de un entorno SaaS multi-tenant.

La meta no es solo disponer de entidades persistidas.

La meta es contar con una arquitectura común que permita:

ordenar operación
mantener aislamiento por tenant
centralizar gobierno transversal
integrar módulos sin pérdida de identidad
crecer sin rehacer arbitrariamente el sistema

La lectura estructural aprobada del producto incorpora además estos principios estables:

la complejidad operativa debe ser absorbida por el sistema y no trasladada al usuario como burocracia digital innecesaria

la experiencia de uso debe tender a simplicidad, claridad y bajo roce aun cuando por detrás existan procesos, validaciones, documentos o trazabilidad más complejos

la automatización constituye un pilar estructural del proyecto y debe orientar la evolución del sistema cuando permita reducir fricción, errores y carga manual repetitiva

seguridad y trazabilidad forman parte de la base obligatoria del producto y no deben tratarse como extras opcionales

la operación real del cliente tiene prioridad sobre formalismos digitales que no agreguen valor operativo genuino

El proyecto debe poder servir como base duradera para múltiples frentes operativos del negocio sin degradarse en una colección de soluciones puntuales desconectadas.

<<SECTION: ARQUITECTURA SAAS MULTI-TENANT>>
SECTION_VERSION: 00001

ARQUITECTURA SAAS MULTI-TENANT

El sistema utiliza una arquitectura SaaS multi-tenant.

Cada tenant representa una empresa o espacio de trabajo operativo independiente dentro de una misma aplicación.

El aislamiento entre empresas no se resuelve mediante aplicaciones separadas ni bases separadas por defecto, sino mediante una arquitectura común con separación lógica de datos y contexto operativo.

Principios estructurales:

cada tenant constituye una unidad operativa propia
el sistema mantiene una única plataforma común
la operación válida siempre ocurre dentro de un contexto tenant determinado
la pertenencia del usuario y el tenant activo forman parte de la lectura correcta del sistema

<<END SECTION>>

<<SECTION: MODELO DE AISLAMIENTO OPERATIVO>>
SECTION_VERSION: 00004

MODELO DE AISLAMIENTO OPERATIVO

Las entidades operativas del negocio pertenecen a un tenant y deben respetar aislamiento lógico entre empresas.

Ese aislamiento no es solo un detalle de persistencia.

Forma parte de la arquitectura conceptual del sistema.

Principios estables:

los datos de un tenant no deben mezclarse con los de otro
la operación visible y válida del usuario depende del tenant activo
las entidades globales del sistema no deben leerse como entidades del negocio
el contexto tenant no es opcional ni decorativo dentro de la operación interna

El aislamiento alcanza también a capas internas de soporte operativo que sostienen trazabilidad, agrupación contextual o ejecución de acciones dentro de módulos funcionales.

Este criterio incluye movimientos, operaciones internas y líneas operativas cuando participan como anclajes de ejecución o trazabilidad dentro del tenant.

Este documento reconoce el aislamiento como principio estructural general.

Su desarrollo técnico y sus reglas operativas concretas se resuelven en documentos especializados del proyecto.

<<END SECTION>>

↓

Lectura correcta:

TENANT ACTIVO COMO CONTEXTO OPERATIVO

El uso operativo del sistema ocurre dentro de un tenant activo.

Ese tenant activo constituye el contexto concreto desde el cual deben leerse las entidades operativas, las relaciones del negocio y las capacidades disponibles para la empresa actual.

El tenant activo define, a nivel estructural:

contexto de empresa vigente
marco operativo de los módulos del negocio
lectura correcta de las relaciones del ecosistema
ámbito de operación del usuario dentro del sistema

El tenant activo no debe interpretarse como un dato accesorio de sesión sin relevancia arquitectónica.

DISTINCIÓN ENTRE SISTEMA Y NEGOCIO

La distinción entre entidades del sistema y entidades del negocio es estructural.

Las entidades del sistema sostienen identidad, acceso, onboarding y gobierno general.

Las entidades del negocio sostienen la operación propia de cada tenant.

Regla conceptual:

Regla complementaria:

tampoco debe sobredimensionarse el dominio del negocio hasta absorber piezas cuya función real es global o transversal a la plataforma.

<<SECTION: ARQUITECTURA MODULAR DEL ECOSISTEMA>>
SECTION_VERSION: 00018

ARQUITECTURA MODULAR DEL ECOSISTEMA

La arquitectura modular del proyecto debe leerse como un ecosistema reusable y desacoplado.

Los módulos no deben relacionarse como cajas cerradas sin puntos de contacto ni como núcleos invadibles desde cualquier consumidor.

La lectura correcta distingue tres planos:

núcleo interno del módulo
puerta de enlace modular oficial
consumo externo desde otros módulos

NÚCLEO INTERNO

Corresponde a la lógica, reglas y operación propia del módulo.

No debe reabrirse desde consumidores salvo cambio real del contrato interno del propio módulo.

PUERTA DE ENLACE MODULAR OFICIAL

Corresponde a la forma publicada por el módulo para integrarse con el ecosistema.

Puede expresarse mediante:

acciones compactas
embebidos fuertes
payloads arbitrados
surfaces oficiales
acciones por fila
resolvers reutilizables
representaciones visuales oficiales

Cuando una capacidad reusable requiera arbitraje operativo real, la puerta de enlace no se agota en una vista aislada.

Debe entenderse como publicación completa del módulo oferente.

Regla arquitectónica:

debe centralizarse infraestructura repetida y mantenerse local la semántica del módulo.

Los contratos comunes pueden absorber helpers, shape repetido y utilidades mínimas de montaje.

Los resolvers, payloads, decisiones funcionales, flags UX y reglas del dominio permanecen dentro del módulo oferente.

Reducir líneas de código no constituye objetivo suficiente si se pierde legibilidad contractual, identidad modular o trazabilidad semántica.

CONSUMO EXTERNO

Corresponde al uso de capacidades publicadas por otros módulos.

La secuencia correcta es:

consumer solicita
support del módulo oferente resuelve
componente del módulo oferente representa
host monta

El consumo externo debe expresar intención y contexto, sin absorber la decisión operativa completa del módulo oferente.

HUB VISUAL

Un módulo puede alojar interacción de otro módulo dentro de su superficie sin absorber su lógica operativa.

En esa situación:

el módulo host conserva gobierno de su superficie visual

el módulo invitado conserva gobierno de su lógica operativa

Regla:

la arquitectura modular del ecosistema describe posiciones y fronteras estructurales.

No debe absorber contratos de seguridad específicos, arbitrajes de permiso, validaciones operativas detalladas ni patrones visuales que correspondan a documentos especializados.

<<END SECTION>>

BASE DEL DOMINIO

La base del dominio agrupa piezas conceptuales que sostienen múltiples relaciones relevantes dentro del ecosistema operativo.

Caso actual principal:

parties

Su función no se limita a registrar un dato aislado.

Debe leerse como anclaje relacional estable para múltiples frentes del negocio.

Regla estructural:

que una pieza pertenezca a la base del dominio no la convierte en infraestructura global del sistema.

Sigue perteneciendo al ecosistema operativo del tenant, pero con un rol conceptual más amplio que el de un módulo operativo puntual.

Regla adicional consolidada:

los usuarios internos activos del tenant no pertenecen a la base del dominio del negocio.

Su lectura correcta pertenece al plano tenant y a memberships como pertenencia operativa real.

Por lo tanto, esa necesidad no debe reinterpretarse como expansión de parties ni como apertura de un módulo users dentro de la base del dominio.

<<SECTION: MODELO GENERAL DE CRECIMIENTO ARQUITECTÓNICO>>
SECTION_VERSION: 00009

MODELO GENERAL DE CRECIMIENTO ARQUITECTÓNICO

El crecimiento del sistema debe apoyarse en módulos con identidad clara, contratos explícitos y responsabilidades separadas.

Toda evolución nueva debe tender a:

respetar multi-tenancy
preservar distinción entre sistema y negocio
mantener identidad conceptual
evitar duplicaciones
apoyarse en relaciones reales
consumir capacidades existentes antes de crear variantes paralelas
mantener coherencia documental

Reglas doctrinales:

una necesidad nueva no debe reabrir arbitrariamente el núcleo de otro módulo

si varios frentes requieren colaboradores internos activos, primero debe evaluarse una solución reusable del plano tenant

las capacidades compactas maduras deben converger hacia publicación oficial del módulo oferente

si un módulo necesita acople externo real, conviene concentrarlo en una puerta contractual explícita

si una entidad hija gana relevancia transversal, primero debe fortalecerse como hija del módulo padre antes de promoverla a módulo autónomo

los frentes transaccionales pueden evolucionar hacia estructuras internas de agrupación contextual y registros unitarios persistidos cuando eso mejore trazabilidad y orden operativo

Regla:

crecer no significa mezclar responsabilidades.

La dirección aprobada del proyecto continúa siendo modular, reusable y con fronteras explícitas.

<<END SECTION>>

<<SECTION: CONTRATO MODULAR COMO DIRECCIÓN DE CRECIMIENTO>>
SECTION_VERSION: 00004

CONTRATO MODULAR COMO DIRECCIÓN DE CRECIMIENTO

El crecimiento correcto del sistema debe priorizar evolución contractual y claridad de fronteras antes que expansión desordenada de código.

Cuando una necesidad nueva aparezca en un módulo existente, la primera lectura correcta no es invadir consumidores ni dispersar lógica transversal.

La salida preferente es fortalecer el contrato propio del módulo y ordenar internamente su dominio.

Reglas:

una mejora interna no debe romper contratos externos estables

la complejidad nueva debe resolverse primero dentro del módulo dueño del problema

los consumidores deben depender de capacidades publicadas y no de detalles internos variables

un contrato modular válido debe permitir integrar sin borrar ownership funcional

Caso vigente consolidado:

orders continúa integrándose con inventory mediante servicios y puertas contractuales explícitas, sin absorber lógica interna del frente stock.

Regla final:

la dirección de crecimiento aprobada del proyecto continúa siendo modular, reusable y contractualmente explícita.
