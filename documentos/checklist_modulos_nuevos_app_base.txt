<<SECTION: OBJETIVO>>
SECTION_VERSION: 00002

OBJETIVO

Este documento define el checklist obligatorio para crear, incorporar o reconvertir módulos dentro de app-base.

Su función es:

reducir omisiones
forzar consistencia arquitectónica
alinear seguridad, dominio y UI
evitar módulos incompletos
mantener coherencia entre implementaciones nuevas y existentes

No es una guía opcional.

Es un control mínimo obligatorio antes de considerar un módulo como integrado al sistema.

Aclaración doctrinal consolidada:

este checklist aplica tanto a módulos nuevos como a reconversiones profundas de módulos existentes cuando cambie su contrato funcional, contractual o de seguridad
<<END SECTION>>

<<SECTION: ESTRUCTURA GENERAL>>
ESTRUCTURA GENERAL

Verificar:

modelo creado
migración creada
controller creado
rutas declaradas explícitamente
vistas base creadas
enlace en navegación si corresponde
validaciones mínimas implementadas
tenant resuelto correctamente
autorización revisada
documentación actualizada
<<END SECTION>>

<<SECTION: REGLA GENERAL>>
SECTION_VERSION: 00002

REGLA GENERAL

Ningún módulo debe considerarse terminado por tener tablas, rutas o CRUD funcionando.

Un módulo solo puede considerarse integrado cuando supera este checklist en forma coherente.

Orden correcto:

definir contrato conceptual
definir contrato de seguridad
definir relaciones reales necesarias
definir superficie de integración
implementar
auditar
ajustar documentación afectada

Nunca al revés.

Regla adicional consolidada:

si una integración nueva requiere reabrir arbitrariamente el núcleo de otro módulo, primero debe revisarse si la necesidad puede resolverse consumiendo una capacidad ya publicada por ese módulo o evolucionando su superficie contractual propia
<<END SECTION>>

<<SECTION: BLOQUE 0 — IDENTIDAD DEL MÓDULO>>
SECTION_VERSION: 00001

BLOQUE 0 — IDENTIDAD DEL MÓDULO

Antes de cualquier decisión técnica debe quedar definida la identidad del módulo.

Validar:

nombre del módulo
problema real que resuelve
naturaleza principal del módulo
posición conceptual en el ecosistema
si es módulo nuevo o reconversión de uno existente

Clasificación conceptual posible:

módulo base del dominio
módulo transversal
módulo autónomo
módulo agrupador
módulo derivado

Preguntas obligatorias:

este módulo tiene identidad propia o es solo una entidad hija
es realmente un módulo o una evolución de otro ya existente
debe vivir por sí mismo o principalmente embebido
qué rol ocupa dentro del ecosistema general

Si estas respuestas no están claras, no debe avanzarse a implementación.
<<END SECTION>>

<<SECTION: BLOQUE 1 — CONTRATO MODULAR>>
SECTION_VERSION: 00013

BLOQUE 1 — CONTRATO MODULAR

Todo módulo debe poder describirse mediante un contrato modular mínimo.

Validar:

núcleo definido
contrato de entrada definido
contrato de salida definido
oferta de integración definida o declarada como futura
límites claros del módulo

NÚCLEO

Debe quedar claro qué resuelve por sí mismo.

CONTRATO DE ENTRADA

Debe quedar claro qué puede recibir desde sistema u otros módulos.

CONTRATO DE SALIDA

Debe quedar claro qué devuelve o expone.

OFERTA DE INTEGRACIÓN

Debe quedar claro qué capacidades reutilizables publica o prevé publicar.

Cuando la capacidad reusable sea compleja, debe quedar claro si el módulo publica además:

payload arbitrado desde backend
bloque visual consumidor
ubicación modular coherente con su identidad contractual

Preguntas obligatorias:

qué pertenece realmente al núcleo
qué integraciones deberían vivir fuera del núcleo
qué debería consumir otro módulo sin tocar implementación interna
qué puede publicarse hoy aunque todavía no tenga consumidor real
la capacidad reusable se publica como unidad completa o solo como vista aislada
cuál es la superficie principal real del módulo
si existe una superficie secundaria relevante que no debe leerse como base principal
si la integración reusable requerida es pesada o compacta
si el módulo nació como capacidad transversal y ahora requiere identidad propia

Regla adicional de cierre:

si la capacidad reusable ya fue cerrada doctrinalmente por el módulo oferente, no debe quedar ninguna superficie secundaria relevante reconstruyendo localmente sus estados operativos

antes de considerar cerrado el contrato modular, revisar también tablas, partials, formularios embebidos y demás superficies derivadas que consuman esa integración

la definición del contrato modular también debe dejar explícito si el módulo publica:

embebido reusable oficial para relaciones fuertes

o acción compacta oficial para relaciones puntuales

Regla evolutiva adicional:

si una capacidad operativa madura demuestra identidad funcional propia, puede reconvertirse en módulo explícito manteniendo continuidad con los consumidores existentes.

Regla adicional consolidada:

al diseñar un módulo debe evaluarse explícitamente si publicará una puerta de enlace oficial hacia el ecosistema además de su núcleo interno

debe quedar distinguido:

qué pertenece al núcleo del módulo
qué forma parte de su puerta de enlace modular
qué podrán consumir otros módulos desde esa surface oficial

si el módulo actuará como oferente de capacidad, como hub visual, o como ambos, eso debe quedar declarado desde el contrato

si el módulo admite consumo externo, deben definirse de forma explícita:

entradas aceptadas desde otros módulos
salidas posibles
reglas de arbitraje
límites de seguridad

Validación adicional para frentes operativos por línea:

si el módulo ejecuta operaciones sobre líneas hijas o recursos delegados, debe verificarse que el contrato describa también:

semántica de ejecución neta
reglas de reversión
efecto sobre pendiente
distinción entre stock y avance operativo cuando no sean equivalentes

Validación adicional para surfaces embebidas reusables:

si el módulo publicará una surface reusable embebida, debe dejar explícito al menos:

type
slot
key
label
targets
priority
view
needs
resolver

y el resolver debe devolver al menos:

count
data

Validación adicional para hosts show surface-driven:

si el módulo será consumido dentro de show pages de otros módulos, no alcanza con declarar target correcto y view publicada.

Debe auditarse también:

qué slot espera el host
si el host monta por type, por slot o por ambos
si la relación es compacta o tabular
si la publicación respeta la gramática real del host consumidor

Una offer puede estar bien orientada al target y seguir siendo invisible si omite slot o si combina type + slot de forma incompatible con el host real.

Validación adicional para datasets internos del tenant:

si la necesidad aparente del nuevo módulo es resolver colaboradores internos, pertenencia activa al tenant o dataset reusable de usuarios internos, debe evaluarse primero si corresponde una raíz reusable del plano tenant antes que abrir un módulo artificial del negocio.

Validación adicional para puertas contractuales blandas:

si el módulo necesita intervención externa real sobre recursos propios pero todavía no justifica una abstracción mayor, debe evaluarse una estación contractual explícita del propio módulo como único punto autorizado de acople hacia lógicas externas.

Debe quedar claro:

qué sigue perteneciendo al núcleo
qué delegaciones externas pasarán por esa estación
qué piezas externas especializadas podrán intervenir
qué responsabilidades continúan prohibidas en controllers o vistas

Validación adicional para retorno interno del host:

si una acción reusable nace dentro de tabs internas o surfaces embebidas del host, debe evaluarse explícitamente si el contrato necesita preservar también contexto interno del anfitrión además del trail contextual.

Controlar:

si existe tab activa relevante
si debe usarse `return_tab`
si la key de retorno es técnica y no visual
si el redirect final vuelve al host correcto y al panel correcto

Validación adicional para row actions:

si la integración se consumirá dentro de tablas nativas de otro módulo, debe evaluarse si la capacidad corresponde a una row action oficial del módulo oferente y no a un botón hardcodeado del host consumidor.

Validación adicional para líneas documentales operables:

si un módulo documental posee líneas que pueden disparar operaciones de inventory, debe verificarse que esa capacidad se lea como consumo de una oferta oficial del módulo inventory.

Controlar:

que la línea documental continúe perteneciendo al módulo documents
que inventory conserve la semántica de stock, ejecución, reversión y trazabilidad
que el host documental no reconstruya localmente reglas de operación material
que el contrato contemple origen y línea de origen trazables
que la lectura de ejecutado y pendiente derive de movimientos reales

<<END SECTION>>

<<SECTION: BLOQUE 2 — SEGURIDAD OBLIGATORIA>>
SECTION_VERSION: 00005

BLOQUE 2 — SEGURIDAD OBLIGATORIA

Ningún módulo operativo puede entrar al sistema sin contrato de seguridad explícito.

Validar:

módulo registrado
capacidades válidas
scopes válidos
constraints necesarias
dataset visible
reglas de create
reglas de update
reglas de delete
si delega autorización al padre
si requiere contexto funcional obligatorio

Validar además:

consumo correcto mediante Security
ausencia de lógica paralela en controller
ausencia de inferencias en Blade
coherencia entre view_any y view

Preguntas obligatorias:

cómo autoriza realmente este módulo
qué rompe si falta contexto
qué acciones dependen de constraints
qué dataset debe devolver Security::scope(...)

Regla adicional consolidada:

si el módulo utiliza constraints funcionales, validar explícitamente que arbitren no solo create sino también view, update y delete cuando corresponda al contrato del módulo

no deben tratarse como metadata decorativa ni como validación parcial

Validación adicional para flujos operativos con estado:

si una operación depende del estado del recurso padre, de una orden, de una línea o de una etapa del flujo, ese estado debe auditarse como parte real de la seguridad efectiva

no debe quedar como simple bloqueo visual opcional

Validación adicional para módulos transaccionales o con estados operativos:

Security, Gate/Policy, controller y UI deben devolver la misma lectura operativa

los estados readonly deben bloquear actualización y eliminación en la autorización central

el borrado debe considerar trazabilidad operativa real y no solo permiso nominal del rol

las acciones sobre recursos hijos deben respetar primero el estado del padre

las pruebas Tinker deben incluir matriz por estado y no solo permiso nominal

Validación adicional para administración owner:

si el módulo depende de activación, permisos o configuración tenant, debe existir una vía visible y real para que owner administre ese frente

auditar explícitamente:

si existen permisos del módulo en el catálogo efectivo
si esos permisos pueden asignarse realmente
si la habilitación del módulo produce consecuencia visible coherente en UI
si owner dispone de una surface real para entender y administrar ese comportamiento

<<END SECTION>>

<<SECTION: BLOQUE 3 — RELACIONES E INTEGRACIONES>>
SECTION_VERSION: 00013

BLOQUE 3 — RELACIONES E INTEGRACIONES

Toda relación con otros módulos debe justificarse por necesidad real del dominio.

Validar:

qué módulos consume
qué módulos lo consumen
si la relación depende de módulo habilitado
si existe ya una capacidad reusable equivalente
si la integración respeta contratos existentes
si evita duplicaciones de selectores, tablas o summaries
si la capacidad reusable ajena se consume como unidad completa y no solo como vista suelta
si la relación corresponde a embebido fuerte o a integración compacta
si el consumidor replica el patrón visual ya consolidado del ecosistema

Preguntas obligatorias:

esta relación agrega valor real o solo comodidad técnica
puede resolverse con una capacidad ya publicada
si falta una capacidad reusable, pertenece al módulo oferente
la integración está reabriendo un núcleo ajeno sin necesidad
el arbitraje de estados sigue centralizado en el módulo oferente o quedó disperso entre consumidores

Regla:

ninguna integración válida debe bypassar seguridad ni crear contratos paralelos.

Regla adicional de cierre:

no considerar homogénea una integración mientras existan superficies secundarias del consumidor que sigan resolviendo localmente una relación, acción o flujo ya cerrado por el módulo oferente

si el módulo consumidor ya posee un patrón visual consolidado para relaciones equivalentes, la nueva integración debe tender a reutilizar ese patrón y no introducir gramática visual nueva sin causa real

Regla adicional consolidada:

si un módulo consume otro, debe hacerlo por la surface oficial publicada por el oferente y no por lógica ad hoc

cuando la integración ocurra dentro de la pantalla de un tercer módulo, debe definirse además si ese tercer módulo actúa como simple contenedor visual o como verdadero participante del flujo

si actúa como hub visual, no debe absorber la lógica operativa ni la seguridad del módulo invitado

la integración correcta debe dejar al consumidor expresando intención y al módulo oferente resolviendo la operación completa

Validación adicional de semántica cruzada:

si la integración mezcla stock, ejecución, reversión o pendiente, debe comprobarse explícitamente que cada indicador use la regla correcta de su dominio y no una equivalencia mecánica entre signos o cantidades

Validación adicional de cierre reusable:

si una relación fuerte ya fue migrada a surface oficial del oferente, auditar también tabs fijas, includes manuales y demás superficies residuales del host antes de considerar cerrada la integración

Validación adicional para hosts estructurados:

si la integración aterriza en una show page surface-driven, debe verificarse explícitamente compatibilidad real entre:

type
slot
target
zona de montaje del host
combinación efectiva consumida por el host

No dar por buena una integración solo porque el target coincide.

Debe comprobarse que el host efectivamente consume esa combinación y que la relación se publica en la zona correcta.

Validación adicional para usuarios internos del tenant:

si una integración requiere colaboradores internos del tenant, no debe apoyarse en abilities de módulos inexistentes ni asumir que existe un módulo users formalizado.

Debe evaluarse primero si la necesidad corresponde a una raíz reusable del plano tenant y no a la creación de un módulo nuevo.

Validación adicional para recursos hijos operados:

si la integración interviene líneas hijas o recursos delegados con trazabilidad propia, debe auditarse si el recurso sigue siendo editable libremente o si requiere reglas nuevas de endurecimiento operativo.

Controlar explícitamente:

límites de edición por ejecutado neto
bloqueo de atributos identitarios
readonly por completed o cancelled
reglas de reversión consistentes
subordinación al estado operativo del padre

Validación adicional para modales y micro-superficies:

si la integración incorpora modales, tabs internas o flujos secundarios, no debe darse por cerrada solo por funcionar el show principal.

Es obligatorio auditar también esas superficies derivadas como parte real del contrato integrado.

Validación adicional para retorno al contexto del host:

si la integración ejecuta acciones desde una tab del anfitrión y luego redirige, debe verificarse que el retorno preserve tanto el recurso host como su contexto interno relevante.

Controlar explícitamente:

trail externo
return_tab interno
key técnica correcta de la tab
ausencia de desvío final a recursos delegados

Validación adicional para tablas anfitrionas:

si el host expone acciones de otro módulo dentro de filas propias, debe auditarse que esas acciones provengan de surfaces oficiales del oferente y no de reconstrucciones locales del consumidor.

Validación adicional para integración inventory sobre documents:

si documents expone operaciones de inventory sobre document_items, debe auditarse que:

la acción por fila provenga de inventory
documents no calcule localmente reglas de stock
la operación registre origen documental y línea documental
la línea conserve trazabilidad hacia movimientos reales
el ejecutado y pendiente se calculen desde movimientos persistidos
las restricciones de edición o eliminación se apliquen sin convertir document_items en módulo autónomo
la futura reversión conserve la misma frontera contractual

<<END SECTION>>

<<SECTION: BLOQUE 4 — UI Y EXPERIENCIA OPERATIVA>>
SECTION_VERSION: 00008

BLOQUE 4 — UI Y EXPERIENCIA OPERATIVA

El módulo debe integrarse visualmente al sistema de forma coherente.

Validar:

index consistente
show consistente
formularios consistentes
acciones visibles correctas
mensajes claros
embebidos consistentes
toolbars alineadas
componentes reusables aprovechados

Si publica oferta de integración visual, validar:

selector usable
tabla embebida reusable
summary legible
acciones contextuales claras
consumo visual homogéneo entre módulos

Preguntas obligatorias:

la UI refleja el contrato real del módulo
hay duplicaciones visuales evitables
el módulo se siente parte del sistema o una excepción aislada
la capacidad reusable visual vive en una ubicación coherente con el módulo oferente o quedó globalizada sin necesidad

Regla adicional de cierre:

la UI no debe considerarse alineada solo porque la vista principal consuma la capacidad reusable correcta

también deben auditarse tablas, partials, formularios embebidos y demás superficies secundarias que puedan exponer esa misma integración

Validación adicional para UX operativa:

si el flujo es complejo internamente, la experiencia visible debe seguir siendo simple, clara y de bajo roce para el usuario final

no trasladar burocracia técnica al frontend cuando el sistema puede absorberla

Validación adicional para hosts visuales:

si una pantalla anfitriona consume capacidades embebidas de otros módulos, debe quedar clara la separación visual entre:

núcleo del host
relaciones estructurales propias
capacidades invitadas publicadas por módulos oferentes

la UI del host no debe mezclar en una misma superficie decisiones operativas que pertenecen al módulo invitado

Validación adicional para show pages host:

la show page debe auditarse como host estructurado y no como un único punto genérico de inserción.

Debe verificarse la presencia y coherencia de sus zonas de montaje reales.

Casos vigentes consolidados:

header_actions
summary_items
detail_items
zona tab coordinada

La zona tab no debe evaluarse como dos renders sueltos sin estructura, sino como bloque coordinado del host.

Validación adicional para formularios con datasets internos:

un formulario no debe quedar bloqueado por depender de una fuente de datos no formalizada del catálogo modular.

Si el flujo admite fallback operativo sano, la UI debe expresarlo con claridad y bajo roce.

Caso consolidado:

cuando una tarea no recibe colaborador explícito, el sistema puede asumir asignación al usuario actual en lugar de volver el formulario inviable.

Validación adicional para acciones compactas:

si el espacio operativo lo requiere, deben priorizarse acciones iconográficas compactas consistentes en tablas embebidas o micro-superficies, siempre que la semántica siga siendo clara y el estado operable venga resuelto por el módulo oferente.

Validación adicional para embebidos maduros:

si el módulo consumidor ya invita una surface oficial suficientemente estructurada del oferente, no debe envolverla nuevamente con cards, wrappers o capas visuales redundantes que dupliquen jerarquía o confundan ownership funcional.

<<END SECTION>>

BLOQUE 5 — DOCUMENTACIÓN Y CIERRE

Ningún módulo debe cerrarse sin impacto documental revisado.

Validar:

contexto_fijo actualizado si cambia arquitectura
convenciones actualizadas si cambia doctrina
seguridad actualizada si cambia contrato de acceso
mapa_entidades actualizado si cambia ecosistema
mapa_navegacion actualizado si cambia recorridos reales
base_visual actualizada si cambia patrón visual
todo actualizado con pendientes reales

Validar además:

tests manuales o técnicos realizados
contradicciones documentales inexistentes
estado del módulo declarado

Estados posibles:

CONTRATO CERRADO
EN REVISIÓN

Regla:

solo módulos con contrato cerrado deben usarse como patrón para nuevos desarrollos

Regla adicional de cierre real:

no declarar cerrado un frente reusable mientras existan modales, tabs internas, formularios secundarios o acciones contextuales relevantes sin auditar contra la doctrina vigente.

El cierre documental debe contemplar superficie principal y micro-superficies operativas relacionadas.

<<END SECTION>>

<<SECTION: CRITERIO FINAL DE APROBACIÓN>>
SECTION_VERSION: 00009

CRITERIO FINAL DE APROBACIÓN

Un módulo no se aprueba por cantidad de código implementado.

Se aprueba cuando:

su identidad es clara
su núcleo está delimitado
sus contratos son coherentes
sus seguridad está cerrada
sus integraciones son sanas
su UI es consistente
su documentación quedó alineada

Preguntas finales obligatorias:

este módulo mejora el ecosistema o agrega ruido
podrá crecer sin romper otros módulos
puede integrarse sin refactors innecesarios futuros
podría usarse como patrón de calidad para el próximo módulo
sus superficies secundarias también quedaron alineadas
sus consumers externos dejaron de depender de variantes legacy
si actúa como host visual, mantiene separado su núcleo de las capacidades invitadas
si las relaciones fuertes ya migraron a surface oficial, desaparecieron las tabs manuales residuales equivalentes
si la necesidad original realmente justificaba un módulo nuevo y no una raíz reusable del plano tenant

Validaciones finales obligatorias para módulos transaccionales:

la reversión fue probada
el recálculo neto posterior fue validado
los estados visibles quedaron consistentes
las alertas automáticas siguen siendo legibles si hubo deduplicación
el resultado fue probado con dataset realista y no solo caso ideal
Security y Gate/Policy quedaron alineados
los estados terminales o readonly bloquean actualización y eliminación
el borrado respeta trazabilidad operativa real
los recursos hijos no contradicen el estado operativo del padre

Validaciones finales adicionales para integraciones embebidas:

si la acción nace en una tab o micro-superficie del host, el retorno vuelve al contexto correcto del anfitrión

si la integración publica row actions, el host dejó de hardcodear botones equivalentes

si la integración opera sobre líneas documentales, la lectura de estado, ejecutado y pendiente deriva de movimientos reales y no de columnas duplicadas o cálculos locales del host

si la integración con inventory quedó implementada para ejecución, debe quedar identificado si la reversión correspondiente ya fue probada o si permanece como pendiente real

<<END SECTION>>

<<SECTION: RUTAS Y CONTROLADOR>>
SECTION_VERSION: 00002

RUTAS Y CONTROLADOR

Verificar:

rutas CRUD declaradas explícitamente
no usar Route::resource(...) mientras siga vigente la convención actual del proyecto
nombres de ruta consistentes
controller delgado
validación separada cuando haga falta
authorize(...) o authorizeResource(...) aplicado cuando corresponda
no duplicar permisos en controller si la policy ya resuelve la acción
si el módulo tiene listados o datasets con visibilidad variable, resolverlos desde la capa central de autorización y no con lógica paralela ad hoc
si el módulo requiere constraints funcionales, declararlas y consumirlas explícitamente desde la capa central

Regla adicional para create contextual:

si create depende de contexto funcional obligatorio, debe quedar definido explícitamente en controller cuál es el canal válido de autorización

no debe asumirse que authorize('create', Model::class) resuelve el caso

debe consumirse Security con el contexto real requerido por el contrato

<<END SECTION>>

VISTAS

Todo módulo nuevo debe contar, como mínimo, con superficies visuales coherentes con el sistema.

Verificar:

index

partials reutilizables necesarias
componentes modulares propios cuando publiquen capacidades reusables

Reglas:

las vistas deben respetar layouts y patrones visuales vigentes

no duplicar bloques ya resueltos por componentes existentes

si una capacidad reusable pertenece al módulo, su bloque visual debe vivir dentro del propio módulo y no en una carpeta global ambigua

si el módulo publica una relación navegable reusable, no alcanza con implementarla en su show principal

también deben contemplarse tablas, sublistados y superficies secundarias consumidoras relevantes

<<END SECTION>>

<<SECTION: CHECKLIST DE INTEGRACIÓN>>
SECTION_VERSION: 00008

CHECKLIST DE INTEGRACIÓN

Cuando un módulo se relacione con otros, verificar:

consumo de contratos existentes antes de crear variantes nuevas

uso de capacidades reusables oficiales del módulo oferente

trail y retorno consistentes

flags modulares correctamente propagados

homogeneidad entre show pages, tablas y partials secundarias

ausencia de links locales remanentes hacia relaciones ya cerradas doctrinalmente por otro módulo

diferenciación explícita entre integración pesada embebida e integración compacta

alineación con el patrón visual ya consolidado en el módulo consumidor cuando exista

si el módulo reemplaza surfaces históricas dispersas, plan de migración claro hacia la nueva surface oficial

si la relación reusable se apoya en registry o mecanismo equivalente, el host consumidor no recompone localmente payload ni naming ad hoc

compatibilidad real entre target, type, slot y gramática efectiva del host consumidor

Regla de cierre:

no considerar terminada una integración por haber corregido solo la pantalla principal del consumidor.

Debe revisarse también el ecosistema visible que consume esa relación.

Regla adicional para show pages surface-driven:

si el host monta surfaces por zonas explícitas, el checklist debe verificar que cada offer publique slot correcto y que el host tenga montaje compatible con esa combinación.

Una offer sin slot o con slot incoherente puede quedar invisible aunque el resto del contrato parezca correcto.

Regla adicional para usuarios internos del tenant:

si una integración necesita usuarios internos activos, validar explícitamente que el dataset provenga del plano tenant reusable correspondiente y no de un módulo inexistente ni de una query duplicada reconstruida por cada consumidor.

<<END SECTION>>

VALIDACIONES BACKEND VS UI

la UI no sea la única barrera de control

SUPERFICIES SECUNDARIAS Y PARTIALS

Verificar:

partials profundas alineadas con la pantalla principal

calendarios alineados con reglas del módulo

formularios embebidos alineados con permisos reales

componentes reutilizables alineados con contrato del módulo

no considerar cerrada una integración reusable solo porque funciona en la vista principal del host.

sublistados
cards compactas
partials de summary
details
tabs secundarias
bloques relacionales derivados

Si una relación puede aparecer en más de una zona del host, cada una debe auditarse según la gramática real de montaje y no por similitud visual aparente.

<<SECTION: ENTIDADES HIJAS Y SUBRECURSOS>>
SECTION_VERSION: 00004

ENTIDADES HIJAS Y SUBRECURSOS

Verificar:

si el recurso hijo tiene autonomía real o depende funcionalmente del padre

si corresponde delegar autorización en el recurso padre

si evita crear un módulo artificial donde alcanza con una entidad hija bien modelada

si la UI del hijo no compite con la jerarquía del padre sin necesidad

si las restricciones reales del hijo provienen del padre o de un frente operativo relacionado ya existente

Preguntas obligatorias:

este recurso necesita identidad modular propia o sigue siendo detalle del padre
su seguridad puede delegarse completamente
su navegación debe ser estructural o contextual
su edición libre sigue siendo válida cuando ya existe trazabilidad operativa iniciada
si gana relevancia operativa, esa relevancia exige módulo nuevo o solo endurecer su contrato como hijo delegado

Regla adicional consolidada:

un subrecurso delegado puede ganar relevancia operativa transversal sin dejar de ser hijo

eso ocurre cuando pasa a actuar como ancla explícita de ejecución, trazabilidad o restricciones dentro de otro frente operativo

caso vigente relevante:

order_items respecto de inventory

En esos casos, validar además:

que no se le inventen permisos autónomos innecesarios
que la trazabilidad no rompa la delegación doctrinal al padre
que la UI exprese su nuevo peso operativo sin promoverlo falsamente a módulo
que edición y borrado dejen de ser libres cuando ya existan movimientos ejecutados o equivalentes

<<SECTION: NAVEGACIÓN Y BREADCRUMB>>
SECTION_VERSION: 00003

NAVEGACIÓN Y BREADCRUMB

Definir desde el inicio si el módulo necesita solo navegación estructural o también navegación contextual.

Preguntas de control:

la entidad puede abrirse desde otra entidad relacionada como continuación natural de trabajo

el usuario necesita volver al origen sin perder el hilo operativo

ese origen es semántico y estable, no solo la página anterior

Si alguna de estas respuestas es sí, el módulo debe evaluar e implementar navegación contextual explícita.

Si corresponde, el módulo debe:

tener builder o resolutor de trail propio compatible con el mecanismo central vigente del proyecto

aceptar contexto de navegación de forma explícita

resolver ese contexto desde la capa reutilizable central vigente

propagar ese contexto en tablas, tabs, acciones por fila, links relacionales, paginación, filtros, botones volver, botones cancelar y redirects relevantes

resolver create -> store -> show sin persistencia de nodos transitorios de “Nuevo …”

Validar además:

cuál es la superficie principal real del módulo

si existe una superficie secundaria que permanece accesible pero no debe actuar como base promovida del trail

si calendarios, tabs o embebidos contextuales necesitan actuar como entrada principal efectiva del módulo

Regla adicional consolidada:

cuando una capacidad reusable oficial del módulo oferente se consuma dentro de otro módulo, también debe propagar el trail válido del contenedor y no reiniciar navegación desde una lectura estructural local del oferente

<<END SECTION>>

<<SECTION: CIERRE>>
SECTION_VERSION: 00002

CIERRE

Un módulo no debe considerarse terminado solo porque funciona.

Debe quedar:

consistente con el sistema
alineado con permisos reales
alineado con multi-tenancy
alineado con la base visual
alineado con la documentación vigente

Además debe quedar explícitamente resuelto:

si el dataset se gobierna desde la capa central de autorización

si el módulo usa solo scopes o también constraints funcionales

si las entidades hijas o embebidas se autorizan por delegación al padre

si un create contextual requiere contexto obligatorio y cuál es su canal válido de autorización

si la validación manual del módulo puede reproducirse de forma canónica con Tinker sin ambigüedad operativa

<<END SECTION>>
