<<SECTION: OBJETIVO>>
SECTION_VERSION: 00005

OBJETIVO

Este documento define el checklist obligatorio para crear, incorporar o reconvertir módulos dentro de app-base.

Su función es:

reducir omisiones
forzar consistencia arquitectónica
alinear seguridad, dominio y UI
evitar módulos incompletos
mantener coherencia entre implementaciones nuevas y existentes

No es una guía opcional.

Es un control mínimo obligatorio antes de considerar un módulo, reconversión profunda, capacidad transversal o capa satélite como integrada al sistema.

Aclaración doctrinal consolidada:

este checklist aplica tanto a módulos nuevos como a reconversiones profundas de módulos existentes cuando cambie su contrato funcional, contractual, de modelado o de seguridad.

También aplica, con la adaptación correspondiente, a:

capacidades transversales
capacidades tenant no modulares
surfaces tenant/transversales no modulares
piezas satélite productivas
frentes operativos especializados
universos semánticos
entradas públicas externas
automatizaciones relevantes
integraciones futuras con IA

Regla:

si una capacidad no es módulo ordinario, el checklist no debe forzarla a convertirse en módulo.

Debe ayudar a clasificarla correctamente antes de implementar.

Casos vigentes:

Servicios y Mantenimiento debe evaluarse como capa satélite productiva ERP/CMMS liviana.

Shopping Autoservicio debe evaluarse como capa satélite comercial autogestiva por tenant.

Regla:

una capa satélite, un frente operativo, un universo semántico o una entrada pública externa no debe considerarse cerrada por tener rutas, tablas o vistas.

Debe validarse su contrato conceptual, seguridad, ownership, navegación, runtime real y documentación afectada.

<<SECTION: REGLA GENERAL>>
SECTION_VERSION: 00018

REGLA GENERAL

Ningún módulo debe considerarse terminado por tener tablas, rutas o CRUD funcionando.

Un módulo, capacidad, frente contextual o universo semántico solo puede considerarse integrado cuando supera este checklist en forma coherente.

Orden correcto:

definir contrato conceptual
definir contrato de seguridad
definir relaciones reales necesarias
definir superficie de integración
definir si participa en trazabilidad operativa transversal
definir si consume lectura contextual de actividad operativa, directa, agregada o contextual especial cuando corresponda
definir si la necesidad corresponde a módulo funcional ordinario, capacidad transversal, pieza satélite, entidad hija, subcapa interna de módulo existente, surface tenant/transversal, frente operativo especializado, universo semántico o entrada pública externa
definir si una necesidad material corresponde a stock físico, ledger formal, cierre operativo, contrato de entrega/aplicación/devolución, producción formal por etapas o modo simple instantáneo
implementar
auditar
ajustar documentación afectada

Nunca al revés.

Regla adicional consolidada:

si una integración nueva requiere reabrir arbitrariamente el núcleo de otro módulo, primero debe revisarse si la necesidad puede resolverse consumiendo una capacidad ya publicada por ese módulo o evolucionando su superficie contractual propia.

Regla adicional para reconversiones profundas:

este checklist también aplica cuando no se crea un módulo nuevo, pero se reconvierte un frente funcional, visual, contractual o de seguridad ya existente.

Caso vigente:

tenant/profile/accesses fue reconvertido en relación con owner/admin y edición de funciones de memberships.

Regla:

una reconversión de acceso o gobierno interno debe validarse con el mismo rigor que un módulo sensible.

Debe incluir UI correcta, backend centralizado, requests manipulados, persistencia final y rollback o limpieza de pruebas.

Regla adicional para subcapas internas de módulos existentes:

si una capacidad contextual pertenece a un módulo host, no posee CRUD, no tiene navegación propia, no requiere ModuleCatalog propio, no entra al dashboard general y no posee rutas propias, debe evaluarse como subcapa interna del módulo antes de tratarla como módulo nuevo.

Caso vigente:

ProductOperationalSummary se define como subcapa contextual de Products.

No debe tratarse como módulo nuevo.

No posee CRUD.

No posee navegación propia.

No posee ModuleCatalog propio.

No entra al dashboard general.

No posee rutas propias.

Se publica mediante ProductSurfaceService como summary_items dentro de products.show.

Regla:

una subcapa interna puede tener contrato propio de acceso y visibilidad sin convertirse en módulo funcional ordinario.

Debe quedar claro:

qué módulo host la aloja
qué módulos aporta datos
qué módulo conserva ownership de cada dato
qué condiciones de visibilidad aplican
qué surfaces la publican
qué vistas representan el payload
qué queda fuera de su alcance inicial

Regla adicional para capas satélite productivas:

si una necesidad organiza un frente operativo amplio sobre módulos existentes, no debe asumirse que corresponde crear un módulo CRUD ordinario.

Debe evaluarse si corresponde:

módulo funcional
pieza satélite
capacidad transversal
surface tenant/transversal
dashboard operativo especializado
experiencia enfocada
lectura consolidada
contrato nuevo del módulo dueño
subcapa contextual interna de un módulo existente
universo semántico de navegación y uso
entrada pública externa

Caso vigente:

Servicios y Mantenimiento debe leerse como capa satélite productiva ERP/CMMS liviana y no como CRUD aislado.

Reglas:

no duplicar ownership de módulos existentes

no absorber Orders, Assets, Inventory, Products, Appointments, Tasks, Documents, Attachments, Parties ni OperationalActivity

si una necesidad es propia de mantenimiento pero no corresponde al núcleo de un módulo existente, debe evaluarse como pieza satélite antes de reabrir módulos cerrados

si una necesidad puede resolverse mediante lectura consolidada, surfaces, contratos existentes o capacidades oficiales, no debe abrirse una pieza nueva por reflejo automático

Caso operativo vigente:

service_maintenance existe como frente habilitable no CRUD.

service_maintenance.view_any permite acceso al área Servicio y mantenimiento.

Orders sostiene la primera base operativa mediante:

group = service

Universo semántico vigente:

/service

Reglas:

el acceso al área no reemplaza permisos operativos de Orders ni de otros módulos participantes

las acciones reales continúan bajo el contrato del módulo dueño

el frente habilitable no CRUD no debe tratarse como CRUD ordinario

el universo semántico no debe tratarse como ownership funcional

la entrada visual no debe convertirse por sí sola en duplicación de navegación, tablas o rutas si puede apoyarse en contratos vigentes

Regla adicional para capas satélite comerciales autogestivas:

si una necesidad organiza una experiencia comercial externa por tenant, no debe asumirse que corresponde crear un módulo CRUD ordinario ni reutilizar directamente el plano interno del tenant.

Debe evaluarse si corresponde:

entrada pública externa
registro externo
capacidad satélite comercial
entidad auxiliar de registro pendiente
login externo separado
tienda pública controlada
administración interna separada de catálogo o tienda
contratos propios de validación
separación frente a memberships
relación con Parties
relación futura con Products, Inventory y Documents
integración futura con pasarela de pago
integración futura con QR o controlador externo
validación posterior de identidad operativa
habilitación operativa externa separada de la confirmación inicial

Caso vigente:

Shopping Autoservicio debe leerse como capa satélite comercial autogestiva por tenant.

Key técnica:

self_service_sales

Universo público tenant-scoped vigente:

/shop/{tenant:slug}

Entrada pública general vigente:

/tienda

Label visible:

Tienda

Módulo interno estándar relacionado:

shops

Administración interna vigente:

/shops

Rutas internas vigentes:

shops.*

Reglas:

self_service_sales no debe tratarse como CRUD ordinario

self_service_sales no debe tratarse como panel interno completo del tenant

self_service_sales consume catálogo publicado por shops

self_service_sales no administra internamente shops

shops administra configuración interna de tienda y artículos publicados

la administración interna de shops no debe mezclarse con el plano externo customer

self_service_sales no debe reemplazar Parties

self_service_sales no debe reemplazar Products

self_service_sales no debe reemplazar Inventory

self_service_sales no debe reemplazar Documents

self_service_sales no debe reemplazar Security

el cliente externo no debe convertirse en User

el cliente externo no debe convertirse en Membership

el cliente externo no debe recibir roles internos

el cliente externo no debe acceder al panel interno

el cliente externo no debe administrar tienda

el cliente externo no debe administrar shops

la tienda pública no debe habilitar operación interna del tenant

el flujo externo no debe reutilizar controllers internos cuando pertenezcan a otro plano de seguridad

la confirmación de email no debe leerse como habilitación operativa completa

la habilitación para operar requiere identidad operativa suficiente según contrato vigente o futuro específico

Caso operativo vigente:

self_service_sales inicia con registro externo pendiente, confirmación por token y creación de Party customer person.

Entidad auxiliar vigente:

self_service_customer_registrations

Regla:

la existencia de una entrada pública externa no autoriza navegación interna.

La resolución del tenant por slug público no debilita el aislamiento multi-tenant.

Regla adicional para administración interna de shops:

si una capa satélite externa consume una configuración interna, debe distinguirse entre:

configuración interna del tenant
publicación comercial
consumo público externo
operación transaccional futura

Caso vigente:

Shop representa cabecera/configuración interna de tienda.

ShopItem representa producto publicado.

ProductLineItemSelector o capacidad equivalente de Products puede usarse como base para seleccionar productos activos publicables.

Reglas:

Shop no reemplaza Tenant

ShopItem no reemplaza Product

publicar producto no equivale a mover stock

publicar producto no equivale a vender

activar tienda no equivale a habilitar operación comercial completa

administración interna no debe tocar sesión externa customer

administración interna no debe autorizarse por SelfServiceExternalSession

customer externo no debe operar rutas shops.*

Regla adicional para contrato material Inventory/Orders:

si una capacidad mueve, entrega, aplica, consume, devuelve o regulariza materiales, debe distinguirse entre:

stock físico
movimiento físico
entrega formal
aplicación formal
devolución formal
saldo formal entregado
cierre material
modo simple instantáneo
producción formal por etapas
modo formal por etapas

Casos vigentes:

InventoryMovement = libro físico de stock.

InventoryMaterialFlow = ledger formal del subcontrato material.

InventoryMaterialFlow no reemplaza InventoryMovement.

InventoryMaterialFlow no mueve stock por sí mismo.

Tipos formales vigentes:

formal_delivery
formal_application
formal_return

Reglas:

Inventory conserva ownership material

Orders conserva ownership de la orden, líneas y lifecycle

Orders no calcula saldos materiales

Orders no administra el ledger material formal

Orders no reconstruye lógica de stock

Orders no decide consumos ni aplicaciones materiales

si una operación implica stock físico, debe pasar por Inventory

si una operación implica regularización material formal, debe pasar por InventoryMaterialFlow o frontera oficial de Inventory

no usar InventoryMovement.kind como sustituto del contrato formal por etapas cuando se requiere saldo trazable

no resolver entrega/aplicación/devolución formal solo con metadata

no aplicar más de lo entregado disponible

no devolver más de lo entregado disponible

no crear movimientos parciales si un guard material bloquea la operación

no cerrar contrato material formal con movimientos manuales sueltos

la aplicación formal no crea InventoryMovement

la devolución formal puede crear movimiento físico de ingreso

la UI no debe elegir kind ni reconstruir disponibilidad

Blade debe renderizar payload preparado por backend

Regla adicional para producción:

si producción funciona en modo simple, puede consumir componentes e ingresar producto terminado en una única acción validada.

Si producción funciona en modo formal por etapas, debe aplicar componentes previamente entregados y no descontarlos nuevamente desde depósito.

Debe validarse la cantidad producible contra el componente físico entregado disponible más restrictivo.

Los componentes service no participan en consumo ni aplicación material física.

Regla adicional para cierres:

si el cierre depende de condiciones de otro módulo, debe consultar blockers o fronteras oficiales del módulo dueño.

Caso vigente:

el cierre de orden consulta blockers materiales de Inventory.

Si existe saldo formal pendiente, el cierre debe bloquearse.

Regla:

la UI no puede ser la única barrera operativa.

Un botón oculto o deshabilitado no reemplaza validación backend.

Regla adicional para pruebas runtime:

cuando una operación tiene efectos materiales, no alcanza con validar creación de registros.

Debe validarse también:

ausencia de efectos colaterales no deseados

ausencia de movimientos parciales ante bloqueo

rollback limpio cuando corresponda

estado final de saldos formales

estado final de stock físico

bloqueo de cierre cuando corresponde

posibilidad de cierre cuando el material queda regularizado

conteos antes/después medidos bajo el mismo contexto tenant

Regla:

en proyectos tenant-scoped, los conteos de validación deben medirse con el mismo contexto de tenant antes y después.

No comparar conteos globales contra conteos tenant-scoped ni mezclar scopes en pruebas de delta.

Regla adicional para registro externo liviano:

si el flujo externo inicial usa registro liviano, debe distinguirse entre:

identificación inicial de contacto
confirmación de email
identidad operativa suficiente
habilitación para operar

Datos mínimos vigentes del registro inicial liviano:

nombre visible
email
teléfono

Lectura:

el email funciona como vínculo fuerte inicial.

El teléfono es dato obligatorio de contacto, pero no reemplaza al email como vínculo principal.

Regla:

DNI, nombre real completo, datos fiscales, contraseña externa completa, saldo, fichas, QR, comprobantes y facturación no deben forzarse en el registro inicial liviano si el contrato vigente los reserva para validación posterior o fases futuras.

Regla adicional para vínculos gestionados opcionales y referencias propias:

si un módulo posee un vínculo gestionado opcional y una referencia textual propia, el checklist debe auditar también sus salidas visuales y contextuales.

Debe revisarse:

formularios
listados
tablas principales
tablas embebidas
show summaries
prints
surfaces
summary_items
row actions
componentes linked
seeders cuando creen registros directamente

Reglas:

si existe vínculo gestionado autorizado, la salida visual debe permitir link oficial

si no existe vínculo gestionado, debe mostrar referencia propia

no debe inventar link

no debe ocultar la referencia propia por ausencia del vínculo gestionado

no debe interpretar record_metadata desde Blade

la lectura contextual o de manualidad debe exponerse mediante helper, presenter o payload preparado

Caso vigente:

Orders representa Contraparte y Activo mediante helpers propios y componentes linked oficiales cuando existe vínculo gestionado.

Regla adicional para flags visuales de tablas reusables:

si un SurfaceService o payload publica un flag visual para columnas, debe validarse que el partial final lo represente realmente.

Caso vigente:

showCounterparty

showAsset

Regla:

un flag visual no debe quedar como contrato fantasma.

Si no tiene representación real, debe implementarse o eliminarse.

Regla adicional para SurfaceService:

cuando una reconversión afecte un SurfaceService, debe distinguirse:

lógica propia del módulo

mecánica transversal reusable

configuración contextual

hosts soportados

offers publicadas

payload preparado

vistas consumidas

Regla:

no vaciar un SurfaceService por reflejo.

No migrar masivamente sin validación runtime.

Extraer solo mecánica reusable cuando corresponda.

Caso vigente:

OrderSurfaceService conserva configuración propia.

La mecánica transversal reusable se extrajo a:

app/Support/Modules/Concerns/BuildsHostPacks.php
app/Support/Modules/Concerns/ResolvesRelatedEmbeddedSurfaces.php

Regla adicional para clasificaciones:

si una clasificación mezcla dimensiones conceptuales distintas, debe frenarse la implementación hasta revisar el contrato funcional.

No deben mezclarse en un mismo campo:

propiedad o relación del objeto
ubicación de ejecución

lifecycle

pertenencia operativa

Regla adicional para actividad operativa:

no todo módulo debe registrar actividad por defecto.

La participación en OperationalActivity debe justificarse por valor operativo real.

No debe registrarse actividad solo por existir CRUD, eventos técnicos o cambios persistidos sin significancia operativa.

La integración con OperationalActivity debe definirse como parte del contrato del módulo dueño.

Debe quedar claro:

qué operaciones emiten eventos
qué campos son observables
qué campos quedan excluidos por ruido técnico
qué cambios tienen valor operativo real
si existen relaciones hijas que deban expresarse como metadata operativa
si existe entidad agrupadora que deba actuar como ancla operativa
si el módulo consume actividad operativa contextual
si la lectura contextual será directa, agregada o contextual especial

Casos validados:

parties puede registrar cambios de party_roles como metadata operativa de Party.

inventory debe registrar InventoryOperation como ancla operativa agrupada, no InventoryMovement como actividad principal.

projects puede declarar un activity_record_set para lectura agregada de actividad contextual de Project + Tasks.

parties puede declarar un activity_context para lectura contextual especial de Party employee con membership activa vinculada.

Regla adicional para lectura contextual de actividad operativa:

la lectura directa no requiere provider adicional.

Se limita al record host autorizado.

La lectura agregada requiere que el módulo dueño declare explícitamente su frontera mediante un provider compatible con ActivityRecordSetProvider.

La lectura contextual especial requiere que el módulo dueño declare explícitamente su criterio mediante un provider compatible con ActivityContextProvider.

Antes de declarar un activity_record_set debe quedar claro:

qué records forman parte de la actividad contextual del host
por qué esos records pertenecen funcionalmente al host
si esa agregación altera o no ownership
si el host ya autorizó el record principal
si la lectura agregada no equivale a lectura general del tenant
si no se está reconstruyendo historia desde campos actuales en lugar de leer OperationalActivity persistido

Antes de declarar un activity_context debe quedar claro:

qué criterio amplía el dataset
qué módulo dueño define la frontera
qué condición habilita esa lectura especial
si respeta tenant activo
si no habilita lectura general del tenant
si no reemplaza permisos del host
si no contamina OperationalActivity con lógica del módulo

Regla:

OperationalActivity no debe modificarse con condiciones específicas por módulo.

No corresponde agregar lógica como:

if Project
if Order
if Document
if Party employee

La frontera debe vivir en el módulo dueño o en su contrato publicado.

Caso validado de activity_record_set:

projects.show
ProjectActivityRecordSet
Project + Tasks

Caso validado de activity_context:

parties.show
PartyEmployeeActivityContext
Party employee con membership activa vinculada
actividad directa de Party
actividad donde el user vinculado sea actor o sujeto

Regla adicional para capacidades tenant no modulares:

no toda capacidad tenant-scoped debe incorporarse a ModuleCatalog.

Si una capacidad:

no es módulo funcional
no tiene navegación propia
no tiene CRUD completo
no se habilita ni deshabilita como módulo ordinario

debe evaluarse como capacidad tenant no modular.

Caso validado:

operational_activity.view_any

Regla:

las capacidades tenant no modulares pueden ser evaluadas por Security sin convertirse en módulos funcionales ordinarios.

Regla adicional para surfaces tenant/transversales no modulares:

una capacidad tenant-scoped o transversal puede publicar una surface hacia hosts funcionales sin convertirse en módulo funcional.

Esta integración no representa un módulo nuevo si la capacidad:

no posee navegación principal
no posee CRUD ordinario
no se incorpora a ModuleCatalog
no pasa por TenantModuleAccess
no se habilita ni deshabilita como módulo ordinario

Caso validado:

OperationalActivitySurfaceService publica una tab contextual Actividad para hosts autorizados mediante TenantSurfaceRegistry.

Antes de crear una surface para una capacidad tenant/transversal se debe verificar:

si la capacidad es módulo funcional o no
si corresponde ModuleCatalog o TenantSurfaceRegistry
si necesita navegación propia o solo consumo embebido
si la autorización proviene de permiso propio o del host
si la surface representa lectura general o contextual
si el host ya autorizó el registro antes de consumir la surface

Regla:

no debe quedar como cierre estable un push manual de tabs en vistas host si la capacidad ya puede publicarse mediante registry reusable.

Regla adicional para surfaces modulares:

una capacidad interna de un módulo host puede publicarse mediante el surface service del módulo cuando deba integrarse de forma natural al show del host.

Caso vigente:

ProductOperationalSummary se publica mediante ProductSurfaceService.

Publica tres summary_items:

Compras
Ventas
Entradas / salidas

Regla:

si la infraestructura surface-driven vigente puede publicar la capacidad, no debe dejarse como cierre estable un include directo en el Blade del host.

Regla adicional para IA:

si una capacidad produce información relevante para IA futura, debe exponerla mediante contratos claros, datos estructurados, catálogos, surfaces o lecturas autorizadas.

No debe depender de scraping visual ni labels dispersos.

BLOQUE 0 — IDENTIDAD DEL MÓDULO

Antes de cualquier decisión técnica debe quedar definida la identidad del módulo o capacidad.

Validar:

nombre del módulo o capacidad
problema real que resuelve
naturaleza principal
posición conceptual en el ecosistema
si es módulo nuevo o reconversión de uno existente
si es módulo funcional, entidad hija, capacidad transversal de soporte, capacidad tenant no modular, pieza interna del tenant, pieza satélite productiva, subcapa interna de módulo existente, frente operativo especializado, universo semántico, entrada pública externa o surface tenant/transversal no modular
si configura datos internos
si consume datos internos
si publica datos hacia plano externo
si opera en plano interno
si opera en plano externo

Clasificación conceptual posible:

módulo base del dominio
módulo transversal
módulo autónomo
módulo agrupador
módulo derivado
capacidad transversal de soporte
capacidad tenant no modular
subcapa interna de módulo existente
capa satélite
frente operativo especializado
universo semántico
entrada pública externa
administración interna de una capacidad externa
configuración interna consumida por una capa externa

Reglas:

no todo CRUD técnico es módulo funcional

no toda ruta pública es módulo

no toda tabla auxiliar es entidad navegable

no toda administración interna de una capa externa pertenece a la capa externa

no todo consumo público externo autoriza administración interna

no todo cambio de clasificación requiere migrar nombres físicos de tablas

Caso vigente:

shops es módulo interno estándar del tenant.

self_service_sales es capa satélite comercial externa que consume Shop publicado.

Lectura:

Shop configura catálogo interno.

Self-Service Sales consume publicación externa.

Regla:

la capa satélite no debe apropiarse de la administración interna de la entidad configurada por un módulo estándar.

BLOQUE 1 — CONTRATO MODULAR

Antes de aprobar un módulo nuevo, reconversión profunda, capacidad transversal, pieza satélite o subcapa interna debe poder explicarse con claridad:

núcleo propio
contrato de entrada
contrato de salida
contrato de seguridad
oferta de integración
participación o no en trazabilidad operativa transversal
participación o no en lectura contextual de actividad operativa
relación con módulos dueños existentes
límite de ownership funcional
si requiere datos estructurados para reportes, trazabilidad, automatización o IA futura

Validar además:

si realmente es módulo y no dataset reusable del tenant
si resuelve un problema propio del dominio
si evita invadir núcleos existentes
si sus integraciones futuras pueden publicarse por contrato
si el ownership funcional queda claro
si debe emitir eventos operativos livianos después de acciones propias autorizadas

CRITERIO FINAL DE APROBACIÓN

Una capacidad nueva no se considera madura solo porque renderiza correctamente.

contrato modular

multi-tenancy

No alcanza con una vista compartida ni con una refactorización parcial.

Se considera cierre sólido cuando desaparecen duplicaciones relevantes sin perder claridad, ownership funcional, trazabilidad ni seguridad.

Criterio adicional para reconversiones profundas:

una reconversión se considera cerrada solo cuando la semántica anterior deja de operar en todas las capas relevantes.

Validaciones mínimas esperadas:

código aplicado

Criterio adicional para reconversión de constraints funcionales:

una reconversión de permisos no se considera cerrada si la constraint anterior sigue operando como fallback legacy.

Criterio adicional para autonomía modular:

una reconversión estructural no se considera cerrada si el módulo todavía depende en controller, request, vista, seeder, print, surface o servicio de un vínculo que fue definido como opcional por doctrina y esquema.

Validaciones mínimas esperadas:

modelo coherente con el contrato
migraciones en orden real de dependencias

seeders cargando snapshots propios cuando no pasan por controllers
vistas usando método de presentación propio cuando exista snapshot
surfaces sin dependencia exclusiva de relaciones opcionales
surfaces agrupadas por capacidad real cuando aplique
prints mostrando dato propio correcto
helpers linked genéricos sin métodos específicos por consumidor
hooks o services de frontera para bloqueos externos
selectors o providers cuando varios módulos consumen un dominio externo
catálogos definiendo significado de negocio propio del módulo
servicios transversales limitados a mecánica común
prueba runtime web del flujo principal
prueba Tinker cuando corresponda
php -l sobre archivos modificados
php artisan optimize:clear
php artisan migrate:fresh --seed cuando el saneamiento sea estructural

Caso vigente:

Orders puede existir sin party_id usando counterparty_name.

Documents puede derivarse desde Orders autónomas usando counterparty_name propio.

Tasks puede asociarse a Orders desde tasks.order_id.

Orders no debe conservar task_id persistente.

Order::tasks() fue eliminado porque la dirección funcional vigente es Task -> Order.

Criterio adicional para flujos derivados:

un flujo derivado no se considera cerrado solo porque crea el registro destino.

Debe validar además:

autorización del módulo receptor
datos propios del registro destino
líneas copiadas cuando corresponda
cálculos o totales reales
lifecycle inicial correcto
salida visual posterior
ausencia de dependencia a métodos inexistentes
prueba runtime web real

Caso vigente:

Order → Document debe respetar documents.create, copiar contraparte, copiar ítems y recalcular totales desde document_items sin depender de métodos inexistentes.

Criterio adicional para cierre de fase:

una fase de saneamiento no debe declararse cerrada sin auditar al menos:

controller principal
controllers hijos relevantes
modelo
helpers linked
surface services
hooks
services/support vinculados
catálogos
rutas principales
validaciones runtime o smoke tests relevantes
grep de residuos cuando existan nombres legacy conocidos

Caso vigente:

Fase 0 de saneamiento evidente de Orders quedó cerrada funcionalmente luego de auditar y sanear controller, modelo, linked helper, surfaces, hooks, selección de productos y numeración.

Regla final:

cerrar una intervención significa que el sistema queda más coherente, autónomo, seguro y alineado con su doctrina vigente.
