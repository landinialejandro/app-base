<<SECTION: OBJETIVO DEL DOCUMENTO>>
SECTION_VERSION: 00003

OBJETIVO DEL DOCUMENTO

Este documento define la doctrina oficial de seguridad, acceso y autorización del proyecto app-base.

Su función es establecer reglas obligatorias para controlar:

ingreso
pertenencia
tenant activo
visibilidad
capacidades operativas
constraints funcionales
contratos de seguridad modulares
acciones críticas

Este documento es la fuente especializada de verdad para toda materia relativa a acceso y autorización.

Si existiera contradicción con un documento no especializado, prevalece este documento en su materia.

No corresponde definir aquí:

navegación visual
patrones UI
roadmap
estructura física del código
detalle funcional completo de cada módulo
historial de implementación
contexto arquitectónico general ajeno a seguridad

Regla:

este documento debe permanecer breve, fuerte y operativo.

Debe contener doctrina de seguridad, no inventario completo de casos implementados.

<<END SECTION>>

<<SECTION: PRINCIPIOS INMUTABLES DE AUTORIZACIÓN>>
SECTION_VERSION: 00034

[AUTORIDAD CENTRAL]
Decisión final = backend / capa central de seguridad.
Nada visual, contextual, satélite, transversal, público, automatizado o IA reemplaza esa decisión.

[TENANT]
Toda autorización ocurre dentro de tenant correcto.
No operar registros ajenos.
Slug público puede resolver tenant, pero no habilita acceso interno.

[OWNER / ADMIN]
owner = gobierno estructural vía memberships.is_owner.
admin = rol delegable fuerte.
admin ≠ owner.
Regla: admin opera fuerte; owner gobierna.

[PLANOS SEPARADOS]
Dataset disponible ≠ visibilidad UI ≠ permiso efectivo ≠ trazabilidad ≠ automatización ≠ IA ≠ autorización.
Plano interno ≠ plano externo.
Gestión interna de tienda ≠ consumo público de tienda.

[NO AUTORIZAN]

* UI visible
* dashboard
* universo semántico
* entrada pública
* capa satélite
* subcapa contextual
* vínculo opcional
* query param
* hidden input
* JS
* log operativo
* IA
* automatización
* sesión externa mínima
* middleware externo pasivo
* tienda pública
* catálogo publicado
* customer externo habilitado visualmente

[MÍNIMO ACOPLE]
Consumidor no reinterpreta seguridad de otro módulo.
Usar capacidades oficiales.
Respetar ownership y contrato del módulo dueño.

[FRONTERAS EXTERNAS]
Si acción propia depende de condición externa:
módulo dueño consulta hook/service/frontera explícita.
No reconstruir lógica externa.
Falla en bloqueo crítico = bloquear.

[AUTONOMÍA MODULAR]
Módulo autónomo ≠ acceso libre.
Vínculo opcional ≠ permiso.
Snapshot textual ≠ permiso.
Contexto request ≠ permiso.

[ACTIVIDAD]
Lectura general tenant-scoped requiere:
operational_activity.view_any

Lectura contextual requiere:
host autorizado + frontera declarada si es agregada/especial.

[CAPAS SATÉLITE]
Permiso área ≠ permiso operativo.
Universo semántico ≠ contrato del módulo dueño.
Entrada pública ≠ acceso interno.
Administración interna de una capa satélite requiere usuario interno autenticado, tenant activo y autorización backend.

Casos:

* /service: universo Servicio y mantenimiento; acciones reales = Orders u otros módulos dueños.
* self_service_sales: plano externo customer separado del plano interno tenant.

[SHOPPING AUTOSERVICIO]
Puede ingresar ≠ puede operar.
Cuenta externa válida ≠ operación comercial habilitada.
Sesión externa mínima ≠ operación comercial habilitada.
Catálogo publicado visible ≠ compra habilitada.
Customer externo no administra tienda.
operation_enabled = false bloquea operación comercial.

[ACCIONES CRÍTICAS]
Requieren autorización efectiva + validación backend + confirmación humana o regla explícita cuando corresponda.

Ejemplos:
crear/cerrar/cancelar/reabrir orden
aprobar/cerrar documento
consumir/revertir stock
modificar permisos
habilitar módulos
eliminar registros
modificar datos sensibles
confirmar token externo
crear customer externo
habilitar operación comercial externa
activar tienda pública
publicar artículos de tienda
acreditar saldo futuro
consumir ficha futura
ejecutar cobro futuro
emitir comprobante futuro

[ESTADOS TERMINALES]
Order closed/cancelled no permite operación ordinaria.
Reapertura = acción restringida.
Estado satélite no puede saltar lifecycle base.

[REGLA SÍNTESIS]
Todo plano auxiliar debe volver a autorización backend efectiva.

<<END SECTION>>

<<SECTION: ACCESO PÚBLICO Y ACCESO AUTENTICADO>>

[ACCESO PÚBLICO]
Previo o externo a operación interna autenticada.
Puede mostrar onboarding, login, tienda pública, entrada externa, selector externo, salida externa o formularios externos.
No habilita operación interna.

[ACCESO INTERNO AUTENTICADO]
Requiere:

User autenticado
Membership vigente
Tenant activo
Autorización efectiva

[TENANT POR SLUG]
Puede resolver tenant.
No debilita aislamiento.
No habilita acceso interno.
Operaciones públicas se asocian al tenant_id correspondiente.

[ENTRADA PÚBLICA GENERAL]
Entrada pública externa ≠ login interno.
No autentica User.
No crea Membership.
No habilita panel interno.
No reemplaza /login.
No reemplaza /solicitar-empresa.
No habilita operación comercial completa por sí misma.

[SELECTOR EXTERNO]
Selector externo ≠ selector interno de tenants.
No pertenece al panel interno.
No abre navegación tenant interna.
No habilita dashboard.
No habilita tenant/profile.
No habilita operación comercial completa.

[TOKEN TEMPORAL]
Token temporal externo ≠ sesión interna.
No concede pertenencia interna.
No concede roles internos.
No habilita operación comercial.
No reemplaza autorización externa posterior.
Token usado no puede reutilizarse.

[SESIÓN EXTERNA]
Sesión externa ≠ sesión interna app-base.
Sesión externa ≠ tenant activo interno.
Sesión externa ≠ Membership.
Sesión externa ≠ User.
Sesión externa ≠ guard externo definitivo.
Sesión externa ≠ operación comercial habilitada.
Sesión externa ≠ autorización para administración interna.

Debe usar namespace propio cuando corresponda.

No debe:

usar Auth::login()
usar Fortify interno
crear User
crear Membership
crear roles internos
escribir session('tenant_id')
habilitar panel interno
habilitar dashboard
habilitar tenant/profile
habilitar operación comercial completa
habilitar administración interna de tienda

[MIDDLEWARE EXTERNO PASIVO]
Middleware externo pasivo puede resolver contexto externo.
No es autorización obligatoria.
No bloquea visitantes públicos.
No exige sesión externa para ver tienda pública.
No redirige por falta de sesión externa.
No habilita operación comercial.
No reemplaza guard externo definitivo.
No reemplaza middleware externo definitivo de autorización obligatoria.
No escribe session('tenant_id').
No usa Auth.
No crea User.
No crea Membership.

[REGISTRO EXTERNO]
Puede crear registro pendiente externo.
Puede confirmar email por token.
Puede crear Party customer person al confirmar.

No debe:

usar flujos internos de invitations
usar contratos internos de alta de Parties
usar contratos internos de memberships
usar login interno
crear User
crear Membership
iniciar sesión interna

[CONFIRMACIÓN POR TOKEN]
Acción sensible del plano externo.

Debe validar:

tenant
token
estado
vencimiento
duplicados

antes de crear Party.

[HABILITACIÓN OPERATIVA]
Confirmar email no habilita operación comercial completa.
Completar identidad operativa no habilita operación comercial completa.
Crear contraseña externa no habilita operación comercial completa.
Validar credencial externa propia no habilita operación comercial completa.
Seleccionar tienda no habilita operación comercial completa.
Iniciar sesión externa mínima no habilita operación comercial completa.
Resolver contexto externo pasivo no habilita operación comercial completa.
Redirigir a tienda pública no habilita operación comercial completa.
Ver catálogo publicado no habilita operación comercial completa.

Para operar deberá existir:

identidad operativa suficiente
relación customer-tienda válida
habilitación operativa explícita
autorización externa correspondiente al flujo

[TIENDA PÚBLICA Y CATÁLOGO]
Tienda pública y catálogo publicado no habilitan:

compra
pago
saldo
fichas operativas
QR
órdenes
documentos
movimientos de stock
carrito
selección transaccional
administración interna de tienda

[ADMINISTRACIÓN INTERNA DE TIENDA]
Administración interna de tienda pertenece al plano interno.

Debe requerir:

User autenticado
Membership vigente
tenant activo
autorización backend efectiva

No puede ser autorizada por:

customer externo
sesión externa
middleware externo pasivo
tienda pública
entrada pública

[ERRORES DE ACCESO EXTERNO]
Los fallos de acceso externo deben usar mensajes genéricos cuando corresponda.

No deben revelar causa exacta del fallo.

[REGLA GENERAL PARA FLUJOS EXTERNOS]
Todo flujo externo debe validar identidad externa, tenant, relación externa, estado operativo y alcance antes de permitir cualquier operación.

[REGLA FINAL]
Autenticación externa ≠ autorización interna.
Sesión externa ≠ sesión interna.
Middleware externo pasivo ≠ autorización obligatoria.
Operar en tienda ≠ pertenecer al tenant.
Entrada pública ≠ panel interno.
Selector externo ≠ selector interno de tenants.
Token temporal ≠ sesión interna.
Selección de tienda ≠ operación comercial habilitada.
Contexto externo pasivo ≠ operación comercial habilitada.
Catálogo publicado ≠ operación comercial habilitada.
Customer externo ≠ administrador interno de tienda.
Salida externa ≠ logout interno.
