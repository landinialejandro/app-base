<<SECTION: REGLA CENTRAL DE TRABAJO>>
REGLA CENTRAL DE TRABAJO

No inventar nada.

No asumir patrones por memoria cuando deban validarse.
No completar huecos críticos con supuestos.
No proponer estructuras, flujos, archivos o convenciones no verificadas como si fueran hechos.

Primero entender con precisión:

- el pedido puntual
- el código real entregado
- el estado real verificable del proyecto

Luego contrastar eso con la documentación vigente.

La documentación es contexto obligatorio, pero el pedido puntual y el código real tienen prioridad para determinar el estado efectivo del proyecto.

Si el código o el pedido muestran una actualización todavía no reflejada en la documentación:

- no forzar la documentación por encima del código
- señalar la diferencia con claridad
- trabajar sobre el estado real verificable
- indicar si corresponde o no actualizar documentación
<<END SECTION>>

<<SECTION: CRITERIO DE AVANCE>>
CRITERIO DE AVANCE

No bloquear la respuesta por faltantes menores si igualmente se puede aportar algo útil, concreto y razonable.

Si el contexto alcanza para responder, proponer, corregir o implementar parcialmente, avanzar.

Solo frenar o pedir archivos cuando falte contexto realmente crítico para no romper el proyecto, contradecir el código o inventar una solución no verificable.

Ante duda menor, priorizar mejor esfuerzo útil y explícito antes que silencio o bloqueo innecesario.
<<END SECTION>>

<<SECTION: REGLAS ESPECÍFICAS DEL PROYECTO>>
REGLAS ESPECÍFICAS DEL PROYECTO

* no usar Route::resource(...) en el core mientras siga vigente la convención actual
* no proponer modales para create/edit operativos salvo excepción muy justificada
* si hay navegación contextual, reutilizar el mecanismo central vigente y no inventar variantes paralelas
* no reconstruir navegación funcional desde url()->previous(), historial o supuestos implícitos
* no inventar clases CSS, estructuras visuales ni patrones frontend fuera de la base visual vigente
* no exponer tenant_id como dato de formulario del usuario
* no asumir tenant_id como foreignId() sin verificar el patrón real del proyecto
* si el tema toca permisos, tomar como fuente de verdad obligatoria el documento seguridad_acceso_proyecto_app_base
* no proponer lógica de permisos fuera del contrato persistido en base de datos
* no usar autorización abstracta cuando la capacidad dependa de constraints
* auditar cualquier cambio de permisos contra el documento de seguridad antes de proponer implementación
* validar policy + authorize + @can cuando corresponda, sin contradecir la capa central ni el contrato persistido

<<SECTION: REGLA PARA ARCHIVOS COMPLETOS DE CÓDIGO>>
REGLA PARA ARCHIVOS COMPLETOS DE CÓDIGO

- cuando devuelvas un archivo completo, la primera línea debe ser siempre un comentario válido para ese lenguaje con este formato:
FILE: ruta/del/archivo.ext | VX
- si el archivo es nuevo, usar V1
- si el archivo ya existe y no tiene comentario de versión, al devolverlo completo agregarlo como V2
- si el archivo ya tiene comentario de versión, incrementar en 1 la última versión existente
- esta regla aplica solo cuando se devuelve el archivo completo
- si se devuelve solo un fragmento o bloque parcial, no agregar ni modificar versión
- el comentario debe escribirse con la sintaxis correcta del archivo correspondiente
<<END SECTION>>

<<SECTION: SI NECESITÁS ARCHIVOS>>
SI NECESITÁS ARCHIVOS

Si necesitás archivos, primero determiná exactamente cuáles son necesarios para entender o auditar el caso.

- no pedir archivos de más
- no pedir bloques genéricos si no hacen falta
- no pedir archivos si el caso ya puede resolverse razonablemente con lo disponible

Cuando los necesites, generame un único comando de consola con todos los archivos exactos y suficientes.

Ejemplo:

(
find app/Models -type f -name "Product.php";
find app/Http/Controllers -type f -name "ProductController.php";
find resources/views/products -type f -name "*.blade.php";
) | sort | while read file; do
    echo "===== $file ====="
    cat "$file"
    echo
    echo
done > documentos/auditoria/auditoria-[nombre-de-referencia].txt

Indicame todos los archivos necesarios así te los paso en un único bloque.
