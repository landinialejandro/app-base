INSTRUCCIÓN INICIAL OBLIGATORIA

Antes de proponer cambios, auditar código o escribir implementación, revisar los documentos vigentes del proyecto y tomarlos como contexto.

Documentos base a considerar:

* contexto_fijo_proyecto_app_base
* convenciones_proyecto_app_base
* checklist_modulos_nuevos_app_base
* mapa_entidades_app_base
* mapa_navegacion_relaciones_app_base
* base_visual_proyecto_app_base
* todo_proyecto_app_base
* deploy_compatibilidad_proyecto_app_base
* seguridad_acceso_proyecto_app_base

Documento operativo adicional cuando el caso involucre permisos, constraints, multi-tenancy, runtime real o validaciones manuales:

* guia_tinker_proyecto_app_base

Regla documental adicional:

cuando el pedido involucre navegación o automatización sobre la base documental del proyecto, tratar DOC_SLUG como identificador canónico del documento y no usar nombres de archivo legacy ni títulos humanos como clave técnica

Regla de cierre adicional:

un módulo no debe considerarse cerrado solo porque controller, rutas o vista principal parezcan correctos

el cierre válido exige revisar también superficies secundarias relevantes del módulo, por ejemplo:

partials
tablas embebidas
calendarios
formularios embebidos
acciones reutilizables
flujos hijos
redirects
cancelar
breadcrumbs
navegación profunda

el veredicto de cierre debe basarse en código real y comportamiento verificable del sistema

Regla adicional de validación relacional:

no asumir modelado de relaciones por intuición

si una integración entre módulos parece implicar una foreign key, primero verificar columnas, relaciones y flujo real en código o runtime antes de documentarla o implementarla

Regla adicional de cierre modular:

antes de dar por cerrado un módulo, reanclar explícitamente cuál es su superficie principal real

si existe una superficie secundaria relevante, dejarla nombrada como tal y no leerla como base promovida del módulo por inercia

<<SECTION: REGLA CENTRAL DE TRABAJO>>

REGLA CENTRAL DE TRABAJO

No inventar nada.

No asumir patrones por memoria cuando deban validarse.
No completar huecos críticos con supuestos.
No proponer estructuras, flujos, archivos o convenciones no verificadas como si fueran hechos.

Primero entender con precisión:

* el pedido puntual
* el código real entregado
* el estado real verificable del proyecto

Luego contrastar eso con la documentación vigente.

La documentación es contexto obligatorio, pero el pedido puntual y el código real tienen prioridad para determinar el estado efectivo del proyecto.

Si el código o el pedido muestran una actualización todavía no reflejada en la documentación:

* no forzar la documentación por encima del código
* señalar la diferencia con claridad
* trabajar sobre el estado real verificable
* indicar si corresponde o no actualizar documentación

Aplicación operativa explícita de la regla:

NO SE ASUME.
NO SE SUPONE.

No asumir tenant activo en Tinker.
No asumir que una prueba abstracta reproduce el runtime real.
No asumir que Gate o policy abstracta equivalen a autorización válida cuando el contrato exige contexto.
No asumir relaciones simétricas o foreign keys del lado intuitivo sin verificar código o base real.

Regla operativa adicional:

no dar por cerrado un módulo por inspección parcial

si el pedido es cerrar, alinear o auditar un módulo, debe revisarse también si quedaron remanentes en superficies secundarias o derivadas antes de emitir veredicto de cierre
<<END SECTION>>

<<SECTION: CRITERIO DE AVANCE>>
SECTION_VERSION: 00004

CRITERIO DE AVANCE

Avanzar en pasos pequeños, verificables y reversibles.

Priorizar primero:

contrato doctrinal
alineación real de implementación
validación runtime cuando aplique
homogeneidad de superficies consumidoras
documentación final

No declarar cerrado un frente por inspección parcial.

Un módulo puede quedar cerrado en su núcleo y mantener pendientes residuales explícitos.

Cuando el cambio toque permisos, constraints, datasets o multi-tenancy, priorizar prueba real antes de declarar cierre.

Regla adicional consolidada:

si el frente incluye consumo transversal desde otros módulos, el avance correcto tiende a seguir este orden:

reanclar superficie principal del módulo
publicar oferta reusable oficial desde el propio módulo
integrar consumidores respetando el patrón visual ya consolidado
barrer trail, permisos, dataset y superficies secundarias residuales
cerrar documentación

<<END SECTION>>

<<SECTION: REGLAS ESPECÍFICAS DEL PROYECTO>>

REGLAS ESPECÍFICAS DEL PROYECTO

La documentación vigente es fuente formal de verdad salvo que se esté auditando una divergencia con código real.

No inventar comportamiento.

No asumir estructuras, tablas, columnas, rutas, APIs ni flujos no verificados.

NO SE ASUME. NO SE SUPONE.

Si falta un dato crítico, pedir exactamente ese dato y no otro.

Leer primero los documentos involucrados antes de proponer cambios documentales.

Si se detecta contradicción entre documentos, señalarla antes de corregir.

Si una capacidad reusable ya fue publicada por el módulo oferente, no proponer duplicaciones locales en módulos consumidores salvo justificación real.

El cierre de un frente modular requiere revisar también superficies secundarias relevantes y no solo la pantalla principal.

Regla adicional consolidada:

cuando un módulo consumidor ya tenga un patrón visual consolidado para relaciones equivalentes, las nuevas integraciones deben replicar ese patrón y no introducir gramática visual nueva sin justificación real.

Regla adicional sobre integraciones:

distinguir explícitamente entre relación embebida fuerte y relación compacta

no forzar tabs o tablas pesadas cuando la relación real solo requiere una acción contextual compacta

SI NECESITÁS ARCHIVOS

Si necesitás archivos, primero determiná exactamente cuáles son necesarios para entender o auditar el caso.

* no pedir archivos de más
* no pedir bloques genéricos si no hacen falta
* no pedir archivos si el caso ya puede resolverse razonablemente con lo disponible

Cuando el caso sea de permisos, constraints, Security, policies o multi-tenancy, pedir además lo mínimo necesario para reproducir el runtime real y no una abstracción incompleta.

Si el caso requiere Tinker, seguir la guía operativa vigente del proyecto.

Cuando los necesites, generame un único comando de consola con todos los archivos exactos y suficientes.

Ejemplo:

(
find app/Models -type f -name "Product.php";
find app/Http/Controllers -type f -name "ProductController.php";
find resources/views/products -type f -name "*.blade.php";
) | sort | while read file; do
echo "===== $file ====="
cat "$file"
echo
echo
done > documentos/auditoria/auditoria-[nombre-de-referencia].txt

Indicame todos los archivos necesarios así te los paso en un único bloque.
