<<SECTION: OBJETIVO>>
SECTION_VERSION: 00002

OBJETIVO

Este documento define el mapa vigente de navegación, recorridos funcionales y relaciones visibles entre módulos de app-base.

Su función es:

describir recorridos reales del sistema
ordenar navegación principal y contextual
documentar accesos entre módulos
evitar flujos inconsistentes
guiar futuras ampliaciones de navegación

No define:

seguridad doctrinal
modelo conceptual completo de entidades
contratos modulares internos
detalle visual de componentes

Esos aspectos se resuelven en otros documentos.

Regla consolidada:

este documento describe navegación real o navegación objetivo ya validada

no debe cargarse con teoría arquitectónica general que corresponda a otros documentos
<<END SECTION>>

<<SECTION: REGLA GENERAL>>
SECTION_VERSION: 00007

REGLA GENERAL

La navegación del sistema debe ser coherente con la arquitectura modular vigente.

Navegar no equivale a integrar.

Integrar no equivale a duplicar.

Toda relación entre módulos debe resolverse preguntando primero:

es un salto de navegación
es una capacidad reusable
es una relación compacta
es una relación fuerte tabular
es una acción contextual
es parte del núcleo del host

La respuesta correcta define dónde y cómo debe aparecer.

Regla adicional:

si una relación ya fue cerrada como capacidad reusable oficial del módulo oferente, no corresponde sostener una variante local equivalente como camino principal.

Regla adicional para show pages host:

antes de agregar una tab, validar si la necesidad real se resuelve mejor en summary, details o acción compacta.

La pestaña debe reservarse para relaciones densas o flujos con entidad propia dentro del show.

Regla adicional de compatibilidad:

si una integración depende de host surface-driven, no basta con definir target narrativo.

Debe existir compatibilidad real entre:

type publicado
slot publicado
zona consumida por el host
estructura efectiva del show

<<END SECTION>>

<<SECTION: REGLAS DE LECTURA>>
SECTION_VERSION: 00002

REGLAS DE LECTURA

Una relación en este documento significa acceso, recorrido o vínculo visible para el usuario dentro del sistema.

No implica automáticamente:

foreign key obligatoria
permiso concedido
implementación ya terminada
visibilidad universal
módulo siempre habilitado

Toda navegación real sigue condicionada por:

módulo habilitado para el tenant
permiso efectivo
estado funcional del recurso
flujo operativo aplicable

Tipos de navegación relevantes:

estructural
contextual
embebida
derivada

NAVEGACIÓN ESTRUCTURAL

Recorrido estable propio del módulo.

Ejemplo:

menú principal → index → show → edit

NAVEGACIÓN CONTEXTUAL

Recorrido originado desde otro módulo conservando contexto.

Ejemplo:

appointment.show → crear order contextual

NAVEGACIÓN EMBEBIDA

Consumo de tablas, bloques o relaciones dentro de otra pantalla.

Ejemplo:

orders embebidas dentro de asset.show

NAVEGACIÓN DERIVADA

Flujo donde una acción en un módulo genera o abre otro módulo como consecuencia operativa.

Ejemplo:

orders.show → crear document

Regla:

cuando existan varias rutas posibles, debe priorizarse la más consistente con NavigationTrail y con el flujo real del usuario
<<END SECTION>>

<<SECTION: NAVEGACIÓN PRINCIPAL TENANT>>
SECTION_VERSION: 00006

NAVEGACIÓN PRINCIPAL TENANT

El tenant autenticado accede a una navegación principal compuesta por módulos habilitados para su empresa y visibles según permiso efectivo.

Recorrido base:

dashboard
↓
módulos habilitados
↓
index del módulo
↓
show del recurso
↓
acciones permitidas

Módulos operativos principales actuales:

projects
tasks
appointments
assets
products
orders
documents
inventory

Módulos base o transversales visibles según implementación:

parties
attachments principalmente como navegación embebida o contextual

Reglas:

no mostrar módulos no habilitados
no mostrar acciones no permitidas
no asumir igualdad de menú entre tenants distintos
la navegación principal debe reflejar contrato real del sistema y no solo existencia técnica de rutas

Regla adicional consolidada:

la aparición visible de un módulo en navbar, dashboard o accesos rápidos no debe resolverse por enlaces forzados en UI

debe resultar de la combinación real de:

módulo habilitado para el tenant
permiso efectivo del usuario
estado de configuración aplicable cuando corresponda

Aclaración consolidada para inventory:

inventory ya forma parte de la lectura principal del ecosistema tenant

inventory.index funciona como entrada estructural del frente

inventory.show funciona como ficha operativa por producto

los accesos contextuales desde products, orders o documents complementan esa navegación y no la reemplazan

si inventory aún no resulta visible para un tenant o usuario concreto, esa ausencia debe leerse primero como efecto de contrato efectivo o configuración pendiente y no como invitación a forzar accesos manuales en UI

<<END SECTION>>

<<SECTION: NAVEGACIÓN CONTEXTUAL ENTRE MÓDULOS>>
SECTION_VERSION: 00009

NAVEGACIÓN CONTEXTUAL ENTRE MÓDULOS

La navegación entre módulos debe responder al flujo real de trabajo y no a enlaces arbitrarios entre entidades.

Reglas:

cada salto debe tener sentido operativo
el retorno debe preservar contexto cuando corresponda
no multiplicar accesos equivalentes sin necesidad
priorizar recorridos frecuentes
no confundir navegación estructural con integración reusable

Casos vigentes:

tasks puede abrir orders contextualmente
appointments puede abrir orders contextualmente
orders puede abrir documents relacionados
products puede abrir inventory como frente operativo de stock
assets puede abrir appointments relacionados
parties puede abrir appointments relacionados

Regla adicional consolidada:

cuando una relación reusable ya resuelva apertura, create contextual o navegación contextual completa, debe priorizarse esa surface oficial antes que links locales aislados.

Regla adicional para hosts show:

una relación contextual visible dentro de un show page puede resolverse por distintas zonas del host:

header
summary
details
tabs

La elección debe responder a densidad real, frecuencia de uso y peso operativo de la relación.

No toda navegación contextual merece una tab.

Las relaciones puntuales o 1 a 1 deben priorizar summary, detail o acción compacta antes que pestañas pesadas.

Caso consolidado actual:

appointments.show consume relaciones compactas publicadas por assets, parties y orders dentro de su zona summary.

<<END SECTION>>

<<SECTION: RELACIONES EMBEBIDAS EN SHOW PAGES>>
SECTION_VERSION: 00010

RELACIONES EMBEBIDAS EN SHOW PAGES

Las relaciones embebidas dentro de show pages deben organizarse según relevancia operativa y peso real del contenido.

Tipos conceptuales:

relación compacta
relación de detalle
relación tabular fuerte
acción contextual

RELACIÓN COMPACTA

Corresponde a vínculos puntuales, 1 a 1 o de baja densidad.

Debe priorizarse en:

summary
details
acciones contextuales

RELACIÓN DE DETALLE

Corresponde a información complementaria útil pero no dominante.

Debe resolverse en bloques detail o cards secundarias.

RELACIÓN TABULAR FUERTE

Corresponde a datasets densos, historiales, tablas operativas o flujos con navegación interna.

Debe priorizarse en la zona tab del host.

ACCIÓN CONTEXTUAL

Corresponde a navegación, apertura, create o acción rápida dependiente de contexto.

Debe vivir cerca del dato que le da sentido.

Reglas:

no toda relación visible justifica una tab
no duplicar la misma relación en varias zonas sin causa real
si existe surface oficial del módulo oferente, priorizarla
el host compone la experiencia visual, no reemplaza el contrato del oferente

Regla adicional para show pages surface-driven:

la zona tab no debe leerse como panel huérfano separado de su navegación.

Debe entenderse como item tabular completo del host que coordina:

toolbar
botones
estado activo
panel visible

Caso consolidado actual:

las relaciones fuertes publicadas en tabs deben entrar como unidad tabular completa.

Las relaciones compactas hacia appointments.show deben entrar como summary linked y no como tabs pesadas.

<<SECTION: FLUJOS DERIVADOS RELEVANTES>>

FLUJOS DERIVADOS RELEVANTES

task
→ order

appointment
→ order

order
→ document

order_items
→ document_items por copia funcional cuando corresponda

signup_request aprobado
→ invitation owner_signup
→ tenant inicializado
→ owner onboarding

Reglas:

un flujo derivado expresa continuidad operativa entre módulos con identidad propia

no implica subordinación conceptual absoluta del módulo destino respecto del origen

task → order y appointment → order representan derivación hacia un módulo autónomo ya consolidado

si el flujo derivado requiere contexto, debe preservar seguridad y trail correspondiente

si existe una order ya vinculada al origen funcional, debe priorizarse apertura consistente de esa order antes de ofrecer alta duplicada

Aclaración relacional consolidada:

en task → order el vínculo estructural persistido se resuelve desde orders.task_id

en appointment → order el vínculo operativo real se resuelve desde appointments.order_id y no requiere orders.appointment_id

<<SECTION: REGLAS DEL TRAIL>>
SECTION_VERSION: 00006

REGLAS DEL TRAIL

El trail debe preservar orientación del usuario entre módulos relacionados.

Objetivos:

saber dónde está
entender desde dónde llegó
poder volver sin fricción
mantener contexto operativo

Reglas:

el trail debe construirse desde origen real y no desde rutas genéricas
los retornos deben respetar contexto cuando exista
no perder filtros o query útil sin motivo
las integraciones reusables deben propagar trail cuando corresponda

Regla adicional para hosts embebidos:

si una relación se consume dentro de un show page sin salir de pantalla, no siempre requiere breadcrumb nuevo.

Puede requerir solo contexto interno del host.

Regla adicional para acciones compactas:

si una relación compacta abre otro módulo o crea contextualmente desde el host, debe conservar trail suficiente para retorno limpio al recurso anfitrión.

Caso vigente:

relaciones compactas desde appointments.show hacia assets, parties y orders deben mantener retorno coherente al appointment anfitrión cuando navegan fuera del host.

<<END SECTION>>

<<SECTION: MAPA DE NAVEGACIÓN PRINCIPAL>>
SECTION_VERSION: 00005

MAPA DE NAVEGACIÓN PRINCIPAL

La navegación principal del sistema organiza acceso a frentes funcionales y no solo a tablas CRUD.

Entradas vigentes del ecosistema:

dashboard
projects
tasks
appointments
parties
products
assets
orders
documents
inventory

Reglas:

cada entrada principal debe corresponder a un frente reconocible del negocio
dashboard resume o deriva, no reemplaza módulos
los módulos con identidad propia deben tender a tener acceso claro
la navegación principal no obliga a que toda relación secundaria tenga menú propio

Aclaración adicional:

un módulo puede tener navegación principal y además publicar capacidades reutilizables consumidas dentro de otros hosts.

Aclaración sobre hosts avanzados:

la centralidad de un módulo en el menú no lo convierte automáticamente en patrón rector de estructura show.

La madurez contractual de cada host se evalúa por su propia gramática de montaje.

Caso vigente:

appointments.show expresa una estructura host más avanzada que otros shows aún en transición.

<<END SECTION>>

<<SECTION: INVENTORY>>
SECTION_VERSION: 00004

INVENTORY

Lectura estructural vigente:

inventory.index
tabla principal de saldos por producto

inventory.show
ficha operativa por producto

Relaciones de navegación relevantes:

products.show
→ inventory.show del producto cuando corresponda

orders.show
→ surface operativa de inventory por línea sin absorber su navegación principal

documents.show
→ trazabilidad de movimientos cuando exista relación documental

inventory.show
→ orden contextual
→ documento contextual
→ producto relacionado

Reglas:

inventory debe publicar sus propias surfaces oficiales desde el propio módulo

las tablas de movimientos y formularios operativos no deben quedar dispersos como resolución oficial en products u orders

products conserva la navegación del catálogo maestro

inventory asume la navegación del saldo, historial y movimiento

cuando inventory sea consumido de forma embebida desde otros módulos, esa experiencia no debe reemplazar ni borrar su navegación estructural propia

Aclaración operativa consolidada:

cuando orders.show consuma inventory por línea, esa lectura embebida debe poder resolver por fila:

stock
ejecutado
pendiente
estado operativo
acción rápida contextual

sin degradar inventory.show como ficha estructural del producto

Aclaración adicional:

la navegación contextual desde una línea hacia inventory no debe leerse como simple apertura genérica del producto

puede requerir preservar el order_item como contexto operativo de origen cuando eso siga siendo útil para continuar el trabajo

Aclaración adicional de naming visual:

cuando inventory se consuma como capacidad embebida en orders.show, la lectura correcta del frente ya no debe reducirse a “movimientos” genéricos

debe expresar operación por línea, diferenciada de la capa estructural de ítems del host

<<END SECTION>>

<<SECTION: REGLA PARA LINKS RELACIONALES DESDE TABLAS CONTEXTUALES>>
SECTION_VERSION: 00004

REGLA PARA LINKS RELACIONALES DESDE TABLAS CONTEXTUALES

Cuando una tabla enlaza desde una entidad fila hacia una entidad relacionada, debe evaluarse cuál es el contexto real que el breadcrumb necesita preservar.

Regla práctica:

si el trabajo sigue anclado en la fila desde la que se abrió el vínculo, el breadcrumb esperado puede ser:

módulo origen
↓
entidad fila
↓
entidad relacionada

Esto aplica especialmente en:

tablas embebidas dentro de show

listados contextuales dentro de tabs

acciones por fila hacia entidades relacionadas

sublistados donde el contenedor no alcanza por sí solo para conservar el contexto de origen

Regla adicional consolidada:

en una tabla contextual, el link principal de la fila debe consolidar primero el trail completo de esa fila

los links relacionales secundarios de esa misma fila deben reutilizar ese trail ya construido y no reiniciar contexto desde el contenedor

Regla adicional para integraciones reusables:

si la fila expone una capacidad compleja ya publicada por otro módulo, no debe reconstruirse localmente por conveniencia dentro de la tabla

debe consumirse la capacidad oficial del módulo oferente también en esa superficie contextual

Casos vigentes relevantes:

linked-order-action
linked-party-action
linked-asset-action

<<END SECTION>>

<<SECTION: DETALLE DE FLUJOS RELEVANTES>>
SECTION_VERSION: 00009

DETALLE DE FLUJOS RELEVANTES

APPOINTMENTS -> PARTIES

Uso típico:

desde un turno se consulta o abre el contacto relacionado

Contexto esperado:

si la party se abre desde el turno, puede conservarse retorno útil al appointment de origen cuando siga siendo relevante para la tarea actual

Regla adicional validada:

si parties no está habilitado para el tenant, la UI no debe mostrar ni mantener operativo ese acceso relacional

APPOINTMENTS -> ASSETS

Uso típico:

desde un turno se consulta o abre el activo relacionado

Contexto esperado:

si el asset se abre desde el turno, puede conservarse retorno útil al appointment de origen cuando siga siendo relevante para la tarea actual

Regla adicional validada:

si assets no está habilitado para el tenant, la UI no debe mostrar ni mantener operativo ese acceso relacional

APPOINTMENTS -> ORDERS

Uso típico:

desde un turno se crea o abre una orden vinculada

Contexto esperado:

si ya existe orden relacionada, debe priorizarse su apertura consistente

si no existe, debe resolverse create contextual desde la capacidad reusable oficial de orders

ORDERS -> DOCUMENTS

Uso típico:

desde una orden se crea o abre documentación derivada

Contexto esperado:

la creación o apertura documental debe conservar el flujo host de la orden cuando siga siendo relevante

Regla adicional consolidada:

la lectura embebida de documents en orders.show debe provenir de la surface oficial publicada por documents y no de una tab manual reconstruida por el host

ORDERS -> INVENTORY POR LÍNEA

Uso típico:

desde la tabla de ítems se ejecuta operación material sobre una línea

Contexto esperado:

la línea sigue siendo el anclaje funcional del trabajo

la operación visible debe resolver stock, ejecutado, pendiente, estado y acción rápida contextual desde inventory

Regla adicional consolidada:

orders actúa como host visual del flujo, pero la lógica operativa y la semántica material pertenecen a inventory

PRODUCTS -> INVENTORY

Uso típico:

desde un producto se consulta stock, historial o ficha operativa de inventario

Contexto esperado:

products conserva la lectura de catálogo maestro

inventory concentra saldo, historial y operación

DOCUMENTS -> ORDERS

Uso típico:

desde un documento derivado se vuelve a la orden origen cuando exista

Contexto esperado:

el retorno debe preservar la lectura del flujo documental derivado y no romper el contexto operativo útil

<<SECTION: CRITERIOS DE AUDITORÍA FUTURA>>
CRITERIOS DE AUDITORÍA FUTURA

Al auditar breadcrumb, trail o navegación entre módulos, verificar:

si la pantalla debía abrirse en modo estructural o contextual

si el trail se propagó correctamente

si hubo colapso de nodos repetidos

si volver y cancelar respetan el nodo previo real

si redirects post acción mantienen el contexto útil

si el breadcrumb visible sigue siendo compacto

si navbar y sidebar marcan el área funcional esperada y no solo la hoja final abierta

si en tablas contextuales el trail correcto era el del contenedor o el de la fila

si hay pantallas donde el contexto se pierde por filtros, tabs, paginación o acciones secundarias

si los enlaces, acciones y campos relacionales hacia módulos relacionados verifican primero que el módulo destino esté habilitado para el tenant

si la deshabilitación de módulos destino elimina completamente:

accesos
bloques visuales
referencias indirectas

especialmente en flujos desde appointments hacia orders y assets
<<END SECTION>>

<<SECTION: CRITERIO FINAL>>
SECTION_VERSION: 00002

CRITERIO FINAL

La navegación correcta del sistema no es la que muestra más enlaces.

Es la que permite al usuario llegar de forma clara, segura y coherente a la acción que necesita.

Una relación visible entre módulos se justifica cuando:

mejora continuidad operativa
reduce fricción real
respeta permisos
respeta módulos habilitados
mantiene contexto útil
no duplica caminos innecesarios

Si una relación existe técnicamente pero no cumple estos criterios, no debe priorizarse como navegación visible.
