OBJETIVO

Este documento define el modelo operativo obligatorio de seguridad y acceso del sistema app-base.

Este documento es:

la única fuente de verdad para permisos V2
el contrato de implementación obligatorio para cualquier módulo
la referencia normativa para resolver autorización en todo el sistema
la base doctrinal para arbitrar contratos de seguridad modulares

Regla doctrinal consolidada:

cada módulo del sistema debe poseer su propio contrato de seguridad

ese contrato no reemplaza la capa central

la capa central de seguridad arbitra, valida y consume los contratos modulares dentro del marco general del sistema

<<SECTION: PRINCIPIOS INMUTABLES DE AUTORIZACIÓN>>
SECTION_VERSION: 00006

PRINCIPIOS INMUTABLES DE AUTORIZACIÓN

Este bloque define reglas que no admiten excepción.

REGLA 1 — FUENTE DE VERDAD

La base de datos es la única fuente de verdad del contrato de permisos V2 persistido.

Esto incluye:

role_permission
constraints
scopes
capabilities

Este documento define la normativa de implementación del sistema.

Los catálogos definen el vocabulario técnico permitido.

La capa central resuelve y aplica el contrato persistido.

Cualquier definición de permisos fuera de la base de datos debe considerarse una desviación arquitectónica inválida.

Aclaración obligatoria:

la expresión fuente de verdad refiere al permiso persistido efectivo

no reemplaza:

la función normativa de este documento
la función de vocabulario controlado de los catálogos
la función de arbitraje y consumo de la capa central

REGLA 2 — CAPA CENTRAL

Toda autorización debe resolverse exclusivamente mediante la capa central.

No se permite:

definir permisos en controllers
definir permisos en policies
definir permisos en Blade

Estas capas solo pueden consumir la capa central.

REGLA 3 — PROHIBICIÓN DE EVALUACIÓN ABSTRACTA

Ninguna capacidad puede evaluarse sin contexto si el módulo utiliza constraints obligatorias o si la acción depende de datos funcionales previos.

Evaluar capacidades sin contexto debe considerarse implementación inválida.

Ejemplo inválido:

authorize('create', Order::class)

cuando el módulo requiere kind.

Gate o policy abstracta no equivalen a autorización válida cuando el contrato exige contexto funcional explícito.

REGLA 4 — CONSTRAINTS OBLIGATORIOS

Si un módulo define constraints:

las constraints son obligatorias
no son opcionales
forman parte del contrato del permiso

Un permiso sin constraints completas debe considerarse inválido.

Regla adicional consolidada:

si el contrato del módulo exige que una constraint arbitre segmentación funcional real, esa constraint debe impactar efectivamente create, view, update y delete según corresponda

no debe tratarse como metadata decorativa ni como validación parcial de una sola acción

REGLA 5 — NO DUPLICACIÓN

No puede existir lógica de permisos en más de un nivel.

la base de datos define el contrato persistido
el contrato de seguridad del módulo define el comportamiento esperado del módulo
la capa central arbitra el contrato persistido
policy consume esa resolución
controller consume y ejecuta la autorización correspondiente
Blade muestra solo el resultado permitido

Ninguna de estas capas puede redefinir permisos.

REGLA 6 — UI SIN LÓGICA

La UI no puede:

inferir permisos
evaluar capacidades abstractas
reconstruir lógica

Solo puede consumir resultados del backend.

REGLA 7 — CONSISTENCIA GLOBAL

view_any debe ser igual o más amplio que view.

No se permiten configuraciones inconsistentes.

REGLA 8 — CATÁLOGOS COMO VOCABULARIO CONTROLADO

Los catálogos del sistema definen el conjunto válido de:

capabilities
scopes
módulos
perfiles

No se permite:

definir valores fuera de los catálogos
interpretar valores no registrados
introducir enums implícitos en código

El contrato persistido debe utilizar exclusivamente valores válidos definidos en los catálogos.

REGLA 9 — CREATE CONTEXTUAL SIN CLASS-STRING OBLIGATORIO

Cuando una capacidad create sea contextual, la autorización debe poder resolverse por:

módulo
capacidad
contexto funcional

La presencia de subject class-string no es condición doctrinal para evaluar constraints en create contextual.

Si el contrato requiere contexto funcional, ese contexto es obligatorio.

Si el contexto falta o es inválido:

la autorización debe devolver no autorizado

sin depender de parches por módulo ni de lógica especial en controller o policy.

REGLA 10 — CONTRATO DE SEGURIDAD PROPIO POR MÓDULO

Todo módulo operativo real del sistema debe poseer un contrato de seguridad propio.

Ese contrato debe dejar definido, como mínimo:

universo visible
capacidades aplicables
scopes válidos
uso o no de constraints
reglas de create
reglas de update
reglas de delete
si consume o no contexto funcional obligatorio
si delega o no autorización en recurso padre

No se permite operar un módulo sin contrato explícito.

REGLA 11 — SECURITY COMO ÁRBITRO CONTRACTUAL

La capa central de seguridad actúa como árbitro de los contratos de seguridad modulares.

Esto implica que Security debe verificar, como mínimo:

si el módulo está habilitado
si la capacidad existe
si el scope es válido
si las constraints requeridas están presentes y son válidas
si el contexto funcional exigido por el módulo fue provisto
si el consumo solicitado respeta el contrato persistido del módulo

La capa central no reemplaza el contrato del módulo.

Lo arbitra.

REGLA 12 — INTEGRACIÓN MODULAR BAJO ARBITRAJE

Cuando un módulo consuma capacidades, relaciones o superficies de integración ofrecidas por otro módulo, ese consumo no puede bypassar la capa central de seguridad.

Toda integración válida entre módulos debe seguir respetando:

módulo habilitado
capacidad efectiva
scope
constraints
contexto funcional cuando corresponda

No se permite que una integración modular cree un canal paralelo de autorización.
<<END SECTION>>

REGLAS DEL LOGIN PÚBLICO

El login autentica identidad global.

No concede acceso operativo a un tenant por sí mismo.

Reglas:

el login resuelve únicamente autenticación de user

no existe acceso operativo válido sin membership activa

no existe acceso operativo válido sin tenant seleccionado

no debe permitirse autoingreso a tenants no activos o no seleccionados

no debe existir fallback automático a tenant si el usuario posee múltiples memberships

el login debe aplicar protección contra abuso e intentos masivos

el login no debe revelar información diferenciada sobre:

existencia del usuario
validez parcial de credenciales
estado interno

Reglas de continuidad de flujo:

cuando el login se origine desde un flujo previo válido (por ejemplo invitación o recuperación):

el sistema debe reanudar ese flujo original

no debe aplicarse la redirección estándar posterior al login

no debe perderse el contexto del flujo iniciado

Regla crítica:

el login no define contexto tenant

el contexto tenant solo puede definirse posteriormente mediante selección o resolución válida

Regla consolidada:

ninguna redirección posterior al login puede:

asignar tenant implícitamente
resolver acceso operativo sin validación completa
ignorar el flujo funcional previo si existe

LINKS SENSIBLES E INVITACIONES

Incluye:

Reglas generales:

todo link debe ser de uso único

todo link debe expirar

todo link debe invalidarse al consumirse

no debe permitirse reutilización

ningún link sensible puede derivar directamente en acceso operativo sin autenticación y pertenencia válida

Reglas de consistencia de identidad:

el flujo iniciado por un link debe completarse únicamente por la identidad correspondiente

no debe permitirse aceptar o completar un flujo con una identidad distinta a la originalmente vinculada

no debe permitirse transferencia implícita del acceso entre usuarios

no debe permitirse incorporación a tenant sin invitación válida

la aceptación de invitación debe generar o validar una membership consistente

no debe permitirse cerrar el flujo si la membership resultante no es válida

Reglas de alta de owner (owner_signup):

owner_signup puede originarse sin tenant previo

la finalización del flujo debe crear:

tenant válido
membership owner válida

no debe existir tenant sin owner válido

Reglas de recuperación de credenciales:

no debe permitir escalar privilegios

no debe alterar memberships existentes

no debe modificar pertenencia a tenant

Regla crítica consolidada:

ningún flujo basado en links puede:

crear acceso operativo directo
bypassar autenticación
bypassar validación de membership

<<SECTION: CAPA V2 DE PERMISOS>>
SECTION_VERSION: 00006

CAPA V2 DE PERMISOS

Componentes:

RolePermissionResolver
RecordScopeResolver
CapabilityCatalog
ModuleCatalog
TenantModuleAccess
Security

Capacidades:

view_any
view
create
update
delete

Scopes válidos:

tenant_all
own_assigned
limited

Scopes inválidos:

all
mine

Reglas:

view_any no es booleano
create puede depender de contexto
constraints son obligatorias
el acceso a un módulo depende previamente de su habilitación mediante TenantModuleAccess

Regla operativa consolidada:

Security es la puerta central de consumo de autorización y el árbitro contractual de los módulos del sistema.

Roles de la capa:

la base de datos define el contrato persistido
los contratos doctrinales de módulo definen el comportamiento esperado
los resolvers consumen ese contrato persistido
Security orquesta la resolución final y arbitra su validez
policies actúan como wrapper mínimo
controllers ejecutan la autorización

No se permite:

consumir resolvers directamente desde controllers
evaluar permisos fuera de Security

Aclaración doctrinal consolidada para create contextual:

cuando una capacidad create depende de constraints funcionales, la resolución válida debe poder hacerse por:

module + capability + context

sin exigir subject class-string como condición estructural de evaluación.

Si el contexto requerido falta o no cumple constraints:

allowed = false

La semántica correcta debe resolverse en la capa central y no mediante parches puntuales por módulo.

Aclaración adicional consolidada:

cuando un módulo publique capacidades reutilizables para otros módulos, la capa V2 no trata esa publicación como excepción arquitectónica

la considera una oferta modular consumible

pero su consumo sigue quedando bajo arbitraje normal de Security y del contrato de seguridad del módulo oferente

Aclaración sobre constraints funcionales efectivas:

cuando el contrato persistido incluya constraints que segmentan el universo funcional del módulo, su efecto debe mantenerse también sobre registros existentes y no solo sobre create

Caso ya validado:

orders con constraints.allowed_kinds

Aclaración sobre módulo base intrínseco:

si un módulo forma parte de la base intrínseca del sistema y no debe tratarse como módulo opcional ordinario, esa condición debe quedar explicitada en su contrato modular propio

no debe quedar librada a interpretación implícita al consumir TenantModuleAccess en otras capas del sistema
<<END SECTION>>

<<SECTION: FLUJO DE GESTIÓN Y CONSUMO DE PERMISOS>>
SECTION_VERSION: 00006

FLUJO DE GESTIÓN Y CONSUMO DE PERMISOS

superadmin habilita módulos
owner configura permisos
backend resuelve
UI consume

Flujo operativo consolidado:

base de datos → contrato persistido
↓
contrato de seguridad doctrinal del módulo
↓
resolvers → lectura del contrato
↓
Security → orquestación y arbitraje
↓
policy → wrapper mínimo
↓
controller → authorize(...) o Security::scope(...) o inspect(...)
↓
Blade → renderiza según resultado

Reglas:

la UI no define permisos
el backend define dataset
las acciones dependen de capability real

Regla crítica:

cuando la visibilidad del dataset depende de permisos, el controller debe usar Security::scope(...)

No se permite:

queries paralelas para listados
conteos fuera del scope efectivo
datasets auxiliares no alineados con permisos

Regla adicional consolidada para create contextual:

cuando create depende de contexto funcional obligatorio, el flujo correcto de consumo no pasa por Gate o policy abstracta.

Debe pasar por Security con contexto explícito.

En esos casos:

policy o Gate abstracta pueden actuar solo como fail-closed
controller y validación deben consumir Security con el contexto real requerido por el contrato

Ejemplo doctrinal del patrón correcto:

orders.create con kind explícito

Patrón doctrinal inválido:

Gate::allows('create', Order::class)

cuando create requiere constraints obligatorias.

Regla modular consolidada:

cuando un módulo consuma una oferta publicada por otro módulo, por ejemplo selector, create contextual, relación operativa o bloque derivado, el flujo correcto sigue siendo:

contrato del módulo oferente
↓
contrato persistido
↓
Security
↓
consumo permitido

No se permite:

consumir ofertas modulares saltando el arbitraje central
redefinir en el módulo consumidor lo que el módulo oferente ya cerró como contrato de seguridad

Regla adicional consolidada para módulos deshabilitados:

la ocultación visual de relaciones, links, campos o acciones no constituye validación suficiente

si una referencia operativa hacia otro módulo depende de que ese módulo esté habilitado para el tenant, el backend también debe rechazarla cuando el módulo destino no esté disponible

Regla adicional sobre segmentación efectiva:

si el contrato persistido incorpora constraints funcionales, el flujo de consumo debe preservarlas también al operar sobre registros reales existentes

no deben perderse al pasar de create a view, update o delete
<<END SECTION>>

<<SECTION: REGLAS OPERATIVAS POR MÓDULO>>
SECTION_VERSION: 00004

REGLAS OPERATIVAS POR MÓDULO

Cada módulo debe definir su propio contrato de seguridad.

Ese contrato debe dejar explícito, como mínimo:

naturaleza del módulo
universo visible
scopes válidos
uso de constraints
reglas de create
reglas de update
reglas de delete
si requiere contexto funcional
si delega autorización a un recurso padre
si publica o no ofertas de integración consumibles por otros módulos

No se permite:

interpretación implícita
uso genérico de limited
delegar lógica a UI
operar un módulo sin contrato explícito
consumir un módulo desde otro sin arbitraje central de seguridad

Regla adicional consolidada:

la publicación de capacidades de integración por parte de un módulo no altera su contrato de seguridad

lo amplía como superficie consumible

pero siempre bajo arbitraje de Security

Regla adicional para módulos base intrínsecos:

si un módulo ocupa una posición base transversal o intrínseca dentro del sistema, su contrato debe indicar explícitamente cómo se comporta respecto de la habilitación modular del tenant

esa condición no debe quedar repartida entre documentos ni inferirse desde relaciones indirectas

Regla adicional sobre entidades hijas delegadas:

cuando una entidad hija dependa funcionalmente de una cabecera y no posea autonomía modular real, su contrato de seguridad no debe independizarse artificialmente

debe delegar autorización en el recurso padre

esta delegación aplica a create, update y delete del recurso hijo salvo decisión doctrinal expresa en contrario

Regla adicional sobre relaciones reales:

el contrato del módulo debe describir las relaciones operativas reales tal como existen en código vigente

no deben inferirse foreign keys simétricas inexistentes solo por la existencia de navegación o integración entre módulos
<<END SECTION>>

<<SECTION: CONTRATO DE SEGURIDAD MODULAR>>
SECTION_VERSION: 00001

CONTRATO DE SEGURIDAD MODULAR

Todo módulo operativo del sistema debe poseer un contrato de seguridad propio.

Ese contrato expresa cómo debe comportarse el módulo dentro del marco general de seguridad del sistema.

Debe dejar explícito, como mínimo:

estado del contrato
naturaleza del módulo
universo visible
capacidades aplicables
scopes válidos
constraints requeridas si aplica
reglas de create
reglas de view
reglas de update
reglas de delete
si requiere contexto funcional obligatorio
si delega autorización en recurso padre
si publica ofertas modulares consumibles por otros módulos

Reglas:

el contrato de seguridad del módulo no reemplaza la capa central
no autoriza por sí mismo
no puede consumirse por fuera de Security
no puede contradecir los principios estructurales del documento

Su función es definir el comportamiento esperado del módulo.

La función de Security es arbitrar si ese comportamiento puede o no consumirse válidamente en cada caso.

Regla adicional consolidada:

si el módulo publica capacidades de integración para otros módulos, esa publicación forma parte de su superficie de consumo válida

pero no constituye una excepción de seguridad ni un canal paralelo de autorización
<<END SECTION>>

REGLAS DE IMPLEMENTACIÓN SEGURA

la base de datos es la única fuente de verdad del contrato de permisos V2

contrato doctrinal del módulo → define comportamiento esperado del módulo
resolver → resuelve el contrato persistido
Security → orquesta y arbitra la evaluación
policy → wrapper mínimo de consumo
controller → consume y ejecuta la autorización correspondiente
Blade → muestra solo el resultado permitido

Prohibido:

lógica paralela
evaluación abstracta
redefinir permisos
crear integraciones modulares que bypassen arbitraje central

Distinción:

view_any → dataset
view → registro

Regla crítica adicional:

cuando el dataset depende de permisos, debe resolverse mediante Security::scope(...)

Regla crítica adicional para create contextual:

cuando create depende de contexto funcional obligatorio, Gate o policy abstracta no constituyen el canal válido principal de autorización

debe consumirse Security con contexto explícito

Regla crítica adicional para UI relacional:

si un recurso relacionado existe pero el usuario no posee permiso real de vista sobre el módulo destino, la UI no debe renderizar un link operativo hacia ese recurso

puede mostrarse texto o referencia no navegable cuando resulte útil para lectura, pero no navegación efectiva

si el módulo destino está deshabilitado para el tenant, la UI no debe renderizar links, acciones ni bloques residuales hacia ese módulo aunque exista la relación cargada

las integraciones visuales entre módulos deben consumir resultados arbitrados por backend y no reconstruir autorización local en Blade
