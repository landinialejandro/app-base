<<SECTION: RELACIONES CLAVE DEL ECOSISTEMA>>
SECTION_VERSION: 00007

RELACIONES CLAVE DEL ECOSISTEMA

projects agrupa tasks sin absorber su autonomía

tasks puede originar orders

appointments puede originar orders

assets contextualiza orders, appointments y documents

orders puede derivar documents

orders contiene detalle operativo mediante order_items

products alimenta order_items y document_items

products sostiene la base operativa actual de inventory V1 mediante inventory_movements

parties se relaciona transversalmente con múltiples módulos

attachments puede integrarse transversalmente sobre múltiples recursos

Reglas:

estas relaciones expresan lectura conceptual prioritaria del ecosistema actual

task y appointment actúan como orígenes funcionales legítimos hacia orders sin degradar la autonomía conceptual del módulo orders

la relación tasks → orders posee además persistencia estructural desde orders.task_id

la relación appointments → orders opera como vínculo contextual y operativo real desde appointments.order_id sin requerir orders.appointment_id

order_items debe leerse como detalle dependiente de orders y no como módulo autónomo paralelo

la relación entre order_items y product es opcional y forma parte del detalle operativo del padre

products ya no debe leerse frente a inventory V1 como promesa futura difusa

la relación products → inventory_movements expresa una capacidad operativa actual, validada y todavía no cerrada como módulo autónomo independiente

orders no solo ocupa posición central por su entidad principal

también publica oferta de integración consumible por otros módulos del ecosistema

esa oferta puede incluir arbitraje backend propio y representación visual modular sin dejar de pertenecer contractualmente al módulo orders

Aclaración relacional consolidada adicional:

parties no solo participa como relación transversal pasiva

debe leerse también como módulo base del dominio que puede publicar oferta reusable propia para el ecosistema

caso inmediato previsto:

linked-party-action como capacidad oficial de integración para arbitrar apertura navegable, referencia readonly o ausencia de acción operable según soporte modular, permiso efectivo y existencia de party relacionada

Aclaración adicional consolidada para assets:

assets no solo participa como entidad relacionada consumida por otros módulos

también puede publicar oferta reusable propia para resolver navegación relacional homogénea dentro del ecosistema

Caso consolidado actual:

linked-asset-action como capacidad reusable oficial del módulo assets

no invalidan otras relaciones técnicas menores ya existentes o futuras

<<SECTION: FUNCIÓN DE CADA UNA>>
FUNCIÓN DE CADA UNA

users

Representa la identidad global del usuario en todo el sistema.

Un user puede pertenecer a múltiples tenants.

No pertenece a un tenant específico.

tenants

Representa cada empresa o espacio de trabajo dentro del SaaS.

Cada tenant contiene sus propios datos operativos.

Además de datos generales, puede contener configuración funcional en:

tenants.settings

Esto permite persistir contexto operativo del tenant cuando todavía no justifica entidad propia.

memberships

Representa la pertenencia de un user a un tenant.

Define acceso real por empresa.

Es la entidad correcta para resolver:

a qué empresa pertenece un usuario
qué estado tiene dentro de esa empresa
qué roles tiene dentro de esa empresa
qué perfil funcional persistido tiene dentro de esa empresa

signup_requests

Representa solicitudes iniciales de alta o interés desde la landing pública.

No pertenece al dominio operativo interno del tenant.

invitations

Representa invitaciones dentro del flujo de acceso y onboarding del sistema.

Puede operar en dos modos:

owner_signup
member_invite

Cuando type = owner_signup puede existir sin tenant asociado.

Cuando type = member_invite sí se vincula operativamente a un tenant existente.

Por esta razón, no debe tratarse como una entidad tenant-scoped pura del negocio.
<<END SECTION>>

<<SECTION: ACLARACIÓN SOBRE TENANT-SCOPED>>
ACLARACIÓN SOBRE TENANT-SCOPED

El trait TenantScoped debe usarse en entidades cuyo ciclo de vida pertenece completamente a un tenant.

Ejemplos de uso correcto:

Product
Order
Document
Branch
Party
Project
Task
Appointment
Asset
Attachment

Aclaración sobre Invitation

Invitation no debe tomarse como ejemplo simple de entidad tenant-scoped pura, porque su comportamiento depende del tipo de invitación.

Cuando type = owner_signup puede operar sin tenant asociado.

Cuando type = member_invite sí se vincula operativamente a un tenant existente.

No debe usarse TenantScoped en entidades del sistema como:

User
Tenant
Membership
SignupRequest

ORDERS

Tipo conceptual:

módulo autónomo de núcleo operativo fuerte

Representa operación interna principal del tenant.

Relaciones consumidas:

tenant
party opcional
asset opcional
task opcional mediante orders.task_id

Relaciones ofrecidas:

historial operativo
seguimiento de trabajo

Superficie contractual conceptual posible:

selector de orden
tabla embebida reusable
summary de orden
acción create contextual
bridge hacia documents
lookup mínimo
acción enlazada reusable

Reglas:

orders debe tratarse como entidad central operativa del ecosistema actual

puede recibir contexto desde tasks y appointments sin perder autonomía conceptual

tasks y appointments pueden originar orders, pero no la absorben como subrecurso ni redefinen su identidad funcional

su núcleo no debe reabrirse por cada nueva integración externa

actúa como hub operativo relevante dentro del ecosistema actual

su create requiere contexto funcional válido mediante kind y no debe interpretarse como create abstracto genérico

Aclaración contractual consolidada:

la oferta de integración de orders ya no debe leerse solo como posibilidad abstracta o futura

incluye una capacidad reusable concreta para arbitrar apertura de orden existente, create contextual, estado readonly o requisito faltante según el contexto recibido

esa capacidad pertenece al módulo orders como parte de su oferta de integración

su resolución válida puede combinar backend propio del módulo y bloque visual modular consumible por otros módulos sin perder identidad contractual

Caso consolidado actual:

linked-order-action forma parte de esa oferta reusable oficial

Reglas de consistencia:

tasks y appointments constituyen orígenes funcionales válidos hacia orders dentro del ecosistema actual

<<SECTION: MEMBERSHIPS, OWNER Y ROLES>>
MEMBERSHIPS, OWNER Y ROLES

La pertenencia operativa de un usuario a un tenant se resuelve por memberships.

Reglas actuales:

memberships.is_owner expresa la condición estructural de owner

owner no debe modelarse como un rol asignable ordinario dentro del flujo normal de edición

owner es único por tenant

la pivote membership_role se utiliza para roles delegables del tenant

admin es el rol delegable de gestión amplia

administrator es el rol delegable administrativo u oficina

operator es el rol operativo base por defecto

sales permanece como rol operativo específico

una membership activa no-owner no debe quedar sin rol

puede existir una membership owner sin registros en membership_role y eso no constituye un error del modelo

como evolución activa de la autorización, membership podrá incorporar además un profile_slug para refinar capacidades funcionales sin multiplicar roles base

La jerarquía efectiva debe entenderse así:

owner como condición estructural
roles delegables como capa operativa base
perfil funcional como especialización de trabajo
permisos concretos como capa de autorización

La complejidad funcional futura no debe resolverse creando nuevos roles base fuera de este conjunto cerrado.
<<END SECTION>>

REGLAS DE LECTURA

Las relaciones aquí expresadas son conceptuales y funcionales.

No implican automáticamente:

foreign keys obligatorias
pantallas existentes
botones visibles
navegación activa
permisos concedidos

Regla adicional consolidada:

cuando una relación se describa aquí como origen funcional, vínculo contextual o flujo operativo real, eso no obliga a una foreign key simétrica en ambos sentidos

puede existir relación estructural persistida desde un solo lado del vínculo y aun así existir integración válida y navegación real entre módulos

appointments.order_id expresa vínculo operativo real con orders sin requerir orders.appointment_id

CLASIFICACIÓN CONCEPTUAL DE MÓDULOS

No todos los módulos del sistema cumplen el mismo rol conceptual.

Tipos principales:

módulo base del dominio
módulo transversal
módulo autónomo
módulo agrupador
módulo derivado

MÓDULO BASE DEL DOMINIO

Caso actual:

parties

Representa la base relacional principal del negocio.

MÓDULO TRANSVERSAL

Caso actual:

attachments

Representa una capacidad reusable embebida o delegada al recurso padre.

MÓDULO AUTÓNOMO

Casos actuales relevantes:

tasks
appointments
assets
products
orders

Poseen identidad operativa propia.

MÓDULO AGRUPADOR

Caso actual:

projects

Agrupa y da lectura agregada sobre tasks.

MÓDULO DERIVADO

Caso actual:

documents respecto de orders

Formaliza o prolonga documentalmente una operación más nuclear.

Regla:

esta clasificación debe utilizarse para decidir integraciones y evitar jerarquías falsas

CONTRATO MODULAR CONCEPTUAL

Cada módulo principal del sistema debe poder describirse conceptualmente mediante:

núcleo
relaciones que consume
relaciones que ofrece
superficie contractual visible

NÚCLEO

Lo que el módulo resuelve por sí mismo.

RELACIONES QUE CONSUME

Lo que toma del ecosistema para ampliar su operación.

RELACIONES QUE OFRECE

Lo que deja disponible a otros módulos.

SUPERFICIE CONTRACTUAL VISIBLE

Piezas conceptuales reutilizables que otros módulos podrían consumir.

Ejemplos:

selector
tabla embebida
summary reusable
acción contextual
lookup mínimo

Regla:

esta sección no define implementación técnica obligatoria

define cómo pensar el crecimiento del ecosistema modular
<<END SECTION>>

<<SECTION: REGLA GENERAL DE CRECIMIENTO>>
REGLA GENERAL DE CRECIMIENTO

Antes de crear una nueva entidad, verificar:

si el caso puede resolverse con una entidad ya existente
si parties puede absorber el nuevo caso
si la relación puede modelarse como extensión y no como nueva tabla
si la necesidad corresponde realmente al dominio o solo a una vista específica
<<END SECTION>>
