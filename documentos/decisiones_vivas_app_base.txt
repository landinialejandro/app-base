DECISIONES VIGENTES

No abrir módulo users por necesidades de colaboradores internos.

La resolución vigente es:

users = identidad global
memberships = pertenencia operativa
raíces tenant reusables para datasets internos

Inventory debe leerse como frente funcional real y no como efecto lateral de products.

Orders conserva centralidad operativa pero puede hostear capacidades de otros módulos sin absorber su lógica.

La evolución reusable correcta prioriza surfaces oficiales del módulo oferente antes que soluciones locales del consumidor.

Security permanece como kernel central de autorización efectiva.

Las show pages deben converger hacia slots explícitos y montaje coordinado.

NavigationTrail resuelve contexto externo.

return_tab complementa contexto interno del host cuando aplica.

No backward patching como cierre estable de migraciones.

Para parties, la constraint funcional de seguridad es allowed_party_roles.

Lectura vigente:

allowed_party_roles se evalúa contra party_roles.role.

allowed_kinds no aplica a parties.

Motivo:

parties.kind representa únicamente tipo de contacto y no relación comercial ni vínculo interno.

La gestión de contactos con relación employee se autoriza mediante PartyEmployeeContactAuthorization.

Regla vigente:

employee es relación sensible.

No se protege solo en UI.

Debe bloquearse en backend.

No se notifica al owner por cada intento individual de manipulación.

Criterio acordado:

primero registrar o auditar eventos sensibles
notificar solo ante patrón, reiteración o severidad

Este criterio no implica implementación vigente de auditoría o notificación.

OperationalActivity queda decidido como capacidad transversal tenant-scoped de trazabilidad operativa.

Lectura vigente:

no es CRUD ordinario
no es navegación principal
no pertenece a parties, tasks, orders, documents ni appointments
no autoriza operaciones
no reemplaza Security
no reemplaza policies
no valida la operación original
no debe bloquear el flujo principal si falla el registro

Regla vigente:

el módulo dueño autoriza, valida y ejecuta su operación.

Luego emite un evento operativo liviano.

OperationalActivity registra si corresponde.

La escritura de OperationalActivity es system-owned y append-only.

No se implementa create manual desde UI.

No se implementa edit.

No se implementa update.

No se implementa delete.

No se implementa destroy.

No se implementa carga manual por formularios.

Decisión vigente sobre lectura:

la autorización de lectura del log todavía no está definida.

No se debe asumir acceso de lectura por existir operational_activities.

Lecturas futuras posibles:

vista general desde perfil de empresa
actividad por colaborador desde parties.show para employee con membership activa vinculada

Project Lab es la herramienta operativa local vigente para aplicar, auditar y validar intervenciones técnicas.

Regla vigente:

todo snippet destinado a Project Lab debe respetar el contrato real de la herramienta.

FILE corresponde a archivo completo.

TARGET corresponde a operación parcial.

No se deben mezclar FILE y TARGET.

No se deben entregar encabezados ambiguos que obliguen al usuario a corregir el formato.

Ante falta de datos reales:

NO SE ASUME
NO SE SUPONE
