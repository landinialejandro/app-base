CHECKLIST OFICIAL PARA CREAR MÓDULOS NUEVOS – app-base

OBJETIVO

Este documento define el procedimiento estándar para crear nuevos módulos dentro del proyecto app-base.

Su finalidad es:

mantener consistencia entre módulos
garantizar compatibilidad multi-tenant
evitar errores estructurales
acelerar el desarrollo
mantener coherencia en todo el sistema

Este checklist aplica a todos los módulos CRUD del sistema.

EVALUACIÓN DE LA ENTIDAD (ANTES DE CREAR EL MÓDULO)

Antes de crear una nueva entidad verificar:

si el caso puede resolverse utilizando la entidad parties
si corresponde a una extensión funcional de parties
si realmente es una entidad nueva del dominio del sistema

Recordar:

parties es la entidad base del negocio.

Si la entidad no es necesaria en esta etapa:

registrarla en TODO-DEL-PROYECTO.

CREAR MIGRACIÓN

Crear la migración correspondiente.

Ejemplo:

php artisan make:migration create_products_table

Reglas obligatorias para entidades del negocio:

toda tabla debe incluir:

tenant_id
timestamps
softDeletes

Ejemplo mínimo:

$table->uuid('tenant_id')->index();
$table->string('name');
$table->timestamps();
$table->softDeletes();

CREAR MODELO

Crear modelo:

php artisan make:model Product

El modelo debe incluir los traits:

TenantScoped
ResolvesTenantRouteBinding
SoftDeletes

Ejemplo:

use App\Models\Concerns\TenantScoped;
use App\Models\Concerns\ResolvesTenantRouteBinding;
use Illuminate\Database\Eloquent\SoftDeletes;

DEFINIR CAMPOS FILLABLE

Definir los campos permitidos.

Ejemplo:

protected $fillable = [
'tenant_id',
'name',
'description'
];

CREAR CONTROLLER

Crear controlador tipo resource.

php artisan make:controller ProductController --resource

Métodos esperados:

index
create
store
show
edit
update
destroy

DEFINIR RUTAS

Registrar rutas en:

routes/web.php

Ejemplo:

Route::resource('products', ProductController::class);

Las rutas deben ejecutarse dentro de los middlewares:

auth
tenant

CREAR VISTAS

Crear carpeta del módulo:

resources/views/products

Vistas mínimas:

index.blade.php
create.blade.php
edit.blade.php
show.blade.php
_form.blade.php

ESTRUCTURA VISUAL OBLIGATORIA

Todas las páginas internas deben utilizar:

<x-page> <x-breadcrumb /> <x-page-header /> <x-card /> </x-page>

PATRÓN DE LISTADOS

Las vistas index deben utilizar:

page.list-page
card.list-card
table-wrap.list-scroll

Esto garantiza:

scroll interno
layout estable
coherencia visual

FORMULARIOS

Los formularios deben reutilizar el parcial:

_form.blade.php

Este archivo debe ser utilizado por:

create
edit

VALIDACIONES

Las validaciones deben definirse inicialmente en el controlador.

Ejemplo:

$request->validate([
'name' => ['required','string','max:255']
]);

PRUEBAS FUNCIONALES MÍNIMAS

Verificar funcionamiento del módulo:

crear
editar
eliminar
listar

VERIFICACIÓN MULTI-TENANT

Confirmar que:

un tenant no puede acceder a datos de otro tenant.

Este comportamiento es gestionado por:

TenantScoped
ResolvesTenantRouteBinding

DATOS DE PRUEBA (OPCIONAL)

Si el módulo es relevante:

crear factory.

Ejemplo:

php artisan make:factory ProductFactory

y agregar datos al DemoSeeder.

ACTUALIZAR DOCUMENTACIÓN DEL PROYECTO

Si se crea una entidad nueva actualizar:

MAPA-DE-ENTIDADES
TODO-DEL-PROYECTO

REGLA DE COHERENCIA

Todo módulo nuevo debe seguir el mismo patrón que:

projects
parties

REGLA FINAL

Si una decisión afecta a toda la arquitectura:

no improvisar.

Registrar la decisión en:

CONVENCIONES-DEL-PROYECTO.
