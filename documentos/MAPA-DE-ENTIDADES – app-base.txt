MAPA-DE-ENTIDADES – app-base

Objetivo

Este documento define el mapa conceptual de entidades del proyecto app-base.

Su finalidad es:

visualizar la estructura general del sistema
evitar duplicación de modelos
orientar la creación de nuevos módulos
mantener coherencia en el crecimiento del SaaS

Criterio general de modelado

El sistema distingue entre:

entidades del sistema
entidades del negocio
entidades operativas
entidades documentales
entidades futuras de soporte

Reglas principales:

users es entidad del sistema
parties es la entidad base del negocio
antes de crear una entidad nueva evaluar si puede resolverse con una entidad existente

Entidades del sistema

Estas entidades forman el núcleo técnico del SaaS.

SIGNUP_REQUESTS

Entidad que representa la solicitud pública inicial para crear una nueva empresa en el sistema.

Se utiliza como etapa previa de validación manual o comercial antes de crear un tenant.

Campos principales:

id
requested_name
requested_email
company_name
phone_whatsapp
status
review_notes
approved_at
rejected_at
approved_by
created_at
updated_at

Notas:

No crea por sí sola ni un user ni un tenant.

Representa un proceso previo de validación administrativa o comercial.

Puede dar origen a una o más invitations.

Relación de flujo:

signup_requests aprobada
↓
genera invitation owner_signup
↓
owner acepta invitación
↓
se crea tenant y membership owner

INVITATIONS

Entidad que representa una invitación formal de ingreso o vinculación al sistema.

Es el mecanismo central de onboarding y alta de usuarios relevantes dentro del sistema.

Campos principales:

id
tenant_id (nullable)
type
status
email
token
signup_request_id (nullable)
invited_by_user_id (nullable)
sent_at
expires_at
accepted_at
accepted_ip
user_agent
meta
created_at
updated_at

Relaciones:

belongsTo tenant (nullable)
belongsTo signupRequest (nullable)
belongsTo invitedBy user (nullable)

Dominios cerrados:

type

owner_signup
member_invite

status

pending
accepted
expired
cancelled

Interpretación de campos del flujo:

sent_at
momento en que el superadmin marca la invitación como enviada al owner

accepted_at
momento en que el owner acepta la invitación

Notas importantes:

tenant_id puede ser NULL cuando type = owner_signup.

owner_signup se utiliza para completar el alta de una nueva empresa aprobada desde signup_requests.

member_invite se utiliza para sumar un usuario a un tenant existente.

No existe registro público libre en el sistema; el alta de usuarios relevantes parte desde invitations.

TENANTS

Entidad que representa una organización dentro del sistema.

Define el aislamiento principal del modelo multi-tenant.

Campos principales:

id
name
slug
settings
created_at
updated_at

Notas importantes:

id utiliza UUID (char(36)) como primary key.

slug es único.

Todas las relaciones hacia tenants deben utilizar CHAR(36).

Esto evita incompatibilidades con migraciones que utilicen bigint por defecto.

settings se utiliza para configuración básica editable de empresa cuando no justifica entidad propia.

Relaciones conceptuales relevantes:

tenants → memberships
tenants → document_sequences

Las invitaciones de miembros se relacionan operativamente con el tenant mediante invitations.tenant_id.

USERS

Entidad que representa la identidad global del sistema.

Responsabilidades:

autenticación
identidad única del usuario
acceso al sistema

Campo global relevante:

is_superadmin

Define acceso administrativo global al sistema.
No depende de memberships ni de pertenencia a tenants.

Notas:

email es único global.

Un mismo user puede pertenecer a muchos tenants.

Un mismo user puede ser owner de muchos tenants.

La pertenencia real no se resuelve desde users, sino desde memberships.

MEMBERSHIPS

Entidad que representa la pertenencia real entre un user y un tenant.

Es la tabla puente central del modelo multi-tenant.

Campos principales:

id
tenant_id
user_id
status
is_owner
joined_at
blocked_at
blocked_reason
created_at
updated_at

Notas:

Define si un usuario pertenece o no a una empresa.

Permite múltiples memberships para un mismo user en distintos tenants.

Es la entidad de acceso por tenant.

El acceso operativo debe resolverse a través de membership y su estado.

membership_role

Tabla pivote para asignación de roles por membership.

Campos relevantes conceptuales:

membership_id
role_id
branch_id nullable
created_at
updated_at

Notas:

el proyecto ya permite asignación de roles por membership

branch_id puede utilizarse null como asignación general al tenant

si en el futuro se implementa gestión por sucursal, branch_id permitirá especialización más fina del alcance del rol

Regla sobre TenantScoped

El trait TenantScoped se utiliza únicamente en entidades cuyo ciclo de vida pertenece completamente a un tenant.

Ejemplos de uso correcto:

Product
Order
Document
Branch
Role
Invitation
Party
Project
Task
Asset

No debe utilizarse en entidades del sistema como:

User
Tenant
Membership
SignupRequest

Entidades operativas

PROJECTS

Agrupa trabajos o iniciativas.

Relaciones principales:

tenant → projects
projects → tasks

Nota:

projects no posee catálogo funcional propio por el momento.

TASKS

Unidad operativa de trabajo.

Representa actividades concretas dentro del sistema.

Relaciones principales:

tenant → tasks
project → tasks
party → tasks
assigned user → tasks
order → task (relación operativa opcional 1 a 1 resuelta desde orders.task_id)

Dominios funcionales:

status
priority

Notas operativas:

toda tarea nace asignada a un usuario del tenant

assigned_user_id referencia al usuario responsable operativo de la tarea

una tarea puede derivar en una única orden, normalmente de servicio

PRODUCTS

Representa productos, artículos o insumos.

Relaciones posibles:

tenant → products
products → orders
products → inventory
products → documents

Dominio funcional:

kind

ASSETS

Entidad base reusable para representar una unidad identificable y operable del negocio.

Campos base relevantes:

id
tenant_id
party_id
kind
relationship_type
name
internal_code
status
notes
created_at
updated_at
deleted_at

Notas:

representa un activo operativo genérico, no contable ni patrimonial

permite especialización futura por tablas complementarias

ejemplo futuro:

asset_motorcycles
asset_machinery

party_id representa el vínculo actual principal del activo con un contacto o cliente

el historial de cambios de titularidad o vinculación no se resuelve aún en v1

Relaciones principales:

tenant → assets
party → assets
assets → orders
assets → documents

DOCUMENT_SEQUENCES

Entidad que controla la numeración automática de órdenes y documentos dentro de cada tenant.

Finalidad:

gestionar secuencias independientes por tipo documental o tipo de orden.

Campos relevantes:

tenant_id
branch_id nullable
doc_type
prefix
point_of_sale
padding
next_number

Interpretación:

la secuencia se define por tenant + tipo documental + punto de venta.

prefix identifica el tipo documental o tipo de orden.

point_of_sale identifica el punto de venta base de la empresa.

padding define la cantidad de dígitos del número correlativo.

next_number guarda el próximo número a emitir.

Uso actual:

orders utiliza secuencias del tipo:

order.sale
order.purchase
order.service

documents utiliza secuencias del tipo:

quote
delivery_note
invoice
work_order
receipt
credit_note

Relaciones:

tenant → document_sequences

ORDERS

Representa pedidos u órdenes comerciales.

Características:

representan intención comercial
los totales se calculan dinámicamente a partir de los ítems

Relaciones principales:

tenant → orders
orders → parties
orders → assets
orders → products
orders → documents
orders → task (nullable)

Campos relevantes adicionales:

number
sequence_prefix
point_of_sale
sequence_number

Aclaración:

la orden se numera automáticamente al guardarse.

el número no se ingresa manualmente.

el tipo de orden no debe modificarse una vez numerada.

Dominios funcionales:

kind
status

Notas operativas:

belongsTo party

belongsTo asset (nullable)

belongsTo task (nullable)

una orden vinculada a un activo debe mantener consistencia con el contacto del activo

cuando una orden nace desde una tarea, la relación operativa 1 a 1 se expresa mediante orders.task_id

ORDER_ITEMS

Detalle de productos dentro de una orden.

Relaciones principales:

order → order_items
product → order_items

Características:

heredan tenant desde la cabecera
no exponen tenant_id en fillable

Dominio funcional:

kind

Regla importante:

kind reutiliza el mismo dominio definido en products.

Entidades documentales

DOCUMENTS

Entidad genérica para documentos comerciales u operativos.

Ejemplos:

factura
remito
presupuesto
orden de trabajo
nota de crédito
recibo

Características:

representan documentos formales
los totales se persisten en la cabecera

Relaciones principales:

tenant → documents
documents → parties
documents → assets
documents → products
documents → projects
documents → orders

Campos relevantes adicionales:

number
sequence_prefix
point_of_sale
sequence_number

Aclaración:

el documento se numera automáticamente al guardarse.

el número no se ingresa manualmente.

el tipo de documento no debe modificarse una vez numerado.

Relación con secuencias:

document.kind → document_sequences.doc_type

Dominios funcionales:

kind
status

Notas operativas:

belongsTo party

belongsTo order (nullable)

belongsTo asset (nullable)

cuando un documento se genera desde una orden, hereda el activo si la orden lo tiene

DOCUMENT_ITEMS

Detalle de líneas dentro de un documento.

Relaciones principales:

document → document_items
product → document_items

Características:

heredan tenant desde cabecera
eliminación de cabecera elimina ítems

Dominio funcional:

kind

Regla:

kind reutiliza el mismo dominio de products.
