<<SECTION: OBJETIVO>>
SECTION_VERSION: 00002

OBJETIVO

Este documento define el checklist obligatorio para crear, incorporar o reconvertir módulos dentro de app-base.

Su función es:

reducir omisiones
forzar consistencia arquitectónica
alinear seguridad, dominio y UI
evitar módulos incompletos
mantener coherencia entre implementaciones nuevas y existentes

No es una guía opcional.

Es un control mínimo obligatorio antes de considerar un módulo como integrado al sistema.

Aclaración doctrinal consolidada:

este checklist aplica tanto a módulos nuevos como a reconversiones profundas de módulos existentes cuando cambie su contrato funcional, contractual o de seguridad
<<END SECTION>>

<<SECTION: ESTRUCTURA GENERAL>>
ESTRUCTURA GENERAL

Verificar:

modelo creado
migración creada
controller creado
rutas declaradas explícitamente
vistas base creadas
enlace en navegación si corresponde
validaciones mínimas implementadas
tenant resuelto correctamente
autorización revisada
documentación actualizada
<<END SECTION>>

<<SECTION: REGLA GENERAL>>
SECTION_VERSION: 00002

REGLA GENERAL

Ningún módulo debe considerarse terminado por tener tablas, rutas o CRUD funcionando.

Un módulo solo puede considerarse integrado cuando supera este checklist en forma coherente.

Orden correcto:

definir contrato conceptual
definir contrato de seguridad
definir relaciones reales necesarias
definir superficie de integración
implementar
auditar
ajustar documentación afectada

Nunca al revés.

Regla adicional consolidada:

si una integración nueva requiere reabrir arbitrariamente el núcleo de otro módulo, primero debe revisarse si la necesidad puede resolverse consumiendo una capacidad ya publicada por ese módulo o evolucionando su superficie contractual propia
<<END SECTION>>

<<SECTION: BLOQUE 0 — IDENTIDAD DEL MÓDULO>>
SECTION_VERSION: 00001

BLOQUE 0 — IDENTIDAD DEL MÓDULO

Antes de cualquier decisión técnica debe quedar definida la identidad del módulo.

Validar:

nombre del módulo
problema real que resuelve
naturaleza principal del módulo
posición conceptual en el ecosistema
si es módulo nuevo o reconversión de uno existente

Clasificación conceptual posible:

módulo base del dominio
módulo transversal
módulo autónomo
módulo agrupador
módulo derivado

Preguntas obligatorias:

este módulo tiene identidad propia o es solo una entidad hija
es realmente un módulo o una evolución de otro ya existente
debe vivir por sí mismo o principalmente embebido
qué rol ocupa dentro del ecosistema general

Si estas respuestas no están claras, no debe avanzarse a implementación.
<<END SECTION>>

<<SECTION: BLOQUE 1 — CONTRATO MODULAR>>
SECTION_VERSION: 00004

BLOQUE 1 — CONTRATO MODULAR

Todo módulo debe poder describirse mediante un contrato modular mínimo.

Validar:

núcleo definido
contrato de entrada definido
contrato de salida definido
oferta de integración definida o declarada como futura
límites claros del módulo

NÚCLEO

Debe quedar claro qué resuelve por sí mismo.

CONTRATO DE ENTRADA

Debe quedar claro qué puede recibir desde sistema u otros módulos.

CONTRATO DE SALIDA

Debe quedar claro qué devuelve o expone.

OFERTA DE INTEGRACIÓN

Debe quedar claro qué capacidades reutilizables publica o prevé publicar.

Cuando la capacidad reusable sea compleja, debe quedar claro si el módulo publica además:

payload arbitrado desde backend
bloque visual consumidor
ubicación modular coherente con su identidad contractual

Preguntas obligatorias:

qué pertenece realmente al núcleo
qué integraciones deberían vivir fuera del núcleo
qué debería consumir otro módulo sin tocar implementación interna
qué puede publicarse hoy aunque todavía no tenga consumidor real
la capacidad reusable se publica como unidad completa o solo como vista aislada
cuál es la superficie principal real del módulo
si existe una superficie secundaria relevante que no debe leerse como base principal
si la integración reusable requerida es pesada o compacta

Regla adicional de cierre:

si la capacidad reusable ya fue cerrada doctrinalmente por el módulo oferente, no debe quedar ninguna superficie secundaria relevante reconstruyendo localmente sus estados operativos

antes de considerar cerrado el contrato modular, revisar también tablas, partials, formularios embebidos y demás superficies derivadas que consuman esa integración

la definición del contrato modular también debe dejar explícito si el módulo publica:

embebido reusable oficial para relaciones fuertes

o acción compacta oficial para relaciones puntuales

<<END SECTION>>

<<SECTION: BLOQUE 2 — SEGURIDAD OBLIGATORIA>>
SECTION_VERSION: 00002

BLOQUE 2 — SEGURIDAD OBLIGATORIA

Ningún módulo operativo puede entrar al sistema sin contrato de seguridad explícito.

Validar:

módulo registrado
capacidades válidas
scopes válidos
constraints necesarias
dataset visible
reglas de create
reglas de update
reglas de delete
si delega autorización al padre
si requiere contexto funcional obligatorio

Validar además:

consumo correcto mediante Security
ausencia de lógica paralela en controller
ausencia de inferencias en Blade
coherencia entre view_any y view

Preguntas obligatorias:

cómo autoriza realmente este módulo
qué rompe si falta contexto
qué acciones dependen de constraints
qué dataset debe devolver Security::scope(...)

Regla adicional consolidada:

si el módulo utiliza constraints funcionales, validar explícitamente que arbitren no solo create sino también view, update y delete cuando corresponda al contrato del módulo

no deben tratarse como metadata decorativa ni como validación parcial
<<END SECTION>>

<<SECTION: BLOQUE 3 — RELACIONES E INTEGRACIONES>>
SECTION_VERSION: 00004

BLOQUE 3 — RELACIONES E INTEGRACIONES

Toda relación con otros módulos debe justificarse por necesidad real del dominio.

Validar:

qué módulos consume
qué módulos lo consumen
si la relación depende de módulo habilitado
si existe ya una capacidad reusable equivalente
si la integración respeta contratos existentes
si evita duplicaciones de selectores, tablas o summaries
si la capacidad reusable ajena se consume como unidad completa y no solo como vista suelta
si la relación corresponde a embebido fuerte o a integración compacta
si el consumidor replica el patrón visual ya consolidado del ecosistema

Preguntas obligatorias:

esta relación agrega valor real o solo comodidad técnica
puede resolverse con una capacidad ya publicada
si falta una capacidad reusable, pertenece al módulo oferente
la integración está reabriendo un núcleo ajeno sin necesidad
el arbitraje de estados sigue centralizado en el módulo oferente o quedó disperso entre consumidores

Regla:

ninguna integración válida debe bypassar seguridad ni crear contratos paralelos.

Regla adicional de cierre:

no considerar homogénea una integración mientras existan superficies secundarias del consumidor que sigan resolviendo localmente una relación, acción o flujo ya cerrado por el módulo oferente

si el módulo consumidor ya posee un patrón visual consolidado para relaciones equivalentes, la nueva integración debe tender a reutilizar ese patrón y no introducir gramática visual nueva sin causa real

<<END SECTION>>

<<SECTION: BLOQUE 4 — UI Y EXPERIENCIA OPERATIVA>>
SECTION_VERSION: 00003

BLOQUE 4 — UI Y EXPERIENCIA OPERATIVA

El módulo debe integrarse visualmente al sistema de forma coherente.

Validar:

index consistente
show consistente
formularios consistentes
acciones visibles correctas
mensajes claros
embebidos consistentes
toolbars alineadas
componentes reusables aprovechados

Si publica oferta de integración visual, validar:

selector usable
tabla embebida reusable
summary legible
acciones contextuales claras
consumo visual homogéneo entre módulos

Preguntas obligatorias:

la UI refleja el contrato real del módulo
hay duplicaciones visuales evitables
el módulo se siente parte del sistema o una excepción aislada
la capacidad reusable visual vive en una ubicación coherente con el módulo oferente o quedó globalizada sin necesidad

Regla adicional de cierre:

la UI no debe considerarse alineada solo porque la vista principal consuma la capacidad reusable correcta

también deben auditarse tablas, partials, formularios embebidos y demás superficies secundarias que puedan exponer esa misma integración
<<END SECTION>>

<<SECTION: BLOQUE 5 — DOCUMENTACIÓN Y CIERRE>>
SECTION_VERSION: 00001

BLOQUE 5 — DOCUMENTACIÓN Y CIERRE

Ningún módulo debe cerrarse sin impacto documental revisado.

Validar:

contexto_fijo actualizado si cambia arquitectura
convenciones actualizadas si cambia doctrina
seguridad actualizada si cambia contrato de acceso
mapa_entidades actualizado si cambia ecosistema
mapa_navegacion actualizado si cambia recorridos reales
base_visual actualizada si cambia patrón visual
todo actualizado con pendientes reales

Validar además:

tests manuales o técnicos realizados
contradicciones documentales inexistentes
estado del módulo declarado

Estados posibles:

CONTRATO CERRADO
EN REVISIÓN

Regla:

solo módulos con contrato cerrado deben usarse como patrón para nuevos desarrollos
<<END SECTION>>

CHECKLIST MÍNIMO EXISTENTE

Además de los bloques obligatorios anteriores, verificar como mínimo:

<<SECTION: CRITERIO FINAL DE APROBACIÓN>>
SECTION_VERSION: 00002

CRITERIO FINAL DE APROBACIÓN

Un módulo no se aprueba por cantidad de código implementado.

Se aprueba cuando:

su identidad es clara
su núcleo está delimitado
sus contratos son coherentes
su seguridad está cerrada
sus integraciones son sanas
su UI es consistente
su documentación quedó alineada

Preguntas finales obligatorias:

este módulo mejora el ecosistema o agrega ruido
podrá crecer sin romper otros módulos
puede integrarse sin refactors innecesarios futuros
podría usarse como patrón de calidad para el próximo módulo

Si la respuesta es no o dudosa, el módulo debe seguir en revisión.
<<END SECTION>>

<<SECTION: MODELO Y MIGRACIÓN>>
MODELO Y MIGRACIÓN

Verificar:

la tabla usa naming consistente
la entidad incluye tenant_id si corresponde al negocio
la entidad no incluye tenant_id si es del sistema
soft delete aplicado cuando corresponda
fillable revisado
casts revisados
relaciones definidas
trait TenantScoped aplicado si corresponde
<<END SECTION>>

<<SECTION: RUTAS Y CONTROLADOR>>
SECTION_VERSION: 00002

RUTAS Y CONTROLADOR

Verificar:

rutas CRUD declaradas explícitamente
no usar Route::resource(...) mientras siga vigente la convención actual del proyecto
nombres de ruta consistentes
controller delgado
validación separada cuando haga falta
authorize(...) o authorizeResource(...) aplicado cuando corresponda
no duplicar permisos en controller si la policy ya resuelve la acción
si el módulo tiene listados o datasets con visibilidad variable, resolverlos desde la capa central de autorización y no con lógica paralela ad hoc
si el módulo requiere constraints funcionales, declararlas y consumirlas explícitamente desde la capa central

Regla adicional para create contextual:

si create depende de contexto funcional obligatorio, debe quedar definido explícitamente en controller cuál es el canal válido de autorización

no debe asumirse que authorize('create', Model::class) resuelve el caso

debe consumirse Security con el contexto real requerido por el contrato

<<END SECTION>>

<<SECTION: VISTAS>>
SECTION_VERSION: 00003

VISTAS

Todo módulo nuevo debe contar, como mínimo, con superficies visuales coherentes con el sistema.

Verificar:

index
show
create
edit cuando corresponda
partials reutilizables necesarias
componentes modulares propios cuando publiquen capacidades reusables

Reglas:

las vistas deben respetar layouts y patrones visuales vigentes

no duplicar bloques ya resueltos por componentes existentes

si una capacidad reusable pertenece al módulo, su bloque visual debe vivir dentro del propio módulo y no en una carpeta global ambigua

si el módulo publica una relación navegable reusable, no alcanza con implementarla en su show principal

también deben contemplarse tablas, sublistados y superficies secundarias consumidoras relevantes

<<END SECTION>>

<<SECTION: AUTORIZACIÓN Y UI>>
SECTION_VERSION: 00003

AUTORIZACIÓN Y UI

Verificar coherencia entre backend y superficie visible.

Controlar:

acciones visibles alineadas con permisos reales

dataset proveniente de la capa central cuando aplique

ausencia de botones inaccesibles

ausencia de links residuales hacia módulos no habilitados

create contextuales resueltos con contexto válido

capacidades reusables sin canal paralelo de autorización

Regla adicional:

si una acción reusable navega hacia otro módulo, la visibilidad final debe seguir arbitrada por el contrato del módulo oferente y no por decisiones locales del consumidor

<<END SECTION>>

<<SECTION: CHECKLIST DE INTEGRACIÓN>>
SECTION_VERSION: 00004

CHECKLIST DE INTEGRACIÓN

Cuando un módulo se relacione con otros, verificar:

consumo de contratos existentes antes de crear variantes nuevas

uso de capacidades reusables oficiales del módulo oferente

trail y retorno consistentes

flags modulares correctamente propagados

homogeneidad entre show pages, tablas y partials secundarias

ausencia de links locales remanentes hacia relaciones ya cerradas doctrinalmente por otro módulo

diferenciación explícita entre integración pesada embebida e integración compacta

alineación con el patrón visual ya consolidado en el módulo consumidor cuando exista

Regla de cierre:

no considerar terminada una integración por haber corregido solo la pantalla principal del consumidor.

Debe revisarse también el ecosistema visible que consume esa relación.

<<END SECTION>>

<<SECTION: VALIDACIONES BACKEND VS UI>>
SECTION_VERSION: 00002

VALIDACIONES BACKEND VS UI

Verificar:

la UI no sea la única barrera de control

campos ocultos también estén validados en backend

acciones no visibles también estén protegidas en backend

datasets respeten scope real desde backend

links relacionales dependan de permiso y módulo activo

si un módulo relacionado está deshabilitado, el backend rechace referencias hacia ese módulo

tabs, tablas parciales, calendarios y formularios embebidos respeten la misma lógica que la pantalla principal

no existan rutas operables residuales aunque la UI ya no las muestre
<<END SECTION>>

<<SECTION: SUPERFICIES SECUNDARIAS Y PARTIALS>>
SECTION_VERSION: 00004

SUPERFICIES SECUNDARIAS Y PARTIALS

Verificar:

partials profundas alineadas con la pantalla principal

calendarios alineados con reglas del módulo

formularios embebidos alineados con permisos reales

componentes reutilizables alineados con contrato del módulo

cancelar y redirects respetan trail vigente

no existen botones, filtros o enlaces residuales en vistas secundarias

el módulo puede considerarse cerrado también fuera de index/controller/show principal

si una capacidad reusable publicada por un módulo se consume en partials de otros módulos, se consume desde la ubicación modular oficial del oferente y no desde copias o variantes globales ambiguas

si la partial reutilizada depende de payload arbitrado, ese payload proviene del módulo oferente y no de composición local dispersa

si la superficie secundaria es una tabla o partial consumida dentro de un contenedor contextual, debe heredar trailQuery contextual y no reiniciar navegación desde una base local del módulo oferente

Regla adicional consolidada:

una integración no debe considerarse cerrada mientras exista хотя sea una superficie secundaria relevante con resolución local residual del mismo flujo o capacidad reusable

<<END SECTION>>

<<SECTION: ENTIDADES HIJAS Y SUBRECURSOS>>
SECTION_VERSION: 00003

ENTIDADES HIJAS Y SUBRECURSOS

Verificar:

si el recurso hijo tiene autonomía real o depende funcionalmente del padre

si corresponde delegar autorización en el recurso padre

si evita crear capabilities propias sin necesidad real

si evita ownership artificial en el hijo

si evita dataset independiente cuando solo es detalle operativo

si no agrega responsabilidades de cabecera por comodidad técnica

si las policies del hijo, cuando existan, delegan explícitamente en el padre

si el ciclo de vida del hijo queda claramente subordinado a la cabecera

Validar además:

pertenencia real del hijo al padre antes de operar create, edit, update o delete

navegación de cancelar y volver coherente con el padre y con el trail vigente

alineación de controllers, vistas embebidas y formularios al mismo criterio de delegación

Caso patrón ya validado:

order_items como subrecurso delegado a orders bajo orders.update
<<END SECTION>>

<<SECTION: NAVEGACIÓN Y BREADCRUMB>>
SECTION_VERSION: 00003

NAVEGACIÓN Y BREADCRUMB

Definir desde el inicio si el módulo necesita solo navegación estructural o también navegación contextual.

Preguntas de control:

la entidad puede abrirse desde otra entidad relacionada como continuación natural de trabajo

el usuario necesita volver al origen sin perder el hilo operativo

ese origen es semántico y estable, no solo la página anterior

Si alguna de estas respuestas es sí, el módulo debe evaluar e implementar navegación contextual explícita.

Si corresponde, el módulo debe:

tener builder o resolutor de trail propio compatible con el mecanismo central vigente del proyecto

aceptar contexto de navegación de forma explícita

resolver ese contexto desde la capa reutilizable central vigente

propagar ese contexto en tablas, tabs, acciones por fila, links relacionales, paginación, filtros, botones volver, botones cancelar y redirects relevantes

resolver create -> store -> show sin persistencia de nodos transitorios de “Nuevo …”

Validar además:

cuál es la superficie principal real del módulo

si existe una superficie secundaria que permanece accesible pero no debe actuar como base promovida del trail

si calendarios, tabs o embebidos contextuales necesitan actuar como entrada principal efectiva del módulo

Regla adicional consolidada:

cuando una capacidad reusable oficial del módulo oferente se consuma dentro de otro módulo, también debe propagar el trail válido del contenedor y no reiniciar navegación desde una lectura estructural local del oferente

<<END SECTION>>

<<SECTION: CIERRE>>
SECTION_VERSION: 00002

CIERRE

Un módulo no debe considerarse terminado solo porque funciona.

Debe quedar:

consistente con el sistema
alineado con permisos reales
alineado con multi-tenancy
alineado con la base visual
alineado con la documentación vigente

Además debe quedar explícitamente resuelto:

si el dataset se gobierna desde la capa central de autorización

si el módulo usa solo scopes o también constraints funcionales

si las entidades hijas o embebidas se autorizan por delegación al padre

si un create contextual requiere contexto obligatorio y cuál es su canal válido de autorización

si la validación manual del módulo puede reproducirse de forma canónica con Tinker sin ambigüedad operativa

<<END SECTION>>
