OBJETIVO

Este documento define el mapa vigente de navegación, recorridos funcionales y relaciones visibles entre módulos de app-base.

Su función es:

describir recorridos reales del sistema
ordenar navegación principal y contextual
documentar accesos entre módulos
evitar flujos inconsistentes
guiar futuras ampliaciones de navegación

No define:

seguridad doctrinal
modelo conceptual completo de entidades
contratos modulares internos
detalle visual de componentes

Esos aspectos se resuelven en otros documentos.

Regla consolidada:

este documento describe navegación real o navegación objetivo ya validada

no debe cargarse con teoría arquitectónica general que corresponda a otros documentos
<<END SECTION>>

<<SECTION: REGLA GENERAL>>
REGLA GENERAL

El sistema distingue entre:

navegación estructural
navegación contextual

navegación estructural:
expresa la ubicación normal de una pantalla dentro de su módulo

navegación contextual:
expresa el origen funcional desde el cual se llegó a una pantalla relacionada cuando ese origen sigue siendo útil para continuar el trabajo

Como regla general:

toda pantalla debe poder abrirse en modo estructural

solo ciertos flujos aceptan además trail contextual heredado

Regla adicional validada:

la navegación hacia módulos relacionados depende del estado activo del módulo destino

si el módulo destino está deshabilitado:

no deben renderizarse links
no deben renderizarse accesos
no debe existir navegación residual hacia ese módulo
<<END SECTION>>

<<SECTION: REGLAS DE LECTURA>>
SECTION_VERSION: 00002

REGLAS DE LECTURA

Una relación en este documento significa acceso, recorrido o vínculo visible para el usuario dentro del sistema.

No implica automáticamente:

foreign key obligatoria
permiso concedido
implementación ya terminada
visibilidad universal
módulo siempre habilitado

Toda navegación real sigue condicionada por:

módulo habilitado para el tenant
permiso efectivo
estado funcional del recurso
flujo operativo aplicable

Tipos de navegación relevantes:

estructural
contextual
embebida
derivada

NAVEGACIÓN ESTRUCTURAL

Recorrido estable propio del módulo.

Ejemplo:

menú principal → index → show → edit

NAVEGACIÓN CONTEXTUAL

Recorrido originado desde otro módulo conservando contexto.

Ejemplo:

appointment.show → crear order contextual

NAVEGACIÓN EMBEBIDA

Consumo de tablas, bloques o relaciones dentro de otra pantalla.

Ejemplo:

orders embebidas dentro de asset.show

NAVEGACIÓN DERIVADA

Flujo donde una acción en un módulo genera o abre otro módulo como consecuencia operativa.

Ejemplo:

orders.show → crear document

Regla:

cuando existan varias rutas posibles, debe priorizarse la más consistente con NavigationTrail y con el flujo real del usuario
<<END SECTION>>

NAVEGACIÓN PRINCIPAL TENANT

Recorrido base:

↓

↓

↓

↓

Módulos operativos principales actuales:

projects
tasks
appointments
assets
products
orders
documents

Módulos base o transversales visibles según implementación:

parties

Reglas:

no mostrar módulos no habilitados
no mostrar acciones no permitidas
no asumir igualdad de menú entre tenants distintos
la navegación principal debe reflejar contrato real del sistema y no solo existencia técnica de rutas

<<SECTION: NAVEGACIÓN CONTEXTUAL ENTRE MÓDULOS>>
SECTION_VERSION: 00009

NAVEGACIÓN CONTEXTUAL ENTRE MÓDULOS

Las relaciones entre módulos deben priorizar navegación contextual cuando mejora continuidad operativa del usuario.

Flujos relevantes actuales:

tasks.show
→ crear / abrir order relacionada

appointments.show
→ crear / abrir order relacionada

appointments.calendar
→ crear / abrir order relacionada

assets.show
→ ver órdenes relacionadas
→ crear order contextual
→ ver documents relacionados
→ ver party relacionada

parties.show
→ ver relaciones operativas
→ navegar assets relacionadas
→ navegar orders relacionadas
→ navegar documents relacionados

orders.show
→ crear document derivado
→ ver party relacionada
→ ver asset relacionado
→ gestionar items
→ gestionar attachments si aplica

documents.show
→ volver a order origen cuando exista
→ ver party relacionada
→ ver asset relacionado

projects.show
→ navegar tareas relacionadas

Reglas:

tasks participa ya como origen contextual pleno hacia orders y no solo como antecedente conceptual del flujo

appointments participa como origen contextual pleno hacia orders tanto desde show como desde calendario y superficies equivalentes

debe preservarse NavigationTrail cuando exista contexto válido

cancelar o volver debe respetar origen funcional cuando corresponda

si no existe contexto válido, usar navegación estructural estándar

no crear variantes manuales de breadcrumbs por módulo si ya existe motor común

cuando un flujo transversal hacia orders deba arbitrar si corresponde abrir una orden existente, crear una nueva, informar requisito faltante o no mostrar capacidad, debe priorizarse el consumo de la capacidad reusable publicada por orders y no lógica duplicada por vista

la capacidad reusable consolidada actual para este frente es linked-order-action

cuando un flujo transversal hacia assets deba arbitrar si corresponde abrir el asset relacionado, mostrar referencia readonly o no exponer acción operable, debe priorizarse el consumo de la capacidad reusable publicada por assets y no links locales dispersos

la capacidad reusable consolidada actual para este frente es linked-asset-action

Aclaración operativa consolidada:

linked-order-action ya no debe leerse solo como idea doctrinal ni como bloque aislado reutilizado por conveniencia

debe leerse como superficie publicada por orders y consumida desde módulos como appointments y tasks

su consumo válido incluye tanto la resolución contextual proveniente del propio módulo orders como su bloque visual modular correspondiente

linked-asset-action debe leerse con el mismo criterio doctrinal para relaciones visibles hacia assets desde módulos consumidores

la navegación contextual hacia módulos relacionados no debe reconstruirse localmente por cada vista consumidora cuando esa superficie ya fue cerrada por el módulo oferente

Regla adicional de homogeneidad:

mientras exista una superficie secundaria relevante que todavía resuelva localmente el acceso hacia orders o assets, el frente de navegación contextual no debe considerarse homogéneamente cerrado

Aclaración consolidada para parties:

cuando el flujo transversal hacia parties deba arbitrar si corresponde abrir la party relacionada, mostrarla como referencia readonly o no exponer acción operable, debe tender a priorizarse el consumo de la capacidad reusable publicada por parties y no links locales dispersos por vista

la capacidad reusable prevista para este frente es linked-party-action

parties.show no debe leerse solo como ficha aislada

debe tender a operar como pantalla transversal de relaciones operativas relevantes según módulos activos, permisos efectivos y contexto vigente

<<END SECTION>>

RELACIONES EMBEBIDAS EN SHOW PAGES

Las show pages pueden exponer relaciones mediante bloques embebidos consistentes.

Casos actuales o consolidados:

asset.show
orders relacionadas
documents relacionados
attachments

party.show
relaciones operativas según módulos activos

project.show
tasks relacionadas
métricas agregadas

task.show
capacidad reusable hacia order relacionada

appointment.show
capacidad reusable hacia order relacionada

order.show
items
attachments
acciones derivadas
relaciones principales

document.show
items
attachments cuando aplique
relaciones principales

Reglas:

priorizar componentes reusables

evitar duplicación de tablas equivalentes

si una relación ya posee bloque reusable válido, reutilizarlo

las relaciones embebidas no deben reemplazar navegación estructural cuando abrir el recurso completo agrega valor

los accesos embebidos hacia orders desde tasks o appointments deben consumir la misma lógica reusable del módulo orders

los accesos embebidos hacia assets desde módulos consumidores deben consumir la misma lógica reusable del módulo assets cuando corresponda

Regla adicional consolidada:

si una relación embebida depende de un módulo deshabilitado para el tenant, ese bloque no debe permanecer visible como pestaña operativa residual

la pantalla debe resolver fallback limpio hacia una pestaña activa disponible

Regla adicional de cierre:

también las tablas parciales y superficies secundarias embebidas deben consumir esas mismas capacidades reusables oficiales y no resolver localmente el vínculo relacionado

<<SECTION: REGLAS DEL TRAIL>>
SECTION_VERSION: 00002

REGLAS DEL TRAIL

Cuando exista trail contextual válido:

el controller resuelve el contexto

la vista consume el contexto

el trail se propaga en links, forms y redirects relevantes

el trail debe colapsar nodos repetidos

el breadcrumb visible no debe duplicar nodos ya recorridos

los botones volver, cancelar y redirects post acción deben respetar el nodo previo real cuando ese contexto siga siendo útil

El trail no debe reconstruirse desde:

url()->previous()
historial del navegador
supuestos implícitos del modelo

El trail no debe serializarse dentro de las URLs internas de sus propios nodos.

Regla adicional consolidada:

si una apertura entre módulos se resuelve mediante una capacidad reusable oficial del módulo oferente, esa capacidad también debe respetar y propagar el trail válido del flujo activo

esto aplica tanto a módulos operativos como a módulos base del dominio

casos relevantes actuales:

linked-order-action
linked-party-action

<<SECTION: REGLA PARA LINKS RELACIONALES DESDE TABLAS CONTEXTUALES>>
SECTION_VERSION: 00004

REGLA PARA LINKS RELACIONALES DESDE TABLAS CONTEXTUALES

Cuando una tabla enlaza desde una entidad fila hacia una entidad relacionada, debe evaluarse cuál es el contexto real que el breadcrumb necesita preservar.

Regla práctica:

si el trabajo sigue anclado en la fila desde la que se abrió el vínculo, el breadcrumb esperado puede ser:

módulo origen
↓
entidad fila
↓
entidad relacionada

Esto aplica especialmente en:

tablas embebidas dentro de show

listados contextuales dentro de tabs

acciones por fila hacia entidades relacionadas

sublistados donde el contenedor no alcanza por sí solo para conservar el contexto de origen

Regla adicional consolidada:

en una tabla contextual, el link principal de la fila debe consolidar primero el trail completo de esa fila

los links relacionales secundarios de esa misma fila deben reutilizar ese trail ya construido y no reiniciar contexto desde el contenedor

Regla adicional para integraciones reusables:

si la fila expone una capacidad compleja ya publicada por otro módulo, no debe reconstruirse localmente por conveniencia dentro de la tabla

debe consumirse la capacidad oficial del módulo oferente también en esa superficie contextual

Casos vigentes relevantes:

linked-order-action
linked-party-action
linked-asset-action

<<END SECTION>>

<<SECTION: DETALLE DE FLUJOS RELEVANTES>>
SECTION_VERSION: 00006

DETALLE DE FLUJOS RELEVANTES

APPOINTMENTS -> PARTIES

Uso típico:

desde un turno se consulta o abre el contacto relacionado

Contexto esperado:

si la party se abre desde el turno, puede conservarse retorno útil al appointment de origen cuando siga siendo relevante para la tarea actual

Regla adicional validada:

si parties no está habilitado para el tenant, la UI no debe mostrar ni mantener operativo ese acceso relacional

APPOINTMENTS -> ASSETS

Uso típico:

desde un turno se consulta o abre el activo relacionado

Contexto esperado:

si el asset se abre desde el turno, puede conservarse retorno útil al appointment de origen cuando siga siendo relevante para la tarea actual

Regla adicional validada:

si assets no está habilitado para el tenant, la UI no debe mostrar ni mantener operativo ese acceso relacional

APPOINTMENTS -> ORDERS

Uso típico:

un turno deriva en una orden operativa o abre la ya existente

Contexto esperado:

si la orden se abre desde appointment relacionado, puede conservarse retorno útil al turno o al módulo de agenda

Observación actual validada:

el acceso al dataset completo de appointments depende de view_any

la navegación desde appointments hacia entidades relacionadas ya no se ve restringida por errores de visibilidad

la coherencia entre listado, acceso y navegación quedó validada en funcionamiento real

el acceso hacia orders debe mantenerse consistente entre show, tablas, formularios embebidos y calendario

la resolución abrir existente / crear nueva / estado no disponible debe priorizar la capacidad reusable publicada por orders

esa capacidad debe consumirse desde el propio módulo orders, tanto en su resolución contextual como en su bloque visual modular

la relación operativa real del flujo se resuelve desde appointments.order_id

si orders no está habilitado para el tenant:

no debe mostrarse el acceso
no debe renderizarse el bloque asociado
no debe existir navegación residual hacia ese módulo

PROJECTS -> TASKS

Uso típico:

desde project.show o desde listados relacionados se accede a tareas vinculadas

Contexto esperado:

tasks puede necesitar retorno al proyecto de origen si se abrió desde allí

TASKS -> ORDERS

Uso típico:

una tarea deriva en una orden

Contexto esperado:

si el flujo empezó en project o task, la orden puede conservar ese origen funcional cuando siga siendo útil

Observación actual validada:

tasks debe tratarse como origen contextual legítimo del trail hacia orders con el mismo criterio doctrinal aplicado a appointments

la resolución abrir existente / crear nueva / estado no disponible debe apoyarse en la capacidad reusable publicada por orders cuando ese flujo se consuma desde show u otras vistas relacionadas

esa capacidad debe consumirse como superficie oficial del módulo orders y no como reconstrucción local de la vista consumidora

la relación estructural persistida del flujo se resuelve desde orders.task_id

ORDERS -> PARTIES

Uso típico:

desde una orden se consulta o abre el contacto asociado

Contexto esperado:

si la party se abre desde la orden, el breadcrumb puede conservar la orden de origen cuando esa relación siga siendo parte del trabajo en curso

Regla adicional consolidada:

cuando el acceso a parties deba resolver si la referencia es navegable o solo informativa, debe tender a priorizarse una capacidad reusable oficial del módulo parties y no criterios dispersos por tabla, show o partial

DOCUMENTS -> PARTIES

Uso típico:

desde un documento se consulta o abre el contacto asociado

Contexto esperado:

si la party se abre desde el documento, el breadcrumb puede conservar el document de origen cuando siga siendo útil para la continuidad del trabajo

PARTIES -> ASSETS

Uso típico:

desde el contacto se consultan activos vinculados

Contexto esperado:

si el flujo parte del contacto, los activos relacionados pueden abrirse preservando party.show como origen contextual útil

PARTIES -> ORDERS

Uso típico:

desde el contacto se consultan órdenes vinculadas

Contexto esperado:

si el flujo parte del contacto, las órdenes relacionadas pueden abrirse preservando party.show como origen contextual útil

PARTIES -> DOCUMENTS

Uso típico:

desde el contacto se consultan documentos vinculados

Contexto esperado:

si el flujo parte del contacto, los documentos relacionados pueden abrirse preservando party.show como origen contextual útil

Aclaración adicional consolidada:

party.show debe tender a comportarse como show transversal con relaciones operativas embebidas relevantes y no solo como ficha aislada del contacto

<<SECTION: CRITERIOS DE AUDITORÍA FUTURA>>
CRITERIOS DE AUDITORÍA FUTURA

Al auditar breadcrumb, trail o navegación entre módulos, verificar:

si la pantalla debía abrirse en modo estructural o contextual

si el trail se propagó correctamente

si hubo colapso de nodos repetidos

si volver y cancelar respetan el nodo previo real

si redirects post acción mantienen el contexto útil

si el breadcrumb visible sigue siendo compacto

si navbar y sidebar marcan el área funcional esperada y no solo la hoja final abierta

si en tablas contextuales el trail correcto era el del contenedor o el de la fila

si hay pantallas donde el contexto se pierde por filtros, tabs, paginación o acciones secundarias

si los enlaces, acciones y campos relacionales hacia módulos relacionados verifican primero que el módulo destino esté habilitado para el tenant

si la deshabilitación de módulos destino elimina completamente:

accesos
bloques visuales
referencias indirectas

especialmente en flujos desde appointments hacia orders y assets
<<END SECTION>>

<<SECTION: CRITERIO FINAL>>
SECTION_VERSION: 00002

CRITERIO FINAL

La navegación correcta del sistema no es la que muestra más enlaces.

Es la que permite al usuario llegar de forma clara, segura y coherente a la acción que necesita.

Una relación visible entre módulos se justifica cuando:

mejora continuidad operativa
reduce fricción real
respeta permisos
respeta módulos habilitados
mantiene contexto útil
no duplica caminos innecesarios

Si una relación existe técnicamente pero no cumple estos criterios, no debe priorizarse como navegación visible.
