CHECKLIST OFICIAL PARA CREAR MÓDULOS NUEVOS – app-base

OBJETIVO

Este documento define el procedimiento obligatorio para crear nuevos módulos dentro del sistema app-base.

Su finalidad es:

mantener consistencia entre módulos
garantizar compatibilidad con la arquitectura multi-tenant
evitar errores estructurales
facilitar el mantenimiento del sistema
asegurar coherencia en el crecimiento del proyecto

EVALUACIÓN DE LA ENTIDAD (ANTES DE CREAR EL MÓDULO)

Antes de crear una nueva entidad verificar:

si el caso puede resolverse utilizando la entidad parties
si corresponde a una extensión funcional de parties
si realmente es una entidad nueva del dominio del sistema

Recordar:

parties es la entidad base del negocio. 

Si la entidad no es necesaria en esta etapa:

registrarla en TODO-DEL-PROYECTO.

VERIFICAR PATRÓN REAL ANTES DE CREAR LA MIGRACIÓN

Antes de crear una nueva migración revisar migraciones existentes del proyecto.

Verificar específicamente:

tipo real utilizado para tenant_id
si las foreign keys se definen mediante foreignId() o manualmente
si existe un patrón de definición ya validado en módulos existentes

Regla:

no asumir tipos por convención de Laravel si el proyecto ya utiliza otro patrón validado.

CREAR MIGRACIÓN

Crear la migración correspondiente.

Ejemplo:

php artisan make:migration create_products_table

Reglas obligatorias para entidades del negocio:

todas las tablas deben incluir:

tenant_id
timestamps
softDeletes

CREAR MODELO

Crear modelo:

php artisan make:model NombreModelo

El modelo debe incluir los traits obligatorios del proyecto:

TenantScoped
ResolvesTenantRouteBinding
SoftDeletes

DEFINIR CAMPOS FILLABLE

Definir los campos permitidos para asignación masiva.

Reglas:

no incluir tenant_id en fillable si no es necesario
evitar permitir asignación manual de tenant_id desde formularios

CREAR CONTROLLER

Crear controlador tipo resource.

php artisan make:controller NombreController --resource

Métodos esperados:

index
create
store
show
edit
update
destroy

REGLA DE TENANT EN CONTROLADORES

Verificar la correcta asignación de tenant según el tipo de entidad.

Si la entidad es padre:

confirmar que utiliza TenantScoped
no asignar tenant_id manualmente en store()

Si la entidad es hija:

asignar tenant_id explícitamente desde el contexto del padre
asignar también la foreign key del padre

CONSULTAS TENANT-AWARE

Revisar store() y update() para asegurar que:

las verificaciones exists(...) o validaciones de relaciones
sean tenant-aware

Ejemplo conceptual:

verificar que el registro referenciado pertenezca al tenant activo.

DEFINIR RUTAS

Registrar rutas en:

routes/web.php

Ejemplo:

Route::resource('nombre', NombreController::class);

Las rutas deben ejecutarse dentro de los middlewares:

auth
tenant

CREAR VISTAS

Crear carpeta del módulo:

resources/views/nombre

Vistas mínimas:

index.blade.php
create.blade.php
edit.blade.php
show.blade.php
_form.blade.php

ESTRUCTURA VISUAL OBLIGATORIA

Todas las páginas internas deben utilizar:

<x-page>  
<x-breadcrumb />  
<x-page-header />  
<x-card />  
</x-page>

PATRÓN DE LISTADOS

Las vistas index deben utilizar:

page.list-page
card.list-card
table-wrap.list-scroll

FORMULARIOS

Los formularios deben reutilizar el parcial:

_form.blade.php

Este archivo debe ser utilizado por:

create
edit

VALIDACIONES

Las validaciones deben definirse inicialmente en el controlador.

Ejemplo:

$request->validate([
'name' => ['required','string','max:255']
]);

REGLA PARA MÓDULOS CON CABECERA + DETALLE

Si el módulo tiene estructura cabecera/detalle (por ejemplo orders o documents):

implementar primero la entidad cabecera
luego implementar el manejo de ítems mediante rutas anidadas

Ejemplo conceptual:

orders
orders/{order}/items

Reglas:

no mezclar, salvo necesidad justificada, la creación de cabecera e ítems en un único store()
el flujo debe permitir crear cabecera primero y luego gestionar los ítems

PRUEBAS FUNCIONALES

Verificar funcionamiento básico del módulo:

crear
editar
eliminar
listar

PRUEBAS FUNCIONALES AVANZADAS

Verificar adicionalmente:

verificar relaciones hijas
verificar eliminación cuando existen registros hijos
verificar route model binding multi-tenant
verificar sublistados internos si el módulo tiene detalle

VERIFICACIÓN MULTI-TENANT

Confirmar que:

un tenant no puede acceder a datos de otro tenant.

Esto debe estar garantizado mediante:

TenantScoped
ResolvesTenantRouteBinding

DATOS DE PRUEBA (OPCIONAL)

Si el módulo es relevante para el sistema:

crear factory.

Ejemplo:

php artisan make:factory NombreModeloFactory

y agregar datos al DemoSeeder.

LIMPIEZA DE MÓDULOS ABSORBIDOS

Si un módulo previo fue absorbido por otro módulo más general:

eliminar restos técnicos del módulo anterior
eliminar migraciones innecesarias
eliminar rutas, modelos o controladores que ya no tengan uso

ACTUALIZAR DOCUMENTACIÓN DEL PROYECTO

Si se crea una nueva entidad actualizar:

MAPA-DE-ENTIDADES
TODO-DEL-PROYECTO

REGLA DE COHERENCIA

Todo módulo nuevo debe seguir el mismo patrón utilizado en:

projects
parties

REGLA FINAL

Si durante el desarrollo aparece una decisión que afectará a toda la arquitectura del sistema:

no improvisar.

Registrar la decisión en:

CONVENCIONES-DEL-PROYECTO

antes de continuar el desarrollo.
