MAPA-DE-ENTIDADES – app-base

Objetivo

Este documento define el mapa conceptual de entidades del proyecto app-base.

Su finalidad es:

visualizar la estructura del sistema
evitar duplicación de modelos
orientar la creación de nuevos módulos
mantener coherencia en el crecimiento del SaaS

Criterio general de modelado

El sistema distingue entre:

entidades del sistema
entidades del negocio
entidades operativas
entidades documentales
entidades futuras de soporte

Reglas principales:

users es entidad del sistema
parties es la entidad base del negocio
antes de crear una entidad nueva evaluar si puede resolverse con una entidad existente

Entidades del sistema

Estas entidades forman el núcleo técnico del SaaS.

users

Representa usuarios con acceso al sistema.

Responsabilidades:

autenticación
acceso al sistema
gestión de sesión

tenants

Representa empresas dentro del SaaS.

Responsabilidades:

aislamiento lógico de datos
contexto empresarial de operación

memberships

Relaciona usuarios con empresas.

Responsabilidades:

pertenencia de un usuario a uno o más tenants
gestión futura de roles o permisos dentro del tenant

Regla sobre TenantScoped

El trait TenantScoped se utiliza únicamente en entidades cuyo ciclo de vida pertenece completamente a un tenant.

Ejemplos de uso correcto:

Product
Order
Document
Branch
Role
Invitation

No debe utilizarse en entidades del sistema como:

User
Tenant
Membership

Entidad base del negocio

parties

Entidad general para personas u organizaciones relacionadas con el negocio.

Nombre técnico:

parties

Nombre visible en interfaz:

contactos

Ejemplos de uso:

cliente
proveedor
empleado
contratista
tercero

La diferenciación inicial se realiza mediante un campo de tipo.

Dominio funcional:

kind es un dominio cerrado que define el tipo de actor representado por la entidad.

Campos conceptuales esperados

tenant_id
kind
name
display_name opcional
document_type nullable
document_number nullable
tax_id nullable
email nullable
phone nullable
address nullable
notes nullable
is_active
timestamps
soft deletes

Entidades operativas

Representan la operación diaria del negocio.

projects

Agrupa trabajos o iniciativas.

Relaciones principales:

tenant → projects
party → projects
projects → tasks

Nota:

projects no posee catálogo funcional propio por el momento.

tasks

Unidad operativa de trabajo.

Representa actividades concretas dentro del sistema.

Relaciones principales:

tenant → tasks
project → tasks
party → tasks
user → tasks

Dominio funcional:

status es un dominio cerrado que define el estado operativo de la tarea.

products

Representa productos, artículos o insumos.

Relaciones posibles:

tenant → products
products → orders
products → inventory
products → documents

Dominio funcional:

kind es un dominio cerrado que define la naturaleza del producto.

orders

Representa pedidos u órdenes comerciales.

Características funcionales:

representan intención comercial
los totales se calculan dinámicamente a partir de los ítems

Relaciones posibles:

tenant → orders
orders → parties
orders → products
orders → documents

Dominios funcionales:

kind define el tipo de orden
status define el estado del proceso comercial

order_items

Representa el detalle de productos dentro de una orden.

Relaciones principales:

order → order_items
product → order_items

Características:

heredan el tenant desde la cabecera
no exponen tenant_id en fillable
la posición del ítem es responsabilidad del módulo

Dominio funcional:

kind reutiliza el mismo dominio definido para products.

Entidades documentales

documents

Entidad genérica para documentos comerciales u operativos.

Ejemplos de documentos:

factura
remito
presupuesto
orden de trabajo
nota de crédito
recibo

Características funcionales:

representan documentos formales
los totales se persisten en la cabecera

Relaciones posibles:

tenant → documents
documents → parties
documents → products
documents → projects
documents → orders

Dominios funcionales:

kind define el tipo documental
status define el estado del documento

document_items

Representa el detalle de líneas dentro de un documento.

Relaciones principales:

document → document_items
product → document_items

Características:

heredan el tenant desde la cabecera
la eliminación de la cabecera elimina sus ítems

Dominio funcional:

kind reutiliza el mismo dominio definido para products.

Entidades financieras futuras

payments

Representa pagos o cobros asociados a documentos.

Relaciones posibles:

tenant → payments
payments → documents
payments → parties

Entidades de inventario futuras

inventory

Representa stock de productos.

inventory_movements

Registra entradas, salidas o ajustes de stock.

Entidades futuras de soporte

addresses
attachments
notes
roles
permissions
branches
document_sequences
party_profiles

Nota importante

Aunque la interfaz utiliza el término contactos, no se creará una tabla contacts.

La entidad real continúa siendo:

parties

Regla de racionalización

Antes de crear una nueva entidad base evaluar:

si pertenece al sistema o al negocio
si puede resolverse con parties
si puede resolverse con documents
si es realmente necesaria ahora
