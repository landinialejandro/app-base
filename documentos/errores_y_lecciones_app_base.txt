LECCIONES VIGENTES

Las lecciones vigentes del proyecto deben orientar decisiones futuras para evitar repetir errores ya detectados.

NO ASUMIR UTILITIES CSS NO VALIDADAS

Una clase visual tipo utility no debe asumirse disponible si no está validada en la base CSS real.

Caso detectado:

class="d-block"

No produjo el salto de línea esperado en los Blades compactos de ProductOperationalSummary.

Corrección aplicada:

reemplazar span + d-block por div

Lección:

cuando se necesita comportamiento estructural HTML seguro, preferir markup semántico simple antes que depender de una utility CSS no validada.

Regla:

si una clase CSS no está confirmada en la base visual real, no debe usarse como solución estructural obligatoria.

PRIMER PILOTO FUNCIONAL NO SIEMPRE ES CIERRE ARQUITECTÓNICO

Una primera implementación puede funcionar visual o técnicamente y, aun así, no representar el cierre arquitectónico correcto.

Caso detectado:

la primera implementación de ProductOperationalSummary como include directo funcionaba, pero no respetaba plenamente el patrón surface-driven ni la identidad de subcapa contextual.

Corrección aplicada:

mover salida visual a ProductSurfaceService
eliminar include directo
organizar archivos en carpeta propia

Ubicaciones finales:

app/Support/Products/OperationalSummary/
resources/views/products/operational-summary/

Lección:

si existe infraestructura surface-driven para el host, la salida estable debe integrarse mediante surface del módulo oferente y no quedar como include local disperso.

Regla:

funcionar no alcanza.

Debe respetar arquitectura, ownership, superficie de integración, seguridad y documentación vigente.

NO CALCULAR MÉTRICAS EN BLADE

Las vistas deben representar payload ya preparado por servicios o capas de lectura.

Caso vigente:

ProductOperationalSummary publica vistas compactas para summary_items.

Regla:

los Blades compactos no deben calcular compras, ventas, entradas, salidas, proveedor actual, proveedor anterior ni promedios.

Deben recibir datos ya resueltos.

Lección:

cuando una vista necesita métricas operativas, debe existir una capa de lectura previa.

El Blade no debe transformarse en servicio, query builder ni autorizador.

NO CONFUNDIR SUBCAPA CONTEXTUAL CON MÓDULO NUEVO

Una capacidad visual o analítica alojada dentro de un módulo host no debe convertirse automáticamente en módulo nuevo.

Caso vigente:

ProductOperationalSummary

Lectura correcta:

subcapa contextual interna de Products
sin CRUD
sin navegación propia
sin ModuleCatalog propio
sin dashboard general
sin rutas propias

Lección:

antes de abrir módulo, tabla, ruta o dashboard, debe revisarse si la necesidad puede vivir como subcapa interna del módulo host y publicarse por surface.

NO USAR OPERATIONALACTIVITY COMO FUENTE DE MÉTRICAS CUANDO EXISTE MÓDULO DUEÑO

OperationalActivity registra trazabilidad transversal.

No debe usarse como fuente principal de métricas si existen módulos dueños de los datos transaccionales u operativos.

Caso vigente:

ProductOperationalSummary debe usar Orders e Inventory como fuentes principales.

OperationalActivity conserva trazabilidad, pero no aporta la métrica principal.

Lección:

trazabilidad no reemplaza datos operativos del módulo dueño.

MIGRATE FRESH SEED NO CIERRA FLUJOS RUNTIME

Un flujo puede pasar lint, migrate:fresh y seeders, pero fallar en runtime web real.

Caso detectado:

Order → Document falló por dependencia a DocumentTotalsCalculator::apply(), método inexistente para ese flujo.

El error apareció en prueba web real.

Corrección aplicada:

el flujo Order → Document recalcula totales desde document_items luego de copiar ítems.

Lección:

migrate:fresh --seed no alcanza para cerrar flujos derivados con controllers, requests, autorización, copia de líneas, cálculos y redirecciones.

Regla:

todo flujo derivado relevante debe validarse en runtime web real antes de considerarse cerrado.

UN VÍNCULO NULLABLE NO DEBE REINTRODUCIRSE COMO REQUIRED SIN DOCTRINA

Un campo relacional nullable puede ser vínculo opcional y no dependencia obligatoria.

Caso detectado:

Orders necesitaba poder crearse sin party_id usando counterparty_name.

Documents necesitaba poder formalizarse desde Orders autónomas sin depender rígidamente de party_id.

Corrección aplicada:

se separó vínculo gestionado opcional y snapshot textual propio.

Patrón vigente:

party_id = vínculo gestionado opcional

counterparty_name = snapshot textual propio

Lección:

un `*_id` no debe tratarse automáticamente como required por existir en el modelo o por conveniencia visual.

Regla:

antes de exigir un vínculo en controller o request, validar esquema, contrato funcional, doctrina y autonomía del módulo.

NO UBICAR RELACIONES DE ORIGEN O SEGUIMIENTO EN EL MÓDULO RECEPTOR POR COMODIDAD

La relación persistente debe vivir en el módulo que conserva ownership real del concepto.

Caso detectado:

orders.task_id ubicaba en Orders una relación de seguimiento que pertenece funcionalmente a Tasks.

Corrección aplicada:

la relación se movió a tasks.order_id.

Lectura vigente:

Tasks organiza pendientes y seguimiento.

Orders ejecuta intervención operativa.

Una Order puede tener múltiples Tasks asociadas desde la lectura funcional del ecosistema.

Lección:

si una relación expresa origen, planificación o seguimiento, no debe ubicarse automáticamente en el módulo receptor.

Regla:

validar siempre qué módulo conserva ownership conceptual antes de persistir el vínculo.

LOS SNAPSHOTS DEBEN LLEGAR A SEEDERS, PRINTS, LISTADOS Y SURFACES

Agregar un snapshot al modelo no cierra el saneamiento si las demás salidas siguen dependiendo solo del vínculo gestionado.

Caso vigente:

Orders y Documents incorporan counterparty_name.

Riesgo detectado:

listados, prints, show summaries, surfaces o componentes linked pueden seguir mostrando solo party->name y dejar vacío cuando party_id sea null.

Lección:

la autonomía del módulo debe verse en todas sus salidas.

Regla:

cuando exista método de presentación propio, debe usarse en listados, show summaries, prints, surfaces y salidas contextuales.

Casos vigentes:

Order::displayCounterpartyName()

Document::displayCounterpartyName()

PARTYLINKED NO ES FUENTE PRINCIPAL DE CONTRAPARTE

PartyLinked representa vínculo gestionado con Parties.

No debe usarse como fuente principal de texto visible cuando el módulo dueño posee snapshot propio de contraparte.

Caso vigente:

Orders y Documents pueden tener counterparty_name aunque party_id sea null.

Lección:

un componente linked no debe ocultar la autonomía visual del módulo dueño.

Reglas:

PartyLinked se usa cuando existe party_id y corresponde mostrar vínculo gestionado

si no existe party_id pero existe counterparty_name, debe mostrarse texto propio

no inventar link hacia Party

no mostrar vacío si el módulo dueño posee snapshot

ASSETLINKED NO ES FUENTE PRINCIPAL DE ACTIVO SI EXISTE REFERENCIA PROPIA

AssetLinked representa vínculo gestionado con Assets.

No debe usarse como fuente principal de texto visible cuando el módulo dueño posee referencia propia de activo.

Caso vigente:

Orders puede representar Activo mediante asset_id y asset_reference.

Lección:

un componente linked no debe ocultar la autonomía visual del módulo dueño.

Reglas:

AssetLinked se usa cuando existe asset_id y corresponde mostrar vínculo gestionado

si no existe asset_id pero existe asset_reference, debe mostrarse texto propio

no inventar link hacia Asset

no mostrar vacío si el módulo dueño posee referencia propia

SHOWPARTY NO DEBE SEGUIR COMO CONTRATO PRINCIPAL SI EL DATO ES CONTRAPARTE

Un nombre de contrato visual incorrecto genera confusión conceptual y acoplamiento.

Caso detectado:

showParty podía interpretarse como columna de vínculo gestionado a Party, cuando el dato visible real era Contraparte.

Corrección vigente:

showCounterparty es el contrato visual para columnas de contraparte en Orders y Documents.

Lección:

el nombre del contrato visual debe expresar el dato operativo visible, no necesariamente la entidad gestionada relacionada.

Regla:

usar Contraparte cuando se representa dato propio del registro.

Usar Contacto cuando se filtra o selecciona Party gestionada.

NO DEJAR FLAGS VISUALES SIN REPRESENTACIÓN EN TABLAS REUSABLES

Un contrato visual no queda cerrado si el flag viaja por SurfaceService y partials intermedios, pero la tabla final no lo representa.

NO VALIDAR MIDDLEWARE SOLO CON INVOCACIÓN MANUAL SI EL RUNTIME HTTP USA PIPELINE LARAVEL

Un middleware puede funcionar en una prueba manual o Tinker si se lo invoca con argumentos adicionales, pero fallar en navegador real cuando Laravel lo ejecuta mediante su pipeline HTTP.

Caso detectado:

ResolveSelfServiceExternalCustomer funcionaba en prueba manual pasando SelfServiceExternalSession como tercer argumento.

Pero en runtime web real Laravel llamó el middleware como:

handle($request, $next)

y se produjo ArgumentCountError por dependencia esperada como parámetro adicional de handle.

Causa:

la dependencia de servicio fue modelada como argumento de handle en lugar de inyectarse por constructor.

Corrección aplicada:

inyectar SelfServiceExternalSession mediante constructor del middleware.

Lectura correcta:

el middleware debe recibir dependencias por constructor injection cuando son servicios del contenedor.

Los parámetros adicionales de handle deben reservarse para parámetros explícitos declarados en la definición del middleware, no para servicios internos.

Lección:

Tinker puede validar lógica aislada, pero no reemplaza prueba runtime HTTP del pipeline real.

Reglas:

no dar por cerrado un middleware solo por invocación manual

no pasar servicios como argumentos extra de handle salvo contrato explícito de middleware parametrizado

inyectar servicios del contenedor por constructor

validar middleware en navegador o request HTTP real

validar rutas públicas con y sin sesión cuando el middleware sea pasivo

validar que el middleware no convierta una ruta pública en privada si su contrato es solo contextual

Caso vigente:

ResolveSelfServiceExternalCustomer es middleware externo pasivo de self_service_sales.

No autoriza.

No bloquea visitantes públicos.

No exige login externo.

No reemplaza guard externo definitivo.

No habilita operación comercial.

Solo resuelve contexto externo cuando existe sesión externa válida para el tenant público de la ruta.


Caso detectado:

Orders ya propagaba showAsset desde OrderSurfaceService y embedded-tabs, pero orders.partials.table no renderizaba la columna Activo.

Corrección aplicada:

orders.partials.table incorporó render condicional de Activo mediante showAsset y usa Order::displayAssetReference() como helper de lectura visible.

Lección:

cuando una tabla reusable recibe flags de columnas, debe auditarse que cada flag tenga representación real o eliminarse si no corresponde.

Regla:

un flag visual no debe quedar como contrato fantasma.

NO MOSTRAR TEXTO PLANO CUANDO EXISTE VÍNCULO GESTIONADO ENLAZABLE

Mostrar el texto correcto no cierra el patrón si existe un vínculo gestionado autorizado que debería representarse como link oficial.

Caso detectado:

Orders mostraba Parties y Assets como texto aunque existieran party_id y asset_id.

Corrección aplicada:

orders.partials.table incorporó PartyLinked y AssetLinked cuando existen vínculos gestionados correspondientes.

Lección:

si el registro dueño tiene vínculo gestionado opcional y el enlace es seguro/autorizado, la salida visual debe permitir navegación oficial.

Regla:

si existe vínculo gestionado autorizado, mostrar link mediante helper o componente oficial.

Si no existe vínculo gestionado, mostrar referencia textual propia.

NO INTERPRETAR RECORD_METADATA DESDE BLADE

record_metadata puede guardar condiciones contextuales o excepcionales, pero no debe ser interpretado directamente por vistas.

Caso vigente:

Orders guarda manualidad de contraparte y activo en:

record_metadata.relationships.counterparty.managed
record_metadata.relationships.asset.managed

Corrección aplicada:

la lectura de manualidad se encapsuló en helpers del modelo:

Order::hasManualCounterpartyReference()
Order::hasManualAssetReference()

Lección:

si una condición contextual debe representarse visualmente, el modelo dueño o una capa de presentación debe exponer un helper o payload explícito.

Regla:

Blade representa.

No interpreta JSON operativo.

NO CONSERVAR FALLBACKS LEGACY CUANDO EL CONTRATO VIGENTE YA ESTÁ VALIDADO

Un fallback transitorio debe eliminarse cuando se confirma que los consumidores vigentes usan el contrato nuevo.

Caso detectado:

Orders mantenía showParty como fallback transitorio para showCounterparty en:

resources/views/orders/partials/table.blade.php
resources/views/orders/partials/embedded-tabs.blade.php

Corrección aplicada:

showParty fue eliminado de Orders luego de validar que no quedaban usos vigentes.

Lección:

la compatibilidad transitoria no debe convertirse en convivencia indefinida.

Regla:

cuando el grep confirma ausencia de consumidores legacy, debe eliminarse el fallback.

FUNCIONAR NO ALCANZA SI LA LÓGICA TRANSVERSAL QUEDA DENTRO DE UN SERVICE PARTICULAR

Un service puede funcionar y aun así contener mecánica transversal que conviene extraer para evitar duplicación futura.

Caso detectado:

OrderSurfaceService contenía lógica reusable de armado de hostPack y resolución de relacionados embebidos.

Corrección aplicada:

se extrajo la construcción genérica de hostPack a:

app/Support/Modules/Concerns/BuildsHostPacks.php

se extrajo la resolución reusable de relacionados embebidos a:

app/Support/Modules/Concerns/ResolvesRelatedEmbeddedSurfaces.php

Lección:

los SurfaceService deben declarar ofertas, hosts y configuración propia.

La mecánica transversal reusable debe vivir en Support/Modules cuando corresponda.

Regla:

no vaciar services por reflejo.

Separar solo lógica transversal reusable de lógica propia del módulo.

UN FRENTE HABILITABLE NO CRUD NO REEMPLAZA PERMISOS DE MÓDULOS DUEÑOS

Una capa satélite productiva puede tener entrada visual y permiso propio de acceso al área, pero eso no autoriza acciones internas sobre los módulos participantes.

Caso vigente:

service_maintenance existe como frente habilitable no CRUD.

service_maintenance.view_any permite acceso al área Servicio y mantenimiento.

Orders sostiene la primera base operativa mediante:

group = service

Lección:

acceder al área no equivale a operar sus recursos internos.

Regla:

las acciones reales deben seguir autorizándose por el módulo dueño.

En órdenes de servicio, las acciones reales siguen perteneciendo a Orders.

NO TRASLADAR DECISIONES TÉCNICAS INTERNAS AL USUARIO

Un formulario puede funcionar técnicamente y aun así exponer al usuario decisiones que el sistema debe resolver.

Caso detectado:

el intento inicial de resolver carga gestionada/manual con botones visibles sobrecargó la UX del formulario.

Patrón descartado:

Usar carga manual

Usar contacto vinculado

Usar referencia manual

Usar activo vinculado

Corrección aplicada:

si existen componentes gestionados, se usa carga gestionada.

Si no existen, se muestra carga manual.

Lección:

no debe trasladarse al usuario una decisión técnica interna cuando el sistema puede resolverla por disponibilidad de componentes.

Regla:

la UI debe reducir fricción y no convertir alternativas técnicas internas en pasos visibles innecesarios.

NO NOMBRAR JS GENÉRICO CON ENTIDADES PARTICULARES

Un comportamiento frontend reusable no debe nombrarse por un caso puntual si la mecánica real es general.

Caso detectado:

app-party-asset-sync

El nombre acoplaba la mecánica a Party/Asset.

Corrección aplicada:

app-linked-select-sync

Lección:

el nombre del contrato JS debe expresar la mecánica reusable.

Regla:

si la interacción es sincronización de selects vinculados, el contrato no debe fijar entidades concretas.

UNA RUTA CONTEXTUAL NECESITA RAÍZ NAVEGACIONAL REAL

Un conjunto de rutas contextuales puede quedar débil si no posee raíz clara para navegación y breadcrumb.

Caso detectado:

/service/orders sin una raíz clara /service generaba duda en el breadcrumb y podía enganchar el flujo al listado general de Orders.

Corrección aplicada:

crear dashboard mínimo en:

/service

Lección:

cuando un universo funcional tiene rutas internas propias, debe evaluarse una raíz navegacional real para sostener jerarquía, breadcrumb y comprensión del usuario.

Regla:

un universo semántico debe preservar su contexto desde la entrada hasta el show o cierre de la acción cuando corresponda.

QUERY PARAMS NO SON SEGURIDAD

Ocultar, eliminar u ofuscar un query param no constituye seguridad.

Caso detectado:

?group=service podía expresar contexto, pero no debía sostener por sí mismo la regla de órdenes de servicio.

Corrección aplicada:

crear rutas semánticas /service y forzar group = service desde backend.

Lección:

la seguridad y coherencia funcional deben sostenerse en backend, autorización, contexto y validación de datos.

Regla:

si un valor contextual no debe depender del usuario, debe forzarse o validarse del lado servidor.

PATCH PARCIAL PUEDE DEJAR FIRMAS INCONSISTENTES

Modificar solo una llamada sin revisar la firma del método receptor puede generar errores runtime.

Caso detectado:

OrderNavigationTrail::create() fue actualizado para pasar Request a contextualBase(), pero contextualBase() conservó la firma vieja.

Resultado:

TypeError.

Corrección aplicada:

actualizar contextualBase() y show() para alinear firma y consumidores.

Lección:

cuando se modifica una firma, deben auditarse llamada, método receptor y consumidores relacionados.

Regla:

un patch parcial no debe cerrarse sin grep o revisión específica de firmas impactadas.

UN PATCH PUEDE INFORMAR AVANCE PERO NO HABER APLICADO EL CAMBIO ESPERADO

La auditoría general puede pasar aunque una cadena crítica siga igual.

Caso detectado:

el primer intento de corregir redirect post-store no modificó realmente la redirección y seguía existiendo orders.show.

Corrección aplicada:

grep específico y reemplazo completo del método store().

Lección:

después de un patch sensible, debe confirmarse la presencia o ausencia exacta del cambio esperado.

Regla:

cuando se busca eliminar una ruta, método, flag o cadena legacy, debe hacerse grep posterior específico.
