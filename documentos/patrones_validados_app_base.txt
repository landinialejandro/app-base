RECONVERSIÓN LIMPIA DE CONSTRAINTS

Cuando cambia la dimensión real de autorización, la constraint funcional debe migrar a esa dimensión y eliminar la convivencia legacy.

Caso validado:

parties:
antes → allowed_kinds sobre parties.kind
ahora → allowed_party_roles sobre party_roles.role

Regla:

la constraint anterior no debe quedar como fallback operativo si ya no representa la semántica vigente del módulo.

AUTORIZACIÓN SENSIBLE CENTRALIZADA

Cuando una regla sensible aparece duplicada en controller y requests, debe extraerse a una pieza reusable del dominio correspondiente.

Caso validado:

PartyEmployeeContactAuthorization

Ubicación vigente:

app/Support/Parties/PartyEmployeeContactAuthorization.php

Regla:

la UI puede expresar disponibilidad, pero la autorización sensible debe resolverse y bloquearse en backend.

VALIDACIÓN RUNTIME DE SEGURIDAD

Una validación sensible debe combinar:

UI correcta
Security runtime
request validation
intento manipulado
verificación final de persistencia

Caso validado:

protección de party vinculada a membership activa.

La validación no se considera cerrada solo por ocultar campos en UI.

Debe probarse que el backend bloquea intentos manipulados y que el registro persiste intacto.

EVENTOS OPERATIVOS TRANSVERSALES

Patrón validado:

módulo dueño
↓
autoriza, valida y muta
↓
emite evento operativo liviano
↓
listener queued con ShouldQueue y afterCommit
↓
prestador transversal aplica sanidad
↓
persistencia append-only

Reglas:

el evento operativo no autoriza
el evento operativo no valida la operación original
el evento operativo no reemplaza Security ni policies
el módulo dueño conserva ownership funcional
la capacidad transversal registra solo si corresponde
el listener queued no debe bloquear el flujo principal

Caso vigente:

OperationalActivity

CAMPOS OBSERVABLES POR CATÁLOGO DEL MÓDULO DUEÑO

Cuando una capacidad transversal registra actividad operativa, los campos observables deben pertenecer al catálogo del módulo dueño.

Casos vigentes:

AppointmentCatalog::activityTrackedFields()
TaskCatalog::activityTrackedFields()
OrderCatalog::activityTrackedFields()
DocumentCatalog::activityTrackedFields()

Regla:

el prestador transversal no debe conocer en exceso la semántica interna de cada módulo.

SANIDAD DE CAMBIOS OPERATIVOS

La trazabilidad operativa debe aplicar sanidad antes de persistir.

Reglas:

no registrar cambios vacíos
ignorar campos técnicos
no registrar por registrar
registrar solo cambios operativamente significativos
clasificar actividad desde el prestador transversal cuando corresponda

Clasificaciones vigentes posibles:

assigned
reassigned
unassigned
status_changed
updated

Regla:

la clasificación no debe dispersarse en controllers.

EVENT DISCOVERY SIN DUPLICACIÓN MANUAL

Cuando Laravel event discovery ya registra un listener, no debe duplicarse el registro manual mediante Event::listen().

Validación requerida:

event:list debe mostrar la asociación esperada.

Regla:

si event discovery ya resuelve la asociación, duplicar el listener genera jobs y registros repetidos.

VALIDACIÓN QUEUED DE ACTIVIDAD OPERATIVA

Secuencia validada para probar eventos queued y OperationalActivity:

disparar evento
verificar jobs > 0
ejecutar queue:work --stop-when-empty --tries=1
verificar operational_activities
verificar jobs = 0
verificar failed_jobs = 0

Regla:

una trazabilidad queued no se considera cerrada solo por disparar el evento.

Debe validarse procesamiento real de cola y persistencia final.
