CONVENCIONES-DEL-PROYECTO – app-base

Objetivo

Definir las convenciones técnicas y de desarrollo que deben seguir todos los módulos del proyecto app-base.

Estas reglas permiten mantener consistencia, previsibilidad y facilidad de mantenimiento del sistema.

Arquitectura base

El sistema utiliza la arquitectura estándar de Laravel basada en:

Modelos Eloquent
Controladores
Vistas Blade
Migraciones
Middleware
Rutas REST

Multi-tenancy

El sistema es SaaS multi-tenant.

El aislamiento de datos se realiza mediante la columna:

tenant_id

Todas las entidades del negocio deben pertenecer a un tenant.

Fuente del tenant operativo

La única fuente válida del tenant operativo es:

app('tenant')

No se debe utilizar directamente:

auth()->user()->tenant_id
request()->tenant_id

Todas las operaciones del sistema deben resolver el tenant actual mediante:

app('tenant')

Acceso al tenant actual

El tenant activo puede obtenerse mediante:

app('tenant')

Los controladores deben utilizar siempre este tenant como contexto de operación.

CONVENCIONES ESPECÍFICAS DEL NÚCLEO SAAS

Estas reglas describen el funcionamiento estructural del núcleo multi-tenant del sistema.

IDENTIDAD GLOBAL

users.email es único global en el sistema.

No existe relación directa user -> tenant mediante un tenant_id en users.

La pertenencia real entre usuario y empresa se resuelve exclusivamente mediante memberships.

TENANTS Y CLAVES

tenants.id utiliza UUID (char(36)) como primary key.

Por lo tanto:

Toda relación hacia tenants debe utilizar CHAR(36).

No deben asumirse bigint ni foreignId() por defecto en relaciones hacia tenants.

Antes de crear una migración debe verificarse el patrón real utilizado en el proyecto.

MEMBERSHIPS

memberships es la tabla puente central del modelo multi-tenant.

Reglas:

Un mismo user puede pertenecer a múltiples tenants.
Un mismo user puede ser owner de múltiples tenants.

memberships define la pertenencia real entre usuario y empresa.

Importante:

Membership no debe tratarse como una entidad tenant-scoped al estilo de los módulos internos.

Por lo tanto:

Membership no debe usar TenantScoped

INVITATIONS

No existe registro libre público de usuarios.

Toda alta o vinculación relevante dentro del sistema debe originarse desde una invitación válida.

Dominios cerrados:

invitations.type

owner_signup
member_invite

invitations.status

pending
accepted
expired
cancelled

REGLA DE tenant_id EN INVITATIONS

invitations.tenant_id puede ser NULL cuando la invitación es de tipo:

owner_signup

Esto permite emitir invitaciones para onboarding de nuevas empresas antes de crear el tenant real.

Cuando:

type = member_invite

entonces:

tenant_id debe existir
y debe apuntar a un tenant válido.

SIGNUP_REQUESTS

signup_requests representa una solicitud pública inicial para crear una empresa en el sistema.

Reglas:

signup_requests no crea users ni tenants por sí sola.

Su función es separar la fase comercial/manual del alta real del sistema.

Una signup_request aprobada puede originar una invitation de tipo:

owner_signup

VALIDACIÓN DE IDENTIDAD EN INVITACIONES

Una invitación nunca debe aplicarse sobre una sesión autenticada con otro email.

Regla:

Si existe conflicto entre el email de la invitación y la sesión activa:

debe cerrarse la sesión actual
debe forzarse autenticación con el email correcto

Comportamiento esperado:

member_invite

si el email no existe, el usuario completa el alta

si el email ya existe, debe pasar por login obligatorio

owner_signup

el flujo continúa por alta/onboarding del owner

si el user ya existe, se reutiliza ese user

no se debe crear un usuario duplicado

Regla de reanudación del flujo de invitación

Cuando una invitación requiere autenticación (por ejemplo porque el usuario ya existe):

el sistema debe redirigir al usuario al flujo original de la invitación inmediatamente después del login.

La redirección posterior al login no debe utilizar la redirección estándar del sistema (dashboard o selector de empresa).

El flujo correcto debe ser:

login requerido
↓
autenticación exitosa
↓
reanudar aceptación de la invitación pendiente

Esto garantiza que:

la invitación no se pierda
el usuario complete el flujo correcto
no se generen inconsistencias en memberships o creación de tenants.

CREACIÓN DE TENANT

En el flujo:

owner_signup

el tenant se crea al momento de aceptar la invitación, no durante la solicitud pública.

Reglas:

el slug del tenant debe generarse automáticamente

el sistema debe resolver posibles colisiones de slug

Trait obligatorio en modelos empresariales

Los modelos del negocio deben utilizar:

TenantScoped
ResolvesTenantRouteBinding
SoftDeletes

Objetivos:

TenantScoped
aplica automáticamente el filtro por tenant.

ResolvesTenantRouteBinding
protege el acceso por URL.

SoftDeletes
permite eliminación lógica.

Uso de ResolvesTenantRouteBinding

Todo modelo accesible mediante route model binding que pertenezca a un tenant debe utilizar el trait:

ResolvesTenantRouteBinding

Esto evita acceso cruzado entre tenants mediante URLs manipuladas.

Convenciones de base de datos

Todas las tablas del negocio deben incluir:

tenant_id
timestamps
softDeletes

Reglas para tenant_id

tenant_id no debe crearse con:

foreignId('tenant_id')

Debe definirse manualmente con un tipo compatible con tenants.id.

Asignación de tenant_id en entidades

Entidades padre

Cuando el modelo utiliza el trait TenantScoped, tenant_id no debe asignarse manualmente en el controller.

Entidades hijas

En entidades hijas dependientes de una cabecera:

tenant_id existe para facilitar índices

no debe exponerse en $fillable

debe heredarse desde la cabecera o desde el contexto del tenant

Convenciones de validación multi-tenant

Las validaciones que utilicen exists o claves foráneas deben restringirse al tenant activo.

Esto evita referencias cruzadas entre tenants.

Convenciones de controladores

Los controladores deben seguir el patrón REST estándar de Laravel.

Métodos esperados:

index
create
store
show
edit
update
destroy

Convenciones de rutas

Las rutas deben definirse en:

routes/web.php

Utilizar resource routes siempre que sea posible.

Ejemplo:

Route::resource('projects', ProjectController::class);

Las rutas deben ejecutarse dentro de los middlewares:

auth
tenant

Convenciones de modelos

Los modelos deben:

usar Eloquent
definir relaciones explícitas
definir fillable o guarded correctamente
utilizar los traits de multi-tenancy

Convenciones de vistas

Cada módulo debe tener su carpeta de vistas:

resources/views/nombre_modulo

Ejemplos:

resources/views/projects
resources/views/parties
resources/views/tasks

Estructura visual de páginas

Todas las páginas internas deben seguir esta estructura:

<x-page> <x-breadcrumb /> <x-page-header /> <x-card /> </x-page>

Componentes Blade disponibles

x-page
x-breadcrumb
x-page-header
x-card

Convención de listados

Las vistas index deben utilizar el patrón de lista del sistema.

page.list-page
card.list-card
table-wrap.list-scroll

Esto garantiza:

scroll interno en tablas
encabezado visible
layout estable

Convención de formularios

Los formularios deben:

utilizar parciales _form
agrupar campos de forma consistente
mostrar errores cerca del campo
ubicar botones dentro de form-actions

CONVENCIONES DE VISTAS Y CSS

Las vistas Blade deben utilizar exclusivamente las clases y patrones visuales definidos en:

public/css/app-base.css

y en el documento:

BASE VISUAL DEL PROYECTO – app-base.

No deben incorporarse clases de frameworks externos o ajenos al proyecto si esas clases no están definidas explícitamente en la base visual propia.

Reglas para formularios

Los formularios deben seguir el patrón visual común del proyecto.

Clases permitidas:

form
form-group
form-label
form-control
form-help
form-actions

Los mensajes de error deben utilizar clases definidas dentro del sistema visual del proyecto.

No deben utilizarse clases externas o utilidades no soportadas por app-base.css.

Ejemplos de clases que no deben utilizarse si no están definidas en el sistema visual:

invalid-feedback
is-invalid
text-danger
form-check
d-flex
gap-*
btn-sm

Acciones compactas con íconos

Las acciones compactas en tablas o sublistados deben utilizar el patrón visual propio del proyecto.

No deben utilizarse utilidades externas de frameworks para compactar botones o alinear iconos.

Deben utilizarse los componentes de iconos definidos en el sistema.

Visualización de catálogos

Cuando una entidad utilice catálogos funcionales (Catalogs), las vistas deben mostrar siempre las etiquetas legibles del catálogo.

No deben mostrarse directamente los valores internos persistidos en la base de datos.

Esto evita exponer códigos internos al usuario y mejora la claridad de la interfaz.

Convención para catálogos funcionales

Los estados, tipos y variantes funcionales del sistema no deben repetirse como strings sueltos.

La implementación recomendada es centralizar estos catálogos en:

app/Support/Catalogs

Si un dominio ya existe en otro módulo debe reutilizarse y no duplicarse.

Gestión de módulos reemplazados

Cuando un módulo sea reemplazado por otro más general:

el módulo anterior debe eliminarse

no deben mantenerse módulos obsoletos

Terminología de interfaz

La interfaz utiliza términos orientados al usuario.

Equivalencias:

Tenant → Empresa
Party → Contacto

La nomenclatura técnica puede mantenerse en el código.

Selector de empresa

Un usuario puede pertenecer a múltiples empresas.

Reglas:

si tiene una sola empresa se accede directamente

si tiene varias puede cambiar mediante selector en el header

Páginas públicas

Algunas páginas no utilizan el layout completo.

Ejemplos:

welcome
login
register
selección de empresa

Estas páginas utilizan:

$publicPage = true

Principios generales

Simplicidad
Evitar complejidad innecesaria.

Consistencia
Seguir siempre las convenciones definidas.

Escalabilidad
Permitir evolución futura del sistema.

Claridad
El código debe ser fácil de entender y mantener.
