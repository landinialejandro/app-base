INSTRUCCIÓN INICIAL OBLIGATORIA

Proyecto: app-base (Laravel multi-tenant)

Idioma de trabajo: español

Regla máxima permanente:

NO SE ASUME
NO SE SUPONE

Antes de proponer cambios:

leer contexto vigente
trabajar sobre código real
pedir evidencia mínima necesaria
no inventar estructura
no inferir APIs
no suponer tablas, rutas o contratos

Regla adicional consolidada:

ante integraciones entre users, memberships y parties, separar siempre:

identidad global
pertenencia operativa
ficha ampliada opcional

users mantiene identidad global.

memberships resuelve pertenencia operativa al tenant.

party puede funcionar como ficha ampliada tenant-scoped solo cuando exista vínculo explícito.

No debe asumirse creación automática ni edición libre de datos identitarios.

Estado base reciente a considerar:

Orders y Documents comparten lifecycle persistido:

draft
pending_approval
approved
closed
cancelled

La primera línea válida promueve automáticamente:

draft → pending_approval

Inventory ya opera integrado con Orders y Documents mediante surfaces oficiales y mantiene ownership funcional propio.

La convergencia reusable de line items quedó avanzada y validada en múltiples capas.

Las show pages principales ya cuentan con patrón común para tabs host mediante x-host-tabs y helper HostTabs.

return_tab se usa como contexto interno del host para preservar el panel activo cuando corresponde.

trail continúa representando contexto externo entre recursos.

Parties fue reconvertido conceptualmente y en seguridad.

Lectura vigente:

party.kind = tipo de contacto

Valores:

person
company

party_roles.role = relación con la empresa

Valores:

customer
supplier
employee
other

Seguridad vigente de parties:

parties usa allowed_party_roles como constraint funcional.

allowed_party_roles se evalúa contra party_roles.role.

orders conserva allowed_kinds.

allowed_kinds no aplica a parties.

Reglas de continuidad para parties:

no usar customer, supplier ni employee como parties.kind

no reintroducir KIND_CUSTOMER, KIND_SUPPLIER ni KIND_EMPLOYEE como valores de kind

no usar allowed_kinds como permiso o constraint funcional de parties

no usar helpers legacy basados en kind como permiso de parties

no usar rol en UI de parties si puede confundirse con roles de permisos o memberships

usar preferentemente:

Tipo de contacto
Relación con la empresa

Lectura vigente de employee:

colaborador con membership activa → ficha bloqueada parcialmente

colaborador manual sin user → contacto común con relación employee, solo gestionable por usuarios autorizados

La autorización sensible de employee está centralizada en:

app/Support/Parties/PartyEmployeeContactAuthorization.php

El formulario progresivo de parties ya está implementado y funcionando.

No debe revertirse a formulario plano sin justificación basada en código real y necesidad validada.

OperationalActivity quedó incorporado como capacidad transversal tenant-scoped de trazabilidad operativa.

Lectura vigente:

no es CRUD ordinario
no es navegación principal
no autoriza operaciones
no reemplaza Security
no reemplaza policies
no valida la operación original
no debe bloquear el flujo principal si falla el registro del log

La secuencia correcta es:

módulo dueño autoriza
módulo dueño valida
módulo dueño ejecuta
módulo dueño emite evento operativo liviano
OperationalActivity registra si corresponde

La escritura del log es system-owned y append-only.

No debe proponerse create manual, edit, update, delete, destroy ni carga manual desde UI para OperationalActivity.

La lectura futura del log todavía requiere autorización explícita.

No asumir acceso de lectura por existir operational_activities.

Módulos actualmente integrados con OperationalActivity:

appointments
tasks
orders
documents

Cuando una tarea involucre Project Lab, automatizador, Tinker, runtime real, colas, eventos, permisos, constraints, multi-tenancy, actualización de código o actualización documental, se debe respetar la guía operativa vigente de Project Lab.

Regla crítica para snippets:

usar FILE solo para archivos completos.

usar TARGET solo para operaciones parciales.

no mezclar FILE con operadores de método.

no entregar formatos ambiguos.

REGLAS OPERATIVAS DEL CHAT

Trabajar siempre con foco en coherencia doctrinal del sistema.

Preferencias operativas vigentes:

auditar antes de modificar

pedir solo el contexto faltante imprescindible

no rehacer diagnósticos generales ya cerrados

proponer cambios concretos y verificables

priorizar bajo riesgo y alto impacto

mantener seguridad y tenancy como núcleo no negociable

si una capacidad pertenece a un módulo oferente, no mover ownership al consumidor

si dos módulos comparten gramática equivalente, evaluar reutilización antes de duplicar

validar con evidencia real cuando corresponda:

syntax
routes
runtime
browser informado por usuario

Regla operativa adicional:

ante mejoras UI o refactors visuales, no asumir estructura de archivos.

Primero deben ubicarse archivos reales mediante CLI o evidencia equivalente.

Regla operativa adicional:

si un dato ya existe en un catálogo estructural, debe consumirse desde allí y no duplicarse localmente.

Antes de modificar múltiples módulos, services o views por una misma necesidad, debe buscarse el punto común de normalización.

Esto aplica especialmente cuando la necesidad pueda repetirse en el futuro.

No declarar cierres por intuición.

El cierre debe apoyarse en evidencia técnica suficiente.

Regla operativa adicional para parties:

no asumir la semántica anterior de kind.

Antes de tocar parties, verificar la lectura vigente:

parties.kind = person/company

party_roles.role = customer/supplier/employee/other

allowed_party_roles = constraint funcional de parties sobre party_roles.role

allowed_kinds no aplica a parties

Si la tarea involucra employee, membership o ficha ampliada, distinguir siempre:

employee con membership activa
employee manual sin user del sistema

No mezclar esa distinción con roles de permisos.

Si la tarea involucra employee, consumir o preservar la autorización sensible desde:

PartyEmployeeContactAuthorization

No duplicar esa regla en controller, requests o vistas.

La UI de parties expresa permisos resueltos, pero no autoriza.

tools/project-lab

Project Lab permite:

ejecutar Tinker sobre runtime real
ejecutar Tinker desde textarea
ejecutar Tinker desde clipboard del sistema mediante backend
ejecutar comandos Artisan rápidos
aplicar actualizaciones de código
aplicar actualizaciones documentales
ejecutar auditorías simples
usar snippets rápidos
ver salida en consola unificada
copiar resultados para compartir en el chat

Herramientas Lab disponibles:

code
docs
audit

Criterios de uso:

usar code para aplicar actualizaciones de código compatibles con el automatizador vigente
usar docs para aplicar actualizaciones documentales por bloques completos de sección
usar audit para ejecutar auditorías simples o comandos de inspección del proyecto

Capacidades actuales del automatizador de código:

creación o reemplazo de archivos completos PHP
creación o reemplazo de archivos completos Blade
creación o reemplazo de archivos completos CSS
creación o reemplazo de archivos completos JS
actualización parcial de métodos PHP existentes
alta de métodos nuevos dentro de clases PHP
actualización parcial de funciones JS
actualización de secciones CSS/JS mediante bloques SECTION
normalización automática de encabezado FILE y versión
mantenimiento de log operativo de cambios
salida de consola con estado y trazabilidad
validaciones básicas de ruta para evitar escrituras accidentales

Capacidades actuales del automatizador documental:

actualización por DOC_SLUG
reemplazo completo de secciones delimitadas
validación de nombre exacto de sección
rechazo de secciones inexistentes
salida explícita OK o ERROR

Reglas:

la existencia de Project Lab no reemplaza auditoría técnica ni criterio arquitectónico

la herramienta ejecuta cambios

la decisión de usarlos sigue perteneciendo al proceso técnico

si una tarea consiste principalmente en aplicar código ya definido sobre archivos existentes, priorizar formato compatible con Project Lab antes que edición manual fragmentada

si una prueba requiere runtime real, tenant activo, Security o contexto funcional, debe respetarse la guía operativa vigente de Project Lab

si una prueba requiere eventos queued, debe validarse también cola, jobs, failed_jobs y persistencia final cuando corresponda

si una intervención involucra OperationalActivity, no alcanza con disparar el evento: debe validarse procesamiento real y ausencia de duplicación de listeners

SI NECESITÁS ARCHIVOS

Pedir solo archivos mínimos necesarios.

Usar formato claro y copiable.

Cuando se requiera auditoría masiva, solicitar salida ordenada hacia carpeta auditoria.

Ejemplo:

(
find app/Models -type f -name "*.php";
find app/Http/Controllers -type f -name "*.php";
find app/Http/Middleware -type f -name "*.php";
find database/migrations -type f -name "*.php";
find resources/views/orders -type f -name "*.blade.php";
find resources/views/documents -type f -name "*.blade.php";
) | sort | while read file; do
echo "===== $file ====="
cat "$file"
echo
echo
done > documentos/auditoria/auditoria-modulo.txt

<<SECTION: REGLA PARA ARCHIVOS COMPLETOS DE CÓDIGO>>
SECTION_VERSION: 00003

REGLA PARA ARCHIVOS COMPLETOS DE CÓDIGO

Cuando devuelvas un archivo completo, la primera referencia del archivo debe usar siempre este formato:

FILE: ruta/del/archivo.ext | VX

Reglas:

si el archivo es nuevo, usar V1
si el archivo ya existe y no tiene comentario de versión, al devolverlo completo agregarlo como V2
si el archivo ya tiene comentario de versión, incrementar en 1 la última versión existente
esta regla aplica solo cuando se devuelve el archivo completo
si se devuelve solo un fragmento o bloque parcial, no agregar ni modificar versión
el comentario debe escribirse con la sintaxis correcta del archivo correspondiente

Formato canónico adicional para PHP completo:

la salida debe respetar esta estructura exacta:

<?php

// FILE: ruta/del/archivo.php | VX

...resto del código...

Es decir:

- apertura PHP en primera línea
- una línea en blanco
- comentario FILE con versión
- una línea en blanco
- resto del archivo
