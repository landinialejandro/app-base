Objetivo del documento

Este archivo registra:

funcionalidades pendientes

decisiones postergadas

mejoras futuras

profundizaciones arquitectónicas

El objetivo es no bloquear el avance actual del sistema, dejando registradas las mejoras que se evaluarán en etapas posteriores.

Este documento debe crecer con el proyecto.

1. Principio de desarrollo del proyecto

El proyecto seguirá esta estrategia:

Fase 1
Sistema funcional mínimo.

Fase 2
Mejoras estructurales y ampliación de funcionalidades.

Fase 3
Optimización, seguridad avanzada y escalabilidad.

Regla importante:

No profundizar prematuramente en arquitectura si bloquea la construcción del sistema funcional.

2. Multi-tenant (pendientes)

El núcleo multi-tenant ya está operativo, pero quedan mejoras futuras.

Pendientes
Validaciones adicionales de tenant

Evitar posibles inconsistencias si:

se accede a recursos sin tenant activo

se manipulan rutas manualmente

Protección extra en queries críticas

Hoy el TenantScoped global scope protege los modelos.

En el futuro evaluar:

auditoría automática de tenant_id

logs de acceso entre tenants

Middleware de verificación de membresía

Verificar que el usuario pertenece realmente al tenant activo.

Hoy:

Futuro posible:

EnsureTenantMembership
