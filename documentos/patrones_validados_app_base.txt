ADUANA FUNCIONAL ENTRE MÓDULOS

Patrón validado:

un módulo consumidor solicita una capacidad
↓
una frontera explícita procesa la necesidad
↓
el módulo oferente conserva ownership de su lógica
↓
el consumidor recibe payload, decisión, bloqueo, link, selector o surface

Formas válidas de frontera:

hooks
selectors
services
linked helpers
surface services
catálogos
registries
contratos explícitos

Reglas:

el consumidor no reconstruye lógica interna del oferente

el oferente no interpreta modelos internos del consumidor

el controller consumidor no debe acumular conocimiento de dominios externos si existe frontera apropiada

una relación de modelo no autoriza por sí misma a gobernar un módulo externo

Caso vigente:

Orders no debe interpretar Task, Appointment, Party, Asset o Document más allá de referencias pasivas, relaciones necesarias o contratos explícitos.

AISLAMIENTO OPERATIVO RAZONABLE

Patrón validado:

un módulo saneado no requiere aislamiento absoluto artificial.

Requiere:

núcleo propio claro
relaciones pasivas justificadas
dependencias activas saneadas
contratos explícitos para integración
controllers sin lógica interna de dominios externos
helpers genéricos del módulo oferente
hooks o services para bloqueos externos
selectors para selección o validación de dominios externos
surface services por capacidad real

Caso vigente:

Orders queda como isla funcional saneada apta para iniciar Fase 1 de mantenimiento.

Reglas:

las relaciones pasivas pueden conservarse si corresponden a columnas propias, derivaciones aceptadas o patrones transversales

las dependencias activas deben pasar por frontera explícita

no debe confundirse relación pasiva con ownership funcional

no debe confundirse aislamiento razonable con desconexión artificial del ecosistema

HOOKS COMO FRONTERA FUNCIONAL

Patrón validado:

el módulo dueño del flujo consulta un hook en puntos estratégicos.

El módulo externo puede:

bloquear
ajustar
dejar pasar
informar condición externa
aportar validación contextual

Reglas:

el módulo dueño del flujo no conoce la lógica interna del módulo externo

el módulo externo conserva ownership de su regla

si una verificación de bloqueo crítico falla, debe asumirse el escenario seguro

el hook no reemplaza autorización principal del módulo dueño

Caso vigente:

OrdersHooks funciona como frontera estratégica del flujo de Orders.

InventoryOrderItemHooks conserva la lógica material de bloqueo de cierre por ejecución física de líneas.

LINKED HELPER GENÉRICO

Patrón validado:

el módulo oferente arma links para su propio modelo o null.

No interpreta modelos consumidores.

Caso vigente:

OrderLinked::forOrder(?Order $order, ...)

Reglas:

si Tasks necesita mostrar una Order, Tasks entrega Order|null

si Appointments necesita mostrar una Order, Appointments entrega Order|null

Orders arma el link de su propio recurso

no deben existir métodos linked específicos por consumidor

No deben reaparecer:

OrderLinked::forTask()
OrderLinked::forAppointment()
OrderLinked::forDocument()
OrderLinked::forAsset()
OrderLinked::forParty()

SURFACE AGRUPADA POR CAPACIDAD REAL

Patrón validado:

una misma capacidad con igual formato y distinto host, filtro o configuración contextual debe representarse como una sola offer con múltiples targets.

Caso vigente:

OrderSurfaceService ofrece orders.related.embedded como surface embebida única para assets.show y parties.show.

Configuración contextual:

relatedHosts()

Puede concentrar:

clase esperada
columna de filtro
tabsId
flags visuales
mensaje vacío
parámetros de montaje

Reglas:

no duplicar offers por host si la capacidad es la misma

la surface debe expresar la capacidad real

el host aloja

el oferente prepara payload o vista

SURFACE SERVICE DECLARATIVO CON LÓGICA TRANSVERSAL EXTRAÍDA

Patrón validado:

un SurfaceService debe declarar ofertas, hosts y configuración propia.

La lógica transversal reusable no debe quedar incrustada dentro de un único SurfaceService si puede servir como mecánica común para otros módulos.

Piezas vigentes:

app/Support/Modules/Concerns/BuildsHostPacks.php
app/Support/Modules/Concerns/ResolvesRelatedEmbeddedSurfaces.php

Lectura:

BuildsHostPacks concentra la construcción genérica de hostPack.

ResolvesRelatedEmbeddedSurfaces concentra la resolución reusable de relacionados embebidos.

El SurfaceService conserva:

ofertas propias
hosts soportados
configuración contextual
relaciones propias
flags visuales propios
mensajes propios

Caso validado:

OrderSurfaceService conserva su configuración propia de hosts y relaciones, pero ya no contiene toda la mecánica genérica de validación, consulta y armado de payload.

Hosts validados:

orders.show
assets.show
parties.show

Surface validada:

orders.related.embedded

Payload validado:

orders
tabsId
trailQuery
emptyMessage
showCounterparty
showAsset

Reglas:

no vaciar un SurfaceService por reflejo

no mover lógica propia del módulo a un concern transversal

extraer solo mecánica reusable

aplicar el patrón progresivamente y con validación runtime

no migrar services de forma masiva sin evidencia

SELECTOR O PROVEEDOR DE DOMINIO

Patrón validado:

cuando varios módulos necesitan seleccionar o validar entidades de un mismo dominio, debe existir una capacidad central del módulo dueño.

Caso vigente:

ProductLineItemSelector

Responsabilidad:

opciones de productos
reglas de validación de productos
verificación de acceso a productos para líneas

Consumidores vigentes:

OrderItemController

DocumentItemController

Reglas:

Products ofrece selección y validación de productos

Orders administra sus propios order_items

Documents administra sus propios document_items

los controllers consumidores no deben reconstruir Product::query(), products.viewAny ni Rule::exists('products') cuando existe selector oficial

NUMERACIÓN TRANSVERSAL CON SIGNIFICADO EN CATÁLOGOS

Patrón validado:

un servicio transversal puede ejecutar mecánica común de numeración, pero no decidir significado de negocio propio de cada módulo.

Regla síntesis:

el módulo define el significado.

el servicio transversal ejecuta la mecánica.

Caso vigente:

RecordNumberGenerator ejecuta la mecánica de numeración.

OrderCatalog define prefijos de Orders.

DocumentCatalog define prefijos de Documents.

Caso eliminado como lectura vigente:

DocumentNumberGenerator

Reglas:

los prefijos pertenecen al catálogo del módulo

el generador transversal no debe decidir prefijos por sí mismo

no debe usarse un service nombrado por un módulo si la mecánica es transversal y compartida

RELACIONES PASIVAS EN MODELOS

Patrón validado:

los modelos pueden conservar relaciones pasivas, estructurales o transversales si corresponden al contrato real.

Eso no habilita controllers invasivos.

Reglas:

una relación pasiva no equivale a ownership funcional

una relación de consulta no equivale a autorización

una relación inversa no necesaria debe eliminarse si induce navegación conceptual incorrecta o acoplamiento activo indebido

Caso vigente:

Order ya no expone tasks().

La dirección conceptual es Task -> Order mediante tasks.order_id.

Relaciones aceptadas actualmente en Order:

party
asset
documents
attachments

Lectura:

estas relaciones son pasivas, de consulta, derivación o transversales mientras no habiliten que OrderController gobierne módulos externos.

RECONVERSIÓN LIMPIA DE CONSTRAINTS

Cuando cambia la dimensión real de autorización, la constraint funcional debe migrar a esa dimensión y eliminar la convivencia legacy.

Caso validado:

parties:
antes → allowed_kinds sobre parties.kind
ahora → allowed_party_roles sobre party_roles.role

Regla:

la constraint anterior no debe quedar como fallback operativo si ya no representa la semántica vigente del módulo.

AUTONOMÍA MODULAR CON VÍNCULOS OPCIONALES

Patrón validado:

un módulo puede conservar autonomía funcional aunque posea vínculos opcionales hacia otros módulos.

Lectura:

el vínculo opcional enriquece, contextualiza o formaliza

no define por sí solo existencia funcional

no concede autorización

no debe convertirse automáticamente en required

no debe forzar dependencia visual

Casos vigentes:

Orders puede existir sin party_id

Documents puede existir o derivarse sin party_id cuando conserva contraparte textual propia

Regla:

un campo `*_id` debe interpretarse según contrato funcional real, no por reflejo técnico.

SNAPSHOT TEXTUAL PROPIO + VÍNCULO GESTIONADO OPCIONAL

Patrón validado:

cuando un módulo puede vincularse opcionalmente con Parties pero necesita conservar dato propio de contraparte, debe separar:

party_id = vínculo gestionado opcional

counterparty_name = snapshot textual propio del registro

Casos vigentes:

Orders:

party_id
counterparty_name

Documents:

party_id
counterparty_name

Reglas:

el snapshot no reemplaza Party

el snapshot no reemplaza Security

el snapshot no crea permiso

el snapshot permite lectura autónoma del registro

la salida visual debe usar método de presentación propio cuando exista

Casos vigentes:

Order::displayCounterpartyName()

Document::displayCounterpartyName()

Regla visual:

no mostrar solo party->name si el módulo posee snapshot propio y fallback controlado.

CONTRAPARTE SNAPSHOT CON VÍNCULO GESTIONADO OPCIONAL

Patrón validado:

módulo dueño conserva snapshot propio
↓
party_id funciona como vínculo gestionado opcional
↓
displayCounterpartyName() resuelve texto visible
↓
PartyLinked solo representa enlace a Party cuando existe y está autorizado

Casos validados:

Orders
Documents

Reglas:

la visualización de contraparte no debe depender rígidamente de party_id

los listados deben mostrar Contraparte cuando representan el dato operativo visible

los filtros por party_id pueden seguir usando Contacto

las surfaces deben publicar showCounterparty cuando corresponda mostrar la columna de Contraparte

PartyLinked no reemplaza displayCounterpartyName()

si no existe party_id pero existe snapshot propio, se muestra texto propio

si existe party_id y corresponde link, se muestra vínculo gestionado oficial

no inventar link hacia Party

no mostrar vacío si el módulo dueño posee snapshot

TABLAS/LISTADOS CON VÍNCULOS GESTIONADOS Y REFERENCIAS PROPIAS

Patrón validado:

cuando un módulo conserva vínculos gestionados opcionales y referencias textuales propias, las tablas y listados deben representar el dato operativo visible desde el módulo dueño.

Lectura:

*_id = vínculo gestionado opcional

*_reference o snapshot textual vigente = referencia textual propia del registro

record_metadata = condición contextual o excepcional

Reglas:

la tabla no debe reconstruir lógica interna del módulo oferente

la tabla no debe interpretar directamente JSON operativo

la tabla debe consumir helpers de presentación del modelo dueño cuando existan

el módulo dueño conserva la lectura visible de sus referencias propias

los SurfaceService pueden decidir qué columnas mostrar según host mediante flags explícitos

los hosts no deben duplicar información que ya está representada por el recurso anfitrión

si existe vínculo gestionado autorizado, debe mostrarse link mediante helper o componente oficial

si no existe vínculo gestionado, debe mostrarse referencia textual propia del registro

si la carga fue manual, puede mostrarse una indicación visual discreta

no inventar links cuando no existe vínculo gestionado

no reconstruir Party ni Asset desde Orders

Caso validado:

Orders.

Helpers vigentes:

Order::displayCounterpartyName()
Order::displayAssetReference()
Order::hasManualCounterpartyReference()
Order::hasManualAssetReference()

Helpers y componentes vinculados vigentes:

PartyLinked::forParty()
parties.components.linked-party
AssetLinked::forAsset()
assets.components.linked-asset

Contratos visuales vigentes:

showCounterparty
showAsset

Reglas específicas:

showCounterparty indica que una tabla o surface de Orders debe mostrar Contraparte.

showAsset indica que una tabla o surface de Orders debe mostrar Activo.

En orders.index se muestran Contraparte y Activo.

En assets.show, la surface embebida de Orders muestra Contraparte y no muestra Activo, porque el host ya es el activo.

En parties.show, la surface embebida de Orders muestra Activo y no muestra Contraparte, porque el host ya es el contacto.

record_metadata puede indicar manualidad o excepción, pero no debe convertirse en fuente principal del dato visible.

Caso validado de manualidad:

record_metadata.relationships.counterparty.managed = false
record_metadata.relationships.asset.managed = false

La lectura visual de esa condición debe resolverse mediante helpers del modelo dueño y no desde Blade.

Regla de convergencia:

showParty no debe permanecer como contrato visual de Orders.

Caso validado:

showParty fue eliminado de Orders luego de confirmar que los consumidores vigentes usan showCounterparty.

FORMULARIO HOST CON FRONTERAS EXPLÍCITAS

Patrón validado:

cuando un formulario host tiene campos que pertenecen a su contrato propio y puede recibir enriquecimiento de módulos oferentes, el host no debe renderizar ofertas externas mediante un foreach abierto.

Debe declarar fronteras explícitas.

Patrón:

si existe proveedor externo válido
renderiza componente del proveedor

si no existe proveedor externo
renderiza campo manual propio

Caso validado:

orders.form

Fronteras vigentes:

Party disponible → selector de contacto

Party no disponible → campo manual propio de contraparte

Asset disponible → selector de activo

Asset no disponible → campo manual propio o referencia propia equivalente vigente

Ofertas vigentes:

party.order.form-context

asset.order.form-context

Reglas:

el formulario host conserva su contrato propio

el módulo oferente prepara payload o componente

el host decide cómo encajar la frontera

una carga manual no debe transformarse automáticamente en gestionada sin acción explícita y trazable

RECORD_METADATA COMO METADATA CONTEXTUAL O EXCEPCIONAL

Patrón validado:

record_metadata puede conservar metadata contextual o excepcional del registro.

No reemplaza columnas centrales.

No reemplaza datos operativos principales.

No reemplaza vínculos gestionados.

No debe ser fuente principal de salida visual.

Criterio vigente:

si todo ocurre por vía gestionada normal
record_metadata puede permanecer null

si alguna frontera cae a carga manual
record_metadata puede registrar solo esa excepción

Caso validado:

record_metadata.relationships.counterparty.managed = false
record_metadata.relationships.asset.managed = false

Reglas:

la lectura de record_metadata debe encapsularse en helper, presenter o payload preparado

Blade representa

Blade no interpreta JSON operativo

CAPA SATÉLITE PRODUCTIVA HABILITABLE NO CRUD

Patrón validado:

una capa satélite productiva puede tener entrada operativa y permiso propio de acceso al área sin convertirse en módulo CRUD ordinario.

Caso vigente:

Servicios y Mantenimiento.

Lectura:

service_maintenance existe como frente habilitable no CRUD

service_maintenance.view_any permite acceso al área

Orders sostiene la primera base operativa mediante group = service

Reglas:

el permiso de acceso al área no reemplaza permisos operativos de módulos dueños

las acciones reales continúan bajo el contrato del módulo dueño

la capa satélite no absorbe ownership de Orders, Assets, Inventory, Products, Appointments, Tasks, Documents, Attachments, Parties ni OperationalActivity

la entrada visual no implica CRUD ordinario

el frente especializado puede orientar la experiencia sin crear módulos o tablas por reflejo automático

UNIVERSO SEMÁNTICO DE URL

Patrón validado:

una URL raíz puede representar un frente funcional especializado sin convertirlo automáticamente en módulo dueño.

El universo semántico expresa:

contexto funcional
intención de uso
navegación especializada
entrada operativa
rutas propias del frente

No expresa por sí mismo:

ownership funcional
permiso operativo sobre recursos participantes
CRUD ordinario nuevo
duplicación del módulo dueño

Caso validado:

/service

Lectura:

/service representa el frente Servicio y mantenimiento.

/service no reemplaza Orders.

/service no absorbe ownership de órdenes.

/service no convierte órdenes de servicio en entidad ajena a Orders.

Orders conserva ownership funcional de las órdenes.

Rutas validadas:

GET /service
GET /service/orders
GET /service/orders/create
POST /service/orders
GET /service/orders/{order}

Nombres de ruta validados:

service.index
service.orders.index
service.orders.create
service.orders.store
service.orders.show

Reglas:

el universo puede tener rutas propias cuando el contexto funcional lo justifique

las rutas internas pueden consumir controllers o capacidades del módulo dueño

el contexto funcional fuerte debe aplicarse desde backend

el universo no debe depender de query params manipulables como única fuente de contexto

las vistas pueden adaptarse al contexto del universo

los breadcrumbs deben reflejar el universo

las redirecciones físicas deben conservar el universo cuando el usuario entra por ese frente

RUTA CONTEXTUAL FUERTE SIN QUERY PARAM MANIPULABLE

Patrón validado:

cuando un frente especializado requiere contexto funcional fuerte, no debe depender únicamente de un query param manipulable.

Caso validado:

/service/orders no depende de ?group=service.

El contexto service se define desde el universo /service.

Regla:

si una orden se crea desde /service/orders, group = service debe resolverse desde backend.

Si el request intenta enviar un group incompatible con el universo, el backend debe preservar el contexto funcional correcto.

Lectura:

ocultar, eliminar u ofuscar un query param no es seguridad.

La seguridad y coherencia funcional deben sostenerse en backend.

CAMPO BLOQUEADO POR CONTEXTO

Patrón validado:

un formulario puede bloquear visualmente un campo cuando el contexto del universo define un valor fijo.

La UI puede expresar:

campo deshabilitado
hidden complementario
texto fijo
estado visual bloqueado

Pero esto no reemplaza backend.

Caso validado:

en /service, el campo Tipo puede mostrarse bloqueado visualmente como Servicio.

Reglas:

la UI comunica contexto

el backend valida o fuerza el valor

un input manipulado no debe romper el contexto del universo

el bloqueo visual no debe considerarse seguridad

JS GENÉRICO PARA SELECTS VINCULADOS

Patrón validado:

si una interacción frontend expresa una mecánica reusable, el contrato JavaScript debe nombrarse por esa mecánica y no por entidades de un caso puntual.

Caso reemplazado:

app-party-asset-sync

Contrato vigente:

app-linked-select-sync

Uso:

data-action="app-linked-select-sync"
data-source-select
data-target-select
data-source-value

Lectura:

app-linked-select-sync sincroniza selects vinculados de forma genérica.

Reglas:

no fijar nombres Party/Asset si la mecánica puede reutilizarse

el JS no decide permisos

el JS no reemplaza validación backend

el JS no debe ser la única defensa de consistencia para datos críticos

TINKER CON REQUEST MANIPULADO Y ROLLBACK

Patrón validado:

una prueba Tinker puede simular request manipulado para verificar que el backend sostenga el contrato real.

Uso validado:

enviar un valor manipulado
confirmar que backend fuerza o valida el valor correcto
confirmar redirección esperada
confirmar persistencia esperada
aplicar rollback si la prueba no debe dejar datos finales

Caso validado:

POST /service/orders con group manipulado debe preservar group = service desde backend y redirigir al universo /service.

Reglas:

la prueba debe usar usuario real, tenant real, permisos reales y runtime real cuando corresponda

el rollback debe confirmar que no quedan datos de prueba

la salida debe servir como evidencia para cierre técnico y documental
