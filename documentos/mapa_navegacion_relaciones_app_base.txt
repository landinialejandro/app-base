OBJETIVO

Este documento define el mapa vigente de navegación, recorridos funcionales y relaciones visibles entre módulos de app-base.

Su función es:

describir recorridos reales del sistema
ordenar navegación principal y contextual
documentar accesos entre módulos
evitar flujos inconsistentes
guiar futuras ampliaciones de navegación

No define:

seguridad doctrinal
modelo conceptual completo de entidades
contratos modulares internos
detalle visual de componentes

Esos aspectos se resuelven en otros documentos.

Regla consolidada:

este documento describe navegación real o navegación objetivo ya validada

no debe cargarse con teoría arquitectónica general que corresponda a otros documentos

<<SECTION: REGLA GENERAL>>
SECTION_VERSION: 00010

REGLA GENERAL

Toda navegación del sistema debe responder a una necesidad funcional real.

No debe crearse navegación por simetría visual, comodidad técnica o existencia de tablas auxiliares.

Reglas obligatorias:

navegación principal expresa estructura estable
navegación contextual expresa relaciones reales de trabajo
dataset interno no equivale a módulo navegable
host visual no equivale a ownership funcional
el trail preserva contexto entre pantallas relacionadas
la UI no debe prometer accesos no operables

Regla adicional consolidada:

si una necesidad puede resolverse como pieza reusable del plano tenant o como capacidad publicada por un módulo existente, esa salida debe evaluarse antes de abrir una nueva ruta mental en el sistema mediante navegación adicional.

Regla adicional para actividad operativa:

la existencia de operational_activities no habilita por sí sola navegación principal nueva.

OperationalActivity debe leerse primero como capacidad transversal de soporte.

Cualquier lectura visual debe responder a una necesidad funcional real, autorización explícita y superficie definida.

La integración backend de nuevos módulos con OperationalActivity no modifica por sí misma la navegación principal, las show pages, las tabs ni los recorridos visibles.

Regla:

no crear tab, vista general, menú ni acceso de actividad sin contrato de lectura autorizado y decisión visual específica.

Regla adicional para lectura contextual:

una tab Actividad dentro de un host autorizado no crea navegación principal de OperationalActivity.

La navegación visible permanece dentro del show del host.

La lectura contextual debe limitarse a la actividad asociada al registro host autorizado, al conjunto explícito de records definido por el módulo dueño cuando exista activity_record_set o al criterio contextual especial definido por el módulo dueño cuando exista activity_context.

Si no existe activity_record_set ni activity_context, la lectura contextual es directa sobre el record host.

Si existe activity_record_set, la lectura contextual es agregada sobre el conjunto definido por el módulo dueño.

Si existe activity_context, la lectura contextual se adapta al criterio especial definido por el módulo dueño.

Regla:

la existencia de una tab Actividad no implica por sí misma lectura agregada ni lectura contextual especial.

La lectura agregada debe estar declarada explícitamente por el módulo dueño mediante activity_record_set.

La lectura contextual especial debe estar declarada explícitamente por el módulo dueño mediante activity_context.

<<SECTION: REGLAS DE LECTURA>>
SECTION_VERSION: 00002

REGLAS DE LECTURA

Una relación en este documento significa acceso, recorrido o vínculo visible para el usuario dentro del sistema.

No implica automáticamente:

foreign key obligatoria
permiso concedido
implementación ya terminada
visibilidad universal
módulo siempre habilitado

Toda navegación real sigue condicionada por:

módulo habilitado para el tenant
permiso efectivo
estado funcional del recurso
flujo operativo aplicable

Tipos de navegación relevantes:

estructural
contextual
embebida
derivada

NAVEGACIÓN ESTRUCTURAL

Recorrido estable propio del módulo.

Ejemplo:

menú principal → index → show → edit

NAVEGACIÓN CONTEXTUAL

Recorrido originado desde otro módulo conservando contexto.

Ejemplo:

appointment.show → crear order contextual

NAVEGACIÓN EMBEBIDA

Consumo de tablas, bloques o relaciones dentro de otra pantalla.

Ejemplo:

orders embebidas dentro de asset.show

NAVEGACIÓN DERIVADA

Flujo donde una acción en un módulo genera o abre otro módulo como consecuencia operativa.

Ejemplo:

orders.show → crear document

Regla:

cuando existan varias rutas posibles, debe priorizarse la más consistente con NavigationTrail y con el flujo real del usuario
<<END SECTION>>

NAVEGACIÓN PRINCIPAL TENANT

La navegación principal tenant expresa los módulos funcionales visibles de la empresa activa.

Debe responder al contrato real del sistema y no a mera existencia técnica de código.

Casos funcionales actuales:

dashboard
projects
tasks
appointments
parties
products
assets
orders
documents
inventory

La aparición real de cada entrada depende de:

módulo habilitado
permiso efectivo
estado operativo cuando corresponda

Regla adicional consolidada:

users no debe aparecer por esta necesidad como módulo navegable autónomo dentro del tenant

La gestión interna de usuarios, pertenencias, roles y permisos continúa anclada al frente tenant/profile/accesses y no a una navegación principal users separada

Regla adicional consolidada:

tenant/profile es frente de gobierno interno del tenant.

No es módulo funcional ordinario del negocio.

Su acceso visible puede estar disponible para owner y admin según autorización central.

Lectura vigente:

owner puede acceder a tenant/profile como autoridad estructural de gobierno

admin puede acceder a tenant/profile con restricciones de gobierno

sales/operator no deben ver ni acceder a ese frente si no poseen autorización efectiva

La autorización de acceso debe resolverse por la pieza central correspondiente y no por condición visual local.

Caso vigente:

TenantProfileAccess::canViewProfile

Regla adicional consolidada:

la navegación principal puede incorporar iconografía por módulo para mejorar lectura visual

esa iconografía debe provenir de metadata modular (ModuleCatalog) y no definirse localmente en la vista

Aclaración vigente para parties:

parties mantiene su lugar como módulo visible de contactos.

El alta de contacto puede resolverse mediante experiencia progresiva dentro de la misma pantalla, sin crear una navegación nueva.

Aclaración vigente para OperationalActivity:

OperationalActivity no se agrega como entrada de navegación principal tenant por su mera existencia técnica.

Su lectura vigente es capacidad transversal de trazabilidad operativa.

La integración de OperationalActivity con appointments, tasks, orders, documents, projects, assets, products, attachments, parties e inventory no crea navegación principal propia.

La lectura general inicial desde tenant/profile mediante tab Actividad tampoco crea navegación principal nueva.

La lectura contextual desde show pages autorizadas mediante tab Actividad tampoco crea navegación principal nueva.

La lectura agregada mediante activity_record_set tampoco crea navegación principal nueva de OperationalActivity.

La navegación principal no debe abrir un menú propio de actividad salvo decisión futura validada, autorizada y documentada.

Regla:

la navegación principal no reemplaza flujos contextuales ni relaciones entre módulos

Su función es dar acceso estructural al ecosistema operativo vigente

<<END SECTION>>

<<SECTION: NAVEGACIÓN CONTEXTUAL ENTRE MÓDULOS>>

NAVEGACIÓN CONTEXTUAL ENTRE MÓDULOS

Los recorridos entre módulos deben respetar identidad propia, contexto funcional y contratos vigentes.

Relaciones vigentes destacadas:

orders consume capacidades oficiales de inventory sin absorber su lógica interna

orders puede actuar como host visual para inventory, documents y attachments

products consume lectura contextual de inventory mediante saldos e historial

documents puede contextualizar trazabilidad de inventory cuando exista vínculo documental

documents puede actuar como host visual de acciones oficiales de inventory sobre document_items

appointments, tasks, assets y parties mantienen sus relaciones propias dentro del ecosistema general

tenant/profile actúa como frente de gobierno interno del tenant para usuarios, memberships, roles y permisos

Regla de frontera:

que un módulo hospede visualmente una capacidad ajena no lo convierte en dueño de su lógica operativa.

El módulo oferente conserva reglas, validaciones y evolución interna.

Aclaración vigente para tenant/profile:

tenant/profile no abre un módulo users.

La gestión de usuarios, memberships, roles y permisos continúa anclada al frente tenant/profile/accesses.

Owner y admin pueden visualizar ese frente cuando la autorización central lo permita.

La diferencia entre ambos no se resuelve creando navegaciones separadas, sino aplicando restricciones dentro del mismo frente.

Lectura vigente:

owner gobierna

admin opera fuerte con restricciones

Aclaración vigente para edición de funciones:

la edición de funciones de una membership puede resolverse como micro-interacción modal dentro de la pantalla de roles/accesos.

Patrón visible:

acción por fila
apertura de modal
edición del conjunto de funciones
cancelar o guardar
recarga natural tras submit cuando corresponda

Esta interacción no constituye navegación nueva ni cambio de recurso.

Aclaración vigente para inventory:

la capa inventory_operations no modifica por sí misma los recorridos principales entre módulos.

Su función es enriquecer el contexto interno y la trazabilidad del frente inventory.

Frente a OperationalActivity, InventoryOperation actúa como ancla operativa agrupada.

InventoryMovement conserva la trazabilidad fina de stock y no genera por sí mismo navegación contextual adicional en el log transversal.

InventoryOperation no tiene todavía show canónico confirmado para navegación desde el log.

Aclaración vigente para documents:

cuando documents hospeda operaciones de inventory por línea documental, la navegación sigue perteneciendo al flujo documental anfitrión salvo que el usuario decida abrir una surface canónica de inventory.

Aclaración vigente para lifecycle de cabeceras:

la creación de una cabecera en orders o documents no implica por sí sola que el recurso esté listo para operar.

La cabecera nace en draft y puede pasar a pending_approval cuando recibe la primera línea válida.

Este cambio pertenece al flujo natural del recurso y no crea una navegación nueva.

Aclaración vigente para membership → party:

la ficha ampliada de colaborador puede abrir recorrido contextual hacia parties cuando exista vínculo explícito.

El flujo desde perfil o membership hacia party debe preservar la distinción entre:

identidad global
pertenencia operativa
ficha ampliada tenant-scoped

Regla:

crear o vincular una ficha ampliada no debe ocurrir de forma silenciosa por navegación.

Debe existir acción o confirmación explícita.

Aclaración vigente para OperationalActivity:

OperationalActivity puede ofrecer lectura como actividad general del tenant o como actividad contextual asociada a un registro host autorizado.

Esa lectura no modifica por sí misma los recorridos principales de los módulos que originaron la actividad.

La navegación hacia el registro relacionado debe resolverse desde la propia fila de actividad, integrando el enlace en la columna Registro.

No debe agregarse una columna adicional solo para link.

Lectura visual general vigente:

tenant/profile
tab Actividad

Estado actual:

la tab Actividad muestra los últimos 50 registros de actividad operativa del tenant

tenant/profile puede ser accesible por owner y admin según contrato vigente, con restricciones diferenciadas de gobierno

Lectura contextual validada:

show del host autorizado
tab Actividad
tabla de actividad contextual del record, del record set declarado o del activity_context declarado

Hosts validados:

tasks.show
parties.show
projects.show
assets.show
products.show
appointments.show
orders.show
documents.show

La tab se accede mediante:

return_tab=activity

Reglas:

esta tab no representa navegación principal de OperationalActivity

no crea ruta propia nueva de OperationalActivity

no habilita lectura general del tenant

no habilita acceso a otros registros fuera del host autorizado, del record set declarado por el módulo dueño o del activity_context declarado por el módulo dueño

La lectura directa por defecto aplica cuando el host tiene view autorizado y consume la surface de actividad.

La lectura agregada aplica solo cuando el módulo declara activity_record_set.

La lectura contextual especial aplica solo cuando el módulo declara activity_context.

Caso validado de activity_record_set:

projects.show muestra actividad del Project y de sus Tasks mediante ProjectActivityRecordSet.

Caso validado de activity_context:

parties.show puede mostrar actividad ampliada de una Party employee con membership activa vinculada mediante PartyEmployeeActivityContext.

Lectura validada para parties.show:

Party común
→ actividad directa del registro Party

Party employee manual sin user o sin membership activa
→ actividad directa del registro Party

Party employee con membership activa vinculada
→ actividad del colaborador:

* actividad directa de Party
* actividad donde el user vinculado sea actor
* actividad donde el user vinculado sea sujeto

Aclaración vigente:

orders.show y documents.show quedan con actividad directa en el estado actual.

Si más adelante requieren actividad agregada o contextual especial, deberán definir su propio contrato.

No se amplía por simetría automática.

La navegación desde una actividad hacia el registro destino debe existir solo cuando haya ruta show canónica segura.

Caso validado:

una actividad de Task visible desde Project enlaza a tasks.show mediante OperationalActivityLinkResolver.

Trail validado para lectura general:

Inicio
Perfil de empresa · Actividad
Registro destino

Casos actuales sin link seguro:

InventoryOperation
Attachment

Regla:

no se inventan links para records sin ruta canónica segura.

RELACIONES EMBEBIDAS EN SHOW PAGES

Las relaciones embebidas dentro de show pages deben mejorar continuidad operativa sin confundir ownership funcional.

Reglas:

el host conserva identidad principal
el módulo invitado conserva su lógica y semántica
las acciones del invitado no se reinterpretan en el host
la navegación derivada debe mantener contexto cuando corresponda
las relaciones compactas y las relaciones fuertes no deben mezclarse sin criterio

Criterios de uso:

summary_items para relaciones puntuales o de baja densidad
detail_items para bloques breves informativos
tabs o paneles amplios para relaciones fuertes
row actions oficiales para operaciones dentro de tablas anfitrionas

Caso vigente en orders.show:

inventory puede operar por línea dentro del host sin perder identidad modular

documents y attachments pueden consumirse como capacidades invitadas

Caso vigente en documents.show:

inventory puede operar por línea documental dentro del host sin perder identidad modular

Las acciones por fila pertenecen a inventory y documents conserva la tabla documental como superficie anfitriona

Caso vigente en parties.show:

parties puede hospedar relaciones embebidas o compactas publicadas por otros módulos mediante surfaces oficiales.

La lectura de parties.show debe respetar la nueva semántica de parties:

tipo de contacto
relación con la empresa

La definición de qué relaciones embebidas resultan coherentes en parties.show debe analizarse desde party_roles.role y no desde parties.kind.

Criterio vigente para ese análisis:

customer
supplier
employee
other

Reglas para parties.show:

no decidir embebidos por parties.kind
no inventar relaciones
no agregar embebidos manuales
consumir surfaces oficiales de los módulos oferentes
detectar brechas cuando una relación necesaria no tenga surface oficial publicada
mantener el ownership funcional en el módulo oferente

No se abre navegación nueva para party_roles.

party_roles sigue siendo entidad hija o complementaria de parties y no módulo navegable propio.

Aclaración vigente para actividad en show pages:

la tab Actividad puede publicarse como lectura contextual heredada cuando el host ya autorizó el registro actual.

No debe leer actividad desde campos actuales ni inferir historia desde created_by, updated_by, assigned_user_id o party_id.

Debe consumir OperationalActivity persistido y autorizado.

La lectura contextual se resuelve mediante:

OperationalActivityContextReader
OperationalActivitySurfaceService
TenantSurfaceRegistry
ModuleSurfaceRegistry
x-host-tabs

Hosts validados:

tasks.show
parties.show
projects.show
assets.show
products.show
appointments.show
orders.show
documents.show

Reglas:

la tab Actividad contextual no habilita lectura general del tenant

no usa operational_activity.view_any

no convierte OperationalActivity en módulo navegable

no convierte al host en dueño de la lógica de actividad

no debe agregarse manualmente con push local si ya puede publicarse desde registry reusable

La lectura contextual directa se limita al record host autorizado.

La lectura contextual agregada requiere activity_record_set declarado por el módulo dueño.

Caso validado:

projects.show utiliza lectura agregada Project + Tasks mediante ProjectActivityRecordSet.

La lectura contextual especial requiere activity_context declarado por el módulo dueño.

Caso validado:

parties.show utiliza PartyEmployeeActivityContext para Party employee con membership activa vinculada.

Regla:

la lectura agregada y la lectura contextual especial no deben resolverse con inferencias internas de OperationalActivity.

El módulo dueño define la frontera funcional.

Aclaración vigente para parties.show y employee:

Party común
→ actividad directa del registro Party

Party employee manual sin user o sin membership activa
→ actividad directa del registro Party

Party employee con membership activa vinculada
→ actividad directa de Party y actividad donde el user vinculado sea actor o sujeto

Esta lectura no crea módulo users.

No crea módulo memberships.

No crea módulo party_roles.

No crea navegación propia de OperationalActivity.

Aclaración vigente para tenant/profile:

tenant/profile incorpora tab Actividad como lectura visual general de OperationalActivity.

Esta lectura general se diferencia de la lectura contextual por registro.

Orden visual recomendado para tablas anfitrionas:

primero debe mantenerse el núcleo del recurso actual

las capacidades transversales o invitadas deben conservar jerarquía secundaria

la actividad operativa debe mantener lectura compacta y no competir con el recurso principal

<<SECTION: REGLAS DEL TRAIL>>
SECTION_VERSION: 00008

REGLAS DEL TRAIL

El trail representa contexto externo entre recursos y pantallas.

No reemplaza estados internos del host.

Reglas:

cada salto relevante debe poder reconstruir origen útil
no usar trail como contenedor ambiguo de todo estado UI
preservar breadcrumb comprensible
evitar cadenas innecesarias o redundantes
el destino final debe coincidir con la intención original del usuario

Cuando además exista contexto interno del host, debe complementarse con return_tab u otro mecanismo explícito equivalente.

Distinción doctrinal:

trail = contexto externo entre recursos
return_tab = contexto interno de una pantalla anfitriona

Caso vigente:

acción disparada desde una tab de orders.show puede requerir ambos:
trail para volver a la order
return_tab para restaurar el panel correcto

Regla adicional:

las transiciones especiales de estado no deben confundirse con edición ordinaria.

Si una acción cambia estado terminal o readonly hacia uno operable, debe tratarse como acción específica del flujo y no como simple navegación a edit.

Caso vigente:

reapertura closed → approved en orders.

Regla adicional sobre tabs host:

return_tab debe usarse para conservar el contexto visual interno de una show page cuando el host posea tabs principales.

El host es responsable de validar que la key recibida exista dentro de sus tabs disponibles.

Si la key no existe, debe resolver un fallback seguro sin romper la navegación.

<<END SECTION>>

<<SECTION: MAPA DE NAVEGACIÓN PRINCIPAL>>
SECTION_VERSION: 00006

MAPA DE NAVEGACIÓN PRINCIPAL

La navegación principal del sistema organiza acceso a frentes funcionales y no solo a tablas CRUD.

Entradas vigentes del ecosistema:

dashboard
projects
tasks
appointments
parties
products
assets
orders
documents
inventory

Reglas:

cada entrada principal debe corresponder a un frente reconocible del negocio
dashboard resume o deriva, no reemplaza módulos
los módulos con identidad propia deben tender a tener acceso claro
la navegación principal no obliga a que toda relación secundaria tenga menú propio

Aclaración adicional:

un módulo puede tener navegación principal y además publicar capacidades reutilizables consumidas dentro de otros hosts.

Aclaración sobre hosts avanzados:

la centralidad de un módulo en el menú no lo convierte automáticamente en patrón rector de estructura show.

La madurez contractual de cada host se evalúa por su propia gramática de montaje.

Caso vigente:

appointments.show expresa una estructura host más avanzada que otros shows aún en transición.

Aclaración vigente sobre OperationalActivity:

OperationalActivity no forma parte del mapa de navegación principal vigente.

Su base técnica puede existir sin abrir una entrada principal.

Quedan como posibles lecturas futuras:

vista general desde perfil de empresa
vista filtrada desde parties.show para employee con membership activa vinculada

Estas lecturas no equivalen a navegación principal salvo decisión futura validada.

<<END SECTION>>

<<SECTION: INVENTORY>>
SECTION_VERSION: 00009

INVENTORY

La navegación principal del frente inventory debe apoyarse en sus surfaces oficiales y no en accesos residuales dispersos.

Superficies vigentes:

inventory.index
inventory.show

Lectura esperada:

inventory.index como tablero operativo de saldos y acceso a productos

inventory.show como historial y detalle contextual de movimientos para un producto

Reglas:

los consumidores externos deben enlazar a inventory mediante estas surfaces cuando requieran lectura completa

si el caso requiere solo contexto breve, debe priorizarse embebido oficial o acción compacta

las vistas filtradas secundarias deben tender a reutilizar surfaces canónicas y no abrir pantallas paralelas innecesarias

Cuando inventory opere dentro de orders.show:

la navegación no debe romper continuidad del host salvo intención explícita del usuario

las acciones por línea deben permanecer en contexto local cuando sea posible

la apertura de historial filtrado puede usar surface oficial de inventory conservando trail de retorno

return_tab puede conservar el panel host correspondiente cuando la operación nació dentro de una tab principal

Cuando inventory opere dentro de documents.show:

la navegación no debe romper continuidad del host documental salvo intención explícita del usuario

las acciones por línea documental deben permanecer en contexto local cuando sea posible

la apertura de historial filtrado por document_item puede usar surface oficial de inventory conservando trail de retorno

return_tab puede conservar el panel host correspondiente cuando la operación nació dentro de una tab principal

Regla adicional por estado superior:

si la order anfitriona está closed o cancelled, inventory no debe exponer desde ese host acciones operativas incompatibles con dicho estado.

Aclaración adicional vigente:

la incorporación de inventory_operations no crea por sí misma una navegación principal nueva ni una pantalla obligatoria separada.

La operación se incorpora como capa contextual interna del frente inventory.

El flujo visible del usuario puede seguir centrado en acciones como surtir, devolver, ingresar o consultar movimientos.

Internamente, Inventory puede resolver o reutilizar una agrupación operativa contextual y luego registrar uno o múltiples movimientos unitarios de stock.

Cuando un movimiento tenga operación asociada, inventory.movements.show puede exponer esa información como lectura contextual adicional sin desplazar la navegación principal hacia la operación.

REGLA PARA LINKS RELACIONALES DESDE TABLAS CONTEXTUALES

Cuando una tabla enlaza desde una entidad fila hacia una entidad relacionada, debe evaluarse cuál es el contexto real que el breadcrumb necesita preservar.

Regla práctica:

si el trabajo sigue anclado en la fila desde la que se abrió el vínculo, el breadcrumb esperado puede ser:

módulo origen
↓
entidad fila
↓
entidad relacionada

Esto aplica especialmente en:

tablas embebidas dentro de show

listados contextuales dentro de tabs

acciones por fila hacia entidades relacionadas

sublistados donde el contenedor no alcanza por sí solo para conservar el contexto de origen

Regla adicional consolidada:

en una tabla contextual, el link principal de la fila debe consolidar primero el trail completo de esa fila

los links relacionales secundarios de esa misma fila deben reutilizar ese trail ya construido y no reiniciar contexto desde el contenedor

Regla adicional para integraciones reusables:

si la fila expone una capacidad compleja ya publicada por otro módulo, no debe reconstruirse localmente por conveniencia dentro de la tabla

debe consumirse la capacidad oficial del módulo oferente también en esa superficie contextual

Casos vigentes relevantes:

linked-order-action
linked-party-action
linked-asset

Regla adicional para tablas anfitrionas con row actions:

si la tabla pertenece al host pero expone micro-acciones de otro módulo, los links y acciones secundarios deben respetar al mismo tiempo:

contexto de fila
trail del host
retorno interno del panel cuando aplique

Esto evita que la navegación termine desviando al usuario hacia el recurso delegado como destino final de trabajo.

DETALLE DE FLUJOS RELEVANTES

TASKS → ORDERS

La tarea puede derivar una orden dentro del flujo operativo.

El usuario conserva contexto mediante trail.

APPOINTMENTS → ORDERS

El turno puede originar una orden contextual asociada a la atención planificada.

ASSETS → APPOINTMENTS

Desde un activo puede consultarse o gestionar agenda relacionada.

ASSETS → ORDERS

Desde un activo puede abrirse historial operativo o crear nueva orden contextual.

ASSETS → DOCUMENTS

Desde un activo puede abrirse documentación relacionada cuando corresponda.

PARTIES → APPOINTMENTS

Desde un contacto puede consultarse agenda asociada.

PARTIES → ORDERS

Desde un contacto puede abrirse o crear orden contextual.

PROJECTS → TASKS

El proyecto actúa como agrupador y punto de acceso contextual a tareas relacionadas.

ORDERS → INVENTORY POR LÍNEA

La orden puede consumir inventory como frente operativo embebido sobre sus líneas.

El usuario puede ejecutar acciones materiales contextualizadas sin abandonar orders.show cuando el flujo lo permita.

Inventory conserva ownership funcional de la operación, el cálculo de stock, las validaciones materiales y la trazabilidad.

La capa inventory_operations permite agrupar contextualmente uno o múltiples movimientos generados dentro de una misma acción operativa temporal.

Esta agrupación no reemplaza la trazabilidad individual de inventory_movements ni modifica la condición de orders como host visual consumidor.

ORDERS → DOCUMENTS

La orden puede derivar documentos cuando corresponda al flujo operativo.

DOCUMENTS → ORDERS

El documento puede conservar relación con la orden que le dio origen cuando exista derivación funcional.

PRODUCTS → INVENTORY

El producto actúa como catálogo maestro para la lectura de stock, saldos e historial.

Inventory no debe confundirse con products.

Products ofrece el artículo.

Inventory registra la operación material sobre ese artículo.

<<SECTION: CRITERIOS DE AUDITORÍA FUTURA>>
CRITERIOS DE AUDITORÍA FUTURA

Al auditar breadcrumb, trail o navegación entre módulos, verificar:

si la pantalla debía abrirse en modo estructural o contextual

si el trail se propagó correctamente

si hubo colapso de nodos repetidos

si volver y cancelar respetan el nodo previo real

si redirects post acción mantienen el contexto útil

si el breadcrumb visible sigue siendo compacto

si navbar y sidebar marcan el área funcional esperada y no solo la hoja final abierta

si en tablas contextuales el trail correcto era el del contenedor o el de la fila

si hay pantallas donde el contexto se pierde por filtros, tabs, paginación o acciones secundarias

si los enlaces, acciones y campos relacionales hacia módulos relacionados verifican primero que el módulo destino esté habilitado para el tenant

si la deshabilitación de módulos destino elimina completamente:

accesos
bloques visuales
referencias indirectas

especialmente en flujos desde appointments hacia orders y assets

<<SECTION: CRITERIO FINAL>>
SECTION_VERSION: 00002

CRITERIO FINAL

La navegación correcta del sistema no es la que muestra más enlaces.

Es la que permite al usuario llegar de forma clara, segura y coherente a la acción que necesita.

Una relación visible entre módulos se justifica cuando:

mejora continuidad operativa
reduce fricción real
respeta permisos
respeta módulos habilitados
mantiene contexto útil
no duplica caminos innecesarios

Si una relación existe técnicamente pero no cumple estos criterios, no debe priorizarse como navegación visible.
