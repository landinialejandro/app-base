NATURALEZA DE PROJECT LAB

Project Lab es la estación operativa local de desarrollo, auditoría, validación y mantenimiento técnico del proyecto.

Ubicación vigente:

tools/project-lab

Project Lab no forma parte del sistema SaaS productivo.

No debe publicarse ni exponerse en entornos accesibles desde internet.

Su uso corresponde únicamente en entorno local controlado.

Funciones principales:

ejecución de Tinker
ejecución de Tinker desde clipboard
ejecución de comandos Artisan permitidos
herramienta Lab de código
herramienta Lab documental
herramienta Lab de auditoría
snippets rápidos
búsqueda operativa de métodos o fragmentos
consola unificada
guardado de consola como auditoría
copiado de salida para compartir evidencia
persistencia auxiliar local cuando corresponda
procesamiento local de cola pendiente cuando se requiera validar eventos queued

Project Lab debe leerse como herramienta técnica local del proyecto.

No debe confundirse con una funcionalidad del producto final.

Reglas:

Project Lab no define reglas de negocio.
Project Lab no define autorización del producto.
Project Lab no define navegación del usuario final.
Project Lab no reemplaza documentación rectora.
Project Lab no reemplaza criterio arquitectónico.
Project Lab no reemplaza validación runtime.
Project Lab no reemplaza pruebas en navegador cuando correspondan.

Project Lab define únicamente el contrato operativo local para aplicar, auditar, ejecutar y validar tareas técnicas sobre el proyecto.

Regla específica para colas:

Project Lab puede facilitar el procesamiento manual de colas en local cuando sea necesario validar eventos queued, jobs y persistencia final.

Esta capacidad no debe trasladarse como botón, acción o pantalla dentro del producto SaaS.

HERRAMIENTAS DISPONIBLES

Project Lab dispone de tres herramientas Lab principales:

code
docs
audit

CODE

Se utiliza para aplicar actualizaciones de código compatibles con los contratos vigentes de Project Lab.

Puede operar sobre:

archivos PHP completos
archivos Blade completos
archivos CSS completos
archivos JS completos
métodos PHP existentes
métodos PHP nuevos dentro de clases existentes
funciones JS existentes
secciones CSS o JS delimitadas

Reglas:

debe trabajar siempre dentro del projectRoot actual
debe resolver rutas mediante la aduana central de rutas vigente
debe escribir archivos mediante la aduana central de escritura vigente
debe rechazar rutas fuera del proyecto
debe validar antes de escribir cuando el error sea detectable
debe validar sintaxis PHP cuando corresponda
no debe aplicar parches ambiguos
no debe mezclar contratos de archivo completo con contratos parciales

DOCS

Se utiliza para aplicar actualizaciones documentales mediante bloques completos de sección.

Debe operar con:

encabezado REEMPLAZAR EN: [doc_slug]
doc_slug existente
nombre exacto de sección
bloque completo delimitado
SECTION_VERSION
contenido compatible con el documento destino

Reglas:

no debe inventar secciones
no debe reemplazar fragmentos internos
no debe aplicar cambios sin SECTION_VERSION
no debe aceptar nombres aproximados
no debe mezclar explicación narrativa con bloque automatizado
debe devolver OK o ERROR explícito

AUDIT

Se utiliza para ejecutar auditorías locales, inspecciones y comandos de apoyo sobre el proyecto real.

Lectura vigente:

audit ejecuta bash libre local desde projectRoot.

No es sandbox del sistema operativo.

Su uso queda a riesgo del usuario técnico local.

Reglas:

debe ejecutarse desde el projectRoot actual
debe utilizarse para obtener evidencia real
puede ejecutar auditorías fuertes cuando el usuario técnico lo solicite
no debe endurecerse con bloqueos heurísticos que rompan auditorías necesarias
no debe tratarse como protección de sistema operativo
no debe ejecutar prompts de chat como si fueran comandos

Uso esperado:

inspeccionar archivos reales
buscar métodos o residuos
generar auditorías en documentos/auditoria
validar rutas
validar contratos de código
obtener evidencia antes de intervenir

Regla general:

Project Lab no decide qué cambio es correcto.

Project Lab ejecuta, valida, registra y muestra salida operativa.

La interpretación técnica sigue dependiendo del proceso de trabajo, la documentación vigente, el código real y la evidencia obtenida.

REGLA ESTRICTA DE FORMATOS

Project Lab debe rechazar formatos ambiguos.

Un snippet solo puede aplicarse si coincide claramente con un contrato válido.

Contratos válidos principales:

archivo PHP completo
archivo Blade completo
archivo CSS completo
archivo JS completo
reemplazo de método PHP
alta de método PHP
reemplazo de función JS
reemplazo de sección CSS o JS
reemplazo de sección documental

Regla crítica:

Project Lab debe fallar antes de escribir cuando el error sea detectable.

Si un bloque no cumple exactamente un contrato válido:

no debe aplicar cambios
no debe crear archivos
no debe crear directorios
no debe escribir fuera del projectRoot
no debe normalizar silenciosamente casos peligrosos
debe devolver ERROR explícito

No se aceptan cambios ambiguos como éxito parcial.

No se aceptan escrituras preventivas para luego validar.

Orden obligatorio:

validar contrato
resolver ruta segura dentro del projectRoot
validar existencia cuando corresponda
aplicar cambio
validar resultado cuando corresponda
mostrar salida operativa

Regla:

un formato pensado para archivo completo no debe usarse para reemplazar métodos.

Un formato pensado para operación parcial no debe usarse para crear archivos completos.

Un bloque documental no debe usarse para modificar código.

Un bloque de código no debe usarse para modificar documentación.

<<SECTION: REGLA ANTIAMBIGÜEDAD EN HEADERS>>
SECTION_VERSION: 00002

REGLA ANTIAMBIGÜEDAD EN HEADERS

Un encabezado FILE identifica archivo completo.

Un encabezado TARGET identifica operación parcial.

Regla crítica:

FILE no debe contener operadores de método.

No son válidos como archivo completo los encabezados que mezclen ruta de archivo con operadores de método.

Estos casos deben resolverse de una sola manera:

si corresponde operación parcial, debe usarse TARGET
si corresponde archivo completo, debe usarse FILE sin operadores parciales
si el parser no puede clasificarlo con seguridad, debe devolver ERROR

Nunca debe crearse un archivo cuya ruta incluya operadores de método.

Regla obligatoria:

si un header FILE contiene operadores parciales, Project Lab no puede tratarlo como archivo completo.

Debe devolver ERROR explícito o normalizarlo hacia TARGET solo si el parser lo reconoce de forma segura como operación parcial válida.

Regla recomendada:

para operaciones parciales usar siempre TARGET.

Para archivos completos usar siempre FILE.

No mezclar ambos contratos en un mismo snippet.

<<END SECTION>>

<<SECTION: ERRORES OBLIGATORIOS>>
SECTION_VERSION: 00004

ERRORES OBLIGATORIOS

Project Lab debe devolver ERROR y no escribir cambios cuando detecte fallas validables antes de la escritura.

Casos obligatorios:

contenido vacío
herramienta Lab no reconocida
header no compatible
ruta destino vacía
ruta fuera del projectRoot
ruta con operadores inválidos para archivo completo
archivo inexistente en operación parcial
método inexistente en reemplazo parcial
método ya existente en alta de método
función JS inexistente en reemplazo parcial
sección documental inexistente
doc_slug inexistente
sección CSS o JS inexistente
intento de usar formato de archivo completo para operación parcial ambigua
bloque documental sin encabezado REEMPLAZAR EN cuando la herramienta vigente lo requiere
bloque documental con DOC_SLUG como encabezado operativo si Project Lab Docs espera REEMPLAZAR EN
header FILE que combine operadores de método
snippet que mezcle FILE y TARGET de forma ambigua

Regla:

un error debe ser explícito, legible y accionable.

Debe indicar:

qué falló
por qué falló
qué formato correcto corresponde usar cuando aplique

Regla adicional:

un ERROR no debe dejar cambios parciales aplicados cuando la operación pueda validarse antes de escribir.

Regla sobre sintaxis PHP:

si la validación de sintaxis PHP posterior a un cambio falla, la intervención no debe considerarse cerrada.

Debe corregirse o revertirse según corresponda.

Regla adicional sobre errores posteriores a la aplicación:

si la herramienta aplicó un cambio pero una validación posterior falla, el resultado no debe tratarse como cierre exitoso.

Debe corregirse o revertirse antes de avanzar.

<<END SECTION>>

<<SECTION: SALIDA OPERATIVA ESPERADA>>
SECTION_VERSION: 00009

SALIDA OPERATIVA ESPERADA

Toda ejecución de Project Lab debe producir salida clara.

Estados esperados:

OK
WARN
ERROR
INFO

OK

Indica operación aplicada correctamente.

Ejemplos:

archivo actualizado
método reemplazado
documento actualizado
auditoría generada
sintaxis PHP válida

WARN

Indica que la operación se aplicó o se procesó, pero existe una advertencia relevante.

ERROR

Indica que la operación no debe considerarse aplicada correctamente.

INFO

Indica orientación complementaria.

Reglas:

no debe considerarse cierre técnico una salida sin OK verificable

no debe considerarse seguro un cambio aplicado con errores no resueltos

una salida OK de aplicación no reemplaza validaciones posteriores necesarias para el frente trabajado

si Project Lab informa OK pero una validación posterior falla, la intervención no está cerrada

Las validaciones posteriores dependen del tipo de cambio.

Pueden incluir:

php -l
optimize:clear
route:list
migrate:fresh --seed
Tinker
auditoría
grep preciso
validación en navegador
jobs
failed_jobs
persistencia final
rollback o limpieza de prueba
smoke test del servicio o modelo afectado

Regla:

la consola debe permitir reconstruir qué se ejecutó, qué resultado produjo y qué validación quedó pendiente si corresponde.

USO DE TINKER DENTRO DE PROJECT LAB

Tinker es una capacidad interna de Project Lab para reproducir comportamiento real del sistema.

Debe usarse especialmente para:

permisos efectivos
constraints
tenant activo
Security
datasets visibles
relaciones activas
validaciones backend
flujos controlados
auditoría de runtime
eventos operativos
validación de persistencia derivada
requests fabricados
controllers con autorización
rollback de pruebas sensibles

Regla crítica:

en el editor Tinker de Project Lab debe pegarse solo código PHP interno.

No debe pegarse el comando completo:

php artisan tinker --execute

Project Lab ya ejecuta internamente el proceso correspondiente contra el proyecto actual.

Reglas:

si se quiere visualizar el resultado de una expresión, debe utilizarse return o salida explícita

si una salida no muestra el resultado esperado, puede requerirse echo, dump o equivalente

si un bloque falla por el uso de use, no debe asumirse automáticamente que falló la lógica funcional

ante errores de wrapper, puede usarse namespace completo

cuando una prueba dispare eventos queued, Tinker solo valida el disparo inicial

si la prueba requiere persistencia final, debe procesarse cola y verificar resultado

Regla para pruebas sensibles:

si la prueba involucra permisos, roles, memberships, tenant/profile o acciones críticas, debe utilizarse usuario real, tenant real, autenticación real y rollback o limpieza cuando corresponda.

Regla:

Tinker ayuda a reproducir runtime.

No reemplaza navegador cuando el cierre dependa de interacción visual.

<<END SECTION>>

<<SECTION: REGLA DE TENANT ACTIVO EN TINKER>>
SECTION_VERSION: 00003

REGLA DE TENANT ACTIVO EN TINKER

En app-base, muchas resoluciones dependen de:

app('tenant')

Por lo tanto, una prueba Tinker no reproduce correctamente el sistema si no existe tenant activo bindeado en el contenedor.

Regla operativa:

antes de probar permisos, visibilidad, queries tenant-aware o lógica que consume Security, debe resolverse y bindearse un tenant activo real.

Secuencia esperada:

obtener un usuario real
obtener una membership activa real de ese usuario
tomar el tenant de esa membership
bindear el tenant en el contenedor

Sin ese paso, la prueba puede:

fallar artificialmente
devolver falsos negativos
no representar el comportamiento real del sistema

Regla adicional:

no fijar tenant_id como número ni como dato estable copiado desde otra sesión.

Debe obtenerse desde runtime real.

Regla:

tenant activo bindeado no es solo comodidad de prueba.

Es condición necesaria para reproducir el comportamiento real de capacidades tenant-scoped.

<<END SECTION>>

<<SECTION: REGLA DE AUTORIZACIÓN EN TINKER>>
SECTION_VERSION: 00002

REGLA DE AUTORIZACIÓN EN TINKER

La autorización debe probarse desde la capa central vigente.

Regla operativa:

para permisos, visibilidad o acciones permitidas, debe usarse Security o el contrato central correspondiente.

Usos válidos:

inspect
allows
authorize
scope

No probar permisos reconstruyendo lógica manualmente.

No probar permisos desde UI como fuente final.

No inferir permisos desde campos visibles.

Regla adicional:

si el módulo utiliza constraints, la prueba debe incluir el contexto funcional real requerido.

Ejemplos:

si la capacidad requiere kind, probar con kind explícito

si la capacidad requiere una relación funcional, probar contra esa relación real

si la capacidad no requiere contexto, no inventarlo

Regla:

la prueba debe reproducir el contrato de seguridad real, no una versión simplificada creada para pasar.

La salida debe interpretarse según el frente técnico que se está revisando.

Regla adicional:

si una auditoría muestra una referencia residual, debe analizarse si es:

código activo
comentario
documentación
backup
log
archivo generado
residuo real

No se debe corregir sin distinguir el tipo de hallazgo.

Regla adicional:

el grep debe ser preciso.

No debe confundirse una coincidencia textual válida con un residuo legacy real.

Regla para cierres de fase:

antes de declarar cerrado un frente de saneamiento, la auditoría debe incluir las piezas relevantes del frente intervenido.

La lista concreta depende del caso.

Project Lab ayuda a obtener evidencia.

No reemplaza el análisis técnico de esa evidencia.

USO DE CODE

La herramienta code debe usarse para aplicar cambios de código cuando el snippet cumpla un contrato válido.

Reglas:

usar archivo completo solo cuando se quiera reemplazar el archivo completo
usar TARGET con operador de reemplazo para reemplazar método PHP existente o función JS existente

usar secciones para CSS o JS cuando corresponda

no usar formatos ambiguos
no aplicar snippets generados sin revisar el encabezado

Si el objetivo es reemplazar un archivo completo, el snippet debe usar FILE.

Si el encabezado mezcla ambas intenciones, Project Lab debe devolver ERROR o normalizar de forma explícita y segura cuando el contrato lo permita.

Nunca debe crear un archivo nuevo por interpretar mal una operación parcial.

VALIDEZ DE RESPUESTAS DESTINADAS A PROJECT LAB

Cuando una respuesta del chat entregue contenido para Project Lab, debe cumplir simultáneamente tres condiciones:

coherencia doctrinal
compatibilidad técnica con la herramienta
aplicabilidad real sobre el proyecto vigente

No alcanza con que el contenido sea conceptualmente correcto.

No alcanza con que el objetivo sea adecuado.

No alcanza con que el bloque parezca razonable.

Si el bloque no puede ser procesado por Project Lab, la respuesta debe considerarse incorrecta para el flujo operativo del proyecto.

Reglas:

para código, usar únicamente encabezados compatibles con la herramienta de código vigente

para documentación, usar únicamente el formato documental aceptado por Project Lab

para auditorías, entregar comandos compatibles con la herramienta audit vigente

para Tinker, entregar solo código interno PHP cuando se use el editor Tinker

para Artisan, entregar comandos Artisan sin envolverlos en ejecución externa innecesaria

Si se desconoce el formato exacto, no se debe entregar el bloque final.

Primero debe pedirse evidencia o auditarse el contrato real.

Regla crítica:

una respuesta no debe obligar al usuario a corregir el formato para que la herramienta lo acepte.

Regla crítica para documentación:

antes de entregar bloques documentales destinados a Project Lab, debe verificarse que:

DOC_SLUG sea exacto
el nombre de sección exista exactamente si se trata de reemplazo
SECTION_VERSION corresponda al incremento correcto
el bloque sea completo
no se usen nombres conceptuales o aproximados
no se creen secciones nuevas salvo mecanismo explícito compatible

Si el nombre exacto de sección no está confirmado, no debe generarse el bloque final.

Debe solicitarse el listado de secciones o auditarse el documento vigente.

Regla:

una sección con buen contenido pero nombre incorrecto es inválida para Project Lab.

No debe considerarse entrega útil.

Regla adicional:

los errores de aplicación devueltos por Project Lab deben incorporarse al análisis posterior.

Si una sección no se encontró, no debe insistirse con el mismo nombre ni normalizarlo por intuición.

Debe buscarse el tag real vigente del documento y entregar nuevamente el bloque con nombre exacto.

<<SECTION: FALLAS OPERATIVAS QUE PROJECT LAB DEBE PREVENIR>>

FALLAS OPERATIVAS QUE PROJECT LAB DEBE PREVENIR

Project Lab debe prevenir errores derivados de ambigüedad, formato incorrecto o interpretación excesivamente flexible.

Casos que deben considerarse fallas:

crear un archivo cuando la intención era reemplazar un método
interpretar como ruta completa un encabezado que contiene operadores de método
aceptar un bloque documental sin formato real válido
crear secciones documentales inexistentes sin autorización explícita
aplicar snippets incompletos
mezclar contratos de archivo completo y operación parcial
procesar silenciosamente formatos dudosos
devolver OK ante una operación que no cumplió el contrato esperado
escribir fuera del projectRoot
ejecutar texto narrativo como comando
ocultar el contenido ejecutado desde clipboard

Regla:

ante ambigüedad, Project Lab debe fallar.

Regla complementaria:

ante ambigüedad, el chat también debe detenerse y pedir o auditar la evidencia mínima necesaria.

No se debe resolver por intuición.

Regla:

una ejecución desde clipboard debe dejar visible la entrada efectivamente ejecutada cuando la herramienta lo permita.

La consola muestra la salida.

El textarea debe mostrar la entrada.

Ninguna ejecución desde clipboard debe quedar oculta para el usuario.

DOCTRINA OPERATIVA DE PROJECT LAB

Project Lab no es una herramienta accesoria ni un simple panel de utilidades.

Dentro del proceso técnico del proyecto, Project Lab funciona como estación operativa local para ejecutar intervenciones controladas.

Project Lab gobierna operativamente:

cómo se aplican snippets de código
cómo se aplican actualizaciones documentales
cómo se ejecutan auditorías simples
cómo se reproducen pruebas Tinker
cómo se ejecutan comandos Artisan controlados
cómo se comparte evidencia técnica en el chat
cómo se guarda una salida de consola como auditoría
cómo se valida si una intervención fue aplicada por la herramienta

Regla central:

toda salida destinada a Project Lab debe respetar el contrato real vigente de Project Lab.

Si el contenido entregado no respeta el formato aceptado por la herramienta, ese contenido es inválido aunque la intención conceptual sea correcta.

Reglas:

no improvisar formatos
no asumir que una variante será aceptada
no mezclar contratos de archivo completo y operación parcial
no entregar bloques documentales sin formato real compatible
no transformar una convención deseada en contrato real sin validación previa
no usar Project Lab como excusa para aplicar doctrina no auditada

Cuando exista duda sobre el contrato de entrada de Project Lab, debe auditarse primero la herramienta o solicitarse la evidencia mínima necesaria.

Project Lab debe ser estricto.

El chat también debe ser estricto cuando genera contenido para Project Lab.

Regla de frontera documental:

esta sección no debe incorporar doctrina funcional extensa de módulos del sistema.

Solo debe documentar reglas de uso de la herramienta.

<<END SECTION>>

VALIDACIONES POSTERIORES A CAMBIOS

Después de aplicar código, debe validarse según el tipo de cambio.

Validaciones posibles:

php -l sobre archivos PHP modificados
php artisan optimize:clear
php artisan route:list cuando toque rutas
Tinker cuando toque runtime, permisos, constraints o datos reales
navegador cuando toque UI
grep cuando se eliminen residuos legacy
auditoría simple cuando se requiera evidencia transversal
migrate:fresh --seed cuando el cambio afecte arquitectura, migraciones, seeders o contratos estructurales
smoke test del modelo o servicio afectado cuando el cambio sea de saneamiento modular

Regla:

no declarar cierre por intuición.

Un cierre técnico debe apoyarse en evidencia suficiente.

Regla adicional:

si Project Lab informa OK pero la validación posterior falla, el frente no está cerrado.

Debe corregirse o revertirse según corresponda.

Regla adicional para saneamiento:

si se elimina o reemplaza una dependencia activa, debe validarse que no queden consumidores activos usando la vía anterior.

Regla final:

las validaciones deben demostrar que el patrón nuevo opera y que el patrón anterior no quedó como fallback activo indebido.

<<SECTION: SEGURIDAD DE USO>>
SECTION_VERSION: 00003

SEGURIDAD DE USO

Project Lab debe permanecer limitado al entorno local.

Reglas:

no exponer Project Lab públicamente
no usar Project Lab como funcionalidad productiva
no compartir URLs externas de Project Lab
no relajar validaciones por comodidad
no ejecutar código desconocido sin revisión
no pegar snippets destructivos sin entender alcance
no asumir que una salida visual equivale a autorización efectiva
no confundir herramienta local con permiso del producto

Límite operativo vigente:

Project Lab debe operar dentro del root del proyecto actual.

Toda escritura persistente controlada por Project Lab debe resolverse contra el projectRoot.

Project Lab puede trabajar sobre distintos proyectos locales, Laravel o no Laravel.

Las capacidades Laravel son herramientas disponibles cuando el proyecto las soporta.

El límite de seguridad operativo es el projectRoot actual, no Laravel.

Regla sobre audit:

la herramienta audit ejecuta bash libre local desde el projectRoot.

Audit no constituye un sandbox del sistema operativo.

El usuario técnico local es responsable del contenido ejecutado.

No debe ejecutarse como audit un prompt de chat, texto narrativo o contenido no pensado como comando.

Regla sobre Tinker:

Tinker ejecuta código PHP sobre el proyecto actual.

No es sandbox.

Debe utilizarse únicamente en entorno local controlado y con código comprendido.

Regla sobre Artisan:

Artisan debe ejecutarse mediante comandos permitidos por Project Lab.

No debe usarse como shell libre.

Regla:

la herramienta facilita la operación técnica, pero no elimina la obligación de seguridad.

<<SECTION: CRITERIO DE CIERRE OPERATIVO>>

CRITERIO DE CIERRE OPERATIVO

Una intervención realizada con Project Lab se considera cerrada solo cuando existe evidencia suficiente.

Según el caso, esa evidencia puede incluir:

salida OK de Project Lab
validación de sintaxis
auditoría simple
Tinker runtime
Security real
grep de residuos
optimize:clear
route:list
migrate:fresh --seed
smoke test del modelo o servicio afectado
validación visual en navegador
actualización documental

No alcanza con que el snippet haya sido aceptado.

No alcanza con que no aparezca un error visual.

No alcanza con que la UI oculte una acción.

No alcanza con revisar solo una pieza si el frente incluye varias piezas conectadas.

Regla:

el cierre debe demostrar que el sistema quedó más coherente, no solo que el cambio fue aplicado.

Regla adicional para cierre de fase:

una fase no debe declararse cerrada sin auditar las piezas relevantes del frente.

Las piezas mínimas dependen del alcance real del cambio.

Pueden incluir:

controller principal
controllers hijos relevantes
modelo
helpers
surface services
hooks
services o support vinculados
catálogos
rutas principales
validaciones runtime
smoke tests
grep de residuos conocidos

Regla final:

cerrar una intervención significa que el sistema queda más coherente, autónomo, seguro y alineado con su doctrina vigente.

<<SECTION: ERRORES Y LECCIONES INCORPORADAS>>
SECTION_VERSION: 00004

ERRORES Y LECCIONES INCORPORADAS

Lección incorporada:

un snippet ambiguo puede generar daño si el parser interpreta como ruta lo que en realidad era una operación parcial.

Regla consolidada:

Project Lab debe validar antes de escribir.

FILE corresponde a archivo completo.

TARGET corresponde a operación parcial.

Si aparece operador de método dentro de FILE, debe fallar o normalizarse explícitamente hacia TARGET de forma segura.

Nunca debe crear un archivo con operadores parciales en la ruta.

Lección incorporada adicional:

una ejecución desde clipboard no debe ser invisible para el usuario.

Regla consolidada:

cuando Project Lab ejecute desde clipboard, debe devolver y mostrar el contenido efectivamente leído.

Esto aplica especialmente a:

code
docs
audit
Tinker desde clipboard

Regla operativa:

el textarea correspondiente debe reflejar el contenido ejecutado desde clipboard.

La consola muestra la salida.

El textarea muestra la entrada.

Ninguna ejecución desde clipboard debe quedar oculta para el usuario.

Lección incorporada adicional:

Project Lab no debe recibir texto narrativo como si fuera script ejecutable.

Regla:

las respuestas narrativas no son comandos.

Para Project Lab deben enviarse exclusivamente:

comandos shell válidos
patches completos compatibles
formatos aceptados por el automatizador
bloques documentales válidos
snippets de auditoría realmente ejecutables

Lección incorporada adicional:

un cierre técnico no debe apoyarse solo en una pieza aislada cuando el frente involucra varias capas.

Regla:

si una fase declara saneamiento modular, Project Lab debe ayudar a producir evidencia sobre todas las piezas relevantes, no solo aplicar un patch parcial.
