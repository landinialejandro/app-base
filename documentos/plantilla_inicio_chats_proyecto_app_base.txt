Inventory ya opera integrado con Orders y Documents mediante surfaces oficiales y mantiene ownership funcional propio.

Inventory cuenta con inventory_operations como capa interna de agrupación operativa contextual.

InventoryOperation representa la acción operativa agrupada.

InventoryMovement conserva la trazabilidad fina y atómica de stock.

Estado base reciente de contrato material Inventory:

Inventory incorpora contrato material formal mínimo mediante InventoryMaterialFlow.

InventoryMovement conserva el libro físico de stock.

InventoryMaterialFlow clasifica el subcontrato material formal.

Tipos formales vigentes:

formal_delivery
formal_application
formal_return

InventoryMaterialFlow no reemplaza InventoryMovement.

InventoryMaterialFlow no mueve stock por sí mismo.

InventoryMaterialFlowService crea y valida flows formales.

InventoryMaterialBalanceService lee saldos formales y conserva lectura ambigua para movimientos anteriores o no formalizados.

InventorySurfaceService publica payload de Material formal para surfaces embebidas.

InventoryOrderItemHooks bloquea cierre si queda saldo material formal pendiente.

ProductionOrderInventoryOperationService soporta producción instantánea y producción formal por etapas.

Producción instantánea sigue vigente cuando no existen flows formales pendientes.

Producción instantánea debe bloquearse si existen entregas formales pendientes de componentes físicos contra la misma orden/línea.

La entrega formal puede crear movimiento físico de salida y flow formal_delivery.

La aplicación formal crea flow formal_application y no crea InventoryMovement.

La aplicación formal no descuenta stock físico dos veces.

La devolución formal puede crear movimiento físico de ingreso y flow formal_return.

Aplicación y devolución no pueden superar saldo entregado disponible.

Inventory conserva ownership material.

Orders conserva ownership de orden, líneas y lifecycle.

Orders no calcula saldos materiales.

Orders no administra ledger material formal.

Orders no reconstruye lógica de stock.

Orders no decide consumos ni aplicaciones materiales.

No usar InventoryMovement.kind ni metadata como sustituto del subcontrato material formal cuando se requiere saldo trazable.

Modelo mental validado:

el depósito entrega.

la orden consume o aplica.

el cierre regulariza.

Fase 1 cerrada:

lectura consolidada inicial + guard defensivo contra doble consumo.

Fase 2 cerrada:

ledger material formal InventoryMaterialFlow + InventoryMaterialFlowService.

Fase 3 operativa mínima cerrada:

acciones internas y UI mínima de entrega, aplicación y devolución formal.

Producción formal por etapas cerrada en corte funcional inicial:

con entregas formales, ingresa producto terminado y aplica componentes previamente entregados.

No crea consumo físico doble de componentes ya entregados.

Los componentes service no participan en consumo ni aplicación material física.

La cantidad producible se limita por el componente físico entregado disponible más restrictivo.

Caso validado:

si se entregan 50 franelas y 40 siliconas, producir 50 bloquea por silicona.

producir 40 ingresa terminado, aplica 40 componentes y deja 10 franelas disponibles.

Cierre material vigente:

el cierre de orden consulta blockers materiales de Inventory.

si queda saldo formal pendiente, el cierre se bloquea.

la acción Devolver debe permanecer disponible aunque la línea esté completed si la orden está operable y existe saldo formal disponible.

quedó sobrante

REGLA PARA ARCHIVOS COMPLETOS DE CÓDIGO

Cuando devuelvas un archivo completo, la primera referencia del archivo debe usar siempre este formato:

FILE: ruta/del/archivo.ext | VX

Reglas:

si el archivo es nuevo, usar V1
si el archivo ya existe y no tiene comentario de versión, al devolverlo completo agregarlo como V2
si el archivo ya tiene comentario de versión, incrementar en 1 la última versión existente
esta regla aplica solo cuando se devuelve el archivo completo
si se devuelve solo un fragmento o bloque parcial, no agregar ni modificar versión
el comentario debe escribirse con la sintaxis correcta del archivo correspondiente

Formato canónico adicional para PHP completo:

la salida debe respetar esta estructura exacta:

<?php

// FILE: ruta/del/archivo.php | VX

...resto del código...

Es decir:

- apertura PHP en primera línea
- una línea en blanco
- comentario FILE con versión
- una línea en blanco
- resto del archivo
